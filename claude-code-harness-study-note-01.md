## 0. 한 줄 요약

> Claude Code는 강력한 "엔진"이지만, 그것을 실제 프로젝트에 잘 부려먹으려면 **하니스(Harness)** 라는 마구(馬具)/배선층이 필요하다. 하니스는 `Skills + Agents + Commands` 3가지로 이루어진 설정 묶음이다.

비유: 모델 = 엔진 / 하니스 = 조향장치·브레이크·계기판이 갖춰진 완성차

## 2. 핵심 개념: 하니스가 왜 필요한가

화자의 비유 [00:00]:
> "여러분이 새 직원을 뽑았다고 생각해 보세요. 이 직원이 아무리 머리가 좋아도 첫날부터 알아서 일 잘하진 않잖아요."

- 범용 AI인 Claude는 **우리 회사·우리 팀·우리 프로젝트 방식을 모름**
- 매번 "우리는 FastAPI 써, 컨벤션은 이렇게…" 설명을 반복하면 비효율적
- **하니스는 그 설명들을 파일로 굳혀서 한 번에 주입하는 시스템**

### 보강(공식 문서)
- Claude Code는 **영구 기억이 없음** → 세션이 끊기면 다시 0부터
- 그래서 `CLAUDE.md`, `AGENTS.md`, `skills/`, `agents/` 같은 파일 묶음으로 컨텍스트를 **외부 저장**해두는 것이 표준 패턴
- `AGENTS.md`는 Linux Foundation에 기증된 **AAIF 표준** — Codex CLI, Cursor도 같은 파일을 네이티브로 읽음 (도구 바뀌어도 호환)

## 3. 하니스의 3대 구성 요소

| # | 구성요소 | 한 줄 정의 | 호출 방식 |
|---|---|---|---|
| 1 | **Skill (스킬)** | "이 분야 전문 지식으로 답해줘" — 가이드라인·베스트 프랙티스 문서 묶음 | `/스킬명 + 요청` 슬래시 명령어 |
| 2 | **Agent (에이전트)** | "이 문제 스스로 해결해줘" — 자율적으로 일하는 보조 직원 | 자동/명시적 호출 |
| 3 | **Command (커맨드)** | 자주 하는 복잡한 작업을 한 줄 명령으로 압축 | `/`로 시작 |

### 3-1. 결정적 차이: Skill vs Agent [03:23]
- **Skill** = **지식 제공자** (passive). "FastAPI 가이드라인" 같은 책을 펴서 보여줌
- **Agent** = **문제 해결자** (active). 주제 받고 알아서 검색·계획·실행

## 6. 화자의 핵심 추천/팁

### ⭐ 팀 규칙은 스킬 파일에 직접 추가 [04:38]
> 하니스를 기본 그대로 쓰지 말고 **우리 회사/팀 규칙**으로 커스터마이즈. 파일 하나만 수정해도 훨씬 강력해짐.

### ⭐ 나만의 에이전트 만들기 [04:44]
> 파일 하나 추가로 — 우리 서비스 특성 맞춤 디버거, 우리 회사 방식 코드 리뷰어 등 가능

→ **같은 설명을 반복할 필요가 없어지고, 매번 일관된 결과물**이 나온다는 점

## 7. ASUS ROG Zephyrus G16 (Windows + RTX 4080) 환경 셋업 가이드

> 영상이 Mac/Linux 기준 분위기여서, G16 환경에 맞춰 보강

### 7-1. 필수 사전 설치
```powershell
# Windows에서는 WSL2 + Ubuntu 사용을 강력 권장 (Claude Code는 Linux 환경에서 가장 안정적)
wsl --install -d Ubuntu-22.04
```

WSL2 안에서:
```bash
# Node.js (LTS)
curl -fsSL https://deb.nodesource.com/setup_20.x | sudo -E bash -
sudo apt install -y nodejs

# Git
sudo apt install -y git

# Claude Code 본체
npm install -g @anthropic-ai/claude-code
```

설치 확인:
```bash
claude --version
```

### 7-2. 첫 프로젝트 부트스트랩
```bash
mkdir my-project && cd my-project
claude              # 클로드 코드 실행
> /init             # CLAUDE.md 자동 생성 (프로젝트 규칙 골격)
```

### 7-3. 하니스 디렉터리 구조 (표준 패턴)
```
my-project/
├── .claude/
│   ├── agents/             # 에이전트 정의
│   │   ├── planner.md
│   │   ├── plan-reviewer.md
│   │   └── web-researcher.md
│   ├── skills/             # 스킬 정의
│   │   ├── fastapi-backend/
│   │   ├── nextjs-frontend/
│   │   └── mermaid-diagram/
│   └── commands/           # 슬래시 커맨드
├── CLAUDE.md               # Claude 전용 프로젝트 규칙
├── AGENTS.md               # AAIF 표준 (Codex/Cursor 호환)
└── docs/                   # 도메인별 상세 컨텍스트
```

### 7-4. RTX 4080을 살리는 방향 (Claude Code는 클라우드라 GPU 직접 안 씀)
GPU는 클로드 자체 추론에는 안 쓰이지만, 다음 부분에서 살릴 여지:
- **로컬 LLM 보조**: Codex/Cursor 대신 로컬 모델(예: Qwen2.5-Coder)을 보조 에이전트로 → Claude Code의 MCP 서버로 연결
- **병렬 테스트 실행**: 학부 때 들으신 ECE 408 패러다임 그대로, 빌드/테스트 잡을 GPU 가속 (CUDA 기반 단위테스트, PyTorch 모델 학습 루프) 으로 돌리고 Claude는 오케스트레이션만
- **임베딩 기반 코드 검색**: 코드베이스 전체를 GPU로 임베딩 → 벡터 DB → Claude의 컨텍스트 사전 검색

> 비유: ECE 391에서 배운 OS 스케줄러처럼, Claude를 **마스터 프로세스**로 두고 GPU는 **워커 풀**로 두는 그림

## 8. 영상 외부 보강 — 알아두면 좋은 것

### Claude Code의 컨텍스트 윈도우 구성 (공식 문서)
1. 대화 이력
2. 파일 내용
3. 명령 실행 결과
4. `CLAUDE.md`
5. **Loaded skills** ← 스킬은 여기 로드됨
6. System instruction

→ 컨텍스트 가득 차면 auto-compact로 오래된 도구 출력부터 삭제. 초기 지시사항이 날아갈 수 있으니 **중요 규칙은 CLAUDE.md에 박아둘 것**

### Subagent의 비밀
- 서브에이전트는 **메인 대화와 분리된 새 컨텍스트**를 가짐
- 작업 끝나면 **요약만 가져옴** → 긴 세션에서 컨텍스트 절약에 유리
- 영상의 "플래너 에이전트"가 이 패턴

### 안전장치 두 가지
1. **파일 수정 전 스냅샷** → `Esc` 두 번으로 롤백 가능
2. **권한 모드** → 위험한 작업 전 사용자 확인
