## 한 줄 요약
AI에게 일을 시키기 전에 "주문서(스펙)"를 먼저 완벽하게 작성하는 Spec Driven Development(SDD) 개념을 설명하고, 기존 SDD 도구(Kiro·Spec Kit·BMAD)의 아쉬운 점을 짚은 뒤 도구에 묶이지 않는 하니스 기반 워크플로우를 대안으로 제시한다.

## 핵심 개념

### 1. 바이브 코딩(vibe coding)의 한계
화자는 AI를 처음 쓸 때의 흔한 방식을 "바이브 코딩"으로 정의한다.

> "이거 만들어 줘. 아, 이거 좀 다르게 해 줘. (...) 이걸 개발자들이 바이브 코딩이라고 불러요. 느낌대로 흥이 가는 대로 AI한테 던지는 방식이에요." [00:26]

치명적 문제 두 가지를 짚는다:

> "같은 말도 표현을 조금 다르게 하면 완전히 다른 결과가 나오고 수정 요청할수록 처음[의도]랑 점점 멀어지기 시작해요." [00:38]

[의견] 화자는 이것이 "단순히 불편함의 문제가 아니"며, Amazon·GitHub 같은 글로벌 기업이 직접 나서서 새 방법을 만들 정도의 한계라고 본다. [00:46]

### 2. Spec Driven Development(SDD) 정의
> "AI한테 일시키기 전에 주문서를 먼저 완벽하게 써라. 그게 전부예요." [01:02]

**인테리어 리모델링 비유** [01:07]:
- 방식 A: "화이트톤으로 해주세요, 책상도 하나 넣어주시고..." (즉흥 요청)
- 방식 B: 메모장에 미리 정리 — 분위기(북유럽 미니멀), 필수 가구(책상·침대·옷장), 수납(벽면 전체 활용), 색상(화이트+원목), 예산(1천만 원)

[의견] 당연히 방식 B가 결과가 좋다 — 업체 입장에서 뭘 원하는지 명확하기 때문. AI도 똑같다. [01:28]

### 3. 주문서(스펙)에 적어야 할 세 가지
화자가 제시한 스펙의 3요소 [01:39]:

1. **무엇을 원하는가** — "로그인 화면을 만들어라"가 아니라 구체적으로:
   > "사용자가 이메일과 비밀번호로 로그인할 수 있어야 된다. 틀리면 에러 메시지를 보여줘야 된다. 세 번 틀리면 잠금 처리가 돼야 된다." [01:46]
2. **어떻게 만들 것인가** — 전체 구조, 순서, 방향성. ("인수[?] 순서대로 만들어라" 같은 것)¹
3. **무엇을 하나씩 해야 하는가** — 체크리스트. 단계가 1·2·3으로 나뉘면 AI가 훨씬 안정적으로 따라간다. [02:01]

> ¹ [불명확: 원문 "인수 순서대로 만들어라" — 자동자막 오인식 추정. 문맥상 "정해진 순서대로"의 의미로 보이나 정확한 단어 불명]

### 4. SDD가 실제로 잘 작동하는 이유 (세 가지)
[03:52~]

1. **AI의 기억 문제를 해결** — AI의 가장 큰 약점은 기억력이 없다는 것. 새 대화를 열면 처음부터 다시 설명해야 한다. 주문서 문서가 그 기억을 대신한다.
   > "마치 알바생한테 업무 매뉴얼을 만들어 주는 것처럼요. 매뉴얼이 있으면 사람이 바뀌어도 일이 흔들리지 않잖아요." [04:12]
2. **'뭘 만들지'와 '어떻게 만들지'를 분리** — AI에게 가장 힘든 요청은 "뭘 만들지 생각하면서 동시에 만들어 줘야" 하는 것. SDD는 먼저 무엇을 정하고(주문서), 그다음 AI가 어떻게만 고민하게 한다. [04:24]
3. **반복해도 흔들리지 않음** — 수정 요청을 반복해도 주문서가 기준 역할을 한다.
   > "AI가 어디로 가려[고] 해도 주문서가 '아니야, 이 방향이야' 하면서 잡아주거든[요]." [04:46]

## 카탈로그 — SDD 구현 도구 3종 (화자 평가)

화자가 "SDD 개념을 구현한 도구"로 소개한 세 가지와 각각에 대한 평가 [02:15~]:

| 도구 | 만든 곳 | 화자 평가 [의견] |
|---|---|---|
| **Kiro**² | Amazon | "아마존 서비스에 묶여야 해요. 내가 원하는 환경에서 자유롭게 쓰기가 어렵거든요." |
| **Spec Kit**³ | GitHub | "써보면 방법도 복잡하고 그닥 성능이 좋다고 못 느꼈어요." |
| **BMAD**⁴ | 오픈소스 커뮤니티 | "개념[취지]는 좋은데 배우는 데 걸리는 시간이 상당해요. 게다가 그냥 역할 놀이하는 기분이에요." |

[의견] 세 도구 공통 문제:
> "내 프로젝트, 내 작업 방식에 딱 맞게 설정하기가 너무 어려웠습니다. 도구에 내가 맞춰야 하는 느낌이랄까요?" [02:44]

> ² [자막 보정: "키로/K로" → Kiro (Amazon)]
> ³ [자막 보정: "스펙트/스페킷/스펙기" → Spec Kit (GitHub)]
> ⁴ [자막 보정: "비매드/비메드/빌메드" → BMAD(-METHOD)]

## 대안 — 하니스 기반 워크플로우

[의견] 화자는 "결국 핵심은 도구가 아니라 구조"라는 결론에 도달해 직접 하니스 기반 워크플로우를 만들었다고 한다. [02:50]

**하니스(harness) 비유** [02:56]:
> "안전 장치 같은 거예요. 예를 들어 번지 점프 할 때 하니스를 착용하잖아요. 점프는 자유롭게 하되 떨어지지 않게 잡아주는 구조. AI 작업도 마찬가지예요. AI한테 자유롭게 일을 시키되 방향이 흔들리지 않도록 잡아주는 구조."

화자가 쓰는 하니스의 세 구성요소 [03:11]:
1. **프로젝트 맥락 파일** — AI가 이 프로젝트에서 항상 알고 있어야 할 것들(목표·제약 조건·이미 결정된 것들)을 담은 파일.
2. **작업 요청 템플릿** — 매번 AI에게 부탁할 때 같은 형식으로 전달. (배달앱 주문 시 주소·요청사항 틀에 비유)
3. **결과 검증 체크리스트** — AI 결과물이 "내가 원한 게 맞아"를 빠르게 확인하는 기준 목록.

> "기존 [Kiro]처럼 특정 도구에 묶이지 않아요. [ChatGPT]든 [Claude]든 어떤 AI 도구에서든 그대로 붙여서 쓸 수 있고 내 작업 방식에 맞게 처음부터 설계할 수 있어요." [03:37]

## 팁 — 비개발자도 지금 당장 시작하는 법

[04:50] AI에게 뭔가 부탁하기 전에 메모장에 세 가지를 먼저 적어라:
1. 내가 원하는 최종 결과가 정확히 뭔가
2. 어떤 조건이 있나? 예외 상황은?
3. 어떤 순서로 만들면 좋을까?

> "이것만 해도 결과가 진짜 달라져요." [05:04]

[의견] 화자의 마무리 메시지 — 이 방식이 모든 상황에서 필요한 건 아니지만 "복잡하고 중요한 프로젝트일수록 이 방식이 빛을 발한다." 또한 개발자만의 얘기가 아니라 ChatGPT로 기획서 쓰는 사람, AI로 마케팅 콘텐츠·업무 자동화하는 사람 모두에게 적용되는 사고방식이다. [05:18]

> "AI한테 시키기 전에 먼저 명확하게 적어라. 이것 하나만 기억하셔도 오늘 영상 본 보람이 있어요." [05:33]

## 외부 보강

영상에서 평가만 하고 구체 사양은 생략한 SDD 도구 3종의 객관 정보 (출처: 아래 링크):

- **Kiro** (Amazon): VS Code/Code OSS 포크 기반 AI IDE. 스펙 작성(요구사항 → 유저 스토리 → 인수 기준) 후 기술 설계·태스크 분할을 돕고, 파일 변경/커밋에 반응하는 이벤트 기반 "hooks" 에이전트를 지원. 2025년 중반 기준 preview 단계.
- **GitHub Spec Kit**: Python CLI, GitHub 스타 9.3만+ (2026-05-07 v0.8.7 기준). Claude Code·GitHub Copilot·Amazon Q·Gemini CLI 등 30개 이상 AI 에이전트 지원. 모든 워크플로우의 토대로 "constitution"(모든 변경에 적용되는 불변 원칙 markdown 파일)을 둔다.
- **BMAD-METHOD**: MIT 라이선스 오픈소스. 12개 이상의 전문 AI 에이전트(PM·아키텍처·UX·개발·QA·스크럼 마스터 등)를 SDLC 전체에 걸쳐 오케스트레이션. 2026-04-29 v6.6.0, GitHub 스타 4.67만+. 화자가 "역할 놀이 같다"고 평한 것은 이 다중 에이전트 역할 구조와 맞닿아 있다.
