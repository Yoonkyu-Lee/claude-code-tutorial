## 한 줄 요약

## 핵심 개념

### 1. 잘 짜인 코드 베이스 = 잘 설계된 집의 도면
화자는 코드 베이스 품질을 "집의 도면"에 비유한다.

> "도면 잘 짜인 집은 콘센트 위치 적당하고 화장실 동선도 자연스럽고 가구들이기도 편해요. 근데 도면 없이 그냥 막 지은 집은 사람 사는 것만 고생해요. 콘센트가 침대 뒤에 숨어 있고 화장실 가는데 거실 가로 찔러야 되고요." [00:32]

> "코드 베이스가 잘 세팅되어 있으면 새 기능 하나 추가할 때 여기에 이걸 끼우면 되겠다 하는 자리가 명확해요. 근데 코드 베이스가 엉망이면 새 기능 하나 넣을 때마다 이거 어디다 넣어야 되지 하고 고민붙어 해야 되거든요." [00:51]

화자 본인 경험:

> "저희도 처음엔 그냥 빠르게 만드는데만 집중했어요. 근데 그렇게 몇 개월 지나니까 작은 기능 하나 추가하는 것도 일주일씩 걸리는 거예요. 왜? 어디를 건드리면 어디가 터질지 모르니까요. 그때부터 코드 베이스 자체를 다시 짜기 시작했어요." [01:07]

### 2. DDD (Domain-Driven Design) — 비즈니스 단위 분리
화자가 가장 먼저 적용한 원칙.

> "쉽게 말해서요. 코드를 이건 결제 영역, 이건 회원 영역, 이건 주문 영역 이런 식으로 비즈니스 단위를 나눠 놓는 거예요." [01:28]

회사의 부서 비유:

> "큰 회사에 부서가 있잖아요. 인사팀, 재무팀, 영업팀[^1]. 각 팀이 자기 일에만 집중하고 필요할 때만 다른 팀이랑 협업하죠. 만약 한 사무실에 모든 직원이 섞여 있으면 어떨까요? 누가 무슨 일을 하는지도 모르고 책임 소재도 흐릿하고 정말 혼란스럽겠죠." [01:35]

> "DDD가 딱 그거예요. 결제 로직은 결제 도메인 안에, 회원 로직은 회원 도메인 안에. 각자 자기 영역에서 자기 일만 하니까 코드가 깔끔해져요. 새로 들어온 개발자도 아, 결제 관련 문제면 결제 폴더만 보면 되겠구나 하고 바로 찾아갈 수 있어요." [01:50]

[^1]: 자동자막 "영어팀"을 컨텍스트(인사팀·재무팀과 병렬되는 부서)로 보아 "영업팀"으로 정정.

### 3. 기술 스택 분리 — Next.js + Python
> "프론트 엔드를 Next.js로 백엔드를 Python으로 가져갔어요. 왜 이렇게 나눴느냐? 각자 잘하는 걸 시키려고요." [02:04]

각 스택의 역할에 대한 화자의 정의:

- **Next.js (프론트엔드)**:
  > "Next.js는 화면 그리는데 정말 강해요. 사용자가 보는 부분 클릭하고 입력하고 결과 보는 부분이에요. 서버 사이드 렌더링도 되고 라우팅도 자동이고 요즘 앱에 필요한 거의 다 들어 있어요." [02:13]

- **Python (백엔드)**:
  > "Python은 그 반대편에 있어요. 데이터 처리 머신 러닝 모델 돌리기, 외부 API랑 통신하기 이런 작업들은 Python이 좀 더 편한 거 같습니다. 라이브러리도 많고 데이터 다루는 문법이 좀 더 직관적이에요." [02:22]

- Python 선택의 추가 근거 [의견]:
  > "[의견] 다른 언어에 비해 가지고 굉장히 읽고 쓰기도 편하고 어 AI가 코드를 짤 때 코드량이 적어 가지고 좀 더 효과적으로 작업을 할 수 있어요." [02:40]

분리의 효과:

> "프론트 작업하는 사람은 백엔드 신경 안 써도 돼요. 백엔드 작업하는 사람은 화면 디자인 신경 안 써도 되고요. 각자 자기 영역에서 빠르게 움직일 수 있어요." [02:51]

화자의 실제 구성:

> "물론 저 같은 경우에는 모노레포[^2]에서 프론트엔드와 백엔드를 한 곳에다가 모아놓은 그런 레포지터리를 쓰고 있습니다." [02:59]

[^2]: 자동자막 "모노래"를 컨텍스트(프론트엔드/백엔드를 한 곳에 모은 레포지터리)로 보아 "모노레포(monorepo)"로 정정.

### 4. 기능 추가 시나리오 — 잘 짜인 베이스의 실제 작동
> "회원이 결제하면 알림 보내는 기능 하나 추가한다고 쳐요. 잘 깔린 코드 베이스에서는 어떻게 되냐면요. 결제 도메인에 가서 결제 완료됐을 때 이벤트 하나 발생시켜요. 알람 도메인에서 그 이벤트 받아서 처리하고요. 끝이에요. 다른 영역 건드릴 일이 거의 없어요." [03:13]

레고 비유:

> "마치 레고 블록 같은 거예요. 기반 블록이 잘 깔려 있으면 그 위에 새 모양 만드는 건 금방이거든요." [03:34]

> "버그도 적어요. 왜? 영역이 분리되어 있으니까. 결제 코드 고쳤다고 회원 가입이 망가지는 일이 거의 없어요. 영역끼리 영향을 안 주거든요." [03:41]

### 5. 계획 없는 바이브 코딩 — 천천히 무너지는 패턴
화자의 정의:

> "저는 아무런 베이스도 없이 맨 땅에서 코드를 짜는 걸 계획 없는 바이브 코딩이라고 불러요." [04:16]

초기 속도의 함정:

> "계획 없는 바이브 코딩이 처음에는 진짜 빨라요. 설계 안 해도 되고 구조 안 잡아도 되고 그냥 떠오른 대로 막짜니까요. 한 일주일 정도는 결과물이 척척 나오는 거 같아요. 이거 왜 사람들이 어렵다고 하지 싶을 정도예요." [04:24]

한 달 뒤의 붕괴:

> "근데 문제는 한 달 뒤에요. 코드가 점점 엉키기 시작해요. 결제 로직이 화면 코드 안에 들어가 있고 회원가입이 주문 코드에서 직접 수정되고 한마디로 부서가 없는 회사가 되는 거예요. 누가 무슨 일 하는지 아무도 모르고 한 사람 일 바꾸면 옆 사람 일까지 다 망가져요." [04:38]

집 비유:

> "도면 없이 집을 지으면요. 처음엔 빠르거든요. 벽 세우고 지붕 얹고. 근데 2층 올리려고 보면 1층 벽이 그 무게를 못 버텨요. 결국 다 부수고 새로 지어야 해요. 코드도 똑같아요. 기초가 부실한 채로 기능 계속 쌓으면 어느 순간 무너져 버려요." [04:53]

디버깅 지옥:

> "바이브 코딩으로 만든 프로그램에서 버그가 나면요. 어디서 시작된 건지 추적이 안 돼요. 결제 화면에서 오류 났는데 원인은 회원 쪽에 있고 그 회원은 또 주문 코드에서 건드리고 거미줄[^3]처럼 엉켜서 한 줄 풀면 또 다른 줄이 엉켜요." [05:10]

[^3]: 자동자막 "7타리"를 컨텍스트(엉킨 줄, 풀면 또 엉키는 구조)로 보아 "거미줄"로 정정. [불명확 가능성 있음]

## 카탈로그 — 두 접근의 5축 비교
화자가 영상 후반부에서 정리한 비교표.

| 축 | 잘 짜인 코드 베이스 | 계획 없는 바이브 코딩 |
|---|---|---|
| 1. 초반 속도 | 처음 짤 때 시간이 좀 든다 (인정) | 훨씬 빠르다 |
| 2. 한 달 뒤 속도 | 새 기능 추가가 점점 더 쉬워진다 (기반이 탄탄) | 점점 더 어려워진다 (어디 건드릴지 모름) |
| 3. 버그 | 결제 버그는 결제 안에서만 영향 | 버그 하나가 사방으로 퍼짐. "화면 버그인 줄 알았는데 데이터베이스까지 손대야 하는 일 흔하다" |
| 4. 신규 인원 합류 | 일주일이면 자기 일 시작. 폴더 구조만 봐도 보임 | 한 달 지나도 헤맴. "이 변수는 왜 여기 있는 거예요?" 물어볼 사람도 만든 본인뿐 |
| 5. 결과물 퀄리티 | 안정적 | "화면은 예뻐 보일 수 있어요. 근데 잠깐만 써 보면 느려지고 오류 나고 이상한 동작이 보여요. 사용자는 그걸 다 느끼거든요" |

원문 인용:

> "둘째, 한 달 뒤 속도 여기서 뒤집혀요. 잘 짜인 코드 베이스는 새 기능 추가가 점점 더 쉬워져요. 기반이 탄탄하니까요. 바이브 코딩은 반대예요. 점점 더 어려워져요." [05:38]

> "다섯째, 결과물 퀄리티. 이게 가장 중요해요. 기초가 부실하면 결국 퀄리티 낮은 결과물이 나와요." [06:17]

## 팁 — 스킬화로 코드 베이스 자체를 자산화
화자는 영상 막바지에 "스킬 기반"이라는 개념을 짧게 언급한다.

> "또한 스킬 기반으로 해 가지고 그동안의 어떤 템플릿들 그런 것들을 다 포매팅화시키고 스킬화를 시켜 가지고 그 스킬 기반으로 코드를 짜게 되면 코드가 굉장히 안정적으로 잘 짜여집니다." [06:43]

[의견] 그리고 다음과 같이 강조:

> "[의견] 한 번에 무너지는 게 아니라 천천히 무너집니다. 그래서 더 위험해요. 아직 괜찮네 하다가 어느 날 손 쓸 수 없게 되거든요." [07:00]

자기 점검 질문:

> "한 번쯤은 이 코드 베이스가 1년 뒤에도 빠르게 움직일 수 있는 구조인가 자문해 보시면 좋아요." [07:12]

코드 베이스 위에서의 바이브 코딩 [의견]:

> "[의견] 특히나 이러한 코드 베이스가 있다면 이제 코드에 대해 가지고 전혀 모르시는 분들이라더라도 이 코드를 기반으로 해서 바이브 코딩을 하신다면 좀 더 well-architected 된 그러니까 유지보수 가능한 코드를 만들 수 있다는 점 그 점 명심해 가지고 이제 사용을 하시면 될 거 같습니다." [08:06]
