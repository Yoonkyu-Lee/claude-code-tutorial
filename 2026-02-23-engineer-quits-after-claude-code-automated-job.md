## 핵심 개념

### 1. Claude Code는 채팅 도구가 아니라 "직접 들어오는" 에이전트
화자는 처음에 ChatGPT처럼 질문→답변→복붙으로 썼다고 한다.

> "ChatGPT는 채팅이에요. 이 코드 어떻게 짜요? 물어보면 답해 줘요. 그럼 그걸 복사해서 내가 직접 붙여야 돼요. Claude Code는 직접 들어와요. 내 컴퓨터 안으로. 내 프로젝트 안으로 직접 들어와서 파일 읽고 수정하고 새로 만들어요. 자동화 프로그램이랑 AI가 좀 합쳐진 느낌이라고 보시면 돼요." [02:00]

[보정: 자막 "채집t", "채 GPT" → ChatGPT, "클로드 코드" → Claude Code. 이하 동일 보정 적용.]

### 2. 컨텍스트 한계 — "갑자기 멍청해지는 순간"
화자가 가장 중요한 포인트라고 직접 강조한 부분.

> "여기가 진짜 중요한 포인트예요. Claude Code를 오래 쓰다 보면 갑자기 멍청해지는 순간이 와요. 아까까지 잘하다가 갑자기 엉뚱한 소리를 해요. 아까 말한 걸 완전히 잊어버린 것처럼." [02:24]

원인은 AI의 기억 한계. 대화가 길어지면 앞 내용을 잊는 것 = "컨텍스트가 터진다".

비유 (화자 제시):
> "직원한테 일 시키는데 그 직원이 메모장이 있어요. 메모장 한 페이지가 다 차하면 앞에 적어 둔 거 지워지는 거예요. 그럼 어떻게 해야겠어요? 중요한 내용은 따로 파일로 저장해서 필요할 때마다 보여 주는 거죠. 그게 컨텍스트 관리예요." [02:41]

해결책: 프로젝트 구조, 코드 스타일 규칙, 주의 사항을 **파일에 정리해서 주입**. 대화가 새로 시작돼도 파일만 다시 넣으면 이어서 작업 가능.
> "직원이 바뀌어도 인수인계 문서가 있으면 되는 것처럼요." [03:04]

### 3. 스킬(Skill) = 레시피
반복되는 지시를 파일로 만들어 두는 것.

> "이런 상황엔 이렇게 매번 말해 줘야 하는 게 반복되는 거예요. 근데 이걸 파일로 만들어 두면 (...) 매번 설명 안 해도 돼요. 이 파일 보고 해라라고 하면 되거든요. 이게 스킬이에요." [03:11]

비유 (화자 제시):
> "요리사한테 매번 소금 이만큼 설탕 이만큼 설명 안 해도 레시피 카드 하나 주면 되잖아요." [03:22]

### 4. 레퍼런스 코드 = 기준점
화자가 직접 잘 짠 코드 하나를 "기준(reference)"으로 삼아 스킬 파일에 담고, 나머지를 그 기준에 맞춰 고치게 했다.

> "제가 직접 코드 하나를 정말 잘 짰어요. 컨벤션도 완벽하고 구조도 깔끔하고 (...) 이게 레퍼런스예요. 이게 기준이에요. 그걸 스킬 파일에 담아서 Claude Code한테 줬어요. 이 코드처럼 짜라. 이게 기준이다. 그러면 다른 코드들도 이 기준에 맞춰 고쳐져요." [03:30]

### 5. 개발자의 역할 변화
> "개발자가 코드를 덜 짜고 있어요. 대신 뭘 만들지 결정하고 어떻게 구조를 잡을지 판단하고 결과를 검사하는 역할로 바뀌고 있어요." [04:45]

[의견] 화자는 이 역할조차 계속 줄어들 것이라 본다:
> "근데 이 역할도 계속 줄어들면요. (...) 지금 이 속도라면 제가 지금 하는 업무에 90%는 5년 안에 AI가 대체할 것 같았어요. 어쩌면 더 빠를 수도 있어요." [04:50] / [06:23]

## 실제 사례 (카탈로그)

### 사례 1 — 레거시 리팩토링을 반나절 만에
프로덕션에서 돌아가는 프로젝트의 리팩토링.

화자의 리팩토링 정의:
> "실제로 서비스 중인 코드를 동작은 그대로 유지하면서 내부를 깔끔하게 정리하는 거예요." [03:47]

비유: 사람이 살고 있는 집을 리모델링 하는 것 — 잘못 건드리면 실제 서비스가 터지므로 굉장히 조심스러운 작업, 보통 몇 주씩 잡는다. [03:50]

방법:
1. **테스트 코드를 먼저 짠다.** 테스트 = "자동으로 제대로 돌아가나 확인해 주는 코드". 리팩토링 전에 깔아두면 리팩토링 후 "버튼 한 번"으로 검증 가능. [04:07]
2. 안전망(테스트)을 깔고 스킬도 설정한 뒤 Claude Code에 맡김.
3. **반나절 만에 끝남. 버그 하나 없이.**

> "그 과정에서 제가 직접 짠 코드는 딱 하나예요. 처음에 레퍼런스로 만든 코드 그게 전부예요. 저는 그냥 결과 확인하고 괜찮으면 배포하면 됐어요." [04:33]

### 사례 2 — 어드민 시스템을 하루 만에
회사 내부용 관리자 페이지(데이터 조회·설정 변경·통계 확인). 백엔드+프론트엔드 모두 필요.

설계 선택 2가지:
- **모노레포(monorepo)**: 한 폴더 안에 백엔드와 프론트엔드를 같이 둠.
  > "이게 Claude Code한테 엄청 유리해요. 왜냐하면 백엔드 코드 보면서 바로 프론트엔드 코드를 짤 수 있거든요. (...) 이걸 스스로 판단할 수 있어요. 제가 일일이 (...) 설명 안 해도 돼요." [05:20]
- **SDD (Spec Driven Development)**: 코드 짜기 전에 문서로 먼저 정의.
  > "이 기능은 이렇게 동작해야 한다. 이 버튼 누르면 이런 결과가 나와야 한다. 문서를 먼저 쓰고 그게 맞는지 확인하고 그다음에 Claude Code한테 구현을 맡기는 거예요. 제가 할 일은 문서 리뷰뿐이에요." [05:43]

결과:
> "어드민 구축에 하루가 걸렸습니다. 그 이후 새로운 기능 요청 들어오면 2~3시간 이내에 처리했어요." [05:54]

[불명확: 자막 "3세시간" — 문맥상 "두세 시간(2~3시간)"으로 추정되나 정확한 수치는 단정하지 않음. 후반 [07:26]에도 동일 표현 등장.]

## 핵심 교훈 (화자가 명시적으로 정리한 3가지 + 1)

화자가 "제가 배운 핵심은 세 가지"라고 직접 번호를 매긴 부분이다.

1. **컨텍스트 관리가 전부.** 파일로 관리하고 스킬로 관리하라. "이게 안 되면 Claude Code가 제 실력을 못 내요." [08:29]
2. **테스트를 먼저 짜라.** "Claude Code한테 마음껏 맡기려면 안전망이 있어야 해요. 테스트 없이 맡기면 버그 나도 모르거든요." [08:35]
3. **SDD로 스펙을 먼저 잡아라.** "구현보다 설계가 더 중요한 시대예요. 스펙이 명확하면 구현은 Claude Code가 해요." [08:42]

추가 (+1):
> "이걸 잘 쓸 줄 알게 되면 선택지가 생겨요. 회사에서 더 효율적으로 일할 수도 있고 저처럼 퇴사해서 내 걸 만들 수도 있어요. (...) 선택지가 생기는 것 자체가 강점이에요." [08:47]

## 화자의 의견·서사 (사실과 분리)

[의견] 자동화가 "꿀이 아니라 공포"였다는 것이 영상 전체의 정서적 축이다.
> "이게 꿀이 아니에요. 이게 공포였어요. 아, 이러다 나 진짜 필요 없어지겠다." [00:13]

[의견] 퇴사·창업 결정의 논리:
> "제가 Claude Code를 잘 다루는 게 회사의 서비스를 만드는 데 쓰이고 있잖아요. 근데 이걸 왜 남의 걸 만드는 데 써요? 내 걸 만들면 안 되나?" [06:43]

[의견] 개발자 창업의 최대 장벽이 사라졌다는 주장:
> "개발자가 창업하는 데 제일 큰 장벽이 개발 시간이었거든요. (...) 근데 지금은 그 장벽이 사라졌어요. Claude Code 잘 쓰면 서비스 하나 만드는 데 며칠이면 돼요. 그러면 아이디어만 있으면 되는 거잖아요." [06:53]

[의견] 비개발자에게도 해당된다는 확장:
> "지금 일어나고 있는 일이 개발자만의 이야기가 아니에요. (...) 디자인, 글쓰기, 마케팅, 법률, 회계 다 AI가 들어오고 있거든요." [07:39] — 대응 원칙은 "AI가 할 수 있는 건 넘기고 AI가 못 하는 판단·결정에 집중", 그러려면 "AI를 먼저 잘 알아야 한다. 막연하게 무서워하거나 반대로 무시하면 안 된다." [08:07] / [08:14]

## 외부 보강

용어 보강 (영상은 정의를 일상 비유로만 설명하므로, 정식 정의를 보강):

- **SDD (Spec-Driven Development)**: 구현 전에 명세(스펙) 문서를 먼저 작성하고, 그 스펙을 기준으로 구현·검증하는 개발 방식. 이 저장소의 기존 노트 `2026-04-30-design-md-google-spec-explained.md`에서 Google의 spec 문서 사례를 다룬 바 있어 교차 참조 가능. (출처: 본 저장소 내부 노트)
- **모노레포(monorepo)**: 여러 프로젝트(여기선 백엔드·프론트엔드)를 단일 저장소에 두는 구조. 영상의 주장(에이전트가 백엔드를 보며 프론트엔드를 함께 생성하기 유리)은 단일 컨텍스트 내에서 두 계층의 코드를 동시에 참조할 수 있다는 점에 근거한다. (출처: 일반 소프트웨어 공학 정의)

[보강 범위 제한: 화자가 사용한 구체적 도구·구성(어떤 테스트 프레임워크, 어떤 모노레포 도구인지)은 영상에서 언급되지 않아 보강하지 않음.]
