## 한 줄 요약
바이브 코딩 결과물이 "AI가 만든 티"가 나는 건 디자인 감각이 아니라 규칙 부재 탓이며, 컬러 팔레트(Coolors)·2색 제한·Stitch·MCP 회피·Skills·shadcn/ui 같은 지키기만 하면 되는 규칙 여섯 가지로 해결할 수 있다는 실전 가이드.

## 메타
- 영상 유형: 튜토리얼/실습형 (단계별 도구·설정 위주)
- 자막: 한국어 자동생성. 타임스탬프 제공
- 게시일 출처: 디스패처 확정값 (yt-dlp)
- 자막 정정: 자동자막의 영문 고유명사 오인식을 컨텍스트로 복원함 (아래 각주 참조)

> **자막 정정 각주**: 한국어 자동자막이 영문 도구명을 음차로 잘못 잡은 것을 컨텍스트로 복원했다.
> `클러즈/클러즈.co` → **Coolors (coolors.co)**, `스티치` → **Stitch**, `무의/무이` → **MUI (Material UI)**,
> `샤드 CN/샤드시/샤드시즌 UI` → **shadcn/ui**, `테일윈드 컨피 JS` → **tailwind.config.js**,
> `핵스코드` → **hex code**, `스킬즈` → **Skills**, `넥스트 JS` → **Next.js**, `토셋/토스` → **Toss**,
> `MCP/모델 컨텍스트 프로토콜` → **MCP (Model Context Protocol)**.

## 핵심 개념

### 문제 정의: 기능은 완벽한데 화면을 열면 한숨이 나온다
> "바이브 코딩으로 만든 결과물이요. 3초 만에 티납니다. 디자인 때문에요." [00:00]

> "바이브 코딩으로 앱 하나 뚝딱 만들어 오면 기능은 기가 막혀요. 동작도 완벽해요. 근데 화면을 딱 열면요. (색깔이) 세 개 다섯 개, 여섯 개 들어가 있고요. 버튼 크기도 제각이고요. 여백은 들쑥날쑥하고요." [00:15]

화자의 핵심 주장: 해결책은 디자인 공부가 아니다.

> "디자인 공부해야지 하면서 피그마 강의 듣고 컬러 이런 공부하고 UIUX 책 사고 근데 솔직히 그거 다 하실 거면 왜 바이브 코딩을 하겠어요? 바이브 코딩한테 필요한 건요. 디자인 감각이 아닙니다. 그냥 지키기만 하면 되는 규칙이에요." [00:32]

> [의견] "핵심은요. 바이브 코딩으로 만든 결과물이 구려 보이는 건 실력 문제가 아니라 규칙을 안 지켜서 그렇습니다." [06:40]

> **참고**: 영상은 "여섯 가지"라고 예고하지만 실제 발화에서는 일곱 번째 규칙(shadcn/ui)까지 다룬다. [04:38 "일곱 번째"] 아래 카탈로그는 화자가 번호를 매긴 순서를 그대로 따른다.

## 카탈로그 — 디자인 규칙

### 규칙 1: 컬러를 직접 고르지 마세요 (Coolors 사용)
> "바이브 코딩 결과물이 구려 보이는 이유 1순위가 뭔지 아세요? 바로 색깔입니다. AI한테 파란색으로 해줘 하면요. 대충 아무 파란색이나 가져오거든요. (...) AI도 색 조합은 잘 못 맞춰요." [00:56]

방법:
- **Coolors (coolors.co)** 접속 → 스페이스바 한 번 누르면 조화로운 컬러 팔레트가 자동 생성됨 [01:11]
- 마음에 드는 팔레트의 **hex code**를 그대로 복사해 프롬프트에 넣는다 [01:23]

> "절대 이거 좀 더 진하게 해 볼까 하면서 수정하지 마세요. 그 순간 조화가 깨져요. (Coolors)가 준 그대로 쓰는게 핵심이에요." [01:28]

### 규칙 2: 다섯 색 다 쓰지 말고 딱 두 개만
> "(Coolors)에서 팔레트를 받으면요. 색이 다섯 개가 나와요. 근데 다섯 개 다 쓰시면 안 됩니다. 딱 두 개만 쓰세요. 메인 컬러 하나, 서브 컬러 하나." [01:36]

근거 (화자가 제시한 예시):
> "잘 만들어진 사이(트)들 보세요. 흰색이랑 검정 포인트로 옅은 파란색 사실상 두 색에서 세 개의 색밖에 사용하지 않아요. (Toss) 보세요. 흰색 바탕에 파란색 하나. 깔끔하죠?" [01:49]

> "AI한테 여기는 빨간색, 저기는 초록색, 이건 노란색 이렇게 시키거든요. 이러는 순간 축제 현수막이 돼요." [02:01]

운용 규칙:
- **메인 컬러**: CTA 버튼·강조에만
- **서브 컬러**: 배경·보조 요소에만
- 나머지는 전부 흰색·검정·회색 계열 [02:10]

### 규칙 3: AI한테 디자인 시킬 때 Stitch를 써라
> "스티치(Stitch)가 뭐냐면요. 웹사이트나 앱 UI를 AI로 뚝딱 만들어 주는 도구예요. 프롬프트 한 줄이면 화면이 나오거든요." [02:23]

Stitch 사용 시 반드시 지킬 두 가지:
1. **Coolors에서 받은 hex code를 정확히 넣는다** — "파란색으로 해 줘"가 아니라 hex code를 넣어야 일관된 컬러가 나옴 [02:33]
2. **레퍼런스 사이트 묘사는 영문으로** — "토스 앱처럼 깔끔하게" 같은 한국어 묘사는 AI가 잘 못 알아들음 [02:42]

> "이렇게 영문으로 구체적으로 묘사해 주셔야 돼요. 결과물이 확 달라집니다." [02:50]

### 규칙 4: MCP로 디자인하지 마라
> "클로드 코드 같은 AI 코딩 도구 쓸 때요. MCP로 디자인하지 마세요. MCP가 뭐냐면요. 모델 컨텍스트 프로토콜이라고 해서 AI한테 외부 도구를 연결해 주는 건데요. 피그마 MCP 같은 걸로 디자인을 시키면 이론적으로는 굉장히 좋아 보이거든요." [02:53]

문제 메커니즘 (화자 설명):
> "근데 실제로 하다 보면 토큰이 터져서 디자인의 일관성이 떨어집니다. 즉 코드의 품질이 계속해 가지고 떨어진다고 보시면 될 것 같습니다. 토큰이 순식간에 소진되면서 AI가 앞에서 했던 맥락을 까먹기 시작해요." [03:12]

> "처음에는 깔끔했던 디자인이 수정할수록 점점 일관성이 무너지는 거예요. 버튼 스타일이 페이지마다 달라지고요. 간격이 제각이 되고요. 컬러도 흔들리기 시작합니다." [03:27]

> [의견] "MCP로 디자인 직접 조작하는 거는 저는 개인적으로는 비추입니다." [03:36]

### 규칙 5: MCP 대신 Skills를 써라
> "스킬즈(Skills)가 뭐냐면요. 클로드 코드에서 특정 작업에 대한 가이드라인을 미리 문서로 만들어 놓는 거예요. 예를 들어서요. 웹 디자인 가이드라인이라는 스킬을 만들어 놓으면요. 거기에 컬러는 이거 써라. 여백은 이만큼 줘라. 폰트 사이즈는 이 체계를 따라라. 이런 규칙들이 다 적혀 있는 거예요." [03:44]

MCP 대비 이점:
> "MCP처럼 매번 API를 호출하는게 아니라 그냥 텍스트 문서를 읽는 거라서 토큰 소모도 훨씬 적고요. 일관성이 유지돼요." [04:06]

> "스킬즈는 AI한테 디자인 규칙을 잘 알려 주는 겁니다." [04:17]

### 규칙 6: Stitch용 영문 프롬프트는 Claude로 추출하라
> "아까 스티치 사용할 때 웹사이트에 대한 묘사를 넣으라고 했잖아요. 그때 웹사이트에 대한 묘사 영문 프롬프트 같은 경우에는 클로드를 통해 가지고 영문 프롬프트를 추출하는게 좀 더 효과가 좋더라입니다." [04:21]

> [의견] "이거는 굉장히 개인적인 팁이었습니다." [04:38]

### 규칙 7 (보너스): Next.js 프로젝트는 MUI 말고 shadcn/ui
> "바이브 코딩으로 넥스트 JS 프로젝트 만들 때요. (MUI)보다는 샤드 CN (shadcn) 추천드립니다." [04:41]

MUI의 문제 (화자 설명):
> "(MUI)가 뭐냐면요. 커스터마이징이 좀 고통입니다. (MUI) 버튼색 하나 바꾸려면요. (테마 파일) 열고 팔레트 수정하고 (override) 넣고 SX 프로(퍼티) 넣고 근데 이 컴포넌트는 또 다른 방식으로 해야 되네. AI도 이거 헷갈려해요." [04:49]

> "(MUI) 특유의 머터리얼 디자인 느낌 있잖아요. 그게 좋을 때도 있는데 벗어나려고 하면 프레임워크랑 싸워야 돼요." [05:02]

shadcn/ui가 바이브 코딩과 잘 맞는 이유:
> "샤드시즌(shadcn) UI는요. 접근 방식이 조금 다릅니다. 컴포넌트는 npm으로 설치하는게 아니라 소스 코드를 직접 프로젝트에 복사해 넣는 방식이거든요. 그러니까 버튼닷t tx(button.tsx)가 내 프로젝트 안에 있는 거예요. 바이브 코딩이랑 궁합이 최고입니다. AI가 코드를 직접 수정할 수 있으니까요." [05:07]

> "메인 컬러 하나, 서브 컬러 하나 (config)에 넣으면 끝. (...) 디자인 감각이 없어도 일관되게 나오거든요." [05:23]

## 워크플로 — 실제 적용 순서 (화자가 정리한 단계)
[05:31 ~ 06:11 발화 기준]

1. **Coolors** 접속 → 스페이스바로 마음에 드는 팔레트 선택 → 메인 컬러 1개, 서브 컬러 1개만 뽑아 hex code 복사
2. 프로젝트 생성 시 **Next.js + shadcn/ui** 세팅
3. **tailwind.config.js**에 두 컬러를 primary / secondary로 넣기
4. 화면 레이아웃이 필요하면 **Stitch** 사용 → primary hex / secondary hex 입력 + 레퍼런스 사이트는 영문으로 구체 묘사 (영문 프롬프트는 Claude로 추출)
5. **Claude Code**로 바이브 코딩할 때는 **Skills의 디자인 가이드라인**을 넣어 둔다 (MCP로 디자인 도구 연결 X)

## 외부 보강

영상에서 도구명만 언급되고 정의가 짧아, 객관 사실을 공식 출처로 보강한다. (영상 내용과 분리)

- **Coolors (coolors.co)**: 컬러 팔레트 생성 도구. 스페이스바로 무작위 조화 팔레트 생성, hex code 복사 지원. [출처: coolors.co]
- **Stitch (Google)**: Google이 공개한 AI 기반 UI 디자인 도구. 프롬프트로 웹/모바일 UI 화면과 코드를 생성. [출처: stitch.withgoogle.com]
- **shadcn/ui**: Radix UI + Tailwind CSS 기반 컴포넌트 모음. npm 패키지로 설치하는 대신 컴포넌트 소스 코드를 프로젝트에 직접 복사해 넣어 완전한 커스터마이징·소유권을 갖게 하는 방식이 특징. [출처: ui.shadcn.com]
- **MUI (Material UI)**: Google Material Design 기반 React 컴포넌트 라이브러리. 테마(theme)·SX prop·override로 커스터마이징하며, 영상에서 "프레임워크와 싸워야 한다"고 지적한 부분이 이 추상화 계층을 가리킴. [출처: mui.com]
- **MCP (Model Context Protocol)**: Anthropic이 공개한, AI 모델에 외부 도구·데이터를 연결하는 개방형 프로토콜. [출처: modelcontextprotocol.io]
- **Skills (Claude Code)**: 특정 작업의 가이드라인을 마크다운 문서로 미리 정의해 두면 Claude Code가 작업 시 참조하는 기능. 영상의 "MCP는 매번 호출, Skills는 텍스트 문서를 읽는 것"이라는 토큰 비교는 이 차이를 가리킴.

> 위 보강은 도구의 객관적 정의 확인용이며, "토큰이 터진다"는 인과 주장은 화자의 경험·의견(`[의견]` 표기)이다.
