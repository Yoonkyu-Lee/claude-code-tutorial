## 핵심 개념

### 1. 세 도구의 역할 분담

화자는 영상 도입부에서 자신의 워크플로우가 "맥퍼커 + G-Stack + Superpowers" 세 도구의 조합이라고 못박는다.

> "스킬 하나만 쓰면 한 개가 분명히 있어요. 그래서 저는 여러 가지 스킬을 콤비네이션에서 쓰고 있습니다." [00:07]

각 도구의 주된 목적은 다음과 같이 분리된다.

#### 1.1 맥퍼커 — 도메인 매핑 (+ 화자 커스텀: DDD 아키텍처링)

> "먼저 맥퍼커의 스킬은 도메인 맵핑이 주된 목적이에요. 우리 도메인에서 이 단어가 정확히 어떤 뜻인지 AI한테 박아 주는 거예요. 용어가 흔들리면 코드도 흔들리거든요." [00:11]

화자는 비유로 신입 직원 용어집을 든다.

> "신입 직원 들어왔을 때 회사 용어집 줘야 되잖아요. 우리는 이 단어를 이런 의미로 써요. 이런 거요. 맥버컷이 그 용어집을 자동으로 만들어 주는 도구라고 보시면 돼요." [01:32]

여기까지가 기본인데, 화자는 한 단계 더 나간다 ([의견] 표시):

> "[의견] 저는 여기에 한 발 더 나아가서 DDD 아키텍처링을 커스텀으로 얹었어요. 도메인-드리븐-디벨로프먼트를 더 강하게 가져가는 거예요. 용어 정리해서 끝나는 게 아니라 그 도메인 모델이 폴더 구조랑 코드 경계까지 이어지도록요." [00:26]

구체적으로 어떻게 다른가:

> "도메인 모델이 폴더 경계, 모듈 경계, 함수 시그니처까지 이어져요. 주문 도메인은 주문 모듈 안에서 끝나고 결제 도메인은 결제 모듈로 분류되도록 해요. 용어가 단순한 약속이 아니라 코드 아키텍처를 결정하는 1차 자료가 되는 거예요." [01:55]

그 효과 ([의견]):

> "[의견] 이게 나중에 결정적이에요. AI가 같은 단어를 같은 뜻으로 알아먹는 데서 끝나는 게 아니라 도메인 경계를 따라 코드를 짜니까 모듈이 섞이지 않습니다." [02:07]

화자에 따르면 맥퍼커 스킬은 **약 16개** 정도라고 한다.

> "맥퍼커 스킬은 16개 정도 되는데 주된 목적은 하나예요. 도메인 맵핑을 통해서 정확한 용어를 쓰도록 유도하는 거예요." [01:08]

도메인 매핑 자체에 대한 설명 ([01:13]):

> "도메인 맵핑이 뭐냐면요. 우리가 만든 서비스에 등장하는 단어, 개념, 관계 이걸 AI한테 정확하게 알려 주는 거예요. 예를 들어 주문이라는 단어가 있다고 해 봐요. 일반 쇼핑몰의 주문이랑 음식 배달 주문이랑 B2B 발주는 다르잖아요. 근데 그냥 주문 한 단어로 던지면 AI가 헷갈려요. 맥퍼커의 스킬은 이걸 잡아 주는 거예요. 계속 캐묻고 정리해서 도메인 파일로 만들어요." [01:19]

#### 1.2 G-Stack — 브레인스토밍 (특히 제품 스펙)

> "G-Stack은 결이 달라요. 브레인스토밍에 진짜 강해요. 그중에서도 제품 스펙 적을 때가 핵심이에요. 이 스펙에서 뭘 빼야 하고 뭘 살려야 하는지 진짜 신도(심도) 깊게 캐물어요. 스펙 한 줄을 그냥 통과 안 시키거든요. 밀도 있게 다듬어서 내보내요." [00:38]

핵심은 "**뭘 빼야 하는지**"라는 점을 화자가 강조한다.

> "**스펙에서 뭘 빼야 하는지가 진짜 중요하거든요. 보통 빼는 게 더 어려워요. 다 넣고 싶으니까요. G-Stack은 거기서 안 봐줘요. 빼야 할 건 빼라고 끝까지 밀어붙여요.**" [02:39]

대표 스킬: **Office Hours**.

> "대표적인 게 오피스아워즈 스킬이에요. YC 오피스아워 있잖아요. 창업자가 가서 멘토한테 깨지는 거. 그거 그대로 옮겨 놨어요. 여섯 가지 강제 질문이 있어요." [02:47]

화자가 열거한 6가지 질문:

> "누가 정말 이걸 원해? 지금 이걸 어떻게 해결하고 있어? 이게 진짜 절박한 문제야? 가장 작은 단위가 뭐야? 직접 본 적 있어? 미래가 보여?" [02:55]

이 질문 흐름의 효과 ([의견]):

> "[의견] 이 질문 흐름이 진짜 잘 짜여 있어서 한 번 거치면 스펙에서 핵심이 뭐고 노이즈가 뭔지 먼저 확 갈려요." [03:03]

그리고 답변 받은 뒤에도 끝이 아니라는 점:

> "그리고 답변 받은 다음에 그냥 끝이 아니에요. 플랜 CEO 리뷰, 플랜 리뷰 이런 후속 스킬로 넘어가서 스펙을 다시 한 번 다듬어요. 브레인스토밍 단계에서 뽑아낼 수 있는 걸 끝까지 뽑아내는 구조예요." [03:08]

#### 1.3 Superpowers — 실제 구현

> "슈퍼파워즈는 실제 구현 단계예요. 저는 슈퍼파워즈의 브레인스토밍이랑 서브에이전트 드리븐 디벨로프먼트를 실제 코드 짜는 단계에서 써요. 구현 단(계)에 들어가면 이게 메인 엔진이에요." [00:50]

화자가 메인으로 쓰는 3가지 ([03:17]):

> "저는 여기서 세 가지를 메인으로 써요. 브레인스토밍이랑 라이팅 플랜(즈) 그리고 서브에이전트 드리븐 디벨로프먼트."

##### Superpowers의 브레인스토밍

화자는 이걸 **G-Stack 결과를 받아 더 디테일하게 가는 단계**로 쓴다.

> "저 같은 경우에는 이걸 코드로 어떻게 풀 거야에 대한 스펙을 좀 확정 짓는, 상세 스펙을 확정 짓는 용도로 많이 쓰고 있어요." [03:38]

##### Writing Plans + 커스텀 Plan Review 에이전트

> "플랜을 쓴 다음에 단순히 플랜을 쓰는 걸로 끝내는 게 아니라 제 커스텀 에이전트를 사용해 가지고 플랜 리뷰를 돌립니다. 그래서 리뷰 에이전트를 사용해 가지고 플랜을 리뷰를 돌린 다음에 리뷰 결과에 대해 가지고 다시 한 번 플랜을 수정해 줘요." [03:55]

##### Subagent-Driven Development

> "큰 작업을 여러 개로 쪼갠 다음 각각을 별도 에이전트한테 맡겨 버리는 거예요. 비유하자면요. 큰 공사 할 때 한 사람이 다 하는 게 아니잖아요. 전기 따로, 배관 따로, 도배 따로, 각자 전문가가 와서 하죠. 그것처럼 코드도 작업 단위로 쪼개서 각 서브에이전트가 처리해요. 작업 끝나면 다른 에이전트가 와서 검수해요. 이거 통과돼야지 다음 단계로 넘어가요." [04:13]

화자 평가 ([의견]):

> "[의견] 이 흐름이 실제 구현 속도를 진짜 많이 올려 줍니다." [04:31]

UI 작업의 경우 목업 이미지를 같이 분석한다는 점:

> "UI 만들 때도 같은 방식이에요. 슈퍼파워즈는 목업 이미지를 같이 보면서 분석해 줘요. 이 버튼은 여기 들어가고 이 색깔은 이렇게 이 동작은 이런 식으로 이미지에서 구현 가이드를 뽑아낸 다음 서브에이전트한테 분배해요." [04:36]

작업 단위 크기:

> "코드 짤 때도 그래요. 스펙 한 줄 던지고 끝이 아니라 파일 어디, 어떤 함수 넣을지, 입력은 뭐고 출력은 뭐고, 검증은 어떻게 할지 **2에서 5분짜리 작업 단위**까지 쪼개 주고 그 단위를 서브에이전트가 가져가요." [04:46]

### 2. 세 도구의 합치 공식

> "도메인 DDD는 맥퍼커, 스펙 브레인스토밍은 G-Stack, 실제 구현은 Superpowers. 이렇게 역할이 자연스럽게 나뉘어요." [00:59]

## 카탈로그: 전체 워크플로우 한 사이클

화자가 영상 중반과 마지막에 두 번 동일 순서로 정리한 전체 흐름.

> "전체 흐름 먼저 보여 드릴게요. 그릴미로 깊은 인터뷰 → 도메인 파일 작성 → (오피스아워) → 라이트스펙 → 플랜 → 플랜 리뷰 → 서브에이전트 코드 작성 → 코덱스 리뷰 → 임프로브 코드베이스 아키텍처로 마무리. **이거 한 번 돌리면 두세 시간 걸려요. 진짜로요.** 근데 그만큼 결과물이 다르거든요." [04:59]

각 단계 상세:

### Step 1. 맥퍼커 GrillMe — 깊은 인터뷰로 도메인 파일 작성

> "처음에 무조건 맥퍼커의 그릴미로 시작해요. 이런 거 만들고 싶어. 한 줄 던지고 클로드한테 인터뷰 해 달라고 해요. 그럼 진짜 30분 길면 한 시간 인터뷰가 시작돼요." [05:15]

여기서 화자가 강조하는 포인트:

> "**중요한 건 여기서 적당히 답하면 안 돼요. 진짜 깊게 답해야 돼요. 왜냐하면 이 인터뷰 결과가 그대로 도메인 파일이 되거든요.**" [05:23]

도메인 파일 = `context.md`:

> "이걸 context.md에 박아 두면 이후 모든 작업에서 클로드가 이 파일을 보면서 일해요." [05:34]

비유 — 새 직원 위키:

> "세(새) 직원 들어왔을 때 회사 위키 안 읽혀 주면 계속 이 용어 뭐예요? 이 프로세스 뭐예요? 물어보잖아요. 근데 위키 한 번 정독시키면 다음부터 알아서 일해요. 도메인 파일이 딱 그 역할이에요." [05:38]

### Step 2. G-Stack Office Hours — 비즈니스 임팩트 검증

> "도메인 파일 다 만들었으면 이제 G-Stack으로 넘어가요. 오피스(아워) 돌리는 거예요. 여기서 중요한 게요. 그릴(미가) 어떻게 만들지를 깊게 파는 거였다면 오피스 아워는 **이걸 진짜 만들어야 해**를 묻는 거예요. 질문 결이 달라요." [05:48]

두 단계의 효과:

화자가 짚는 결정적 포인트 — **여기까지 코드 한 줄도 안 짠 상태**:

### Step 3. Superpowers Light Spec → Light Plan → Plan Review

> "자, 이제 본격적으로 코드 짜기 시작합니다. 여기서부터 슈퍼파워즈로 넘어가요. 근데 **한 번에 풀 스펙 안 만듭니다. 라이트 스펙 먼저 짜요.** 핵심 기능 위주로 가볍게요. 그다음에 라이트 플랜 어떻게 구현할지 단계별로 쪼개요." [06:24]

Plan Review (커스텀 에이전트):

> "여기서 진짜 중요한 게 있어요. 저는 플랜 리뷰라는 커스텀 에이전트를 만들어 놨거든요. 에이전트 리뷰 기능을 이용해서요. 이 에이전트가 뭐 하는 애냐면 제가 짠 플랜을 보고 '여기 2단계 빼먹었어, 이거 순서 바꿔야 돼, 이거 너무 큰 작업이야, 저쪽에' 이렇게 빨간 펜으로 수정해 줘요." [06:34]

비유:

### Step 4. Subagent-Driven Development — 실제 코드 작성

> "플랜이 정해졌으면 이제 코드를 짤 시간이에요. 여기서 슈퍼파워즈의 서브에이전트 드리븐 디벨로프먼트를 돌려요. 플랜에 있는 작업들을 클로드가 알아서 서브에이전트한테 분배해요. 그러면 진짜 신기해요. 백그라운드에서 몇 개씩 동시에 돌아가기도 하거든요." [06:58]

화자 페이스 ([의견]):

> "[의견] 저 같은 경우에 보통 두 시간 정도 걸리는 거 같습니다. 그동안 저는 다른 일 하거든요. 영상 편집을 한다든지, 커피를 마신다든지, 일정 단계 끝날 때마다 알림서(알려 줘서) 검수하면 돼요." [07:09]

### Step 5. Codex 플러그인으로 중간 검수

> "**근데 그냥 다 맡겨 놓으면 안 됩니다. 중간 검수가 핵심이거든요.** 저는 코덱스 플러그인을 클로드 코드 안에서 같이 써요. GPT 코덱스랑 클로드가 서로 다른 모델이잖아요. 그래서 한쪽이 짠 코드를 다른 쪽이 보면 사각 지대가 좀 잡히는 거 같아요. 이거 이상한데 하고 짚어 주는 게 정말 많아요." [07:17]

화자의 인상 ([의견]):

> "[의견] 심지어 요즘 드는 기분으로는 **세세한 어떤 구현에 있어서는 코덱스가 훨씬 잘하는 거 같습니다.** 이걸 거치면 코드 퀄리티가 진짜 많이 올라가는 거 같아요. 대다수의 버그가 여기서 잡히는 거 같습니다." [07:31]

### Step 6. 맥퍼커 Improve Codebase Architecture — 도메인 기반 리팩토링

> "여기까지 오면 코드는 거의 완성이에요. 근데 마지막 한 끝이 남아 있거든요. 이때 맥퍼커에 임프로브 코드베이스 아키텍처를 돌려요. 이게 뭐냐면 이미 짜 놓은 코드를 보고 여기 더 깊게 모델링할 수 있는 부분이 있어 짚어 주는 스킬이에요. **도메인 언어 기반으로요.**" [07:43]

context.md를 기준으로 작동한다:

> "앞에서 만들어 놓은 context.md 도메인 파일 있잖아요. 그걸 기준으로 '지금 이 함수는 도메인 개념이랑 좀 어긋나, 이렇게 바꾸자. 이 둘은 사실 같은 개념인데 따로 떨어져 있어, 합치자.' 이런 식으로 리팩토링 포인트를 잡아 줘요." [07:54]

화자 평가 ([의견]):

> "[의견] 솔직히 말해서요, 이 스킬 쓰기 전후로 코드 퀄리티가 진짜 다르더라고요. 그냥 동작하는 코드에서 도메인 모델이 잘 잡힌 코드로 바뀌어요. 읽었을 때 '아, 이 도메인이 이렇게 생겼구나' 한 눈에 들어와요. 나중에 다른 사람이 들어와도 빨리 적응할 수 있고요." [08:05]

### Step 7. (추가) 스킬 기반 리팩토링 — 프런트엔드 / 백엔드

> "여기다 추가적으로 제가 하고 있는 작업들 같은 경우에는 프런트엔드 같은 경우에는 리액트, 버셀 베스트 프랙티스 같은 스킬들을 쓰거나 혹은 제가 커스텀으로 만든 이제 FastAPI 백엔드 가이드라인 같은 스킬들을 이용해 가지고 좀 잘못 짜여진 코드들을 알맞게 리팩토링하는 작업을 거칩니다." [08:19]

구체적 예시:
