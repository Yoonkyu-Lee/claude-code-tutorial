## 요약 (주제별)

### Vistara가 무엇인가 (칩의 정체)
- 메타가 ISCA 2026(2026-06-29, 미국 롤리, 산업 세션)에서 발표한 **Vistara**는 서버 운용 방법이 아니라 메타가 직접 설계한 **CXL Type 3 메모리 확장용 ASIC**이다. 학회 프로그램과 카메라 레디 논문이 공개돼 있다.
- 칩 구조: 한쪽에 PCIe 5세대 기반 CXL 인터페이스, 반대쪽에 DDR4 메모리 컨트롤러. 그 사이에서 CPU의 메모리 요청을 DDR4 명령으로 바꿔 주는 "통역사" 역할. 어댑터에 대충 꽂은 게 아니라 구형 DDR4를 정상 시스템 메모리처럼 쓰게 해 주는 완전한 메모리 컨트롤러 칩.
- 목적: 수명을 다한 **구형 서버에서 DDR4를 뽑아 새 서버에 재활용**. 최신 HBM도 차세대 DDR5도 아닌 구형 DDR4 재사용이 핵심.

### 실측 성능과 서버 구성 (메타 논문 수치)
- Figure 6 서버 구성: CPU 직결 DDR5 768GB + Vistara 카드 2장으로 재활용 DDR4 256GB 추가 → 서버 1대 총 1TB.
- 대역폭 격차: 로컬 DDR5 이론적 최대 약 **614 GB/s** vs CXL DDR4 피크 약 **76 GB/s** → 피크만 봐도 약 **8배** 차이. CXL은 PCIe 링크를 한 번 거쳐 대역폭도 낮고 지연도 길다.
- 그런데도 성과: 일부 ML 파라미터 서버에서 필요 서버 수 **최대 25% 감소**, 캐시 서비스에서 평균 지연 처리 시간 **29% 감소**(메타 보고).

### 왜 느린데도 이득인가 (메모리 계층 논리)
- CXL이 빨라서 나온 결과가 결코 아니다. 메모리가 부족하면 캐시에서 밀려난 데이터가 플래시/원격 저장장치(디램보다 수십~수백 배 느림)로 내려가고, 모델 파라미터는 더 많은 서버에 분산해야 한다.
- 느린 DDR4라도 용량을 붙여 두면 플래시로 떨어질 데이터를 디램 계층에 붙잡아 두고 모델도 더 적은 서버에 담을 수 있다.
- **핵심은 비교 대상**: CXL DDR4를 DDR5와 비교하면 당연히 느리지만, 플래시/원격 저장장치나 추가 투입해야 했던 서버와 비교하면 유리하다. [의견] CXL이 연산을 빠르게 만든 게 아니라 메모리 부족으로 생기던 더 큰 손실을 피한 것으로 봐야 한다.
- Figure 2 실사용 패턴: 모든 워크로드에서 CXL 대역폭 사용량이 로컬 메모리보다 훨씬 낮음(특히 ML 파라미터 서버). 자주 쓰는 데이터는 로컬 DDR5에 남고, 접근 빈도 낮은 페이지만 CXL로 배치.

### "HBM 다음은 CXL"이라는 프레임 반박
- [의견] "HBM 다음은 CXL"은 틀린 표현에 가깝다. HBM은 계산 중 데이터를 빠르게 공급하고, CXL은 그 바깥의 용량·활용률을 조정한다 — 푸는 문제 자체가 다르다.
- HBM은 고속 메모리 그 자체, CXL은 메모리가 오갈 수 있는 연결 통로. CXL 카드 안에 들어가는 것도 결국 대부분 DDR 계열 디램.
- 지연 민감·반복 접근 데이터(모델 가중치, 활성 데이터, KV 캐시)는 HBM에 남기는 게 유리. CXL은 접근 빈도가 더 낮은, 당장 안 쓰는 모델 조각 보관용 아래 계층에 적합.

### 메모리 쇼티지를 풀어 주나 (공급 관점)
- 구형 DDR4 재활용으로 신규 DDR5 추가 구매 부담을 일부 줄일 수 있다. 논문 인트로덕션: 서버 수명은 보통 5~7년인데 메모리는 10~14년까지 쓸 수 있어 서버가 한물 가도 딥(DIMM)은 더 쓸 수 있다(메타 주장).
- 다만 CXL이 디램 공급 자체를 늘리는 게 아니다. 새 CXL 모듈을 사면 그 안에도 결국 새 디램이 들어간다. **공급 부족 해법이라기보다 이미 만든 디램을 덜 낭비하는 방법**에 가깝다.
- "거의 비용이 들지 않는 방식"이라는 논문 표현과 달리, Vistara 2장 + 별도 카드 펌웨어·냉각·장애 감시가 필요하고 서버당 CXL 확장부가 약 **50W** 전력을 추가로 쓴다. 재활용할 DIMM 재고가 크고 자체 칩·서버 OS까지 설계할 수 있는 메타 같은 기업에서 경제성이 가장 잘 나오는 구조.

### "메모리 풀"이라는 표현의 주의점
- Vistara를 "메모리 풀"로 부르는 건 조심해야 한다. 실제로는 공동 수영장이 아니라 **서버마다 자기 전용 보조 물탱크를 붙인 구조**. 서버 한 대마다 카드 2장을 직접 연결해 DDR4 확장 메모리를 제공하며, 다른 서버와 공유하지 않는다.
- 진짜 메모리 풀 공유는 CXL 스위치·패브릭 매니저·주소 할당·접근 제어·장애 격리를 함께 관리해야 한다. 스위치 기반 CXL을 측정한 2026년 연구에서도 공유 경로 트래픽이 겹치자 특정 조건에서 접근 지연이 몇 배 넘게 늘고 전송 계층까지 제한됐다.
- [의견] 공유할 수 있다는 것과 예측 가능한 성능으로 공유할 수 있다는 것은 전혀 다른 문제.

### 소프트웨어 복잡성 (진짜 난제)
- CXL 업체는 "앱 수정 없이 일반 메모리처럼 쓸 수 있다"고 강조하지만, 복잡성이 사라지는 게 아니라 애플리케이션 바깥(OS)으로 이동할 뿐이다.
- Linux는 CXL DDR4를 CPU에 직접 붙지 않은 별도 NUMA 노드(먼 메모리)로 인식. 자주 쓰는 페이지는 가까운 DDR5에, 오래 안 쓴 페이지는 먼 CXL DDR4로 내리고 다시 뜨거워지면 끌어올린다.
- 메타도 DDR5/CXL 분배용 별도 관리 기능을 붙여야 했다. 여러 작업을 한 서버에 넣자 빠른 DDR5가 한쪽으로 치우쳐 다른 작업 처리량이 **최대 65% 하락**, 메모리 부족으로 종료되는 일까지 발생 → 작업마다 빠른 메모리 몫을 정하는 분배 정책을 추가.
- AI 서버는 더 복잡: GPU HBM, CPU DDR5, CXL 메모리, 플래시 계층이 겹치고 AI 런타임과 OS가 서로 다른 기준으로 데이터 위치를 관리. OS는 곧 필요할 데이터를 알기 어렵고, AI 런타임은 서버 전체 메모리 상황을 다 보지 못한다. 계층이 하나 늘 때마다 배치 결정 복잡성도 함께 는다.

### CXL이 잘 맞는 영역 vs 안 맞는 영역
- 잘 맞음: 대용량 캐시, 데이터 분석, 개발용 VM, 빌드 서버, 추천 시스템의 거대한 임베딩 테이블처럼 **용량이 먼저 부족해지고 상당 부분을 오래 읽지 않는** 워크로드.
- 안 맞음: 대규모 AI 학습에서 GPU가 계속 읽어야 하는 가중치·활성값을 CXL에 두고 직접 계산하는 것. 메타의 ML 사례도 GPU HBM을 CXL로 대체한 게 아니라 추천 시스템·분산 추론용 CPU 파라미터 서버 용량을 늘린 것.
- LLM 추론 후보: 긴 대화의 오래된 KV 캐시, MoE 구조에서 자주 호출 안 되는 전문가를 잠시 내려두는 용도. 단 이번 논문이 LLM 사례를 직접 검증한 건 아니고 업계에서 논의되는 방향(주장/미검증). 매 토큰마다 다시 가져와야 하면 느린 링크가 병목.

### 한국 기업(삼성·SK하이닉스·파네시아)과 시사점
- 삼성전자: CXL 2.0 + PCIe 5세대 기반 CMM-D 제품군 공개. 용량 확장 + 메모리 디스어그리게이션(메모리를 특정 CPU에 묶지 않고 떼어내 필요한 서버가 필요한 만큼 사용; 여러 모듈+스위치 필요). 어떤 데이터가 자주 쓰이는지 장치가 직접 추적하는 CHMU 기능 개발.
- SK하이닉스: DDR5 기반 CXL 메모리 + 지원 소프트웨어 개발 도구 공개, 용량 확장 시장 준비.
- 파네시아(Panmnesia, 국내 팹리스): ISCA'26에서 CXL 컨트롤러 + 포트 기반 라우팅 스위치 발표. Vistara가 서버별 메모리 확장이라면 파네시아는 여러 장치를 연결하는 패브릭·실제 메모리 풀에 더 가까운 영역.
- [의견] CXL 시장 성장이 한국 메모리 기업 디램 판매 증가로 곧바로 이어진다고 단정할 수 없다. 재활용은 새 DDR5 구매를 피하게 하지만, 새 CXL 모듈 채택이 늘면 서버당 디램 용량이 늘 수도 있다. 기회는 단순 디램 판매가 아니라 메모리 컨트롤러·CXL 스위치·데이터 온도 측정·페이지 배치 소프트웨어·펌웨어·장애 관리 등 이질적 메모리를 하나로 묶는 능력에 있다.

### 총평
- [의견] CXL은 많이 붙일수록 좋은 기술이 아니라 "느린 메모리로 내려도 되는 데이터가 얼마나 많은지"에 따라 가치가 갈린다. 상용화 여부도 모듈 용량보다 실제 워크로드·성능·전력·스위치 비용을 함께 봐야 한다.
- 메타의 기여는 논문·시제품에 머물던 CXL을 실제 서버·OS·프로덕션 서비스까지 연결한 것. 다만 확인된 사용처는 HBM을 밀어내거나 AI 연산을 폭발적으로 빠르게 하는 게 아니라, 다 쓴 DDR4를 다시 쓰고 메모리 부족으로 남아 있던 서버 자원을 더 활용하는 기술. HBM과 CXL은 이전/그다음 관계가 아니라 각각 계산 속도와 바깥 메모리 활용을 담당.

## 3줄 오리엔테이션
- 메타가 ISCA 2026에서 공개한 Vistara는 수명 다한 구형 DDR4를 새 서버에 재활용하기 위해 직접 설계한 CXL Type 3 메모리 확장 칩이다.
- 성과(서버 25% 절감, 캐시 지연 29% 감소)는 CXL이 빨라서가 아니라, 데이터가 훨씬 느린 플래시로 밀려나거나 서버를 추가 투입하는 손실을 피했기 때문이다.
- 화자는 "HBM 다음은 CXL"·"쇼티지 해법"·"메모리 풀" 같은 프레임을 반박하며, CXL의 본질은 용량 확장과 배치 소프트웨어 복잡성 관리라고 정리한다.

### [00:56–01:39] 논문의 정체 — 구형 DDR4 재활용, 그리고 영상의 목적
학회 발표 내용은 최신 HBM도 차세대 DDR5도 아닌, 이미 사용을 마친 구형 서버의 DDR4를 뽑아 새 서버에 다시 꽂겠다는 것이다. 단순 재활용 캠페인이 아니라 이를 위해 직접 반도체 칩까지 만들었다 — 오래된 DDR4를 새 서버에 연결하는 CXL 메모리 확장 칩이다. 세계 최대 데이터센터가 메모리 문제를 어떻게 푸는지, 이것이 램 시장과 메모리 기술 방향에 어떤 힌트가 될지 살펴보는 게 오늘의 목적이라고 밝힌다.

### [01:41–02:19] 통념 vs 화자의 시각, 그리고 발표 배경
한편에서는 이 사례를 두고 "HBM 다음의 미래는 CXL이다", "메모리 부족을 해결할 새로운 기술이다"라는 이야기가 나온다. [의견] 화자는 이번 발표를 보며 조금 다른 생각이 들었다고 한다 — CXL이 메타 실제 서비스에서 효과를 낸 건 확인됐지만, 우리가 기대했던 방식과는 상당히 다르다는 것. 기술 이름은 **Vistara**. 2026년 6월 29일 미국 롤리에서 열린 컴퓨터 구조 학회 ISCA 2026 산업 세션에서 발표됐고, 학회 프로그램과 카메라 레디(최종) 논문이 현재 공개돼 있다.

### [02:19–03:14] 칩인가 소프트웨어인가 — CXL과 Vistara 구조
Vistara는 단순한 서버 운용 방법이 아니라 메타가 직접 설계한 CXL Type 3 메모리 확장용 ASIC이 실제로 들어간다. CXL은 CPU·메모리·가속기 같은 장치를 하나의 규격으로 연결하는 인터커넥트로, PCIe 위에서 동작하며 외부에 붙인 메모리도 CPU가 자기 메모리처럼 다루게 해 주는 게 핵심이다. Vistara 칩은 그 CXL을 실제로 구현한 물건 — 한쪽에 PCIe 5세대 기반 CXL 인터페이스, 반대쪽에 DDR4 메모리 컨트롤러가 있어 CPU가 보는 메모리 요청을 DDR4 명령으로 바꿔 주는 통역사 역할을 한다.
> "오래된 메모리를 어댑터에 그냥 대충 꽂은 게 아니라 DDR4를 새로운 서버가 정상적인 시스템 메모리처럼 쓸 수 있도록 만든 완전한 메모리 컨트롤러 칩" [03:05]

### [03:14–03:52] 서버 구성과 8배의 대역폭 격차
Figure 6 서버 구성: CPU에 직접 연결된 빠른 DDR5 768GB에, Vistara 카드 2장을 통해 재활용 DDR4 256GB를 추가해 서버 한 대 메모리를 총 1TB로 만든다. 성능 차이가 상당하다 — 로컬 DDR5의 이론적 최대 대역폭은 초당 약 614GB인 반면, 논문이 제시한 CXL DDR4 쪽 피크 대역폭은 약 76GB. 피크만 봐도 8배 가까이 차이 나고, CXL 메모리는 PCIe 링크를 한 번 거쳐 대역폭도 낮고 지연도 더 길다.

### [03:52–05:11] 느린데도 이득인 이유 — 메모리 계층과 비교 대상
이렇게 느린 메모리를 붙였는데도 메타는 일부 ML 파라미터 서버에서 필요 서버 수를 최대 25% 줄였고, 캐시 서비스에서 평균 지연 처리 시간을 29% 낮췄다고 보고했다. 메모리 하이어라키 관점에서 보면 이는 CXL이 빨라서 나온 결과가 결코 아니다. 메모리가 부족하면 캐시에서 밀려난 데이터는 플래시나 원격 저장장치까지 내려가고 모델 파라미터는 더 많은 서버에 나눠 담아야 하는데, 플래시는 디램보다 수십~수백 배 느리다. 느린 DDR4라도 용량을 더 붙여 두면 플래시로 떨어질 데이터를 디램 계층에 붙잡아 두고 모델도 더 적은 서버에 담을 수 있다. [의견] 핵심은 비교 대상이다 — CXL DDR4를 빠른 DDR5와 비교하면 당연히 느리지만, 플래시/원격 저장장치까지 밀려날 데이터나 추가 투입해야 했던 서버와 비교하면 훨씬 유리하다. CXL이 연산을 빠르게 만든 게 아니라 메모리 부족으로 생기던 더 큰 손실을 피한 것으로 봐야 한다.

### [05:11–06:25] HBM은 고속 메모리, CXL은 연결 통로 — Figure 2 실사용 패턴
HBM이 고속 메모리 그 자체라면 CXL은 메모리가 오갈 수 있도록 만든 연결 통로에 가깝고, CXL 카드 안에 들어가는 것도 결국 대부분 DDR 계열 디램이다. AI 학습·추론에서 매 순간 읽어야 하는 모델 가중치·활성 데이터, 그리고 앞서 처리한 토큰을 다시 계산하지 않으려 저장하는 KV 캐시처럼 반복 접근·지연 민감 데이터는 가능하면 HBM에 남기는 게 유리하다. CXL은 그보다 접근 빈도가 낮은, 당장 안 쓰는 모델 조각을 보관하는 아래 계층에 어울린다. Figure 2에서 파란색은 로컬 DDR, 주황색은 CXL 메모리의 용량 대비 실제 대역폭 사용량인데, 모든 워크로드에서 CXL 쪽 막대가 훨씬 낮게 나온다(특히 ML 파라미터 서버). 자주 쓰는 데이터는 로컬 DDR5에 남고 CXL에는 접근 빈도가 낮은 페이지가 배치됐기 때문이다.

### [06:25–06:41] "HBM 다음은 CXL"이라는 표현 반박
[의견] 그래서 "HBM 다음은 CXL"이라는 표현은 어떻게 보면 틀린 표현에 가깝다. HBM은 계산 중에 데이터를 빠르게 공급하고, CXL은 그 바깥의 용량과 활용률을 조정하는 것이니 푸는 문제 자체가 다르다.

### [06:41–07:33] 쇼티지를 푸는가 — 재활용 vs 공급 증가
그렇다면 메모리 쇼티지를 CXL이 풀어 줄 수 있을까? 여기서도 구분이 필요하다. Vistara처럼 구형 서버 DDR4를 다시 쓰면 용량 확장을 위해 새 DDR5를 추가 구매하는 부담은 일부 줄일 수 있다. 논문 첫 페이지 인트로덕션에서 메타는 서버 수명 5~7년, 메모리 수명 10~14년이라 서버는 한물 가도 딥은 더 쓸 수 있다고 설명한다. 하지만 CXL이 디램 공급 자체를 늘려 주는 게 아니다 — 기존 DDR4 재활용은 신규 구매를 일부 줄이지만, 새 CXL 모듈을 사면 그 안에도 결국 새 디램이 들어간다. 즉 공급 부족 해법이라기보다 이미 만들어진 디램을 조금 덜 낭비하는 방법에 가깝다. 논문은 재활용을 통한 확장을 "거의 비용이 들지 않는 방식"으로 표현하지만, 시스템 전체가 무료인 것도 아니다.
> "비스타라 두 개랑 별도의 카드 펌웨어·냉각·장애 감시가 필요하고 서버당 CXL 확장부가 약 50W의 전력을 또 추가로 쓰게 돼요" [07:44]

### [07:33–08:00] 경제성이 나오는 조건
[의견] 이 방식은 재활용할 딥 재고가 크고 자체 칩과 서버 운영체제까지 설계할 수 있는 메타 같은 기업에서 그나마 경제성이 가장 잘 나오는 구조라고 정리한다.

### [08:00–09:09] "메모리 풀" 표현 주의 — 전용 물탱크 비유
Vistara를 "메모리 풀"이라 부르는 표현도 조심해야 한다. 이름만 보면 여러 서버가 하나의 거대한 메모리 공간을 함께 쓰는 구조처럼 들리지만, 실제로는 공동 수영장이라기보다 서버마다 자기 전용 보조 물탱크를 하나씩 붙인 구조에 가깝다. Vistara는 서버 한 대마다 카드 2장을 직접 연결해 DDR4 확장 메모리를 제공하며, 다른 서버와 메모리를 공동으로 나눠 쓰는 구조가 아니다. 여러 서버가 하나의 메모리 풀을 실제로 공유하려면 CXL 스위치·패브릭 매니저는 물론 주소 할당·접근 제어·장애 격리까지 함께 관리해야 한다. 실제 스위치 기반 CXL을 측정한 2026년 연구에서도 공유 경로에서 트래픽이 겹치자 특정 조건에서 접근 지연이 거의 몇 배 넘게 늘었고, 이를 관리하는 별도 전송 계층까지 제한됐다. [의견] 메모리를 한 곳에 다 모으면 효율적일 것 같지만 그 메모리로 가는 길도 여러 서버가 함께 써야 하니, 공유할 수 있다는 것과 예측 가능한 성능으로 공유할 수 있다는 것은 전혀 다른 문제다.

### [09:09–10:57] 소프트웨어 복잡성 — OS로 이동한 문제
소프트웨어 문제도 있다. CXL 업체들은 애플리케이션을 수정하지 않아도 일반 메모리처럼 쓸 수 있다고 강조하는데, 틀린 말은 아니지만 복잡성이 사라지는 게 아니라 애플리케이션 바깥으로 이동하는 것이다. 원래 서버는 CPU에 붙은 메모리 한 종류만 다루면 됐지만, 속도가 다른 메모리가 둘로 늘면 무엇을 어디에 둘지 판단하는 일이 새로 생기고 이 판단은 운영체제 몫이 된다. Linux는 CXL DDR4를 CPU에 직접 붙지 않은 별도 NUMA 노드(먼 메모리 구역)로 인식하고, 자주 쓰는 페이지는 가까운 DDR5에 남기고 오래 안 쓴 페이지는 먼 CXL DDR4로 내리며 다시 자주 쓰이면 끌어올린다. 문제는 소프트웨어가 생각보다 단순하지 않다는 점 — 메타도 CXL과 DDR5 사이에서 데이터를 나눠 쓰기 위해 별도 관리 기능을 붙여야 했다.
> "여러 가지 작업을 한 서버에 넣어 버리자 빠른 DDR5가 한쪽으로 치우치면서 다른 작업의 처리량이 최대 65% 떨어지고 메모리 부족으로 심지어 종료되는 일까지 생겼습니다" [10:09]

결국 이를 막기 위해 작업마다 빠른 메모리의 몫을 따로 정하는 분배 정책까지 추가해야 했다. AI 서버에서는 문제가 더 복잡해진다 — GPU의 HBM, CPU DDR5, CXL 메모리, 플래시까지 여러 계층이 겹치고, AI 런타임과 운영체제가 각각 다른 기준으로 데이터 위치를 관리하기 때문이다. 운영체제는 어떤 데이터가 곧 다시 필요할지 정확히 알기 어렵고 AI 런타임도 서버 전체 메모리 상황을 다 보지 못한다. CXL은 용량을 늘려 주지만 메모리 계층이 하나 늘 때마다 데이터를 어디 둘지 결정하는 복잡성도 함께 늘어나는 기술이다.

### [10:57–12:10] CXL이 잘 맞는 영역과 안 맞는 영역
그렇다고 CXL이 전혀 필요 없다는 뜻은 아니다. 이번 메타 사례에서 CXL이 잘 맞는 영역은 꽤 분명하다 — 대용량 캐시, 데이터 분석, 개발용 가상 머신, 빌드 서버, 추천 시스템의 거대한 임베딩 테이블처럼 메모리 용량이 먼저 부족해지고 전체 데이터 중 상당 부분을 오랫동안 읽지 않는 워크로드다. 반면 대규모 AI 학습에서 GPU가 계속 읽어야 하는 가중치·활성값을 CXL 메모리에 두고 직접 계산하는 건 어렵다. 메타가 이번 논문에서 보여 준 ML 사례도 GPU HBM을 CXL로 대체한 게 아니라 추천 시스템·분산 추론에 쓰이는 CPU 파라미터 서버의 용량을 늘린 것으로 봐야 한다. LLM 추론에서도 활용 후보로 거론되는데, 긴 대화에서 오래된 KV 캐시나 모델을 여러 전문가로 나눠 필요한 것만 불러오는 MoE 구조에서 자주 호출 안 되는 전문가를 잠시 내려두는 용도다. 다만 이번 논문이 이런 LLM 사례까지 직접 검증한 건 아니고 업계에서 논의되는 방향(주장/미검증)이다. CXL에 내려둔 데이터를 매 토큰마다 자주 다시 가져와야 한다면 느린 링크가 병목이 되고, [의견] 데이터를 내려놓는 것보다 다시 필요해질 시점을 정확히 예측하는 소프트웨어가 더 중요해질 수도 있다.

### [12:10–13:16] 삼성·SK하이닉스·파네시아 현황
우리나라 삼성전자와 SK하이닉스 제품·기술 현황도 살펴본다. 삼성전자는 CXL 2.0과 PCIe 5세대 기반 CMM-D 제품군을 공개했고, 주요 가치로 용량 확장과 메모리 디스어그리게이션을 제시한다. 디스어그리게이션은 메모리를 특정 CPU에 묶지 않고 떼어내 필요한 서버가 필요한 만큼 끌어다 쓰게 만드는 방향인데, 모듈 하나만 꽂으면 되는 게 아니라 여러 모듈과 스위치를 묶어야 가능한 구조다. 삼성은 여기에 더해 어떤 데이터가 자주 쓰이는지 장치가 직접 추적하는 CHMU 기능도 개발해 왔다. SK하이닉스도 DDR5 기반 CXL 메모리와 이를 지원하는 소프트웨어 개발 도구를 공개하며 용량 확장 시장을 준비하고 있다. 국내 팹리스 파네시아(Panmnesia)는 이번 ISCA'26에서 CXL 컨트롤러와 포트 기반 라우팅 스위치를 발표했다 — 메타 Vistara가 서버별 메모리 확장에 가까웠다면 파네시아는 여러 장치를 연결하는 패브릭과 실제 메모리 풀에 더 가까운 영역을 겨냥한다고 한다.

### [13:16–13:53] 한국 메모리 기업의 기회는 디램 판매만이 아니다
[의견] 다만 CXL 시장 성장이 한국 메모리 기업의 디램 판매 증가로 곧바로 이어진다고 단정할 수는 없다. 메타처럼 기존 DDR4를 재활용하면 오히려 새 DDR5 구매를 피할 수 있고, 반대로 삼성·SK하이닉스의 새 CXL 모듈이 본격 채택되면 서버당 장착 가능한 디램 용량이 늘 수도 있다. 따라서 한국 기업의 기회도 단순히 디램을 더 파는 데만 있지 않다 — 메모리 컨트롤러, CXL 스위치, 데이터 온도 측정, 페이지 배치 소프트웨어, 펌웨어와 장애 관리까지 이질적인 메모리를 하나의 시스템으로 묶는 능력이 함께 필요하다.

### [13:53–14:52] 총평 — CXL의 본질과 메타 기여의 의미
[의견] CXL은 많이 붙일수록 좋은 기술이 아니라, 느린 메모리로 내려도 되는 데이터가 얼마나 많은지에 따라 가치가 갈리는 기술이다. 그래서 상용화 여부도 모듈 용량보다 실제 워크로드·성능·전력·스위치 비용을 함께 봐야 하고, 데모에서 메모리가 인식되는 것과 데이터센터에서 안정적으로 운영되는 것은 전혀 다른 문제다. 메타가 이번에 보여 준 건 분명 의미가 있다 — 논문·시제품에 머물던 CXL을 실제 서버·운영체제·프로덕션 서비스까지 연결했으니까. 하지만 이번에 확인된 사용처는 HBM을 밀어내거나 AI 연산을 폭발적으로 빠르게 만드는 기술이 아니라, 사용을 다한 DDR4를 다시 쓰고 메모리 용량이 부족해 남아 있던 서버 자원을 더 활용하는 기술이었다. 그래서 HBM과 CXL은 이전과 그다음의 관계가 아니다 — HBM이 계산에 필요한 속도를 담당한다면 CXL은 그 바깥의 메모리를 어디까지 활용할 수 있는지를 다룬다. 이 부분을 헷갈리지 말라는 당부로 마무리한다(안될공학 패치).
