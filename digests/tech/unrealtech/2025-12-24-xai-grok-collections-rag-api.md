## 요약 (주제별)

### 발표 개요 — Grok Collections API
- xAI가 개발자용 **그록 컬렉션스(Collections) API**를 내놨다. 핵심은 이 API를 통해 **RAG(Retrieval-Augmented Generation)를 손쉽게 구현**할 수 있게 하는 것.
- 배경 시장 상황: 12월 OpenAI GPT-5.2 등장, 11월 제미나이 3 Pro, 이번 달 제미나이 3 Flash까지 나오며 AI 경쟁이 치열한 가운데 그록이 "차별화된 접근"을 취한다는 것이 화자 관점.
- 활용 예: 개발자가 API를 호출해 주식·파이낸셜 데이터나 법적 데이터의 성능을 끌어올린 앱을 손쉽게 만들 수 있고, 이를 이용한 수많은 앱이 나올 것 [의견].

### RAG란 무엇인가 (해설)
- RAG는 LLM이 자체 학습 데이터로만 답하는 게 아니라, 별도로 모아 둔 **데이터베이스(최신 정보 포함)를 참고**해 답하고 인용 출처를 연결해 주는 방식. 제미나이·챗봇 검색에서 링크를 달아 주는 게 그 예.
- 일반적으로 RAG 구축은 번거롭다: 문서 수집 → 실시간 데이터 반영 → **벡터 DB**에 업로드(권한·버전 관리 포함) → PDF·코드·HTML 등 **파싱** → **청킹**으로 분할 저장. 그래야 질의 시 유사도 기반으로 근거를 추출해 LLM 프롬프트에 주입하고 출처까지 낼 수 있다.
- 이렇게 하면 작년까지만 학습된 모델이라도 벡터 DB를 통해 실시간 정보를 가져올 수 있다.

### 그록의 강점 포지셔닝 — 실시간성
- 그록이 늘 내세우는 건 X(트위터)가 세계에서 가장 빠르게 정보를 발행·전달하는 플랫폼이라는 점. 그 정보를 바로 RAG로 업데이트해 실시간 정보를 잘 찾는다는 정평 [의견].
- 발표에서도 "X 플랫폼의 리얼타임 글로벌 인사이트"를 강조 — SNS를 갖지 못한 기업도 실시간 바이럴 정보를 잘 트래킹할 수 있게 하고, **초저지연**으로 실사용에 쓸모 있는 것을 내세운다.

### API·컬렉션 작동 방식
- API는 개발자가 이미 만들어진 좋은 기능을 호출해 자기 앱·웹에 바로 붙여 쓰는 것(네이버·카카오·구글 지도가 모두 API인 것과 같은 원리). 개발자가 모든 걸 다 만들 수 없으니 돈을 내고 잘 만들어진 걸 쓰라는 것.
- 그록은 이를 **컬렉션(Collection)** 단위로 제공. 원하는 문서·정보를 컬렉션에 넣어 두면, 그 컬렉션을 대상으로 검색하거나 질문하면 된다. 앞단의 파싱 등을 개발자가 직접 안 해도 되고, "API에 내장된 RAG"로 포지셔닝.
- 텍스트 추출, PDF·엑셀 등 여러 파일 포맷을 지원. 재무 정보를 넣어 주식 정보 앱이나 추천 앱을 손쉽게 만들 수 있다.

### 가격 모델
- **검색 1,000건당 2.5달러 고정** (주장/미검증). 첫 주에는 무료로 풀었다.
- 통상적인 토큰당 과금과 달리, 몇 개의 토큰을 쓰든 상관없이 **검색 횟수 기준**으로 과금 → 개발자가 앱 사용 비용을 예측하기 쉬워 합리적이라는 것이 화자 평가.

### 성능 주장 (그록 측 벤치마크)
- 재무 데이터·코딩·법적 자문 데이터에서 대체로 점수가 높다. 특히 주식·재무 정보는 매우 높아 "압살"하고, GPT-5.1과 비교했다 (주장/미검증).
- 법적 데이터는 제미나이 3 Pro가 다소 앞서지만 거의 유사한 수준. 구글의 유사 기능, OpenAI의 벡터 스토어류보다 그록 컬렉션이 훨씬 높다고 주장.
- 테스트 조건(그록 제기): 30개 태스크에 문서 13개(전체 ~2,500페이지) 투입. DeepCode 벤치에서는 코드베이스 RAG로 8개 레포·레포당 ~8,000개 파일을 넣어 테스트했더니 정확도가 매우 높게 나왔다 (주장/미검증, 화자도 "다른 경우도 있을 수 있다"고 단서).

### 개발자 전략 — SDK 호환·에이전트 스택
- RAG를 **API 퍼스트클래스 오브젝트**로 전면에 내세워, 문서를 컬렉션으로 묶고 그 컬렉션에 대한 검색·응답을 반복하는 형태로 개발자 워크플로를 구성.
- OpenAI SDK·앤트로픽 SDK와 **호환**되면서 그록만의 실시간성을 얹어, RAG 포함 **에이전트 스택을 갈아타기 쉽게** 해 개발자를 잡으려는 전략 [의견].

### 경쟁사 비교 (구글·OpenAI)
- 구글: **Vertex AI Search로 그라운딩**(=RAG). 구글 클라우드 콘솔에서 데이터 스토어를 만들어(최대 10개) 제미나이를 사용자 데이터로 그라운딩, 응답에 그라운딩 메타데이터(근거 청크)·컨피던스 스코어까지 포함 가능. 다만 여전히 **토큰당 검색**이고 구글 클라우드 안에서 해야 한다. 제미나이 플래시 컨텍스트 언급: 1M 또는 64K [불명확].
- OpenAI: **벡터 스토어 API** 제공. 다만 다 풀어 놓고 벡터 DB·RAG 구성을 개발자가 알아서 하게 하는 형태여서 여러 단계 조율이 필요.
- 그록의 차별점: 인덱싱·검색 인프라를 **자동 제공**, **단일 API 호출**만으로 구성 가능. 운영 관리 측면에서 API 하나만 관리하면 되는 반면 경쟁사는 여러 서비스를 직접 연동·관리해야 함 → 이 점에서 그록이 앞선다고 볼 수 있음 [의견].

### 리스크·유보

### [01:00–01:33] RAG 개념 해설
이 API의 핵심은 **API를 통해 RAG를 구현**할 수 있다는 것. API도 RAG도 모르는 시청자를 위해 설명한다. RAG란 제미나이·챗봇이 검색해 링크를 달아 주듯, LLM이 직접 학습한 데이터로만 답하는 게 아니라 **특정 데이터 파일을 모아 둔 데이터베이스**(최신 정보 포함)를 참고해 답하는 것. 물어보면 그 DB를 참고해 LLM에 질의하므로, 실시간 정보를 가져오고 인용 데이터도 직접 연결해 준다.

### [01:33–02:20] RAG 구축이 원래 번거로운 이유
이런 시스템을 구축하려면 일반적으로 문서를 수집하고, 어떤 데이터를 쓸지 정하고, 계속 쏟아지는 실시간 데이터를 **벡터 DB**에 지속 업데이트해야 한다(파일 업로드, 권한 정리, 버전 관리 포함). 업로드만으로 끝이 아니라 PDF·코드·HTML 등 데이터를 전부 **파싱**하고 **청킹**으로 구분해 벡터 DB에 저장해 둬야, 질의 시 유사도 관점에서 근거를 추출해 LLM 프롬프트에 주입하고 출처까지 낼 수 있다. 그래서 작년까지만 학습된 모델이라도 벡터 DB로 실시간 정보를 가져올 수 있다.

### [02:20–02:44] 그록의 실시간성 강점

### [02:44–03:14] API란 무엇인가 (지도 비유)

### [03:14–03:56] 컬렉션(Collection) 단위 작동 방식
XAI의 컬렉션스 API는 결국 **컬렉션이라는 단위로 묶어** 제공하는 형태다. 개발자가 원하는 문서·정보를 컬렉션에 넣어 두면, 그 컬렉션을 대상으로 검색하거나 질문하면 된다. 개발자는 컬렉션에 자료만 넣어 두면 되고, 앞단의 파싱 등을 직접 안 해도 RAG가 손쉽게 구축된다. "API에 내장된 RAG"로 포지셔닝함으로써, 재무 정보를 토대로 주식 정보 앱이나 추천 앱을 손쉽게 만들 수 있다.

### [03:56–04:16] 지원 포맷과 성능 주장

실제 성능을 보면 재무 데이터·코딩·법적 자문 데이터가 대체로 다 높다. 주식·재무 정보는 매우 높아 "압살"하며, 비교는 GPT-5.1과 했다. 법적 데이터는 제미나이 3 Pro가 좀 더 앞서지만 거의 유사한 수준. 구글도 파이낸셜 관련 기능이 있고 OpenAI VS(벡터 스토어) 같은 것도 있는데, 그록 컬렉션이 훨씬 더 높다고 주장한다. 이는 그록이 제기한 것으로, **30개 태스크에 문서 13개(전체 2,500페이지)**를 넣어 테스트한 결과다 (주장/미검증). 화자도 "다른 경우도 있을 수 있다"고 단서를 단다.

### [06:12–06:28] DeepCode 벤치 결과
DeepCode 벤치에서도 **코드베이스 RAG** 기반으로, **8개 레포에 레포당 약 8,000개 파일**을 넣어 테스트했더니 정확도가 매우 높게 나왔다고 한다 (주장/미검증).

### [06:28–06:57] 포지션 — 실시간 데이터·초저지연
그록의 에이전트 API는 결국 **도구 호출과 실시간 데이터 검색을 전면**에 둔다. XAI 발표에서도 "X 플랫폼의 리얼타임 글로벌 인사이트"를 강조하는데, SNS를 갖지 못한 기업도 X의 실시간·바이럴 정보 트래킹 장점을 활용하게 하고, 이를 기반으로 RAG API를 **초저지연**으로 지원해 실사용에서 쓸모 있는 것을 내세운다.

### [06:57–07:53] 경쟁사도 유사 기능 보유 — 구글 Vertex AI Search
물론 구글·OpenAI에도 이런 게 없는 건 아니다. 구글은 **Vertex AI Search로 그라운딩**(그라운딩=RAG)한다. 구글 클라우드 콘솔에서 **데이터 스토어**를 만들어(내가 원하는 데이터를 넣어 구성) 제미나이를 사용자 데이터로 그라운딩할 수 있다. **최대 10개 데이터 스토어**를 붙일 수 있고, 응답에 그라운딩 메타데이터(근거 청크)나 정확도를 나타내는 **컨피던스 스코어**까지 포함할 수 있다고 문서에 정리돼 있다. 제미나이 플래시 컨텍스트는 1M 또는 64K로 언급 [불명확]. 다만 버텍스 AI도 여전히 **토큰당 검색**이고 구글 클라우드 기반 안에서 해야 한다는 제약이 있다.

### [07:53–08:39] OpenAI 벡터 스토어 vs 그록 — 운영 단순성
OpenAI도 **벡터 스토어 API**가 있다. 전체를 비교하면, OpenAI에서 잘 안 되는 부분(인덱싱, 검색 인프라 자동 제공 등)을 그록이 잘 차지한 것 같다. OpenAI는 벡터 DB·RAG 구축 API를 다 풀어 놓고 나머지는 개발자가 알아서 구성하게 하는 형태여서 여러 단계를 조율해야 하고, 제미나이도 여러 서비스를 연동해야 한다. 반면 그록은 **단일 API 호출**만으로 구성되고, 운영 관리 측면에서 API 하나만 관리하면 되므로 이 점에서 그록이 훨씬 앞선다고 볼 수 있다 [의견].

### [08:39–09:08] 유보·클로징
다만 이 사용성은 아직 검증돼야 하며, 현재는 그록 측 주장일 뿐이다. 호스팅 측면에서 **xAI 서비스에 종속**되는 면이 있으나 호환되는 부분이 있으니, 얼마나 잘 쓰일지 지켜볼 필요가 있다. 진행자는 인스타그램 '언리얼테크 에러'에서 유튜브에 올리지 않는 빠른 소식·다양한 영상도 올린다며 관심을 부탁하고 마무리한다.
