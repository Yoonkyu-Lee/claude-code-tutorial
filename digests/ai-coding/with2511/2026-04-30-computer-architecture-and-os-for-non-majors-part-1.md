## 요약 (주제별)

### 컴퓨터 구조 기본 골격 (폰노이만)
- 컴퓨터는 크게 **CPU · 메모리 · 디스크 · IO**로 구성된다. IO는 Input/Output으로, 인풋에는 키보드·마우스, 아웃풋에는 모니터·프린터가 속한다.
- CPU와 메모리 사이, 메모리와 디스크 사이에는 데이터를 실어 나르는 **버스(bus)**가 있다. 화자는 버스를 "데이터를 싣고 다니는 버스" 개념으로 비유.
- 이 CPU·메모리·디스크·IO를 버스로 연결한 형태를 **폰노이만이 제시**했고, 현대 컴퓨터 구조의 근간이 되었다. [의견] 최근 컴퓨터는 이보다 훨씬 복잡하지만, 어찌 됐든 CPU·메모리·디스크로 구성된다는 큰 틀은 그대로다.

### 프로그램과 언어 (고급 언어 · 컴파일 vs 인터프리터)
- **프로그램 = 어떤 목적을 가진 소프트웨어**. 웹 브라우저, 워드프로세서 모두 프로그램이다.
- C·C++·자바는 **고급 언어**: 사람이 이해하기 쉽고(`if` 같은 문법) 컴퓨터가 직접 이해하긴 어렵다. 문법은 컴퓨터가 아니라 사람 편의를 위한 것.
- **컴파일 언어**(C/C++/자바): 소스 코드가 그대로 실행되는 게 아니라 다른 형태로 **변환(컴파일)**돼 실행된다. C/C++는 컴파일하면 기계어 **exe 파일**, 자바는 **바이트 코드**(클래스 파일)로 변환돼 **JVM(자바 가상 머신)** 위에서 실행된다.
- **인터프리터 언어**(자바스크립트/파이썬): 소스 코드를 **한 줄 한 줄 해석해 실행**한다. 파이썬은 파이썬 모듈이 실행하고, 자바스크립트는 브라우저가 실행한다.
- 브라우저 예시: URL 입력 → 해당 서버 접속 → 웹 서버가 HTML·CSS·자바스크립트를 응답으로 내려줌 → 브라우저가 화면에 렌더링. 브라우저에는 자바스크립트를 해석·실행하는 **인터프리터 엔진**이 내장돼 있다.

### 프로세스 (실행 중인 프로그램)
- 프로그램은 파일 형태로 저장돼 있고, 더블 클릭하면 실행된다. **실행된 상태 = 프로세스**. 즉 "실행되고 있는 프로그램 = 프로세스".
- 컴퓨터 구조상 표현: 설치된 프로그램을 더블 클릭 → 프로그램 **일부가 메인 메모리로 올라감** → 메모리의 일부가 **CPU로 올라가 명령어가 실행됨**.
- 소스 코드는 결국 **명령어들의 집합**이다. 소스 코드 = 명령어들을 만들어 놓은 것으로 봐도 된다.

### 메모리 계층 구조 (왜 프로그램을 다 안 올리나)
- 프로그램 "일부"만 메모리에 올린다. 이유를 설명하는 개념이 **메모리 계층 구조**.
- 디스크·메모리 둘 다 저장 역할을 하지만, **메모리는 비싸고 고성능·고속**(집적회로), **디스크는 상대적으로 쌈**. 용량당 단가는 메모리가 비싸지만 읽고 쓰는 속도가 압도적으로 빠르다.
- 많이 저장하고 싶으면서 빠르게 처리도 하고 싶은 요구를 절충한 결과, **CPU와 디스크 사이에 메모리를 둬서** 필요한 데이터를 올려놓고 빠르게 제공한다.
- 비유: 디스크=여러 책장, 메모리=내 책상 위 책꽂이. 자주 보는 책(=곧 CPU가 실행할 것들)을 책상 책꽂이에 두면 바로 뺄 수 있다.

### 시분할과 동시 실행
- 실제로는 한 프로그램만이 아니라 **수많은 프로그램이 동시에**(백그라운드 포함) 실행된다. 브라우저·워드·유튜브·AI를 동시에 띄워도 유튜브 끊김 없이 워드를 칠 수 있다.
- 여러 프로그램의 일부가 각각 메모리에 올라가 있고, CPU가 명령어들을 **엄청나게 빠른 속도로** 하나씩 처리한다. CPU = "엄청난 속도의 명령어 처리".
- 너무 빨라서 마치 동시에 실행되는 것처럼 보인다. 프로그램을 번갈아 반복 실행하는 이 모습을 **시분할(time-sharing)**이라 한다. 시분할 = 시간(T=time)을 쪼갠다. 하나의 CPU에 시간을 쪼개 1·2·3·4 작업을 번갈아 시키며, 1초도 안 되는 시간에 다른 일로 바꾼다.

### 메모리 한계와 교체 기법
- 메모리는 충분히 크지 못하다. 많은 프로그램은커녕 요즘 몇 GB짜리 한 프로그램도 통째로 못 올려서 일부만 올린다.
- 실행 중, 올라오지 않은 다른 영역이 실행에 필요해지는 경우가 생긴다. 그때는 **메모리에 없는 그 프로그램의 다른 영역을 찾아 다시 올려야** 한다.
- 이렇게 메모리의 어떤 영역이 교체되는 상황에서 **메모리 교체 기법(replacement)**이 등장한다. (2부 예고 주제)

### [00:00–00:32] 인트로 — 오늘 주제는 컴퓨터 구조와 운영 체제
채널 화자(기솔루트 알렉)가 인사하며 오늘 주제가 **컴퓨터 구조와 운영 체제**라고 밝힌다. 지난 설문에서 거의 동시에 두 번째로 많이 나온 요청 주제였다고 한다. 어렵고 복잡해 보일 수 있는 내용을 "한 장을 통해서 누구나 이해할 수 있게" 설명하겠다고 예고한다.

### [00:32–01:13] 컴퓨터 구조의 구성 요소 — CPU·메모리·디스크·IO
컴퓨터 구조는 **CPU, 메모리, 디스크, 그리고 IO**로 이루어진다. IO의 I는 인풋(input), O는 아웃풋(output)이다. 인풋에는 키보드·마우스, 아웃풋에는 모니터·프린터가 속한다. CPU와 메모리 사이, 메모리와 디스크 사이에는 데이터를 주고받는 **버스**가 있는데, 화자는 이를 "데이터를 싣고 다니는 버스" 같은 개념으로 설명한다.

### [01:13–01:49] 폰노이만 구조 — 현대 컴퓨터의 근간
CPU·메모리·디스크·IO를 버스로 연결한 이 모양을 **폰노이만이 제시**했고, 이것이 현대 컴퓨터 구조의 근간이 되었다고 정리한다.
> "이것은 현대 컴퓨터 구조에서의 근간이 되었다라고 보시면 됩니다." [01:25]

[의견] 최근 컴퓨터는 이렇게 단순하지 않고 더 복잡하지만, 어찌 됐든 CPU·메모리·디스크로 구성된다는 것은 맞고, "좀 더 복잡하게 되어 있다" 정도로 이해하면 된다고 한다. 이 틀 자체가 컴퓨터 구조의 내부 구조라는 것.

### [01:49–02:19] 프로그램이란 — 목적을 가진 소프트웨어
CPU 얘기를 바로 하면 복잡해지므로, 더 쉬운 것부터 짚는다며 **프로그램** 개념을 먼저 꺼낸다. 프로그램은 "어떤 목적을 가진 소프트웨어"로, 우리가 쓰는 웹 브라우저도, 워드프로세서도 다 프로그램이다.

### [02:19–03:10] 고급 언어 — 사람이 읽기 쉬운 문법
개발을 생각해 보면 C, C++, 자바 같은 언어로 어떤 목적을 만들고, 그 결과물이 프로그램이 된다. 이 언어들은 **고급 언어**로, 사람은 이해하기 쉽고 컴퓨터는 이해하기 어려운 언어다. `if` 같은 단어처럼 이들 언어는 문법을 갖는데, 이 문법은 컴퓨터가 아니라 **사람이 이해하기 쉽게 만든 것**이라고 설명한다.

### [03:10–03:39] 컴파일 — 기계어 exe와 자바 바이트 코드
고급 언어를 실행하려면 **컴파일**을 한다. C·C++는 컴파일하면 기계어 **exe 파일**로 변환된다. 자바는 **바이트 코드**라는 형태(클래스 파일 등)로 만들어지고, **JVM(자바 가상 머신)**, 즉 자바가 실행될 수 있는 환경 위에서 실행된다.

### [03:39–04:01] 실행의 큰 그림 — 저장→메모리→CPU
프로그램이 저장돼 있다가 실행되면, **실행된 일부가 메모리에 올라가고, 메모리의 일부가 CPU에 의해 실행된다**. 이 흐름을 말하기 위해 프로그램 개념을 먼저 꺼냈다고 밝힌다.

### [04:01–05:20] 컴파일 언어 vs 인터프리터 언어
실행 종류는 두 가지다. 소스 코드가 그대로 실행되지 않고 **다른 형태로 변환돼 실행되는** 것이 **컴파일 언어**(C/C++/자바). 반면 **자바스크립트·파이썬**은 **인터프리터 언어**로, 소스 코드를 **한 줄 한 줄 해석해 실행**한다. 브라우저 예시가 이어진다: 브라우저에서 URL을 치면 그 URL에 해당하는 서버로 접속하고, 웹 서버가 주소에 맞는 **HTML·CSS·자바스크립트를 응답으로 내려주며**, 이들이 화면에 보이게 된다. 즉 코드가 다 내려오고, 그 코드를 실행하는 곳이 브라우저다. 그래서 브라우저에는 **자바스크립트를 해석·실행하는 인터프리터 엔진**이 포함돼 있다.

### [05:20–06:10] 파이썬 실행 · 프로그램은 파일 · 프로세스의 등장
파이썬은 파이썬 모듈이 실행하며, 이런 식으로 코드 자체가 내려와 그대로 실행되는 형태가 인터프리터 언어다. C/C++/자바는 컴파일을 통해 형태가 바뀌어 실행된다. 이렇게 만들어진 프로그램은 **파일 형태로 저장**되고, 마우스로 더블 클릭하면 실행된다. 실제로 그 **실행된 상태를 프로세스**라고 한다.

### [06:10–07:01] 프로세스의 정의 — 실행되고 있는 프로그램
**프로세스 = 실행되고 있는 프로그램**이라고 정의하면 딱 맞는다고 한다. 컴퓨터 구조상으로 다시 표현하면: 설치된 프로그램을 더블 클릭해 실행하면 **프로그램 중 일부가 메인 메모리로 올라오고**, 메모리의 일부가 **CPU로 올라가 명령어들이 실행**된다.

### [07:01–07:34] 프로그램의 목적 · 소스 코드 = 명령어
프로그램은 각자 목적을 가진다. 워드는 타이핑을 칠 수 있게, 웹 브라우저는 URL을 입력해 응답을 받고 화면을 보여 주는 목적으로 만들어졌다. 그 목적을 소스에서 만들었고, **소스 코드의 것들은 다 명령어로 정리**된다. 즉 소스 코드는 명령어들을 만들어 놓은 것으로 봐도 된다.

### [07:34–08:12] 왜 프로그램을 다 안 올리나 — 메모리 계층 구조 도입
"일부가 올라간다"고 했는데 왜 프로그램을 다 메모리에 올리지 못하냐는 질문을 던지고, 여기서 **메모리 계층 구조** 개념을 꺼낸다. 디스크도 메모리도 사실 **저장하는 역할**을 한다는 점에서 출발한다.

### [08:12–09:42] 메모리 vs 디스크 — 비싸고 빠른 메모리, 싸고 느린 디스크
둘 다 저장을 하는데 차이는, **메모리는 굉장히 비싸고 고성능·빠른 처리를 하는 집적회로**이고 **디스크는 상대적으로 싸다**는 점이다. 용량당 단가는 메모리가 비싸지만, 그만큼 읽고 쓰는 속도가 엄청나게 빠르다. 많이 저장하면서도 빠르게 처리하고 싶은 요구를 어떻게 풀 것인가 하는 연구 끝에, **CPU와 디스크 사이에 메모리를 둠으로써** 필요한 데이터를 올려놓고 빠르게 제공하는 역할을 하게 됐다. 비유로, **디스크는 여러 개의 책장**이고, 자주 보는 책은 **내 책상 책꽂이(=메모리)**에 넣어 바로 뺄 수 있다. 그래서 곧 실행될 것들이 메모리에 올라오는데, 괜히 올리는 게 아니라 그 올라간 부분 중 일부가 **CPU에 의해 실행될 것들이기 때문에** 올린다.

### [09:42–10:24] 동시에 도는 수많은 프로그램
지금까지는 프로그램 하나 기준이지만, 실제로는 **엄청나게 많은 프로그램이 동시에** 실행된다. 백그라운드로도 돌고, 눈에 보이는 것만 해도 브라우저·워드·유튜브·AI를 동시에 띄운다. 그럼에도 유튜브에 끊김을 느끼지 않고 워드를 칠 수 있다. 이 상황은 여러 프로그램의 일부가 각각 메모리에 올라가 있는 것이다.

### [10:24–11:27] CPU의 초고속 처리 → 겉보기 동시 실행 = 시분할
메모리에 올라간 각 프로그램의 명령어들이 하나씩 올라가 실행되고, 다음 명령어, 그 다음 명령어가 **엄청나게 빠른 속도로** 처리된다. CPU는 "엄청난 속도의 명령어 처리"로 봐도 된다. 이렇게 빠르게 처리하다 보니 마치 동시에 실행되는 것처럼 보인다. 한 프로그램, 다음 프로그램을 번갈아 반복 실행하는 이 모습을 **시분할**이라고 한다.

### [11:27–11:53] 시분할의 의미 — 시간을 쪼갠다
시분할의 의미는 T가 타임(time), 즉 **시간을 쪼갠다**는 것이다. 일할 수 있는 하나의 CPU에게 시간을 쪼개서 1·2·3·4 작업을 번갈아 계속 시킨다. 1초도 안 되는 시간에 다른 일로 바꾼다.

### [11:53–12:40] 메모리 한계 → 메모리 교체 기법
그러는 동안 **메모리의 한계**가 있다. 메모리는 충분히 크지 못해서, 많은 프로그램은커녕 한 프로그램도 제대로 다 올리지 못한다. 요즘 몇 GB짜리 프로그램은 한 번에 못 올리고 일부만 올라온다. 그런데 올라온 명령어들이 실행되다가, **올라오지 않은 다른 부분이 실행에 필요해지는 경우**가 생긴다. 그러면 메모리에 없는 그 프로그램의 다른 영역을 찾아 다시 올려야 하는 상황이 온다. 이렇게 메모리의 어떤 영역이 교체되므로 **메모리 교체 기법**이 등장한다.
