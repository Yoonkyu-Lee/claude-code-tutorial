## 요약 (주제별)

### 인터넷 · 웹 · 네트워크의 구분
- 인터넷 = 수많은 컴퓨터가 촘촘히 연결된 그 연결 자체. 하나의 선이 아니라 그물망 같은 다양한 망들의 연결.
- 웹 = 인터넷 위에서 동작하는 애플리케이션. 웹 브라우저로 네이버·구글 같은 도메인에 접속하는 것.
- 네트워크 = 컴퓨터 간 연결을 가능하게 하는 기술 자체(어떻게 연결할지). 브라우저가 도메인을 넣고 서버 접속→서버 응답, 그 사이를 잇는 통신.

### 핵심 프로토콜 (IP / TCP / HTTP)
- 프로토콜 = 서로 간의 약속·규약·규칙.
- IP(Internet Protocol): 웹 서버에도 IP가 설정돼 있고, 데이터가 그 IP를 근거로 해당 서버를 찾아가는 역할의 약속. 라우팅의 근거가 IP.
- TCP: 한 덩어리 데이터를 작게 나눠 보내고 받는 구조. 인터넷 망은 다양한 경로로 가므로, 데이터를 정상 수신했는지·오류 있었는지 체크해 재전송하는 역할.
- HTTP(HyperText Transfer Protocol): HTML을 주고받는 약속. 클라이언트-서버가 웹인 경우 HTTP로 요청·응답. (P = 프로토콜)

### 네트워크 핵심 장비
- 라우터: WAN(Wide Area Network, 국가·지역 간 초고속 광케이블 망)과 LAN(사무실·집 인터넷 구간)을 연결. 이기종 간 네트워크 연결에 씀.
- 스위치: LAN 내부의 기기/서버들을 서로 연결하는 장비.
- 방화벽(Firewall, 파이어월): 특정 IP·프로토콜·포트만 들어오게 허용하거나 막는 보안 장비. 보통 필요한 포트만 열고(예: 443 HTTPS는 열고, 21 FTP는 안 엶) 나머지는 차단.
- ISP: Internet Service Provider(인터넷 제공 업자)의 라우터.

### CDN (Content Delivery Network)
- 콘텐츠(주로 이미지·영상)를 사용자 가까이에 두고 더 빠르게 제공하기 위한 네트워크/서버.
- 예: 미국 서비스에 한국 사용자가 접속할 때, 한국에 서버를 두고 한국 사용자는 그 가까운 서버로 접속.
- 캐시 히트: 요청한 이미지가 캐시(가까운 서버)에 있으면 바로 클라이언트로 내려줌. 없으면 원래 서버(오리진 서버)에서 인터넷을 통해 가져와 갖다 놓고 빠르게 제공.

### DMZ와 포트(Port)
- DMZ(Demilitarized Zone): 외부 접속을 허용하는 서버(대표적으로 웹 서버)와, 외부 접속을 막는 서버(데이터베이스 등, 데이터 보호 목적)를 나눠 두는 구역. 접속 가능한 영역을 한 구역으로 분리.
- 포트: 서버 컴퓨터 안에서 구동 중인 여러 서버 프로그램(웹·메일·DB 등)을 구분하기 위한 식별 번호. 물리 장비(컴퓨터)가 서버이고, 웹/메일/DB 서버는 그 안에서 실행되는 서버 프로그램.
- 주요 포트: 80(웹/HTTP), 443(HTTPS), 3306(DB, MySQL), 22(SSH), 8080(WAS 디폴트), 21(FTP). 예: 80포트로 접속되면 웹 서버가 떠 있는지 확인 가능.

### 인프라 구성도 흐름 (요청 → 응답 경로)
- 클라이언트(웹 브라우저) → 도메인 입력 → 인터넷(그물망 라우팅, 근거는 IP) → ISP 라우터 → LAN 진입 → 방화벽(허용된 포트/IP만 통과) → 로드 밸런서/스위치 → 리버스 프록시(=웹 서버) → WAS → DB, 그리고 응답이 역방향으로 돌아오는 과정.
- 중간에 CDN이 이미지·영상 요청을 가까이서 처리(캐시 히트/미스).

### 웹 서버 vs WAS, 리버스 프록시
- 웹 서버: 이미지·HTML 등 정적 웹을 제공하는 서버.
- WAS(Web Application Server): 웹 서버에 애플리케이션이 탑재된 것. 우리가 만드는 백엔드 API 서버가 여기 들어감.
- 리버스 프록시: 웹 서버가 트래픽을 WAS까지 전달만 해주는 역할. 이미지 등은 웹 서버가 직접 제공하고, 애플리케이션 요청은 WAS로 연결.
- Nginx: 웹 서버 + 리버스 프록시 기능을 둘 다 제공하는 웹 서버.
- 포트 예시: 웹 서버가 443(HTTPS)로 받으면, WAS는 디폴트 8080 포트로 트래픽을 받음.

### 이중화(Redundancy)와 DB 구성
- 이중화: 동일한 서버 기능을 양쪽에 두 대 두는 것. 목적은 (1) 로드 분산, (2) 장애 시 한쪽이라도 구동돼 서비스 무중단.
- 웹/WAS는 양쪽을 다 바라보게(웹 서버 1·2가 각각 WAS 두 곳을 바라봄) 구성해 한 군데가 안 돼도 데이터 전달 가능.
- DB는 보통 프라이머리(Primary) 한 곳만 바라보게 함. 이유: 데이터는 쓰기·읽기가 빈번해 두 군데에 쓰면 동기화 문제가 생길 수 있어서.
- 스탠바이(Standby) DB: 리플리케이션으로 백업 데이터만 동기화해두다가, 프라이머리 DB가 중단되면 스탠바이가 프라이머리로 승격되고 WAS들이 그쪽을 쓰게 됨. [의견] 상황·솔루션에 따라 두 군데에 쓸 수도 있음.
- 내부망(Internal Network): WAS와 DB가 있는 구간. 철저히 보안으로 외부 접속을 거의 차단하고 필요한 IP만 열어 내부 관리자가 관리. 내부에 추가 방화벽을 두기도 함.

### [00:35–01:11] 인터넷이란 무엇인가
처음부터 어렵게 가면 힘드니 쉬운 것부터 시작한다며, "인터넷이 무엇인가요?"라는 질문을 던진다. 인터넷은 수많은 컴퓨터가 아주 촘촘히 연결되어 있는 그 연결 자체라고 정의한다.
> "아주 촘촘히 연결되어 있는 모든 컴퓨터의 연결 자체가 인터넷이에요." [00:53]

### [01:11–02:17] 인터넷 vs 웹, 그리고 네트워크
우리는 인터넷과 웹을 같은 것처럼 여기기 쉽지만, 인터넷은 컴퓨터들이 연결된 그 자체이고 웹은 애플리케이션이라고 구분한다. 웹은 웹 브라우저로 네이버·구글 같은 도메인에 접속하는, 인터넷 기반으로 동작하는 애플리케이션이다. 이렇게 인터넷 위에서 동작하는 웹 같은 것들이 서로 연결되는 것이 네트워크이며, 네트워크는 "어떻게 연결할지" 그 연결 자체를 말한다.

### [01:36–02:17] 네트워크 = 컴퓨터 간 연결 기술
브라우저가 도메인을 넣고 서버에 접속해 응답을 받는 과정을 추상적으로 그리면, 그 사이에는 네트워크라는 통신이 있어 서로 데이터를 주고받는다. 결국 컴퓨터 간 연결을 가능하게 하는 기술 자체가 네트워크다.

### [02:17–03:12] 대표 프로토콜과 클라이언트-서버
대표 기술로 IP, TCP/IP라 불리는 프로토콜, 그리고 HTTP가 있다고 소개한다. 클라이언트와 서버가 요청·응답을 주고받을 때, 웹인 경우 HTTP 프로토콜로 요청하고 응답받으며 이때 클라이언트가 브라우저가 된다. 프로토콜의 'P'는 Protocol이고, 프로토콜은 서로 간의 약속·규약·규칙이다. HTTP는 HyperText Transfer Protocol로 HTML을 주고받는 규약이다.

### [03:12–03:35] IP (Internet Protocol)
IP를 먼저 설명한다. 웹 서버에도 IP가 설정돼 있고, 이 IP를 통해 데이터들이 해당 서버를 찾아가는 역할의 약속이다.

### [03:35–04:43] TCP
TCP는 클라이언트가 웹 서버로 데이터를 요청하고 응답받을 때, 데이터를 한 번에 전달하는 게 아니라 작게 나눠 보내고 작게 나눠 받는 구조를 담당한다. 인터넷 망은 다양한 경로로 갈 수 있는데, 이때 서버나 클라이언트에서 데이터를 정상적으로 받았는지, 오류가 있었는지 체크해 다시 보내주는 역할을 하는 프로토콜이다.

### [04:43–06:08] 라우터와 스위치
클라이언트-서버 사이 네트워크 구간에는 다양한 장비가 있는데, 연결을 돕는 핵심 장비들을 소개한다. 라우터는 WAN(Wide Area Network, 국가·지역 간 초고속 인터넷 광케이블로 데이터를 주고받게 구성된 망)과 우리가 쓰는 LAN(사무실·집에서 인터넷 쓸 때의 구간)을 연결하는 장비다. 라우터 안쪽 내부 컴퓨터들을 서로 연결해주는 것은 스위치다. 즉 LAN 내부 기기 연결엔 스위치, LAN과 WAN 같은 이기종 간 연결엔 라우터를 쓴다.

### [06:08–06:26] 방화벽 (Firewall)
방화벽은 말 그대로 파이어월이다. 특정 LAN 구간 안에 특정 IP나 프로토콜만 들어올 수 있게 하는 등의 설정을 할 수 있는 보안 장비다.

### [06:26–07:44] CDN (Content Delivery Network)
CDN은 Content Delivery Network로, 콘텐츠를 더 빠르게 제공하는 기술/네트워크/서버다. 서버가 특정 위치에 있어 접속이 매우 멀 수 있는데(예: 미국 서비스인데 한국 사람이 접속해야 하는 상황), 한국에 서버를 두고 한국 사용자는 그 서버로 접속하게 한다. 프로그램을 구동하는 서버라기보다 대부분 이미지·영상을 더 빠르게 제공하기 위한 서버다. 이런 망을 구성해 서버에 이미지·영상을 올려두고, 만약 거기 없으면 오리진 서버(원래 서버)에서 다시 가져와 갖다 놓고 빠르게 제공한다.

### [07:44–08:21] DMZ
DMZ는 Demilitarized Zone(비무장지대)이다. 서버 중 어떤 것은 외부 접속을 허용하고(대표적으로 웹 서버 — 외부에 접속돼야 하므로), 데이터베이스 같은 서버는 데이터 보호를 위해 외부 접속을 막는다. 이렇게 외부 접속이 가능한 영역을 한 구역으로 나눈 것을 DMZ라 한다.

### [08:21–10:14] 포트 (Port)
포트는 매우 중요하며 컴퓨터(특히 서버) 안에 있는 개념이다. 서버 컴퓨터의 특정 포트로 접속한다는 것은, 그 컴퓨터 안에서 구동 중인 여러 서버 프로그램을 구분하기 위한 식별 번호를 지정하는 것이다. 여기서 '서버'라 부르는 물리 장비(컴퓨터) 자체가 서버이고, 웹 서버·메일 서버·DB 서버는 그 컴퓨터 안에서 실행되는 서버 프로그램을 뜻한다. 데이터를 전달할 때 이 서버 프로그램들을 구분하는 식별 번호가 포트다. 예: 80으로 접속하면 웹 서버, 3306으로 접속하면 DB 서버, 이메일/SSH는 22번, HTTPS는 443번을 쓴다. 특정 웹 서버가 구동 중인지 보려면 80포트로 접속해보는 식으로 표현할 수 있다.

### [10:14–11:09] 구성도 도입 · 클라이언트-서버 접속
이제 구성도를 보며 설명한다. 요소를 하나씩 먼저 설명한 이유는, 구성도를 처음부터 보면 어렵게 느껴지기 때문이다. 앞에서 다 이해 못 했더라도 구성도를 보며 다시 설명하겠다고 한다. "클라이언트-서버가 접속한다"는 것은 사용자가 브라우저를 켜고 도메인을 치는 행위이며, 그 요청이 WAS 서버까지 갔다가 응답이 돌아오는 과정이다.

### [11:09–13:13] 요청 경로 · 인터넷 라우팅
중간에 여러 단계와 장비를 거쳐 간다. 인터넷 상황이 있고, 라우터는 WAN과 LAN 사이를 연결한다. ISP는 Internet Service Provider(인터넷 제공 업자)의 라우터이며, 그 안쪽은 LAN(Local Area Network) 구간으로 랜선을 꽂거나 와이파이를 쓰는 구간이다. CDN은 클라이언트가 이미지·영상을 받을 때 가까운 서버에 두고 바로 응답해주는 형태이고, 원하는 이미지가 캐시 안에 있으면 바로 내려주는 것을 "캐시 히트"라 한다. 없으면 인터넷을 통해 실제 서버로 가서 가져온다. 인터넷은 하나의 선이 아니라 그물망 같은 다양한 망들의 연결이며, 그곳을 지나 목표 서버로 향해 가는 과정을 라우팅이라 한다. 라우팅의 근거가 되는 것은 IP이고, IP로 해당 서버를 찾아간다.

### [13:13–14:28] 방화벽 진입 · 포트 허용/차단
네트워크로 진입하면 첫 번째로 만나는 것이 방화벽(파이어월)이다. 허용된 IP나 포트만 열리고, 접속할 수 없는 것은 이 단계에서 막혀 접속 자체가 안 된다. 방화벽은 보통 허용 방식으로, 해당 포트만 접속되게 하고 다른 포트는 막는다. 예를 들어 21번 FTP(파일을 주고받는 프로토콜)는 열어두지 않고, HTTPS가 쓰는 443 포트는 열어둔다. 이런 식으로 어떤 IP 대역을 열고 막을지, 어떤 포트를 열지 설정한다.

### [14:28–15:03] 로드 밸런서/스위치 · 리버스 프록시 · 이중화 도입
허용된 접속이 들어오면 로드 밸런서(여기서는 스위치라고도 표현 가능)를 거친다. 스위치는 내부 서버들을 연결하는 장비다. 이어서 리버스 프록시(어렵게 표현됐지만 웹 서버로 봐도 됨)로 연결된다. 웹 서버 뒤의 WAS 인프라를 구성할 때 이중화를 한다.

### [15:03–15:48] 이중화의 목적 · DMZ · 내부 방화벽
이중화는 동일한 서버 기능을 양쪽에 두 대 두는 것이다. 가장 큰 목적은 로드 분산과, 장애 시 한쪽이라도 구동돼 서비스에 문제가 없도록 하는 것이다. 여기까지를 DMZ 구간으로 설정해뒀다. 내부에도 방화벽을 두는 경우가 있는데, 내부 네트워크 구성은 다르지만 전체 구성 이해 차원에서 언급한다.

### [15:48–17:08] 웹 서버 vs WAS · 리버스 프록시 · Nginx
웹 서버와 WAS의 차이를 설명한다. 웹 서버는 이미지나 HTML 등 웹을 제공하는 서버다. WAS는 Web Application Server로, 웹 서버에 애플리케이션이 탑재된 것이며 우리가 만드는 백엔드 API 서버가 여기 들어간다. 그래서 웹과 WAS가 연결돼, 이미지 등은 웹 서버가 제공하고 WAS로 연결돼야 할 트래픽은 WAS까지 전달·연결해주는 역할을 한다. 그것이 리버스 프록시(전달만 해주는 서버 역할)다. 이 둘을 다 제공하는 것이 Nginx라는 웹 서버다.

### [17:08–18:28] WAS 연결 구성 · 포트 · 내부망
웹 서버 1이 WAS 두 군데를, 웹 서버 2도 WAS 두 군데를 바라보게 구성하면, 한 군데가 안 돼도 두 군데로 데이터를 전달할 수 있다. 이는 아키텍처 설명이 함께 들어가는 부분으로, 네트워크만 이해하기보다 실질적 구성으로 함께 이해하는 게 낫다고 덧붙인다. 웹 서버가 443 포트로 HTTPS를 받았다면, WAS는 디폴트 포트가 8080이다. 백엔드 애플리케이션들이 자신의 데이터로 패킷을 받을 수 있게 포트와 함께 떠 있다. WAS와 DB가 있는 곳을 내부망(Internal Network)이라 하며, 철저히 보안으로 외부 접속을 거의 차단하고 필요한 IP만 열어 내부 관리자 수준으로 사용한다.

### [18:28–20:40] WAS-DB 연동 · 프라이머리/스탠바이 · 리플리케이션
WAS는 데이터베이스와 연동되는 경우가 많다. DB에서 데이터를 가져오고 전달하거나, 데이터 요청을 받아 처리하는 것이 주로 API 서버가 하는 일이며, 이 API 서버가 WAS에 탑재돼 있어 DB에 접속해 데이터를 가져오거나 생성한다. DB는 프라이머리·스탠바이로 표현되는데, WAS 두 개 모두 프라이머리 DB만 바라보도록 구성된다. 웹은 양쪽을 다 바라보게 했는데 DB는 왜 한 군데냐면, 데이터는 쓰기·읽기가 빈번해 두 군데에 쓰면 동기화 문제가 생길 수 있기 때문이다. [의견] 물론 솔루션이나 상황에 따라 두 군데에 쓸 수도 있다. 보통은 프라이머리 DB로 두고 양쪽 WAS가 여기에 쓰며, 스탠바이 쪽은 리플리케이션으로 백업 데이터만 동기화해두다가, 프라이머리 DB가 중단되면 스탠바이가 프라이머리로 승격되고 WAS들이 그쪽 DB를 쓰게 된다. 즉 DB 양쪽에 동시에 쓰는 게 아니라 한쪽에만 쓰고, 다른 쪽은 동기화해두다 실패 시 승격되는 구조다.

### [20:40–22:27] 전체 요소 복습
주제가 네트워크임을 다시 상기시키며 흐름 관점에서 이해하라고 한다. 주요 요소를 정리한다: CDN은 가까이에 콘텐츠(이미지·영상)를 두고 클라이언트에 빠르게 제공하는 것이고, 없으면 접속해 가져온다. 라우터는 이기종 간 네트워크 연결, 방화벽은 막고 열어 보안하는 역할, 로드 밸런서/스위치는 뒤의 서버 여러 대를 연결할 때 쓰는 장비다. 웹·WAS·DB가 있고, 웹과 WAS 사이 연결에서 중요한 것이 포트(서버가 떠 있는 것을 식별하는 구분 번호)다. WAS는 8080, DB는 3306, 웹은 80을 쓰되 HTTPS로 맞이하는 것은 443이다. 이러한 장비들과 통신의 흐름으로 이해하면 좋다고 정리한다.
