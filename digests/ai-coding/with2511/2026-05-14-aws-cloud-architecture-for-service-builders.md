## 요약 (주제별)

### 클라우드가 필요한 이유 (배경)
- 앱/웹 서비스는 클라이언트만으로 동작하는 경우가 거의 없고, 대부분 인터넷망 → 서버 → 서버의 데이터베이스 구조다. 데이터가 DB에 있고 클라이언트가 그걸 다시 보는 흐름.
- 요즘 스타트업·기업 대부분은 서버를 클라우드에 둔다. 클라우드는 "구름처럼 어딘가에 실제로 존재하는" 서버를 서비스처럼 빌려 쓸 수 있게 만든 것.
- [의견] 바이브 코딩으로 서비스를 만들려는 사람이라면 클라우드 개념과 대략적 구성도 흐름을 알고 실제 AWS·Azure를 쓰면 좋다. 전체를 다 알 필요는 없고 대표 기능 몇 가지만 이해해도 사용 가능.

### 핵심 기술: 가상화(Virtualization)
- 클라우드 서비스를 가능하게 하는 가장 핵심 기술이 가상화. 물리적 장비가 뭔지는 알 수 없지만 그 자원 일부를 쓸 수 있게 제공하는 기술.
- 대표적으로 컴퓨팅 가상화 — 실제 컴퓨터 한 대를 통째로 주는 게 아니라, 자원 일부를 마치 한 대인 것처럼 제공한다. AWS에서는 이 가상 컴퓨터가 EC2.
- 네트워크, 스토리지(디스크), 보안, 데이터베이스, 모니터링 모두 가상으로 서비스처럼 제공된다.

### AWS 주요 서비스 이름·역할 (요소 정리)
- **EC2 (Elastic Compute Cloud)**: 가상 서버. Windows/Linux OS까지 설치해 접속·사용 가능. C가 두 개라 "2". AWS 서비스 이름 앞에 "Elastic(유연하게 늘고 줆)"이 붙어 2가 많이 붙는다.
- **오토 스케일링(Auto Scaling)**: 트래픽이 많을 때 서버가 자동으로 늘고, 적을 때 줄어드는 기능. 미리 설정 가능.
- **Route 53**: AWS의 DNS 서비스. 도메인을 클라우드에 띄운 서버와 연결.
- **CloudFront**: AWS의 CDN(Content Delivery Network). 이미지·동영상을 전 세계 각지 서버에 두고 사용자와 가까운 곳에서 더 빠르게 전달.
- **VPC (Virtual Private Cloud)**: 가상 사설 네트워크(내부망). 안에서는 서버 간 접속이 되고 외부에서는 제한적으로만 접속.
- **ELB (Elastic Load Balancer)**: 뒤에 연결된 여러 서버로 로드(트래픽/요청)를 유사한 수준으로 분산하는 스위치. 단 뒤 서버들은 동일 서버여야 함. 바깥쪽은 External ELB, 안쪽은 Internal ELB.
- **S3 (Simple Storage Service)**: S가 세 개라 "S3". 파일 저장소. 이미지·음성·영상 파일 저장. 저장 단위는 버킷(폴더 개념).
- **EBS (Elastic Block Store)**: 컴퓨터에 붙는 디스크 같은 개념의 저장 서비스.
- **IAM**: 역할·그룹·사용자 권한 관리 도구.
- **시큐리티 그룹(Security Group)**: 가상 방화벽. 허용된 IP 대역/포트만 들어오게 함. 실제 방화벽이 내부망 초입에서 관리한다면, 시큐리티 그룹은 인스턴스(서버)별로 관리 가능.
- **WAF (Web Application Firewall)**: 애플리케이션 레벨의 방화벽.
- **RDS**: 데이터베이스를 서비스로 제공. R=Relational(정형 DB). MySQL·Oracle 등을 쓸 수 있게 제공. 테이블 형태로 관계를 맺어 저장하는 게 RDB.

### 배포(서비스 오픈) 관점의 구성도 흐름
- 서비스 오픈 = 도메인 하나 만들어 클라우드에 띄운 서버와 연결하는 과정. 도메인 접속 시 Route 53을 통해 연결.
- 흐름: 도메인 → Route 53 → WAF(앱 레벨 방화벽) → CloudFront(CDN). CloudFront에 콘텐츠가 없으면 원본이 저장된 S3에서 가져와 캐시해 두고 제공, 없으면 다시 원본에서 가져오는 방식.
- VPC 내부망 안에 서버들을 구성하는 경우가 많다.
- 메인 영역에 EC2 가상 서버가 여러 개 떠 있고, 웹(Web) 서버와 앱(WAS) 서버로 구분. 백엔드 애플리케이션은 앱(WAS) 쪽에 올라간다.
- External ELB가 웹 서버들로 트래픽을 분산 → 다시 모여 Internal ELB가 WAS 쪽으로 전달. 웹·WAS 모두 오토 스케일링 그룹으로 서버 수를 늘렸다 줄였다 가능.
- 데이터 저장은 DB(RDS), Primary/Secondary 구성으로 저장.

### AWS 콘솔 실제 화면 데모
- google에 "AWS" 검색 → 계정 생성. 처음 쓰면 프리티어(Free Tier)로 200달러 크레딧 제공 (주장/미검증).
- **가장 주의할 점**: 200달러를 준다지만 유료 과금이 발생하는 경우가 있다. 인스턴스 생성 시 "프리티어 사용 가능"이라고 찍혀 있어야 프리티어이고, 그 표시가 없으면 그냥 과금된다.
- EC2: "인스턴스 시작" → 이름·OS·사양 선택 → 생성하면 서버가 뜨고 "연결"을 누르면 실제 서버(Linux)에 바로 접속. 여기에 백엔드 애플리케이션이 올라감.
- RDS: "RDS" 검색 → 생성 → 빠른 구성 또는 전체 구성으로 한 단계씩 설정하면 DB 생성. MySQL·Oracle 등 선택 가능.
- S3: 버킷(폴더 개념) 생성 → 이미지·음성·영상 파일 저장.

### 화자의 안내·세일즈 구조
- [의견] 실제 실습을 이 영상에서 일일이 다루긴 한계가 있으니, 제대로 서비스를 만들려면 공부·학습이 좀 필요하다며 본인의 "AWS 클라우드 2열 실습" 강의를 소개. 가상화·Docker·Kubernetes·MSA, AWS 주요 서비스, 인스턴스 생성·로드 밸런스 연결·오토 스케일링·Lambda(서버리스)까지 실습 포함 (강의 내용은 화자 주장).
- 마무리: 서비스를 한다면 메인은 클라우드이며, 주요한 것은 서버·스토리지·데이터베이스. AWS에서 이들을 주요하게 이용할 수 있음을 이해하고 실제 해 보면 이해가 잘 된다.

그 서버는 어디에 있느냐고 물으며, 요즘 많은 스타트업과 기업은 클라우드를 이용한다고 말한다. 클라우드(구름)는 상당한 상징성이 있어서, 정말 구름처럼 어딘가에 실제로 존재하는 것이라 상상해도 된다고 한다. 일종의 서비스로, 서버들을 서비스처럼 쓸 수 있게 만든 것이라고 정의한다.

### [02:41–03:02] 핵심 기술 — 가상화
클라우드 서비스가 가능한 가장 핵심적인 기술이 가상화라고 강조한다. 가상화란 어떤 물리적 장비인지는 알 수 없지만 그 자원의 일부를 쓸 수 있게 제공해 주는 기능·기술이다.

### [03:02–04:24] 대표 요소 기능들 — 컴퓨팅·네트워크·스토리지·보안·DB·모니터링
가상화의 대표는 컴퓨팅 기능이다. 컴퓨터를 쓸 수 있게 제공하되 실제 한 대를 통째로 주는 게 아니라 자원 일부를 한 대인 것처럼 제공한다. 이 가상 컴퓨터가 AWS에서는 EC2다. AWS는 서비스 이름 앞에 "2"가 많이 붙는데, Elastic이라 EC의 C가 겹쳐(ECC → EC2) 그렇게 부른다고 설명한다. 이 외에 네트워크, 스토리지(디스크)·저장 공간, 보안, 데이터베이스, 그리고 서버·네트워크·DB가 정상인지 보는 모니터링 기능이 대표적이다.
> "전체를 다 안다기보다는 몇 가지를 어 알고 어 사용하면 된다 이렇게 생각하셔도 됩니다." [04:15]

### [04:24–05:00] EC2 상세 + 오토 스케일링
컴퓨팅의 대표 EC2는 Windows나 Linux 설치까지 하고 접속해 쓸 수 있는 컴퓨터 한 대를 제공받는 개념이다. 오토 스케일링은 클라우드에서 종종 듣게 되는 표현으로, 자동으로 서버가 많이 필요할 때 늘어나고 적게 필요할 때 줄어드는 서비스라고 기억하고 넘어가라고 한다.

### [05:01–05:38] Route 53(DNS) · CloudFront(CDN)
Route 53은 AWS 상에서의 DNS 서비스를 부르는 이름이다. CloudFront는 CDN(Content Delivery Network)으로, 이미지나 동영상을 사용자와 가까운 곳에 서버를 두고 더 가깝게 제공하는 서비스이며, AWS가 자체적으로 제공한다. 이미지·동영상을 더 빠르게 전달하는 역할이다.

### [05:39–06:11] VPC — 가상 사설 네트워크
VPC는 Virtual Private Cloud, 가상 사설 네트워크다. 내부망 같은 것으로, 안에서 주로 서버 간 접속이 되고 외부에서는 제한적으로 접속할 수 있게 만든 네트워크다. 이것 역시 가상으로 제공된다.

### [06:11–07:16] ELB — 로드 밸런서
ELB(Elastic Load Balancer)도 클라우드에서 자주 언급된다. Elastic은 유연하게 늘고 주는 개념이고, Load Balancer는 뒤의 데이터를 전달해 주는 스위치로 보면 된다. ELB 뒤에는 서버(대표적으로 EC2)가 여러 개 연결될 수 있고, 로드(부하)를 분산시킨다. 단 이렇게 하려면 EC2 안에서 구동되는 서버들은 동일한 서버여야 한다. 하나·두 개·세 개의 서버가 돌아갈 때 그 앞단에 ELB를 두고 전달한다.

### [07:16–08:03] S3 · EBS — 저장 서비스
S3는 Simple Storage Service로 S가 세 개라 S3다(EC2는 Elastic Compute Cloud로 C가 두 개). 파일들을 저장할 수 있는 저장소다. EBS는 Elastic Block Store로, 컴퓨터에 붙는 디스크 같은 개념을 클라우드에서 저장 서비스로 제공하는 것이다.

### [08:03–09:05] IAM · 시큐리티 그룹 vs 방화벽
IAM은 역할·그룹·사용자 권한 등을 관리하는 도구다. 시큐리티 그룹은 이전 영상에서 다룬 방화벽과 같다고 보면 된다. 방화벽은 서버 내부망에 들어가기 전에 막아 두고 가능한 포트·대역폭만 들어가게 하는 개념인데, 시큐리티 그룹이 그런 역할의 가상 방화벽이다. 차이가 있다면 방화벽은 내부망 초입에서 관리하고, 시큐리티 그룹은 인스턴스(서버)별로 관리할 수 있다는 점. 역할은 동일하게 허용된 IP 대역 또는 포트만 들어오게 한다.

### [09:05–11:24] 배포 관점 구성도 흐름 — Route 53 → WAF → CloudFront → S3
앞서 요소 기술을 설명해 이제 화면에 보이는 게 좀 있을 거라며(Route 53, CloudFront, S3, EC2, ELB), 무언가 서비스를 하나 배포(서버에 올림)할 때 무엇을 알아야 하는지 관점으로 본다. 서비스를 오픈한다는 것은 도메인 하나를 만들어 클라우드에 띄운 서버와 연결하는 과정이며, 도메인 접속 시 Route 53을 통해 이뤄진다. 도메인을 타고 WAF(Web Application Firewall, 애플리케이션 레벨 방화벽)로 연결되고, 이어 CloudFront(CDN)로 간다. CloudFront는 이미지·동영상을 저장하고 전 세계 각지에 서버를 두어 가까운 서버에서 빠르게 다운로드하게 한다. 화살표가 S3로도 향하는 이유는, CloudFront(CDN)에 콘텐츠가 없으면 원본이 저장된 S3에서 가져와 캐시해 두고 제공하며, 없으면 다시 원본에서 가져오는 방식이기 때문이다.

### [11:24–11:49] VPC 내부망 구분
VPC(Virtual Private Cloud)로 선을 그어 내부망처럼 구분해 둔 영역을 보여준다. 내부 네트워크로 연결되도록 묶어, 한 서비스를 구성할 때 VPC 안에서 서버들을 구성하는 경우가 많다.

### [11:49–13:01] EC2 웹/앱 서버 · DB Primary/Secondary
메인 영역의 EC2는 쉽게 말해 서버, 곧 가상 서버다. 그림에 EC2가 여러 개로 표현된 것은 가상 서버가 여러 개 떠 있다는 뜻이다. 한쪽 EC2 묶음은 웹(Web), 다른 쪽은 앱(WAS)으로 구분했다. 즉 웹이 기동되는 서버와 WAS가 기동되는 서버로 나뉘며, 백엔드 애플리케이션을 만든다면 앱(WAS) 쪽에 올라간다. DB는 Primary/Secondary로 나뉘어 저장된다.

### [13:01–14:21] External/Internal ELB · 오토 스케일링 그룹
더 세세히 보면, ELB(Elastic Load Balancer)가 여기서 분기를 한다. 로드(트래픽·요청)를 거의 유사한 수준으로 양쪽에 분산해 보내고, 한 번 모였다가 WAS 쪽으로 다시 전달한다. 바깥쪽에 있는 것을 External ELB, 안쪽에 있는 것을 Internal ELB라 부른다. 웹 서버도 WAS도 얼마든지 늘릴 수 있는데, 이는 오토 스케일링 그룹과 연결된다. 오토 스케일링은 서버가 한 개였다가 트래픽이 많으면 두 개·세 개·네 개로 늘리도록 미리 설정해 둘 수 있고, 트래픽이 줄면 줄일 수 있는 개념이다.

### [14:21–15:11] RDS — 관계형 데이터베이스
데이터를 저장하는 곳은 DB이며, 데이터베이스를 서비스로 제공하는 형태가 아마존의 RDS다. R은 Relational(정형 DB)로 보면 된다. RDB(관계형 DB)는 데이터베이스 안에 테이블 형태로 관계를 맺어 저장하는 것을 뜻한다. 여기까지가 클라우드로 백엔드 서버를 올리고 데이터를 저장하는 구성도 레벨의 설명이라고 정리한다.

### [15:11–16:02] AWS 콘솔 데모 시작 · 프리티어 200달러
실제 서비스에서 어떻게 보이는지 잠깐 보여준다. google에 "AWS"를 검색해 들어가 계정을 생성하면 되고, 계정 생성은 기본적으로 쭉 진행하면 된다. 처음 AWS를 쓰면 프리티어가 있어 200달러 크레딧을 제공하는데 꽤 큰 금액이다(주장/미검증). 다만 유료로 발생하는 경우가 있으니 주의하며 사용해야 한다고 언급한다. 화자는 이미 계정이 있어 최근 이용 서비스가 보이며, "EC2"를 검색하면 화면이 뜬다.

### [16:02–17:08] EC2 인스턴스 생성 · 프리티어 과금 주의
화자는 실행 중인 인스턴스가 하나 있어 표시된다고 설명한다. 한 번도 생성한 적 없으면 목록이 비어 있으며, "인스턴스 시작"으로 시작하면 된다. EC2는 서버를 하나 띄우는 것이다. 이름을 적고 OS와 사양을 선택하는데, 여기서 유의할 점을 강조한다.
> "프리티어 사용 가능이라고 찍혀 있어야 프리티어인 거고요. 그렇지 않다면 ... 그냥 과금이 되는 겁니다. 가장 주의하셔야 될 부분입니다." [16:37]
"프리티어 사용 가능" 표시가 있는 것만 이용해야 하며, 없으면 그냥 과금된다는 것이 가장 주의할 부분이다.

### [17:08–18:04] 서버 접속 · RDS 진입
생성하고 나면 서버가 하나 뜨고, "연결"을 누르면 실제 서버에 바로 접속까지 된다. 접속되는 것은 Linux 서버로, 그냥 Linux 서버를 쓰면 되고 여기에 백엔드 애플리케이션이 올라오고 저장되며, 클라이언트에서 이 서버로 접속하는 구조다. 이렇게 EC2 서버를 띄우고 서버에 접속하는 것을 잠깐 보여준 뒤, DB와 S3도 보겠다며 "RDS"를 검색한다.

### [18:04–18:58] RDS 데이터베이스 생성
데이터베이스가 없어 목록이 비어 있는데, "생성"을 눌러 빠른 구성 또는 전체 구성으로 하나씩 고르면 된다. RDS는 우리가 아는 MySQL이나 Oracle 등을 쓸 수 있게 제공하는 서비스다. 데이터베이스가 필요하면 한 단계씩 진행하며 설정하면 DB가 하나 만들어진다. 이렇게 연결되고 도메인과 연결되면 서비스가 실제 클라우드에 올라가 구성된다. 서버가 더 필요하거나 트래픽에 따라 늘렸다 줄였다 하고 싶으면 오토 스케일 그룹을 설정해 늘릴 수 있다.

### [18:58–19:48] S3 버킷
S3를 보면 버킷이 있다. 버킷은 폴더 같은 것으로 생각하면 되며, 폴더를 만들고 거기에 저장한다. 이미지·음성·영상 파일을 넣을 수 있고, 접속할 수 있게 한다. 스토리지 서비스로, 이용할 파일들을 여기에 저장해 두고 쓰는 대표적 파일 서비스가 S3다.

### [19:48–20:39] 구성 재정리
전체를 다 말하기보다, 클라우드에 서비스를 올릴 때 대표적 기능이 Route 53, CloudFront, 가상 서버, 데이터베이스, 파일 저장소 식으로 구성된다고 이해하라고 한다. 본격적으로 하려면 AWS 사이트에 들어가 계정 가입하고 서버를 한번 띄워 보라고 권한다.

### [20:39–21:34] 강의 소개 (세일즈)
서비스이기 때문에 한번 보면 되지만 실제 실습까지 일일이 여기서 다루긴 한계가 있다고 말한다. [의견] 정말 제대로 서비스를 만들려면 공부·학습이 좀 필요하다며 본인의 "AWS 클라우드 2열 실습" 강의를 소개한다. 반응이 좋았다고 하며(주장/미검증), 전반적 기술 내용, 가상화·Docker·Kubernetes, MSA 설명, AWS 주요 서비스, 그리고 실무 실습(인스턴스 생성, 로드 밸런서 연결, 오토 스케일링, Lambda 서버리스까지)을 담았다고 안내한다.

### [21:34–22:08] 마무리 정리
정리하면, 서비스를 한다면 가장 많이 이용할 메인은 클라우드이며, 주요한 것은 서버·스토리지·데이터베이스라고 말한다. AWS에서 이들을 주요하게 이용할 수 있음을 이해하고 실제 해 보면 이해가 더 잘 될 것이라며, 좋은 내용으로 다시 찾아오겠다고 마친다.
