### 활용 사례 1 — 유튜브 추천 알고리즘
- [의견] 유튜브가 지금처럼 성장한 핵심은 오른쪽의 **추천 영상**이라는 것이 화자 견해. 검색하러 들어갔다가 추천 때문에 한두 시간씩 "시간 여행"을 하게 되는 경험이 그 증거.
- 서비스의 목표는 사용자를 **얼마나 오래 머무르게(체류) 할 것인가**이고, 추천 알고리즘이 그 수단.
- 데이터 소스 = 사용자 활동(어떤 영상에 오래 체류했나, 많이 봤나, 좋아요를 눌렀나). 이를 모으고 → 필요한 데이터로 정제 → 필터링/딥러닝 알고리즘으로 학습 → 관심도를 **랭킹(순위)**으로 매겨 보여줌.

### 활용 사례 2 — 장바구니 분석 (market basket analysis)
- 빅데이터 공부 시 가장 많이 보는 예제. 데이터 소스는 영수증/거래 목록. [의견] 가장 좋은 퀄리티의 데이터는 결제 내역(체류보다 확실한 증거).
- 대표 예: **기저귀와 맥주**의 연관관계 — 하나를 사면 다른 하나를 살 가능성. [의견] 아이가 잠들면 부모가 맥주 한잔하며 넷플릭스/유튜브를 보기 때문일 수 있다는 해석(화자 추정).
- 이 상관관계는 상품 진열 배치·동선 분석 → 어떤 코너에 무엇을 놓을지 → 세트/묶음/번들 추천 캠페인으로 확장.

### 활용 사례 3 — 소상공인(SOHO) 할인·이익 최적화
- 커피숍·음식점 등 매장의 원플러스원/할인 이벤트에서 "할인을 얼마나 하는 것이 최적인가"를 데이터로 판단.
- 핵심은 매출이 아니라 **이익**. 50% 할인처럼 무작정 싸게 팔면 많이 팔려도 마진이 적어 전체 이익이 줄 수 있음. 데이터 분석으로 고객 확보와 이익을 동시에 최적화하는 지점을 찾을 수 있다는 주장.

### 빅데이터 분석의 기본 흐름(파이프라인)
- 흐름: **데이터 소스 → 수집 → 저장 → 분석 → 비즈니스 활용**. 반드시 이 모양대로 다 구축할 필요는 없고 약식 구성도 가능(대략의 틀).
- 역할 분담: 데이터 기획자(무엇으로부터 데이터를 가져올지 기획) → 데이터 엔지니어(시스템 구축) → 데이터 분석가. 단, 기획은 꼭 전문 기획자가 아니어도 우리가 생각할 수 있는 범위.

### 데이터 소스·수집·정제
- 데이터베이스 = 정확히 저장된 데이터 / 시스템 로그 = 흘러가는 데이터(접속·페이지 체류·상품 클릭 등). 물류 데이터, 판매 데이터, 서비스 사용 패턴도 모두 소스.
- 디바이스/IoT(센서, RFID 칩 등)로부터의 수집은 좀 더 빅데이터에 가까운 영역.
- 정제(preprocessing) = 없는 값·불필요한 값을 정리하는 전처리. 도구로 **Logstash** 같은 모듈. 저장/검색 엔진으로 **Elasticsearch**를 많이 씀(속도·검색이 빠름).
- 저장 개념: 처음 모으는 곳 **데이터 레이크**, 변환해 두는 곳 **데이터 웨어하우스**. 단, 이 전부를 구축하는 건 대형 서비스·큰 회사 상황이고, 작게는 Elasticsearch만으로도 이 역할을 포함할 수 있음.
- **ETL** = Extract(추출)·Transform(변환)·Load(적재). 데이터를 변환해 저장하는 과정으로 기억.

### 분석 기법
- **상관관계 분석**: 많은 데이터를 보다 보니 두 상품(예: 기저귀·맥주)에 관계성이 있더라는 것을 사후에 규명. 커피숍 판매량과 **날씨**의 상관관계도 예 — 더운 날 아이스 아메리카노가 잘 팔리면 재료 준비에 반영.
- **군집 분석(clustering)**: 데이터를 묶음으로 나눔. ① 시스템 오류 로그에서 특정 패턴이 생기다가 **전혀 없던 새 패턴**이 나오면 이상 상황(특이점)으로 발견(화자가 실제 해 본 사례). ② 고양이/개 분류 — 특징(feature)들에 점을 찍어두고 새 이미지를 던지면 어느 쪽에 가까운지로 분류.
- 모델링 절차: 분석 목적에 맞는 **피처(feature)** 선택 → **알고리즘**(상관관계/군집 등) 선택 → 데이터로 학습 → 값을 조정하며 반복 테스트로 결과를 개선 → 최종 모델을 **API**로 제공(값을 넣으면 결과값을 반환, 예: 특징 넣으면 "고양이/개" 응답).

### 비즈니스 활용처
- **수요 예측**: 지점/나라별 판매량 예측 → 물량을 미리 확보. 엉뚱한 곳에 두면 물류비·재고 부담이 크지만, 예측을 잘하면 큰 경비 절감. 작은 가게·편의점도 요일별·시간별 예측으로 적정 재고 관리 가능.
- **개인화 추천**: [의견] 지금 서비스들이 이렇게까지 발전한 핵심. 서비스에 잘 적용하면 체류 시간 증가.
- **장애 예측**: 군집 분석으로 없던 패턴이 나오면 알람을 받아 처리 → 장애를 빠르게 캐치.

### 간단한 실전 아키텍처(구형/약식)
- 시스템 로그가 파일로 떨어짐 → **Filebeat**로 파일 수집 → **Logstash**로 정제(형태 변환) → **Elasticsearch**에 저장 → **Python**으로 분석·모델링 → **Flask** 또는 **FastAPI**(파이썬 기반)로 API 제공.
- 활용 흐름 예: 유저가 접속하면 그 유저를 API에 넣고 "이 사람에게 추천할 만한 것"을 받아 리스트로 활용.
- [의견] 시스템 한두 대로도 당장 만들 수 있는 수준. 도구 이름(Filebeat, Logstash, Elasticsearch)만 기억해 둬도 도움이 된다는 조언.

### 마무리 메시지
- 내부 모델·알고리즘·저장/수집 방식은 조금씩 바뀌어도 **전체 틀은 유사**하다는 것을 기억하면, ADSP 공부에도 서비스 접목에도 도움이 된다.
- 세일즈 장치: 별도 유료자료·단톡방 언급 없음. 표준 CTA(좋아요 누르면 관련 주제를 계속 올리겠다)만 있음.

### [01:01–02:50] 유튜브 추천의 데이터 처리 구조
추천으로 인해 사용자는 자기가 관심 있는 것에 계속 머무를 수밖에 없고, 결국 유튜브라는 서비스의 목표는 사용자를 **얼마나 많이 머무르게 할 것인가**라고 정리한다. 서비스를 만들 때 어떤 알고리즘을 적용해 최대한 많이 쓰게/머무르게 할 것인가가 데이터 분석 활용의 대표 사례다. 구조를 간단히 보면, 데이터 소스(어떤 영상에 오래 체류했나·많이 봤나·좋아요를 눌렀나 같은 사용자 활동)를 쭉 모으고, 불필요한 데이터를 **정제**해 실제 필요한 데이터로 모은 뒤, 필터링 또는 딥러닝 알고리즘으로 학습해 이 사람이 어떤 영상에 관심 있는지를 **랭킹(순위)**으로 매겨 보여준다. 그래서 사용자는 계속 관심 영상이 보여지며 머무르게 된다.

> "서비스를 만드는 사람들도 관심을 가져야 될 것이 결국은 데이터 분석이다." [02:44]

### [02:52–05:16] 장바구니 분석 — 기저귀와 맥주, 상관관계
데이터 분석/빅데이터 공부에서 가장 많이 보는 예제로 **장바구니 분석**을 든다. 데이터 소스는 영수증 또는 거래 목록이며, [의견] 가장 좋은 퀄리티의 데이터는 결제 내역이다(체류보다 확실한 증거). 대표 예로 **기저귀와 맥주**의 연관관계 — 하나를 사면 다른 하나를 살 가능성 — 를 설명한다. [의견] 화자는 그 이유를 아기를 키우는 사람이 아기가 잠들면 자유시간을 즐기며 맥주 한잔하고 넷플릭스·유튜브를 보기 때문일 수 있다고 추정한다. 이런 "무언가를 샀을 때 나머지 하나를 살 확률" 분석은 **상품 진열 배치·동선 분석**으로까지 이어져, 어떤 코너에 무엇을 놓을지(인접 배치로 소비 유도), 진열 변경, 세트·묶음·번들 추천 캠페인의 아이디어를 준다. 이것이 모두 데이터 분석에서 나온다는 점을 강조한다.

### [05:19–06:42] 소상공인(SOHO) 사례 — 할인율과 이익 최적화
커피숍뿐 아니라 음식점·여러 매장 형태에도 적용된다며, 매장이 하는 원플러스원·할인 이벤트를 예로 든다. 화자는 "할인을 얼마나 하는 것이 가장 좋을까"를 고민한 적이 있다고 한다. 무작정 50% 할인하면 싸게 팔려 매출은 오르겠지만, 가장 중요한 건 **이익**이다 — 많이 팔아도 마진이 적어 전체 이익이 적을 수 있다. 데이터 분석을 통해 이 비즈니스를 어떻게 했을 때 고객도 많이 확보하고 가장 큰 이익을 낼지 판단할 수 있다는 것. 내 가게 데이터를 가지고 분석해 볼 만하다고 말한다.

### [06:42–08:27] 빅데이터 분석의 기본 흐름 개관
다양한 활용처를 어떻게 시스템으로 구성할지의 이야기로 넘어간다. 꼭 이 모양대로 다 구축해야 하는 건 아니지만 대략의 흐름을 이해해 두면 좋다고 전제한다. 첫 단계는 **데이터 소스**(유튜브면 사용 패턴, 장바구니/커피숍이면 어떤 물건과 어떤 물건을 같이 샀는지). 이 단계에서 **데이터 기획자**가 어디서 데이터를 가져올지 아이디어와 기획을 해 줘야 한다. 그 뒤 시스템 구축은 **데이터 엔지니어(엔지니어링)**, 그다음이 **데이터 분석가**의 몫이다. 단, 데이터 기획은 꼭 전문 기획자가 아니어도 우리가 생각할 수 있는 범위라고 덧붙인다. 흐름을 한 줄로 보면 데이터 소스 → 수집(모으기) → 저장 → 분석 → 비즈니스 활용이다.

### [08:27–10:28] 데이터 소스의 종류 — 로그·DB·IoT
약식 구성도 가능하다며 데이터 소스를 구체화한다. **시스템 로그**가 대표적인데, 데이터베이스와의 차이는 DB에는 정확한 데이터가 저장되어 있고 시스템 로그는 흘러가는 데이터(사용자 접속, 페이지 체류, 상품 클릭)라는 점이다. 이런 것도 모두 데이터가 될 수 있다. 물류 데이터, 서비스 사용 패턴, 판매 데이터도 대표적 소스다. 나아가 **디바이스나 IoT 기기**(센서, RFID 칩 등)로부터 로그를 모아 분석하는 것은 좀 더 빅데이터에 가까운 영역이다. 어떤 물건이 오는지 등을 기기로부터 수집하는 차원. 시스템/서비스를 만들 사람은 데이터를 모으는 과정으로 보면 되고, 물류·디바이스까지 확대하면 IoT 기기까지 생각할 수 있다.

### [10:28–13:01] 데이터 수집·정제와 저장 — Logstash, Elasticsearch, ETL
**수집**은 모으는 것 자체이고, **정제**는 모으다 보면 생기는 없는 값·불필요한 값을 정리해 주는 과정(전처리, preprocessing)이다. 기술적으로 **Logstash** 같은 모듈이 데이터를 전처리해 준다. 로그에는 몇 시 몇 분에 접속, 어떤 사용자가 클릭, 어떤 IP에서 접속 등 다양한 데이터가 있는데, 이 중 굳이 필요 없는 부분을 정제 처리한다. 데이터를 담는 DB/검색 엔진으로는 **Elasticsearch**를 많이 쓴다. 처음 모으는 곳을 **데이터 레이크**, 조금 더 변환해 두는 곳을 **데이터 웨어하우스**라 부르는데, 이 전부를 구축하지 않을 수도 있다 — 화자가 설명하는 정도는 대형 서비스·큰 회사의 상황이고, 작게 구축한다면 Elasticsearch만으로 이 역할까지 포함할 수 있다. 기억해 둘 것으로 **ETL**을 든다: Extract(추출)·Transform(변환)·Load(적재). 데이터를 변환해 저장·적재하는 과정으로 이해하면 된다.

### [13:01–16:00] 분석 단계 — 상관관계 분석과 군집 분석
적재된 데이터에서 일부를 뽑아 어떤 관계가 있는지 여러 모델로 분석한다. 대표적으로 **상관관계 분석**과 **군집 분석**이 있다. 장바구니 예에서 기저귀와 맥주가 같이 있더라는 것은 처음부터 안 게 아니라, 많은 데이터를 보다 보니 두 상품에 관계성이 있다고 사후에 규명한 것이다. 이렇게 묶어 추천하거나 이벤트를 할 수 있다. 커피숍 판매량은 **날씨와의 상관관계**를 분석해 볼 수 있는데, 더운 날 아이스 아메리카노가 잘 팔린다면 재료 준비에 감안한다. **군집 분석**은 데이터를 묶음으로 나누는 것으로, 화자가 실제 해 본 사례로는 시스템을 만든 뒤 오류 로그를 찍어 보니 특정 패턴이 생겼고, 전혀 나오지 않던 **새 패턴**이 발견되면 그것을 특이점(이상 상황)으로 잡을 수 있었다. 또 고양이/개 분류 예처럼, 각 특징들을 놓고 점을 찍어 두었다가 새 이미지를 던지면 그 특징이 어느 쪽에 가까운지에 따라 개/고양이로 분류할 수 있다.

### [16:00–17:40] 모델링 절차 — 피처·알고리즘 선택, 학습·테스트, API
결국 모델/알고리즘은 분석하고자 하는 것과 맞는 것을 선택해야 한다. 어떤 **특징(피처)**을 가지고 할지(예: 고양이·개의 어떤 요소를 가져올지)의 선택이 하나 있고, 어떤 **알고리즘**(상관관계 분석/군집 분석 등)을 쓸지의 선택이 또 있다. 그 선택으로 데이터를 학습시키고, 알고리즘에 들어가는 값들을 조정하며 반복 테스트해 결과가 적정한지 보면서 계속 발전시킨다. 최종 모델을 **API**로 제공하는데, API란 어떤 값을 넣으면 결과값을 주는 것이다. 고양이/개 예로, 새 이미지가 오면 특징 몇 개를 뽑아 API로 전달하고 API가 "고양이/개"로 응답한다. 간단한 예지만, 더 다양한 것을 만들어 실제 서비스와 연동할 수 있다는 것이 API 제공의 측면이다.

### [17:40–19:24] 비즈니스 활용처 — 수요 예측·개인화 추천·장애 예측
**수요 예측**은 매우 중요하고 잘 쓰이는 대표 예다. 어떤 지점·나라의 판매량을 예측해 그곳에 물량을 더 확보한다. 엉뚱한 곳에 두면 물류 비용도 크고 재고도 문제지만, 수요 예측을 잘할수록 큰 경비 절감이 가능하다. 작은 가게나 편의점도 어떤 물건을 얼마나 떼올지를 요일별·시간별로 예측해 적정 물량을 두는 것이 가장 좋은 재고 관리다. **개인화 추천**은 [의견] 지금 서비스들이 이렇게까지 발전할 수 있었던 핵심이며(유튜브에 빠져든 것도 결국 오른쪽 추천 영상 때문), 서비스에 잘 활용하면 더 많은 체류 시간을 가져올 수 있다. **장애 예측**은 앞의 군집 분석으로, 나오지 않던 새 패턴이 나오면 알람을 받아 처리하는 개념이다.

### [19:24–21:26] 간단한 실전 아키텍처 (약식 구성)
당장 시스템 한두 대로도 만들 수 있는 구형/약식 관점을 제시한다. 시스템 로그는 파일로 떨어지는데, 그 파일을 수집하는 모듈이 **Filebeat**(파일을 자꾸 가져옴)다. 형태를 바꿔 정제하는 것이 **Logstash** — 이름만이라도 기억해 두면 도움이 된다. 정제한 데이터를 저장하는 곳이 **Elasticsearch**(속도·검색이 빠름)이고, 여기서 바로 데이터를 가져와 분석할 수도 있다. 분석에는 **Python**이 상당히 좋다. 파이썬 분석으로 알고리즘을 적용해 모델을 만들고, 이를 API로 제공할 때는 **Flask**를 많이 쓰고 **FastAPI**도 많이 쓴다(둘 다 파이썬 기반). 정리하면 로그를 쌓고 → 수집 → 정제 → 저장 → 분석 알고리즘 → 모델 완성 → API 적용의 체계이며, 활용 예로 유저가 접속하면 그 유저를 API에 넣어 "이 사람에게 추천할 만한 것"의 리스트를 받아 쓰는 흐름이다.

### [21:26–23:03] 마무리 — 공부·서비스 접목에의 도움과 CTA
이 구성은 실제 서비스에 적용할 수도 있고, ADSP 같은 자격증 공부 중이면 "실제로는 이렇게 구축하는가 보다"라고 간접 이해를 도울 수도 있으며, 회사에서 빅데이터 분석을 구축해 보자고 할 때 제안해 볼 수도 있다. 데이터를 분석하는 것까지 해 보면 서비스 확장, 체류 시간 증가, 장애 발생 시 빠른 원인 파악에 도움이 된다. 내부 모델·알고리즘·저장/수집 방식은 조금씩 바뀔 수 있지만 **전체 틀은 이런 형태**임을 기억하면 공부에도 서비스 접목에도 도움이 될 것이라고 정리한다. 도움이 되었다면 좋아요를 눌러 달라, 그러면 관련 주제를 계속 올리겠다는 CTA로 마친다.
