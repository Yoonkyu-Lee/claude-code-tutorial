# 클로드코드 잘 쓰는 10단계 — 순서 틀리면 90%가 포기합니다

> maker-evan · 2026-03-10 · 길이 약 11:55 · [영상 링크](https://youtu.be/d-iNXwtwdFU)
> 정리 방식: 시간순 심층 워크스루 (paraphrase) + 주제별 상세 요약
> 검증 배너: 수익·비용·성능 관련 수치는 화자 주장이며 미검증. 원제·표현은 그대로 두되 자동자막 오인식은 문맥 교정함.

## 요약 (주제별)

### 핵심 프레임: "치매 걸린 아인슈타인"
- 화자는 클로드 코드를 "치매에 걸린 아인슈타인"으로 규정한다. 복잡한 코드·아키텍처 설계·에러 디버깅은 천재적으로 하지만 **기억을 못 한다**는 것. 오늘 대화 내용을 내일 새 대화에서 완전히 잊고, "이 방식 쓰지 마"라던 지시도 새 세션에서 사라지며, 어제 고친 버그를 오늘 다시 만든다. 대화 중간에도 "치매(컴팩션)"가 온다.
- [의견] 사람들이 클로드 코드에 실패하는 건 AI가 멍청해서가 아니라 **순서 문제**라는 게 영상 전체 주장. 이해 없이 도구(플러그인·MCP·프레임워크)만 쌓아 올린 게 원인.
- 10단계의 본질: 이 "천재이지만 기억을 못 하는 존재"를 어떻게 세팅하고 관리하느냐.

### 10단계 로드맵 (순서가 핵심)
- **1단계**: 플러그인 없이 기본 상태로 시작. 일주일간 참으며 클로드가 어떻게 일하는지 관찰.
- **2단계**: 튜토리얼 지옥 탈출, 지금 당장 작은 것 하나 만들기.
- **3단계**: 반복 작업을 스킬로 저장.
- **4단계**: MCP 연결(팔다리 달아주기).
- **5단계**: 자주 안 쓰는 기능은 MCP 대신 스킬/CLI로, 안 쓰는 MCP는 끄기(토큰 절약).
- **6단계**: 컨텍스트 엔지니어링 + CLAUDE.md(장기 메모리) + 컴팩션 회피. **화자가 10단계 중 제일 중요하다고 꼽음.**
- **7단계**: 토큰 절약 방법론(필요 파일만·모델 선택·CLI/스킬).
- **8단계**: 내 소모량 직접 측정.
- **9단계**: Max 플랜에서 한계까지 밀어붙여 보기.
- **10단계**: 하니스 찾기.
- [의견] 핵심은 "순서대로 해야 한다". 10단계를 1단계에서 하면 기초가 없어 흔들린다.

### 초기 3단계 (기초 감각)
- **1단계 — 기본 상태 관찰**: 클로드 코드를 처음 켜면 까만 터미널 창뿐. 허전해 보여 다들 바로 플러그인을 찾지만, 딱 일주일만 참고 클로드가 파일을 어떤 순서로 읽는지·복잡한 작업을 어떻게 쪼개는지·오류 시 어떻게 복구하는지·언제 이상하게 행동하는지를 눈으로 봐야 한다. 이걸 모르면 플러그인이 뭘 보완하는지도 모르는 "블랙박스"가 된다. "왜 대화가 길어지면 이상해지지?" 같은 질문이 생기는 게 성장.
- **2단계 — 튜토리얼 지옥 탈출**: 영상·블로그·강의로 다섯 시간 공부해도 만든 게 없으면 소용없다. 클로드 코드는 이론이 아니라 실습 도구(자전거·수영 비유). 가계부 앱이든 일정 도구든 **내가 쓸 것 하나**를 정해 같이 만들어라. 완성도 50%짜리를 만들며 배우는 게 완벽한 튜토리얼보다 빠르다.
- **3단계 — 스킬 만들기**: 배포·코드리뷰·커밋 메시지처럼 매번 같은 순서를 설명하는 반복이 보이면, 그 순서를 레시피처럼 저장한 게 스킬(볶음밥 레시피 노트 비유). 판별 기준: **두 번 이상 반복한 작업이 있으면 스킬로 만들어라.**

### MCP와 토큰 비용 (4·5단계)
- **4단계 — MCP 연결**: MCP(Model Context Protocol)는 클로드 코드에 "팔다리"를 붙여 도구를 쓰게 하는 것. GitHub MCP → PR 생성·이슈·코드리뷰, 브라우저 MCP → 웹페이지 열기·클릭·스크린샷, 데이터베이스 MCP → DB에 직접 쿼리.
- **5단계 — MCP 대신 스킬/CLI**: MCP를 켜는 순간 대화 시작 전부터 비용이 나간다. 토큰 = 클로드의 "연료"이자 치매 아인슈타인의 기억 용량이며, 꽉 차면 오래된 기억부터 지워진다. MCP 서버는 세션 시작 시 "이런 기능들이 있다"는 목록을 통째로 밀어 넣어, 한 글자 쓰기 전부터 소비된다.
- **수치(주장/미검증)**: 커뮤니티 측정으로 **MCP 서버 5개를 켜면 전체 용량의 41%가 사라진다**고 함. 해결책 두 가지: 자주 안 쓰는 기능은 스킬로(호출 때만 로드), 안 쓰는 MCP는 끄기.

### 컨텍스트 엔지니어링과 CLAUDE.md (6단계)
- 컨텍스트 엔지니어링 = 상황에 딱 맞는 정보만 골라 주는 기술. 치매 아인슈타인에게 뭘 기억/망각시킬지 설계하는 것. [의견] 화자가 10단계 중 가장 중요하다고 봄.
- 인수인계 문서 비유: 매일 처음부터 설명하는 대신 인수인계 문서 한 장을 잘 써 두고 출근 때마다 읽게 하는 것 = **CLAUDE.md**. 프로젝트 폴더에 두면 대화 시작 시 자동으로 읽힌다. 프로젝트 구조·코딩 규칙·자주 쓰는 명령어·절대 하면 안 되는 것을 적어 둔다.
- 흔한 실수: CLAUDE.md를 너무 길게(1,000줄 등) 쓰는 것. 길수록 토큰을 먹고, 규칙이 100개면 클로드도 사람처럼 다 못 따른다. [의견] `/init`으로 자동 생성된 것보다 **AI가 잘 인지 못 하는 부분**을 골라 직접 기입해야 효과가 좋다.

### 컨텍스트 컴팩션 (6단계 후반)
- 컨텍스트가 일정량 쌓이면 클로드 코드가 자동으로 지금까지의 대화를 강제 요약·압축한다("치매 발작"). 컴팩션 후엔 이전에 100% 따르던 지시를 위반하고, 처음부터 다시 설명하며, 고쳤던 버그가 재발하고, 읽던 파일을 잊어 다시 읽는다.
- 회피 3원칙: ① CLAUDE.md를 짧고 핵심적으로(컴팩션 후에도 자동 재로딩되는 장기 메모리이므로), ② 관련 없는 작업 시작 시 `/clear`를 적극 사용해 대화를 쌓지 않기, ③ 작업을 잘게 쪼개 한 세션에 하나의 작업만. [의견] 컴팩션은 선택이 아니라 피하거나 최소화해야 하는 대상.

### 토큰 절약과 자기 측정 (7·8단계)
- **7단계 — 토큰 절약 방법론**: ① 필요 없는 파일은 주지 않기(로그인 기능 고칠 때 전체 프로젝트 대신 로그인 관련 파일만). 읽는 파일 수를 줄일수록 토큰 절약·컴팩션 지연. ② 모델 선택 — 복잡한 설계는 Opus, 간단한 코드 수정은 Sonnet이 빠르고 저렴. ③ MCP 대신 CLI 도구·스킬(파일 검색은 grep, 간단한 배포는 스킬). MCP는 진짜 필요할 때만.
- **8단계 — 소모량 측정**: 6~7단계 최적화 기법을 실제로 적용해 복잡한 작업 하나를 처음부터 끝까지 완주하며 내 소비 패턴을 파악. "한 달 식비를 모르면 절약 못 한다"는 비유. 이 규모면 대략 이만큼 쓴다는 감각이 생기면 플랜 선택도 현명하게 할 수 있다.

### 한계 밀어붙이기와 하니스 (9·10단계)
- **9단계 — Max 플랜 한계까지**: Max 플랜은 프로 플랜 위에 두 단계 — 5배짜리 월 100달러, 20배짜리 월 200달러(주장/미검증). 엄청 복잡한 프로젝트 하나를 처음부터 끝까지 클로드 코드로만 해 봐야 진짜 능력이 보이고, "이건 클로드 코드로 하면 안 되는구나"도 알게 되어 모든 걸 맡기려는 집착에서 벗어난다(고속도로 안 타 본 차 비유).
- **10단계 — 하니스 찾기**: 하니스 = 등산 안전벨트처럼 클로드가 내 방식대로 일관되게·실수 없이 일하도록 잡아주는 구조. 클로드 코드의 자유로움이 단점(세션마다 다르게 행동)이 될 때 이를 막아준다. Oh My Claude Code, Superpowers 등 다양하니 취향껏. 10단계에 배치한 이유: 1~9단계 경험 없이 처음부터 쓰면 기능이 너무 많아 뭘 왜 쓰는지 모르는 채 도구만 쓰게 되기 때문.

### 세일즈·커뮤니티 구조
- 하드 세일즈나 유료자료·단톡방 유입 장치는 두드러지지 않음. 대신 CTA로 "지금 몇 단계인지 댓글에 숫자 하나 남겨 달라", 어디서 막혔는지 함께 쓰면 제일 많이 나온 단계로 심화 영상을 만들겠다고 함. 구독·좋아요와 "커뮤니티가 있으면 공유해 달라"는 요청으로 마무리.

## 3줄 오리엔테이션
- 클로드 코드가 안 느는 건 AI가 멍청해서가 아니라 **배우는 순서가 틀려서**라는 주장 — 처음부터 플러그인·MCP·프레임워크부터 쌓는 게 실패 원인.
- 핵심 멘탈모델은 "치매 걸린 아인슈타인": 천재적이지만 세션·컴팩션마다 기억을 잃으므로, 10단계는 이 존재를 세팅·관리하는 순서다.
- 1(기본 관찰)→2(실전 제작)→3(스킬)→4(MCP)→5(MCP 최적화)→6(컨텍스트/CLAUDE.md)→7(토큰 절약)→8(측정)→9(한계 테스트)→10(하니스) 순서를 지키는 게 전부.

## 전문 — 시간순 워크스루

### [00:00–01:15] 도입 — "AI 탓이 아니라 순서 문제"
클로드 코드를 처음 쓰면 "와, 이게 되네" 하고 신기하지만 얼마 지나면 막히기 시작한다고 연다. 플러그인은 뭘 써야 할지, MCP는 뭔지 모르겠고, 유튜브에선 다들 잘 쓰는 것 같은데 나는 왜 느린지 답답해진다. 화자는 이게 AI 탓이 아니라 **순서 문제**라고 단언하며, 오늘 그 순서를 10단계 로드맵으로 알려주겠다고 한다. 영상을 끝까지 보면 세 가지가 명확해진다고 예고한다.
> "AI탓이 아닙니다. 순서 문제예요." [00:17]
- [의견] 끝까지 보면 ① 왜 안 늘었는지 이유, ② 지금 내가 어느 단계인지, ③ 오늘부터 뭘 해야 하는지가 보인다고 함.

### [01:15–01:20] 커뮤니티의 가장 흔한 실수
커뮤니티에서 사람들이 제일 많이 하는 실수는 처음부터 플러그인 깔고 MCP 다섯 개 연결하고 프레임워크 설치하고 시작하는 것. 세팅이 빵빵해 보이지만 막상 쓰면 오류가 나는데 어디서 나는지 모르고, 클로드가 이상하게 행동하는데 이유를 모른 채 "AI가 멍청한가 보다" 하고 포기하게 된다.

### [01:20–01:38] 왜 실패하나 — 이해 없이 도구만 쌓음
사실 AI가 멍청한 게 아니라 **이해 없이 도구만 쌓아 올린 것**이 문제다. 수영 못 하는 사람에게 올림픽 선수용 최신 수경·전신 수영복을 다 사줘도 물에서 팔을 어떻게 젓고 언제 숨 쉬는지 모르면 소용없다는 비유. 기초부터 쌓아야 나중에 뭔가 이상해졌을 때 직접 고칠 수 있다.
> "이해 없이 도구만 쌓아올린 거예요." [00:53]

### [01:38–02:00] 근본 사실 — "치매 걸린 아인슈타인"
도구를 쓰기 전에 알아야 할 근본 사실을 제시한다. 클로드 코드는 "치매에 걸린 아인슈타인"이다. 복잡한 코드 작성·아키텍처 설계·에러 디버깅은 아주 잘하지만 기억을 못 한다. 오늘 대화를 내일 새 대화에서 완전히 잊고, "이 방식 절대 쓰지 마"라던 지시도 새 세션엔 없던 일이 되며, 어제 함께 고친 버그를 오늘 또 만든다. 내 코딩 스타일·프로젝트 규칙·쌓아온 맥락이 새 대화를 여는 순간 다 사라진다. 게다가 대화 중간에도 치매가 오는 경우가 있다. 그래서 이 존재를 어떻게 세팅·관리하느냐가 잘 쓰는 사람과 못 쓰는 사람의 차이라며 10단계로 진입한다.
> "클로드 코드는요. 치매에 걸린 아인슈타인이에요." [01:22]

### [02:00–02:46] 1단계 — 플러그인 없이 기본 상태로 관찰
클로드 코드를 처음 켜면 그냥 까만 터미널 창에 텍스트뿐이라 허전해 보여 다들 바로 플러그인을 찾고 세팅을 만지고 싶어 한다. 딱 일주일만 참으라고 한다. 이 기간에 치매 아인슈타인이 어떻게 일하는지 직접 봐야 한다 — 파일을 어떤 순서로 읽는지, 복잡한 작업을 어떻게 쪼개는지, 오류 시 어떻게 혼자 복구하려 하는지, 언제 이상하게 행동하는지. 이걸 모르면 플러그인이 뭘 해주는지도 모르는 블랙박스가 된다. 기본 상태로 쓰다 보면 "왜 이건 고치고 저건 못 고치지? 왜 대화가 길어지면 이상해지지?" 같은 질문이 생기는데, 그 질문이 생기는 게 성장이다.

### [02:46–03:18] 2단계 — 튜토리얼 지옥 탈출
유튜브·블로그·강의로 다섯 시간을 공부해도 정작 만든 게 없는 상태를 "튜토리얼 지옥"이라 부른다. 클로드 코드는 이론으로 넣는 도구가 아니다. 자전거 타는 법을 책으로 100시간 공부해도 실제로 타 넘어져 보지 않으면 못 하듯. 그러니 지금 당장 만들 것 하나를 정하라 — 아무리 작아도 좋고, 가계부 앱이든 일정 정리 도구든 내가 쓸 걸 정해 클로드와 만들어 보라. 완성도 50%짜리를 만들며 배우는 게 완벽한 튜토리얼을 보는 것보다 훨씬 빠르게 는다.

### [03:18–04:08] 3단계 — 스킬 써보고 만들기
어느 정도 쓰다 보면 "나 이 작업 또 하는 거 아니야?" 하는 생각이 든다. 배포할 때마다 같은 순서를 설명하고, 코드리뷰할 때마다 같은 기준을 얘기하고, 커밋 메시지마다 같은 형식을 알려주는 반복이 생긴다. 여기서 스킬이 등장한다 — 반복되는 작업 순서를 레시피처럼 저장하는 것. 볶음밥 만드는 법을 매번 구글링하는 대신 레시피 노트에 적어 두고 꺼내 쓰는 것과 같다. 코드 저장 시 특정 규칙이 있다면 스킬 파일 하나 만들어 두면 끝. 또 다른 장점은 토큰을 거의 안 먹는다는 점 — 호출할 때만 로드되므로 스킬 열 개를 설치해도 기본 소모가 거의 없다. 판별 기준: **두 번 이상 반복한 작업이 있으면 스킬로 만들어라.**

### [04:08–04:37] 4단계 — MCP 연결
MCP는 Model Context Protocol의 약자로, 쉽게 말해 클로드 코드에 "팔다리"를 붙여 도구를 쓸 수 있게 하는 것. GitHub MCP를 연결하면 클로드가 직접 PR을 만들고 이슈를 달고 코드리뷰를 할 수 있고, 브라우저 MCP를 연결하면 실제로 웹페이지를 열고 클릭하고 스크린샷을 찍으며, 데이터베이스 MCP를 연결하면 DB에 직접 쿼리를 날릴 수 있다. 이걸 간편하게 해주는 것이 MCP다.
> "즉 팔다리를 달아줘서 도구를 사용할 수 있게 하는 거죠." [04:32]

### [04:37–05:22] 5단계 — MCP 대신 스킬/CLI (토큰 비용)
MCP를 켜는 순간 대화 시작 전부터 비용이 나간다. 토큰은 클로드 코드가 쓰는 연료이자 치매 아인슈타인의 기억 용량이라, 꽉 차면 오래된 기억부터 지워진다. MCP 서버는 세션 시작 시 "나 이런 기능들이 있어요"라는 목록을 통째로 클로드 머릿속에 밀어 넣어, 사용자가 한 글자도 쓰기 전에 이미 소비된다. 커뮤니티에서 측정한 수치로 **MCP 서버 5개를 켜면 전체 용량의 41%가 사라진다(주장/미검증)**. 해결책은 두 가지 — 자주 안 쓰는 기능은 MCP 대신 스킬로 만들고(호출 때만 로드), 안 쓰는 MCP는 끈다.

### [05:22–05:39] 기초 단계 마무리 → 고수 영역 전환
1~5단계까지 했으면 기본 세팅은 끝, 튜토리얼 레벨이 끝난 것이다. 어떻게 작동시키고, 반복 작업을 어떻게 저장하고, MCP 비용을 어떻게 관리하는지까지가 기초. 6단계부터는 클로드를 진짜 내 방식으로 길들이는 "고수의 영역"이라고 예고한다.

### [05:39–06:23] 6단계 — 컨텍스트 엔지니어링과 CLAUDE.md
6단계는 컨텍스트 엔지니어링으로, [의견] 화자가 전체 10단계 중 제일 중요하다고 보는 단계다. 컨텍스트 엔지니어링은 클로드에게 상황에 딱 맞는 정보만 골라 주는 기술 — 치매 아인슈타인에게 뭘 기억하게 하고 뭘 잊게 할지 설계하는 것. 신입 직원 비유: 매일 처음부터 회사와 프로젝트를 설명하는 대신, 인수인계 문서 한 장을 잘 써 두고 출근 때마다 읽게 하는 편이 낫다. 클로드 코드에서 그 인수인계 문서가 **CLAUDE.md 파일**이다. 프로젝트 폴더에 만들어 두면 클로드가 대화 시작 시 자동으로 읽는다. 프로젝트 구조·코딩 규칙·자주 쓰는 명령어·절대 하면 안 되는 것들을 써 두면 매번 설명할 필요가 없다.

### [06:23–06:52] 6단계 — CLAUDE.md의 흔한 실수
쉽게 하는 실수는 CLAUDE.md를 너무 길게 쓰는 것이다. 일단 다 써 놓겠다고 1,000줄, 2,000줄 쓰는 사람이 있는데 그러면 안 된다 — 길수록 토큰을 먹고, 규칙이 100개면 사람도 다 기억 못 하듯 클로드도 다 못 따라온다. [의견] 그리고 굉장히 중요한 점으로, `/init`(슬래시 이닛)을 통해 자동 생성된 CLAUDE.md보다는 **AI(클로드)가 잘 인지하지 못하는 부분**에 대해 CLAUDE.md에 직접 기입해 둬야 효과가 좋다.

### [06:52–07:22] 6단계 — 컨텍스트 컴팩션
예고했던 컨텍스트 컴팩션. 클로드 코드는 치매 노인이라 컨텍스트가 일정량 쌓이면 자동으로 압축을 해버린다 — 지금까지 나눈 대화를 강제로 요약해버리는 "치매 발작". 컴팩션이 발동되면 그 전에는 지시를 100% 따르던 클로드가 이후엔 그 지시를 위반하고, 처음부터 다시 설명하게 되며, 이미 고쳐 달라던 버그가 다시 나타나고, 어떤 파일을 읽고 있었는지 잊어 다시 읽기 시작한다. 그래서 컴팩션은 최대한 피해야 한다.

### [07:22–07:57] 6단계 — 컴팩션 회피 3원칙
회피법 세 가지. 첫째, CLAUDE.md를 짧고 핵심적으로 써 둔다 — 컴팩션이 발동돼도 CLAUDE.md는 다시 자동으로 읽히므로, 결국 CLAUDE.md가 이 치매 아인슈타인의 장기 메모리다. 둘째, `/clear` 명령어를 적극적으로 쓴다 — 관련 없는 작업을 시작할 때는 무조건 `/clear`로 대화를 쌓지 않고 깨끗하게 시작하면 컴팩션이 발동될 일이 잘 없다. 셋째, 작업을 잘게 쪼갠다 — 한 대화에서 너무 많은 걸 하면 컴팩션이 오므로 한 세션에 하나의 작업만 하는 습관. [의견] 컴팩션은 선택이 아니라 피하거나 최소화해야 한다.

### [07:57–08:34] 7단계 — 토큰 줄이는 방법론
7단계는 토큰 절약 방법론. 첫째, 필요 없는 파일은 주지 않는다 — 로그인 기능을 고치는데 전체 프로젝트를 다 읽힐 필요 없이 로그인 관련 파일만 주면 된다. 읽는 파일 수를 줄일수록 토큰이 절약되고 컴팩션도 늦게 온다. 둘째, 모델 선택 — 클로드에는 여러 모델이 있어 복잡한 설계는 Opus가 낫지만 코드 수정이나 간단한 작업은 Sonnet이 훨씬 빠르고 저렴하다. 작업 난이도에 맞는 모델을 골라 쓰면 토큰도 아끼고 속도도 빠르다. 셋째, MCP 대신 CLI 도구와 스킬 — 파일 검색은 grep 명령으로, 간단한 배포는 스킬로 만들 수 있고, MCP는 진짜 필요할 때만 쓰고 나머지는 가볍게 처리하는 편이 효율적이다.

### [08:34–09:11] 8단계 — 내 소모량 측정
8단계는 한 세션에 얼마나 쓰는지 직접 확인. 6~7단계에서 최적화 기법을 다 배웠으니 이제 실제로 적용하고 소비 패턴을 파악한다. 복잡한 작업 하나를 정해 처음부터 끝까지 돌려보며 스킬도 쓰고 컨텍스트 관리도 하고 최적화 기법을 전부 적용해 진짜 어려운 작업 하나를 완주한다. 요리 예산 비유 — 한 달 식비를 얼마 쓰는지 모르면 절약할 수 없듯, 먼저 내가 얼마 쓰는지 알아야 줄일 수 있다. 이 규모 프로젝트면 대략 이만큼 쓴다는 감각이 생기면 플랜 선택도 현명해진다. 실제 소모량을 모른 채 플랜을 고르면 너무 많이 사거나 너무 적게 사게 된다.

### [09:11–09:53] 9단계 — Max 플랜 한계까지 써보기
9단계는 클로드 코드 Max 플랜을 쓴다면 한계까지 써보기. Max 플랜 구조는 기본 프로 플랜 위에 두 가지 — 5배짜리가 월 100달러, 20배짜리가 월 200달러(주장/미검증). 왜 해야 하냐면, 차를 처음 사서 주차장에서만 타다 고속도로를 한 번도 안 타보면 이 차가 어디까지 가는지 모르듯, 안전한 범위에서만 쓰면 그 모습밖에 못 본다. 엄청 복잡한 프로젝트 하나를 처음부터 끝까지 클로드 코드로만 해봐야 그 능력이 진짜로 보인다. 그리고 "이건 클로드 코드로 하면 안 되는 거구나"를 알게 되는 것도 중요한데, 모든 걸 클로드 코드로 하려는 집착에서 벗어나 뭘 맡기고 뭘 내가 해야 하는지가 명확해진다.

### [09:53–10:31] 10단계 — 하니스 찾기
마지막 10단계는 하니스 찾기. 하니스는 등산할 때 몸에 묶는 안전벨트로, 절벽에서 떨어져도 잡아준다. 클로드 코드에서 하니스는 클로드가 내 방식대로 일관되게 실수 없이 일하도록 잡아주는 구조다. 클로드 코드는 자유롭게 뭐든 하는데 그게 단점이 되기도 한다 — 내가 원하는 방식이 있는데 클로드가 다른 방식으로 하거나 세션마다 다르게 행동할 때 하니스가 그걸 막아준다. Oh My Claude Code, Superpowers 등 다양하니 취향껏 골라보라고 한다. 하니스를 10단계에 배치한 이유는, 1~9단계 경험 없이 처음부터 하니스를 쓰면 기능이 너무 많아 뭘 왜 쓰는지 모르는 채 도구만 쓰게 되기 때문. 충분히 기초를 쌓고 써야 진짜 효과가 난다.

### [10:31–11:38] 전체 정리 (10단계 요약)
클로드 코드는 치매에 걸린 아인슈타인 — 천재인데 기억을 못 하며, 10단계는 이 천재를 어떻게 세팅·관리할 것인가다. 1단계 플러그인 없이 기본 상태로 시작해 작동 방식 파악, 2단계 실전 프로젝트로 튜토리얼 지역 탈출해 지금 당장 만들기, 3단계 반복 작업을 레시피(스킬)로 저장, 4단계 필요한 팔다리(MCP) 붙이기, 5단계 안 쓰는 MCP는 끄고 자주 안 쓰는 기능은 스킬로 만들어 기억 용량 지키기, 6단계 컨텍스트 엔지니어링 — CLAUDE.md가 장기 메모리이니 짧고 핵심적으로 쓰고 `/clear`로 컴팩션 피하기, 7단계 토큰 절약 — 작업에 맞는 모델 선택·MCP 대신 CLI와 스킬·필요한 파일만 주기, 8단계 내 소모량을 숫자로 파악, 9단계 한계까지 써보며 진짜 능력 확인, 10단계 내 스타일에 맞는 하니스 찾기. 핵심은 하나 — **순서대로 해야 한다**. 10단계를 1단계에서 하면 안 되고, 성급하게 올라가면 기초가 없어 흔들린다.
