> 검증 배너: 이 영상은 도구 홍보성 요소가 있고, 화자 개인 워크플로우의 효과·소요시간은 본인 경험 진술이라 미검증. 브랜드/스킬명은 자동자막 음차를 문맥으로 교정했다.

## 요약 (주제별)

### 1. 세 도구의 역할 분담
- 화자는 스킬 하나만으로는 한계가 분명해서 여러 스킬을 조합해 쓴다고 밝힌다. 핵심 3종은 **맥퍼커(Matt Pocock 계열 스킬) · G스택(gstack) · 슈퍼파워즈(Superpowers)**.
- 역할이 자연스럽게 나뉜다: **도메인/DDD는 맥퍼커, 스펙 브레인스토밍은 G스택, 실제 구현은 슈퍼파워즈.**

### 2. 맥퍼커 — 도메인 맵핑 + 커스텀 DDD
- 맥퍼커 스킬은 약 16개 정도이며 주목적은 하나, **도메인 맵핑**. 서비스에 등장하는 단어·개념·관계를 AI에게 정확히 알려 주는 것. 용어가 흔들리면 코드도 흔들린다는 게 전제.
- 예시: "주문"이라는 단어도 일반 쇼핑몰 주문 / 음식 배달 주문 / B2B 발주가 다 다른데, 그냥 "주문"으로 던지면 AI가 헷갈린다. 계속 캐물어 정리해 도메인 파일로 만든다. 비유: 신입에게 회사 용어집 주는 것.
- 화자는 여기에 한 발 더 나가 **커스텀으로 DDD 아키텍처링**을 얹었다. 도메인 용어를 사전으로만 쓰는 게 아니라 폴더 경계·모듈 경계·함수 시그니처까지 반영되게 했다. "주문" 도메인은 주문 모듈 안에서 끝나고 "결제"는 결제 모듈로 분리. 용어가 코드 아키텍처를 결정하는 1차 자료가 된다. [의견] AI가 도메인 경계를 따라 코드를 짜니 모듈이 섞이지 않는다는 점을 "나중에 결정적"이라고 평가.

### 3. G스택 — 스펙 브레인스토밍과 오피스아워 검증
- G스택의 진짜 강점은 **브레인스토밍, 특히 제품 스펙 작성**. "같이 생각해 줘" 수준이 아니라, 스펙에서 뭘 빼고 뭘 살릴지 심도 깊게 캐묻는다. "이 기능 핵심 가치에 붙는가? 누가 써? 없으면 어떻게 돼? 이미 비슷한 걸 쓰는 사람 있어?" 식으로 한 줄 한 줄 흔든다.
- [의견] 스펙에서 **빼는 게 더 어렵고 더 중요**하다(다 넣고 싶으니까). G스택은 뺄 건 빼라고 끝까지 밀어붙인다.
- 대표 스킬 **오피스아워(office hours)** = YC 오피스아워(창업자가 멘토에게 깨지는 자리)를 그대로 옮긴 것. **여섯 가지 강제 질문**: 누가 정말 이걸 원해 / 지금 어떻게 해결하고 있어 / 진짜 절박한 문제야 / 가장 작은 단위가 뭐야 / 직접 본 적 있어 / 미래가 보여. 한 번 거치면 핵심과 노이즈가 갈린다.
- 답변 후에도 끝이 아니라 **플랜 CEO 리뷰 / 플랜 리뷰** 같은 후속 스킬로 스펙을 다시 다듬어, 브레인스토밍에서 뽑을 수 있는 걸 끝까지 뽑는다.

### 4. 슈퍼파워즈 — 실제 구현 엔진
- 실제 코드 작성 단계의 메인 엔진. 화자가 주로 쓰는 세 가지: **브레인스토밍, 라이트 라이팅 플랜(writing plans), 서브에이전트 드리븐 디벨롭먼트(sub-agent driven development)**.
- 브레인스토밍: G스택 브레인스토밍 결과를 받아 슈퍼파워즈에서 한 번 더 돌려, "이걸 코드로 어떻게 풀 거야"에 대한 **상세 스펙 확정** 용도로 쓴다.
- 라이팅 플랜: 플랜을 쓴 뒤 그냥 끝내지 않고 **커스텀 리뷰 에이전트로 플랜 리뷰**를 돌린다(에이전트 리뷰 기능 활용). "2단계 빠졌어, 순서 바꿔, 이 작업 너무 커" 식으로 지적받고 플랜을 수정. 비유: 시니어에게 작업 계획서 검토받기. [의견] 혼자 짠 플랜의 빈틈이 절반 이상 여기서 잡힌다.
- 서브에이전트 드리븐 디벨롭먼트: 큰 작업을 여러 개로 쪼개 각각을 별도 에이전트에 맡기고, 끝나면 다른 에이전트가 검수해 통과해야 다음으로 넘어간다. 비유: 공사에서 전기·배관·도배 각자 전문가. UI도 동일 — 슈퍼파워즈가 목업 이미지를 함께 보며 버튼 위치·색·동작을 뽑아 서브에이전트에 분배. 코드도 "파일 어디·어떤 함수·입력·출력·검증"을 **2~5분짜리 작업 단위**까지 쪼개 배분.

### 5. 전체 워크플로우 (end-to-end 순서)
- 그릴미(grill-me)로 깊은 인터뷰 → 도메인 파일 작성 → (오피스아워로 비즈니스 검증) → 라이트 스펙 → 플랜 → 플랜 리뷰 → 서브에이전트 코드 작성 → 코덱스 리뷰 → 임프로브 코드베이스 아키텍처로 마무리 → (추가로 스킬 기반 리팩토링).
- 한 번 돌리면 **두세 시간, 그 이상** 걸린다(주장/미검증).
- 시작은 무조건 맥퍼커+그릴미. 한 줄 던지고 클로드에게 인터뷰를 요청하면 **30분~1시간 인터뷰**가 시작된다. 이 인터뷰 결과가 그대로 도메인 파일이 되므로 적당히 답하면 안 되고 깊게 답해야 한다. 도메인 파일은 **CONTEXT.md에 박아두는 AI용 사전**. 비유: 신입에게 회사 위키 정독시키기.
- [의견] 1~2단계(도메인+비즈니스 검증)까지 약 1시간, 코드는 한 줄도 안 짠 상태지만 이게 핵심 — 바로 코드부터 짜면 나중에 헛수고가 드러난다.

### 6. 코덱스 교차 리뷰 (중간 검수)
- 서브에이전트에 다 맡겨 놓기만 하면 안 되고 **중간 검수가 핵심**. 화자는 클로드 코드 안에서 **코덱스 플러그인**을 함께 쓴다.
- [의견] GPT 코덱스와 클로드가 서로 다른 모델이라 한쪽이 짠 코드를 다른 쪽이 보면 사각지대가 잡힌다. 요즘은 세세한 구현에서 코덱스가 훨씬 잘하는 것 같고, 대다수 버그가 이 단계에서 잡힌다(주장/미검증).

### 7. 코드 품질 마무리 — 임프로브 코드베이스 아키텍처 + 스킬 리팩토링
- 코드가 거의 완성되면 맥퍼커의 **임프로브 코드베이스 아키텍처(improve codebase architecture)** 스킬을 돌린다. 앞서 만든 CONTEXT.md 도메인 파일을 기준으로 "이 함수는 도메인 개념과 어긋나, 이 둘은 같은 개념인데 떨어져 있어 합치자" 식 리팩토링 포인트를 도메인 언어 기반으로 잡아 준다.
- [의견] 이 스킬 전후로 코드 품질이 크게 다르다 — 그냥 동작하는 코드에서 도메인 모델이 잘 잡힌 코드로 바뀌고, 읽으면 도메인 구조가 한눈에 들어와 신규 인력이 빨리 적응한다(주장/미검증).
- 추가로 프런트엔드는 **React·Vercel 베스트 프랙티스** 스킬, 백엔드는 **커스텀 FastAPI 백엔드 가이드라인** 스킬로 리팩토링. 대표 예: 라우트 경로에 있는 서비스 로직을 서비스 계층으로 옮기기.

### 8. 결과·효과 (화자 주장)
- [의견] 워크플로우가 길게 느껴져도 결과물이 다르다 — 코드 한 줄 한 줄이 도메인 개념과 맞물려, 손볼 때 어디를 봐야 할지 보인다.
- [의견] 결정적으로, 이렇게 나온 코드는 **사람이 나중에 수작업으로 버그를 고칠 수 있는 상태**가 된다.
- 마무리: 다음 영상 예고 — 이 워크플로우의 코드베이스 소개·사용 가이드, 그리고 코덱스 플러그인 셋업하는 법.

## 3줄 오리엔테이션
- 화자가 실제로 쓰는 AI 코딩 워크플로우를 처음부터 끝까지 공개하는 영상 — 스킬 하나가 아니라 여러 스킬을 조합한다는 게 핵심.
- 도구 3종(맥퍼커=도메인/DDD, G스택=스펙 브레인스토밍, 슈퍼파워즈=구현)의 역할을 먼저 정의하고, 이어서 이를 하나로 엮은 두세 시간짜리 실제 순서를 보여 준다.
- 반복 강조점: 코드부터 짜지 말고 도메인·비즈니스 검증을 먼저 하고, 서로 다른 모델(코덱스)로 교차 검수하고, 도메인 언어 기반으로 마무리 리팩토링하라는 것.

## 전문 — 시간순 워크스루

### [00:00–00:12] 인트로 — 조합형 워크플로우 예고
화자는 자신이 실제로 쓰는 워크플로우를 처음부터 끝까지 보여 주겠다며, **맥퍼커 + 슈퍼파워즈 + G스택을 조합한 워크플로우**라고 소개한다. 스킬 하나만 쓰면 분명한 한계가 하나씩 있어서, 자신은 여러 스킬을 콤비네이션으로 쓴다고 말한다.

### [00:13–00:22] 맥퍼커의 주목적 = 도메인 맵핑
맥퍼커 스킬은 도메인 맵핑이 주된 목적으로, "우리 도메인에서 이 단어가 정확히 어떤 뜻인지 AI에게 박아 주는" 역할이다.
> "용어가 흔들리면 코드도 흔들리거든요. 그걸 잡아주는 도구라고 보시면 돼요." [00:19]

화자는 이를 위해 **그릴미(grill-me) 스킬**을 사용해 도메인 문서화를 처리한다고 밝힌다.

### [00:22–00:37] 커스텀 DDD 얹기 + 세 도구 결 소개
화자는 여기서 한 발 더 나아가 **커스텀으로 DDD 아키텍처링을 얹어** 도메인 드리븐(domain-driven)을 더 강하게 가져간다. 용어 정리로 끝나는 게 아니라 그 도메인 모델이 폴더 구조·코드 경계까지 이어지게 하는 것. 이어서 G스택은 결이 달라 **브레인스토밍, 특히 제품 스펙 작성**에 강하고 스펙에서 뭘 빼고 살릴지 깊게 캐묻는다고 예고하고, 슈퍼파워즈는 **실제 구현 단계** 도구로 브레인스토밍과 서브에이전트 드리븐 디벨롭먼트를 코드 작성 단계에서 쓴다고 소개한다.

### [00:35–00:59] 역할 분담 정리
셋을 어떻게 합치냐에 대해 역할이 자연스럽게 나뉜다고 정리한다: **도메인 DDD는 맥퍼커, 스펙 브레인스토밍은 G스택, 실제 구현은 슈퍼파워즈.**

### [00:59–01:13] 맥퍼커 스킬 규모와 목적
맥퍼커 스킬은 약 16개 정도인데 주된 목적은 하나, **도메인 맵핑을 통해 정확한 용어를 쓰도록 유도하는 것**이라고 다시 강조한다.

### [01:14–01:42] 도메인 맵핑이란 + "주문" 예시
도메인 맵핑은 서비스에 등장하는 단어·개념·관계를 AI에게 정확히 알려 주는 것이다. 예로, "주문"이라는 단어도 일반 쇼핑몰 주문·음식 배달 주문·B2B 발주가 다 다른데, 그냥 "주문"으로 던지면 AI가 헷갈린다. 맥퍼커 스킬이 이걸 계속 캐묻고 정리해 **도메인 파일**로 만든다.
> "신입 직원 들어왔을 때 회사 용어집 줘야 되잖아요. … 맥퍼커가 그 용어집을 자동으로 만들어 주는 도구라고 보시면 돼요." [01:32–01:42]

### [01:42–02:07] 커스텀 DDD — 용어를 코드 구조로
화자는 여기서 한 단계 더 가, 맥퍼커가 뽑은 도메인 용어를 사전으로만 쓰지 않고 **실제 코드 구조에 그대로 반영**되게 했다. 도메인 모델이 폴더 경계·모듈 경계·함수 시그니처까지 이어져, "주문" 도메인은 주문 모듈 안에서 끝나고 "결제" 도메인은 결제 모듈로 분류된다. 용어가 단순한 약속이 아니라 **코드 아키텍처를 결정하는 1차 자료**가 되는 것.

### [02:07–02:15] DDD의 결정적 효과 [의견]
[의견] 이게 나중에 결정적이라고 평가한다 — AI가 같은 단어를 같은 뜻으로 알아먹는 데서 끝나지 않고, **도메인 경계를 따라 코드를 짜니 모듈이 섞이지 않는다**는 것.

### [02:15–02:47] G스택 — 스펙 브레인스토밍의 결
G스택의 진짜 강점은 브레인스토밍, 그중에서도 제품 스펙 작성이다. "그냥 아이디어 같이 생각해 줘" 식이 아니라, 스펙에서 뭘 피하고 뭘 살려야 하는지 심도 깊고 밀도 있게 캐묻는다. "이 기능 넣을게요" 하면 그냥 통과시키지 않고 "이거 핵심 가치에 붙는 기능이야? 누가 써? 없으면 어떻게 돼? 이미 비슷한 걸 쓰는 사람 있어?" 식으로 한 줄 한 줄을 다 흔든다.
> "스펙에서 뭘 빼야 하는지가 진짜 중요하거든요. 보통 빼는 게 더 어려워요. 다 넣고 싶으니까요. G스택은 거기서 안 봐줘요." [02:41–02:45]

### [02:47–03:07] 오피스아워 스킬 — 여섯 강제 질문
대표적인 게 **오피스아워(office hours) 스킬**로, YC 오피스아워(창업자가 멘토에게 깨지는 자리)를 그대로 옮긴 것이다. **여섯 가지 강제 질문**이 있다: 누가 정말 이걸 원해? / 지금 이걸 어떻게 해결하고 있어? / 이게 진짜 절박한 문제야? / 가장 작은 단위가 뭐야? / 직접 본 적 있어? / 미래가 보여? [의견] 이 질문 흐름이 잘 짜여 있어 한 번 거치면 스펙에서 핵심과 노이즈가 먼저 확 갈린다.

### [03:07–03:15] 후속 리뷰 스킬
답변을 받은 다음에도 끝이 아니라 **플랜 CEO 리뷰·플랜 리뷰** 같은 후속 스킬로 넘어가 스펙을 다시 다듬는다. 브레인스토밍 단계에서 뽑아낼 수 있는 걸 끝까지 뽑아내는 구조.

### [03:15–03:43] 슈퍼파워즈 — 구현 엔진, 세 가지 주력
슈퍼파워즈는 실제 구현 단계에서 쓰는 엔진이다. 화자는 세 가지를 메인으로 쓴다: **브레인스토밍, 라이트 라이팅 플랜, 서브에이전트 드리븐 디벨롭먼트.** 먼저 브레인스토밍은 보통 G스택 브레인스토밍 결과를 받아 슈퍼파워즈 브레인스토밍을 한 번 더 돌리는데, "이걸 코드로 어떻게 풀 거야"에 대한 **상세 스펙을 확정**하는 용도로 많이 쓴다.

### [03:43–04:06] 라이팅 플랜 + 플랜 리뷰 에이전트
다음은 라이팅 플랜. 플랜을 쓴 뒤 단순히 끝내는 게 아니라 **커스텀 리뷰 에이전트로 플랜 리뷰**를 돌리고, 리뷰 결과에 따라 플랜을 다시 수정한다.

### [04:06–04:33] 서브에이전트 드리븐 디벨롭먼트
그다음이 서브에이전트 드리븐 디벨롭먼트로, 실제 구현 단계에서 이걸로 코드를 짠다. 큰 작업을 여러 개로 쪼갠 뒤 각각을 별도 에이전트에 맡기고, 작업이 끝나면 다른 에이전트가 검수해 **통과해야 다음 단계로 넘어간다.**
> "큰 공사 할 때 한 사람이 다 하는 게 아니잖아요. 전기 따로, 배관 따로, 도배 따로 … 그것처럼 코드도 작업 단위로 쪼개서 각 서브에이전트가 처리해요." [04:17–04:25]

[의견] 이 흐름이 실제 구현 속도를 많이 올려 준다.

### [04:33–04:53] UI·코드 작업 단위 쪼개기
UI를 만들 때도 같은 방식으로, 슈퍼파워즈가 **목업 이미지를 함께 보며** "이 버튼은 여기, 이 색깔은 이렇게, 이 동작은 이렇게" 구현 가이드를 뽑아 서브에이전트에 분배한다. 코드도 스펙 한 줄 던지고 끝이 아니라 "파일 어디, 어떤 함수, 입력은 뭐고 출력은 뭐고 검증은 어떻게" **2~5분짜리 작업 단위**까지 쪼개 서브에이전트가 가져간다.

### [04:53–05:12] 전체 흐름 개요 + 소요시간
도구 세 개를 소개했으니 지금부터가 진짜라며, 실제로 어떻게 섞어 쓰는지 전체 흐름을 먼저 보여 준다: **그릴미로 깊은 인터뷰 → 도메인 파일 작성 → 스펙·플랜·플랜 리뷰 → 서브에이전트 코드 작성 → 코덱스 리뷰 → 임프로브 코드베이스 아키텍처로 마무리.** 이걸 한 번 돌리면 **두세 시간** 걸리지만 그만큼 결과물이 다르다고 말한다(주장/미검증).

### [05:12–05:48] 1단계 — 그릴미 인터뷰로 도메인 파일 만들기
처음엔 무조건 맥퍼커+그릴미로 시작한다. "이런 거 만들고 싶어" 한 줄 던지고 클로드에게 인터뷰를 요청하면 **30분, 길면 한 시간 인터뷰**가 시작된다. 여기서 적당히 답하면 안 되고 깊게 답해야 하는데, 이 인터뷰 결과가 그대로 도메인 파일이 되기 때문이다. 도메인 파일은 AI에게 주는 사전 같은 것으로, **CONTEXT.md에 박아두면** 이후 모든 작업에서 클로드가 이 파일을 보며 작업한다.
> "새 직원 들어왔을 때 회사 위키 안 읽혀 주면 계속 '이 용어 뭐예요?' 물어보잖아요. 근데 위키 한번 정독시키면 다음부터 알아서 이래요. 도메인 파일이 딱 그 역할이에요." [05:38–05:46]

### [05:48–06:14] 2단계 — G스택 오피스아워로 비즈니스 검증
도메인 파일을 다 만들었으면 G스택으로 넘어가 오피스아워를 돌린다. [의견] 그릴미가 "어떻게 만들지"를 깊게 판 것이라면, 오피스아워는 "이걸 진짜 만들어야 해?"를 묻는 것으로 질문 결이 다르다 — 그릴미는 도메인 깊이로, 오피스는 비즈니스 임팩트로 빠진다. 두 개를 차례로 거치면 기술적으로 깊이 이해됐고 비즈니스적으로 만들 이유가 있는 두 가지가 동시에 잡혀, 그제서야 진짜 만들 게 명확해진다. 이 과정 자체가 G스택에서 말하는 브레인스토밍이다.

### [06:14–06:22] 아직 코드 0줄 — 그게 핵심 [의견]
여기까지 1~2단계에 약 한 시간이 지났고 코드는 한 줄도 안 짰다. [의견] 근데 이게 핵심으로, **바로 코드부터 짜기 시작하면 나중에 헛쓴 게 드러난다.**

### [06:22–06:56] 3단계 — 라이트 스펙·플랜·플랜 리뷰
이제 본격적으로 슈퍼파워즈로 넘어가 코드를 짜는데, 한 번에 풀 스펙을 만들지 않고 **라이트 스펙**을 핵심 기능 위주로 가볍게 먼저 짠다. 그다음 **라이트 플랜**으로 구현을 단계별로 쪼갠다. 여기서 중요한 건 **플랜 리뷰 커스텀 에이전트**(에이전트 리뷰 기능 활용) — 화자가 짠 플랜을 보고 "여기 2단계 빼먹었어, 이거 순서 바꿔야 돼, 이건 너무 큰 작업이야" 식으로 빨간 펜으로 수정해 준다.
> "시니어한테 제 작업 계획서 들고 가서 검토받는 거랑 똑같아요. 혼자서 짠 플랜에는 항상 빈틈이 있거든요. … 그게 절반 이상 잡히는 거 같습니다." [06:47–06:56]

### [06:56–07:16] 4단계 — 서브에이전트로 코드 작성
플랜이 정해지면 슈퍼파워즈의 서브에이전트 드리븐 디벨롭먼트를 돌린다. 플랜의 작업들을 클로드가 알아서 서브에이전트에 분배하고, 백그라운드에서 몇 개씩 동시에 돌아가기도 한다. 화자는 보통 **두 시간 정도** 걸리며 그동안 영상 편집·커피 같은 다른 일을 하다가 단계가 끝날 때마다 알림을 받고 검수한다고 말한다.

### [07:16–07:43] 5단계 — 코덱스 교차 리뷰
다만 다 맡겨 놓기만 하면 안 되고 **중간 검수가 핵심**이다. 화자는 클로드 코드 안에서 **코덱스 플러그인**을 함께 쓴다. [의견] GPT 코덱스와 클로드가 다른 모델이라, 한쪽이 짠 코드를 다른 쪽이 보면 사각지대가 잡히고 "이거 이상한데" 하고 짚어 주는 게 많다. 요즘 드는 기분으로는 세세한 구현에서는 코덱스가 훨씬 잘하는 것 같고, **대다수 버그가 이 단계에서 잡힌다**고 말한다(주장/미검증).

### [07:43–08:17] 6단계 — 임프로브 코드베이스 아키텍처
여기까지 오면 코드는 거의 완성이지만 마지막 한 끝이 남는다. 이때 맥퍼커의 **임프로브 코드베이스 아키텍처** 스킬을 돌린다. 이미 짜 놓은 코드를 보고, 앞에서 만든 CONTEXT.md 도메인 파일을 기준으로 "이 함수는 도메인 개념과 어긋나 이렇게 바꾸자, 이 둘은 사실 같은 개념인데 떨어져 있어 합치자" 식 리팩토링 포인트를 도메인 언어 기반으로 잡아 준다. [의견] 이 스킬 전후로 코드 품질이 진짜 다르다 — 그냥 동작하는 코드에서 도메인 모델이 잘 잡힌 코드로 바뀌고, 읽으면 도메인 구조가 한눈에 들어와 나중에 다른 사람이 들어와도 빨리 적응한다(주장/미검증).

### [08:17–08:45] 추가 — 스킬 기반 리팩토링
추가로 화자는 프런트엔드는 **React·Vercel 베스트 프랙티스** 같은 스킬을, 백엔드는 **커스텀으로 만든 FastAPI 백엔드 가이드라인** 스킬을 이용해 잘못 짜인 코드를 알맞게 리팩토링한다. 대표적으로 라우트 경로에 있는 서비스 로직을 서비스 계층으로 옮기는 리팩토링을 계속 유도하고 있다.

### [08:45–09:18] 전체 정리 (도구 + 워크플로우)
정리하면 도구 세 개의 역할이 명확하다: 맥퍼커는 도메인 맵핑에 DDD 아키텍처링을 얹어 코드 경계까지 잡고, G스택은 제품 스펙 브레인스토밍으로 뭘 빼고 살릴지 밀도 있게 다듬고, 슈퍼파워즈는 실제 구현 브레인스토밍·서브에이전트 드리븐으로 코드를 짠다. 워크플로우는 그릴미 인터뷰 → 도메인 파일 작성 → 결정 잡기 → 라이트 스펙 → 플랜 → 플랜 리뷰 검수 → 서브에이전트 코드 작성 → 코덱스 리뷰 → 임프로브 코드베이스 아키텍처 마무리 → 추가 스킬 기반 리팩토링. **두세 시간, 그 이상** 걸린다(주장/미검증).

### [09:18–09:37] 길지만 결과가 다르다 [의견]
[의견] 길게 느껴지지만 한번 결과물을 받아 보면 생각이 많이 달라질 것이라며, 코드 한 줄 한 줄이 도메인 개념과 맞물려 있어 나중에 손볼 때도 어디를 손대야 할지 보인다고 말한다.
> "결정적으로 이 코드 같은 경우에는 사람이 나중에 수작업으로 버그를 고칠 수가 있는 상태가 됩니다." [09:27–09:33]

### [09:37–10:01] 마무리 + 다음 영상 예고
여러분도 한번 시도해 보길 권하며(처음엔 복잡해 보여도 한두 번 돌리면 왜 이렇게 짜였는지 감이 온다), 어떤 워크플로우를 쓰는지 댓글로 남겨 달라고 한다. 다음 영상에서는 이번에 소개한 워크플로우의 **코드베이스를 소개**하고 사용법 가이드를 올리며, **코덱스 플러그인 셋업하는 법**도 다뤄 보겠다고 예고한다.
