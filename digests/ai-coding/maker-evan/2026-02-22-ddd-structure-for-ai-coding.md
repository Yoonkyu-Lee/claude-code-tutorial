# AI 잘 쓰는 팀 vs 망하는 팀, 딱 하나 차이입니다

> maker-evan · 2026-02-22 · [영상 링크](https://youtu.be/7GjRM2uv-6E)
> 정리 방식: 시간순 심층 워크스루 (paraphrase) + 주제별 상세 요약

## 요약 (주제별)

### 핵심 주장 — 같은 AI, 갈리는 결과의 원인은 "코드 구조"
- 같은 AI를 써도 어떤 팀은 거의 완벽한 코드를 뽑고, 어떤 팀은 AI가 만든 코드를 다시 AI에게 고쳐 달라는 악순환에 빠진다.
- 화자는 그 차이가 "코드를 담아두는 구조"에 있다고 주장하며, AI가 잘 작동하는 구조로 DDD(도메인 주도 설계, Domain-Driven Design)를 제시한다.
- 화자 경험담: AI에게 백엔드를 만들게 했더니 일주일 뒤 기능 하나 추가하려는데 아무도 손댈 수 없는 코드가 됐고, AI조차 "이 코드는 어떤 구조인지 파악하기 어렵습니다"라고 답했다. (주장/미검증)

### 문제 정의 — 구조 없는 바이브 코딩의 붕괴
- 바이브 코딩 = 개발자가 코드를 직접 쓰는 대신 AI에게 말로 설명하면 AI가 대신 짜 주는 방식. 개발자는 방향만 잡는다.
- 처음엔 신기하지만 "회원가입 만들어 줘 → 결제 추가해 줘 → 둘을 연결해 줘"처럼 쌓이면 AI가 이상한 걸 만들기 시작한다.
- 원인: AI는 정해진 틀이 없으면 매번 조금씩 다른 방식으로 코드를 만든다. 구조 없이 쓰면 "열 줄 중 여섯 줄이 나중에 문제가 되는 코드"가 된다. (주장/미검증)

### DDD가 무엇인가 — 편의점 비유
- DDD = 비즈니스 개념을 코드로 그대로 표현하는 것. 편의점에서 음료는 음료 코너, 과자는 과자 코너에 두듯, 유저·결제·주문·배송 같은 실제 비즈니스 개념(도메인) 단위로 코드를 나눈다.
- 음료 코너에 칫솔이 있으면 손님도 알바생도 헷갈리듯, 코드도 개념대로 정리돼야 한다.
- 각 도메인 안에는 역할이 정해진 기본 3요소가 있다: (1) 개념이 어떻게 생겼는지 정의하는 파일, (2) DB에서 꺼내오고 저장하는 부분, (3) 실제 비즈니스 규칙을 처리하는 부분.
- 핵심: DDD는 단순한 폴더 나누기가 아니라, 비즈니스 사람들이 쓰는 말과 코드 이름을 완전히 일치시키는 것. 영업팀이 "결제 취소"라고 하면 코드에도 같은 이름이 있어야 한다(다만 보통 영어로). 

### 왜 AI에게 유리한가 — 패턴 인식
- 화자 주장: AI가 코드를 잘 짜는 비결은 창의력이 아니라 패턴 인식이다.
- 비유: 새 회사 입사 시 선임이 "우리 팀은 항상 이 양식으로 보고서 써"라며 기존 보고서 10개를 주면, 다음 보고서도 같은 양식으로 쓸 수 있다. AI도 유저 도메인 생김새를 보면 결제 도메인도 같은 패턴으로 자동 인식·복제한다.
- 화자 경험: DDD로 구조를 잡으면 AI가 기존 패턴을 보고 새 기능을 거의 완벽하게 찍어냈고, 같은 패턴이라 코드 검토·확인 포인트도 줄었다. 반대로 구조 없이 쓰면 같은 역할 코드가 여기저기 흩어져, 고칠 때 다 찾아 고쳐야 했다. (주장/미검증) [의견]

### 컨텍스트 관리 — AI에게 필요한 것만 보여주기
- AI도 사람처럼 한 번에 너무 많은 정보를 주면 품질이 뚝 떨어진다. 전체 코드를 다 보여주며 "결제 버그 고쳐 줘" 하면 엉뚱한 파일을 건드리거나 중복 코드를 만든다. 처음과 끝은 잘 기억하고 중간은 흐릿하게 처리하는 경향. [의견]
- DDD면 자연스레 해결: "유저 문제야" → 유저 도메인만, "결제 오류야" → 결제 도메인만 보여주면 된다. AI가 볼 코드가 필요한 것만 남아 항상 집중 상태.
- 비유: 다리 아픈데 머리 엑스레이·심장 초음파를 다 주면 의사도 헷갈린다. 다리 자료만 주면 된다.

### 이름 일치와 환각 방지
- AI에게 하는 말과 코드가 같은 언어를 써야 한다. "유저 쪽 프로필 정보 가져와 줘" 했을 때 유저 폴더 안 파일이라고 바로 안다 — 말과 코드 이름이 일치하기 때문.
- 반대로 "데이터 처리", "관리자"처럼 이름이 모호하면 AI가 어디가 그 역할인지 몰라 추측해 만들어낸다 — 그게 환각.
- 화자 경험: 코드 이름을 대충 지었더니 AI가 엉뚱한 파일에 코드를 만들었고, 비즈니스 언어로 이름을 통일한 뒤 환각이 확 줄었다. (주장/미검증)

### 도구·자동화 — Claude Code 훅 + 도메인 트리거 + 스킬
- Claude Code에는 훅(hook) 시스템이 있어 특정 조건이 맞으면 AI가 자동으로 어떤 행동을 하게 미리 설정할 수 있다.
- 여기에 도메인 트리거를 달아 스킬을 호출한다. 스킬 안에는 각 도메인별로 읽어야 할 파일이 분류돼 있어 그에 맞춰 행동한다.
- 예: "결제 환불 기능 만들어 줘" → AI가 알아서 결제 도메인 파일들을 먼저 읽어 기존 개념 표현·데이터 저장 방식·비즈니스 규칙을 파악한 뒤 코드를 만든다.
- 단, DDD 구조가 잡혀 있어야 작동한다. 구조 없이 코드가 흩어져 있으면 트리거를 달아도 AI가 어디를 읽어야 할지, 결제 맥락을 파악할지 모른다. 도메인 경계가 명확해야 "결제 폴더가 딱 떨어진다".
- 화자가 가장 놀란 부분: 구조를 잡고 트리거를 다니 AI가 자기 코드 스타일을 학습한 것처럼 읽을 코드를 알아서 뽑아냈고, "이 도메인에서는 이렇게 해"라고 매번 설명할 필요가 없어졌다. (주장/미검증) [의견]

### 실전 절차 — 두 부류의 사용자에게
- (1) 개발자에게 작업을 맡기는 사람: "DDD 구조로 짜 주세요, 비즈니스 개념 단위로 폴더 나눠 주세요"라고 요청하면 나중에 AI 도구로 유지보수할 때 훨씬 효율적이고 "이 코드 어디 있어요" 질문이 사라진다(폴더 이름이 곧 비즈니스 개념이라서).
- (2) 직접 바이브 코딩하는 사람: 처음 베이스 코드, 특히 첫 번째 도메인 하나는 손으로 직접 짜라고 강력 권고. AI에게 구조를 맡길 수도 있지만, 개념 정의·데이터 꺼내는 방식을 직접 써 봐야 나중에 AI 코드의 맞고 틀림을 판단할 수 있다. 이게 없으면 이상한 코드도 뭐가 이상한지 모른 채 쓰게 된다.
- 첫 도메인을 직접 짜고 "이 패턴으로 다른 도메인도 만들어 줘" 하면 훨씬 정확히 동작하고, 구조를 유지하며 기능을 더할수록 AI가 점점 더 잘 따라온다. 처음 세팅이 중요.

### 4줄 핵심 정리 (화자 마무리)
- 하나: 정해진 틀이 있으면 AI가 틀을 복사해 일관된 코드를 만든다 — 구조가 패턴이 된다.
- 둘: 도메인 단위로 나누면 AI가 필요한 도메인만 집중해서 본다 — 볼 코드가 작을수록 품질이 오른다.
- 셋: 코드 이름이 실제 비즈니스 언어와 같으면 AI가 어디를 수정할지 바로 찾는다 — 이름이 안 맞으면 추측하고 환각이 생긴다.
- 넷: 훅에 도메인 트리거를 달면 AI가 알아서 관련 코드를 읽고 맥락을 파악한다 — 매번 설명 안 해도 된다.

### 세일즈 구조
- 영상 초반과 말미에 구독자 이벤트를 예고·진행한다.
- 화자가 DDD로 짠 코드 베이스를 제공하며, 시청자는 해당 레포지토리를 가져다 쓰면 된다고 안내. 구독·좋아요·댓글·알림 설정 후 고정 댓글의 가이드를 따르면 코드 베이스 링크가 올라간다는 구조.

## 3줄 오리엔테이션
- 같은 AI를 써도 팀별 결과가 갈리는 이유를 "코드 구조" 하나로 지목하고, 그 해법으로 DDD(도메인 주도 설계)를 비개발자도 이해하도록 편의점 비유로 푼다.
- 핵심 논리: AI는 창의력이 아니라 패턴 인식으로 코드를 짜므로, 비즈니스 개념 단위 폴더 + 비즈니스 언어와 일치하는 이름 + 도메인별 컨텍스트 격리가 AI의 정확도를 끌어올린다는 것.
- Claude Code 훅에 도메인 트리거를 달아 스킬을 호출하는 자동화, 첫 도메인은 직접 짜라는 실전 조언, 그리고 구독자 이벤트 + 코드 베이스 배포 세일즈로 마무리한다.

## 전문 — 시간순 워크스루

### [00:00–00:14] 도입 — AI가 만든 손댈 수 없는 코드
화자는 AI에게 백엔드를 만들어 달라고 했더니 진짜로 만들어 줬지만, 일주일 뒤 기능 하나를 추가하려 보니 아무도 손댈 수 없는 코드가 돼 있었다고 말한다. AI에게 다시 물어봐도 "이 코드는 어떤 구조인지 파악하기 어렵습니다"라는 답이 돌아왔다는 경험담으로 문을 연다. (주장/미검증)

### [00:14–00:27] 문제 제기 — 같은 AI, 갈리는 결과
같은 AI를 써도 어떤 팀은 AI가 거의 완벽한 코드를 뽑아내고, 어떤 팀은 AI가 만든 코드를 또 AI에게 고쳐 달라고 하는 악순환에 빠진다고 지적한다.
> "차이가 뭔지 아세요? 코드를 담아두는 구조예요." [00:23]
AI가 잘 작동하는 구조가 있다는 것이 오늘의 주제다.

### [00:27–00:41] 오늘의 답 = DDD 예고 + 이벤트 안내
화자는 그 구조가 왜 "도메인 드리븐 DDD"인지 개발을 모르는 사람도 완전히 이해하도록 풀어주겠다고 예고한다. 이어 구독자 이벤트를 진행할 예정이니 영상을 끝까지 집중해 달라고 요청한다(세일즈 장치, 1차 언급).

### [00:41–01:07] 바이브 코딩이란 & 구조 없을 때의 붕괴
바이브 코딩을 설명한다: 예전엔 개발자가 코드를 한 줄씩 직접 썼지만, 지금은 AI에게 말로 설명하면 AI가 대신 짜 준다. 개발자는 "이런 기능 필요해"라는 방향만 잡으면 된다. 그러나 실제로 써 보면 처음엔 신기해도, "회원가입 만들어 줘 → 결제 기능 추가해 줘 → 이 두 개 연결해 줘"로 가면 AI가 뭔가 이상한 걸 만들기 시작한다고 한다.

### [01:06–01:20] 원인 — 정해진 틀의 부재
왜 이런 일이 생기냐면, AI는 정해진 틀이 없으면 매번 조금씩 다른 방식으로 코드를 만들기 때문이다.
> "코드 구조 없이 AI를 그냥 쓰면 열 줄 중에 여섯 줄이 나중에 문제가 되는 코드가 돼요." [01:12] (주장/미검증)
그 틀을 잡아 주는 것이 오늘 얘기할 DDD다.

### [01:20–01:39] DDD 비유 — 편의점 진열
DDD(도메인 주도 설계)라는 이름은 어렵게 들리지만, 화자는 편의점 비유로 설명한다. 음료는 음료 코너, 과자는 과자 코너, 도시락은 도시락 코너에 있다. 만약 음료 코너에 칫솔이 있으면 손님도 찾기 힘들고 알바생도 어디 진열할지 헷갈린다. 코드도 똑같다는 것.

### [01:39–01:58] 도메인과 내부 3요소
DDD는 비즈니스 개념을 코드로 그대로 표현하는 것이다. 유저·결제·주문·배송처럼 비즈니스에서 실제로 쓰는 개념이 곧 도메인이다. 각 도메인 안에는 역할이 정해진 세 가지 기본 요소가 있다고 설명한다: 이 개념이 어떻게 생겼는지 정의하는 파일, 데이터베이스에서 꺼내오고 저장하는 부분, 실제 비즈니스 규칙을 처리하는 부분.

### [01:58–02:19] 핵심 원칙 — 말과 코드 이름의 일치
DDD는 단순히 폴더를 나누는 게 아니라, 비즈니스 사람들이 쓰는 말과 코드 이름을 완전히 일치시키는 것이라고 강조한다. 영업팀이 "결제 취소"라는 용어를 쓰면 코드에도 똑같이 "결제"라는 이름이 그대로 있어야 한다(다만 이때는 보통 영어를 쓴다). 이 단순한 원칙이 AI 입장에서는 완전히 다른 세상이라고 말한다.

### [02:19–02:37] AI의 비결은 패턴 인식
> "AI가 코드를 잘 짜는 비결이 뭔지 아세요? 창의력이 아닙니다. 패턴 인식이에요." [02:24] [의견]
비유: 새 회사에 입사했는데 선임이 "우리 팀은 항상 이 양식으로 보고서 써"라며 기존 보고서 10개를 주면, 다음 보고서도 같은 양식으로 쓸 수 있고 뭘 물어볼 필요도 없다.

### [02:37–02:52] 도메인 패턴 복제
AI도 똑같다. DDD 구조로 코드를 짜면 유저 도메인이 어떻게 생겼는지 본 AI가 결제 도메인도 "아, 같은 패턴이구나" 하고 자동 인식한다. "결제 기능 만들어 줘" 한마디에 유저 도메인과 똑같이 생긴 결제 도메인을 그대로 복제해서 만들어 준다.

### [02:52–03:14] 화자 경험 — 구조 있음 vs 없음
직접 해 보니 이게 진짜였다며, DDD로 구조를 잡아두면 AI가 기존 패턴을 보고 새 기능을 거의 완벽하게 찍어냈고 코드 검토도 편해졌으며 확인할 포인트도 줄었다고 한다. 반대로 구조 없이 썼을 때는 초반엔 괜찮다가 어느 순간 같은 역할을 하는 코드가 여기저기 흩어져 있었고, AI가 참고할 패턴이 없어 매번 조금씩 다르게 만든 탓에 나중에 뭔가 고치려면 다 찾아 다 고쳐야 했다. (주장/미검증) [의견]

### [03:14–03:35] AI의 컨텍스트 과부하
재밌는 점 하나: AI도 사람과 비슷하게 한 번에 너무 많은 정보를 주면 품질이 뚝 떨어진다. 전체 코드를 다 보여주며 "결제 버그 고쳐 줘" 하면 AI가 엉뚱한 파일을 건드리거나 이미 있는 코드를 중복으로 만든다. 처음과 끝은 잘 기억하는데 중간 내용을 흐릿하게 처리하는 것 같다고 관찰한다. [의견]

### [03:35–03:57] DDD의 컨텍스트 격리 효과
DDD 구조면 이게 자연스럽게 해결된다. "유저 관련 문제 있어" 하면 유저 도메인만, "결제 오류야" 하면 결제 도메인만 보여주면 된다. AI가 봐야 할 코드가 딱 필요한 것만 남아 항상 집중할 수 있는 상태에서 코드를 만든다. 비유: 다리가 아픈데 머리 엑스레이·심장 초음파를 다 갖다 주면 의사도 헷갈리듯, 다리 관련 자료만 가져가면 된다.

### [03:57–04:29] 말과 코드가 같은 언어 → 환각 방지
AI에게 하는 말과 코드가 같은 언어를 써야 한다. "유저 쪽에서 프로필 정보 가져와 줘" 하면 AI는 유저 폴더 안의 파일을 바로 안다 — 말과 코드 이름이 완전히 일치하기 때문. 반대로 코드가 "데이터 처리"라는 파일에 처리 기능, "관리자"라는 파일에 관리 기능 식으로 돼 있으면, "유저 정보 가져와 줘" 했을 때 데이터 처리가 그 역할인지 관리자가 그 역할인지 AI도 모른다. 그러면 AI가 추측해서 만들어내는데, 그게 환각이다.

### [04:29–04:49] 이름 통일의 실전 효과
> "DDD의 핵심은 바로 이거예요. 비즈니스에서 쓰는 말이랑 코드 이름을 맞추는 거예요." [04:29]
어렵게 말했지만 그냥 현장에서 쓰는 말을 코드 이름으로 쓰라는 것. 화자도 처음엔 코드 이름을 대충 지었더니 AI가 자꾸 엉뚱한 파일에 코드를 만들었고, 비즈니스 언어로 이름을 통일한 뒤부터는 AI가 "아 이거 여기 있는 거네" 하고 바로 찾아 환각이 확 줄었다고 한다. (주장/미검증)

### [04:49–05:11] Claude Code 훅 + 도메인 트리거 + 스킬
DDD의 진짜 위력을 완성시키는 부분을 소개한다. Claude Code에는 훅 시스템이 있어, 특정 조건이 맞으면 AI가 자동으로 어떤 행동을 하도록 미리 설정할 수 있다. 여기에 도메인 트리거를 달아 스킬을 불러 사용하면 된다. 스킬 안에는 각 도메인별로 읽어야 하는 파일들이 분류돼 있고 그에 맞춰 행동한다.

### [05:11–05:35] 트리거 작동 & DDD 전제
예를 들어 "결제 환불 기능 만들어 줘" 하면 AI가 알아서 결제 도메인 파일들을 먼저 쭉 읽어, 기존에 결제 개념을 어떻게 표현하는지·데이터는 어떻게 저장하는지·어떤 비즈니스 규칙이 있는지 다 파악한 다음 코드를 만든다. 단, 이건 DDD 구조가 잡혀 있어야 제대로 작동한다. DDD 없이 코드가 흩어져 있으면 트리거를 달아도 AI가 어디를 읽어야 할지, 결제 맥락을 파악할 수 있는지도 모른다. 도메인 경계가 명확하니 "결제 폴더가 딱 떨어지는" 것이다.

### [05:35–05:51] 화자가 가장 놀란 지점
직접 써 보면서 가장 놀란 게 이 부분이었다며, 구조를 잡고 트리거를 달아두니 AI가 자기 코드 스타일을 학습한 것처럼 읽을 코드를 알아서 뽑아냈고, "이 도메인에서는 이렇게 해"라고 매번 설명할 필요가 없어졌다고 한다. (주장/미검증) [의견]

### [05:51–06:11] 비개발자에게 의미 (1) — 개발자에게 맡기는 경우
"저는 개발자도 아닌데 이게 무슨 의미냐"는 사람을 위해 두 상황을 나눈다. 첫째, 개발자에게 작업을 맡기는 사람은 "DDD 구조로 짜 주세요, 비즈니스 개념 단위로 폴더 나눠 주세요"라고 요청할 수 있다. 이 구조로 짜면 나중에 AI 도구로 유지보수할 때 훨씬 효율적이고, "이 코드 어디 있어요"라는 질문도 사라진다 — 비즈니스 개념이 곧 폴더 이름이기 때문.

### [06:11–06:37] 비개발자에게 의미 (2) — 직접 바이브 코딩
둘째, 직접 바이브 코딩을 시작하는 사람. 여기서 중요한 것은 처음 베이스 코드는 손으로 직접 짜는 게 좋다는 것. AI에게 구조를 잡아 달라고 하면 만들어 주긴 하지만, 첫 번째 도메인 하나는 직접 짜기를 강력 권한다. 어떤 개념을 어떻게 정의하고 데이터는 어떻게 꺼내오는지 손으로 한 번 써 봐야 나중에 AI 코드의 맞고 틀림을 판단할 수 있다. 이게 없으면 AI가 이상한 코드를 만들어도 뭐가 이상한지 모르고 그냥 쓰게 된다.

### [06:37–06:55] 첫 세팅의 중요성 + 코드 베이스 제공
첫 도메인 하나를 직접 짜고 AI에게 "이 패턴으로 다른 도메인도 만들어 줘" 하면 훨씬 정확하게 동작하며, 이 구조를 유지하며 기능을 추가할수록 AI가 점점 더 잘 따라온다. 처음 세팅이 중요하다. 다만 화자가 이미 짜 놓은 코드 베이스를 쓰면 되므로, 시청자는 해당 레포지토리를 가져다 쓰면 된다고 안내한다(코드 안내 사항은 곧 안내 예정 — 세일즈 장치).

### [06:55–07:24] 4줄 핵심 정리
마지막으로 오늘 핵심 네 가지를 정리한다. 하나, 정해진 틀이 있으면 AI가 틀을 복사해 일관된 코드를 만든다 — 구조가 패턴이 된다. 둘, 도메인 단위로 나누면 AI가 필요한 도메인만 집중해서 볼 수 있고, 볼 코드가 작아질수록 품질이 올라간다. 셋, 코드 이름이 실제 비즈니스 언어와 같으면 AI가 어디를 수정할지 바로 찾고, 이름이 맞지 않으면 추측을 하며 환각이 생긴다. 넷, 훅에 도메인 트리거를 달아두면 AI가 알아서 관련 코드를 읽고 맥락을 파악해 매번 설명 안 해도 된다.

### [07:24–07:47] 마무리 비유 + 구독자 이벤트
어렵게 생각할 필요 없이, 편의점 진열처럼 비즈니스 개념 단위로 나눠 같은 종류는 같은 곳에 두고, 진열마다 "여기엔 이것"이라는 표시를 해 두는 것 — 그게 DDD다. AI가 여러분 코드를 처음 봤을 때 "이 팀은 이렇게 일하는구나"를 바로 파악하도록 하는 것이 목적. 마지막으로 구독자 이벤트를 진행한다며, 구독·좋아요·댓글·알림 설정을 하고 고정 댓글을 확인해 가이드를 따르면 DDD로 짠 화자의 코드 베이스가 올라갈 링크가 있을 것이라고 안내하며 끝맺는다(세일즈 구조).
