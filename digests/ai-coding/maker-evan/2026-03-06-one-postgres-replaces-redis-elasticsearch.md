# Redis, Elasticsearch 다 버렸습니다. 결과는?

> 검증 배너: 벤치마크·비용 수치는 화자 주장으로, 이 정리 내에서 독립 검증되지 않음 (주장/미검증).

## 요약 (주제별)

### 핵심 메시지
- 복잡성은 "필요할 때 추가"하는 것이지 "미래를 대비해 미리 사는" 것이 아니다. 지금 당장은 단순하게 해결할 수 있는 아키텍처가 좋은 아키텍처이고, 화자에게 그것은 PostgreSQL 하나다.
- 충분히 커지기 전까지는 단순한 쪽이 훨씬 강하다는 것이 화자의 반복 주장.

### 화자의 경험 근거 (세 회사)
- 두 번째 회사: 검색=Elasticsearch, 메인 데이터=PostgreSQL, 비정형 데이터=MongoDB 구조. 이론상 "각자 잘하는 걸 맡긴다"라 완벽해 보였으나 운영은 악몽이었다는 회고. [의견]
- 겪은 문제: (1) 비용 — Elasticsearch 클러스터 유지에 한 달 수백만 원, MongoDB 별도 비용 (주장/미검증), (2) 인력 — 세 DB를 관리할 사람이 필요해 인력이 거의 두세 배 (주장/미검증), (3) DB 간 데이터 동기화 문제로 Kafka까지 도입, 비용 몇 천만 원 지출 (주장/미검증), (4) 장애 시 세 DB 중 원인을 다 뒤져야 함 — "DB 관리하느라 정작 제품은 못 만드는 느낌". [의견]
- 세 번째 회사: 규모 있는 서비스인데 PostgreSQL 하나만 사용. 처음엔 "이게 맞아?" 의심했으나, 인프라 비용이 전 회사 대비 확연히 줄고, 장애 시 볼 곳이 하나라 대응이 빠르고, 신입 온보딩 시 DB 스택 설명이 바로 끝나고, 비개발자도 SQL만 알면 데이터를 뒤질 수 있어 데이터 공유가 빨랐다는 관찰. (주장/미검증)

### 방법론 — 언제 분리해야 하나
- 화자 확신: DAU 100만 정도 트래픽이 아니면 PostgreSQL 하나로 거의 다 처리할 수 있다. 그 수준에 도달하면 그때 개발자 고용해 분리해도 전혀 늦지 않다. [의견] (주장/미검증)
- 전문 툴이 진짜 필요한 임계값 (화자 기준, 주장/미검증):
  - Elasticsearch — 정말 복잡한 집계 쿼리를 매우 빠른 속도로 처리해야 할 때.
  - Pinecone — 수십억 개 벡터를 밀리초 단위로 검색해야 할 때 (Google·Netflix·TikTok 수준).
  - Redis — 초당 수십만 건의 캐시 읽기가 필요할 때.
- 스타트업·중소기업·스케일업 단계는 이 임계값에 해당할 가능성이 거의 없다는 것이 화자 주장. 지금 미리 복잡성을 사는 것은 "미래의 고통을 지금 당겨오는 것". [의견]

### DB를 늘릴 때 실제로 늘어나는 것 (숨은 비용)
- DB 하나 추가 = 저장소 하나가 아니라: 쿼리 언어가 달라져 각각 배워야 하고, 권한 시스템이 달라 따로 설정하고, 백업 전략·모니터링도 따로 하고, DB 간 데이터 동기화 문제가 생겨 Kafka나 Kinesis 같은 것을 또 써야 한다.
- "팀원 한 명 더 뽑는 것보다 DB 하나 더 붙이는 게 더 비쌀 수도 있다" — 인프라 비용에 더해 개발자 인지 부하 비용이 크다는 주장. [의견] (주장/미검증)

### PostgreSQL의 대체 능력 (익스텐션)
- PostgreSQL 위에 올라가는 플러그인(익스텐션)으로 전문 DB 기능을 흡수:
  - PG Search — Elasticsearch 수준의 풀텍스트 검색.
  - pgvector — AI 임베딩 저장 + 유사도(벡터) 검색.
  - PGMQ — Redis 없이 큐 시스템 구성.
- 결과적으로 "하나의 DB, 하나의 연결, 하나의 백업 전략"으로 수렴.

### 성능·비용 벤치마크 주장
- pgvectorscale(pgvector에서 나온 확장)과 Pinecone 벤치마크: 지연 시간 기준 Pinecone 대비 28배 빠르고, 비용 75% 절감 (주장/미검증).
- 이유(화자 설명): Pinecone은 별도 서비스라 네트워크 레이턴시가 붙지만, pgvector는 PostgreSQL 안에서 직접 실행돼 그 오버헤드가 없다 — "데이터가 이미 있는 곳에서 바로 검색".
- Elasticsearch vs PostgreSQL 풀텍스트 검색: 일반적 검색 쿼리에서는 성능 차이가 거의 없고, 오히려 Elasticsearch의 JVM 메모리 오버헤드가 더 문제가 되는 경우가 많다는 주장. (주장/미검증)

### 단일 DB의 이점 (정리)
- 신규 개발자 온보딩이 빠름 (PostgreSQL 하나만 배우면 됨).
- 장애 대응이 단순 (범인 찾을 곳이 다섯 군데가 아니라 한 군데).
- 비용 절감 (pgvector는 이미 쓰는 PostgreSQL 비용에 포함, Elasticsearch SaaS의 월 수백만 원 절약). (주장/미검증)

### 2026년·AI 에이전트 맥락
- Claude Code·Cursor 같은 도구로 AI가 코드를 짜고 테스트·배포하는 시대. AI 에이전트가 테스트할 때마다 DB 환경 세팅이 필요한데 DB가 다섯 개면 너무 복잡해서 "그냥 안 된다". PostgreSQL 하나면 AI도 바로 테스트 환경을 만들고 검증·작업할 수 있다. "단순함이 곧 속도." [의견]

### 실전 마이그레이션 절차
- 신규 프로젝트: 처음부터 PostgreSQL로 시작. 벡터가 필요해지면 그때 pgvector 설치, 검색이 필요하면 PG Search를 붙인다.
- 기존 프로젝트: (1) 지금 쓰는 전문 DB를 PostgreSQL 확장으로 대체 가능한지 먼저 테스트 — pgvector 설치 후 기존 벡터 데이터 마이그레이션하고 성능 비교, 차이 없거나 빠르면 넘어감. (2) 하나씩 천천히 교체 — 한 번에 다 바꾸면 사고 나므로 가장 덜 중요한 것부터 검증하며 진행. (3) 성능 모니터링 필수 — EXPLAIN ANALYZE로 쿼리 성능 확인, pg_stat_statements로 슬로우 쿼리 포착. PostgreSQL에는 이미 강력한 모니터링 도구가 있다.

### 세일즈 구조
- 마지막에 구독 요청(CTA)과 댓글 유도(현재 DB 몇 개 쓰는지, PostgreSQL로 안 되더라 하는 경험 공유)로 마무리. 별도 유료 자료·단톡방 유입 언급은 없음.

## 3줄 오리엔테이션
- 검색·캐시·벡터를 Elasticsearch·Redis·Pinecone으로 나눠 쓰는 대신, 화자는 PostgreSQL 하나 + 익스텐션(PG Search·pgvector·PGMQ)으로 대부분 대체할 수 있다고 주장한다.
- 세 회사 경험을 근거로 "다중 DB = 비용·인력·동기화 지옥"이었고 "단일 DB = 비용↓·장애대응↑·온보딩↑"였다고 회고하며, DAU 100만 전까지는 단일 PostgreSQL로 충분하다고 본다.
- 핵심 철학은 "복잡성은 필요할 때 사는 것이지 미리 사는 게 아니다"이며, AI 에이전트 시대에 단순한 스택이 곧 개발 속도라는 논지로 이어진다.

## 전문 — 시간순 워크스루

### [00:00–00:12] 도입 — DB 몇 개 쓰세요?
프로젝트 데이터베이스가 몇 개냐고 물으며, PostgreSQL 하나, Redis 하나, Elasticsearch 하나, 거기에 요즘 AI 한다고 Pinecone까지 네다섯 개를 쓰는 상황을 짚는다. 그런데 그걸 다 버리고 PostgreSQL 하나만 써도 된다고, "진짜로" 가능하다고 운을 뗀다.

### [00:12–00:34] 실제로 다 써봤다 — 두 번째 회사 아키텍처
검색은 Elasticsearch, 캐시는 Redis, 벡터는 Pinecone으로 나눠 써야 하나 고민을 많이 하지만, 화자는 이걸 실제로 다 써봤다고 말한다. 두 번째 회사 시절 팀 아키텍처는 검색=Elasticsearch, 메인 데이터=PostgreSQL, 비정형 데이터=MongoDB였다. "각자 잘하는 걸 맡기는" 구조라 이론상 완벽하게 들리지만, 실제 운영해 보니 악몽이었다는 것.
> "이게 이론상으로는 완벽하게 들리잖아요. 각자 잘하는 걸 맡기는 거니까. 근데 실제로 운영해 보니까 이게 악몽이더라고요." [00:30] [의견]

### [00:34–00:57] 다중 DB의 실제 고통 — 비용·인력·동기화·장애
문제는 우선 비용이다. Elasticsearch 클러스터 유지에 한 달 수백만 원, MongoDB도 따로 나간다 (주장/미검증). 이 셋을 관리할 사람이 있어야 하니 인력도 거의 두세 배가 들었다. DB 간 데이터 동기화 문제가 자꾸 생겨 싱크를 맞추려 Kafka까지 도입했고 비용이 몇 천만 원 깨졌다 (주장/미검증). 밤에 장애가 뜨면 세 DB 중 어디가 원인인지 다 뒤져야 했고, "DB 관리하느라 정작 제품은 못 만드는 느낌"이었다고 회고한다. [의견]

### [00:57–01:27] 세 번째 회사 — PostgreSQL 하나의 반전
세 번째 회사는 PostgreSQL 하나만 썼다. 규모 있는 서비스인데 그렇게 버틴다는 게 처음엔 "이게 맞아?" 싶었지만 실제로는 달랐다. 인프라 비용이 전 회사 대비 확연히 줄어 있었고, 장애가 나도 볼 곳이 하나라 대응이 훨씬 빨랐으며, 신입 온보딩 때 DB 스택 설명이 그냥 끝났다(PostgreSQL 하나니까). 심지어 비개발자도 SQL만 알면 데이터를 뒤질 수 있어 데이터 공유도 빨랐다. 여기서 "복잡한 게 좋은 게 아니구나, 충분히 커지기 전까지는 단순한 게 훨씬 강하구나"를 깨달았다고 말한다.
> "아 복잡한 게 좋은 게 아니구나. 충분히 커지기 전까지는 단순한 게 훨씬 강하구나." [01:24] [의견]

### [01:27–01:41] 임계값 — DAU 100만
그래서 화자는 DAU 100만 정도 트래픽이 아니면 PostgreSQL 하나로 거의 다 처리할 수 있다고 확신한다. 그 수준에 도달할 정도가 되면 그때 개발자를 고용해 분리해도 전혀 늦지 않다는 주장. [의견] (주장/미검증)

### [01:41–02:05] DB 추가 = 저장소 하나가 아니다 (숨은 비용)
데이터베이스를 하나 추가한다는 건 저장소 하나 생기는 게 아니라고 강조한다. 쿼리 언어가 달라져 각각 배워야 하고, 권한 시스템이 달라 따로 설정하고, 백업 전략도 달라지고, 모니터링도 따로 해야 한다. 게다가 DB끼리 데이터 동기화 문제가 생기는데 이게 나중에 진짜 지옥이라, Kafka나 Kinesis 같은 걸 또 써야 한다. 그래서 팀원 한 명 더 뽑는 것보다 DB 하나 더 붙이는 게 더 비쌀 수도 있고, 인프라 비용은 물론 개발자 인지 부하 비용이 어마어마하다고 말한다. [의견] (주장/미검증)

### [02:05–02:27] 왜 2026년에 더 중요한가 — AI 에이전트
이게 왜 2026년에 더 중요하냐면, 지금은 AI 에이전트 시대이기 때문이다. Claude Code, Cursor 같은 도구로 AI가 코드를 짜고 테스트하고 배포한다. AI 에이전트가 뭔가 테스트할 때마다 DB 환경 세팅을 해야 하는데 데이터베이스가 다섯 개면 너무 복잡해서 "그냥 안 된다". 반대로 PostgreSQL 하나면 AI도 바로 테스트 환경을 만들고 검증하고 작업할 수 있다. "단순함이 곧 속도"라고 정리한다. [의견]

### [02:27–02:41] 오늘 영상에서 보여줄 것 (예고)
이어 오늘 영상에서 보여줄 항목들을 예고한다: PostgreSQL이 전문 DB를 실제로 대체할 수 있다는 것, pgvector가 Pinecone보다 빠르다는 것, 숨겨진 운영 비용의 실체, 그리고 실제로 어떻게 마이그레이션하는지 등등.

### [02:41–03:01] 익스텐션 — PG Search, pgvector, PGMQ
본론으로, 요즘 PostgreSQL로 뭘 할 수 있는지 나열한다. 검색은 PG Search를 쓰면 Elasticsearch 수준의 풀텍스트 검색이 되고, pgvector로 AI 임베딩을 저장하고 유사도 검색을 하며, PGMQ를 쓰면 Redis 없이 큐 시스템을 만들 수 있다. 이 모두가 익스텐션 — PostgreSQL 위에 올라가는 플러그인이다. 결과는 "하나의 DB, 하나의 연결, 하나의 백업 전략".

### [03:01–03:29] 성능 반론 — pgvectorscale vs Pinecone 벤치마크
"전문 툴보다 성능이 떨어지지 않냐"는 반문에, pgvector에서 나온 pgvectorscale이라는 확장을 든다. 이게 Pinecone과 실제 벤치마크를 했는데, 지연 시간 기준 Pinecone 대비 28배 빠르고 비용은 75% 절감이라는 결과 (주장/미검증). 이유는 아키텍처 차이 — Pinecone은 별도 서비스라 네트워크 레이턴시가 붙지만, pgvector는 PostgreSQL 안에서 직접 실행되니 그 오버헤드가 없다. 데이터가 이미 있는 곳에서 바로 검색하는 셈이다.

### [03:29–03:41] Elasticsearch vs PostgreSQL 풀텍스트
Elasticsearch와 PostgreSQL 풀텍스트 검색을 비교하면, 일반적인 검색 쿼리에서는 성능 차이가 거의 없다고 말한다. 오히려 Elasticsearch의 JVM 메모리 오버헤드가 더 큰 문제가 되는 경우가 많다는 주장. (주장/미검증)

### [03:41–04:11] 전문 툴이 진짜 필요한 순간 (높은 임계값)
"적절한 도구를 사용하라"는 말 자체가 틀린 건 아니지만, 그 기준이 생각보다 훨씬 높다는 게 핵심이라고 한다. Elasticsearch가 진짜 필요한 순간은 정말 복잡한 집계 쿼리를 굉장히 빠른 속도로 처리해야 할 때, Pinecone이 필요한 순간은 수십억 개 벡터를 밀리초 단위로 검색해야 할 때(Google·Netflix·TikTok 수준), Redis가 반드시 필요한 경우는 초당 수십만 건의 캐시 읽기가 필요할 때다. (주장/미검증)

### [04:11–04:25] 스타트업은 그 임계값에 없다

### [04:25–05:00] 단일 DB로 줄이면 좋아지는 것 세 가지
PostgreSQL 하나로 줄이면 뭐가 좋아지는지 정리한다. 첫째, 신규 개발자 온보딩이 훨씬 빨라진다 — 여러 DB를 설명할 필요 없이 PostgreSQL 하나만 배우면 된다. 둘째, 장애 대응이 단순해진다 — 서비스가 죽었을 때 DB가 다섯 개면 다섯 군데를 다 봐야 하지만 하나면 거기만 보면 된다. 셋째, 비용 — Pinecone Pro 플랜과 달리 pgvector는 이미 쓰는 PostgreSQL 비용에 포함이고, Elasticsearch SaaS로 매달 나가던 몇백만 원을 PostgreSQL 검색으로 대체하면 그게 다 절약이다. (주장/미검증)

### [05:00–05:37] 실전 마이그레이션 절차
실제로 어떻게 하는지 설명한다. 새 프로젝트라면 처음부터 PostgreSQL로 시작하고, 벡터가 필요해지면 그때 pgvector를, 검색이 필요하면 PG Search를 붙이면 된다. 기존 프로젝트는 세 단계로: 첫째, 지금 쓰는 전문 DB를 PostgreSQL 확장으로 대체 가능한지 먼저 테스트한다 — pgvector 설치 후 기존 벡터 데이터를 마이그레이션하고 성능을 비교해, 차이가 없거나 오히려 빠르면 넘어간다. 둘째, 하나씩 천천히 바꾼다 — 한 번에 다 바꾸려다 사고 나므로 가장 덜 중요한 것부터 검증하고 넘어간다. 셋째, 성능 모니터링은 필수다 — EXPLAIN ANALYZE로 쿼리 성능을 확인하고 pg_stat_statements로 슬로우 쿼리를 잡는다. PostgreSQL에는 생각보다 강력한 모니터링 도구가 이미 있다.
