## 요약 (주제별)

### 왜 클로드 코드 단독 사용이 줄었나 (배경·문제의식)
- 화자 관찰(주장): 요즘 클로드 코드를 단독으로 쓰는 사람이 크게 줄었다. 이유는 (1) 클로드 코드의 불확실성, (2) 가이드(규칙)를 잘 안 지키는 문제, (3) 성능 저하. 이 때문에 코덱스를 보조로 같이 띄우거나 코덱스로 리뷰를 돌리는 케이스가 늘었고, 아예 코덱스로 옮겨가는 사람도 많아졌다. (주장/미검증)
- 구체 근거로 든 사례: 올해 2월~3월 사이 클로드 코드 성능이 일시적으로 떨어진 시기가 있었고, Anthropic이 직접 사후 보고서를 내 "출력 누락, 메모리 초기화 같은 버그 3개가 겹쳤다"고 밝혔다고 함. (주장/미검증)
- [의견] 이런 일을 겪으니 "한 모델만 믿고 가는 건 위험하다"는 감각이 생겼고, 그래서 워크플로가 코덱스 병행/재검토 쪽으로 이동 중이라고 봄.

### 코덱스 플러그인 CC (환경·도구)
- 2026년 3월 말 OpenAI가 공식 플러그인 "코덱스 플러그인 CC"를 공개. 클로드 코드 안에서 코덱스를 그대로 호출하게 해준다. (주장/미검증)
- 이후 클로드 코드 안에서 슬래시 커맨드 하나로, 같은 프로젝트·같은 컨텍스트에서 코덱스를 부를 수 있게 되어 병행 사용이 더 자연스러워짐.

### 왜 서로 다른 회사 모델을 같이 쓰나 (핵심 논리)
- 핵심 명제: "서로 다른 회사가 만든 모델은 서로 다른 곳에서 실수한다." (주장)
- 비유: 같은 학원·같은 강사에게 배운 두 학생은 틀리는 문제가 비슷하지만, 다른 학원·다른 강사에게 배운 학생을 데려오면 서로 놓친 부분을 잡아준다. 클로드와 코덱스가 그 관계.
- 또 다른 비유: 자기 글은 자기가 교정해도 오탈자를 못 잡고 다른 사람이 봐줘야 보이는 것과 같다. 클로드는 자기가 짠 코드를 자기 시각으로 보지만, 코덱스는 처음 보는 코드를 다른 시각으로 본다.

### 교차 리뷰 사례 — 코덱스가 잡아낸 버그들 (근거 사례)
- MCP 브릿지 코드 사례: 클로드 코드로 만든 걸 코덱스에 리뷰시켰더니 클로드가 못 본 문제 3개가 나옴 — (1) 종료 코드 처리 로직의 구멍, (2) 터미널 출력의 ANSI 이스케이프 문자 필터 누락으로 인한 인젝션 위험, (3) 경로를 두 번 적용하는 버그. (주장/미검증)
- 사이드바 대형 리팩터링 사례: 새 위저드 흐름·상태 관리를 새로 짜고 화면 여러 개를 동시에 수정한 뒤 코덱스 레스큐로 점검. 6분 35초 만에 엣지케이스 버그 4개 발견 — (1) 설정 안 된 상태에서 제출 버튼이 조용히 실패, (2) 특정 단계를 건너뛰고 리뷰 단계로 점프 가능, (3) 실행 중 요청을 뒤로 가기로 끊으면 복구 불가, (4) 동시 요청이 서로 상태를 덮어씀. (주장/미검증)
- [의견] 큰 변화가 한꺼번에 일어나면 엣지케이스는 상태 변화 사이 틈에 숨어 잘 안 보이고, 나중에 사용자가 걸려서야 드러난다 — 그래서 다른 시각이 필요하다는 것이 사례 제공자의 소감.

### 코덱스 레스큐 기능 (도구·기능)
- "코덱스 레스큐(rescue)" = "구조하러 와 달라"는 뜻. 클로드 코드로 작업하다 막힐 때(같은 버그를 두세 번 고쳐도 또 실패, 코드가 점점 복잡해지기만 할 때) 쓰는 기능.
- [의견] 화자는 예전엔 그냥 새 세션 열고 처음부터 다시 시작했다고 함. 레스큐를 쓰면 막힌 그 자리에서 슬래시 커맨드 하나("이 테스트가 왜 깨지는지 조사해 줘")로 코덱스가 백그라운드에서 그 작업을 따로 파본다.
- 그동안 클로드 코드 창에서는 다른 작업을 계속할 수 있고, 코덱스가 끝나면 결과를 가져와 보여준다.

### 모델 조합 & 비용 절감 (방법론·비용)
- 클로드 코드의 "Opus Plan" 모드: 플래닝은 Opus가, 실제 코드 작성은 Sonnet이 하도록 한 모드 안에서 역할이 자동 분리됨.
- 논리: Opus는 똑똑하지만 비싸고 Sonnet은 적당히 똑똑하면서 훨씬 싸다. 코딩에서 진짜 머리 써야 하는 건 초기 설계(구조·데이터 흐름·발생 가능 문제)인데 이건 전체 토큰의 약 10~20%뿐. 나머지 약 80%는 실제 코드 타이핑(함수 작성·테스트·임포트 정리)이라 Sonnet으로 충분. (주장/미검증)
- 비용 비교(인용 사례): 전부 Opus로 돌리면 약 1달러 50센트, Plan만 Opus·구현은 Sonnet 조합이면 약 48센트로 거의 1/3토막. (주장/미검증)
- 비유: 설계도만 베테랑에게 맡기고 시공은 일반 작업장에 맡기는 것 — 설계도가 좋으면 시공은 누가 해도 비슷하게 나온다.

### 핵심 패턴 — 플랜 후 코덱스 적대적 리뷰 (실전 방법론, 화자가 가장 강조)
- 화자가 "제일 말하고 싶었던 내용". 요즘 많이 쓰는 패턴:
  1. Opus에게 플랜을 시킨다 → 단계별 계획을 짜줌 → 이 플랜을 마크다운 파일 하나로 정리해 둔다.
  2. 바로 구현으로 넘어가지 않고 코덱스에게 "adversarial(적대적) 리뷰"를 돌린다 = 플랜을 비판적으로 검토. 문법이 아니라 설계 자체를 의심("이 구조가 맞나? 캐싱 로직 괜찮나? 동시성 문제 없나? 데이터 날아갈 위험 없나?").
- 근거 사례: 이 적대적 리뷰를 3라운드 돌렸더니 14개 문제가 나옴 — 인증 모델 누락, 셸 스크립트 다운로드 처리 버그 등이 플랜 단계에서 잡힘. (주장/미검증)
- [의견] 플랜 단계에서 잡으면 코드 짜기 전이라 수정 비용이 거의 안 들지만, 구현 다 하고 발견되면 다시 짜야 한다 — 그래서 이 순서가 중요.

### 설치·워크플로 (실전 절차)
- 준비물: 클로드 코드 설치, 코덱스 CLI 별도 설치. GPT는 무료로는 부족하고 [의견] 플러스(월 20달러) 정도가 괜찮다고 함.
- 설치 세 줄: (1) 클로드 코드 안에서 슬래시 플러그인 마켓플레이스에 OpenAI 코덱스 저장소 주소 추가, (2) 플러그인 인스톨 명령으로 코덱스 플러그인 설치, (3) 코덱스 셋업 한 번 실행. → 슬래시 커맨드 6개가 새로 생기며 많이 쓰는 건 코덱스 리뷰·코덱스 레스큐 둘.
- 워크플로 한 사이클: Opus가 플랜 → 플랜을 마크다운으로 저장(단일 기준점) → 코덱스 adversarial 리뷰로 플랜 두드리기 → 지적 반영 → 코드 구현 → 마지막에 코덱스 리뷰 한 번 더.

### 리스크·주의 (한계)
- 코덱스 레스큐에 가끔 무한 루프 버그가 있음: 같은 파일을 계속 다시 읽기만 하고 결과를 안 내놓는 경우, Mac에서 가끔 멈추기도 함. 5분 넘게 응답 없으면 코덱스 캔슬로 끊고 다시 시도. (주장/미검증)
- [의견] "아직 완성된 건 아니니 너무 맹신하면 안 된다."

### 세일즈·CTA 구조
- 마무리에 구독 요청("도움이 됐다면 구독 한 번"), 시청자에게 "어떤 조합으로 쓰냐" 질문, 다음 영상에서 코덱스 활용 워크플로를 더 다루겠다는 예고.

## 3줄 오리엔테이션
- 클로드 코드 단독 사용에서 벗어나 코덱스를 "다른 시각의 리뷰어"로 병행하는 흐름과 그 이유를 설명하는 영상.
- 핵심은 두 가지: (1) 서로 다른 회사 모델은 서로 다른 곳에서 실수하니 교차 리뷰가 유효, (2) 구현 전 "플랜 → 코덱스 적대적 리뷰 → 구현 → 코덱스 리뷰" 순서가 수정 비용을 아낀다.
- 부가로 Opus 설계 / Sonnet 구현 조합의 비용 절감, 코덱스 레스큐 기능, 플러그인 설치 3줄, 무한 루프 주의점을 다룸.

## 전문 — 시간순 워크스루

### [00:00–00:26] 도입 — 클로드 코드 단독 사용이 줄었다
화자는 요즘 클로드 코드를 단독으로 쓰는 사람이 크게 줄었다고 관찰을 연다. 이유로 클로드 코드의 불확실성, 가이드를 안 지키는 문제, 그리고 성능 저하 때문에 코덱스를 같이 띄우거나 코덱스로 리뷰를 돌리는 케이스가 많아졌고, 심지어 코덱스로 아예 옮겨가는 사람도 많아졌다고 한다. (주장/미검증) 오늘은 왜 이런 흐름이 생겼는지 알아보겠다고 예고한다.

### [00:26–01:00] 단독 사용의 불안 경험과 2~3월 성능 저하 사건
[의견] 화자는 한동안 클로드 코드만 띄워놓고 작업했고 결과물이 좋아 "이거 하나로 다 되겠다" 싶었다고 회고한다. 하지만 쓰다 보면 불안한 순간이 많았다 — 같은 버그를 두세 번 고쳤다는데 또 깨지고, 어떤 날은 멀쩡하다 어떤 날은 답이 이상하게 짧아졌다. 실제로 올해 2월~3월 사이 클로드 코드 성능이 일시적으로 떨어진 시기가 있었고, 나중에 Anthropic이 직접 사후 보고서를 내 출력 누락, 메모리 초기화 같은 버그 3개가 걸쳐 있었다고 밝혔다고 한다. (주장/미검증)
> "한 모델만 믿고 가는 게 좀 위험하다는 감각이 생긴 거예요." [00:58]

### [01:00–01:26] 워크플로 이동과 코덱스 플러그인 CC 등장
[의견] 그래서 코덱스를 보조로 같이 뛰거나 결과물을 코덱스에게 한 번 더 검토시키는 방식으로 워크플로가 옮겨가고 있다고 본다. 마침 2026년 3월 말 OpenAI가 공식 플러그인 "코덱스 플러그인 CC"를 공개했는데, 클로드 코드 안에서 코덱스를 그대로 호출하게 해주는 플러그인이다. 이게 풀린 뒤로 클로드 코드 안에서 슬래시 커맨드 하나로, 같은 프로젝트·같은 컨텍스트에서 코덱스를 부를 수 있어 병행이 더 자연스러워졌다. (주장/미검증)

### [01:26–01:47] 왜 굳이 코덱스도 쓰나 — 핵심 논리와 비유
첫 질문: 클로드 코드도 잘하는데 왜 코덱스를 또 쓰나. 핵심은 "서로 다른 회사가 만든 모델은 서로 다른 곳에서 실수한다"로 정리된다.
> "서로 다른 회사가 만든 모델은 서로 다른 곳에서 실수해요." [01:31]

비유로, 같은 학원·같은 강사에게 배운 두 학생은 틀리는 문제가 비슷하지만, 다른 학원·다른 강사에게 배운 학생을 데려오면 서로 놓친 부분을 잡아준다. 클로드와 코덱스가 딱 그런 관계라고 설명한다.

### [01:47–02:22] 사례 1 — MCP 브릿지 코드, 코덱스가 잡은 3가지
실제 사례로, 어떤 사람이 클로드 코드로 MCP 브릿지 코드를 만들고 다 됐다 싶었는데 코덱스에게 리뷰를 시켰더니 클로드가 못 본 문제 3개가 나왔다. (1) 종료 코드 처리 로직의 구멍, (2) 터미널 출력의 ANSI 이스케이프 문자 필터가 빠져 인젝션 위험이 있던 것, (3) 경로를 두 번 적용하는 버그. (주장/미검증) 클로드는 자기가 짠 코드니까 자기 시각으로 검토하지만, 코덱스는 처음 보는 코드를 다른 시각으로 봐서 보인다는 것. 사람이 자기 글을 교정하면 오탈자를 못 잡고 다른 사람이 봐줘야 잡히는 것과 비슷하다고 부연한다.

### [02:22–02:57] 코덱스 레스큐 기능 소개
두 번째 포인트로 이 플러그인에서 제일 인상적인 "코덱스 레스큐" 기능을 든다. 이름 그대로 "구조하러 와 달라"는 뜻. 클로드 코드로 작업하다 막힐 때 — 같은 버그를 두세 번 고쳤다는데 또 실패하고 코드가 점점 복잡해지기만 할 때 — 쓴다. [의견] 화자는 예전엔 그냥 새 세션 열고 처음부터 다시 시작했다고 한다. 레스큐를 쓰면 막힌 그 자리에서 슬래시 커맨드 하나("이 테스트가 왜 깨지는지 조사해 줘")만 치면 코덱스가 백그라운드에서 그 작업을 받아 따로 파본다. 그동안 클로드 코드 창에서는 다른 작업을 계속할 수 있고, 끝나면 결과를 가져와 보여준다.

### [02:57–03:34] 사례 2 — 사이드바 대형 리팩터링, 엣지케이스 4개
실제 예시로, 한 사람이 사이드바를 크게 리팩터링했다 — 새 위저드 흐름, 상태 관리를 새로 짜고 화면 여러 개를 동시에 수정. 다 됐다 싶었는데 코덱스 레스큐로 점검시켰더니 6분 35초 만에 엣지케이스 버그 4개가 나왔다. (1) 설정 안 됐을 때 제출 버튼이 조용히 실패, (2) 특정 단계를 안 쓰고 리뷰 단계로 점프 가능, (3) 실행 중인 요청을 뒤로 가기로 끊으면 복구가 안 됨, (4) 동시 요청에 서로 상태를 덮어씀. (주장/미검증) [의견] 이 사람은 큰 변화가 한꺼번에 일어나면 엣지케이스는 잘 안 보이고 상태 변화 사이 틈에 숨어 있다가 나중에 사용자가 걸려서야 드러난다며, 그래서 다른 시각이 필요했다고 했다.

### [03:34–04:27] 모델 조합과 비용 — Opus Plan / Sonnet 구현
세 번째 포인트는 요즘 많이 쓰는 모델 조합. 클로드 코드에는 "모델 Opus Plan" 모드가 있는데, 플래닝은 Opus가 하고 실제 코드 작성은 Sonnet이 하도록 한 모드 안에서 역할이 자동 분리된다. 이유: Opus는 똑똑하지만 비싸고 Sonnet은 적당히 똑똑하면서 훨씬 싸다. 코딩에서 진짜 머리 써야 하는 건 초기 설계(구조·데이터 흐름·발생 가능 문제)인데 이건 전체 토큰의 약 10~20%뿐이고, 나머지 약 80%는 실제 코드 타이핑(함수 작성·테스트·임포트 정리)이라 Sonnet으로 충분하다. (주장/미검증) 비용 비교로, 전부 Opus면 약 1달러 50센트, Plan만 Opus·구현은 Sonnet이면 약 48센트로 거의 1/3토막이 난다고 한다. (주장/미검증) 설계도만 베테랑에게 맡기고 시공은 일반 작업장에 맡기는 것에 비유한다 — 설계도가 좋으면 시공은 누가 해도 비슷하다.

### [04:27–05:13] 핵심 패턴 — 플랜 후 코덱스 적대적 리뷰
네 번째 포인트를 화자는 "사실 제일 말씀드리고 싶었던 내용"이라 강조한다. 요즘 많이 쓰는 패턴은, 먼저 Opus에게 플랜을 시켜("이 기능 만들 건데 어떻게 접근할까") 단계별 계획을 받고 이 플랜을 마크다운 파일 하나로 정리해 두는 것. 그다음 바로 구현으로 안 넘어가고 코덱스에게 adversarial(적대적) 리뷰를 돌린다 — 플랜을 비판적으로 검토시키는 것으로, 단순 문법이 아니라 설계 자체를 의심한다("이 구조가 맞나? 캐싱 로직 괜찮나? 동시성 문제 없나? 데이터 날아갈 위험 없나?"). 실제 어떤 사례에서는 이걸 세 라운드 돌렸더니 14개 문제가 나왔고, 인증 모델 누락·셸 스크립트 다운로드 처리 버그 등이 플랜 단계에서 다 잡혔다. (주장/미검증) [의견] 플랜 단계에서 잡으면 코드 짜기 전이라 수정 비용이 거의 안 들지만 구현 다 하고 발견되면 다시 짜야 하므로 이 순서가 중요하다.

### [05:13–05:44] 설치 방법 — 준비물과 세 줄 설치
이론을 봤으니 설치·사용법을 간단히 보여준다. 준비물은 클로드 코드가 깔려 있어야 하고 코덱스 CLI도 별도 설치되어야 한다. GPT는 무료로는 조금 부족하고, [의견] 플러스 월 20달러짜리 정도가 괜찮다고 본다. 설치는 세 줄: 먼저 클로드 코드 안에서 슬래시 플러그인 마켓플레이스에 OpenAI 코덱스 저장소 주소를 추가, 그다음 플러그인 인스톨 명령으로 코덱스 플러그인 설치, 마지막으로 코덱스 셋업을 한 번 돌리면 끝. 이러면 슬래시 커맨드 6개가 새로 생기고, 가장 많이 쓰는 건 코덱스 리뷰와 코덱스 레스큐 두 가지다.

### [05:44–06:09] 워크플로 한 사이클 정리
워크플로를 한 사이클로 정리한다. 먼저 Opus가 플랜을 짜준다 → 플랜을 마크다운 파일로 저장(나중에 단일 기준점이 됨) → 코덱스 adversarial 리뷰로 그 플랜을 두드려 보고 지적된 부분이 있으면 플랜에 반영 → 코드를 짠다 → 구현이 끝나면 마지막으로 코덱스 리뷰로 한 번 더 돌려 본다.

### [06:09–06:22] 주의 — 코덱스 레스큐 무한 루프 버그
한 가지 주의로, 코덱스 레스큐가 가끔 무한 루프에 빠지는 버그가 있다. 같은 파일을 계속 다시 읽기만 하고 결과를 안 내놓는 경우가 있고 Mac에서 가끔 멈춰버리기도 한다. 5분 넘게 응답이 없으면 코덱스 캔슬로 끊고 다시 시도하라고 안내한다. (주장/미검증)
> "아직 완성된 건 아니니까 너무 맹신하시면 안 됩니다." [06:21]

### [06:22–06:40] 마무리 — CTA와 다음 편 예고
오늘은 코덱스와 클로드 코드의 조합을 알아봤다며, 워크플로가 도움이 됐다면 구독을 한 번 눌러 달라고 요청한다(세일즈/CTA). 시청자에게 "어떤 조합으로 쓰냐"고 묻고, 다음 영상에서는 코덱스 등을 이용해 어떻게 워크플로를 잘 돌릴 수 있는지 알아보겠다고 예고하며 마친다.
