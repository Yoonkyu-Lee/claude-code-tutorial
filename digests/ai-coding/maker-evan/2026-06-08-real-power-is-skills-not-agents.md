# 진짜는 에이전트가 아니라 '스킬'이었다

> maker-evan · 2026-06-08 · 길이 약 12:52 · [영상 링크](https://youtu.be/2oOXHWGn_Cw)
> 정리 방식: 시간순 심층 워크스루 (paraphrase) + 주제별 상세 요약
> 검증 배너: 본 영상의 정확도·비용·시간 수치는 모두 Anthropic 공식 블로그를 화자가 인용·전달한 것으로, 화자가 직접 재현·검증한 값이 아니다. "(주장/미검증)"으로 표시한다. 영상 말미에 화자 유료 상품("스킬샵") 홍보가 포함된다.

## 요약 (주제별)

### 배경 — Anthropic이 공개한 사내 데이터 분석 자동화
- Anthropic이 공식 블로그에 자사 내부에서 Claude를 어떻게 쓰는지 공개했고, 화자는 이 글을 소재로 삼는다. 핵심 주장: 회사 데이터 분석 질문의 95%를 AI에 통째로 넘겼고, 정확도 평균 95%, 일부 영역 99% (주장/미검증).
- 화자는 이 글에서 배운 것을 자기 스킬 설계에 그대로 옮겨 적용 중이라고 밝힌다 [의견].

### 문제 정의 — 기존 세 접근이 다 실패한 이유
- 데이터를 모두가 쓸 수 있게 만드는 건 회사들의 오랜 숙제였고, 세 가지 접근이 모두 벽에 부딪혔다.
- (1) 넓고 평평한 테이블: 모든 데이터를 한 장의 거대 시트에 몰아넣기. 회사가 커지면 비슷한 시트가 늘고 '매출' 같은 정의가 시트마다 달라진다. SQL 모르는 사람에겐 무용.
- (2) 울타리 친 환경: 미리 정해진 대시보드만 보게 함. 자잘한 실제 질문을 못 받아줘 대시보드가 끝없이 늘어나 관리 불능.
- (3) AI 에이전트에 그냥 맡기기: 가장 위험. 그럴듯한 답을 주지만 맞는지 확인할 길이 없다. AI가 실제 데이터 구조·문서 전문가 지식과 분리돼 있어 "겉만 번지르, 속은 빈 가짜 정밀함"이 생긴다.

### 진짜 문제 세 가지 (Anthropic의 진단)
- (1) 개념과 실제 데이터 사이 애매함: '활성 사용자'라는 말 하나에도 정의 질문이 한가득(로그인? 글 작성? 가짜 계정 포함? 관찰 기간?). 정의를 안 주면 신입이 제멋대로 센다.
- (2) 데이터가 낡음: 회사는 살아 있어 테이블 구조·정의·로직이 계속 바뀐다. 어제 맞던 게 오늘 틀린다.
- (3) 정보를 못 찾음: 정답이 회사 어딘가에 있는데 AI가 못 찾는다. "책은 다 있는데 분류표 없는 도서관".

### 해법 구조 — 4층 아키텍처
- 집 짓듯 1층부터 쌓는 구조.
- 1층(데이터 기초 공사): 표준 데이터셋 = 단 하나의 진실의 원천(one source of truth). '매출은 무조건 이 데이터'로 못을 박음. 40개 후보가 아니라 딱 하나. 강제 수단으로 CI/CD를 걸어 규칙 안 지키면 통과를 못 시킴. 메타데이터(설명·정의·출처·소유자)를 부록이 아니라 1등 시민/제품처럼 관리.
- 2층(진실의 원천들): 신뢰도 순으로 쌓음. 맨 위 시맨틱 레이어(회사 공식 인증 지표 계산기 — '활성 사용자는 이렇게 계산'을 시스템이 보장) → 데이터 족보(lineage, 데이터가 어디서 와 어떻게 변했는지) → 과거 쿼리 모음(정리·다듬은 참고 문서) → 맨 아래 회사 맥락 지식(로드맵·의사결정 기록·조직도). 핵심 문장: "매출이 40개 후보가 아니라 딱 하나의 데이터로 연결되면 문제는 거의 사라진다."
- 3층(스킬): 절차적 지식 = 일하는 방법. 이 층이 정확도를 뒤집은 핵심.
- 4층(검증): 답이 맞는지 확인.

### 3층 스킬 — 정확도를 21%에서 95%로 뒤집은 층
- 스킬 = 절차적 지식(어떤 자료를 먼저 볼지, 애매할 때 어떻게 헤쳐 나갈지, 완성 분석이 어떻게 생겨야 하는지). 비유: 신입에게 주는 선배의 업무 매뉴얼.
- 수치(주장/미검증): 스킬 없을 때 정확도 21%를 못 넘음(5번 중 1번). 스킬 투입 후 95% 이상, 일부 영역 99%. 같은 AI인데 매뉴얼 하나 차이.
- 스킬을 둘로 나눔: (1) 지식 스킬 = 안내 데스크/라우터. 질문이 들어오면 "이건 마케팅 영역"이라 판단해 그 영역 참고 문서 30개 정도만 골라옴(수만 개 다 뒤지지 않음) — 도서관의 분류표. (2) 런북 스킬 = 베테랑 분석가의 순서(질문 명확화 → 자료 찾기 → 실행 → 적대적 검토 → 검증된 패턴 재사용(리텐션 곡선, 문제 쪼개기 등)). 매번 새로 짜지 않고 검증된 툴을 가져다 씀.
- 참고 문서 구조: 빠른 참조(맥락·기본 필터) → 차원 설명 → 핵심 테이블 → 주의사항·자주 쓰는 패턴 → 다른 문서로의 연결. 군더더기 없이 필요한 것만.

### 문서 부패(decay)와 CI 훅 해법
- 스킬 완성 후 95%를 찍었으나 딱 한 달 뒤 65%로 하락 (주장/미검증). 원인은 방치가 아니라 오히려 회사가 계속 일해서 테이블·정의가 바뀌었는데 스킬 문서만 그대로였던 것 — 문서가 낡음.
- 해법: 스킬 문서를 데이터 코드와 같은 창고에 넣고 CI 훅을 걸어, 데이터 모델을 바꾸면 스킬 문서도 같이 바꾸도록 강제(안 바꾸면 통과 안 됨).
- 결과: 코드 수정의 90%가 스킬 업데이트를 함께 포함(주장/미검증). 교훈: 낡은 문서는 모델 한 개보다 AI 성능을 더 빨리 죽인다.

### 4층 검증 — 오프라인 평가·반전 실험·실시간 검증
- 오프라인 평가: 정답을 아는 문제를 대량으로 만들어 시험. 문제 출제도 Claude가 대시보드 보고 자동 생성, 사람이 검수. 영역별 90% 미달이면 출시 차단.
- 반전 실험: 과거 SQL 수천 개를 통째로 줬고 그 파일의 80%가 관련 있었음(정답이 다 거기 있었음). 그런데 정확도가 1점도 안 오름 (주장/미검증). 결론: 문제는 정보 접근이 아니라 구조 — 질문을 데이터에 연결하는 '다리'가 없었던 것. "데이터는 쌓는 게 아니라 구조화하는 것."
- 실시간 검증: AI 답을 다른 AI 검토관이 의심·검증(적대적 검토). 트레이드오프(주장/미검증): 정확도 +6%, 토큰 -32%(비용 절감), 응답 시간 +72%(느려짐). "공짜 점심은 없다."
- 출처 푸터: 답 밑에 어느 등급 출처에서 왔는지·신선도·소유자를 표시(식당 메뉴판 원산지 표시 비유).
- 자동 교정 수집: 예약된 에이전트가 채팅 채널을 돌며 "이 숫자 틀렸다"는 지적을 잡아 수정안을 자동 생성 — 시스템이 스스로 학습·개선.

### 결과와 6가지 핵심 교훈
- 결과(주장/미검증): 비즈니스 분석 질문 95% 자동화, 정확도 평균 95%·특정 영역 99%, 데이터 사이언티스트가 단순 질문에서 해방돼 모델링·예측·머신러닝 같은 고부가 업무로 이동.
- 교훈: (1) 데이터는 소프트웨어가 아니다 — 코드는 맞는지 증명 가능하지만 데이터 분석은 정답 증명이 불가, 진짜 문제는 '애매함' 자체. (2) 정확도는 모델 지능이 아니라 맥락과 검증의 문제. (3) 강제 없는 규칙은 빠르게 썩는다. (4) 낡은 문서는 모델 한 개보다 성능을 더 빨리 죽인다(한 달 만에 65% 사례). (5) AI는 문서 만들기는 잘하지만 정의는 사람이 책임져야 한다('활성 사용자' 정의는 사람 몫). (6) 과거 쿼리는 그냥 쌓지 말고 정리된 패턴으로 다듬어야 한다(날것은 1%도 안 오른다 — 영상에서 두 번 강조).

### 최소 시작 지점 (Anthropic 권장)
- 거창하게 시작하지 말고 딱 셋: (1) 표준 데이터셋 몇 개 — 핵심 데이터만 단 하나의 진실로 정함. (2) 오프라인 평가 수십 개 — 정답 아는 시험 문제로 성능 측정. (3) 얇은 지식 스킬 하나 — 처음부터 두껍게 말고 안내 데스크 역할만 하는 아주 얇은 스킬.
- 이후 조건(오류 허용도, 비즈니스 복잡도, 사용자 기술 숙련도, 정확도 vs 속도 우선순위, 접근 관리)에 맞춰 한 층씩 쌓음. 리듬: "잡고 → 측정하고 → 강제하고 → 관리한다."

### 화자 결론과 세일즈 구조
- [의견] 화자 결론: 우리에게 필요한 건 에이전트를 만드는 것이 아니라 스킬을 만드는 것이며, 이런 스킬을 쌓아 테크닉을 늘리는 것이 '해자(moat)'다.
- 세일즈 장치: 검증된 스킬을 얻고 싶으면 화자 유료 상품 "스킬샵"을 방문하라는 홍보로 마무리(정보로 1회 기록).

## 3줄 오리엔테이션
- Anthropic이 공식 블로그에서 공개한 "사내 데이터 분석 95% AI 자동화" 사례를, 화자가 4층 아키텍처(기초 데이터 → 진실의 원천 → 스킬 → 검증) 중심으로 해설한다.
- 핵심 반전은 "정보를 더 준다고 정확도가 오르지 않는다(과거 SQL 수천 개 줘도 +0%)" — 문제는 접근이 아니라 구조·스킬(절차적 지식)이라는 점.
- 결론은 "에이전트가 아니라 스킬이 진짜다"이며, 말미에 화자 유료 상품 홍보가 붙는다. 정확도·비용 수치는 전부 인용(미검증).

## 전문 — 시간순 워크스루

### [00:00–00:35] Anthropic 블로그에서 출발 — 95% 통제 이양
오늘 이야기는 Anthropic 공식 블로그에 직접 올라온 글에서 시작한다. Anthropic은 자사 내부에서 Claude를 어떻게 쓰는지 공개하고 있는데, 요지는 회사 데이터 분석 업무의 95%를 AI에 통째로 넘겼다는 것이다. 정확도가 무려 95%, 어떤 영역에서는 99%까지 나왔고, 데이터 사이언티스트들이 더 이상 단순 질문에 시달리지 않게 됐다(주장/미검증).
> "저도 이 글에서 배운 걸로 지금 제 스킬들을 설계하고 있어요." [00:24] [의견]

화자는 오늘 그 안에 숨은 노하우와 팁을 하나도 빠짐없이 뜯어보겠다고 예고한다.

### [00:35–01:06] 실패한 접근 (1) — 넓고 평평한 테이블
데이터를 모두가 쓸 수 있게 만드는 건 회사들의 오랜 숙제였고, 방법마다 벽에 부딪혔다. 첫 번째는 '넓고 평평한 테이블' — 모든 데이터를 한 장의 거대한 엑셀 시트에 다 때려 박는 것이다. 처음엔 괜찮아 보이지만 회사가 커지면 비슷한 시트가 자꾸 생기고, '매출'이 이 시트에선 이렇게, 저 시트에선 저렇게 계산돼 정의가 서로 안 맞는다. 결국 SQL 모르는 사람에겐 아무 도움이 안 됐다.

### [01:06–01:38] 실패한 접근 (2) 울타리 친 환경, (3) AI 에이전트
두 번째는 '울타리 친 환경' — 미리 정해진 대시보드만 보게 하는 것이다. 그러나 사람들이 진짜 궁금해하는 자잘한 질문을 못 받아줘 대시보드가 끝도 없이 늘어나 관리가 안 된다. 세 번째는 AI 에이전트에게 그냥 맡기는 것인데, 이게 제일 위험했다. 그럴듯한 답을 주지만 그 답이 맞는지 확인할 길이 없고, AI가 회사의 진짜 데이터 구조나 문서 전문가 지식과 분리돼 있어 "겉만 번지르하고 속은 빈 가짜 정밀함"이 생긴다. 셋 다 실패했다.

### [01:38–02:36] Anthropic이 정리한 진짜 문제 세 가지
Anthropic은 진짜 문제를 딱 셋으로 정리했다. 첫째, 개념과 실제 데이터 사이의 애매함. 예컨대 '활성 사용자'라는 말 하나에도 질문이 한가득이다 — 어떤 행동을 해야 활성인가, 로그인이면 되나, 글을 써야 하나, 가짜 계정도 포함하나, 며칠 전까지 본 것을 활성으로 칠 건가.
> "신입 직원한테 '단골 손님 수 좀 세 줘'라고 한 거랑 같아요. 단골의 기준을 안 정해 주면 신입은 자기 맘대로 세요." [02:07]

둘째, 데이터가 자꾸 낡는다는 것. 회사는 살아 있어 테이블 구조·정의·로직이 바뀌고 어제 맞던 게 오늘 틀린다. 셋째, 정보를 못 찾는 것. 정답이 회사 어딘가에 분명히 있는데 AI가 못 찾는다 — "책은 다 읽는데 분류표가 없는 도서관"에서 수만 건 사이를 헤매는 격이다.

### [02:36–03:07] 4층 구조 시작 — 1층 표준 데이터셋(one source of truth)
Anthropic은 이 셋을 하나씩 부쉈다. 이제 유명한 4층짜리 구조를, 집 짓듯 1층부터 올라간다. 1층은 데이터 기초 공사이고 여기서 제일 중요한 단어가 '표준 데이터셋', 즉 단 하나의 진실의 원천(one source of truth)이다. 앞서 매출이 시트마다 다르게 계산된다고 했는데, 표준 데이터셋은 "매출은 무조건 이 데이터를 봐라"고 못을 박는 것이다 — 40개의 그럴듯한 후보가 아니라 딱 하나로.
> "음식점 표준 레시피예요. 알바가 바뀌어도 김치찌개 맛이 똑같으려면 레시피가 하나여야 하잖아요." [03:11]

### [03:07–03:45] CI/CD 강제 + 메타데이터를 1등 시민으로
여기서 진짜 노하우가 나온다. 레시피만 정해 두면 사람들이 안 지키므로, Anthropic은 CI/CD로 강제했다. CI/CD는 코드를 올릴 때 자동으로 검사하는 "컨베이어 검수대"로, 규칙을 안 지키면 아예 통과를 못 시킨다. 또 하나, 메타데이터를 1등 시민으로 대접했다. 메타데이터는 이 데이터가 뭔지 설명하는 꼬리표(설명·정의·출처·소유자)인데, 이를 부록 취급하지 않고 제품처럼 정성껏 관리했다.

### [03:45–04:47] 2층 진실의 원천들 — 시맨틱 레이어·족보·과거 쿼리·맥락
2층은 '진실의 원천들'로, 믿을 수 있는 정보를 신뢰도 순서로 쌓았다(위일수록 더 믿을 만함). 맨 위는 시맨틱 레이어 — 회사가 공식 인증한 지표 계산기로, "활성 사용자는 이렇게 계산한다"를 시스템이 보장한다. 이게 있으면 사람마다 다르게 계산해서 조용히 틀리는 일이 사라진다. 그 아래는 데이터의 족보(lineage) — 데이터가 어디서 와 어떻게 변했는지, 옛날에 버려진 테이블의 맥락까지 보여준다. 그 아래는 과거 쿼리 모음인데, 예전 SQL을 그냥 쌓아둔 게 아니라 잘 정리된 참고 문서로 다듬었다. 여기 중요한 팁이 숨어 있다.
> "날것 SQL을 그냥 검색하게 했더니 정확도가 1%도 안 올랐대요. 다듬어야 의미가 있다는 거예요. 이거 뒤에서 또 나와요." [04:28]

맨 아래는 회사의 맥락 지식(컨텍스트) — 로드맵·의사결정·기록·조직도 같은 것이다. 핵심 한 문장: "매출이라는 단어가 40개 후보가 아니라 딱 하나의 데이터로 연결되면 문제는 거의 사라진다."

### [04:47–05:05] 중간 정리 + 3층 예고
여기까지 1층(기초 공사)과 2층(sources of truth)을 봤다. 화자는 지금부터가 진짜라며, 다음 3층이 "정확도를 21%에서 95%로 뒤집은 마법의 층"이라고 강조한다.

### [05:05–05:44] 3층 스킬 — 절차적 지식, 21%→95%
3층의 이름은 '스킬'이고, 스킬이란 절차적 지식, 쉽게 말해 '일하는 방법'이다. 어떤 자료를 먼저 봐야 하는지, 애매할 땐 어떻게 헤쳐 나가는지, 완성 분석은 어떻게 생겨야 하는지 같은 노하우 묶음이다. 비유하면 신입에게 주는 선배의 업무 매뉴얼로, 데이터만 던져 주는 것과 일하는 법까지 알려 주는 것은 하늘과 땅 차이다.
> "스킬이 없을 때 정확도가 21%를 못 넘었어요. … 스킬을 넣었더니 95% 위로, 어떤 영역은 99%까지요." [05:30] (주장/미검증)

### [05:44–06:29] 스킬을 둘로 — 지식 스킬(라우터)과 런북 스킬
이 스킬을 영리하게 둘로 나눈 것이 진짜 노하우다. 첫째는 '지식 스킬'로 안내 데스크(라우터) 역할이다. 질문이 들어오면 "이건 마케팅 영역이네" 하고 그 영역에 맞는 참고 문서 30개 정도만 골라 불러온다 — 수만 개를 다 뒤지지 않고 잘 고른 30개만. 이게 앞서 도서관 비유의 '분류표'다. 둘째는 '런북 스킬'로 베테랑 분석가의 순서다: 먼저 질문을 명확히 하고, 자료를 찾고, 실행하고, 그다음 핵심인 '적대적 검토'를 한다 — 자기가 낸 답을 자기가 의심하며 다시 따진다. 그리고 자주 쓰는 패턴(리텐션 곡선, 문제 쪼개는 법 등)을 재사용한다. 매번 새로 짜지 않고 검증된 툴을 가져다 쓰는 것이다.

### [06:29–06:46] 참고 문서의 생김새
참고 문서에는 구조가 있다. 맨 위 빠른 참조(이 데이터의 맥락과 기본 필터), 그다음 차원 설명, 핵심 테이블, 주의 사항과 자주 쓰는 패턴, 마지막에 다른 문서로 가는 연결까지. 군더더기 없이 딱 필요한 것만 짜여 있다.

### [06:46–07:41] 부패 이야기 — 한 달 만에 95%→65%, CI 훅으로 해결
화자가 제일 좋아하는 이야기다. 스킬을 다 만들어 정확도 95%를 찍었는데, 딱 한 달이 지나자 65%로 떨어졌다. 아무것도 안 해서가 아니라 오히려 반대로, 회사가 계속 일하며 테이블과 정의가 바뀌었는데 스킬 문서는 그대로여서 문서가 낡아버린 것이다(주장/미검증). "가게 메뉴는 바뀌었는데 신입에게 준 매뉴얼은 작년 것"이라 신입이 열심히 할수록 더 틀린다. 해결책으로 Anthropic은 스킬 문서를 데이터 코드와 같은 창고에 넣고 CI 훅을 걸어, 데이터 모델을 바꾸면 스킬 문서도 같이 바꾸도록 강제했다(안 바꾸면 통과 불가). 그 결과 코드 수정의 90%가 스킬 업데이트를 함께 포함하게 됐다(주장/미검증).
> "낡은 문서는 AI 성능을 모델 한 개보다 더 빨리 죽여요." [07:36]

### [07:42–08:03] 4층 검증 (1) — 오프라인 평가
4층은 검증, 즉 답이 맞는지 확인하는 층이며 크게 세 가지로 했다. 먼저 오프라인 평가 — 미리 정답을 아는 문제를 잔뜩 만들어 AI에게 시험을 보게 한다. 재밌는 건 이 시험 문제도 Claude가 대시보드를 보고 자동으로 뽑고 사람이 검수한다는 점이다. 그리고 영역별로 90%를 안 넘으면 출시를 못 하게 막았다.

### [08:03–08:38] 반전 실험 — 과거 SQL 수천 개를 줘도 +0%
그리고 반전 실험이다. Anthropic은 AI에게 과거 SQL 수천 개를 통째로 주면 어떨지 궁금해 실제로 줘봤고, 그 파일의 80%가 관련이 있었다(정답이 거기 다 있었다). 그런데 정확도가 1점도 안 올랐다(주장/미검증). 즉 문제는 정보 접근이 아니었다 — 정보는 이미 있었다. 진짜 문제는 구조, 즉 질문을 데이터에 어떻게 연결하느냐의 '다리'가 없었던 것이다.
> "신입한테 10년치 영수증을 박스째 던져 준 거예요. '여기 답 다 있어' 하고요. 근데 정리가 안 돼 있으니 못 쓰는 거죠. 데이터는 쌓는 게 아니라 구조화하는 거예요." [08:30]

### [08:38–09:10] 실시간 검증 — 적대적 검토의 트레이드오프
마지막은 실시간 검증으로, 여기서 적대적 검토가 다시 나온다. AI가 답을 내놓으면 다른 AI 검토관이 그 답을 의심하고 따진다. 다만 공짜가 아니라 트레이드오프가 숫자로 나온다(주장/미검증): 정확도는 6% 오르고, 토큰은 32% 줄어(비용 절감) 좋지만, 응답 시간이 72% 늘어 느려진다. 정확도를 얻는 대신 속도를 내주는 것이다 — "공짜 점심은 없다", 무엇을 중요하게 볼지 정해야 한다.

### [09:10–09:35] 출처 푸터 + 자동 교정 수집
또 하나 좋은 장치가 출처 푸터다. 답을 줄 때 밑에 꼬리표를 달아 이 정보가 어느 등급 출처에서 왔는지, 얼마나 신선한지, 누가 주인인지를 표시한다 — 식당 메뉴판에 원산지 표시하듯 믿고 먹을 수 있게. 마지막은 자동 교정 수집이다. 예약된 에이전트가 채팅 채널들을 돌아다니다 누가 "이 숫자 틀렸어요" 하면 그걸 잡아 수정안을 알아서 만든다 — 시스템이 스스로 학습하며 나아진다.

### [09:35–10:00] 결과 정리
4층을 다 봤고, 결과를 숫자로 정리하면 비즈니스 분석 질문의 95%가 자동화됐고, 정확도는 평균 95%, 특정 영역 99%다(주장/미검증). 제일 중요한 건 데이터 사이언티스트들이 매일 들어오는 단순 질문에서 벗어나 모델링·예측·머신러닝 같은 진짜 어려운 일을 하게 됐다는 점이다 — AI가 잡일을 가져가니 사람은 더 가치 있는 일로 올라갔다.

### [10:00–10:53] Anthropic이 짚은 6가지 핵심 교훈
화자는 회사에 그대로 적용 가능하다며 여섯 교훈을 든다. 첫째, 데이터는 소프트웨어가 아니다 — 코드는 맞는지 증명할 수 있지만 데이터 분석은 정답을 딱 증명할 방법이 없고, 진짜 문제는 코드 생성이 아니라 '애매함' 그 자체다. 둘째, 정확도는 결국 맥락과 검증의 문제이지 모델이 똑똑한가의 문제가 아니다. 셋째, 강제가 없는 규칙은 빠르게 썩는다. 넷째, 낡은 문서는 모델 한 개보다 성능을 더 빨리 죽인다(한 달 만에 65% 사례). 다섯째, AI는 문서 만들기는 잘하지만 정의는 사람이 책임져야 한다 — '활성 사용자'가 뭔지는 AI가 아니라 사람이 정한다. 여섯째, 과거 쿼리는 그냥 쌓지 말고 정리된 패턴으로 다듬어야 한다 — 날것으로 주면 1%도 안 오른다(영상에서 두 번 강조).

### [10:53–11:52] 최소 시작 지점 세 가지
Anthropic은 최소 시작 지점도 알려줬다. 거창하게 시작할 필요 없이 딱 셋이다. 첫째, 표준 데이터셋 몇 개 — 모든 걸 정리할 필요 없이 제일 중요한 핵심 데이터 몇 개만 단 하나의 진실(source of truth)로 정한다. 둘째, 오프라인 평가 수십 개 — 정답을 아는 시험 문제 수십 개를 만들어 성능을 측정한다. 셋째, 얇은 지식 스킬 하나 — 처음부터 두껍게 고심해 만들지 말고 안내 데스크 역할만 하는 아주 얇은 스킬 하나만 만든다. 이 셋으로 시작한 뒤 각자 상황(오류 허용도, 비즈니스 복잡도, 사용자의 기술 숙련도, 정확도 vs 속도 우선순위, 접근 관리 방식)에 맞춰 한 층씩 쌓으면 된다. 리듬은 "잡고 → 측정하고 → 강제하고 → 관리한다".

### [11:52–12:52] 화자 결론 + 세일즈
화자는 Anthropic이 회사 데이터 분석에서 한 것을 자기 스킬 설계에 그대로 옮겨오고 있으며, 어떤 일을 하든 이 원리는 똑같이 적용된다고 말한다 [의견]. 시청자에게 "AI에 일 시킬 때 스킬·매뉴얼을 만들어 써봤는지" 댓글로 경험을 묻는다.
> "우리에게 필요한 건 에이전트를 만드는 것이 아닙니다. 스킬을 만드는 거예요. … 이런 스킬을 만들어 테크닉을 늘려 나가는 것이 우리의 해자라고 보실 수 있습니다." [12:29] [의견]

마지막으로 검증된 스킬을 얻고 싶으면 자신의 유료 상품 "스킬샵"을 방문하라며(검증된 스킬을 판매 중) 홍보하고 영상을 마친다.
