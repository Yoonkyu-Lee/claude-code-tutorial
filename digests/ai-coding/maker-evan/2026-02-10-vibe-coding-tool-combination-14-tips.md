# 똑같은 프롬프트인데 결과물이 다른 이유 (바이브 코딩 실전 팁 14)

> maker-evan · 2026-02-10 · 길이 약 5:26 · [영상 링크](https://youtu.be/Jt-rRUC2BMo)
> 정리 방식: 시간순 심층 워크스루 (paraphrase) + 주제별 상세 요약
> 검증 배너: 성능·우열 비교는 대부분 화자의 실전 체감에 근거한 주장(미검증)이다.

## 요약 (주제별)

### 핵심 테제 — 도구 조합이 절반
- 바이브 코딩의 결과물 차이는 "도구 선택이 절반, 나머지 절반은 도구를 쓰는 순서와 조합"에서 온다는 것이 영상 전체의 관통 주장이다. [의견]
- 하나의 도구를 잘 쓰는 게 아니라, 각 도구가 잘하는 영역을 파악해 파이프라인으로 엮는 것이 핵심이라고 반복 강조한다. 최종 정리: 분석은 Gemini, 코딩은 Claude, 음성 추출은 Whisper, 마지막에 스킬로 품질 체크. [의견]

### 모델별 강·약점 (화자 주장/미검증)
- **Gemini**: 성능은 좋지만 한국어 프롬프트를 넣으면 결과물이 확 떨어진다(주장/미검증). 이미지 인식·영상 분석은 Gemini 계열이 현재 가장 잘한다(주장/미검증).
- **Claude**: 코딩이 가장 낫다(주장/미검증). 간단한 로고·미니멀 디자인은 Claude 웹에서 의외로 잘 뽑는다(복잡한 일러스트는 어려움).
- **OpenAI**: 뭘 시켜도 어느 정도 하는 "평타"는 좋으나 한 영역을 압도하지 못한다(주장/미검증). 메인보다 보조로 쓰는 편. 단 Whisper 같은 특정 API는 매우 좋다.

### "Claude 경유" 프롬프트 패턴
- Gemini에 넣을 프롬프트든 Stitch에 넣을 프롬프트든, 사람이 영어로 직접 쓰지 말고 Claude에게 한국어로 설명해 영문 프롬프트로 변환시킨 뒤 그 결과를 넣으라는 패턴이 반복된다(팁 1, 7).
- Stitch의 경우 레퍼런스 URL을 주는 대신 Claude가 웹사이트 정보를 읽고 정리한 묘사 + 영문 프롬프트를 넣는 조합이 현재 가장 효과적이라고 한다. [의견]

### 영상 분석 워크플로
- 이미지·영상 분석은 Gemini가 압도적(주장/미검증). 레퍼런스 웹사이트 스크린샷을 Gemini에 넣어 구조를 분석하고, 그 결과를 Claude에 넘겨 "이 구조로 만들어 줘"로 작업하면 혼자 할 때보다 정확하다.
- 영상은 통째로 AI에 던지지 말 것. 특히 후킹 포인트 찾기는 현재 모델 한계가 분명. 대신 OpenAI Whisper API로 음성을 먼저 텍스트로 추출한 뒤 분석하면 컨셉 분석 정확도가 완전히 다르다고 한다. [불명확: "을도파 컨셉" — 자동자막 오인식, 문맥상 "후킹/컨셉 분석"으로 추정]

### Google Stitch 실전 팁
- 모델을 Pro로 쓸 것: Pro와 Flash의 결과물 차이가 크다. 빠른 테스트는 Flash도 무방하나 실제 프로젝트는 Pro(디자인 퀄리티 차이). (팁 5)
- 레퍼런스 URL을 주지 말 것: URL을 잘 못 알아먹으므로, 레이아웃을 말로 묘사하고 컬러 코드를 정확히 넣어라. (팁 6)
- 결과물을 그대로 쓰지 말 것: Stitch 결과물을 AI Studio에서 빌드·다운로드한 뒤 "Next.js 프론트엔드 가이드라인 스킬" 같은 것으로 컨버팅해 프로덕션 수준 코드로 올려라. (팁 8) [불명확: 스킬 정확 명칭 — 자동자막상 "넥스트 JS 프론트 핸드 가이드라인 스킬"]

### UI 라이브러리·스킬 자동화
- UI 라이브러리는 MUI보다 shadcn을 권장. AI가 shadcn 기반으로 코드를 생성하면 커스텀 없이도 모던하고 깔끔한 결과가 기본으로 나온다(주장/미검증). (팁 9)
- 코드를 짜고 끝내지 말고 마지막에 스킬로 자동 체크: Vercel의 React best practices 스킬로 코드 품질을, 웹 디자인 가이드라인 스킬로 디자인을 체크하면 완성도가 올라간다. 이 두 스킬은 `skills.sh`(추정) 사이트에서 받을 수 있다고 한다. (팁 10) [불명확: 사이트명 — 자동자막 "skill.shh.com / shc.com"]

### 디자인 지시 요령
- 로고·미니멀 디자인은 Claude 웹에서 바로 만들어 보라(별도 디자인 툴 불필요한 경우 많음). (팁 11)
- 디자인 수정 시 디테일을 하나하나 지시하기보다 "미니멀하게 다듬어 줘"처럼 방향만 주는 게 효과적일 때가 있다. "깔끔하게 여백 살려서 미니멀하게" 같은 키워드가 잘 먹힌다. (팁 12) [의견]

### AI가 아직 못하는 것 (한계)
- SNS 글은 AI로 통째로 쓰지 말 것: 티가 난다. 원문(생각·언어)은 직접 쓰고 AI는 문장 정리·오탈자·흐름 개선만 하는 게 사람 냄새 나는 글이 된다. (팁 13) [의견]
- 후킹 포인트 찾기는 아직 AI가 못한다: 시청자가 멈추는 순간·감정이 바뀌는 타이밍은 사람 감각이 필요. AI는 텍스트 기반 분석은 잘하나 영상의 흐름·감정 읽기는 부족하다. (팁 14) [의견]

### 세일즈 구조
- 명시적 유료 상품·단톡방 유입은 없음. 마지막에 구독·좋아요 요청과 "궁금한 건 댓글로" 유도가 전부. 외부 리소스로 `skills.sh`(추정) 스킬 다운로드를 언급.

## 3줄 오리엔테이션
- 같은 프롬프트라도 결과가 다른 이유는 "도구 조합·순서"에 있다는 전제 아래, 화자가 직접 삽질하며 얻은 바이브 코딩 실전 팁 14개를 순서대로 나열하는 영상이다.
- 관통 원리 두 가지: (1) 각 도구의 강점 영역을 나눠 파이프라인으로 엮어라(분석=Gemini, 코딩=Claude, 음성=Whisper), (2) 프롬프트는 Claude에 한국어로 설명해 영문으로 변환시켜 다른 도구에 넣어라.
- 성능·우열 비교는 대부분 화자의 실전 체감(주장/미검증)이며, 후반부(팁 13·14)는 "AI가 아직 못하는 영역"으로 마무리한다.

## 전문 — 시간순 워크스루

### [00:00–00:24] 도입 — 왜 같은 프롬프트인데 결과가 다른가
바이브 코딩에서 똑같은 프롬프트를 써도 결과물 차이가 크게 나 답답했던 경험을 화자도 처음엔 겪었다고 연다. 여러 프로젝트를 직접 만들며 깨달은 결론은 "바이브 코딩은 도구 선택이 절반이고, 나머지 절반은 도구를 쓰는 순서와 조합"이라는 것. [의견] 오늘은 실전에서 체감한 팁 14가지를 한 번에 정리하며, 하나하나가 직접 삽질 끝에 알게 된 것이라 끝까지 보면 작업 시간이 확실히 줄 것이라고 예고한다.

> "바이브 코딩은 도구 선택이 절반이고 나머지 절반은 도구를 쓰는 순서와 조합이라는 겁니다." [00:10]

### [00:25–00:45] 팁 1 — Gemini엔 영어로, 단 직접 쓰지 말고 Claude 경유
Gemini 계열 모델은 성능이 아주 좋지만 한국어 프롬프트를 넣으면 결과물이 확 떨어진다고 한다(주장/미검증). 그렇다고 영어 프롬프트를 사람이 직접 쓸 필요는 없다. Claude에게 먼저 한국어로 설명하고 "영문 프롬프트로 변환해 달라"고 한 뒤, 그 결과를 Gemini에 넣으라는 것. 이 한 단계만 추가해도 결과물 퀄리티가 체감될 정도로 달라진다고 주장한다. [의견]

### [00:46–01:09] 팁 2 — 이미지·영상 분석은 Gemini가 압도적
이미지 인식·영상 분석은 현재 Gemini 계열이 가장 잘한다고 한다(주장/미검증). 화자의 워크플로는 Gemini로 원본을 분석한 다음 그 결과를 Claude에 넘겨 작업하는 것. 예로 레퍼런스 웹사이트 스크린샷을 Gemini에 넣어 구조를 분석하고, 그 분석 결과를 Claude에 주면서 "이 구조로 만들어 줘"라고 하면 혼자 할 때보다 훨씬 정확한 결과물이 나온다. "AI 도구는 하나만 잘 쓰는 게 아니라 잘하는 걸 조합하는 것"이라는 원리를 재확인한다. [의견]

### [01:09–01:35] 팁 3 — 영상은 통째로 던지지 말고 Whisper로 텍스트화
영상을 통째로 AI에 던져 분석시키면 퀄리티가 많이 아쉽고, 특히 후킹 포인트 찾기는 현재 모델의 한계가 분명하다고 한다. 더 효과적인 방법으로 OpenAI Whisper API로 음성을 먼저 추출·텍스트 변환한 뒤 분석하는 루틴을 제시한다. 영상 자체를 주는 것보다 컨셉 분석 정확도가 완전히 다르다며, 영상 작업자는 이 루틴을 꼭 기억하라고 한다. [불명확: "을도파 컨셉" — 자동자막 오인식으로 정확한 표현 불명, 문맥상 후킹/컨셉 분석으로 추정]

### [01:36–01:56] 팁 4 — OpenAI는 평타는 좋지만 압도하는 영역이 없다
OpenAI 모델은 전반적으로 무난해 뭘 시켜도 어느 정도 하지만, 하나를 압도하는 영역이 없다는 평가(주장/미검증). 코딩은 Claude가 낫고 이미지·영상 분석은 Gemini가 나아서, 화자는 OpenAI를 메인보다 보조로 활용하는 편이라고 한다. Whisper 같은 특정 API는 정말 좋지만 코딩 메인으로 쓰기엔 아쉬운 부분이 많다는 단서를 단다. [의견]

### [01:57–02:13] 팁 5 — Google Stitch는 Pro와 Flash 차이가 크다
Google Stitch에는 Pro 모드와 Flash 모드가 있는데 결과물 차이가 꽤 크다고 한다(주장/미검증). 빠르게 테스트할 때는 Flash도 괜찮지만, 실제 프로젝트를 만들 거면 모델을 Pro로 사용하라—디자인 퀄리티가 다르다고 조언한다.

### [02:13–02:28] 팁 6 — Stitch엔 레퍼런스 URL 주지 말고 말로 묘사
Stitch에게 "이 홈페이지 참고해서 만들어 줘"라고 URL을 주면 잘 못 알아먹는다고 한다. 대신 웹페이지 레이아웃을 말로 묘사하고 컬러 코드를 정확하게 넣어 주라는 것. 이렇게 구체적으로 알려주면 결과물이 확 달라진다고 한다. [의견]

### [02:28–02:47] 팁 7 — Stitch도 Claude가 답
앞서 Gemini에 Claude를 경유하라고 한 것과 마찬가지로 Stitch도 그렇다고 한다. Claude에게 웹사이트 정보를 읽게 하고 그걸 영문 프롬프트로 만들어 Stitch에 명령하면, 직접 프롬프트를 쓰는 것보다 결과물이 훨씬 좋다. 정리하면 Stitch를 쓸 때는 레퍼런스 URL 대신 "Claude가 정리한 묘사 + 영문 프롬프트"가 현재 가장 효과적인 조합이라는 것. [의견]

### [02:47–03:06] 팁 8 — Stitch 결과물을 그대로 쓰지 말고 프로덕션 코드로 컨버팅
Stitch 결과물이 괜찮아 보여도 바로 실전에 쓰면 아쉬운 부분이 많다고 한다. 화자가 추천하는 루틴은 Stitch 결과물을 AI Studio에서 빌드·다운로드한 뒤, "Next.js 프론트엔드 가이드라인 스킬" 같은 것으로 컨버팅하는 것. 이렇게 하면 Stitch의 디자인 감각은 살리면서 실제 프로덕션 수준 코드로 올릴 수 있다고 한다. [불명확: 스킬 정확 명칭 — 자동자막 "넥스트 JS 프론트 핸드 가이드라인 스킬"]

### [03:08–03:25] 팁 9 — MUI보다 shadcn을 쓰라
바이브 코딩 시 UI 라이브러리로 MUI를 쓰는 사람이 많지만, shadcn이 훨씬 감각적인 디자인이 나온다고 한다(주장/미검증). AI가 shadcn 기반으로 코드를 생성하면 모던하고 깔끔한 결과물이 기본으로 나오고, 특별히 커스텀하지 않아도 "이거 괜찮은데" 하는 수준이 나온다고 한다. [의견] [불명확: "무고"/"무이" — MUI(Material UI)로 추정]

### [03:25–03:47] 팁 10 — 코드·디자인 체크를 스킬로 자동화
코드를 짜고 그냥 끝내지 말라고 한다. Vercel의 React best practices 스킬로 코드 품질을 체크하고, 웹 디자인 가이드라인 스킬로 디자인을 체크하면 퀄리티가 올라간다. 이 두 스킬을 마지막에 한 번씩 돌리는 습관만 들여도 완성도가 확실히 달라진다는 것. 두 스킬은 특정 사이트에서 다운로드할 수 있다고 언급한다. [불명확: 사이트명 — 자동자막 "skill.shh.com / shc.com", skills.sh 계열로 추정]

### [03:48–04:02] 팁 11 — 로고·미니멀 디자인은 Claude 웹이 최강
간단한 로고나 미니멀한 디자인이 필요하면 Claude 웹에서 바로 만들어 보라고 한다. 복잡한 일러스트는 어렵지만 심플하고 세련된 디자인은 Claude가 의외로 잘 뽑아준다는 것. 별도 디자인 툴을 안 켜도 되는 경우가 많다고 한다. [의견]

### [04:02–04:21] 팁 12 — 디자인 수정은 "미니멀하게"처럼 방향만
Claude Code로 디자인을 수정할 때 디테일한 지시를 하나하나 내리기보다 "미니멀하게 다듬어 줘"라고 방향만 잡아주면 꽤 만족스러운 결과가 나온다고 한다. AI에게 디자인을 시킬 때는 구체적인 수치보다 방향성을 주는 게 효과적인 경우가 있고, "깔끔하게 여백 살려서 미니멀하게" 같은 키워드가 생각보다 잘 먹힌다는 것. [의견]

### [04:25–04:46] 팁 13 — SNS 글은 AI로 통째로 쓰지 마라
