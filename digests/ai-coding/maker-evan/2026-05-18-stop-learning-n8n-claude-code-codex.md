# n8n 이제 안 배우셔도 됩니다

> maker-evan · 2026-05-18 · 길이 약 7분55초 · [영상 링크](https://youtu.be/IA9p_Fe7_oI)
> 정리 방식: 시간순 심층 워크스루 (paraphrase) + 주제별 상세 요약
> 검증 배너: 화자의 개인 경험·수치("10배", "90% 이상", "30분", "일주일")는 모두 화자 주장이며 미검증. [의견] 다수.

## 요약 (주제별)

### 핵심 주장
- 화자는 이제 n8n(자막상 "M8"·"엠파"로 오인식된 것)을 쓰지 않으며, Claude Code와 Codex 두 도구만으로 충분하다고 본다. [의견]
- 자동화 입문 도구로 n8n을 새로 배우는 것은 시간 낭비이고, 그 시간에 Claude Code를 깔아 워크플로를 만들어 보는 게 낫다는 것이 결론. [의견]

### 자동화 시장의 3단계 진화론
- 1단계: 실제 프로그래밍 언어로 자동화를 직접 코딩하던 시대.
- 2단계: 비주얼 워크플로 빌더(Zapier / Make / n8n) — 노드 드래그 앤 드롭으로 코드 없이 자동화. 화자도 이 단계(n8n)로 입문. "진짜 혁명"이라 평가.
- 3단계: 에이전틱 워크플로 — Claude Code 같은 도구에 한국어(자연어)로 지시하면 생성·테스트·디버깅까지 알아서 수행, 사람 개입 최소.
- 2→3단계 전환 속도가 매우 빠르고, 한번 3단계를 맛보면 2단계로 못 돌아간다. "워드프로세서 쓰다 타자기로 돌아가는 격"이라는 비유. [의견]

### 개인 경험(주장/미검증)
- 작년까지는 자동화 하나에 주말이 통째로 날아갔고(노드 연결·webhook 설정·에러 디버깅), 복잡한 건 며칠씩 걸렸다는 주장.
- 지금은 Claude Code에 한 문단 입력하고 커피 한 잔 마시고 오면 끝난다는 주장. "인테리어 도면을 직접 그리던 사람이 이제 '거실 북유럽 스타일로' 한마디만 던지는 격."
- 결과물이 n8n으로 만들던 것과 동급 혹은 더 정교하다는 주장. 6개월간 양쪽을 다 써본 뒤 내린 결론이라고 밝힘.

### 역사적 비유(도구 대체의 필연성)
- 1980년대 "컴퓨터 오퍼레이터"라는 전문직이 있었지만 누구나 할 수 있는 일이 되며 사라졌다.
- 1990년대 웹 개발자가 HTML을 한 줄씩 직접 짜던 시절엔 그게 "슈퍼파워"였지만, Squarespace 같은 도구가 나오며 초등학생도 웹사이트를 만든다.
- 자동화도 같은 길: n8n이 "AI로 만들기" 기능을 내놓는 것도 같은 흐름이며, 결국 누구나 자연어로 자동화를 만들게 된다. 그 단계로 가는 가장 빠른 길이 n8n이 아니라 Claude 같은 에이전틱 도구라는 주장. [의견]

### Claude Code vs Codex
- 요즘은 Claude Code뿐 아니라 Codex도 있고, 둘 다 자연어로 워크플로를 만들고 디버깅까지 해준다. 각자 강한 부분이 조금 다르다.
- n8n으로 만들던 워크플로의 90% 이상을 둘 중 하나로 다 만들 수 있다는 주장. 그래서 n8n에 의존할 이유가 점점 사라진다. [의견]

### 유지보수 비용 (화자가 꼽은 "제일 큰 이유")
- n8n은 만들 때는 노드가 직관적이라 재밌지만, 한두 달 뒤 API 스펙 변경·외부 서비스 응답 포맷 변화·노드 버전 업데이트 중 하나만 생겨도 워크플로가 멈추고, 노드를 하나하나 열어 어디서 데이터가 깨졌는지 찾아야 한다.
- 겉보기엔 비개발자가 쉽게 쓸 수 있어 보이지만, 버그 한 번 터지면 JSON 구조·API 응답 코드 의미·인증 토큰 갱신 로직을 알아야 하고 결국 개발자가 봐줘야 한다. "조립식 가구 — 조립은 누구나, 나사 빠지면 결국 전문가 호출" 비유. 비개발자가 끝까지 혼자 끌고 가는 걸 거의 못 봤다는 주장. [의견]
- Claude Code/Codex는 버그 나면 에러 메시지를 던지고 "이거 고쳐" 한 줄이면 알아서 분석·수정. 비개발자도 끝까지 혼자 운영 가능하다는 주장.

### "통제권/블랙박스" 오해 반박
- n8n 옹호론: "n8n은 각 단계가 다 보이는데 Claude Code는 블랙박스다."
- 반박: Claude Code도 실행 로그가 다 보여 어디서 뭐가 호출됐는지·응답이 뭔지 추적 가능하고, 에러 메시지를 그대로 던지면 원인을 찾아 고쳐준다. n8n 디버깅은 노드마다 클릭해 인풋/아웃풋·빨간 표시·JSON을 들여다봐야 해 오히려 더 번거롭다. "n8n=자동차 본네트 직접 열기, Claude Code=정비사에게 맡기기" 비유. [의견]

### 시작 방법 (3단계 실전 절차)
- 1단계: 지금 n8n에서 돌리는 것 중 가장 단순한 워크플로(예: 구글 폼 응답 오면 슬랙 알림) 하나를 골라, Claude Code에 한국어로 그대로 설명 → 30분 안에 돌아가는 결과를 받을 수 있다는 주장.
- 2단계: 익숙해지면 복잡한 것(API 여러 개 연결·조건 분기·데이터 가공)으로 확장. 결과물을 보며 동작 원리를 익힌다.
- 3단계: Codex 같은 다른 에이전틱 도구도 한 번씩 써보고 본인 작업 스타일에 맞는 걸 찾는다.
- 팁: 처음엔 노드가 없어 막막해 보일 수 있지만 일주일만 쓰면 적응하고, 그때부터 n8n이 답답해진다는 주장. [의견]

### 본질·마무리 메시지
- 비즈니스/클라이언트는 n8n을 쓰든 Claude Code를 쓰든 관심 없다. 오직 "내 문제를 해결해 줄 수 있는가"만 본다. 그러니 도구에 집착하지 말 것.
- 본질은 비즈니스 문제를 정의하고 가장 빠르게 해결하는 방법을 찾는 것인데, 그 가장 빠른 방법이 지금은 명백히 바뀌었다는 주장. n8n으로 며칠 걸리던 걸 Claude Code/Codex로 한 시간이면 끝내 시간 차이가 10배이고, 유지보수까지 더하면 차이가 더 벌어진다. (수치 주장/미검증)
- 결론: n8n은 적어도 입문 도구로는 더 안 배워도 되고, Claude Code+Codex 두 도구면 충분. 자동화 시장은 이미 3단계로 넘어왔고, 2단계에 남는 게 손해는 아니지만 비효율적이라는 것.

## 3줄 오리엔테이션
- 자동화 도구 시장을 "코딩 → 비주얼 빌더(n8n 등) → 에이전틱(Claude Code/Codex)" 3단계로 보고, 지금은 3단계로 넘어왔다는 것이 영상의 뼈대.
- 화자는 자기 경험·역사적 비유·유지보수 비용·통제권 오해 반박을 근거로 "n8n을 새로 배울 필요 없다"고 주장한다(대부분 개인 주장/미검증).
- 마지막엔 도구 집착을 버리고 "문제를 가장 빠르게 해결하는 도구"를 쓰라며 Claude Code부터 깔아 보라고 권한다.

## 전문 — 시간순 워크스루

### [00:00–00:19] 도입 — "n8n 2026년에도 배울 가치가 있나?"
자동화를 만드는 사람들에게 가장 많이 받는 질문이 "n8n을 2026년에도 배울 가치가 있냐"는 것이고 다들 헷갈려 한다고 운을 뗀다. 화자 자신도 한참 고민했지만 솔직한 입장을 밝히겠다며, 이제 자신은 n8n을 쓰지 않고 Claude Code와 Codex만 있으면 충분하다고 선언한다. 무엇이 달라졌는지 풀어주겠다고 예고한다.
(주: 자막의 "M8"·"엠파"·"엠파닝"은 모두 n8n의 오인식 표기로 보고 n8n으로 교정함.)

### [00:19–01:05] 자동화 시장의 3단계 진화
이쪽 일을 오래 하며 진화 단계가 뚜렷이 보인다고 말한다. 1단계는 프로그래밍 언어로 자동화를 직접 짜던 시절, 2단계는 Zapier·Make·n8n 같은 비주얼 워크플로 빌더로 코드 없이 노드를 드래그 앤 드롭하던 시절(화자도 n8n으로 입문, "진짜 혁명"). 3단계는 에이전틱 워크플로로, Claude Code 같은 도구에 한국어로 "이런 거 만들어 줘" 하면 생성·테스트·디버깅까지 알아서 하고 사람이 거의 안 끼어들어도 된다.
> "2단계에서 3단계로 넘어가는 속도가 진짜 빠릅니다. 그리고 한번 3단계 맛보면 2단계로 못 돌아가요." [01:01] [의견]

워드프로세서를 쓰다 타자기로 돌아가는 것과 같다는 비유로 되돌아갈 수 없음을 강조한다.

### [01:05–01:44] 개인 경험 — 주말 vs 커피 한 잔
작년까지는 자동화 하나 만들려면 주말이 통째로 날아갔다고 한다(n8n 켜놓고 노드 하나하나 연결, webhook 설정, 에러 디버깅, 복잡한 건 며칠). 지금은 Claude Code에 한 문단 입력하고 커피 한 잔 마시고 오면 끝나 있다는 것. 인테리어 도면을 직접 그리던 사람이 이제 "거실 북유럽 스타일로 해 주세요" 한마디만 던지면 도면이 나오는 격이라고 비유한다. 처음엔 될 리 없다고 의심했지만 직접 써보니 n8n에서 만들던 것과 동급, 아니 더 정교한 결과물이 나온다고 주장한다. (경험·품질 주장/미검증)

### [01:44–02:05] "그럼 n8n은 왜 배워?"
자연스럽게 떠오르는 이 질문이 궁금해 6개월 동안 양쪽을 다 써봤다고 한다. 결론은 명확하다며, n8n으로 만들 수 있는 것은 Claude Code로 거의 다, 그것도 더 빠르게 만들 수 있다고 말한다. [의견]

### [02:05–02:44] 역사 비유 — 사라진 직업들
"n8n에는 나만의 자리가 남을 것"이라는 반론에 대해 역사를 보면 답이 나온다고 한다. 1980년대엔 컴퓨터를 켜고 프로그램을 돌리는 "컴퓨터 오퍼레이터"가 전문직이었지만 누구나 할 수 있는 일이 되며 사라졌고, 1990년대엔 HTML을 한 줄씩 직접 짜는 게 슈퍼파워였지만 Squarespace 등이 나오며 초등학생도 웹사이트를 만든다. 자동화도 같은 길을 걷고 있고, n8n이 "AI로 만들기" 기능을 내놓는 것도 같은 흐름이며, 결국 누구나 자연어로 자동화를 만들게 되고 그 가장 빠른 길은 n8n이 아니라 Claude 같은 에이전틱 도구라고 주장한다. [의견]

### [02:44–03:19] Codex 등장과 "90% 이상 대체"
요즘은 Claude Code만 있는 게 아니라 Codex라는 도구도 있고, 둘 다 써봤는데 비슷한 일(자연어로 워크플로 생성·디버깅)을 하되 각자 강한 부분이 조금 다르다고 한다. 핵심은 n8n으로 만들던 워크플로의 90% 이상을 이 둘 중 하나로 다 만들 수 있다는 것(주장/미검증). 그래서 n8n에 의존할 이유가 점점 사라지며, 같은 결과를 더 빠르게 내주는 도구가 둘이나 있는 상황에서 굳이 노드를 드래그할 이유가 없다고 본다. [의견]

### [03:19–04:26] 유지보수 비용 — "제일 큰 이유일 수도"
n8n의 가장 큰 문제로 유지보수 비용을 든다. 만들 때는 노드가 직관적이라 재밌지만 한두 달 뒤가 문제다. API 스펙 변경, 외부 서비스 응답 포맷 변화, 노드 버전 업데이트 중 하나만 생겨도 워크플로가 멈추고, 다시 열어 노드를 하나하나 들여다보며 어디서 데이터가 깨졌는지·어떤 분기가 안 탔는지 찾아야 한다. 더 큰 함정은, 겉보기엔 비개발자가 쉽게 쓸 수 있어 보이지만 버그 한 번 터지면 JSON 구조 파악·API 응답 코드 의미·인증 토큰 갱신 로직이 필요해 결국 개발자가 와서 봐줘야 한다는 것.
> "비개발자분이 끝까지 혼자 끌고 나가는 거 솔직히 거의 못 봤어요." [04:13] [의견]

조립식 가구(조립은 누구나, 나사가 빠지면 결국 전문가 호출) 비유로 이를 정리한다. 반면 Claude Code/Codex는 버그가 나면 에러 메시지를 던지고 "이거 고쳤죠" 한 줄이면 알아서 분석·수정하므로 비개발자도 끝까지 혼자 운영할 수 있는 환경이 된다고 주장한다.

### [04:36–05:29] "통제권/블랙박스" 오해 반박
"그래도 n8n이 통제권이 더 좋지 않냐"는 생각은 오해라고 한다. n8n 옹호론은 "n8n은 각 단계가 다 보이는데 Claude Code는 블랙박스"라는 것인데, Claude Code도 실행 로그가 다 보여 어디서 뭐가 호출됐는지·어떤 응답이 왔는지 추적 가능하고, 에러가 나면 그 메시지를 그대로 던지며 "이거 왜 이래?" 한마디면 원인을 찾아 고쳐준다고 반박한다. n8n 디버깅은 노드를 하나하나 클릭하며 인풋/아웃풋 확인, 빨간색 표시 찾기, JSON 구조 들여다보기로 머리가 아프지만 Claude Code는 "왜 안 돼" 한 줄로 끝난다는 것.
> "엠파(n8n)는 자동차 본네트 본인이 열고 부품 하나 만져 보는 거예요. 클로드 코드는 정비사한테 차에서 이상한 소리 나요 하고 맡기는 거." [05:21] [의견]

### [05:29–06:14] 시작 방법 — 3단계 절차
어떻게 시작하냐는 질문에 단계별로 정리한다. 1단계: n8n에서 돌리는 것 중 가장 단순한 워크플로 하나(예: 구글 폼 응답이 오면 슬랙으로 알림)를 골라 Claude Code에 한국어로 그대로 설명하면("구글 폼에 응답 오면 슬랙 채널에 알림 보내는 스크립트 만들어 줘") 30분 안에 돌아가는 걸 받을 수 있다(주장/미검증). 2단계: 익숙해지면 API 여러 개 연결·조건 분기·데이터 가공 같은 복잡한 워크플로로 넘어가고, 만들어진 결과물을 보며 동작 원리를 익힌다. 3단계: Codex 같은 다른 에이전틱 도구도 한 번씩 써보고 자기 작업 스타일에 맞는 걸 찾는다.

### [06:14–06:24] 팁 — 일주일이면 적응
처음엔 노드가 없어 Claude Code가 더 어려워 보이고 막막할 수 있지만, 한 일주일만 쓰면 완전히 적응되고 그때부터는 n8n이 답답해서 못 돌아간다고 한다. [의견]

### [06:24–07:01] 가장 중요한 얘기 — 도구가 아니라 문제 해결
비즈니스와 클라이언트는 여러분이 n8n을 쓰는지 Claude Code를 쓰는지 관심이 없고, 오직 "내 문제를 해결해 줄 수 있는가" 하나만 본다고 강조한다. 그러니 도구에 집착하지 말라는 것. 어떤 도구를 쓰든 본질은 비즈니스 문제를 정의하고 그걸 빠르게 해결하는 방법을 찾는 것인데, 그 가장 빠른 방법이 지금은 명백히 바뀌었다고 한다. n8n으로 며칠 걸리던 걸 Claude Code/Codex로 한 시간이면 끝내 시간 차이가 10배이고, 유지보수 시간까지 잡으면 차이가 더 벌어진다는 주장. (수치 주장/미검증)

### [07:01–07:55] 정리 및 CTA
n8n은 더 이상 안 배워도 된다 — 적어도 자동화 입문 도구로는 시간 낭비이니, 그 시간에 Claude Code를 깔아 워크플로를 한 번 만들어 보는 게 100배 낫다고 정리한다(과장 표현/주장). Codex도 함께 보라며, 둘 다 자연어 기반이고 n8n으로 만들던 워크플로 대부분을 대체할 수 있어 도구 두 개면 충분하다고 한다. n8n은 만들 때는 쉬워 보여도 결국 유지보수에서 발목 잡히고 비개발자가 혼자 끌고 가기 어려운 반면, Claude Code/Codex는 만드는 것도 고치는 것도 빠르다고 재차 강조한다. 자동화 시장은 이미 3단계로 넘어왔고 2단계에 남는 게 손해는 아니지만 비효율적이니 한번 옮겨가 보길 추천한다. 마지막으로 시청자에게 어떤 도구를 쓰는지·왜 옮기는지 댓글로 알려달라 하고(다음 영상에서 다루겠다고 예고), 좋아요·구독을 요청하며 마친다.
