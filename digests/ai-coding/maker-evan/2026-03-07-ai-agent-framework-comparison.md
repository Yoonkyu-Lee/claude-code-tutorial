# 랭그래프 선택하셨나요? 저라면 안 씁니다

> 정리 방식: 시간순 심층 워크스루 (paraphrase) + 주제별 상세 요약
> 검증 배너: 프레임워크 평가·트렌드 전망은 대부분 화자 개인 판단([의견])이며 벤치마크 등 객관 수치 근거는 제시되지 않음. 아래에서 [의견]으로 구분.

## 요약 (주제별)

### 문제 인식: 프레임워크 선택 질문 자체가 잘못됐다
- 2025년에 AI 에이전트 프레임워크가 한꺼번에 쏟아졌다: LangChain, LangGraph, Google ADK, Pydantic AI, Claude Agent SDK 등.
- 겉보기엔 다 비슷하지만 속을 열면 완전히 다르다. 각자 다른 문제를 풀도록 만들어졌다 — 어떤 건 다양한 AI 서비스 연결, 어떤 건 AI 응답의 일관성, 어떤 건 AI에게 컴퓨터 작업 시키기가 목표.
- 그래서 "뭐가 제일 좋아요?"라는 질문 자체가 잘못됐다는 게 화자 입장. [의견]
- 아무거나 골라 시작했다가 6개월 뒤 갈아엎는 팀이 많다(주장/미검증). 나중에 갈아타기 어려운 선택을 지금 해버리면 후회 가능성이 높다는 게 핵심 메시지. [의견]

### LangChain — 관성으로 쓰일 뿐, 새 프로젝트엔 비추
- 검색·튜토리얼이 가장 많고 에이전트 입문 영상 대부분이 여기서 시작한다.
- ChatGPT가 처음 나왔을 때 AI를 뭔가에 붙이려면 연결 도구가 필요했는데 선택지가 없어 다들 LangChain으로 시작했고, 그 관성이 지금까지 이어진다.
- 실사용 시 벽에 부딪힘: 일부만 바꾸려 해도 알 수 없는 용어를 계속 파야 하고, 문제 발생 시 원인 위치를 찾기 어렵고, 업데이트하면 잘 되던 게 갑자기 안 되고, 공식 문서가 안 좋기로 유명하다. [의견]
- 결론: 새 프로젝트를 LangChain으로 시작하는 것은 비추. 이미 쓰고 있고 유지보수만 하는 경우가 아니라면 다른 걸 보라. [의견]

### LangGraph — 더 복잡해졌다, 나중에 고통
- LangChain 팀이 더 복잡한 걸 할 수 있게 업그레이드한 버전.
- 복잡함을 해결하려 만들었는데 오히려 배울 게 더 늘었고, 간단한 것 하나 만드는 데 단계가 너무 많다("달걀 하나 삶는데 설명서가 열 페이지"라는 비유). [의견]
- 그래프 구조 자체가 직관적이지 않다는 게 화자 생각. [의견]
- 결론: 복잡하고 나중에 고통받을 수 있음. 새 프로젝트에 비추. [의견]

### Google ADK — 아이디어는 좋으나 버그·커스터마이즈 벽·구글 신뢰 문제
- 2025년 구글이 공개. 여러 AI를 팀처럼 협업(역할 분담, 순차/동시 실행)시킨다는 아이디어 자체는 나쁘지 않다.
- 잘되는 부분: 사전 정보 가져오기(pre-fetch), 데이터 저장·불러오기가 생각보다 쉽게 붙고 예제 코드도 많다.
- 문제 1 — 커스터마이즈: 정해진 방식대로만 쓰면 되지만 조금이라도 내 양식대로 바꾸려 하면 막힌다. 억지 우회(개발자들이 말하는 monkey patch)를 여러 군데서 많이 해야 한다.
- 문제 2 — 버그: 기본 기능이 안 되거나 어제 되던 게 오늘 안 되는 일이 계속 난다(주장/미검증).
- 문제 3 — 더 근본적: 구글이 만들었다는 것 자체가 신뢰가 안 간다. 구글은 오픈소스를 만들어 놓고 조용히 손 떼버린 전적이 많아서 "구글이 만든 건 믿고 쓰기 무섭다"는 얘기가 개발자 사이에 돈다. ADK도 1~2년 뒤 어떻게 될지 장담 못 한다. [의견]

### Pydantic AI — 화자가 가장 기대하는 도구
- 목적: AI의 응답 결과를 정해진 포맷(형식)으로 받고 싶을 때.
- 만든 팀이 개발자 사이에서 신뢰받는, 좋은 도구 만드는 곳으로 알려져 있다. [의견]
- 해결하는 문제: AI가 매번 다른 형식으로 대답하는 것 — "이 형식으로만 대답해라"고 정해 놓으면 항상 그 형식으로 나오고, 이상하게 대답하면 알아서 다시 한다. 실서비스에 응답 형식 일관성이 필요할 때 유용.
- 써본 사람들 반응이 "드디어 이런 게 나왔다"는 분위기(주장/미검증).
- 단점: 최근 도구라 대형 서비스 적용 사례가 많지 않고, 막히면 검색해도 답이 잘 안 나오는 경향. 다만 문서(docs)가 매우 잘 돼 있어 AI에게 docs를 주고 맡기면 잘한다는 게 화자 평. [의견]

### Claude Agent SDK — 발상이 다르다("어떻게 만들까"가 아니라 "뭘 하게 할까")
- 다른 도구는 "AI를 어떻게 만들까"를, Claude Agent SDK는 "AI가 뭘 할 수 있게 할까"를 이야기한다.
- Claude Code의 능력(파일 열기·수정, 인터넷 검색, 결과 저장)을 그대로 쓴다. "이 폴더에서 문제 찾아서 보고서 만들어 줘"라고 하면 AI가 직접 파일을 읽고 문제를 찾아 보고서를 만든다. 기능을 따로 만들 필요 없이 내장돼 있다.
- 여러 AI 역할 분담 협업, 중간에 멈췄다 이어서 작업하기도 된다.
- 단점: Claude만 쓴다 — Anthropic 유료 계정 없으면 못 쓰고 다른 AI로 교체 불가. 내가 만든 기능을 붙이려면 번거로운 방식이 필요하고, 시작 난도가 다른 도구보다 높다.
- 적합 상황: 파일 작업·코드 관련 자동화. Claude Code 경험이 그대로 들어온다고 보면 된다.

### 최종 추천 매트릭스 (상황별)
- 빠르게 소규모 팀에서 만들어 보고 응답 일관성을 원함 → Pydantic AI.
- 여러 AI가 협업하는 큰 시스템을 실서비스에 올려야 함 → 검색하면 LangGraph가 나오겠지만 화자라면 안 씀(고치기 힘들고 팀 규모가 커야 유의미). 차라리 Pydantic AI로 직접 만드는 게 낫다. [의견]
- 파일·코드 작업 자동화, Claude 기반 → Claude Agent SDK.
- 특정 AI 회사에 묶이기 싫고 나중에 교체 가능성 있음 → Pydantic AI(어떤 AI든 갈아끼움).
- 아직 뭐가 필요한지도 모르겠음 → 아무 도구도 쓰지 말고 AI를 직접 연결해서 써 보라. 도구는 문제를 해결하기도 하지만 더 만들기도 한다. 무턱대고 에이전트 프레임워크부터 쓰는 건 비추. [의견]

### 트렌드 전망: TensorFlow → PyTorch 반복 서사
- 딥러닝 초창기 TensorFlow(구글)가 선택지 없어 다 썼지만 복잡·비직관·디버깅 난해했고, PyTorch가 나오자 자연스럽고 직관적이라 연구자·스타트업·대기업까지 다 넘어갔다.
- 지금 LangChain/LangGraph가 "TensorFlow 자리"라고 봄 — 먼저 나와 사용자 많고 대기업도 쓰지만 쓸수록 불편·비직관. [의견]
- 커뮤니티에 "우리 팀이 LangChain을 버린 이유", "LangChain 없이 직접 만드는 게 낫다" 류 글이 계속 올라온다 — 소수 의견이 아니라 TensorFlow 때와 같은 흐름이라는 주장.
- 승자는 아직 안 정해졌다. Pydantic AI가 그 자리를 차지할 수도, 아직 안 나온 뭔가가 나올 수도. 1~2년 안에 정해질 것이라는 게 화자 전망. [의견]
- 지금 확실한 하나: 나중에 갈아타기 어려운 선택을 지금 하면 후회한다 → 교체 쉬운 선택을 하라. [의견]

- 핵심 주장: "뭐가 제일 좋냐"는 질문이 틀렸다 — 각 도구는 다른 문제를 푼다. LangChain/LangGraph는 관성으로 쓰일 뿐 새 프로젝트엔 비추, ADK는 버그·구글 리스크, Pydantic AI가 가장 기대, 코드/파일 자동화는 Claude Agent SDK.
- 관통 서사: 지금 상황은 TensorFlow→PyTorch 판갈이와 똑같으며, 승자 미정이니 "나중에 갈아타기 쉬운 선택"을 하라.

## 전문 — 시간순 워크스루

### [00:00–00:18] 도입 — 프레임워크가 너무 많다
AI 에이전트를 만들려고 검색하면 LangChain, LangGraph, Google ADK, Pydantic AI, Claude Agent SDK 등이 쏟아진다. 화자는 솔직히 이 중 아무거나 골라 시작했다가 6개월 뒤 갈아엎는 팀이 엄청 많다고 말한다(주장/미검증). 이유는 이것들이 겉보기엔 다 똑같아 보이는데 속을 열면 완전히 다르기 때문. 이 영상을 보고 나면 자기 상황에 딱 맞는 게 뭔지 알 수 있다고 예고한다.

### [00:18–00:41] 문제의 본질 — 각자 다른 문제를 푼다
2025년에 AI 도구, 특히 에이전트 프레임워크가 한꺼번에 나왔다. 문제는 이것들이 각자 완전히 다른 생각으로 만들어졌다는 것이다. 어떤 건 다양한 AI 서비스를 연결하는 게, 어떤 건 AI 대답이 항상 일정하게 나오는 게, 어떤 건 AI에게 컴퓨터 작업을 시키는 게 목표다. 다 다른 문제를 풀고 있으므로 "뭐가 제일 좋아요?"라는 질문 자체가 잘못됐다고 볼 수 있다는 게 화자 입장. [의견]

### [00:41–01:02] LangChain (1) — 왜 다들 여기서 시작하나
가장 먼저 LangChain. 검색하면 제일 많이 나오고 튜토리얼도 제일 많으며 에이전트 입문 영상 대부분이 여기서 시작한다. 하지만 화자는 명확히 "비춘다(비추한다)"고 말한다.
> "명확하게 말할게요. 랭체인이 나온게 채치피pt가 처음 나왔을 때예요." [00:50]
그때는 AI 기능을 뭔가에 붙이려면 연결 도구가 필요했는데 선택지가 없어 다들 LangChain으로 시작했고, 그 관성이 지금도 이어진다는 것이다.

### [01:02–01:20] LangChain (2) — 실사용의 벽
실제로 쓰다 보면 벽에 부딪힌다. 이 부분만 좀 바꾸고 싶은데 알 수 없는 용어를 계속 파야 하고, 뭔가 잘못됐을 때 어디서 문제가 생겼는지 찾기가 너무 어렵다. 업데이트하면 어제까지 잘 되던 게 갑자기 안 되고, 공식 문서(설명서)도 안 좋기로 유명해 읽다가 중간에 포기하게 된다. 화자는 이게 한두 번이 아니며 LangChain 좀 써본 팀이라면 한 번씩 다 겪는다고 한다. [의견]

### [01:20–01:40] LangGraph (1) — 더 복잡해졌다
그래서 LangChain 팀이 LangGraph를 만들었다. 더 복잡한 걸 할 수 있게 업그레이드한 버전인데, 화자는 솔직히 더 복잡하다고 말한다. 복잡한 걸 해결하겠다고 만든 건데 그 과정에서 배울 것이 오히려 늘었고, 간단한 것 하나 만드는데 단계가 너무 많다.
> "달걀을 하나 삶는데 설명서가 열 페이지인 느낌이에요." [01:33] (자동자막 오인식 교정)
그리고 그래프 구조가 그렇게 직관적이지 않다는 게 화자 생각이다. [의견]

### [01:40–02:04] 비유 — TensorFlow와 PyTorch
화자는 비유를 든다. AI 하는 사람은 TensorFlow를 알 것이다. 딥러닝 초창기 도구인데 처음엔 선택지가 없어 다들 썼지만, 막상 써 보면 쓸데없이 복잡하고 잘못됐을 때 원인 찾기가 힘들고 전혀 직관적이지 않았다. 그 사이 PyTorch가 나왔고, 자연스럽고 직관적으로 쓸 수 있으며 뭔가 잘못되면 어디가 잘못됐는지 바로 보였다. 결국 연구자, 스타트업, 큰 회사들까지 다 PyTorch로 갔다 — 구글이 TensorFlow를 만들었는데도. [의견]

### [02:04–02:22] LangChain/LangGraph = 지금의 TensorFlow 자리
화자는 지금 AI 에이전트 도구들이 딱 그 상황이라고 본다. LangChain과 LangGraph가 지금 TensorFlow 자리라는 것 — 먼저 나왔고 쓰는 사람 많고 큰 회사도 있지만 쓸수록 불편하고 비직관적이다. 그래서 지금 LangChain이나 LangGraph로 새 프로젝트를 시작하는 건 비춘다. 이미 쓰고 있고 유지보수만 하는 경우가 아니라 새로 시작하는 거라면 다른 걸 보라고 추천한다. [의견]

### [02:22–02:42] Google ADK (1) — 아이디어와 잘되는 부분
그다음은 Google ADK. 2025년 구글이 공개한 도구다. 아이디어 자체는 나쁘지 않다 — AI를 여러 개 만들어 팀처럼 일하게 한다는 것으로, 각자 역할이 있고 어떤 건 순서대로, 어떤 건 동시에 일하게 할 수 있다. 솔직히 잘되는 부분도 있다: AI가 뭔가 하기 전에 미리 필요한 정보를 가져오거나, 데이터를 저장하고 불러오는 부분이 생각보다 쉽게 붙는다. 참고할 예제 코드도 꽤 많다.

### [02:42–03:08] Google ADK (2) — 커스터마이즈의 벽과 버그
문제는 조금이라도 내 양식대로 바꾸려 하면 막힌다는 것이다. 정해진 방식 그대로 쓰면 되지만 그 틀에서 조금만 벗어나려 하면 갑자기 안 되기 시작한다. 이걸 억지로 우회하는 걸 개발자들 사이에서 monkey patch(몽키패치)라고 부른다 — 원래 안 되는 걸 임시방편으로 꼬아서 억지로 되게 만드는 것. ADK를 쓰다 보면 이 작업을 한 군데가 아니라 여기저기서 엄청나게 많이 해야 한다. 게다가 버그가 진짜 많아서 기본 기능이 안 되거나 어제 됐는데 오늘 안 되는 일이 계속 난다(주장/미검증).

### [03:08–03:27] Google ADK (3) — 더 근본적 문제: 구글 신뢰
화자는 이보다 더 근본적인 문제를 든다. 구글이 만들었다는 것 자체가 신뢰가 안 간다는 것이다. 구글은 오픈소스 도구를 만들어 놓고 나중에 조용히 손 떼버린 전적이 많아서, 잘 쓰고 있었는데 갑자기 지원 안 한다고 하는 일이 있었다. 이게 워낙 유명해 "구글이 만든 건 믿고 쓰기 무섭다"는 얘기가 개발자 사이에 꽤 돈다. ADK도 지금은 열심히 업데이트되지만 1~2년 뒤 어떻게 될지 장담 못 한다는 게 화자 판단. [의견]

### [03:27–04:01] Pydantic AI — 가장 기대하는 도구
그다음은 Pydantic AI. AI의 대답 결과가 포매팅된(정해진 형식의) 걸 받고 싶다면 이걸 추천한다. 화자가 제일 기대하는 도구인데, 만든 팀이 개발자들 사이에서 신뢰받는, 좋은 도구 만드는 곳으로 알려져 있기 때문이다. [의견] 이 도구가 푸는 문제는 AI가 매번 다른 형식으로 대답하는 것 — 오늘은 이렇게, 내일은 저렇게 대답하면 실서비스에 쓰기 곤란하다. Pydantic AI는 "이 형식으로만 대답해라"고 딱 정해 놓으면 항상 그 형식으로 나오고, 이상하게 대답하면 알아서 다시 한다. 써본 사람들 반응이 "드디어 이런 게 나왔다"는 분위기다(주장/미검증).

### [04:01–04:15] Pydantic AI — 단점
단점은 최근에 나온 거라 실제 큰 서비스에서 써 봤다는 사례가 많지 않다는 것이다. 뭔가 막히면 검색해도 답이 잘 안 나오는 경향이 있다. 다만 docs(문서)가 굉장히 잘 돼 있어서 AI에게 이 docs를 주고 알아서 하라고 하면 굉장히 잘한다고 화자는 생각한다. [의견]

### [04:15–04:47] Claude Agent SDK — 발상이 다르다
그다음은 Claude Agent SDK. 이건 나머지 셋과 생각 자체가 다르다. 다른 도구들은 "어떻게 AI를 만들까"를 얘기하는데, Claude Agent SDK는 "AI가 뭘 할 수 있게 할까"를 얘기한다. Claude Code를 써 봤다면 알듯이, AI가 직접 파일을 열고 수정하고 인터넷 검색하고 결과를 저장하는 그 능력을 그대로 쓴다. "이 폴더에서 문제 찾아서 보고서 만들어 줘"라고 하면 AI가 직접 파일을 열고 내용을 읽고 문제를 찾아 보고서를 만든다. 따로 기능을 만들어 줄 필요 없이 다 내장돼 있다. 여러 AI를 역할별로 나눠 협업시키는 것도, 중간에 멈췄다 이어서 작업하는 것도 된다.

### [04:47–05:01] Claude Agent SDK — 단점과 적합 상황
단점은 Claude만 쓴다는 것이다. Anthropic 유료 계정 없으면 못 쓰고 다른 AI로 바꿀 수 없다. 그리고 내가 만든 기능을 붙이려면 좀 번거로운 방식으로 만들어야 하고, 다른 도구들보다 시작하기 어렵다. 언제 쓰기 좋냐면 파일 작업이나 코드 관련 자동화를 할 때다. Claude Code 경험이 그대로 들어온다고 보면 된다.

### [05:01–05:42] 상황별 최종 정리
화자가 상황별로 정리한다.
- 빠르게 소규모 팀에서 만들어 보고 대답이 항상 일정하게 나오길 기대 → Pydantic AI.
- 여러 AI가 협업하는 큰 시스템을 실서비스에 올려야 함 → 검색하면 LangGraph가 나오겠지만 화자라면 안 쓴다. 나중에 고치기 힘들고 팀 규모가 커야 유의미하다고 본다. 차라리 Pydantic AI로 직접 만드는 게 낫다. [의견]
- 파일·코드 작업 자동화 → Claude Agent SDK. 다른 도구에선 이런 기능을 직접 다 만들어야 하는데 여기선 그냥 있다.
- 특정 AI 회사에 묶이기 싫고 나중에 바꿀 수도 있음 → Pydantic AI. 어떤 AI든 갈아끼울 수 있다.
- 아직 뭐가 필요한지도 모름 → 아무 도구도 쓰지 말고 AI를 직접 연결해서 써 보라. 도구는 문제를 해결하기도 하지만 더 만들기도 한다. 처음부터 무턱대고 에이전트 프레임워크를 쓰는 건 추천 안 한다. [의견]

### [05:42–06:23] 트렌드 — TensorFlow 서사의 반복
트렌드 얘기. LangChain이 지금도 많이 쓰이는 건 사실이지만, 좋아서가 아니라 먼저 나와서 다들 거기서 시작했기 때문이다. AI 개발자 커뮤니티에는 요즘 "우리 팀이 LangChain을 버린 이유", "LangChain 없이 직접 만드는 게 낫다" 같은 글이 계속 올라온다. 화자는 이게 소수 의견이 아니라 TensorFlow 때와 똑같이 흘러가는 것이라고 본다. TensorFlow 아니면 바보라는 분위기가 있었지만 결국 PyTorch가 다 뒤집었고, 지금 LangChain/LangGraph가 딱 그 자리라는 것이다.
> "나중에 갈아타기 어려운 선택을 지금 해 버리면 후회할 가능성이 높아요." [06:20] [의견]
솔직히 그 자리를 Pydantic AI가 차지할 수도 있고 아직 안 나온 뭔가가 나올 수도 있다.

### [06:23–06:52] 요약 결론
정리. LangChain은 새 프로젝트에선 쓸 이유가 없어 보인다. LangGraph는 복잡하고 나중에 고통받을 수 있다. ADK는 버그 많고 구글이 언제 손 뗄지 모른다. 지금 새로 시작한다면 — 파일·코드 자동화나 Claude 기반이면 Claude Agent SDK, 그 외 대부분은 Pydantic AI를 추천한다. 하나 더, 지금 이 판은 승자가 아직 안 정해진 상태다. 1~2년 안에 AI 에이전트 도구 하면 이거라는 게 정해지지 않을까 생각한다. 그게 Pydantic AI일 수도, 완전히 새로운 뭔가일 수도 있다. 그때까지는 나중에 바꾸기 쉬운 선택을 해야 한다. [의견]

### [06:52–07:04] 마무리 CTA
도움됐으면 좋아요 한 번 눌러 달라. 다음엔 Pydantic AI로 실제 서비스를 만드는 영상을 만들어 볼까 한다. 댓글에 지금 뭘 쓰고 있는지, LangChain 쓰다 힘들었던 경험이 있으면 남겨 달라, 꼭 읽어 보겠다.
