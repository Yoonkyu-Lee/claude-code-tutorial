## 요약 (주제별)

### 문제 정의 — AI의 "기억 없음"
- 대부분의 AI 사용 방식(파일 올리고 → 질문 → 답 받고 → 채팅창 닫기)은 지식이 전혀 축적되지 않는다. 다음날 같은 질문을 하면 AI는 자료를 처음부터 다시 읽어야 한다.
- 자료 20개가 있으면 매 질문마다 20개를 전부 다시 읽는다. 어제 읽은 것을 기억하지 못한다.
- 많이 읽을수록 토큰이 더 소모되고("돈이 나간다"), 자료가 많아질수록 더 느려진다.

### 카파시의 LLM 위키 개념 (핵심)
- 테슬라 AI 팀 출신 카파시가 공유한 해법으로, 이름은 "LM 위키"(LLM 위키). [불명확: 자동자막 "LM 위키"]
- 비유: 도서관 사서. 책이 들어오면 사서가 읽고 요약 카드(어떤 내용인지, 어떤 책과 연결되는지, 핵심이 뭔지)를 만든다. 이후 질문이 오면 책 전체를 다시 읽지 않고 요약 카드만 본다.
- LLM 위키가 바로 이것 — AI가 파일을 매번 처음부터 읽는 대신, 한 번 읽은 내용을 위키피디아처럼 정리해 둔다. 개념마다 페이지가 있고 페이지끼리 연결된다.
- 새 자료가 들어오면 AI가 그것을 읽고 기존 위키 페이지를 업데이트하거나 새 페이지를 만든다. 이후 질문 시 AI는 원본을 다시 읽지 않고 위키만 본다.

### 카파시가 강조한 "유지보수 비용 거의 제로"
- 카파시의 핵심 문장(주장): 유지보수 비용이 거의 제로에 가깝다.
- 근거: 인간은 위키 유지(내용 업데이트, 앞뒤 안 맞는 내용 정리, 새 내용 반영)에 지쳐 포기하지만, AI는 지치거나 귀찮아하지 않는다. 파일 15개를 한 번에 업데이트해도 불평하지 않는다.
- [의견] 인간이 포기했던 일을 AI가 대신하기 때문에 이 방식이 실제로 작동한다.

### 카파시 방식의 구조 (3요소)
- 원본 파일 / AI가 만든 위키 / 위키 관리법을 알려주는 CLAUDE.md — 이 셋이면 AI가 알아서 위키를 운영한다.
- 실제 구축: `raw` 폴더에 원본(기사·논문·이미지 등)을 넣고, `wiki` 폴더에 AI가 만든 위키 파일이 들어간다. CLAUDE.md에 "새 파일이 들어오면 위키 업데이트"라고 적어 두면 끝.
- 사용 시: AI에게 "이거 처리해줘" 하면 AI가 위키 10~15개(페이지)를 알아서 업데이트한다. 다음 질문 시 AI는 `INDEX.md`(목차)를 먼저 보고 관련 페이지만 읽어 답한다.

### 카파시 방식의 한계
- 질문할 때마다 AI가 원본 파일을 전부 뒤져야 한다 — 목차 읽고, 관련 파일 찾고, 그 파일 전체를 읽는 과정이 매 질문마다 반복된다.
- 소스가 많아질수록 여전히 느려지고 토큰도 그만큼 소모된다.

### Graphify(그래피파이) — 자동 지식 지도 도구
- 파일들을 한 번 자동 분석해 "지식 지도"를 만들어 주는 도구. [불명확: 자동자막 "그래피파이"]
- 두 가지 분석: (1) 구조 분석 — 파일 안 내용을 자동으로 훑음, 빠르고 정확. (2) 의미 분석 — 서브 에이전트(AI 보조 일꾼) 여러 개가 동시에 문서를 읽어 개념 간 관계를 뽑아냄. 문서뿐 아니라 이미지도 분석.
- 두 분석을 합치면 지식 지도가 나오고 `graph.json`(그래프다.제이슨)과 위키 폴더가 자동 생성된다. [불명확: 자동자막 "그래프다.제이슨" → graph.json 추정]
- 캐시 재사용: 한 번 만든 지식 지도는 재사용되며, 파일이 바뀌면 바뀐 파일만 다시 처리하고 변경이 없으면 기존 지도를 그대로 쓴다(전체 재처리 불필요).

### Graphify의 토큰 절감 수치 (주장/미검증)
- 원본 파일 7개를 직접 읽으면 엄청난 양을 AI가 한 번에 처리해야 하지만, Graphify 방식은 전체 요약 한 페이지 + 관련 노드 2~3개만 읽는다. (노드 = 개념 하나하나를 담은 카드)
- 최대 71.5배 절감할 수 있다는 보고가 있다(주장/미검증). [불명확: 자동자막에서 "71.5배 5배"로 들려 정확한 수치 확정 불가]
- 가능한 이유: `graph_report`(그래프 리포트)라는 파일에 전체 지식 지도가 한 페이지로 요약돼 있어(어떤 개념이 핵심인지, 무엇이 묶여 있는지, 어떤 질문에 잘 답할 수 있는지), AI가 요약 페이지를 먼저 보고 관련 노드 2~3개만 읽는다. 이는 카파시의 "한 번 정리한 걸 계속 재사용" 개념과 맞물린다.

### Graphify의 관계·신뢰도·그룹핑 (차별점)
- LLM 위키는 페이지, Graphify는 노드(개념과 관계) 단위.
- 관계 지도: 단순 연결이 아니라 관계의 종류("이걸 바탕으로 만들었다", "이걸 참조한다", "이게 근거다")까지 표현 → AI가 왜 그런지, 근거가 뭔지까지 답할 수 있다.
- 신뢰도 표시: 원본에 명시된 내용은 확실(높음), 합리적으로 추론한 것은 중간, 불확실한 것은 낮음. AI가 "이건 확실한 내용", "이건 추론한 내용"이라고 구분해 답할 수 있다.
- 자동 그룹핑: 비슷한 개념끼리 그룹을 만들어, 질문이 오면 관련 그룹만 보고 전체를 볼 필요가 없다.

### 복리 성장 (주장)
- graph.json에 새 노드가 기존 지도에 연결되며 전체가 풍부해진다. 자료가 쌓일수록 AI 답변 품질이 올라간다 → "복리 성장". (주장)

### 사용 시 주의점 (프롬프팅)
- Graphify 사용 시 CLAUDE.md에서 클로드에게 `grep` 같은 명령어를 쓰지 말라고 요청해야 한다. [불명확: 자동자막 "그래 명령어" → grep 추정]
- 무조건 graph.json 파일을 읽고 관련 노드만 읽으라고 명령해야 한다.
- CLAUDE.md에 이런 세세한 프롬프팅만 있으면 이후 프로세스는 문제없이 작동한다.

### 최종 정리 & CTA
- 카파시 LLM 위키 개념이 핵심 — 위키를 만들어 재사용. Graphify가 그것을 자동화(파일 분석 → graph.json 지식 지도).
- 토큰 최대 71.5배 절감(주장/미검증), 속도 향상, 자료가 쌓일수록 답변 품질이 복리로 상승. CLAUDE.md 한 줄이면 AI가 알아서 지도부터 보고 답한다. 구현은 폴더 + 텍스트 파일이 전부.
- CTA: 지금 AI로 어떤 작업(회사 문서 정리 / 개인 공부 노트 / 책 정리)을 하는지 댓글로 남기면 같이 얘기해 보겠다.

## 3줄 오리엔테이션
- AI가 매번 자료를 처음부터 다시 읽어 지식이 안 쌓이는 문제를, 카파시의 "LLM 위키"(한 번 읽은 걸 위키로 정리해 재사용)로 푼다.
- 카파시 방식은 개념은 좋지만 질문마다 원본을 뒤지는 한계가 있어, 이를 자동화·최적화한 Graphify(파일을 분석해 graph.json 지식 지도 생성)를 소개한다.
- 결과적으로 토큰을 크게 줄이고(최대 71.5배, 주장), 자료가 쌓일수록 답변 품질이 복리로 올라가는 "나만의 지식 비서"를 만들 수 있다는 것이 요지.

## 전문 — 시간순 워크스루

### [00:00–00:26] 문제 제기 — AI는 매번 처음부터
AI에게 오늘 물어보면 잘 대답하지만 다음날 같은 걸 물으면 또 처음부터 설명해야 한다. 맥락도 기억도 없이 매번 새로 시작이라는 것이 문제다. 화자는 이 문제를 근본적으로 해결하는 방법이 있으며, 이 영상을 보면 "AI가 내 모든 걸 기억하는 나만의 지식 비서"를 만들 수 있다고 예고한다. 지금 대부분의 사람이 AI를 쓰는 방식(파일 올리고 → 질문 → 답 받고 → 채팅창 닫기)은 비효율적이며, 이 과정이 반복되면 지식이 전혀 쌓이지 않는다고 지적한다.

### [00:26–00:47] 토큰·속도 비용
구체적 예시로, 회사 자료가 20개 있을 때 "이 프로젝트 어떻게 진행돼?"라고 물으면 AI는 자료 20개를 전부 읽어야 한다. 다음날 같은 질문을 하면 어제 읽은 걸 기억하지 못하고 또 20개를 전부 읽는다. 이것이 토큰 소모로 이어지고("AI가 많이 읽을수록 토큰이 날아간다 = 돈이 나간다"), 자료가 많아질수록 더 느려진다.

> "AI가 많이 읽을수록 토큰이 더 날아가거든요. 돈이 나간다고 생각하면 돼요." [00:41]

### [00:47–01:11] 카파시의 LLM 위키 — 도서관 사서 비유
테슬라 AI 팀 출신 카파시가 이 문제의 해법을 공유했는데, 이름이 "LM 위키"(LLM 위키)라고 한다. 개념은 매우 단순하다. 비유로 도서관 사서를 든다 — 책이 들어오면 사서가 그 책을 읽고 요약 카드(어떤 내용인지, 다른 어떤 책과 연결되는지, 핵심이 뭔지)를 만든다. 이후 누가 그 주제를 알고 싶어 하면 사서는 책 전체를 다시 읽지 않고 요약 카드만 본다. LLM 위키가 딱 이것이다.

### [01:11–01:35] 위키가 스스로 갱신되는 구조
AI가 파일을 읽을 때마다 처음부터 읽는 게 아니라, 한 번 읽은 내용은 위키 형태로 정리해 둔다 — 위키피디아처럼 개념마다 페이지가 있고 페이지끼리 연결된다. 새 자료가 들어오면 AI가 그걸 읽고 기존 위키 페이지를 업데이트하거나 새 페이지를 만들고, 기존 페이지에 내용을 추가하기도 한다. 나중에 질문할 때 AI는 원본 파일을 다시 읽을 필요 없이 위키만 보면 된다.

### [01:35–01:54] "유지보수 비용 거의 제로" — AI는 안 지친다
카파시가 말한 핵심 문장은 유지보수 비용이 거의 제로에 가깝다는 것이다. 이유는, 인간은 위키를 유지(내용 업데이트, 앞뒤 안 맞는 내용 정리, 새 내용 반영)하다 지쳐서 포기하는데 AI는 지치지도 귀찮아하지도 않기 때문이다. 파일 15개를 한 번에 업데이트해도 불평하지 않는다.

> "유지 보수 비용이 거의 제로에 가깝다." [01:36]

[의견] 인간이 포기했던 일을 AI가 대신해 주기 때문에 이 방식이 실제로 작동한다.

### [01:54–02:07] 카파시 방식의 3요소 구조
구조도 간단하다 — 세 가지다. 원본 파일, AI가 만든 위키, 그리고 AI에게 어떻게 위키를 관리할지 알려주는 CLAUDE.md. 이 셋(특히 CLAUDE.md 하나)으로 AI가 알아서 위키를 운영한다.

### [02:07–02:21] 카파시 방식의 한계 — 매 질문마다 원본 뒤짐
다만 여기서 멈추면 아쉽다. 카파시 방식은 개념은 좋지만 문제가 하나 있다 — 질문할 때마다 AI가 원본 파일을 전부 뒤져야 한다. 목차 읽고, 관련 파일 찾고, 그 파일 전체를 읽는 이 과정이 매 질문마다 반복된다. 소스가 많아질수록 더 느려지고 토큰도 그만큼 소모된다.

### [02:21–02:46] Graphify 등장 — 두 가지 분석
그래서 Graphify(그래피파이)가 등장한다. 파일들을 한 번 자동으로 분석해 지식 지도를 만들어 주는 도구다. 두 가지 방식으로 분석한다. 첫 번째는 구조 분석으로, 파일 안 내용을 자동으로 쭉 훑으며 빠르고 정확하다. 두 번째는 의미 분석으로, 서브 에이전트 여러 개가 동시에 문서를 읽어 개념들 사이의 관계를 뽑아낸다. 문서뿐 아니라 이미지도 분석한다. (서브 에이전트는 쉽게 말해 AI 보조 일꾼들이다.) 이 두 가지를 합치면 지식 지도가 나온다.

### [02:46–02:58] graph.json 자동 생성 & 증분 처리
분석 결과로 graph.json과 위키 폴더가 자동으로 생긴다. 한 번 만든 지식 지도는 다음에 또 쓸 수 있고, 파일이 바뀌면 바뀐 파일만 다시 처리하며, 바뀐 게 없으면 그냥 기존 지도를 쓴다. 전체를 처음부터 다시 처리하지 않아도 된다.

### [02:58–03:16] 실제 숫자 — 토큰 절감 (주장/미검증)
여기서 실제 숫자가 나온다. 원본 파일 7개를 직접 읽으면 엄청난 양의 내용을 AI가 한 번에 처리해야 하지만, Graphify 방식으로 하면 전체 요약 한 페이지를 보고 관련 노드 2~3개만 읽는다. (노드는 개념 하나하나를 담은 카드라고 생각하면 된다.) 최대 71.5배 절감할 수 있다는 보고가 있다(주장/미검증).

### [03:16–03:38] 왜 가능한가 — graph_report 요약 페이지
가능한 이유는 "그래프 리포트"라는 파일이 있어서다. 여기에 전체 지식 지도가 한 페이지로 요약돼 있다 — 어떤 개념이 핵심인지, 어떤 것들이 묶여 있는지, 어떤 질문에 잘 대답할 수 있는지까지 적혀 있다. AI는 질문을 받으면 이 요약 페이지를 먼저 보고, 그다음 관련 노드 2~3개만 읽는다. 파일 전체를 다 읽을 필요가 없다. 이것이 카파시가 말한 위키 개념(한 번 정리한 걸 계속 재사용)과 딱 맞는다. LLM 위키는 페이지, Graphify는 노드(개념과 관계)라는 차이가 있다.

### [03:38–03:56] 관계·신뢰도 표시
Graphify가 중요한 이유는 개념들 사이의 "관계"로 찾기 때문이다 — 관계 지도다. 관계에도 종류가 있어서 "이걸 바탕으로 만들었다", "이걸 참조한다", "이게 근거다" 같은 식으로 표현한다. 단순히 연결돼 있다가 아니라 어떤 관계인지까지 알 수 있어, AI가 왜 그렇게 되는지·근거가 뭔지까지 답할 수 있다. 신뢰도 표시도 있다 — 원본에 명시된 내용은 확실(높음), 합리적으로 추론한 것은 중간, 불확실한 것은 낮음. 그래서 AI가 답할 때 "이건 확실한 내용, 이건 추론한 내용"이라고 구분해 말할 수 있다.

### [03:56–04:24] 자동 그룹핑 & 복리 성장
관련 있는 개념들을 자동으로 묶어 준다 — 비슷한 것끼리 그룹을 만든다. 덕분에 AI가 질문을 받으면 관련 그룹만 보고 전체를 볼 필요가 없다. 또한 graph.json은 새 노드가 기존 지도에 연결되면서 전체가 풍부해진다. 자료가 쌓일수록 AI 답변 품질이 올라간다 — 이것이 "복리 성장"이다.

### [04:24–04:53] 카파시 방식 구축 실연
실제로 어떻게 구축하는지 대강 보여준다. 카파시 방식은 구조가 진짜 단순해서 폴더와 텍스트 파일만 있으면 된다. `raw` 폴더에 원본 파일(기사·논문·이미지 뭐든)을 넣고, `wiki` 폴더에 AI가 만든 위키 파일이 들어간다. CLAUDE.md에는 "새 파일이 들어오면 위키 업데이트"라고 적어 두면 끝이다. (예: 자료를 읽어 파일로 저장하고 AI에게 "이거 처리해줘"라고 하면, AI가 위키 페이지 10~15개를 알아서 업데이트한다.) 다음에 질문하면 AI는 `INDEX.md`(어떤 페이지들이 있는지 목차)를 먼저 보고, 관련 페이지만 읽어 답한다.

### [04:53–05:11] Graphify를 더하면
여기에 Graphify를 더하면, 파일이 많거나 이미지가 섞여 있을 때 Graphify가 자동으로 지식 지도를 만들어 준다. graph.json과 위키 폴더가 생기고 전체 요약 페이지가 생긴다. 이걸 CLAUDE.md에 연결하면 된다 — "질문 전에 요약 페이지 먼저 봐"라는 한 줄이면 된다. 그러면 AI가 매 질문 전에 자동으로 지도부터 탐색한다.

### [05:11–05:23] 완성 시 얻는 것 — 나만의 지식 비서
이게 완성되면, 내가 3개월 동안 읽은 자료들을 AI가 전부 위키로 정리하고, 그 위키 기반으로 질문에 답하며, 새 자료가 추가되면 자동으로 업데이트된다. 이것이 "AI가 기억하는 나만의 지식 비서"다.

### [05:23–05:43] 주의점 — CLAUDE.md 프롬프팅
다만 주의할 점이 있다. Graphify 사용 시 클로드에게 `grep` 같은 명령어를 쓰지 말라고 요청해야 하고, 무조건 graph.json 파일을 읽고 관련 노드만 읽으라고 명령해 줘야 한다. CLAUDE.md에 이런 세세한 프롬프팅만 있으면 그 뒤 프로세스들은 문제없이 작동하는 것을 확인할 수 있다. [불명확: 자동자막 "그래 명령어" → grep 추정]

### [05:44–06:11] 최종 정리 & CTA
정리하면, 카파시의 LLM 위키 개념이 핵심으로 위키를 만들어 재사용하는 것이고, Graphify가 그것을 자동화해 파일을 분석해 graph.json 지식 지도를 만든다. 토큰은 최대 71.5배 줄고(주장/미검증) 속도가 빨라지며, 자료가 쌓일수록 AI 답변 품질이 복리로 올라간다. CLAUDE.md 한 줄이면 AI가 알아서 지도부터 보고 답하고, 구현도 어렵지 않다(폴더와 텍스트 파일이 전부). 마지막으로 화자는 시청자에게 지금 AI로 어떤 작업(회사 문서 정리, 개인 공부 노트, 책 정리)을 하는지 댓글로 남기면 같이 얘기해 보겠다고 CTA를 남긴다.
