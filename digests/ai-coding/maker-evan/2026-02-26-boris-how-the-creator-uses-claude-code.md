# 클로드코드 직접 만든 사람이 공개한 사용법, 저도 충격받았습니다

> maker-evan · 2026-02-26 · 길이 약 7분 · [영상 링크](https://youtu.be/2WJzwwvzbBQ)
> 정리 방식: 시간순 심층 워크스루 (paraphrase) + 주제별 상세 요약

## 요약 (주제별)

### 보리스의 핵심 5원칙
영상은 Anthropic의 클로드 코드 총책임자 "보리스"가 공개한 사용법 중 핵심 5가지를 화자가 재정리한 것이다. 화자가 정리한 다섯 가지는: (1) 스스로 확인하게 해라, (2) 계획을 먼저 세워라, (3) 동시에 여러 개를 돌려라, (4) 클로드 전용 메모장에 투자해라, (5) 단축 명령어를 써라. [의견] 화자는 "가장 공감이 됐던 것부터" 순서로 소개한다.

### 동시 다중 실행 (multi-instance orchestration)
- 보리스는 클로드 코드를 하나만 쓰지 않고 화면 창 1~5번을 열어 각 창에 클로드 코드를 하나씩 켜서 동시에 다섯 개를 돌린다 (주장/미검증).
- 창이 서로 뒤엉키지 않도록 창마다 독립된 작업 공간을 따로 만들어 각자 자기 영역에서만 일하게 한다.
- 여기에 더해 웹브라우저 창도 5~10개를 추가로 열고, 아침에 폰으로 시작한 작업을 컴퓨터에서 이어받기도 한다 (주장/미검증).
- 여러 개를 돌릴 때 어느 클로드가 뭘 하는지 알기 위해 알림 설정을 해둔다. 클로드가 다음 지시를 기다리는 상태가 되면 알림이 오고, 사람은 그 창으로 가서 지시를 주고 다른 창으로 넘어간다. [의견] 화자는 이를 "사람이 여러 클로드를 오케스트라처럼 지휘하는 것"에 비유.

### 계획 먼저 세우기 (plan mode)
- 보리스는 바로 시키지 않고 먼저 플랜 모드로 전환한다. 이 모드에서 클로드는 아무것도 건드리지 않고 전체 상황을 분석해 플랜만 짠다 (집 짓기 전 설계도 비유).
- 보리스 인용(화자 전달): "계획의 에너지를 쏟아라. 그래야 클로드가 작업을 한 번에 끝낸다."
- 막힐 때도 계획 모드를 쓴다. 한 클로드가 계획을 세우면 다른 클로드가 경험 많은 선임처럼 그 계획을 비판적으로 검토하게 해, 실제 작업 전에 문제를 미리 찾는다.

### 스스로 확인하게 하기 (self-verification) — 보리스가 꼽은 최중요
- 클로드에게 항상 자기 작업을 스스로 확인할 수 있는 방법을 줘라. 작업을 마친 뒤 맞는지 틀린지 스스로 점검하게 하라는 것 (김치찌개 끓인 요리사가 맛을 봐야 한다는 비유).
- 확인 방법을 안 주면 틀린 줄 모르고 "완료했습니다"라고 해버리고, 주면 스스로 고친다.
- 보리스 인용(화자 전달): "클로드에게 확인 방법을 주면 최종 결과물 품질이 2~3배 향상된다" (주장/미검증).
- 확인 방법은 작업 성격에 따라 다르다: 간단한 건 결과를 직접 실행, 복잡한 건 브라우저를 직접 켜서 화면이 제대로 나오는지 확인. 보리스 팀은 클로드가 직접 브라우저를 열고 화면을 확인하며 될 때까지 반복 수정하기도 한다 (주장/미검증).

### 모델 선택 — 가장 똑똑한 버전을 최대 성능으로
- 클로드 코드 모델에는 빠르고 가벼운 버전과 느리지만 더 똑똑한 버전이 있다. 보리스는 가장 똑똑한 버전을 최대 성능으로 쓴다.
- 보리스 인용(화자 전달): "결국에는 더 빨라요. 왜냐면 덜 고쳐도 되거든요." 싸고 빠른 버전으로 여러 번 수정하는 것보다 처음부터 똑똑한 버전을 한 번 쓰는 게 결국 빠르다는 논리 (주장/미검증).

### 클로드 전용 메모장 ("봉리 학습")
- 프로젝트 폴더에 클로드에게 주는 메모 파일을 하나 만들어 팀 작업·규칙·방향, 그리고 가장 중요한 "클로드가 반복해서 틀리는 실수 목록"을 적는다.
- 잘못된 작업이 생기면 "이 메모장 업데이트해서 같은 실수 다시 하지 마"라고 말한다. 그러면 클로드가 실수를 메모장에 기록하고 다음부터 그걸 보며 같은 실수를 하지 않는다.
- 보리스는 이를 [불명확: 원문 "봉리 학습"] 이라고 부른다 (새 직원이 실수를 반복하며 매뉴얼에 추가하는 것에 비유).

### 단축 명령어 + 분업
- 자주 하는 작업 순서를 명령어 하나로 저장해 둔다 (예: 저장→공유→검토 요청을 한 번에 실행). 스마트폰 단축어 기능과 같은 개념.
- 분업: 지시 끝에 "여러 명이 나눠서 해줘"라고 하면 클로드가 더 많은 처리 능력을 써서 동시에 처리한다.
- 보리스는 작업물 공유 전에 동시에 세 가지를 확인시킨다: 규칙 맞는지 확인 / 작업 이력 파악 / 오류 찾기. 사람이 검토하는 동안 여러 관점에서 미리 다 확인해 둔다.

### 보리스의 마지막 코멘트 — 기본값
- 보리스는 자기 설정이 놀랍도록 기본값 그대로라고 한다. 기본 설정만으로 충분히 잘되고, 맞춤 설정은 필요할 때만 하라고 함.
- 이 팁들은 여러 사이트에 정리돼 있으니 직접 보라고 안내 (구체 출처는 [불명확]).

### 화자(evan) 본인의 추가 팁 5가지
- **팁1 — 대화 내용을 파일로 저장**: 클로드 코드는 새로 시작하면 이전 대화를 완전히 잊는다. "지금까지 한 내용 요약해 줘"로 정리시켜 프로젝트 구조·진행 상황·다음 할 일을 파일로 저장해 두고, 새 세션 시작 때 그 파일을 넣어 주면 클로드가 이전 맥락을 이해하고 이어서 작업한다 (인수인계 문서 비유).
- **팁2 — 도메인 분리**: 프로젝트가 커지면 클로드가 헷갈려 엉뚱한 곳을 건드린다. 처음부터 영역별로 폴더를 나누고(결제/사용자/관리자 등) 각 영역마다 클로드를 하나씩 배치해 자기 영역에만 집중시킨다 (인사팀이 재무팀 일에 안 끼는 것 비유). 실수가 줄고 나중에 찾기도 편하다.
- **팁3 — 설계서 먼저 (SDD)**: 화자는 스펙 주도 개발([불명확: 원문 "스펙드 deop먼 SDD"], Spec-Driven Development) 방식을 쓴다. 보리스와 비슷하되 한 단계 더 나아가, 코드 짜기 전에 무엇을·어떻게 생겼고·어떻게 동작하는지를 문서로 정리한다. 설계서를 "이대로 만들어 줘"로 보여주면 방향이 안 흔들리고 팀 리뷰도 편하다. [의견] "설계서 하나가 수십 번의 수정을 막아 준다."
- **팁4 — 완료 알림 설정 (훅)**: 작업이 끝나면 소리로 알려주는 설정을 훅으로 걸 수 있다. 작업을 맡기고 다른 일(커피, 다른 창 작업)을 하다 소리가 나면 결과를 확인. 화면을 붙잡고 기다리는 시간이 없어지고, 특히 여러 클로드를 동시에 돌릴 때 어떤 게 끝났는지 알기 쉬워진다.
- **팁5 — 스킬 만들기**: 자주 하는 작업의 전체 방법을 스킬로 만든다 (예: 유튜브 썸네일 만들기, 영상 편집, 스크립트). 스킬 파일에는 명령어뿐 아니라 어떤 순서로·어떤 기준으로 판단할지·자주 하는 실수까지 적는다. "썸네일 만들어 줘"라고만 하면 클로드가 스킬 파일을 보고 처음부터 끝까지 알아서 한다. [의견] "나만의 클로드 사용 설명서를 만드는 것."

## 3줄 오리엔테이션
- Anthropic 클로드 코드 총책임자 "보리스"가 공개한 실사용법 중 핵심 5가지를 화자가 뽑아 소개하고, 뒤에 화자 본인의 팁 5가지를 덧붙인 영상이다.
- 보리스의 5원칙: 스스로 확인 / 계획 먼저 / 동시 다중 실행 / 전용 메모장 / 단축 명령어. 화자 팁: 대화 저장 / 도메인 분리 / 설계서 먼저 / 완료 알림 / 스킬 만들기.
- 개발자가 아니어도 이해하도록 비유 위주로 설명하며, 보리스의 수치(품질 2~3배 등)는 모두 화자가 전달한 미검증 주장이다.

## 전문 — 시간순 워크스루

### [00:00–00:27] 도입 — "10%만 쓰고 있다"
화자는 클로드 코드를 그냥 질문하고 답변 받는 식으로 쓰면 "10%만 활용하는 것"이라고 문제 제기한다. 클로드 코드를 직접 만든 사람 "보리스"(Anthropic의 클로드 코드 총책임자, [불명확: 원문 "앤스트로픽"])가 자기가 실제로 어떻게 쓰는지 공개했고, 화자는 그 핵심만 뽑아 소개하며 자기가 직접 쓰며 발견한 팁도 뒤에 추가했다고 예고한다. 개발자가 아니어도 이해할 수 있게 쉽게 설명하겠다고 함.

### [00:27–00:45] 보리스 원칙 ① 동시에 다섯 개 실행
[의견] 가장 공감이 됐던 것부터 시작한다. 보리스는 클로드 코드를 하나만 쓰지 않고 동시에 다섯 개를 쓴다. 화면 창을 1번부터 5번까지 열고 각 창에 클로드 코드를 하나씩 켜서 서로 다른 작업을 동시에 진행한다 (주장/미검증).
> "혼자 일하는 게 아니라 직원 다섯 명을 동시에 부리는 거예요." [00:43]

### [00:45–01:05] 독립 작업 공간 + 브라우저 창 + 폰-PC 이어받기
여러 명에게 각각 다른 일(기획서, 디자인, 정리)을 시키면 서로 뒤엉키지 않도록, 창마다 독립된 작업 공간을 따로 만들어 각자 자기 영역에서만 일하게 한다. 거기에 웹브라우저에서도 창을 5~10개 더 열고, 아침에 폰으로 시작한 작업을 컴퓨터에서 이어받기도 한다 (주장/미검증).

### [01:05–01:22] 알림으로 여러 클로드 지휘하기
여러 개를 돌리면 어느 클로드가 뭘 하는지 알 수 없으므로 알림 설정을 해둔다. 클로드가 다음 지시를 기다리는 상태가 되면 알림이 오고, 그 창으로 가서 지시를 준 뒤 또 다른 창으로 넘어간다.
> "사람이 여러 클로드를 오케스트라처럼 지휘하는 거예요." [01:17]

### [01:22–02:01] 보리스 원칙 ② 계획 먼저 세우기 (플랜 모드)
두 번째는 계획 먼저 세우기로, 보리스가 가장 강조한 것 중 하나다. 보통은 "이거 만들어 줘" 하고 바로 시키지만 보리스는 먼저 플랜 모드로 전환한다. 이 모드에서 클로드는 아무것도 건드리지 않은 채 전체 상황을 분석하고 어떻게 할지 플랜만 짠다 (집 짓기 전 설계도 비유).
> "계획의 에너지를 쏟아라. 그래야 클로드가 작업을 한 번에 끝낸다." [01:44]

막힐 때도 계획 모드를 쓴다. 한 클로드가 계획을 세우면 다른 클로드가 경험 많은 선임처럼 그 계획을 비판적으로 검토하게 해, 실제로 일하기 전에 문제를 미리 찾는다. [의견] 처음엔 귀찮게 느껴지지만 해보면 결과물이 확실히 다르다.

### [02:01–02:38] 보리스 원칙 ③ 스스로 확인하게 하기 (최중요)
세 번째 "스스로 확인하게 해라"가 보리스가 가장 중요하다고 직접 말한 팁이다. 클로드에게 항상 자기 작업을 스스로 확인할 수 있는 방법을 주라는 것. 작업을 마친 뒤 맞는지 틀린지 스스로 점검하게 하라는 뜻이다. 요리사에게 김치찌개를 끓이라고 했는데 완성 후 맛을 한 번도 안 보면 잘 만들었는지 알 수 없다는 비유. 확인 방법을 안 주면 틀린 줄 모르고 "완료했습니다"라고 해버리고, 주면 스스로 고친다.
> "클로드에게 확인 방법을 주면 최종 결과물 품질이 2에서 세 배 향상된다." [02:36] (주장/미검증)

### [02:38–02:53] 확인 방법의 구체 예
확인 방법은 작업 성격에 따라 다르다. 간단한 건 결과를 직접 실행해 보게 하고, 복잡한 건 브라우저를 직접 켜서 화면이 제대로 나오는지 확인하게 한다. 보리스 팀은 클로드가 직접 브라우저를 열고 화면을 확인하면서 될 때까지 스스로 반복해서 고치기도 한다 (주장/미검증).

### [02:53–03:18] 중간 삽입 — 모델 버전 선택
클로드 코드가 쓰는 모델에도 여러 버전이 있다. 빠르고 가벼운 버전, 느리지만 더 똑똑한 버전. 보리스는 가장 똑똑한 버전을 최대 성능으로 쓴다. 느리지 않냐는 물음에 대한 논리:
> "결국에는 더 빨라요. 왜냐면 덜 고쳐도 되거든요." [03:07]

싸고 빠른 버전으로 여러 번 수정하는 것보다 처음부터 똑똑한 버전을 한 번 쓰는 게 결국 빠르다는 것. "싸게 여러 번 수리하느냐, 한 번에 제대로 고치느냐"의 차이 (주장/미검증).

### [03:18–03:57] 보리스 원칙 ④ 클로드 전용 메모장
보리스 팀에서 가장 인상적이었던 것은 클로드 전용 메모장이다. 프로젝트 폴더에 클로드에게 주는 메모 파일을 하나 만들어 팀 작업·규칙·방향, 그리고 가장 중요한 "클로드가 반복해서 틀리는 실수 목록"을 적는다. 잘못된 작업이 생기면 보통은 그냥 다시 시키지만, 보리스는 "이 메모장 업데이트해서 같은 실수 다시 하지 마"라고 말한다. 그러면 클로드가 그 실수를 메모장에 기록하고 다음부턴 그걸 보며 같은 실수를 하지 않는다. 보리스는 이를 [불명확: 원문 "봉리 학습"]이라고 부른다고 한다 (새 직원이 같은 실수를 반복하며 매뉴얼에 추가하는 것 비유). 시간이 지날수록 클로드가 점점 더 팀 방식에 맞게 움직인다.

### [03:57–04:29] 보리스 원칙 ⑤ 단축 명령어 + 분업
단축 명령어: 자주 하는 작업 순서를 명령어 하나로 저장해 둔다 (예: 저장→공유→검토 요청을 한 번에 실행). 매번 같은 말을 반복 안 해도 된다 (스마트폰 단축어 비유). 분업: 지시 끝에 "여러 명이 나눠서 해줘"라고 하면 클로드가 더 많은 처리 능력을 써서 동시에 처리한다. 보리스는 작업물 공유 전에 동시에 세 가지를 확인시킨다 — 규칙 맞는지 확인, 작업 이력 파악, 오류 찾기. 사람이 한 번 검토하는 동안 이쪽에서는 이미 여러 관점에서 다 확인해 둔다.

### [04:29–04:55] 정리 + 보리스의 "기본값" 코멘트
화자가 보리스의 핵심 5가지를 정리한다: 첫째 스스로 확인하게 해라, 둘째 계획을 먼저 세워라, 셋째 동시에 여러 개를 돌려라, 넷째 클로드 전용 메모장에 투자해라, 다섯째 단축 명령어를 써라. 그리고 보리스가 마지막에 한 말 — 자기 설정이 놀랍도록 기본값 그대로라고 한다. 기본 설정만으로 충분히 잘되고, 맞춤 설정은 필요할 때만 하라고. 이 팁들은 여러 사이트에 정리돼 있으니 직접 가서 보라고 안내 (구체 출처 [불명확]).

### [04:55–05:34] 화자 팁 ① 대화 내용을 파일로 저장
여기서부터는 화자 본인이 쓰며 발견한 팁이다. 팁1 — [불명확: 원문 "기억 상식일 문제"] 대화 내용을 파일로 저장하라. 클로드 코드는 새로 시작하면 이전 대화를 완전히 잊어버린다 (매번 처음 만나는 사람 비유). 그래서 중요한 내용이 있으면 "지금까지 한 내용 요약해 줘"로 클로드가 정리하게 해 프로젝트 구조·현재 진행 위치·다음 할 일을 파일로 저장해 둔다. 새 세션 시작 때 그 파일을 넣어 주면 클로드가 바로 이전 맥락을 이해하고 이어서 작업한다 (인수인계 문서가 있으면 새 직원이 바로 일하는 것 비유). 한 번 만들어 두면 시간이 엄청 절약된다.

### [05:34–06:04] 화자 팁 ② 도메인 분리
팁2 — 큰 프로젝트는 영역별로 쪼개라(도메인 분리). 프로젝트가 커지면 클로드가 헷갈려 이것저것 다 섞여 엉뚱한 곳을 건드린다. 그래서 처음부터 영역별로 나눈다 — 결제 관련은 결제 폴더, 사용자 관련은 사용자 폴더, 관리자 관련은 관리자 폴더. 각 영역마다 클로드를 하나씩 배치해 결제 담당 클로드·사용자 담당 클로드가 서로 남의 영역을 안 건들고 자기 영역에만 집중하게 한다 (인사팀이 재무팀 일에 안 끼는 것 비유). 실수가 줄고 나중에 찾아볼 때도 편하다.

### [06:04–06:41] 화자 팁 ③ 설계서 먼저 (SDD)
팁3 — 코드 짜기 전에 설계서 먼저. 화자는 [불명확: 원문 "스펙드 deop먼 SDD"] 스펙 주도 개발(Spec-Driven Development) 방식을 쓴다. 보리스와 비슷하되 한 단계 더 나아갔다. 코드 짜기 전에 뭘 만들 건지·어떻게 생겼는지·어떻게 동작해야 하는지를 문서로 정리한다 (집 짓기 전 설계도 비유). 설계서를 "이대로 만들어 줘"로 클로드에게 보여주면 방향이 안 흔들리고, 이 스펙이 있으면 팀원끼리 리뷰하기도 편하다. 반대로 설계 없이 "이런 기능 추가해 줘"를 계속하면 어느 순간 처음과 완전 다른 게 돼 있다. [의견] 설계서 하나가 수십 번의 수정을 막아 준다.

### [06:41–07:14] 화자 팁 ④ 완료 알림 설정 (훅)
팁4 — 일 끝나면 "띵" 완료 알림 설정. 은근히 삶의 질이 달라지는 팁이다. 클로드가 작업 중일 때 화면을 계속 보며 "다 됐나" 하지 말고, 클로드 코드의 작업 완료 시 소리 알림 설정을 훅으로 걸어 둔다. 작업을 맡기고 커피를 마시거나 다른 창에서 다른 일을 하다가 소리가 나면 결과를 확인한다. 화면을 붙잡고 기다리는 시간이 없어진다. 특히 보리스처럼 여러 클로드를 동시에 돌릴 때 이 알림이 없으면 어떤 클로드가 끝났는지 일일이 확인해야 한다.

### [07:14–07:46] 화자 팁 ⑤ 스킬 만들기
팁5 — 스킬을 만들어라, 나만의 단축 매뉴얼. 보리스가 단축 명령어 얘기를 했는데 화자는 더 나아가 자주 하는 작업의 전체 방법을 스킬로 만든다 (예: 유튜브 썸네일 만들기, 영상 편집, 스크립트). 스킬 파일에는 명령어만 있는 게 아니라 어떤 순서로 할지·어떤 기준으로 판단할지·자주 하는 실수는 뭔지까지 다 적혀 있다. "썸네일 만들어 줘"라고만 하면 클로드가 스킬 파일을 보고 처음부터 끝까지 알아서 다 한다. 처음 만들 때 시간이 좀 걸리지만 한 번 만들어 두면 계속 써먹고 자기에게 맞게 점점 발전시킬 수 있다. [의견] 나만의 클로드 사용 설명서를 만드는 것.

### [07:46–08:00] 마무리 — 하나만 골라서 해보라
[의견] 이것 중 하나라도 오늘 당장 써볼 수 있는 게 있으면 딱 하나만 골라서 해보라고 권한다. 가장 쉬운 두 가지는 계획 먼저 세우기와 완료 알림 설정으로, 둘 다 설정 몇 번이면 끝난다. 해보고 실제로 달라지는지 댓글로 알려 달라고 요청하며 영상을 마친다.
