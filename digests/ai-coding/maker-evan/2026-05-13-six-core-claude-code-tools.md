# 400시간 만에 알게 된 Claude Code의 진짜 핵심 도구 6개

> 정리 방식: 시간순 심층 워크스루 (paraphrase) + 주제별 상세 요약
> 검증 배너: 이 영상은 화자가 직접 운영하는 유료 스킬샵(오픈 베타)을 홍보하는 세일즈 성격이 있다. 속도·세션 유지시간 등 성능 수치는 화자 주장이며 미검증.

## 요약 (주제별)

### 핵심 메시지 — "화려한 것보다 매일 쓰는 것"
- 화자는 Claude Code를 약 10개월(자막 기준) 써 왔고, 유튜브/바이럴용으로 만든 화려한 스킬·플러그인 대다수는 실사용성이 낮다고 본다. [의견]
- 자기가 매일 켜 놓고 쓰는 도구는 6개(보너스 포함 실제로는 그 이상)뿐이며, "화려하지 않고 오히려 지루하지만 진짜 효과적"이라고 주장. [의견]
- 산업(React, Next.js, 자료 제작, 클라이언트 작업)이 달라도 문제 패턴과 해결 도구는 반복된다는 게 선정 근거. [의견]

### GitHub 별점 신뢰성 경고
- 요즘 GitHub가 "인스타그램처럼" 변해서 별점을 부풀리려는 프로젝트가 많다고 주장. [의견]
- 실제로 깔아 보면 성능이 엉망인 경우가 많고, 일부 프로젝트는 설치 스크립트(readme)에 프롬프트 인젝션을 심어 에이전트가 그 파일을 읽으면 성능과 무관하게 별을 누르도록 유도한다고 주장. (주장/미검증)
- 결론: 별 개수 = 좋은 프로젝트 보장 아님. 지금은 "알짜배기를 골라내는 큐레이션 능력"이 가장 중요한 시점. [의견]

### 도구 1 — Superpowers 플러그인
- 스킬이 아니라 여러 스킬을 묶은 플러그인. Claude가 "시니어 개발자처럼" 일하도록 강제하는 워크플로 묶음: 바로 코딩하지 않고 멈춰서 계획 → 테스트 먼저 → 코드 → 끝나면 스펙 부합/코드 품질 두 번 자기 리뷰.
- 해결하는 문제: Claude Code 최대 실패 패턴인 "급하게 짠 코드"(그럴듯하지만 엣지 케이스에서 깨짐)를 속도를 늦춰 단계 건너뛰기를 막는 것.
- 화자는 통째로 다 쓰진 않고 두 스킬을 주로 씀: (1) brainstorming — 기능 만들기 전 무조건 켜서 "누가 쓰나/성공 기준/어떤 문제 해결" 질문으로 요구사항을 정리(헛코드 감소), (2) subagent-driven development — 계획 정리 후 큰 일을 서브 에이전트들에 분배, 각자 깨끗한 컨텍스트에서 작업, 끝나면 다른 서브 에이전트가 스펙/코드 품질 2단계 리뷰. 메인 세션이 안 더러워져 세션이 더 오래 감.
- 설치: 공식 Claude 플러그인 마켓플레이스에서 플러그인 install 명령 한 줄. 화자는 글로벌로 깔아 모든 프로젝트에서 자동 켜지게 해둠.

### 도구 2 — "G스텍" [불명확: 원문 음차 "G스텍"], office + browser
- Y Combinator의 게리 탄(Garry Tan)이 직접 쓰는 셋업이라고 공개한 것. (주장/미검증) 화자가 매일 쓰는 건 그 안의 office와 browser 두 개.
- office: 새 아이디어를 잡을 때 쓰는 스킬. YC 사무실에서 받을 법한 질문 6개("진짜 수요가 있나 / 기존엔 다들 어떻게 해결하나 / 가장 좁은 진입점은")를 던져, 막연한 아이디어를 좁고 날카롭게 만듦. 화자는 새 영상 기획·새 사이드 프로젝트 시작 때도 켬.
- browser: 백그라운드에 크롬 세션을 상주시켜, 매번 새 브라우저를 띄우지 않아 명령당 100~200ms에 끝남 — 기존 방식보다 약 20배 빠르다고 주장. (주장/미검증) 스크린샷·로그인·폼 채우기·콘솔 에러 잡기 등에 쓰며, 화자는 만든 사이트 QA마다 켬. 컨텍스트도 덜 더러워짐.
- 두 개를 같이 쓰면 세션이 오래 감: 보통 Claude가 30분이면 망가지는데 이걸 켜두면 두 시간도 간다고 주장. (주장/미검증)

### 도구 3·4 — 매퍼(Matt Pocock 추정) [불명확: 원문 음차 "매퍼/매포"] 스킬 2종
- 화자가 "타입스크립트 쪽에서 유명하고 강의·유튜브도 하는 분"이라 소개한 사람이 만든 스킬. (게시자가 명시한 이름은 음차만 있어 [불명확])
- grill-me("구워줘"): 짠 코드나 설계를 들이밀면 "여기 이상한데 왜 이렇게 했어?" 식으로 캐물음. 1인 개발자가 동료 코드리뷰가 없어 부끄러운 실수가 그대로 나가는 문제를 해결 — 동료 역할 대행. 예: React 컴포넌트의 useState 위치("부모로 올려야 하는 것 아니냐")를 잡아냄. 화자는 PR 올리기 전 무조건 한 번 돌림.
- improve-architecture: 구조를 통째로 다시 보는 스킬. 기존 코드를 더 좋게 만들 방법을 물으면 "여기 분리하면 테스트 쉬워져 / 이 부분 추상화하면 재사용 가능" 등을 제안. 리팩토링 시 "과하게 분리한 건가/더 해야 하나"처럼 혼자선 안 보이는 판단을 도움. 100% 수용하진 않아도 한두 개 정말 좋은 제안이 나옴. 둘 다 매일 씀.

### 도구 5 — Codex Review
- OpenAI Codex 쪽 스킬로, Claude 안에서 쓸 수 있게 OpenAI 측이 플러그인 형태로 만든 것. (주장/미검증) 내가 짠 코드를 다른 모델(OpenAI 모델) 시각으로 한 번 더 리뷰.
- 근거: 같은 모델로만 보면 자기가 짠 코드를 자기가 보는 것이라 어색한 부분이 안 보임. 다른 모델 관점으로 훑으면 "Claude는 안 잡았는데 Codex가 잡은" 것이 많이 나옴. 화자 최근 체감으로 Codex가 세부적인 부분에서 더 월등하고 Claude Code가 내는 버그를 잘 잡아낸다고 주장. [의견]
- 사용법: 워크플로 마지막 단계에 끼움(Superpowers로 짜고 → 마지막에 Codex Review로 마무리). 이 단계를 거치면 안심하고 머지 가능. 단 토큰은 꽤 많이 든다고 함.

### 도구 6 — Vercel React Best Practices
- Vercel이 만든 스킬. 프론트엔드 하는 사람에겐 필수라고 주장. React 코드 작성 시 Vercel 팀이 정한 모범 사례를 Claude에게 주입.
- 해결하는 문제: 기본 Claude는 React 컴포넌트를 잘 짜지만 useEffect 남용, 클라이언트/서버 컴포넌트 구분 애매, 2년 전 패턴 사용 등이 있음. 이 스킬을 깔면 "이건 서버에서 / 이건 use client 불필요 / 이 패턴은 이제 안 씀"을 알아서 챙겨 코드 톤이 일관됨.
- 설치: npx 계열 명령으로 다운로드(skill.sh 추천 언급). 화자는 React/Next.js 프로젝트 시작 때 무조건 깜.

### 보너스 — PPTX 스킬 + PDF 스킬
- 클라이언트에게 결과를 보여줄 발표자료(보고서/제안서/슬라이드)를 손으로 만들기 귀찮은 문제를 해결. "개발자는 코드는 잘 짜는데 PPT는 못 만든다"는 공감 포인트. [의견]
- PPTX 스킬: 내용만 주면 슬라이드를 만들어 줌. PDF 스킬: 표지·목차·본문 포함 보고서 PDF를 깔끔하게 자동 생성.

### 도입 조언 · 세일즈 구조
- 한 번에 다 깔지 말 것 — 컨텍스트만 무거워짐. 천천히 하나씩 익히며 도입하라. [의견]
- 화자는 "직접 깔아 돌려본 스킬만 올리는" 스킬샵 사이트를 운영 중(자막상 이름 음차 "Skills", [불명확: 정확 도메인/명칭 불명]). 특징 3가지: (1) 본인이 매일 쓰는 스킬 위주 큐레이션, (2) 현재 오픈 베타, (3) 베타 기간 저렴한 가격에 공유(추후 인상 예정). "검증에 시간 안 써도 된다, 그건 내가 한다"가 세일즈 포인트.
- CTA: 좋아요, 여러분이 쓰는 플러그인/스킬 댓글 제보, 구독(다음 영상은 이 도구들로 실제 프로젝트 처음부터 끝까지 만들기 예고, 새 스킬 소식도 사이트에서 전달).

## 전문 — 시간순 워크스루

### [00:00–00:22] 오프닝 — 화려한 스킬 대다수는 실사용이 아니다
Claude Code를 대략 10개월 쓰면서 알게 된 것으로, 유튜브에 멋있어 보이는 스킬·플러그인을 만드는 사람이 많지만 대다수는 "영상용/바이럴용"이지 실제로 쓰는 것이 아니라고 말한다. [의견] GitHub 스타 수만 많고 막상 써 보면 엉망인 프로젝트가 많다고 하며, 자신이 진짜 매일 켜 놓고 쓰는 건 여섯 개 정도이고 "화려하지 않고 오히려 지루하지만 진짜 효과적"이라고 주장. [의견]

### [00:22–01:02] 선정 배경 + 별점 조작·프롬프트 인젝션 경고
본론 전에 자기 배경을 밝힌다 — 영상 제작, 자동화, 클라이언트 작업, React·Next.js 프로젝트, 자료 제작까지 했고, 산업이 달라도 문제 패턴은 똑같이 반복되고 그 문제를 푸는 도구도 반복 등장하더라는 것이 선정 근거다. [의견] 이어 "요즘 GitHub가 좀 이상하다, 인스타그램처럼 변했다"며 별점을 늘려 바이럴을 일으키려는 프로젝트가 너무 많다고 주장한다. [의견]
> "심지어 프로젝트는 readme나 설치 스크립트에 프롬프트 인젝션을 심어놔요. 에이전트가 그 파일을 읽으면 성능이랑 상관없이 무조건 깃허브 별을 누르도록 유도하는 거예요." [00:51] (주장/미검증)

### [01:02–01:17] 결론적 프레임 — 큐레이션 능력이 핵심
별이 많다고 좋은 프로젝트라는 보장이 절대 없으며, 지금은 "알짜배기를 골라내는 큐레이션 능력"이 제일 중요한 시점이라고 정리한다. [의견] Claude Code를 진지하게 쓰고 싶으면 복잡하게 생각하지 말고 자신이 실제로 돌려 검증한 여섯 개부터 시작하라고 권한다.

### [01:17–02:00] 도구 1 — Superpowers 플러그인 (brainstorming + subagent-driven development)
첫 번째는 Superpowers, 스킬이 아니라 여러 스킬이 든 플러그인이다. Claude가 시니어 개발자처럼 일하게 강제하는 워크플로 묶음으로, 바로 코드를 짜지 않고 멈춰 계획을 세우고 → 테스트를 먼저 쓰고 → 코드를 쓰고 → 끝나면 스펙이 맞는지·코드 품질이 어떤지 두 번 자기 리뷰를 한다. 해결하는 문제는 Claude Code의 가장 큰 실패 패턴인 급하게 짠 코드로, 시키면 바로 그럴듯한 코드를 뱉지만 돌리면 엣지 케이스가 깨지고 더 나쁘게는 사용자가 돌릴 때 깨진다. Superpowers는 속도를 충분히 늦춰 문제를 제대로 생각하게 만들고 단계를 건너뛰지 못하게 한다. 화자는 통째로 다 쓰진 않고 안의 두 스킬을 주로 쓴다고 한다: brainstorming과 subagent-driven development. brainstorming은 새 기능을 만들기 전 무조건 켜서 "누가 쓸 거냐 / 성공 기준이 뭐냐 / 어떤 문제를 해결하냐" 질문으로 요구사항을 정리하게 하고, 이 한 단계만으로 헛코드가 크게 준다고 한다. subagent-driven development는 계획이 정리된 다음 큰 일을 서브 에이전트들에 나눠 시키고 각자 가장 깨끗한 컨텍스트에서 자기 일만 하며, 끝나면 다른 서브 에이전트가 스펙 부합/코드 품질 2단계로 리뷰한다. 메인 세션이 더러워지지 않아 한 세션이 훨씬 오래 간다.

### [02:00–02:56] 도구 1 설치
설치는 아주 간단하다고 하며, Claude Code에서 플러그인 install 명령을 한 줄 치면 공식 Claude 플러그인 마켓플레이스에서 Superpowers를 바로 받아준다고 한다(명령어는 화면·설명란에 표기 예고). 화자는 글로벌로 깔아 어떤 프로젝트에 들어가도 자동으로 켜지게 해뒀다.

### [02:56–03:32] 도구 2 — "G스텍"의 office
두 번째는 자막상 "G스텍"으로 음차된 도구 [불명확: 원문 음차 "G스텍"]로, Y Combinator의 게리 탄(Garry Tan)이 직접 쓰는 셋업이라고 공개한 것이라고 소개한다. (주장/미검증) 그 안에서 화자가 매일 쓰는 건 office와 browser 두 개다. office는 새 아이디어를 잡을 때 쓰는 스킬로, "이거 만들면 어떨까?"를 들고 가면 YC 사무실에서 받을 법한 질문 6개("진짜 수요가 있냐 / 기존엔 다들 어떻게 해결하냐 / 가장 좁은 진입점은 뭐냐")를 던져, 막연했던 사업 아이디어가 좁고 날카로워진다. 화자는 새 영상 기획·새 사이드 프로젝트 시작 때도 켠다.

### [03:32–03:57] 도구 2 — browser
그다음 browser가 진짜 핵심이라고 한다. 일반적으로 Claude에게 브라우저 작업을 시키면 매번 새 브라우저를 띄우느라 느린데, browser는 백그라운드에 크롬 세션을 상주시켜 명령 하나당 100~200ms 안에 끝난다 — 기존 방식보다 거의 20배 빠르다고 주장. (주장/미검증) 스크린샷 찍기, 로그인, 폼 채우기, 콘솔 에러 잡기 등에 쓰며 화자는 만든 사이트 QA마다 켠다. 빠르고 컨텍스트도 덜 더러워진다. 두 개를 같이 쓰면 세션이 오래 가서, 보통 Claude가 30분이면 망가지는데 이걸 켜두면 두 시간도 간다고 주장. (주장/미검증)

### [03:59–04:22] 중간 정리 — "차분하게 일하게 만드는 도구들"
여기까지 본 두 도구의 공통점은 둘 다 Claude를 차분하게 일하게 만드는 도구지 빠르게 답하게 만드는 게 아니라는 점이다. 처음엔 "왜 이렇게 뜸 들이지?" 싶고 질문이 많아 답답할 수 있지만, 결과물을 받아보면 한 방에 끝나는 것과 다섯 번 고쳐 받는 것의 차이라고 한다. 처음에 시간을 더 쓰는 대신 나중 디버깅 시간을 크게 줄여준다는 주장이다. [의견] [불명확: 자막 "디버깅 시간을 거의 10분로 줄여" — 원 수치 불명]

### [04:22–05:12] 도구 3 — 매퍼(Matt Pocock 추정)의 grill-me
세 번째는 "매퍼"라고 음차된 사람 [불명확: 원문 음차 "매퍼/매포"]이 만든 스킬로, 타입스크립트 쪽에서 유명하고 강의·유튜브도 한다고 소개한다. 그가 만든 스킬 두 개를 추천하는데, 첫째는 이름이 "그대로 구워줘(grill-me)"라는 뜻이다. 짠 코드나 설계를 들이밀면 "여기 이상한데 왜 이렇게 했어?" 식으로 캐묻는다. 해결하는 문제는 1인 개발자가 동료가 없어 코드 리뷰 받을 데가 없고 혼자 짜다 부끄러운 실수가 그대로 나가는 것으로, grill-me가 그 동료 역할을 한다. 예로 React 컴포넌트를 잘 짠 것 같아도 grill-me를 돌리면 "이 useState 왜 여기 있어? 부모 컴포넌트로 올려야 하는 것 아니야?" 식으로 잡아낸다. 화자는 PR 올리기 전 무조건 한 번 돌려 그런 실수가 거의 안 나가게 설정할 수 있다고 한다.

### [05:12–05:47] 도구 4 — improve-architecture
그다음은 improve-architecture 스킬로, 구조를 통째로 다시 보는 스킬이라고 보면 된다. 이미 있는 코드를 어떻게 더 좋게 만들 수 있을지 물으면 "여기 이렇게 분리하면 테스트 쉬워져 / 이 부분 추상화하면 재사용 가능해" 등을 제안한다. 해결하는 문제는 리팩토링 때 "내가 너무 과하게 분리하는 건가, 아니면 더 분리해야 하나"처럼 혼자 보면 잘 안 보이는 판단으로, 이 스킬을 돌리면 한두 개 정말 좋은 제안이 나온다(100% 다 받아들이진 않음). 화자는 grill-me와 improve-architecture 둘 다 매일 쓴다.

### [05:47–06:49] 도구 5 — Codex Review
다섯 번째는 Codex Review로, OpenAI Codex 쪽 스킬인데 Claude 안에서 쓸 수 있도록 OpenAI 측이 플러그인 형태로 만든 것이라고 한다. (주장/미검증) 내가 짠 코드를 다른 모델(OpenAI 모델) 시각으로 한 번 더 리뷰해 주는 것이다. 중요한 이유는 같은 모델로만 보면 놓치는 게 있기 때문 — Claude에게 코드 리뷰를 시키면 자기 관점으로만 봐서 자기가 짠 코드를 자기가 보는 것과 비슷해 어색한 부분이 잘 안 보인다. Codex Review를 돌리면 완전히 다른 모델 관점에서 훑어 "Claude는 안 잡았는데 Codex가 잡은" 것이 많이 나온다.
> "요즘 들어 드는 생각은, Codex가 세부적인 부분에서는 훨씬 더 월등한 능력을 가진 것 같고, Claude Code가 내는 그런 버그들을 다 잡아내는 것 같습니다." [06:22] [의견]
실용적으로는 Claude Code 워크플로 마지막 단계에 끼운다 — Superpowers로 짜고 마지막에 Codex Review로 리뷰하며 마무리한다. 이 단계를 거치면 안심하고 머지할 수 있으나 토큰은 꽤 많이 든다고 한다.

### [06:49–07:33] 도구 6 — Vercel React Best Practices
여섯 번째는 Vercel이 만든 React best practices 스킬로, 프론트엔드 작업을 하는 사람에겐 필수라고 한다. [의견] React 코드를 짤 때 Vercel 팀이 정해 놓은 모범 사례를 Claude에게 알려주는 스킬이다. 기본 Claude에게 React 컴포넌트를 짜라고 하면 잘 짜지만 어딘가 어색해서 useEffect를 너무 많이 쓰거나 클라이언트/서버 컴포넌트 구분이 애매하거나 2년 전 패턴을 그대로 쓰기도 한다. 이 스킬을 깔면 "이건 서버에서 해야 돼 / 이건 use client 필요 없어 / 이 패턴은 더 이상 안 써" 같은 걸 알아서 챙겨줘, Next 작업하는 사람이면 차이가 크고 코드 톤이 일관돼진다. 설치는 npx 계열에서 다운받으면 되고 skill.sh를 추천한다고 언급. 화자는 React·Next.js 프로젝트 시작 때 무조건 깐다.

### [07:33–08:04] 보너스 — PPTX 스킬 + PDF 스킬
마지막은 두 개를 묶어 말한다: PPTX 스킬과 PDF 스킬로, 보너스라고 한다. 필요한 이유는 클라이언트에게 결과를 보여줄 때 발표자료(보고서·제안서·슬라이드)를 만들어야 하는데 매번 손으로 만들기 귀찮기 때문이다. "개발자는 코드는 잘 짜는데 PPT는 잘 못 만들고 폰트 크기부터 색깔까지 막막하다"고 공감을 유도한다. [의견] PPTX 스킬은 "이 내용으로 슬라이드 만들어 줘" 하면 실제로 PPT를 만들어 주고, PDF 스킬은 표지·목차·본문까지 만들어 보고서 PDF를 깔끔하게 뽑아준다.

### [08:31–09:09] 세일즈 — 스킬샵 사이트(오픈 베타)
