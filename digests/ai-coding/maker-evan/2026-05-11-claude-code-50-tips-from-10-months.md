# 클로드코드 10개월… 매일 12시간 쓰며 알게 된 50가지

> maker-evan · 2026-05-11 · 길이 약 16:48 · [영상 링크](https://youtu.be/MYEYID9Euzs)
> 정리 방식: 시간순 심층 워크스루 (paraphrase) + 주제별 상세 요약
> [검증 배너] 화자 개인 사용 경험·주관적 추천이 다수. "10개월 매일 12시간" 등 사용량 수치는 화자 주장, 미검증.

## 요약 (주제별)

### 0. 핵심 메타포 — "기억력 짧은 신입 직원"
- 화자는 Claude Code를 "일은 잘하지만 매일 출근하면 어제 한 일을 까먹는 신입 직원", "치매에 걸린 아인슈타인"에 비유한다. [의견]
- 그래서 두 가지가 필수: (1) 어떻게 일하는지 적은 **매뉴얼**(CLAUDE.md), (2) 어디까지 했는지 적은 **노트**(작업 기록). 이게 영상 전체를 관통하는 큰 줄기.
- 화자의 직무 전환 주장: 예전엔 직접 코드를 쳤지만("망치 들고 못 박는 사람"), 지금은 신입 여러 명에게 지시하고 결과물을 검수하는 사람이 됐다. 가장 큰 일은 코드 작성이 아니라 신입들 결과물을 다 읽고 확인하는 것. (주장/미검증)
- 영상은 4덩어리 구성: ① 기초, ② 매일 하는 1일 방식, ③ 핵심 4가지 부품(스킬·명령어·MCP·서브에이전트), ④ 고급.

### 1. 프로젝트 시작·매뉴얼(CLAUDE.md) 셋업 (팁 1~5)
- **팁 1** 프로젝트 폴더에서 켜기: 폴더로 들어가 `claude` 실행 → 그 폴더 자료가 신입 책상에 자동으로 깔린다. 처음에 느린 건 신입이 자료를 훑는 시간.
- **팁 2** 새 프로젝트면 `/init`: 신입이 프로젝트를 한 바퀴 둘러보고 매뉴얼(`CLAUDE.md`)을 자동 생성. 새 프로젝트마다 무조건 돌리라고 권장. [의견]
- **팁 3** 매뉴얼은 계층 구조: 회사 전체 사규(전역)와 팀 룰(로컬)이 있고, `/memory`로 현재 적용 룰 확인. 지점 룰(로컬)이 본사 룰보다 우선.
- **팁 4** 매뉴얼은 짧게: 길수록 좋다는 건 반대. 300쪽 사규 다 외우게 하면 헷갈려 정작 중요한 걸 까먹는다. 화자는 **약 50줄**로 유지하고 CLAUDE.md를 여러 개로 분할 저장. [의견]
- **팁 5** 매뉴얼에 넣을 3가지: 무엇을 만드는 프로젝트인지 / 어떤 도구 쓰는지 / **가장 중요한 검증 방법**(결과물 체크리스트 = 공장 검수 라인). 검수 라인이 있으면 신입이 스스로 만들고·검수하고·고치기를 반복. AI 코딩이 안 되는 경우 90%는 이 검수 라인 부재 탓. (주장/미검증)

### 2. 단축키·인터랙션 조작 (팁 6~11)
- **팁 6** Shift+Tab 모드 전환: 바로 실행 / 계획만 짜기 / 알아서 다 하기 3모드. 화자는 세션 시작 시 항상 **계획 모드**부터. 바로 실행은 신입이 잘못 이해한 채 시작하는 위험. [의견]
- **팁 7** ESC 멈추기: 엉뚱한 방향이 보이면 ESC. 끊기를 무서워하지 말 것 — 잘못 가는 걸 두는 게 더 손해. 위 화살표(↑)로 방금 시킨 일을 다시 띄워 수정 후 재전송.
- **팁 8** ESC 2번으로 입력창 비우기: 길게 붙여넣은 문장을 한 번에 깨끗이.
- **팁 9** 빈 입력창에서 ESC 2번 되돌리기: 이전 시점으로 점프(예: 5분 전 대화 지점).
- **팁 10** 스크린샷은 드래그 앤 드롭: 캡처를 터미널 위로 끌어다 놓으면 자동 인식, 별도 업로드 불필요.
- **팁 11** 스크린샷에 설명 같이: 이미지만 던지지 말고 "여기 이 부분 색깔 바꿔줘" 같은 설명 동반. `/help`로 다른 단축키 확인 가능. 이 6개만 외워도 속도 2배. (주장)

### 3. 매일 쓰는 슬래시 명령어 (팁 12~19)
- **팁 12** `/clear` 칠판 지우기: 지금까지 대화를 지우고 새로 시작. 일 하나 끝나고 다른 일 할 때 써서, 앞 일 기억이 다음 일에 간섭하는 걸 막는다.
- **팁 13** `/context` 머리 용량 확인: 신입 책상이 얼마나 찼는지 보기. 명령어 대신 대시보드로 하단에 띄우는 플러그인을 찾아 쓰는 것도 추천. [의견]
- **팁 14** 자동 요약(auto-compact)은 그냥 두기: 대화가 길어지면 시스템이 알아서 요약. 저장하고 싶을 때만 `/compact`. 단, 오토 콤팩션이 일어나면 **성능 저하가 크게** 일어나므로 웬만하면 `/clear`로 컨텍스트를 비우는 습관 추천. (주장/미검증) [의견]
- **팁 15** `/model` 모델 바꾸기: 경험 많은 시니어 vs 빠른 주니어. 회사일=시니어, 사이드=주니어. 화자는 회사일인데 토큰이 부족하거나 빠른 응답이 필요할 때 **Opus Plan 모드**(계획=Opus, 코드=Sonnet)를 써서 토큰 효율을 높인다. [의견]
- **팁 16** `/resume` 이어하기: 실수로 창 닫았을 때 다시 켜고 `/resume` → 어제 시점으로 복귀. 신입에게 어디까지 했는지 다시 안 물어봐도 됨.
- **팁 17** `/mcp` 외부 도구 상태: MCP는 외부 도구와 신입을 잇는 전화기. 전화가 너무 많으면 책상이 어지러우니 진짜 필요한 것만 설치.
- **팁 18** `/help` 매뉴얼 열기: 다른 명령어 설명, 한 번 둘러보기 권장.
- **팁 19** Git을 안전망으로: Git은 시간여행 기계(어제·그제·일주일 전 코드로 복귀). Git 관리와 커밋 메시지 작성까지 Claude에게 맡기는 게 사람보다 안정적이며, Claude가 커밋 내역을 보고 맥락을 추적하는 경우가 많아 더 추천. [의견] — 여기까지가 1단계, 이것만 알아도 80%. (주장)

### 4. 매뉴얼 심화 — 룰·트리거·팀 공유 (팁 20~25)
- **팁 20** 절대 룰 섹션: "이건 절대 하지 마 / 이건 항상 해라" 형태 + 예시. "라면 끓일 때 이 순서로, 절대 물 끓이지 말 것" 식의 구체성. 특히 우리 회사만의 특이 룰을 명시(신입은 다른 회사 경험은 있어도 우리 룰은 모름).
- **팁 21** 룰 업데이트는 신입에게 시키기: 실수가 반복되면 "다시는 안 그러게 매뉴얼 업데이트 해줘"라고 지시. 직접 열어 고치기보다 실패 케이스 반영을 Claude에게 시키는 게 효율적.
- **팁 22** 트리거 단어 박아두기: "빌드해줘" → 정해진 명령어 자동 실행. 키워드 하나로 정해진 동작.
- **팁 23** 매뉴얼을 팀에 공유(컴파운드 엔지니어링): 잘 만든 매뉴얼은 팀 전체가 공유. 주의 2가지 — ① 내 컴퓨터 절대경로 금지(일반화·상대경로), ② 너무 길면 팀 신입도 무거워짐. 화자는 몇 주 써보고 효과 있을 때만 머지. [의견]
- **팁 24** `--dangerously-skip-permissions`(YOLO 모드): 매번 묻는 걸 다 패스. **혼자 쓸 때만** 추천, 팀 공유 시엔 리스크가 큼(마음대로 헤집음). [의견]
- **팁 25** `/permissions` 화이트리스트: YOLO를 쓰더라도 위험 명령어(파일 통째 삭제 등)는 막아둘 것. "식자재 창고는 함부로 못 들어가" 룰.

### 5. 계획·컨텍스트 위생·검증 (팁 26~32)
- **팁 26** 새 일은 무조건 계획 모드부터: 화자는 신입과 "토론"한다 — "이렇게 하는 게 맞아? 왜 그렇게 생각해? 다른 방법은?" 첫 답변을 그대로 받은 적 거의 없음. 요즘은 플랜 모드보다 **Superpowers, gstack** 같은 브레인스토밍 툴을 더 많이 사용. [의견]
- **팁 27** 신선한 컨텍스트가 "컨텍스트 넘친(rot)" 것을 이긴다: 이것저것 계속 던지면 책상에 종이가 산처럼 쌓여 뭘 하려 했는지 까먹음. 처음에 좋은 계획 하나 = 기준점.
- **팁 28** 일 끝나면 노트에 저장: 신입은 매일 기억이 리셋되니 "오늘 한 거 노트에 정리해줘". 다음날 그 노트를 보면 어제 한 일을 앎. 화자는 이를 **세컨드 브레인**(외부 노트장)이라 부름. [의견]
- **팁 29** 노트는 필요할 때만 펴기: 매일 전부 외우면 책상이 무거워짐. "이번 일 관련 노트만 가져와" — 이런 노트를 컨텍스트라고도 부름. 할 일 목록도 이 방식으로 관리(별도 PM 도구 안 씀). [의견]
- **팁 30** 검증 명령어를 무조건 주기: 결과물을 스스로 확인하게. 모바일 앱이면 "Xcode로 빌드해봐", 웹이면 "브라우저 열어 눌러봐". QA/빌드 과정 필수, 디버깅도 가능.
- **팁 31** 복잡한 모델은 시니어 모델: 화자는 거의 항상 Opus만 사용. 느리지만 신입 여러 명을 동시에 돌리므로 하나가 느려도 다른 데 가서 일 시켜 체감 느림이 거의 없음. [의견]
- **팁 32** 신입의 생각 들여다보기(thinking block): 작업 중 자기 생각을 적는데, "확실하지 않은데" 같은 이상한 과정이 보이면 즉시 ESC로 끊기. 특히 초보자가 AI 돌려놓고 아무것도 안 하는 건 금물 — 헛소리가 보이면 그 즉시 멈춰야 함. 매우 중요한 테크닉. [의견]

### 6. 핵심 4가지 부품 — 스킬·명령어·MCP·서브에이전트 (팁 33~40)
- **팁 33** 4가지는 서로 조합됨(레고 블록 4종): 끼워 맞춰 더 큰 걸 만듦. 화자는 이를 "조합성(composability) 프레임워크"라 부를 수 있지 않을까 함. [의견]
- **팁 34** 스킬 = 반복 작업 묶음 + 도메인 전문가 지식: 예) "해커뉴스/네이버 뉴스 가져와서 노트에 정리" = 가져오기+정리하기 2단계를 "스킬 만들어줘"로 묶으면, 다음엔 트리거만으로 간단히 실행.
- **팁 35** 명령어 = 스킬의 단축어: 스킬 만들면 슬래시 명령어도 자동 생성.
- **팁 36** 손으로 만들지 마세요: 스킬·명령어·매뉴얼을 직접 손으로 고치는 건 비추천. Claude에게 시키고(신입이 자기 매뉴얼을 스스로 업데이트) 관리자는 검수만. [의견]
- **팁 37** MCP = 외부 서비스 연결선(전화기): 예) 디자인 툴 연결하면 신입이 디자인 보고 코드 생성. 표준 규격이라 한번 표준이 정해지면 어떤 외부 서비스든 같은 방식으로 꽂힘.
- **팁 38** MCP 설치도 신입에게 시키기.
- **팁 39** 서브에이전트 = 분신술: 신입이 똑같은 신입을 하나 더 불러 따로 일 시킴. **오해 주의** — CEO·디자인·테스팅 에이전트처럼 역할별 분신을 막 만드는 건 최악. [의견] 이유: 테스팅 분신이 일 잘하려면 메인의 작업 내용을 다 알아야 하는데, 분신은 메인 책상 자료를 못 가져가고 새 책상에서 일함. 분신은 **독립적인 일(자료 조사 등)에만** 쓸 것.
- **팁 40** 지시 과부하 피하기: 매뉴얼 잔뜩 쌓고 명령 잔뜩 던지면 혼란. 컨텍스트는 "신선·압축·관련" 원칙, 필요한 자료만 그때그때 꺼내기.

### 7. 병렬 실행·고급 자동화 (팁 41~50)
- **팁 41** 신입 여러 명 띄우기: 화자는 항상 3~4명을 동시에 — 한 명은 기능 작업, 옆은 코드 분석, 옆은 다른 프로젝트. 한 명이 일하는 동안 다른 명에게 다음 일 설명·명령. 멀티태스킹. [의견]
- **팁 42** iTerm으로 책상 나누기: iTerm은 화면을 여러 칸으로 분할(Cmd+D로 세 칸, Cmd+화살표로 이동). tmux 같은 툴도 추천. 화자 본인은 편의상 **Antigravity의 터미널 탭**으로 멀티창 유지. [의견]
- **팁 43** 알람 켜기: "일 끝나면 소리 울리게 해줘" → 신입이 알림 설정도 바꿔줌. 한때 음성 합성으로 결과 요약까지 읽게 했으나 불편해서 뺌. [의견]
- **팁 44** 워크트리로 책상 복제: 신입 여러 명이 같은 프로젝트를 만지면 자료 충돌. git worktree로 책상 분리하면 충돌 이슈 고민 불필요.
- **팁 45** `/chrome`으로 브라우저 주기: 신입이 실제 웹 브라우저를 조작 가능. Playwright 같은 걸 쓰면 자동으로 컴퓨터를 조작.
- **팁 46** 디버깅·QA에 강력: 모든 권한을 주고 QA/디버깅을 시켜도 충분히 강력한 성능. (주장)
- **팁 47** 훅(hook)은 자동 트리거 설정: "이런 상황 되면 자동으로 이거 해라" 규칙. 현관 센서등처럼 어떤 작업 전에 자동 실행되게.
- **팁 48** 작업 끝나면 자동 정리: 파일 수정이 끝날 때마다 코드 정리·검사(린트) 자동 실행. 한 번 설정하면 매번 결과물이 깔끔(요리 끝날 때마다 접시 자동 세척).
- **팁 49** 위험 명령어 차단: 파일 통째 삭제 같은 명령어는 자동 차단(팁 25 재강조).

"이것도 해봐, 저것도 해봐" 식으로 계속 던지면 신입 책상의 종이가 산처럼 쌓이고, 나중엔 자기가 뭘 하려 했는지 까먹는다. 그래서 처음에 좋은 계획 하나를 만드는 게 중요하다 — 그게 기준점.

### [10:06–10:21] 팁 28 — 일 끝나면 노트에 저장
신입은 매일 기억이 리셋된다고 했으니, "오늘 한 거 노트에 정리해줘"라고 시킨다. 다음날 그 노트를 표면(보면) 신입이 어제 자기가 뭘 했는지 알 수 있다. 화자는 이걸 세컨드 브레인, 외부 노트장 같은 것이라 부른다. [의견]

### [10:21–10:36] 팁 29 — 노트는 필요할 때만 펴기
매일 출근하자마자 노트를 다 뽑아 외우면 책상이 무거워진다. "이번 일에 관련된 노트만 가져와"라고 시킨다. 이런 노트를 컨텍스트라고도 부른다. 할 일 목록도 이렇게 관리하며, 따로 프로젝트 관리 도구는 안 쓴다.

### [10:36–10:53] 팁 30 — 검증 명령어를 무조건 주기
신입이 결과물을 만들면 스스로 확인하게 해야 한다. 모바일 앱이면 "Xcode로 빌드해봐", 웹사이트면 "브라우저 열어서 눌러봐". 이런 QA 과정 혹은 빌드 과정이 무조건 필요하며, 심지어 디버깅도 가능하다.

### [10:53–11:07] 팁 31 — 복잡한 건 시니어 모델
화자는 거의 항상 Opus 모델만 쓴다. 좀 느리다. 근데 어차피 신입 여러 명을 동시에 돌리니, 하나가 느려도 다른 데 가서 일 시키면 체감 느림이 거의 없다. [의견]

### [11:07–11:35] 팁 32 — 신입의 생각 들여다보기(thinking block)
신입이 작업하는 동안 자기 생각을 적는데 그게 띵킹 블록이다. 거기서 "확실하지 않은데"라든가 이상한 과정이 보이면 즉시 끊는다 — 잘못된 길을 가고 있는 것. 굉장히 중요한 테크닉으로, 특히 초보자 중 AI를 돌려놓고 아무것도 안 하는 사람들이 있는데 이건 무조건 해야 한다.

> "만약에 얘가 띵킹 블록에서 이상한 헛소리를 하는 거 같다, 그 즉시 ESC 누르고 멈추셔야 됩니다." [11:29] [의견]

### [11:35–11:47] 3단계 예고 — 4가지 부품
여기까지가 일하는 기본기. 다음은 진짜 핵심인 네 가지 부품: 스킬, 명령어, MCP, 서브에이전트. 이름이 어려워 보이지만 하나씩 풀어준다.

### [11:47–11:58] 팁 33 — 네 가지는 다 조합됨
네 가지가 다 서로 조합된다. 레고 블록 네 종류라고 보면 된다 — 각자 모양은 다르지만 끼워 맞춰 더 큰 걸 만들 수 있다. 화자는 이를 "조합성(composability) 프레임워크"라 부를 수 있지 않을까 생각한다. [의견]

### [11:58–12:25] 팁 34 — 스킬은 반복 작업 묶음
스킬은 반복 작업 묶음이자 도메인 전문가 지식이 들어간 묶음. 예를 들어 "해커뉴스 가서 iOS 뉴스 가져와 노트에 정리해줘" 혹은 "네이버 뉴스 일면 정보 가져와줘" — 이게 가져오기·정리하기 두 단계인데, 이걸 "스킬 만들어줘" 하면 다음엔 그 스킬을 트리거하는 것만으로 정말 간단하게 된다.

### [12:25–12:30] 팁 35 — 명령어는 스킬의 단축어
스킬을 만들면 자동으로 슬래시 명령어도 같이 생긴다.

### [12:30–12:43] 팁 36 — 손으로 만들지 마세요
스킬도 명령어도 매뉴얼도 직접 손으로 고치는 건 비추천. Claude Code에게 시켜라. 신입이 자기 매뉴얼을 자기가 업데이트하는 것이며, 관리자는 검수만 하면 된다. [의견]

### [12:43–13:00] 팁 37 — MCP는 외부 서비스 연결선
MCP는 신입이 외부 도구랑 통화하는 전화기 같은 것. 예를 들어 디자인 툴을 연결하면 신입이 디자인을 보고 코드를 만들 수 있다. 표준화돼 있어 규격 같은 것이며, 일단 한번 표준이 정해지면 어떤 외부 서비스든 같은 방식으로 꽂힌다.

### [13:00–13:03] 팁 38 — MCP 설치도 신입에게
MCP 설치도 신입(Claude)에게 시키길 추천.

### [13:03–13:32] 팁 39 — 서브에이전트는 분신술
서브에이전트는 신입이 자기랑 똑같은 신입을 한 명 더 불러 따로 일 시키는 것. "내가 본 작업은 따로 두고 너는 저 일만 해." 그런데 오해가 많다 — 사람들이 CEO 에이전트, 디자인 에이전트, 테스팅 에이전트 식으로 분신을 막 만든다. 화자는 이게 진짜 최악이라고 생각한다. [의견] 왜냐면 테스팅하는 분신이 일을 잘하려면 메인 신입의 작업 내용을 다 알아야 하는데, 분신은 메인 책상 자료를 못 가져가고 새 책상에서 일하기 때문. 분신은 진짜 독립적인 일(자료 조사 같은 것)에만 쓸 것.

### [13:32–13:45] 팁 40 — 지시 과부하 피하기
신입 책상에 매뉴얼 쌓을 거 다 쌓고 명령도 잔뜩 던지면 신입이 혼란스러워한다. 컨텍스트는 신선하게·압축돼 있게·관련 있게가 원칙. 필요한 자료만 그때그때 꺼내서 일 시켜라.

### [13:45–13:54] 4단계 예고 — 여러 명 동시에
여기까지가 신입 한 명 잘 부려먹는 법이었다면, 다음은 신입 여러 명을 동시에 부리는 법. 이걸 한번 익히면 일하는 방식이 달라진다.

### [13:54–14:11] 팁 41 — 신입 여러 명 띄우기
화자는 항상 신입 셋에서 네 명을 동시에 일 시킨다. 한 명한테는 "이 기능 작업해", 옆에서는 "다른 코드 분석해", 또 옆에서는 다른 프로젝트. 한 명이 일하는 동안 다른 명에게 가서 다음 일을 설명하고 또 옆에 가서 명령을 준다. 멀티태스킹.

### [14:11–14:34] 팁 42 — iTerm으로 책상 나누기
iTerm은 터미널 프로그램으로 화면을 여러 칸으로 나눠준다. Cmd+D를 누르면 세 칸이 생기고 Cmd+화살표로 칸 사이를 오간다. tmux 같은 툴을 쓰는 것도 추천. 다만 화자 본인은 편의를 위해 **Antigravity**에 있는 터미널 탭을 이용해 여러 멀티창을 유지한다. [의견]

### [14:34–14:47] 팁 43 — 알람 켜기
"일 끝나면 딸랑 하고 소리 울리게 해줘"라고 하면 신입이 자기 알림 설정도 바꿔준다. 한때 음성 합성으로 결과 요약까지 읽어주게 했었는데 이건 조금 불편해서 뺐다. [의견]

### [14:47–15:02] 팁 44 — 워크트리로 책상 복제
신입 여러 명이 같은 프로젝트를 만지면 자료 충돌이 난다. git worktree 같은 걸 이용해 책상을 분리해주면 문제가 없다. 충돌 이슈도 딱히 고민 안 해도 된다.

### [15:02–15:15] 팁 45 — `/chrome`으로 신입에게 브라우저 주기
`/chrome`을 치면 신입이 진짜 웹 브라우저를 조작할 수도 있다. 혹은 Playwright 같은 걸 이용하면 자동으로 얘가 알아서 컴퓨터를 조작한다.

### [15:15–15:29] 팁 46 — 디버깅·QA에 강력
디버깅과 QA에도 굉장히 강력하다. 이 친구한테 모든 권한을 주고 QA나 디버깅을 시켜도 충분히 강력한 성능을 낼 수 있다. (주장)

### [15:29–15:41] 팁 47 — 훅은 자동 트리거
훅(hook)은 "이런 상황 되면 자동으로 이거 해라"는 규칙. 현관 센서등 같은 것 — 누가 지나가면 자동으로 켜진다. 신입이 어떤 작업을 하기 전에 자동으로 뭔가 실행되게 할 수 있다.

### [15:41–15:53] 팁 48 — 작업 끝나면 자동 정리
신입이 파일 수정을 끝낼 때마다 자동으로 코드 정리·검사(린트)를 돌리게 해라. 한 번 설정하면 매번 결과물이 깔끔하다. 음식점으로 치면 요리가 끝날 때마다 자동으로 접시가 닦이는 것.

### [15:53–15:59] 팁 49 — 위험 명령어는 차단
파일 통째 삭제 같은 명령어는 자동으로 막아둘 것(팁 25 재강조).

### [15:59–16:12] 팁 50 — 플러그인 둘러보기
플러그인은 네 가지 부품을 묶어놓은 세트 — 스킬·외부도구·분신·자동화 스크립트를 한 묶음으로 만들어 공유하는 것. Anthropic 공식 플러그인도 있고 다른 서드파티도 있다.

### [16:12–16:38] 마무리 — 컨텍스트가 왕
50개를 다 봤다. 하나만 기억한다면 "컨텍스트가 왕"이다. 신선하게·압축돼 있게·관련 있게, 신입에게 필요한 것만 주고 그 이상을 주지 말라. 정보를 많이 줄수록 똑똑해지는 게 아니라 헷갈리기만 한다. 다시 정리하면: 계획 모드 혹은 Superpowers의 브레인스토밍이나 gstack의 오피스타워 같은 브레인스토밍 기술로 시작 → 검증 명령어 박아두기 → 트리거 단어로 스킬 호출 → 여러 명 동시에 돌리기. 이 흐름을 한번 익히면 일하는 속도가 진짜 달라진다. (주장) [의견]
