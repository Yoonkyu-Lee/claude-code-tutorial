## 요약 (주제별)

### 1. 하니스 엔지니어링의 정의
- 하니스 엔지니어링은 컨텍스트 엔지니어링을 **포함**해서 AI 에이전트를 감싸는 **모든 설정 파일 = 프로젝트 자체**를 의미한다.
- 구성 요소: (1) `CLAUDE.md` 같은 프로젝트 지시사항, (2) MCP 서버 설정(AI가 쓸 외부 도구 연결), (3) 스킬 파일(전문 기술을 가르치는 설정), (4) 에이전트 설정(유저가 커스텀 전문가를 주입).
- 이 네 가지가 전부 합쳐진 것이 하니스이며, 컨텍스트 엔지니어링은 그 일부다.

### 2. 프롬프트 → 컨텍스트 → 하니스로 이어지는 메타 변화
- 프롬프트 엔지니어링 = "뭘 물어볼까"의 기술(역할 부여, 단계별 지시, 예시 삽입).
- 컨텍스트 엔지니어링 = "뭘 보여줄까"의 기술. 25년에 등장.
- 하니스 엔지니어링 = 컨텍스트+MCP+스킬+에이전트까지 합친 "전체 환경을 어떻게 설계할까"의 기술.
- 근거로 인용된 발언(주장/미검증): 테슬라 AI를 이끌던 카파시(카파스)가 "프롬프트보다 컨텍스트 엔지니어링이 핵심"이라 했고, 가트너가 "프롬프트 엔지니어링 시대는 끝났다"고 공식 선언했다고 화자가 전한다.
- [의견] 화자는 "메타가 빨리 바뀐다"며 최근 또 한 단계 넘어가 하니스 엔지니어링 단계가 왔다고 본다.

### 3. AI 에이전트의 자율 실행 시대 (배경)
- 이제 AI는 질문 한 번·대답 한 번 수준이 아니라 혼자서 수십~수백 단계를 알아서 실행한다(주장).
- 오픈AI는 코덱스라는 AI 에이전트만으로 내부 제품을 만들고 있다(주장).
- 마누스라는 서비스는 AI 하나가 한 작업에서 도구를 평균 50번 스스로 쓴다(주장/미검증).

### 4. 핵심 명제 — "모델이 병목이 아니라 하니스가 병목이다"
- 지금까지는 "이 AI는 똑똑하다/저 AI는 멍청하다"며 모델 성능만 비교했지만, 실제로는 같은 모델이라도 환경 세팅에 따라 결과가 **열 배** 차이 날 수 있다(주장).
- 비유: 똑같은 축구 선수라도 좋은 감독 밑에서는 발롱도르를 타고, 엉망인 팀에서는 벤치만 달군다.

### 5. 오픈AI 코덱스 내부 실험 사례 (주장/미검증)
- 오픈AI가 25년 8월부터 코덱스 에이전트만으로 내부 제품을 만드는 실험을 시작.
- 결과(주장): 수동 작성 코드 0(제로), 5개월 만에 100만 줄 코드 생성, 1500개 PR 머지, 엔지니어 1명이 하루 평균 3.5개 작업 수행.
- 교훈: 처음에는 생각보다 훨씬 느렸는데, 이유는 코덱스가 무능해서가 아니라 하니스 세팅이 제대로 안 됐기 때문(환경 설정 부족, 도구 연결 어설픔, 에러 복구 로직 없음). 이를 하나씩 잡아 나가자 성과가 폭발.
- 오픈AI 내부 발언(주장): "엔지니어링 팀의 주된 역할이 에이전트가 유용한 일을 할 수 있게 만드는 것이 됐다." 즉 코드를 짜는 게 아니라 에이전트가 코드를 잘 짤 수 있는 환경을 만드는 게 일이 됐다.

### 6. 실전 — "그래서 나는 뭘 하면 되는가" 3단계
- 첫째, `CLAUDE.md` 같은 컨텍스트 파일을 제대로 써라(하니스의 기본 중 기본). 프로젝트 구조·규칙·패턴을 알려주는 것.
- 둘째, MCP 서버를 세팅하라. MCP = AI 에이전트가 외부 도구(브라우저 자동화, 문서 검색, 디자인 도구 연결 등)를 쓸 수 있게 하는 연결.
  - 단, MCP는 많이 줄수록 좋은 게 아니다. 또 사용 시 토큰이 터지지 않도록 여러 테크닉이 필요하다(이 부분은 나중에 별도 영상에서 다룬다고 예고).
- 셋째, 스킬 파일을 만들어라. 스킬 = 특정 작업의 전문 지식을 파일로 정리(예: "영상 편집은 이 워크플로우로", "코드 리뷰는 이 기준으로"). 이러면 AI가 그 분야 전문가처럼 일한다.

### 7. 왜 하니스가 지속적 경쟁 우위인가
- 모델은 몇 주 만에 따라잡힌다(GPT가 좋으면 클로드가 쫓아오는 식).
- 하니스는 잘 만들려면 시간이 걸린다: 도구 연결, 에러 처리, 승인 흐름, 개발자 사용 방법론까지 다 익혀야 함.
- [의견] 그래서 지금 하니스에 투자하는 회사가 나중에 진짜 경쟁 우위를 갖게 된다. 모델은 비슷해져도 하니스는 쉽게 복제되지 않는다.
- [의견] 개인도 마찬가지 — AI 도구를 쓸 줄 아는 건 이제 기본이고, 차이를 만드는 건 그 AI를 얼마나 잘 세팅하느냐다. "25년이 컨텍스트 엔지니어링의 해였다면 26년은 하니스의 해."

### 8. 세일즈 구조 (구독자 이벤트)
- 영상 말미에 구독자 이벤트 안내: 구독·좋아요·알림 설정·저장 후 고정 댓글을 확인하면, 화자의 노하우가 담긴 "하니스 코드" 깃허브 주소를 다운로드받는 방법이 적혀 있다고 안내.

## 3줄 오리엔테이션
- 프롬프트→컨텍스트→하니스로 이어지는 "메타 변화"를 정의하고, 하니스 = AI를 감싸는 모든 설정 파일(프로젝트 전체)이라 규정한다.
- 핵심 명제는 "모델이 아니라 하니스가 병목" — 같은 모델도 환경 세팅에 따라 결과가 열 배 차이 난다(주장), 근거로 오픈AI 코덱스 내부 실험을 든다.
- 실천은 3가지(컨텍스트 파일·MCP·스킬)로 정리되고, 마지막에 구독자 이벤트로 하니스 코드 깃허브를 배포한다.

### [00:00–00:13] 도입 — 컨텍스트에서 하니스로
저번 영상에서 컨텍스트 엔지니어링을 다뤘는데, 이번엔 하니스 엔지니어링이라며 "메타가 빨리 바뀐다"고 운을 뗀다. 이번 영상 중에 구독자 이벤트를 안내할 테니 집중해서 시청하라고 예고한다(세일즈 예고).

### [00:13–00:23] 프롬프트 엔지니어링의 시대
챗GPT가 처음 나왔을 때는 "질문만 잘하면 된다 — 역할을 줘라, 단계별로 시켜라, 예시를 넣어라"가 프롬프트 엔지니어링이었다고 회고한다.

### [00:23–00:43] 이제 AI가 하는 일 (자율 에이전트)
지금 AI가 하는 일을 보라며 사례를 든다(주장/미검증): 오픈AI는 코덱스라는 AI 에이전트만으로 내부 제품을 만들고 있고, 마누스라는 서비스는 AI 하나가 한 작업에서 도구를 평균 50번이나 스스로 쓴다. AI가 질문 한 번·대답 한 번 하는 수준이 아니라, 혼자서 수십~수백 단계를 알아서 실행하는 시대가 됐다는 것.

### [00:43–01:03] 컨텍스트 엔지니어링의 등장
> "가트너는 프롬프트 엔지니어링 시대는 끝났다고 공식 선언했거든요." [00:54]

25년에 컨텍스트 엔지니어링이라는 말이 나왔고, 테슬라 AI를 이끌던 카파시가 "프롬프트보다 컨텍스트 엔지니어링이 핵심"이라 했다(주장). 질문을 잘 쓰는 게 아니라 AI한테 뭘 보여줄지 설계하는 게 더 중요해졌다는 정리.

### [01:03–01:37] 하니스 엔지니어링의 정의
최근 또 한 단계 넘어갔다며 하니스 엔지니어링을 정의한다. 하니스 엔지니어링은 컨텍스트 엔지니어링을 포함해 AI 에이전트를 감싸는 **모든 설정 파일 = 프로젝트 자체**를 의미한다. 구체적 구성: (1) `CLAUDE.md` 같은 프로젝트 지시사항, (2) MCP 서버 설정(AI가 쓸 외부 도구 연결), (3) 스킬 파일(전문 기술을 가르치는 설정), (4) 에이전트 설정(유저가 커스텀 전문가를 주입). 이 모두가 합쳐진 것이 하니스이고, 컨텍스트 엔지니어링은 그 일부다.

### [01:37–02:07] 주방 비유
쉬운 비유로 설명한다. 컨텍스트 엔지니어링이 요리사에게 좋은 재료를 골라주는 것이라면, MCP 설정은 외부 업체(수산시장·정육점·농장)와 직통 연결하는 어댑터다. 스킬 파일은 레시피 정리, 에이전트는 셰프를 만드는 것. 이걸 다 합쳐 주방 전체를 설계하는 게 하니스 엔지니어링이다. [의견] 아무리 좋은 재료를 줘도 주방이 엉망이면 좋은 요리가 안 나오고, 주방이 잘 세팅되면 평범한 재료로도 훌륭한 요리가 나온다.

### [02:10–02:35] 핵심 명제 — 하니스가 병목이다
진짜 중요한 얘기라며 명제를 던진다.
> "모델이 병목이 아니라 하니스가 병목이다." [02:12]

지금까지는 "이 AI는 똑똑하다, 저 AI는 멍청하다"며 모델 성능만 비교했지만, 실제로는 같은 모델이라도 환경 세팅에 따라 결과가 열 배 차이 날 수 있다(주장). 비유하면, 똑같은 축구 선수라도 좋은 감독 밑에서는 발롱도르를 타고 엉망인 팀에서는 벤치만 달구는 것과 같다.

### [02:35–03:00] 오픈AI 코덱스 내부 실험
여기까지가 기본이고 진짜 인상적인 사례는 지금부터라며 오픈AI 사례를 든다(주장/미검증). 오픈AI가 25년 8월부터 자기들이 만든 코덱스 에이전트만으로 내부 제품을 만들기 시작했는데, 결과는 수동 작성 코드 0(제로), 5개월 만에 100만 줄 코드 생성, 1500개 PR 머지, 엔지니어 1명이 하루 평균 3.5개 작업 수행. 진짜 교훈은, 처음에는 생각보다 훨씬 느렸다는 것 — 코덱스가 무능해서가 아니라 하니스 세팅이 제대로 안 됐기 때문이다. 환경 설정이 부족하고, 도구 연결이 어설프고, 에러 복구 로직이 없었다. 이걸 하나하나 잡아 나가니 그때부터 폭발적으로 성과가 났다.

### [03:00–03:28] 오픈AI 내부의 역할 변화
오픈AI 내부에서 이런 말이 나왔다고 전한다(주장):
> "엔지니어링 팀의 주된 역할이 에이전트가 유용한 일을 할 수 있게 만드는 것이 됐다." [03:19]

즉 코드를 짜는 게 아니라, 에이전트가 코드를 잘 짤 수 있는 환경을 만드는 게 일이 됐다는 것.

### [03:30–03:48] 3단계 개념 정리
여기까지 정리한다. 프롬프트 엔지니어링은 "뭘 물어볼까"의 기술, 컨텍스트 엔지니어링은 "뭘 보여줄까"의 기술, 하니스 엔지니어링은 컨텍스트+MCP+스킬+에이전트까지 다 합쳐 AI를 감싸는 전체 환경을 어떻게 설계할까의 기술. 그리고 지금부터가 오늘 영상에서 가장 중요한 부분, "그래서 나는 뭐 하면 되는데"라고 넘어간다.

### [03:48–04:01] 실천 ① 컨텍스트 파일
첫째, `CLAUDE.md` 같은 컨텍스트 파일을 제대로 쓰라고 강조한다. 이게 하니스의 기본 중 기본. 이 프로젝트는 이런 구조이고, 이런 규칙을 따르고, 이런 패턴을 쓰라고 알려주는 것.

### [04:01–04:11] 실천 ② MCP 서버
둘째, MCP 서버를 세팅하라. MCP는 AI 에이전트가 외부 도구를 쓸 수 있게 연결해 주는 것 — 브라우저 자동화, 문서 검색, 디자인 도구 연결 등.

### [04:12–04:22] MCP 주의 — 많다고 좋은 게 아니다
다만 MCP는 많이 주면 줄수록 좋지 않다. 또 MCP를 쓸 때 토큰이 터지지 않도록 여러 테크닉을 써야 하는데, 이 부분은 나중에 별도 영상으로 다루겠다고 예고한다.

### [04:23–04:36] 실천 ③ 스킬 파일
셋째, 스킬 파일을 만들어라. 스킬은 AI에게 특정 작업의 전문 지식을 파일로 정리해 주는 것. 예를 들어 "영상 편집은 이 워크플로우로 해", "코드 리뷰는 이 기준으로 해" 같은 걸 파일로 만들어 두면 AI가 그 분야 전문가처럼 일한다.

### [04:38–05:14] 왜 하니스가 지속적 우위인가
마지막으로 하나 더 짚는다. 모델은 몇 주 만에 따라잡을 수 있다 — GPT가 좋으면 클로드가 쫓아오는 식. 하지만 하니스는 잘 만들려면 시간이 걸린다: 도구 연결, 에러 처리, 승인 흐름, 그리고 개발자가 어떻게 쓰느냐 방법론까지 전부 익혀야 한다. [의견] 그래서 지금 하니스에 투자하는 회사가 나중에 진짜 경쟁 우위를 갖게 된다 — 모델은 다 비슷해질 수 있어도 하니스는 쉽게 복제가 안 된다. 개인도 마찬가지로, AI 도구를 쓸 줄 아는 건 이제 기본이고 차이를 만드는 건 그 AI를 얼마나 잘 세팅하느냐다.

### [05:14–05:47] 최종 정리 4가지
정리한다. 하나, 하니스 엔지니어링은 컨텍스트 엔지니어링에 MCP·스킬·에이전트 설정까지 합친 전체 환경 설계다. 둘, 같은 AI 모델이라도 하니스에 따라 열 배 차이 날 수 있다. 셋, 오픈AI도 코드를 짜는 게 아니라 에이전트 환경을 만드는 게 일이 됐다. 넷, 컨텍스트 파일·MCP 연결·스킬 정리 이 세 가지부터 시작하면 된다.
> "25년이 컨텍스트 엔지니어링의 해였다면 26년은 하니스의 해입니다." [05:36]

[의견] AI가 아무리 똑똑해져도 결국 환경을 어떻게 만들어 주느냐가 결과를 결정한다.

### [05:48–06:02] 구독자 이벤트 (세일즈)
마지막으로 구독자 이벤트를 진행한다. 구독·좋아요·알림 설정·저장까지 한 다음 고정 댓글을 확인하면, 화자의 모든 노하우가 담긴 하니스 코드가 있는 깃허브 주소를 다운로드받는 방법을 적어 뒀다고 안내하며 마무리한다.
