## 요약 (주제별)

### 전체 개요 — "에반스 워크플로우"
- 이 강의는 화자(메이커 에반)가 자주 쓰는 스킬 조합을 소개하는 자리다. 총 4가지 스킬을 콤비네이션해서 쓴다: **G스택**(기획) / **슈퍼파워즈(Superpowers)**(실행) / **매트 포콕(Matt Pocock) 스킬**(유지보수) / **코덱스 플러그인 CC**(검증).
- 각 스킬의 역할이 다르기 때문에 조합이 강력하다는 것이 핵심 주장. 목차: 왜 여러 스킬을 조합하나 → 슈퍼파워즈 → 매포(매트 포콕)의 스킬 → G스택 → 코덱스 플러그인 CC → 통합 워크플로우 → 마무리.

### 왜 여러 스킬을 조합해야 하나 — AI 코딩의 4가지 실패 패턴
- 전제: AI 코딩 에이전트는 똑똑하지만 "시니어 엔지니어처럼" 일하지 않는다. "알아서 해 줘"식 요청 때문에 한계가 명확하다. [의견]
- 실패 패턴 4가지: (1) **요구사항 이해 전에 코딩** — 스펙이 흐릿한 상태에서 코드부터 씀. (2) **같은 맥락 반복 설명** — 세션마다 도메인 용어를 다시 설명하고, 설명이 매번 달라짐. (3) **동작 안 하는 코드 생성** — AI가 "완성했다"고 거짓말하는 케이스. (4) **AI 슬롭 양산** — 빠른 속도로 짜다 보니, AI는 "평균에 회귀"하는 특성이 있어 뾰족한 한 점으로 못 가고 특정 구간을 맴돌아 슬롭이 계속 생김. [의견]

### 스킬이란 무엇인가
- 스킬 = 클로드가 작업 전에 반드시 읽고 따르는 매뉴얼. 회사의 매뉴얼에 비유. 모델의 재학습 없이 **행동만** 변경한다.
- 포맷: SKILL.md와 MD 스크립트로 구성. 호출은 슬래시 커맨드로 함. 스킬을 기반으로 도메인 지식을 주입해 알아서 일하게 만드는 것.

### 4개 스킬의 역할 분담
- **G스택 = 기획**("왜 만드는가"에 집중). 없으면 기획이 부재.
- **슈퍼파워즈 = 실행**(효율적 개발 워크플로우, 컨텍스트 안 터지고 개발). 없으면 실행이 부재.
- **매포의 스킬 = 유지보수**(유지보수 가능한 코드로 바꿈, 도메인 지식과 매핑). 없으면 유지보수가 부재.
- **코덱스 플러그인 CC = 검증**(다른 모델·다른 시각으로 리뷰). 없으면 검증이 부재.

### 슈퍼파워즈 (실행) — 화자가 쓰는 3개 스킬
- 슈퍼파워즈는 "시니어 엔지니어의 작업 습관을 마크다운으로 정리한 것". 스타 수가 매우 많고, 그중 화자는 3개만 사용.
- **브레인스토밍(Brainstorming)**: 프로젝트 파일·최근 커밋을 탐색 → 한 번에 질문 하나만 → 목적·제약·성공 기준 수립 → 2~3가지 접근법과 트레이드오프 제시 → 섹션별 승인 후에만 진행 → spec.md에 저장 후 커밋 → Writing Plan을 자동 호출. **핵심: 설계 승인 전까지 어떤 코드도 작성하지 않음.**
- **라이팅 플랜즈(Writing Plans)**: 브레인스토밍 결과로 확정된 스펙을 기반으로 플랜을 작성.
- **서브에이전트 드리븐 디벨롭먼트(Subagent-driven development)**: 3개 코스로 나뉨 — 임플리멘터(구현·테스트·셀프리뷰·커밋) / 스펙 컴플라이언스(요구사항 대비 검증·검수) / 코드 퀄리티(품질 검토, 스펙 통과 후에만 시작). 서브에이전트 기반으로 아주 작은 단위를 새 컨텍스트에서 구현하며, 테스트가 꼼꼼히 붙어 도움이 크다는 평. [의견]

### 매트 포콕의 스킬 (유지보수) — 도메인 언어와 모듈 깊이
- 철학: "에이전트는 코딩만 빠르게 하지 않는다, 코드 부패도 빠르게 만든다". 화자는 이 팩에서 2개 스킬 사용.
- **그릴 위드 독스(gird/build with docs)**: `context.md`와 `ADR` 파일을 참조.
  - `context.md` = **도메인 언어 사전**. 용어의 의미를 적는 파일이며 구현은 쓰지 않음. 모든 스킬이 작업 시작 시 자동으로 읽고, 새 결정이 나오면 대화 도중 바로 갱신 → 매 세션 맥락 재설명이 불필요.
  - `ADR` = **결정 기록**. 되돌리기 어려운 결정만 기록. 맥락 없이 보면 의아한 선택을 설명하고, 대안·트레이드오프를 함께 적음. 이미 끝난 논의는 다시 꺼내지 않음.
  - 동작: 기존 도메인 용어를 먼저 읽음 → 새 용어 발견 시 정의 추가 → 한 번에 질문 하나 → 추측 대신 코드베이스를 봄 → 추천 답변과 근거 제시 → ADR에 기록된 선택은 재논의 차단, 새 결정만 챌린지 → 대화 중 context.md 갱신.
  - 효과: 용어를 통일해 AI에게 계속 주입 → 같은 용어를 쓰는 AI가 확립되어 이격이 사라짐. 특히 **결제 쪽 이슈**에서 효과를 봤다는 경험담. context/ADR을 잘 정의해 두면 화자가 잘못 지시해도 AI가 반박하는 현상이 나옴. [의견]
- **임프로브 코드베이스 아키텍처(Improve codebase architecture)**: 얕은(shallow) 모듈을 **깊은 모듈**(재사용·테스트 가능, 유지보수 쉬운 코드)로 만드는 스킬. 동작: context.md·ADR 로드 → 모듈 경계와 마찰 지점(각 인터페이스의 경계) 탐색 → **삭제 테스트**(지워 봤을 때 복잡도가 주는지 느는지) → 비포/애프터 보고서 생성 → 추천 → 사용자 선택 후 검증.

### G스택 (기획)
- 게리 탄(Y Combinator 사장)이 만든 스킬. 철학: "한 명의 빌더가 올바른 도구를 가지면 팀보다 빠르게 움직인다". [의견]
- **오피스 아워즈(Office Hours)**: YC식 6가지 강제 질문. 코드 작성 없음. (1) 디맨드 리얼리티 — 구체적 수요 증가 증거는? (2) 스테이터스 코(status quo) — 사용자의 현재 임시방편은? (3) 디스퍼릿 스펙(desperate spec) — 지금 이 도구가 가장 필요한 한 명은? (4) 낼리스트 웨이(narrowest way) — 돈 낼 가치가 있는 최소는? (5) 서프라이즈 — 실제 관찰해서 발견한 것은? (6) 퓨처 — 세상이 변할수록 더 필수가 되는 도구인가? 꽤 집요하게 물어보니 체험 추천.
- **브라우즈(Browse)**: 헤드리스 크로미움 처리. 일반 Playwright보다 똑똑하게 동작하는 것 같아 애용. [의견]

### 코덱스 플러그인 CC (검증)
- OpenAI가 공식으로 낸 플러그인. GPT-5 모델을 이용해 앤트로픽 클로드가 짠 코드를 리뷰. [불명확: 자막 "GPTU 5"] 3가지 모드:
  - **코덱스 리뷰**: 변경 사항을 객관적으로 평가, 클로드 리뷰의 보완점으로 사용.
  - **코덱스 애드버서리얼(adversarial)**: 결함이 있다고 가정하고 잡는 기능.
  - **코덱스 레스큐**: 클로드가 막힌 작업을 코덱스에 위임하고 결과를 비교.
- 화자 사용 습관: 보통 코덱스 리뷰를 많이 씀. **코덱스 레스큐는 버그가 있어 가끔 행이 걸려서 안 쓰기를 추천.** 모드별 용도 — PR 마무리 전 일반 검토=코덱스 리뷰, 보안·결제 등 위험 코드=코덱스 애드버서리얼, 버그 3회 이상 실패=코덱스 레스큐, 릴리스 직전 최종 점검=코덱스 애드버서리얼, 새 라이브러리 적용=코덱스 리뷰. (단 레스큐는 애매하다고 재차 언급.) [의견]

### 통합 워크플로우
- 순서: (1) 기획 = office hours + gird-with-docs → (2) 스펙 작성 = 브레인스토밍 + writing plan → (3) 개발 = 서브에이전트 드리븐 디벨롭먼트 → (4) 검증 = 코덱스 + 브라우저 → (5) 코드베이스 유지 = 임프로브 코드베이스 아키텍처.

### 기억할 5가지
- (1) 모델보다 작업 방식이 중요 — 같은 모델이라도 방법이 결과를 결정.
- (2) 단계마다 역할 분리 — 기획·개발·리뷰를 한 페르소나로 섞지 않기.
- (3) 기획 게이트를 먼저 통과 — 수요와 도메인 언어 정리 없이 코드 쓰지 않기.
- (4) 외부 모델로 사각지대 점검 — 클로드 셀프 리뷰만으로는 같은 패턴을 못 봄.
- (5) 실제 화면을 확인해 QA — 테스트 통과가 사용자 작동을 보장하지 않으므로 셀프 QA 필수.

### 실습 절차 (문서로 정리한 예)
- **기획 파트**: G스택 office hours를 켜고 "나는 프롬프트 ○○를 하려 해"라고 입력 → G스택이 계속 질문하면 답변 → 슈퍼파워즈 브레인스토밍을 켜 지금까지 내용 기반으로 스펙 작성 → 매포의 gird-with-docs로 도메인 지식·ADR을 물어보면 답변 → 라이팅 플랜 트리거 → 스펙/플랜 문서 저장.
- **개발 파트**: 다른 세션(또는 클리어)로 전환 권장 → "슈퍼파워즈 플랜스 이어서 작성해 놓고, 서브에이전트 드리븐 디벨롭먼트로 개발하고, 코덱스로 리뷰해 줘" 트리거 → 두세 시간 알아서 처리.
- **유지보수 세션**: 여러 세션을 병렬로 — 임프로브 코드베이스 아키텍처로 리팩토링, 버셀 React 베스트 프랙티스 + Next.js 프론트엔드 가이드라인으로 리팩토링, 패스트API 백엔드 가이드라인으로 리팩토링을 따로따로 실행 → 이상한 코드를 계속 제거. 이 작업은 필수이며 토큰을 많이 쓰지만 안정적 운영을 위해선 필수. [의견]
- 메타프롬프트를 링크로 공유 예정이니 실습을 추천. 훨씬 달라진 것을 체감할 수 있다고 마무리. [의견]

## 3줄 오리엔테이션
- 이 강의는 화자가 실제로 쓰는 4개 스킬(G스택=기획 / 슈퍼파워즈=실행 / 매트 포콕=유지보수 / 코덱스=검증)을 각자의 역할로 나눠 조합하는 "에반스 워크플로우"를 소개한다.
- AI 코딩의 4대 실패 패턴(스펙 없이 코딩, 맥락 반복, 거짓 완성, 슬롭 양산)을 스킬 조합과 역할 분리로 막는 것이 핵심 논리.
- 후반부는 이 스킬들을 기획→스펙→개발→검증→유지보수로 잇는 통합 워크플로우와, 그것을 문서로 정리한 실습 절차다.

## 전문 — 시간순 워크스루

### [00:00–00:20] 오프닝 — 에반스 워크플로우 소개
메이커 에반이 "에반스 워크플로우"라는 이름으로 자주 쓰는 스킬들을 소개하는 자리라고 시작한다. 자신은 총 4가지 정도의 스킬을 콤비네이션해서 쓰고 있다며, 하나씩 설명하겠다고 예고한다.

### [00:20–00:42] 목차와 "왜 조합하나"의 취지
목차는 왜 여러 스킬을 조합해야 하나 → 슈퍼파워즈 → 매포(매트 포콕)의 스킬 → G스택 → 코덱스 플러그인 CC → 통합 워크플로우 → 마무리로 구성된다. 첫 주제부터, 화자는 각 스킬의 **역할이 다르기에** 이 네 개의 조합이 매우 좋다고 생각한다고 밝힌다. [의견]

### [00:42–02:00] AI 코딩의 한계와 4가지 실패 패턴
AI 코딩을 할 때 흔히 "알아서 해 줘"라고 요청하는데, 그 때문에 한계점이 명확하다고 지적한다. AI 코딩 에이전트는 똑똑하지만 시니어 엔지니어처럼 일하지는 않으며, 그래서 생기는 문제가 많다는 것.
그는 4가지 실패 패턴을 든다. 첫째, **요구사항을 이해하기 전에 코딩** — 스펙이 흐릿한 상태에서 코드부터 쓴다. 둘째, **같은 맥락을 반복 설명** — 세션마다 도메인 용어를 다시 설명하게 되고, 그때마다 설명이 달라지기도 한다. 셋째, **동작 안 하는 코드 생성** — AI가 거짓말로 "완성했다"고 말하는 케이스. 넷째, **AI 슬롭 양산** — 빠른 속도로 짜다 보면, AI가 "평균에 회귀"하는 특성 때문에 뾰족한 한 점으로 나아가지 못하고 특정 구간을 맴돌아 슬롭이 계속 생긴다. [의견]

### [02:00–02:41] 스킬이란 무엇인가 + 4개 스킬팩 소개
그래서 "스킬"을 써야 한다며, 스킬은 클로드가 작업 전에 반드시 읽고 따르는 매뉴얼이라고 정의한다. 포맷은 SKILL.md와 MD 스크립트로 구성되고 호출은 슬래시 커맨드로 한다. 비유하면 회사의 매뉴얼이며, 모델의 재학습 없이 **행동만** 변경하는 것이라고 설명한다. 이 스킬을 기반으로 도메인 지식을 주입해 알아서 일하게 만든다는 것. 이어 자신이 쓰는 4가지 스킬팩을 소개한다: G스택, 슈퍼파워즈, 매포(매트 포콕) 스킬, 코덱스.

### [02:41–03:31] 각 스킬의 역할과 "한 픽으로는 부족한 이유"
각각의 역할을 정리한다. **G스택 = 기획**("왜 만드는가"에 집중), **슈퍼파워즈 = 실행**(효율적 개발 워크플로우), **매포의 스킬 = 유지보수를 위한 코드**, 그리고 **코덱스**로 다른 모델·다른 시각에서 검토하는 작업을 거친다. 스킬 하나만으로 부족한 이유는 명료하다 — G스택이 없으면 기획이, 슈퍼파워즈가 없으면 실행이, 매포 스킬이 없으면 유지보수가, 코덱스 플러그인 CC가 없으면 검증이 각각 부재하기 때문이다.

### [03:31–04:08] 각 스킬이 "검증"하는 것
하나씩 뜯어 보면, G스택 스킬은 "정말 꼭 필요한 기능인가"를 기획적으로 다양한 시각에서 충분히 보았는지 검증한다. 슈퍼파워즈는 컨텍스트가 터지지 않고 효율적으로 개발했는지를 알려 준다. 매포의 스킬은 코드를 유지보수 가능한 코드로 바꿔 주며, 이를 도메인 지식과 매핑함으로써 한다. 코덱스 플러그인은 다른 시각으로 리뷰했는지를 본다.

### [04:08–05:00] 슈퍼파워즈 (1) — 브레인스토밍
맨 처음 볼 것은 슈퍼파워즈다. 슈퍼파워즈는 "시니어 엔지니어의 작업 습관을 마크다운으로 정리한 것"으로 스타 수가 매우 많고, 그중 화자는 세 가지만 쓴다. 첫째는 **브레인스토밍**이다. 이 스킬은 맥락(프로젝트 파일과 최근 커밋)을 탐색하고, 한 번에 질문 하나만 하며, 목적·제약·성공 기준을 수립한다. 두세 가지 접근법을 제시하고 트레이드오프를 준 뒤 섹션별로 승인 후에만 다음으로 진행한다. spec.md에 저장하고 커밋하며, Writing Plan을 자동 호출해 플랜을 작성한다. 핵심은 **설계 승인 전까지 어떤 코드도 작성하지 않는다**는 점이라고 강조한다.

### [05:00–05:16] 슈퍼파워즈 (2) — 라이팅 플랜즈
다음은 **라이팅 플랜즈** 스킬이다. 브레인스토밍 결과로 만들어진 스펙을 기반으로 플랜을 작성한다. 즉 브레인스토밍에서 스펙이 확정되면 이 스킬이 작동하는 구조다.

### [05:16–05:58] 슈퍼파워즈 (3) — 서브에이전트 드리븐 디벨롭먼트
세 번째는 **서브에이전트 드리븐 디벨롭먼트**로, 세 가지 코스로 나뉜다: 임플리멘터, 스펙 컴플라이언스, 코드 퀄리티. 임플리멘터는 구현·테스트·셀프 리뷰·커밋을 하고, 스펙 컴플라이언스는 요구사항 대비 검증·검수를 하며, 코드 퀄리티는 코드 품질 검토를 하되 스펙 통과 후에만 시작한다는 룰을 갖는다. 이 스킬을 쓰면 작은 단위 구현을 서브에이전트 기반으로 아주 작은 단위씩 새로운 컨텍스트에서 하게 되며, 테스트가 꼼꼼히 붙어 있어 도움이 많이 된다고 평한다. [의견]

### [05:58–07:11] 매포의 스킬 — 도메인 언어와 모듈 깊이 / context.md·ADR
다음은 매포의 스킬이다. "도메인 언어와 모듈 깊이"라고 적혀 있으며 코드의 유지보수성과 관련된다. 철학은 "에이전트는 코딩만 빠르게 하지 않는다, 코드 부패도 빠르게 만든다"이다. 여기서 쓰는 스킬은 두 가지. 첫째는 **그릴 위드 독스**로, `context.md`와 `ADR` 파일을 참조한다.
`context.md`는 **용어의 의미를 적는 도메인 언어 사전**으로, 구현은 쓰지 않고 도메인 언어를 정의한다. 모든 스킬이 작업 시작 시 자동으로 읽고, 새 결정이 나오면 대화 도중 바로 갱신하므로 매 세션 맥락을 다시 설명할 필요가 없다.
`ADR`은 **결정 기록**으로, 되돌리기 어려운 결정만 기록한다. 맥락 없이 보면 의아한 선택을 설명하고, 대안과 트레이드오프를 함께 적으며, 이미 끝난 논의는 다시 꺼내지 않는다. 유저가 정한 특정 결정 사항을 ADR로 분리해 적는 셈이다.

### [07:11–08:34] 그릴 위드 독스의 동작과 효과
그릴 위드 독스의 동작을 본다: context.md·ADR을 로드해 기존 도메인 용어를 먼저 읽고, 새 용어 발견 시 정의를 붙이며, 한 번에 질문 하나만 하고, 추측 대신 코드베이스를 본다. 추천 답변과 근거를 함께 제시하고, 이미 끝난 결정(ADR에 기록된 선택)은 재논의를 차단하며 새 결정만 챌린지한다. 대화 중 새 용어가 나오면 그 자리에서 context.md에 추가한다.
역할은 유지보수 가능한 코드를 위해 특정 용어를 계속 유지하도록 **용어를 통일**하고 그것을 AI에게 계속 주입하는 것. 이를 기반으로 같은 용어를 쓰는 AI가 확립되어, 용어가 달라져서 생기는 이격이 없어진다. 화자가 가장 많이 느낀 것은 **결제 쪽 이슈**인데, context나 ADR을 잘 정의해 두면 자신이 잘못 지시해도 AI가 그것을 읽고 반박하는 신기한 현상을 볼 수 있다고 한다. [의견]

### [08:34–09:38] 매포 (2) — 임프로브 코드베이스 아키텍처
다음은 **임프로브 코드베이스 아키텍처** 스킬이다. 코드베이스의 얕은(shallow) 모듈을 **깊은 모듈**로 만드는 것으로, 깊은 모듈이란 재사용 가능하고 테스트 가능한, 즉 유지보수가 쉬운 코드를 말한다. 동작은: context.md를 먼저 로드하고 ADR을 로드한 뒤, 모듈의 경계와 마찰 지점(각 모듈·인터페이스의 경계)을 탐색한다. 그다음 **삭제 테스트**(지워 봤을 때 복잡도가 줄어드는지 늘어나는지)를 하고, 비포/애프터 보고서를 생성해 추천한다. 사용자가 선택한 후 검증하는 과정을 거친다. 이를 통해 재사용 가능하고 유지보수가 쉬운 코드로 바꿔 준다.

### [09:38–10:41] G스택 — 오피스 아워즈의 6가지 강제 질문
다음 스킬은 G스택으로, 게리 탄(Y Combinator 사장)이 만든 것이다. 철학은 "한 명의 빌더가 올바른 도구를 가지면 팀보다 빠르게 움직인다"로, 그의 철학이 담긴 스킬이라 본다. [의견] 화자는 **오피스 아워즈** 스킬을 많이 쓰는데, 이는 YC식 6가지 강제 질문을 하고 코드 작성은 없다. (1) 디맨드 리얼리티 — 지금까지 구체적 수요 증가의 증거는 무엇인가? (2) 스테이터스 코 — 지금 사용자들의 임시방편은 무엇인가? (3) 디스퍼릿 스펙 — 지금 이 도구가 가장 필요한 한 명은 누구인가? (4) 돈 낼 가치가 있는 최소는 무엇인가? (5) 서프라이즈 — 실제 관찰해서 발견한 것은 무엇인가? (6) 퓨처 — 세상이 변할수록 더 필수가 되는 도구인가? 꽤 집요하게 물어보니 체험을 추천한다.

### [10:41–11:01] G스택 — 브라우즈
다음은 **브라우즈** 기능으로, 헤드리스 크로미움을 처리해 주는 친구다. 이걸 쓰면 일반적인 Playwright를 쓰는 것보다 똑똑하게 동작하는 것 같아 화자도 애용한다. [의견]

### [11:01–12:22] 코덱스 플러그인 CC — 3가지 모드와 사용 습관
다음은 코덱스 플러그인 CC다. OpenAI가 공식으로 낸 것으로, GPT-5 모델을 이용해 앤트로픽 클로드가 짠 코드를 리뷰해 준다. [불명확: 자막 "GPTU 5"] 세 가지 모드가 있다. **코덱스 리뷰**는 변경 사항을 객관적으로 평가하며 클로드 리뷰의 보완점으로 쓰인다. **코덱스 애드버서리얼**은 결함이 있다고 가정하고 잡는 기능이다. **코덱스 레스큐**는 클로드가 막힌 작업을 코덱스에 위임하고 그 결과를 비교하는 기능이다.
화자는 보통 코덱스 리뷰를 많이 쓴다. 다만 코덱스 레스큐는 버그가 있어 가끔 행(hang)이 걸리는 것 같아 **안 쓰기를 추천**한다. 모드별 사용: PR 마무리 전 일반 검토=코덱스 리뷰, 보안·결제 등 위험 코드=코덱스 애드버서리얼, 버그 3회 이상 실패=코덱스 레스큐, 릴리스 직전 최종 점검=코덱스 애드버서리얼, 새 라이브러리 적용=코덱스 리뷰. 그러면서도 코덱스 레스큐는 버그 때문에 좀 애매하다고 재차 말한다. [의견]

### [12:22–12:58] 통합 워크플로우
통합 워크플로우를 설명한다. 먼저 기획은 office hours와 그릴 위드(docs)로 한다. 그다음 스펙 작성은 브레인스토밍과 writing plan으로 한다. 이어 서브에이전트 드리븐 디벨롭먼트로 개발하고, 코덱스와 브라우저로 검증한다. 마지막으로 코드베이스 유지는 임프로브 코드베이스 아키텍처로 유지보수한다.

### [12:58–13:43] 기억할 5가지
기억할 5가지를 정리한다. (1) 모델보다 작업 방식이 중요하다 — 같은 모델이라도 방법이 결과를 결정한다. (2) 단계마다 역할을 분리해야 한다 — 기획·개발·리뷰를 한 페르소나로 섞지 않는다. (3) 기획 게이트를 먼저 통과해야 한다 — 수요와 도메인 언어 정리 없이 코드를 쓰지 않는다. (4) 외부 모델로 사각지대를 점검해야 한다 — 클로드 셀프 리뷰만으로는 같은 패턴을 못 본다. (5) 실제 화면을 확인해 QA를 해야 한다 — 테스트 통과는 사용자 작동을 보장하지 않으므로 셀프 QA가 필수다. 이렇게 이론 수업은 마무리하고 실습으로 넘어간다고 예고한다.

### [13:43–14:45] 실습 (1) — 기획 파트
실습은 실제 프로덕트를 만들기엔 너무 길어 문서로 정리해 두었다며 하나씩 설명한다. 먼저 기획 파트: G스택에 오피스 아워를 켜고 "나는 지금 프롬프트 ○○를 하려 하고, ○○에 대해 처리해 줘"라고 입력한다. 그러면 G스택이 계속 물어보므로 답변한다. 다음으로 슈퍼파워즈 브레인스토밍을 켜서 "이때까지 한 내용을 기반으로 처리해 줘"라고 하면 스펙을 작성해 준다. 그다음 매포의 gird-with-docs 스킬을 쓰면 스펙에 대해 도메인 지식·ADR 등을 엄청 물어보므로 맞게 처리한다. 마지막으로 라이팅 플랜 스킬을 트리거하면 플랜을 작성한다. 스펙 문서가 나오면 문서로 저장한다.

### [14:45–15:14] 실습 (2) — 개발 파트
그다음에는 다른 세션을 켜거나 클리어 버튼을 누르기를 추천한다. 플랜 문서 이후 "슈퍼파워즈 플랜스 이어서 작성해 놓고, 서브에이전트 드리븐 디벨롭먼트로 개발하고, 코덱스로 리뷰해 줘"라고 트리거하면 처리가 시작된다. 이런 식으로 하면 한두세 시간 정도 돌면서 알아서 처리된다.

### [15:14–16:11] 실습 (3) — 유지보수 세션과 마무리
유지보수 세션은 여러 세션을 병렬로 돌린다. 한 세션에서는 임프로브 코드베이스 아키텍처로 코드베이스 리팩토링을, 또 다른 세션에서는 버셀 React 베스트 프랙티스와 Next.js 프론트엔드 가이드라인 스킬로 리팩토링을, 마지막으로 패스트API 백엔드 가이드라인으로 리팩토링을 따로따로 명령한다. 이렇게 코드베이스를 계속 리팩토링해 이상한 코드를 제거하는 작업을 이어 간다. 이 작업들은 코드 퀄리티·유지보수 퀄리티를 위해 필수이며, 토큰을 많이 쓰지만 안정적 운영을 위해선 필수적이라고 본다. [의견] 마지막으로 이 메타프롬프트를 링크로 공유할 테니 보고 실습해 보길 추천하며, 해 보면 훨씬 달라진 것을 체감할 수 있을 것이라고 마무리한다. [의견]
