# 바이브 코딩 결국 무너집니다

> maker-evan · 2026-05-21 · [영상 링크](https://youtu.be/sSIkZ4-gSrc)
> 정리 방식: 시간순 심층 워크스루 (paraphrase) + 주제별 상세 요약
> 검증 배너: 후반부는 자사 유료 코드 베이스(스킬샵) 홍보. 개발 속도·안정성 관련 서술은 화자의 경험 주장이며 미검증. 할인·이벤트는 세일즈 장치.

## 요약 (주제별)

### 핵심 주장 — 코드 베이스가 속도를 결정한다
- 기획부터 동작 화면까지 빠르게 간 비결은 "잘난 실력"이 아니라 잘 깔린 코드 베이스라는 게 영상의 전제. [의견]
- 잘 짜인 코드 베이스는 처음엔 느려 보여도 길게 보면 훨씬 빠르고 안정적, 감으로만 짜는 바이브 코딩은 처음만 빠르고 시간이 지나면 더 느려지고 무너진다는 것이 결론. (주장/미검증)
- 붕괴는 한 번에 오지 않고 "천천히" 오기 때문에 더 위험하다 — "아직 괜찮네" 하다가 어느 날 손 쓸 수 없게 된다. [의견]

### 방법론 1 — DDD(도메인 주도 설계)로 영역 분리
- 코드를 결제 영역·회원 영역·주문 영역처럼 비즈니스 단위로 나눠 놓는 것. 회사의 부서(인사팀·재무팀 등)가 자기 일에만 집중하고 필요할 때만 협업하는 것에 비유.
- 결제 로직은 결제 도메인, 회원 로직은 회원 도메인에 두면 코드가 깔끔해지고, 새 개발자도 "결제 문제면 결제 폴더" 하고 바로 찾아간다.
- 효과: 새 기능 추가 시 붙일 자리가 명확, 영역이 분리돼 버그가 옆 영역으로 안 번짐(결제 코드 고쳐도 회원 가입이 안 망가짐). (주장/미검증)

### 방법론 2 — 기술 스택 분리 (Next.js + Python)
- 프론트엔드는 Next.js, 백엔드는 Python. "각자 잘하는 걸 시키려고" 나눴다는 논리.
- Next.js: 화면 그리기·사용자 인터랙션에 강함. SSR, 자동 라우팅 등 요즘 앱에 필요한 게 거의 다 들어 있음.
- Python: 데이터 처리·머신러닝 모델 구동·외부 API 통신에 편함. 라이브러리 많고 문법 직관적. [의견] AI가 코드 짤 때 코드량이 적어 더 효과적이라는 주장도 덧붙임. (주장/미검증)
- 분리 이점: 프론트 담당은 백엔드 신경 안 쓰고, 백엔드 담당은 화면 디자인 신경 안 씀 → 각자 빠르게 이동.
- 단, 화자 본인은 프론트·백엔드를 한 곳에 모은 모노레포를 쓴다고 밝힘(스택 분리와 레포 구조는 별개).

### 실전 예시 — "결제 시 알림" 기능 추가
- 잘 깔린 베이스에서는: 결제 도메인에서 결제 완료 이벤트 발생 → 알림 도메인이 그 이벤트 받아 처리 → 끝. 다른 영역 건드릴 일 거의 없음.
- 빠른 이유: 결제·알림 도메인이 각자 이미 잘 정리돼 있어 그 위에 기능만 얹으면 됨. 레고 블록(기반 블록 위에 새 모양 만들기)에 비유.

### 리스크 — 계획 없는 바이브 코딩의 붕괴 경로
- 화자는 "아무 베이스 없이 맨땅에서 짜는" 것을 계획 없는 바이브 코딩이라 부르며 이를 통칭으로 사용.
- 초반 ~일주일은 설계·구조 없이 떠오른 대로 짜니 결과물이 척척 나옴. "왜 어렵다는 거지" 싶을 정도.
- 한 달 뒤: 결제 로직이 화면 코드 안에 들어가고 회원이 주문 코드에서 직접 수정되는 등 코드가 엉킴 = "부서 없는 회사". 한 사람 일 바꾸면 옆 사람 일까지 망가짐.
- 집 비유: 도면 없이 지으면 처음은 빠르지만 2층 올릴 때 1층 벽이 무게를 못 버텨 다 부수고 새로 지어야 함 → "처음부터 다시 짜자"가 나옴.
- 디버깅 지옥: 버그 원인 추적 불가. 결제 화면 오류인데 원인은 회원 쪽, 그 회원은 또 주문 코드에서 건드림 — 실타래처럼 엉켜 한 줄 풀면 또 다른 줄이 엉킴.

### 두 방식 5개 항목 비교
- 초반 속도: 바이브 코딩이 빠름. 잘 짜인 베이스는 처음 세팅에 시간이 듦(인정할 부분).
- 한 달 뒤 속도: 역전. 잘 짜인 베이스는 기능 추가가 점점 쉬워지고, 바이브 코딩은 점점 어려워짐.
- 버그: 잘 짜인 베이스는 영향 범위가 갇힘(결제 버그는 결제 안에서만), 바이브 코딩은 사방으로 퍼짐(화면 버그인 줄 알았는데 DB까지 손대야 함).
- 신규 인원 합류: 잘 짜인 베이스는 폴더 구조만 봐도 일주일이면 시작, 바이브 코딩은 한 달 지나도 헤매고 물어볼 사람이 만든 본인밖에 없음.
- 결과물 퀄리티(가장 중요): 기초 부실이면 결국 퀄리티 낮음. 화면은 예뻐도 써 보면 느려지고 오류 나고 이상 동작 → 사용자가 다 느낌. (주장/미검증)

### 스킬(skill) 기반 개발 언급
- DDD·Next.js·Python 위에, 그동안의 템플릿을 전부 포매팅·스킬화해서 그 스킬 기반으로 코드를 짜면 코드가 매우 안정적으로 짜인다고 주장. (주장/미검증)

### 세일즈 구조
- 오프닝: "초창기 오픈 기념"으로 끝까지 본 사람에게 주는 이벤트가 마지막에 있다고 예고(끝까지 시청 유도).
- 본론에서 설명한 코드 베이스(DDD + Next.js 스킬 + Python 스킬)를 직접 다듬어 스킬샵에 올려뒀고, 설명란 링크로 안내. 빈 폴더부터 구조·도메인·프론트/백 연결을 다 하려면 시간이 많이 드는데 그걸 정리해 뒀다는 소구.
- 이벤트: 해당 코드 베이스를 50% 할인가에 제공 예정. (주장/미검증)
- 마무리 소구: 코드를 전혀 모르는 사람이라도 이 코드 베이스를 기반으로 바이브 코딩하면 더 유지보수 가능한(well-architected) 코드를 만들 수 있다는 점을 강조. [의견]

## 3줄 오리엔테이션
- "잘 짜인 코드 베이스 vs 계획 없는 바이브 코딩"을 초반 속도·장기 속도·버그·인원 합류·퀄리티 5축으로 비교하는 영상.
- 처방은 DDD로 도메인 나누고, 프론트 Next.js / 백엔드 Python으로 역할 분리, 그 위에 스킬(템플릿)을 얹는 것.
- 결론적으로 화자 자신이 만든 코드 베이스를 스킬샵에서 50% 할인 판매하는 홍보 영상.

## 전문 — 시간순 워크스루

### [00:00–00:27] 오프닝 — 빠른 개발의 비결은 실력이 아니라 코드 베이스
새 프로젝트를 기획받고 동작하는 화면까지 가는 데 생각보다 빠르게 끝났다며, 그 비결이 "저희가 잘나서"가 아니라 "코드 베이스가 잘 깔려 있어서"라고 못박는다. 오늘은 잘 짜인 코드 베이스가 왜 중요한지, 기반 없이 감으로만 짜는 코딩이 왜 위험한지 정리하겠다고 예고한다. 또한 초창기 오픈 기념으로 끝까지 본 사람에게 주는 작은 이벤트가 마지막에 있으니 끝까지 봐 달라고 유도한다.

### [00:28–01:07] 집 비유 — 도면 있는 집 vs 막 지은 집
새 집 이사 비유로 시작한다. 도면이 잘 나온 집은 콘센트 위치가 적당하고 화장실 동선도 자연스럽고 가구 들이기도 편한데, 도면 없이 막 지은 집은 콘센트가 침대 뒤에 숨고 화장실 가려면 거실을 가로질러야 한다. 코드도 똑같아서, 코드 베이스가 잘 세팅돼 있으면 새 기능 하나 추가할 때 "여기 끼우면 되겠다" 하는 자리가 명확하지만, 엉망이면 "이걸 어디다 넣지" 하는 고민이 매번 붙는다. 그 고민이 쌓이면 개발 속도가 느려지고 버그도 는다.

### [01:07–01:21] 자기 경험 — 몇 달 뒤 기능 하나에 일주일
화자 팀도 처음엔 빠르게 만드는 데만 집중했는데, 몇 개월 지나니 작은 기능 하나 추가에도 일주일씩 걸렸다고 한다. 이유는 "어디를 건드리면 어디가 터질지 몰라서". 그때부터 코드 베이스 자체를 다시 짜기 시작했다.

### [01:22–02:04] 처방 1 — DDD(도메인 주도 설계)
가장 먼저 적용한 것이 DDD라고 소개한다.
> "도메인 주도 설계라고 하는데 말이 참 어렵죠." [01:26]
쉽게 말해 코드를 결제 영역·회원 영역·주문 영역처럼 비즈니스 단위로 나눠 놓는 것이라 설명한다. 큰 회사의 부서(인사팀·재무팀 등)가 자기 일에만 집중하고 필요할 때만 협업하는 것에 비유하며, 한 사무실에 모든 직원이 섞여 있으면 책임 소재가 흐릿하고 혼란스럽다고 대비시킨다. 결제 로직은 결제 도메인, 회원 로직은 회원 도메인 안에 두면 코드가 깔끔해지고, 새로 들어온 개발자도 "결제 문제면 결제 폴더만 보면 되겠구나" 하고 바로 찾아갈 수 있다. [의견]

### [02:04–02:49] 처방 2 — 기술 스택: Next.js(프론트) + Python(백엔드)
다음은 기술 스택. 프론트엔드는 Next.js, 백엔드는 Python으로 가져갔고, 이유는 "각자 잘하는 걸 시키려고"다. Next.js는 화면 그리기에 강해 사용자가 클릭·입력·결과 보는 부분을 담당하고, SSR도 되고 라우팅도 자동이고 요즘 앱에 필요한 게 거의 다 들어 있다고 설명한다. Python은 반대편에서 데이터 처리·머신러닝 모델 구동·외부 API 통신에 편하고, 라이브러리가 많고 데이터 다루는 문법이 직관적이라고 한다. 생산성도 효과적이라며, 다른 언어에 비해 읽고 쓰기 편하고 AI가 코드를 짤 때 코드량이 적어 더 효과적으로 작업할 수 있다고 덧붙인다. (주장/미검증)

### [02:49–02:59] 스택 분리의 이점
둘로 나눠 놓으면 프론트 작업자는 백엔드를 신경 안 써도 되고 백엔드 작업자는 화면 디자인을 신경 안 써도 돼서, 각자 자기 영역에서 빠르게 움직일 수 있다고 정리한다.

### [02:59–03:07] 단서 — 본인은 모노레포 사용
다만 화자 본인의 경우엔 프론트엔드와 백엔드를 한 곳에 모아놓은 모노레포(monorepo) 형태의 레포지터리를 쓰고 있다고 밝힌다. (스택을 나누는 것과 레포를 하나로 두는 것은 별개라는 취지.)

### [03:07–03:40] 실전 예시 — "결제 시 알림" 기능 추가와 레고 비유
DDD + Next.js/Python 조합을 깔면 새 프로그램 만드는 속도가 진짜 달라진다며 예시를 든다. "회원이 결제하면 알림 보내는 기능"을 추가한다고 치면, 잘 깔린 베이스에서는 결제 도메인에서 결제 완료 시 이벤트 하나를 발생시키고, 알림 도메인이 그 이벤트를 받아 처리하면 끝이다. 다른 영역 건드릴 일이 거의 없다. 빠른 이유는 결제·알림 도메인이 각자 이미 잘 정리돼 있어 그 위에 기능만 살짝 얹으면 되기 때문. 레고 블록처럼 기반 블록이 잘 깔려 있으면 그 위에 새 모양 만드는 건 금방이라는 비유다.

### [03:40–03:50] 버그 격리
버그도 적다고 한다. 영역이 분리돼 있으니 결제 코드를 고쳤다고 회원 가입이 망가지는 일이 거의 없다 — 영역끼리 영향을 안 주기 때문. (주장/미검증)

### [03:50–04:26] 전환 — 바이브 코딩 정의
여기까지가 잘 짜인 코드 베이스의 장점이고, 반대로 기반 없이 감으로 코딩하면 어떻게 되는지로 넘어간다. 요즘 말하는 "바이브 코딩"은 분위기대로·느낌대로 코딩하는 것이라 소개하며, 화자는 아무 베이스 없이 맨땅에서 짜는 것을 "계획 없는 바이브 코딩"이라 부르고 이 영상에선 이를 바이브 코딩으로 통칭하겠다고 한다. (이 구간은 자막상 같은 설명이 반복 재생된다.)

### [04:26–04:52] 한 달 뒤의 붕괴 — 부서 없는 회사
계획 없는 바이브 코딩은 처음엔 진짜 빠르다. 설계도 구조도 안 잡고 떠오른 대로 막 짜니 한 일주일은 결과물이 척척 나와서 "왜들 어렵다고 하지" 싶을 정도다. 문제는 한 달 뒤. 코드가 점점 엉키기 시작해 결제 로직이 화면 코드 안에 들어가고 회원이 주문 코드에서 직접 수정되는 식이 된다. 화자는 이를 "부서가 없는 회사"에 비유한다 — 누가 무슨 일 하는지 아무도 모르고, 한 사람 일을 바꾸면 옆 사람 일까지 다 망가진다.

### [04:52–05:10] 집 비유 재등장 — 2층에서 무너진다
도면 없이 집을 지으면 처음엔 벽 세우고 지붕 얹고 빠르지만, 2층을 올리려고 보면 1층 벽이 그 무게를 못 버텨서 결국 다 부수고 새로 지어야 한다. 코드도 기초가 부실한 채로 기능을 계속 쌓으면 어느 순간 무너지고, 그 시점에 "처음부터 다시 짜자"는 말이 나온다.

### [05:10–05:27] 디버깅 지옥
또 하나는 디버깅. 바이브 코딩으로 만든 프로그램에서 버그가 나면 어디서 시작됐는지 추적이 안 된다. 결제 화면에서 오류가 났는데 원인은 회원 쪽에 있고, 그 회원은 또 주문 코드에서 건드려서, 실타래처럼 엉켜 한 줄 풀면 또 다른 줄이 엉킨다.

### [05:27–06:27] 두 방식 5개 항목 비교
두 상황을 깔끔하게 비교한다.
- 첫째, 초반 속도: 바이브 코딩이 훨씬 빠르다. 잘 짜인 코드 베이스는 처음 짤 때 시간이 든다("인정할 부분은 인정").
- 둘째, 한 달 뒤 속도: 여기서 뒤집힌다. 잘 짜인 베이스는 기능 추가가 점점 쉬워지고, 바이브 코딩은 코드가 엉켜 어디를 건드릴지 몰라 점점 어려워진다.
- 셋째, 버그: 잘 짜인 베이스는 버그 영향이 갇혀서 결제 버그는 결제 안에서만 영향을 주지만, 바이브 코딩은 버그 하나가 사방으로 퍼져 화면 버그인 줄 알았는데 데이터베이스까지 손대야 하는 일이 흔하다.
- 넷째, 신규 인원 합류: 잘 짜인 베이스는 폴더 구조만 봐도 어디서 뭘 하는지 보여 새 개발자가 일주일이면 시작하지만, 바이브 코딩은 한 달 지나도 헤매고 "이 변수는 왜 여기 있어요?" 물어볼 사람이 만든 본인밖에 없다.
- 다섯째, 결과물 퀄리티(가장 중요): 기초가 부실하면 결국 퀄리티 낮은 결과물이 나온다. 화면은 예뻐 보여도 잠깐 써 보면 느려지고 오류 나고 이상한 동작이 보이며, 사용자는 그걸 다 느낀다. (주장/미검증)

### [06:27–06:54] 정리 — 잘 짜인 베이스 + 스킬 기반
오늘 정리하면, 잘 짜인 코드 베이스는 처음엔 좀 느려 보여도 길게 보면 훨씬 빠르고 안정적이다. DDD로 도메인 나누고 프론트는 Next.js, 백엔드는 Python처럼 각자 잘하는 도구 위에 쓰면 만들 수 있는 게 굉장히 많아진다. 또한 그동안의 템플릿들을 다 포매팅·스킬화해서 그 스킬 기반으로 코드를 짜면 코드가 매우 안정적으로 잘 짜인다고 덧붙인다. (주장/미검증)

### [06:54–07:16] 반대편 — 천천히 무너져서 더 위험
반대로 기반 없이 감으로만 가는 바이브 코딩은 처음만 빠르고 시간이 지나면 더 느려지고 더 망가진다.
> "한 번에 무너지는 게 아니라 천천히 무너집니다. 그래서 더 위험해요. 아직 괜찮네 하다가 어느 날 손 쓸 수 없게 되거든요." [07:00]
지금 새 프로젝트를 시작하거나 기존 프로젝트를 손보고 있다면, "이 코드 베이스가 1년 뒤에도 빠르게 움직일 수 있는 구조인가"를 한 번쯤 자문해 보라고 권한다. [의견]

### [07:16–07:52] 세일즈 — 스킬샵에 올린 코드 베이스
오늘 영상엔 또 다른 이유가 있다며 홍보로 전환한다. 지금까지 말한 코드 베이스 — DDD로 도메인 나누고 Next.js 스킬·Python 스킬로 깔아 놓은 그 기반 — 를 직접 다듬어 스킬샵에 올려뒀다고 한다. 새 프로젝트를 빈 폴더에서부터 구조 잡고 도메인 나누고 프론트/백엔드 연결까지 다 하려면 시간이 많이 드는데, 그 시간을 다 들여 정리해 둔 걸 그대로 쓸 수 있게 만들었다는 것. 영상 설명란에 링크를 걸어 둘 테니 둘러보고, 직접 코드를 받아 열어 보면 오늘 말한 구조가 실제로 어떻게 생겼는지 바로 확인할 수 있다고 안내한다.

### [07:52–08:24] 이벤트 — 50% 할인, 초보자 소구
약속한 이벤트로, 잘 짜인 코드 베이스를 필요로 하는 사람에게 50% 할인가에 제공할 예정이라고 밝힌다. (주장/미검증) 특히 코드를 전혀 모르는 사람이라도 이 코드 베이스를 기반으로 바이브 코딩하면 더 well-architected된, 즉 유지보수 가능한 코드를 만들 수 있다는 점을 명심하고 사용하라고 소구한다. [의견]

### [08:24–08:29] 마무리
