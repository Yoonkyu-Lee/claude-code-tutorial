# 개발자가 사라지고 있습니다

> maker-evan · 2026-02-23 · [영상 링크](https://youtu.be/fiVEMLZ0GLU)
> 정리 방식: 시간순 심층 워크스루 (paraphrase) + 주제별 상세 요약
> 검증 배너: 화자의 개인 경험담 + 퇴사·창업 서사. 소요 시간·대체 전망 등 수치는 화자 주장(미검증).

## 요약 (주제별)

### 배경 — 속도 압박 회사와 기술 부채
- 화자는 소프트웨어 엔지니어. 코딩 테스트 통과·취업·실력 축적을 거친 나름 잘한다고 자평한 개발자였다.
- 근무 회사가 특이했다: 해외에서 새 AI(GPT 3, Claude 3, Gemini 3 등)가 나오면 즉시 자사 서비스에 붙여 다음 날 출시하는 업무. AI 업계 속도(거의 매주 신규 출시)를 맞추려다 보니 코드 리뷰 없음, 설계 논의 시간 없음, "일단 돌아가게만" 마인드.
- [의견] 코드는 처음엔 레고 블록처럼 깔끔하지만, 쫓기며 막 던져 넣으면 나중에 어디에 뭐가 있는지 아무도 모르고 하나 고치면 다른 데가 터진다 → 기술 부채(지금 빠르게, 나중에 더 많은 시간으로 갚음). 나쁜 줄 알면서 1년을 버텼고 코드 퀄리티는 계속 떨어지고 버그는 계속 났다.

### Claude Code란 무엇인가 (vs ChatGPT)
- ChatGPT는 채팅: "이 코드 어떻게 짜요?" 물으면 답을 주고, 그걸 복사해 내가 직접 붙여야 함.
- Claude Code는 내 컴퓨터·프로젝트 안으로 직접 들어와 파일을 읽고 수정하고 새로 만듦. "자동화 프로그램 + AI"가 합쳐진 느낌.
- 화자도 처음엔 그냥 채팅처럼(물어보고 답 복붙) 썼다가 뭔가 다르다는 걸 느끼기 시작.

### 컨텍스트 관리 (핵심 1)
- Claude Code를 오래 쓰면 갑자기 멍청해지는 순간이 온다 — 잘하다가 엉뚱한 소리를 하고 아까 말을 잊어버림. AI에 기억 한계가 있어 대화가 길어지면 앞 얘기를 까먹음(컨텍스트가 터진다).
- 비유: 직원에게 메모장이 있는데 한 페이지가 다 차면 앞에 적은 게 지워짐. 해법 = 중요한 내용은 따로 파일로 저장해 필요할 때마다 보여주기.
- 화자는 프로젝트 구조·코드 스타일 규칙·주의사항을 파일에 정리해 주입. 대화가 새로 시작돼도 파일만 다시 넣으면 이어감(직원이 바뀌어도 인수인계 문서가 있으면 되는 것처럼).

### 스킬 (핵심 2, 레시피 비유)
- Claude Code에 일을 시키다 보면 "이 상황엔 이렇게" 반복 설명하는 패턴이 보임. 이걸 파일로 만들어 두면 매번 설명 없이 "이 파일 보고 해라"로 끝 → 이게 스킬.
- 비유: 요리사에게 매번 소금·설탕 양을 설명하지 않고 레시피 카드 하나 주는 것.
- 활용: 화자가 직접 컨벤션·구조가 완벽한 레퍼런스 코드 하나를 잘 짜서 스킬 파일에 담고 "이 코드처럼 짜라, 이게 기준이다"라고 주면 다른 코드들이 그 기준에 맞춰 고쳐짐.

### 테스트 먼저 + 레거시 리팩토링 사례
- 리팩토링 = 서비스 중인 코드의 동작은 그대로 유지하며 내부를 깔끔하게 정리하는 것. 비유: 사람이 살고 있는 집을 리모델링(가족이 그 안에서 밥 먹고 자는 동안 공사) — 잘못 건드리면 실제 서비스가 터짐. 보통 몇 주씩 잡고 함.
- 화자 방법: 먼저 테스트 코드를 깖(자동으로 제대로 도나 확인 — 버튼 누르면 기능 정상/오류를 알려줌). 리팩토링 후 버튼 한 번에 테스트 다 통과하면 버그 없음.
- 안전망(테스트) + 스킬 세팅 후 Claude Code에 맡김 → 프로덕션 프로젝트를 버그 하나 없이 반나절 만에 완료(주장). 이 과정에서 화자가 직접 짠 코드는 처음 레퍼런스 코드 딱 하나뿐, 나머지는 결과 확인하고 배포만.

### 모노레포 + SDD + 어드민 사례
- 다음 프로젝트: 사내 어드민(관리자 페이지 — 데이터 조회·설정 변경·통계 확인). 보통 백엔드(서버 쪽 데이터 처리) + 프론트엔드(화면 쪽)를 다 짜야 해서 시간이 은근히 걸림.
- 모노레포 선택 = 백엔드와 프론트엔드를 한 폴더에. Claude Code에 유리 — 백엔드 코드를 보면서 바로 프론트엔드를 짜고 "여기서 데이터 받으니 화면에 이렇게"를 스스로 판단, 일일이 설명 불필요.
- SDD(Spec Driven Development) = 코드 짜기 전 스펙을 문서로 먼저 정의("이 버튼 누르면 이런 결과"). 문서 먼저 쓰고 맞는지 확인 후 구현을 Claude에 맡김. 화자 역할은 문서 리뷰뿐.
- 결과: 어드민 구축 하루(주장), 이후 신규 기능 요청은 2~3시간 이내 처리(주장). 코드 돌려 놓고 폰(레딧·쓰레드·유튜브)을 봄.

### 개발자 역할 변화와 대체 전망 (주장)
- [의견] 지금 개발자는 코드를 덜 짜고, 대신 뭘 만들지 결정·구조 판단·결과 검사 역할로 바뀌는 중. "그럼 개발자가 할 게 없잖아요" — 맞고 그게 포인트.
- [의견/주장] 이 역할(스킬 짜기·구조 잡기·문서 리뷰)마저 AI가 좋아지면 계속 줄어듦. 화자 계산으로 지금 속도라면 자기 업무의 90%는 5년 안에 AI가 대체할 것 같고(주장, 미검증), 어쩌면 더 빠를 수도.

### 퇴사·창업 결정
- 회사에서 돈 받는데 실제 하는 일은 점점 줄어드는 상황이 "꿀이 아니라 공포"였고 잠이 안 옴.
- 퇴사가 겁 안 났던 건 아니지만(안정적 월급·경력) 생각: 이 실력을 왜 남의 회사 서비스 만드는 데 쓰나, 어드민을 하루에 만든 것처럼 내 서비스도 하루에 만들면 되지 않나.
- 개발자 창업의 제일 큰 장벽이 개발 시간(서비스 하나에 몇 달)이었는데 Claude Code로 며칠이면 됨 → 아이디어만 있으면 되고, 화자에겐 아이디어가 있었음. 그래서 퇴사·창업.
- 현재 남의 것이 아니라 자기 서비스를 만드는 중이고, 회사 다닐 때보다 오히려 더 빠름(눈치·보고·회의가 없어 아이디어→스펙→Claude Code 사이클이 빨리 돎, 기능 추가 2~3시간). 다만 창업이 쉬운 건 아니고 "개발이 느려 못 만든다"는 문제만 사라졌다고 선을 그음.

### 비개발자에게 주는 함의
- [의견] 이건 개발자만의 이야기가 아님. 디자인·글쓰기·마케팅·법률·회계 등 다른 분야에도 AI가 들어오고 있음.
- 화자가 살아남은 방식이 힌트: 코딩은 AI에 맡기고 자신은 뭘 만들지·어떤 구조·스펙·결과 판단(AI가 아직 잘 못하는 것)에 집중. "뭘 만들지는 사람이 결정." 각자 분야에서도 AI가 할 것은 넘기고 AI가 못 하는 판단·결정에 집중. 단 그러려면 AI를 먼저 잘 알아야 하고, 막연히 무서워하거나 무시하면 안 됨.

### 개발자용 핵심 3가지 교훈
- Claude Code를 아직 안 쓰면 빨리 시작하고, 단순히 빠르게 짜는 도구로만 보지 말 것.
- (1) 컨텍스트 관리가 전부 — 파일로, 스킬로 관리. 안 되면 Claude Code가 제 실력을 못 냄.
- (2) 테스트를 먼저 짜라 — 마음껏 맡기려면 안전망 필요. 테스트 없이 맡기면 버그 나도 모름.
- (3) SDD로 스펙을 먼저 잡아라 — 구현보다 설계가 더 중요한 시대, 스펙이 명확하면 구현은 Claude Code가 함.
- [의견] 잘 쓸 줄 알게 되면 선택지가 생김(회사에서 더 효율적으로 일하거나, 퇴사해 내 걸 만들거나). 선택지가 생기는 것 자체가 강점.

## 3줄 오리엔테이션
- Claude Code가 자기 코드를 다 짜주게 되자 "연봉은 그대로인데 내가 필요 없어진다"는 공포를 느낀 엔지니어의 퇴사·창업 고백.
- 컨텍스트 관리 / 스킬 / 테스트 먼저 / 모노레포 / SDD로 레거시 리팩토링(반나절)·어드민(하루)을 처리한 실전 경험을 비개발자도 알아들을 비유로 설명.
- 결론은 "AI가 할 수 있는 건 넘기고 AI가 못 하는 판단·결정에 집중하라" + 개발자용 핵심 3원칙(컨텍스트·테스트·SDD).

## 전문 — 시간순 워크스루

### [00:00–00:24] 고백 — 폰만 보는데 연봉은 그대로, 그게 공포였다
화자는 퇴사한 지 3개월도 안 됐다고 고백하며 시작한다. 회사 다닐 때 레딧·쓰레드·유튜브 등 폰만 봤는데 연봉은 그대로 나왔고, 그게 가능했던 이유는 Claude Code가 코드를 다 짜고 있었기 때문이라고 한다.
> "이게 꿀이 아니에요. 이게 공포였어요." [00:15]

"이러다 나 진짜 필요 없어지겠다"는 생각이 든 순간부터 잠이 안 왔고, 그래서 퇴사하고 창업했다며 오늘 그 이야기를 개발자에게도 개발 모르는 분에게도 중요한 이야기로 풀겠다고 예고한다.

### [00:24–00:44] 배경 — 나름 잘하던 소프트웨어 엔지니어
화자는 앱·웹사이트 안에서 작동하는 코드를 짜는 소프트웨어 엔지니어였다고 자기 직업을 비개발자용으로 설명한다. 코딩 테스트를 통과하고 회사에 들어가 실력을 쌓으며 나름 잘한다고 생각했지만, 근무하던 회사가 좀 특이한 데였다고 넘어간다.

### [00:44–01:05] 특이한 회사 — 신규 AI를 즉시 붙여 다음 날 출시
그 회사는 해외에서 새 AI가 출시되면 빠르게 자사 서비스에 붙여 출시하는 업무를 했다. GPT 3가 나오면 다음 날 연결해 출시, Claude 3·Gemini 3도 마찬가지. AI 업계가 거의 매주 새로운 게 나올 만큼 빠르게 도니 그 속도를 맞춰야 했다.

### [01:05–01:13] 속도의 대가 — 리뷰도 설계도 없이 "일단 돌아가게만"
속도를 맞추다 보니 코드 리뷰가 없고, "이 코드 괜찮아요?" 물어볼 시간도, 설계를 제대로 논의할 시간도 없었다. 일단 빠르게 돌아가게만 만들면 되는 상황이었다.

### [01:13–01:40] 기술 부채 — 레고 블록 던져 넣기 비유
비개발자용 비유로, 코드는 처음엔 레고 블록처럼 깔끔해 어디에 뭐가 있는지 보이고 고칠 수 있지만, 시간에 쫓겨 빠르게만 하면 레고 블록을 "일단 들어가기만 하면 돼"라며 막 던져 넣게 된다고 한다. 그러면 나중에 어디에 뭐가 있는지 아무도 모르고, 뭔가 건드리면 다른 데서 터지며 추적에만 하루가 걸린다.
> "그게 기술 부채라고 하는 건데요. 지금 빠르게 가는데 나중에 더 많은 시간을 갚는 거예요." [01:38]

화자도 이게 나쁜 방향인 줄 알았지만 시간이 없어 어쩔 수 없었다.

### [01:40–01:50] 1년을 버티다 Claude Code를 만남
그렇게 1년을 버텼고 코드 퀄리티는 계속 떨어지고 버그는 계속 났다. 그때 Claude Code를 만났다고 전환한다.

### [01:50–02:14] Claude Code란 — ChatGPT와 뭐가 다른가
ChatGPT는 대화창에 질문하면 답해 주는 것이고, Claude Code도 비슷하지만 하나가 다르다고 설명한다. ChatGPT는 채팅이라 "이 코드 어떻게 짜요?" 물으면 답을 주지만 그걸 복사해 내가 직접 붙여야 한다. 반면 Claude Code는 내 컴퓨터·프로젝트 안으로 직접 들어와 파일을 읽고 수정하고 새로 만든다. 자동화 프로그램과 AI가 합쳐진 느낌이라고 요약한다.

### [02:14–02:23] 처음엔 채팅처럼 쓰다 다름을 느낌
화자도 처음엔 그냥 채팅처럼(물어보고 답 보고 복붙) 썼는데, 어느 순간 뭔가 다르다는 걸 느끼기 시작했다며 "여기가 진짜 중요한 포인트"라고 강조한다.

### [02:23–02:53] 컨텍스트가 터진다 — 메모장 비유
Claude Code를 오래 쓰면 갑자기 멍청해지는 순간이 온다 — 잘하다가 엉뚱한 소리를 하고 아까 말한 걸 완전히 잊어버린 것처럼 군다. AI에 기억 한계가 있어 대화가 너무 길어지면 앞 얘기를 까먹기 시작하는데, 이걸 "컨텍스트가 터진다"고 한다. 비유로, 직원에게 메모장이 있는데 한 페이지가 다 차면 앞에 적어 둔 게 지워지는 것과 같다. 해법은 중요한 내용을 따로 파일로 저장해 필요할 때마다 보여주는 것, 그게 컨텍스트 관리다.

### [02:53–03:07] 파일로 컨텍스트 관리 — 인수인계 문서 비유
이 개념을 파악한 뒤 화자는 Claude Code를 완전히 다르게 쓰기 시작했다. 프로젝트 구조·코드 스타일 규칙·주의사항 같은 것들을 파일에 정리해 넣어 주면, 대화가 새로 시작돼도 파일만 다시 주입하면 바로 이어서 할 수 있다. 직원이 바뀌어도 인수인계 문서가 있으면 되는 것과 같다.

### [03:07–03:26] 스킬 — 레시피 카드 비유
Claude Code에 일을 시키다 보면 "이런 상황엔 이렇게"를 매번 말해 줘야 하는 패턴이 반복된다. 이걸 파일로 만들어 두면 매번 설명 안 해도 "이 파일 보고 해라"로 끝나는데, 그게 스킬이다. 요리사에게 매번 소금·설탕 양을 설명하지 않고 레시피 카드 하나 주면 되는 것에 비유한다. 화자는 이를 리팩토링 업무에 적극 활용했다 — 컨벤션도 완벽하고 구조도 깔끔한, 누가 봐도 이해할 레퍼런스 코드를 직접 하나 잘 짜서 그게 기준이 됐다.
> "이 코드처럼 짜라. 이게 기준이다. 그러면 다른 코드들도 이 기준에 맞춰 고쳐져요." [03:39]

### [03:44–04:00] 리팩토링이란 — 사람 사는 집 리모델링 비유
비개발자용으로, 리팩토링은 실제 서비스 중인 코드의 동작은 그대로 유지하면서 내부를 깔끔하게 정리하는 것이라 설명한다. 사람이 살고 있는 집을 리모델링하는 것과 같아서, 가족이 그 안에서 밥 먹고 자는 동안 공사를 해야 하고 잘못 건드리면 실제 서비스가 터진다. 그래서 굉장히 조심스러운 작업이고 보통 몇 주씩 잡고 한다.

### [04:00–04:27] 테스트 먼저 → 반나절 만에 리팩토링(주장)
화자의 방법은 테스트 코드를 먼저 짜는 것이었다. 테스트 코드는 자동으로 제대로 도는지 확인해 주는 코드로, 버튼을 누르면 이 기능 정상·저 기능 정상·이건 오류를 알려 준다. 리팩토링 전에 이 안전망을 먼저 깔면 리팩토링 후 버튼 한 번에 테스트가 다 통과하면 버그가 없는 것이다. 안전망을 깔고 스킬도 설정한 뒤 Claude Code에 맡겼더니 반나절 만에 끝났다(주장/미검증). 프로덕션에서 돌아가는 프로젝트를 버그 하나 없이 반나절 만에 처리했고, 그 과정에서 직접 짠 코드는 처음 레퍼런스 코드 딱 하나뿐이며 나머지는 결과 확인 후 배포만 했다.

### [04:27–04:50] 개발자 역할의 변화가 시작됨
"그럼 개발자가 할 게 없잖아요? 맞아요. 그게 포인트예요"라며, 지금 개발자가 코드를 덜 짜고 대신 뭘 만들지 결정하고 어떻게 구조를 잡을지 판단하고 결과를 검사하는 역할로 바뀌고 있다고 짚는다. 그리고 이 역할도 계속 줄어들면? 하는 생각을 안고 다음 프로젝트로 넘어간다.

### [04:50–05:34] 어드민 프로젝트 — 모노레포 선택
회사에서 내부용 어드민 시스템(관리자 페이지 — 데이터 조회·설정 변경·통계 확인)을 만들어야 했다. 어드민은 백엔드(서버 쪽 데이터 처리)와 프론트엔드(화면 쪽)를 다 짜야 해서 은근히 시간이 걸린다. 보통 이 둘이 분리돼 있는데 화자는 모노레포(한 폴더 안에 백엔드와 프론트엔드를 같이 두는 구조)를 선택했다. 이게 Claude Code에 엄청 유리한 이유는, 백엔드 코드를 보면서 바로 프론트엔드를 짤 수 있고 "여기서 데이터 받아오니 화면에 이렇게 보여주면 되겠다"를 스스로 판단하기 때문이다. 일일이 설명할 필요가 없다.

### [05:34–05:52] SDD — 스펙을 문서로 먼저
화자는 SDD(Spec Driven Development)도 썼다. 코드 짜기 전에 문서로 먼저 정의하는 방식으로, "이 기능은 이렇게 동작해야 한다, 이 버튼 누르면 이런 결과가 나와야 한다"를 문서로 먼저 쓰고 맞는지 확인한 뒤 Claude Code에 구현을 맡긴다. 화자가 할 일은 문서 리뷰뿐이고 구현은 Claude가 한다.

### [05:52–06:07] 어드민 하루, 기능 요청 2~3시간(주장) → 다시 폰
결과적으로 어드민 구축에 하루가 걸렸고, 이후 새 기능 요청이 들어오면 2~3시간 이내에 처리했다(주장/미검증). 그러고 나서 화자는 코드를 돌려 놓고 폰(레딧·쓰레드·유튜브)을 봤다. 코드가 알아서 돌아가고 자신은 폰을 보는 상황이 처음엔 편했지만, 이게 진짜 꿀이 아니라고 다시 못 박는다.

### [06:07–06:29] 공포의 정체 — 90% 5년 안에 대체(주장)
회사에서 돈을 받고 있는데 실제 하는 일은 점점 줄어들고, 남은 건 스킬 짜고 구조 잡고 문서 리뷰하는 정도인데 이 역할마저 AI가 계속 좋아지면 어떻게 되겠냐고 자문한다. 나름 계산해 보니 지금 속도라면 자기 업무의 90%는 5년 안에, 어쩌면 더 빨리 AI가 대체할 것 같았다(주장/미검증). 그냥 잘리는 걸 기다릴지 지금 움직일지 갈림길에 섰다.

### [06:29–06:55] 퇴사 결심 — 왜 남의 걸 만드나
화자는 지금 움직이는 걸 택했다. 퇴사가 겁이 안 났던 건 아니고 안정적 월급·경력을 버리는 게 쉬운 일은 아니었지만, 이런 생각이 들었다 — Claude Code를 잘 다루는 실력이 회사 서비스를 만드는 데 쓰이고 있는데 왜 남의 걸 만드는 데 쓰나, 내 걸 만들면 안 되나. 회사에서 어드민을 하루에 만든 것처럼 내 서비스도 하루에 만들면 안 되나. 개발자 창업의 제일 큰 장벽이 개발 시간(서비스 하나에 몇 달)이라 일하면서 창업하기 힘들었다고 짚는다.

### [06:55–07:14] 장벽이 사라짐 → 창업
지금은 Claude Code를 잘 쓰면 서비스 하나 만드는 데 며칠이면 되니 그 장벽이 사라졌고, 그러면 아이디어만 있으면 된다. 화자에겐 아이디어가 있었기에 퇴사했다. 지금은 남의 회사 게 아니라 자기 서비스를 Claude Code로 직접 만들고 있다.

### [07:14–07:33] 회사 때보다 더 빠름
재밌는 건 회사 다닐 때보다 오히려 더 빠르게 만들고 있다는 점이다. 회사에선 눈치 보고 보고하고 회의해야 했지만 지금은 그런 게 없다. 아이디어 생각하고 스펙 쓰고 Claude Code에 맡기는 사이클이 엄청 빨리 돌아 추가 기능이 생기면 2~3시간 안에 된다. 물론 다 잘되는 건 아니고 창업이 쉬운 건 아니지만, 최소한 개발이 느려서 못 만든다는 문제는 없어졌다.

### [07:33–07:51] 비개발자에게 — 다른 분야도 마찬가지
개발 모르는 분들에게 이 이야기가 왜 중요하냐면, 지금 일어나는 일이 개발자만의 이야기가 아니기 때문이다. Claude Code 같은 AI 도구가 개발 분야에서 하는 것처럼 디자인·글쓰기·마케팅·법률·회계 등 다른 분야에서도 비슷한 일이 일어나고 있다.

### [07:51–08:16] 살아남은 방식 — 판단·결정에 집중
그럼 어떻게 해야 하나. 화자가 Claude Code를 쓰며 살아남은 방식에 힌트가 있다. 코딩은 AI에 맡기고 대신 뭘 만들지 결정하는 것, 어떤 구조로 갈지·스펙은 어떻게 쓸지·결과가 맞는지 판단하는 것에 집중했다. 이게 AI가 아직 잘 못하는 것이며 뭘 만들지는 사람이 결정해야 한다.
> "AI가 할 수 있는 것들은 AI한테 넘기고 AI가 못 하는 판단과 결정에 집중하는 게 중요해요." [08:09]

단 그러려면 AI를 먼저 잘 알아야 하고, 막연히 무서워하거나 반대로 무시하면 안 된다.

### [08:16–08:44] 개발자용 핵심 3가지
개발자에게는 더 직접적으로, Claude Code를 아직 안 쓰고 있으면 빨리 시작해야 하고 단순히 빠르게 짜는 도구로만 보면 안 된다고 말한다. 배운 핵심 세 가지: 첫째, 컨텍스트 관리가 전부다 — 파일로, 스킬로 관리하라, 안 되면 Claude Code가 제 실력을 못 낸다. 둘째, 테스트를 먼저 짜라 — 마음껏 맡기려면 안전망이 있어야 하고, 테스트 없이 맡기면 버그 나도 모른다. 셋째, SDD로 스펙을 먼저 잡아라 — 구현보다 설계가 더 중요한 시대이고 스펙이 명확하면 구현은 Claude Code가 한다.

### [08:44–09:00] 선택지가 생기는 것 자체가 강점
하나 더, 이걸 잘 쓸 줄 알게 되면 선택지가 생긴다 — 회사에서 더 효율적으로 일할 수도 있고 화자처럼 퇴사해 내 걸 만들 수도 있다. 선택은 각자 몫이지만 선택지가 생기는 것 자체가 강점이라고 강조한다.

### [09:00–09:27] 정리 및 CTA
화자는 정리한다: 소프트웨어 엔지니어로 일하다 Claude Code를 만났고, 처음엔 그냥 채팅 도구인 줄 알았는데 컨텍스트 관리와 스킬을 이해한 뒤 완전히 달라졌다. 레거시 리팩토링을 반나절 만에, 어드민을 하루 만에 만들었고, 폰 보면서 연봉 받는 상황이 꿀이 아니라 공포였다. 자기 직업이 5년 안에 사라질 것 같다는 생각에 퇴사·창업했고 이제는 남의 걸 아니라 제 걸 만들고 있다. 마지막으로 AI를 어떻게 보는지(무서워 피하는지, 어떻게 써야 할지 모르는지) 개발자든 비개발자든 댓글에 현재 상황을 남겨 달라고 요청하며 끝맺는다.
