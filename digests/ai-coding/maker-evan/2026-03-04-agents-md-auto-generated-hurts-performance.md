# AI한테 설명서 줬더니 오히려 망했습니다

> maker-evan · 2026-03-04 · 길이 약 9분 30초 · [영상 링크](https://youtu.be/e7Rl4yEVYAk)
> 정리 방식: 시간순 심층 워크스루 (paraphrase) + 주제별 상세 요약
> 검증 배너: 인용된 연구 수치(성공률·비용 변화)는 화자가 소개한 값으로, 이 정리에서 원논문을 직접 검증하지 않음 (주장/미검증).

## 요약 (주제별)

### 핵심 연구 주장 (취리히 공과대학 실험)
- 화자는 스위스 취리히 공과대학(ETH Zurich로 추정) 연구팀이 AI 코딩 도구의 가이드 파일을 실험 비교했다고 소개한다 (주장/미검증).
- 조건: 같은 AI 도구, 같은 프로젝트를 동일하게 놓고, AI가 자동 생성한 `AGENTS.md`/`CLAUDE.md` vs 사람이 직접 쓴 파일을 비교.
- 결과 (모두 주장/미검증):
  - AI 자동 생성 파일 사용 시 작업 성공률 약 2~3% 하락, 비용 20% 이상 증가.
  - 사람이 직접 쓴 파일 사용 시 성공률 약 4% 상승.
  - 즉 자동 vs 수동에 따라 6~7%p 차이 (100번 중 6~7번 성공/실패가 갈리는 수준), 비용은 20% 격차.

### 왜 자동 생성 파일이 성능을 떨어뜨리나
- 자동 생성 파일에는 "이 프로젝트는 리액트를 쓴다 / 폴더 구조는 이렇다 / 타입스크립트 버전은 5.0이다" 같은, **코드만 읽으면 AI가 이미 다 아는 내용**이 들어간다.
- `package.json` 하나만 봐도 어떤 라이브러리를 쓰는지 다 나오는데, 파일이 그걸 다시 설명하면 AI가 같은 내용을 두 번 처리(중복 작업)하게 된다.
- [의견] 시험 볼 때 이미 아는 내용을 반복해 읽느라 정작 어려운 문제에 시간을 못 쓰는 것과 같다 — 그래서 비용은 오르고 중요한 데 집중 못 해 성능이 떨어진다.

### AGENTS.md / CLAUDE.md 파일의 정의
- AI 코딩 도구(클로드 코드, 커서, GitHub Copilot 등)에게 프로젝트마다 줄 수 있는 가이드 문서 = 일종의 업무 지침서.
- 예: "이 프로젝트에서는 이 도구 써요 / 이 파일은 수정하면 안 돼요 / 테스트는 이 방법으로만 해요".
- `/init` 성격의 명령어 하나로 AI가 프로젝트를 훑어 이 파일을 자동 생성하는 기능이 생겼고, 수십 줄 문서를 10초에 만들어 편리해 보이지만 여기서 문제가 발생했다.

### 좋은 가이드 파일에 써야 할 3가지 (연구 권고)
- 연구는 딱 세 종류만 쓰라고 했다고 화자가 전한다 (주장/미검증):
  1. **도구·컨벤션**: 예) 패키지 설치는 반드시 UV로. pip·conda·poetry 등 대안이 있는데 우리 팀은 UV만 쓴다면, 코드만 봐선 모르니 명시.
  2. **비직관적 제약**: 예) "테스트할 때 반드시 no-cache 옵션". 왜 그래야 하는지는 코드에 안 나오지만 안 하면 문제가 생기는, 과거에 삽질로 발견한 규칙.
  3. **시스템 특화 맥락**: 예) "이 함수는 절대 수정 금지 — 외부 서비스와 연동되어 건드리면 무너진다". 코드만으론 중요도를 알 수 없는 맥락.
- 원칙: AI가 코드를 읽어 알 수 있는 건 쓰지 말고, **코드만 봐선 절대 모를 것만** 써라. 짧아도 되고 정확한 게 중요하다.

### 카페 알바 비유 (전체를 관통하는 설명 장치)
- 나쁜 매뉴얼: "카운터는 입구 정면, 냉장고는 주방 안, 화장실은 왼편" 등 **들어오면 다 보이는** 당연한 것 → 종이만 두꺼워지고 중요한 정보가 묻힌다.
- 좋은 매뉴얼: "홍길동 단골은 아메리카노 주문 시 무조건 디카페인 / 오후 3시에 원두 갈기 / 영수증 기계 오류 시 껐다 켜되 재부팅은 금지 / 토요일 오후엔 원두 2배 준비" 등 **겪거나 알려줘야만 아는 것**.

### 매뉴얼이 길어지는 것은 경고 신호 [의견]
- 매뉴얼이 자꾸 길어지면(1페이지→5페이지) 근본 문제를 문서로 덮고 있다는 신호일 수 있다.
- 예: "커피 머신 버튼 두 번 눌러야 작동"은 머신 고장, "포스기 30분마다 재시작"은 포스기 교체 신호일 수 있다.
- 개발도 동일: AI에 설명을 계속 추가해야 한다면 코드가 너무 복잡하거나 구조가 이상한 건 아닌지 먼저 확인. "이 함수 건드리지 마세요"가 10개를 넘으면 설계가 잘못됐을 수 있다.
- 설명서를 늘리기 전에 **설명이 필요 없는(자기 설명이 되는) 구조**로 만드는 게 먼저 — 좋은 코드/시스템의 특징.

### 일반 AI 사용(ChatGPT·클로드·제미나이)으로의 확장
- 흔한 실수: "저는 30대 마케터이고 이 회사에서 5년 일했고…" 식으로 배경(롤플레잉 페르소나)을 길게 주입 → AI는 그 배경 읽는 데 처리 용량을 쓰느라 진짜 질문에 집중 못 한다.
- 마케터라는 건 쓰는 단어·문체에서 이미 드러나므로 굳이 설명할 필요 없다. 대신 AI가 스스로 알 수 없는 것만 줘라.
- 꼭 줘야 하는 것 예: "이 문서는 CEO에게 보여줄 것 / 경쟁사 얘기는 절대 금지 / 우리 회사 톤이 딱딱해서 친근한 표현은 피할 것".

### 프롬프트 좋은 예 vs 나쁜 예
- 나쁜 예: "저는 중소기업 마케팅 팀장, 직원 50명, B2B 사업, 이메일 마케팅을 하고 싶은데…" → AI가 이미 알거나 추론 가능한 내용이라 불필요.
- 좋은 예: "회계 소프트웨어 회사 마케팅 팀장이에요. 뉴스레터 제목 5개 뽑아 주세요. 단 회사 정책상 '무료·할인'이라는 단어는 절대 못 쓰고, 수신자들이 스타트업이라 너무 기본적인 내용은 피해야 해요." → 배경은 줄이고 AI가 모를 수밖에 없는 제약·맥락만 전달. 짧은 프롬프트가 긴 프롬프트를 이긴다.

### 최종 3대 교훈
- 첫째, AI를 잘 쓰려면 더 많은 정보가 아니라 **AI가 스스로 파악 못 하는 것만 핀포인트로** 준다. AI는 생각보다 맥락을 잘 읽으니 과소평가 말 것.
- 둘째, 자동화가 항상 답은 아니다. AI가 AI용 설명서를 만들었더니 오히려 성능이 떨어졌다 — 편리함 ≠ 효과. 자동화 전에 "정말 효과적인가"를 한 번 더 물을 것.
- 셋째, 설명이 많이 필요하다면 시스템의 문제일 수 있다. 더 설명하기 전에 설명이 필요 없을 만큼 명확하게 만드는 게 먼저 — 코드뿐 아니라 업무 프로세스·조직 구조에도 동일 적용.

## 3줄 오리엔테이션
- 취리히 공과대학 연구(화자 소개, 미검증)에서 AI가 자동 생성한 `AGENTS.md`/`CLAUDE.md`가 사람이 쓴 것보다 성공률을 떨어뜨리고 비용을 올렸다는 게 출발점.
- 이유는 자동 파일이 "코드만 읽으면 AI가 이미 아는 내용"을 중복 서술해 컨텍스트를 낭비하기 때문 — 카페 알바 매뉴얼 비유로 반복 설명된다.
- 결론은 AI(코딩 도구든 ChatGPT든)에게 **코드/문맥으로 알 수 없는 것만** 핀포인트로 주라는 것, 그리고 매뉴얼이 길어지면 시스템 자체를 의심하라는 것.

## 전문 — 시간순 워크스루

### [00:00–00:26] 훅: 설명서를 줬더니 성능이 떨어졌다
클로드 코드·코덱스 같은 AI 코딩 도구를 잘 쓰려고 AI에게 설명서를 줬더니 오히려 성능이 떨어졌다는 연구 결과가 나왔다고 화자가 도입한다. 유명 블로그의 주장이 아니라 스위스 명문대 연구팀이 실험으로 증명한 결과라고 강조한다 (주장/미검증). 왜 이런 일이 생겼는지, AI를 진짜 잘 쓰려면 어떻게 해야 하는지를 개발자가 아니어도 이해할 수 있게 설명하겠다고 예고한다.

### [00:26–01:08] 배경: AI 코딩 도구와 AGENTS.md/CLAUDE.md
요즘 프로그래머는 혼자 코딩하지 않고 클로드 코드·커서·GitHub Copilot 같은 AI 도구를 옆에 두고 작업한다. "버튼 누르면 에러 나는 거 고쳐줘" 하면 AI가 코드를 읽고 분석·수정하고, "이 기능 처음부터 만들어 줘"도 해낸다. 그런데 AI가 프로젝트를 처음 만나면 사람이 새 회사에 적응하듯 낯설어한다 — "이 도구 써요, 이 파일은 건드리면 안 돼요, 이 방식으로 테스트해요"를 누군가 알려줘야 한다. 그래서 나온 게 `AGENTS.md`/`CLAUDE.md` 파일, 즉 프로젝트마다 AI에게 주는 가이드 문서다.
> "프로젝트마다 AI한테 줄 수 있는 일종의 가이드 문서입니다." [01:03]

### [01:08–01:31] 카페 알바 비유 도입
낯선 이름을 카페 비유로 푼다. 새 알바를 고용하면 아무리 똑똑해도 우리 카페만의 규칙은 모르니 업무 매뉴얼을 줘야 한다 — `.md` 파일이 바로 그 역할이다. "이 도구 써요 / 이 파일은 절대 수정 금지 / 테스트는 이 방법으로만" 같은 내용을 미리 알려주는 파일이다.

### [01:31–01:53] 자동 생성 기능의 등장
이 파일을 직접 쓰는 건 개발자에게도 귀찮다 — 프로젝트마다 파일 구조·기술이 다르고 일일이 글로 정리하기 어렵다. 그래서 "이 프로젝트 보고 알아서 매뉴얼 써 줘" 하는 기능이 생겼고, 명령어 하나로 AI가 프로젝트를 훑어 `AGENTS.md`/`CLAUDE.md`를 자동 생성한다. 수십 줄 가이드를 10초에 만들어 편리해 보인다.

### [01:53–02:37] 취리히 공과대학 실험 결과
그런데 여기서 문제가 생겼다. 스위스 취리히 공과대학(세계 최고 수준 공대 중 하나)에서 AI가 자동 생성한 파일과 사람이 직접 쓴 파일을, 같은 AI 도구·같은 프로젝트 조건에서 비교했다 (주장/미검증). 결과가 충격적이었다고 한다.
> "AI가 자동으로 맞는 파일을 쓴 경우 작업 성공률이 2에서 3% 정도 떨어졌어요. 그리고 비용은 20% 이상 늘어났습니다." [02:14]

반대로 사람이 직접 써 준 파일은 성공률이 약 4% 올라, 자동이냐 수동이냐에 따라 6~7%p 차이가 났다 (주장/미검증). 100번 작업하면 6~7번의 성공/실패가 갈리는 차이이고, 비용 20% 격차는 같은 결과에 돈을 20% 더 쓰는 셈이라 크다고 강조한다.

### [02:37–03:01] 왜 나쁜가 (1): 당연한 걸 설명하는 매뉴얼
다시 카페 비유. 알바에게 "카페는 1층 / 카운터는 입구 정면 / 커피 머신은 카운터 뒤 / 냉장고는 주방 안 / 화장실은 왼편"이라고 써 주면, 눈 있는 사람은 들어오면 다 보인다. 당연한 걸 설명하며 종이만 두꺼워지고, 알바는 그걸 다 읽느라 시간을 낭비하며 진짜 중요한 정보가 묻힌다.

### [03:01–03:37] 왜 나쁜가 (2): AI도 코드로 이미 안다
AI가 자동 생성하는 파일에는 "이 프로젝트는 리액트 씁니다 / 폴더 구조는 이렇다 / 타입스크립트 버전은 5.0"이 들어간다. 그런데 AI는 코드를 읽으면 이미 다 안다 — `package.json` 하나만 봐도 어떤 라이브러리를 쓰는지 다 나온다. 자동 생성 파일이 그걸 다시 설명하면 AI는 같은 내용을 두 번 처리해 중복 작업이 생긴다.
> "마치 시험 볼 때 이미 다 아는 내용을 반복해서 읽느라 정작 어려운 문제에 시간을 못 쓰는 것처럼요." [03:27]

그래서 비용은 오르고 정작 중요한 데 집중 못 해 성능이 떨어진다.

### [03:37–04:09] 효과적인 매뉴얼: 겪어야만 아는 것
효과적으로 쓰려면 어떻게 하나. 알바에게 진짜 도움이 되는 매뉴얼은 이런 것이다 — "아메리카노 주문 시 무조건 디카페인으로 드려야 하는 단골 홍길동 씨 / 오후 3시에 원두 갈기, 안 하면 저녁에 맛이 달라짐 / 영수증 기계 오류 시 껐다 켜면 되고 재부팅은 금지 / 토요일 오후엔 줄이 길어 원두 2배 준비". 보기만 해선 절대 알 수 없고 시행착오나 누군가 알려줘야만 아는 것들이다. AI에게도 이런 정보를 줘야 한다.

### [04:09–04:49] 연구 권고: 딱 3가지만 써라
연구에서도 세 가지만 쓰라고 했다고 화자가 전한다 (주장/미검증). (1) **도구·컨벤션**: 예로 프로젝트가 UV 패키지 관리자를 쓴다면 "설치는 반드시 UV로"라고 써야 한다 — pip·conda·poetry 대안이 있어 코드만 봐선 모를 수 있다. (2) **비직관적 제약**: "테스트 시 반드시 no-cache 옵션" 같은, 왜인지 코드엔 안 나오지만 안 하면 문제 생기는, 과거에 삽질로 발견한 규칙. (3) **시스템 특화 정보**: "이 함수는 절대 수정 금지 — 외부 서비스와 연동되어 건드리면 다 무너진다" 같은, 코드로는 중요도를 알 수 없는 맥락.

### [04:49–04:57] 요약 원칙: 모를 것만, 짧아도 정확히
정리하면 AI가 코드를 읽어 알 수 있는 건 쓰지 말고, 코드만 봐선 절대 모를 것만 써라. 짧아도 괜찮고 정확한 게 중요하다.

### [04:57–05:44] 확장: 모든 AI 사용의 공통 실수
여기서 화자가 멈추고 강조한다 — 이 연구는 AI 코딩 도구에만 해당하지 않고, AI에게 뭔가 시킬 때 개발자·마케터·기획자 모두가 범하는 공통 실수를 보여준다. 많은 이가 "정보를 더 주면, 맥락을 길게 줄수록 더 좋은 답이 나온다"고 직관적으로 느끼지만 연구 결과는 반대다.
> "AI한테도 필요한 것만 줘야 해요. 너무 많은 정보는 AI를 오히려 혼란스럽게 만들 수 있어요." [05:24]

당연한 걸 설명하는 데 처리 용량을 쓰면 정말 중요한 걸 처리할 여유가 줄고, 컨텍스트 창은 무한하지 않다. GPT·제미나이·클로드 모두 마찬가지 — 프롬프트를 길게 쓴다고 좋은 게 아니라 핵심만 정확히 전달하는 게 중요하다.

### [05:44–06:32] 흥미로운 관점: 내용 추가는 경고 신호 [의견]
연구가 제시한 흥미로운 관점 하나 — 내용을 계속 추가하는 것 자체가 경고 신호라는 것이다. 카페 매뉴얼이 1페이지에서 5페이지로 늘어난다면, "커피 머신 버튼 두 번 눌러야 작동"은 머신 고장, "냉장고 문 두 번 닫아 확인"은 문이 제대로 안 닫히는 것, "포스기 30분마다 재시작"은 포스기 교체 신호일 수 있다. 매뉴얼이 길어진다는 건 근본 원인을 해결하지 않고 예외 처리를 추가하며 버티는, 문제를 매뉴얼로 덮는 신호일 수 있다.

### [06:32–06:39] 개발 적용: 설계 이상 신호
개발 프로젝트도 마찬가지다. AI에 설명을 계속 추가해야 한다면 애초에 코드가 너무 복잡하거나 구조가 이상한 건 아닌지 먼저 확인해야 한다. "이 함수 건드리지 마세요"가 10개를 넘으면 설계가 잘못된 신호일 수 있다. [의견] 설명서를 늘리기 전에 설명이 필요 없는 구조로 만드는 게 먼저이고, 자기 설명이 되는 코드가 좋은 코드·좋은 시스템의 특징이다.

### [06:39–07:56] 실용편: ChatGPT·클로드·제미나이 프롬프트
이제 코딩 안 해도 적용되는 실용 얘기. 지금 많은 사람이 "저는 30대 마케터이고 이 회사에서 5년 일했고 평소 이런 문서를 쓰고 이런 스타일 좋고 우리 팀은 이런 분위기예요" 식으로 배경을 길게 깔고 시작한다 — 일종의 롤플레잉 페르소나 주입이다. 하지만 AI는 그 긴 배경을 읽는 데 처리 용량을 쓰느라 진짜 질문에 집중 못 하고, 마케터라는 건 쓰는 단어·문체에서 이미 느껴진다. 바꾸는 법: 배경 설명 대신 AI가 스스로 알 수 없는 것만 줘라. "마케터예요"는 안 써도 되고, "30대"도 대부분 불필요하다. 대신 "이 문서는 CEO에게 보여줄 것 / 경쟁사 얘기는 절대 금지 / 우리 회사 톤이 딱딱해 친근한 표현은 피하고 마케터에 어울리는 문체로" 같은 건 AI가 모르므로 꼭 줘야 한다.

### [07:58–08:35] 구체 예: 나쁜 프롬프트 vs 좋은 프롬프트
잘못된 프롬프트는 "저는 중소기업 마케팅 팀장이고 직원 50명, B2B 사업, 이메일 마케팅을 하고 싶은데 블라블라" — AI가 이미 알거나 추론 가능한 내용이라 안 줘도 된다. 좋은 프롬프트는 "회계 소프트웨어 회사 마케팅 팀장이에요. 뉴스레터 제목 5개 뽑아 주세요. 단 회사 정책상 '무료·할인'이라는 단어는 절대 못 쓰고, 수신자들이 스타트업이라 너무 기본적인 내용은 피해야 해요." [불명확: 원문 "스타트업십라는" — 문맥상 '수신자가 스타트업이라'로 교정] 배경은 줄이고 AI가 모를 수밖에 없는 제약·맥락만 줬더니 결과는 두 번째가 훨씬 좋게 나온다 — 짧은 프롬프트가 긴 프롬프트를 이긴다.

### [08:35–09:08] 최종 3대 교훈
핵심 정리. 첫째, AI를 잘 쓰고 싶으면 더 많은 정보가 아니라 AI가 스스로 파악할 수 없는 것만 핀포인트로 줘라 — AI는 생각보다 맥락을 잘 읽으니 과소평가 말 것. 둘째, 자동화가 항상 답은 아니다 — AI가 AI를 위한 설명서를 만들었더니 오히려 성능이 떨어졌고, 편리한 것과 효과적인 것은 다르니 자동화 전에 정말 효과적인가를 한 번 더 물어라. 셋째, 설명이 많이 필요하다면 시스템의 문제일 수 있으니, 더 설명하라고 하기 전에 설명이 필요 없을 만큼 명확하게 만드는 게 먼저다 — 코드뿐 아니라 업무 프로세스·조직 구조에도 똑같이 적용된다.

### [09:08–09:30] 마무리
화자도 처음 이 연구를 봤을 때 "AI한테 더 많이 알려주면 당연히 좋은 거 아니야?"라고 직관적으로 생각했지만, 토큰 컨텍스트를 생각하면 아니더라고 회고한다. AI 도구를 쓰는 사람들에게 오늘부터 배경 설명을 줄이고 AI가 절대 모를 것만 알려주는 걸 시도해 보라고 권하며 결과가 달라질 거라 마무리한다.
