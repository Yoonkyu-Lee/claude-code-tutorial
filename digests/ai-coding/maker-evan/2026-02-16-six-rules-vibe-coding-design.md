# 바이브코딩 결과물, 3초만에 들통나는 이유

> maker-evan · 2026-02-16 · 길이 약 6:08 · [영상 링크](https://youtu.be/uos3e3r6wGY)
> 정리 방식: 시간순 심층 워크스루 (paraphrase) + 주제별 상세 요약
> 검증 배너: 도구·수치·효과 언급은 화자 주장이며 미검증. `[의견]`은 화자 주관.

## 요약 (주제별)

### 문제 정의 — 왜 바이브 코딩 결과물이 "티나는가"
- 바이브 코딩으로 만든 앱은 기능·동작은 완벽한데 화면을 열면 3초 만에 티가 난다. 원인은 **디자인** (주장).
- 증상: 한 화면에 색이 다섯~여섯 개, 버튼 크기 제각각, 여백 들쑥날쑥.
- 해법은 디자인 감각을 기르는 게 아니라(피그마 강의·컬러 공부·UIUX 책은 과함) **지키기만 하면 되는 규칙**이라는 게 핵심 전제. [의견] "디자인 공부 다 할 거면 왜 바이브 코딩을 하겠나."

### 규칙 1~2 — 컬러: 직접 고르지 말고, 두 개만 써라
- **규칙 1**: 컬러를 직접 고르지 마라. 바이브 코딩 결과물이 구려 보이는 1순위 원인은 색깔(주장). AI에게 "파란색으로 해줘" 하면 아무 파란색이나 가져와 옆 회색·배경과 안 어울린다. AI는 색 조합을 잘 못 맞춘다(주장).
- 도구: **Coolors** (자막의 "클러즈" = coolors.co). 스페이스바를 누르면 조화로운 팔레트가 자동 생성된다. 마음에 드는 팔레트의 **hex 코드를 그대로 복사해 프롬프트에 넣는다**. "좀 더 진하게" 식으로 **수정하면 조화가 깨진다** — 준 그대로 쓰는 게 핵심.
- **규칙 2**: 팔레트에서 색이 5개 나와도 **딱 2개만** 쓴다. 메인 컬러 1 + 서브 컬러 1. 나머지는 전부 흰색·검정·회색.
- 배분: 메인 컬러 = CTA 버튼·강조에만, 서브 컬러 = 배경·보조 요소에만. 여기저기 빨강·초록·노랑 시키면 "축제 현수막"이 된다. [의견] 잘 만든 사이트는 사실상 2~3색만 쓴다(예로 토스 언급).

### 규칙 3 — Stitch로 화면 만들 때 두 가지 필수
- **Stitch**(구글): 프롬프트 한 줄로 웹/앱 UI를 만들어 주는 도구.
- 필수 ①: Coolors에서 받은 **컬러 코드(hex)를 정확히** 넣는다. "파란색으로 해줘" 금지 — hex를 넣어야 일관된 컬러가 나온다.
- 필수 ②: 레퍼런스 사이트 묘사는 **영문으로**. "토스 앱처럼 깔끔하게" 같은 한국어는 AI가 잘 못 알아듣는다. 영문으로 구체적으로 묘사하면 결과가 확 달라진다(주장).

### 규칙 4 — MCP로 디자인하지 마라
- MCP(Model Context Protocol) = AI에 외부 도구를 연결하는 것. 피그마 MCP로 디자인시키면 이론적으론 좋아 보인다.
- 실제로는 **토큰이 터져** 디자인 일관성·코드 품질이 계속 떨어진다(주장). 토큰이 순식간에 소진되며 AI가 앞 맥락을 까먹고, 수정할수록 버튼 스타일·간격·컬러가 페이지마다 흔들린다.
- [의견] MCP로 디자인을 직접 조작하는 건 개인적으로 비추.

### 규칙 5 — MCP 대신 Skills를 써라
- **Skills** = 클로드 코드에서 특정 작업의 가이드라인을 미리 **문서로** 만들어 두는 것. 예: "웹 디자인 가이드라인" 스킬에 컬러·여백·폰트 사이즈 체계 규칙을 적어 둔다.
- 클로드 코드가 코드 생성 시 그 규칙을 참고해 일관되게 만든다. MCP처럼 매번 API 호출이 아니라 텍스트 문서를 읽는 것이라 **토큰 소모가 훨씬 적고 일관성이 유지**된다(주장).
- [의견] 바이브 코딩의 핵심은 AI에게 잘 시키는 것 — Skills는 디자인 규칙을 잘 알려 주는 수단.

### 규칙 6 — Stitch용 영문 프롬프트는 클로드로 추출
- 규칙 3의 레퍼런스 사이트 영문 묘사 프롬프트는, **클로드를 통해 영문 프롬프트를 추출**하면 효과가 더 좋았다. [의견] 굉장히 개인적인 팁.

### 규칙 7(보너스) — Next.js는 MUI보다 shadcn/ui
- 자막의 "일곱 번째 … 아는 사람만 아는 것": 바이브 코딩으로 Next.js 프로젝트를 만들 땐 **MUI보다 shadcn/ui** 추천.
- MUI 단점: 커스터마이징이 고통. 버튼색 하나 바꾸려 해도 theme 파일·팔레트·override·sx prop을 만지고, 컴포넌트마다 방식이 달라 AI도 헷갈린다. 머티리얼 디자인 느낌에서 벗어나려면 프레임워크와 싸워야 한다.
- shadcn/ui 장점: 컴포넌트를 npm 설치가 아니라 **소스 코드를 프로젝트에 직접 복사**하는 방식. `button.tsx`가 내 프로젝트 안에 있어 AI가 코드를 직접 수정 가능 → 바이브 코딩과 궁합 최고(주장). 메인/서브 컬러를 config에 넣으면 끝, 디자인 감각 없어도 일관되게 나온다.

### 실전 절차(전체 순서)
1. Coolors 접속 → 스페이스바로 팔레트 고름 → 메인 1·서브 1의 **hex 복사**.
2. 프로젝트: Next.js + **shadcn/ui** 세팅.
3. `tailwind.config.js`에 두 컬러를 primary·secondary로 넣음.
4. 레이아웃 필요하면 **Stitch** 사용 — primary/secondary hex 코드 입력, 레퍼런스 사이트는 (클로드로 추출한) 영문으로 구체 묘사.
5. 클로드 코드로 바이브 코딩할 땐 **Skills의 디자인 가이드라인**을 넣어 둔다. MCP로 디자인 도구 연결은 하지 않는다.

### 리스크·주의 / 흔한 실수
- [의견] "기능 먼저, 디자인 나중" 금지 — 처음부터 컬러·shadcn 세팅하고 시작하라. 나중에 바꾸면 열 배 더 힘들다.
- 컬러 임의 수정 금지, 색 3개 이상 남발 금지, MCP 디자인 조작 금지.
- 마무리 메시지: 결과물이 구려 보이는 건 실력 문제가 아니라 **규칙을 안 지켜서**이고, 오늘의 규칙만 지켜도 꽤 괜찮은 퀄리티가 나온다(주장).

## 전문 — 시간순 워크스루

### [00:00–00:11] 훅 — 3초 만에 들통난다
바이브 코딩으로 만든 결과물은 디자인 때문에 3초 만에 티가 난다고 단언하며 연다. 하지만 오늘 알려 줄 규칙 여섯 개만 지키면 "이거 진짜 바이브 코딩으로 만든 거야?"라는 소리를 듣는다고 약속한다(주장).
> "바이브 코딩으로 만든 결과물이요. 3초 만에 티납니다. 디자인 때문에요." [00:00]

### [00:12–00:46] 문제 정의 — 기능은 되는데 디자인이 문제
바이브 코딩으로 앱을 뚝딱 만들면 기능·동작은 완벽한데 화면을 열면 색이 세~여섯 개, 버튼 크기 제각각, 여백은 들쑥날쑥이라 한숨이 나온다고 증상을 나열한다. 그래서 보통 피그마 강의·컬러 공부·UIUX 책으로 디자인을 배우려 하는데, 화자는 그럴 거면 왜 바이브 코딩을 하냐고 반문한다. 바이브 코딩에 필요한 건 디자인 감각이 아니라 지키기만 하면 되는 규칙이라며 여섯 가지를 예고한다.
> "바이브 코딩한테 필요한 건요. 디자인 감각이 아닙니다. 그냥 지키기만 하면 되는 규칙이에요." [00:40]

### [00:47–01:35] 규칙 1 — 컬러를 직접 고르지 말고 Coolors에서 받아 그대로
결과물이 구려 보이는 1순위 원인은 색깔이라고 짚는다(주장). AI에게 "파란색으로 해줘"라고 하면 아무 파란색이나 가져와 옆 회색·배경과 안 어울리며, AI는 색 조합을 잘 못 맞춘다고 본다. 대안으로 Coolors(자막 "클러즈", coolors.co)를 제시 — 스페이스바를 누르면 조화로운 팔레트가 자동 생성되고, 마음에 드는 팔레트의 hex 코드를 그대로 복사해 프롬프트에 넣으라고 한다. "좀 더 진하게" 식으로 수정하는 순간 조화가 깨지니 준 그대로 쓰는 게 핵심이라고 강조한다.

### [01:36–02:19] 규칙 2 — 다섯 색 중 딱 두 개만
Coolors가 색 다섯 개를 주더라도 다 쓰면 안 되고 메인 1·서브 1, 딱 두 개만 쓰라고 한다. 두 개로 되겠냐는 반문에 잘 만든 사이트를 보라며, 흰색·검정 포인트에 옅은 파란색 정도로 사실상 두세 색만 쓴다고(예로 토스 언급) 답한다. AI에게 여기 빨강·저기 초록·이건 노랑 식으로 시키면 "축제 현수막"이 된다고 경고한다. 메인 컬러는 CTA 버튼·강조에만, 서브 컬러는 배경·보조 요소에만 쓰고 나머지는 전부 흰색·검정·회색으로 채우면 "깔끔하다"는 소리를 듣는다고 정리한다.
> "딱 두 개만 쓰세요. 메인 컬러 하나, 서브 컬러 하나." [01:42]

### [02:19–02:52] 규칙 3 — Stitch: hex 정확히 + 레퍼런스는 영문으로
AI에게 디자인을 시킬 때 Stitch(자막 "스티치")를 쓰라고 한다. Stitch는 프롬프트 한 줄로 웹/앱 UI를 만들어 주는 도구. 쓸 때 두 가지 필수 — ① Coolors에서 받은 컬러 코드를 정확히(hex로) 넣어야 일관된 컬러가 나온다, "파란색으로 해줘"는 금지. ② 레퍼런스 사이트 묘사는 영문으로 하라, "토스 앱처럼 깔끔하게" 같은 한국어는 AI가 잘 못 알아들으니 영문으로 구체적으로 묘사하면 결과가 확 달라진다(주장).

### [02:53–03:40] 규칙 4 — MCP로 디자인하지 마라
클로드 코드 같은 AI 코딩 도구를 쓸 때 MCP로 디자인하지 말라고 한다. MCP(Model Context Protocol)는 AI에 외부 도구를 연결하는 것이며, 피그마 MCP로 디자인을 시키면 이론상 좋아 보인다. 그러나 실제로는 토큰이 터져 디자인 일관성과 코드 품질이 계속 떨어진다고 주장한다. 토큰이 순식간에 소진되며 AI가 앞 맥락을 까먹기 시작하고, 처음엔 깔끔했던 디자인이 수정할수록 버튼 스타일·간격·컬러가 페이지마다 흔들린다. 화자는 MCP로 디자인을 직접 조작하는 건 개인적으로 비추라고 한다. [의견]

### [03:40–04:20] 규칙 5 — MCP 대신 Skills
대안으로 Skills(자막 "스킬즈")를 제시한다. Skills는 클로드 코드에서 특정 작업의 가이드라인을 미리 문서로 만들어 두는 것으로, 예컨대 "웹 디자인 가이드라인" 스킬에 컬러·여백·폰트 사이즈 체계 규칙을 적어 둔다. 그러면 클로드 코드가 코드를 생성할 때 그 규칙을 참고해 일관되게 만든다. MCP처럼 매번 API를 호출하는 게 아니라 텍스트 문서를 읽는 것이라 토큰 소모가 훨씬 적고 일관성이 유지된다고 한다(주장). 바이브 코딩의 핵심은 AI에게 잘 시키는 것이고, Skills는 디자인 규칙을 잘 알려 주는 수단이라고 마무리한다. [의견]

### [04:21–04:40] 규칙 6 — 영문 프롬프트는 클로드로 추출
앞서 Stitch에서 레퍼런스 사이트를 영문으로 묘사하라고 한 것과 연결해, 그 영문 프롬프트를 클로드를 통해 추출하면 효과가 더 좋았다고 덧붙인다. 굉장히 개인적인 팁이라고 단서를 단다. [의견]

### [04:40–05:31] 규칙 7(보너스) — MUI보다 shadcn/ui
"아는 사람만 아는" 일곱 번째로, 바이브 코딩으로 Next.js 프로젝트를 만들 땐 MUI(자막 "무의/무이")보다 shadcn/ui(자막 "샤드CN/샤드시")를 추천한다. MUI는 커스터마이징이 고통스러워 버튼색 하나 바꾸려 해도 theme 파일 열고 팔레트 수정, override 넣고 sx prop 넣고, 컴포넌트마다 방식이 달라 AI도 헷갈려한다고 한다. 머티리얼 디자인 느낌이 좋을 때도 있지만 벗어나려면 프레임워크와 싸워야 한다. 반면 shadcn/ui는 컴포넌트를 npm 설치가 아니라 소스 코드를 프로젝트에 직접 복사하는 방식이라 `button.tsx`가 내 프로젝트 안에 있고, AI가 코드를 직접 수정할 수 있어 바이브 코딩과 궁합이 최고라고 한다(주장). 메인·서브 컬러를 config에 넣으면 끝이며, 디자인 감각이 없어도 일관되게 나온다고 강조한다.
> "컴포넌트는 npm으로 설치하는 게 아니라 소스 코드를 직접 프로젝트에 복사해 넣는 방식이거든요." [05:10]

### [05:31–06:20] 실전 절차 총정리
전체 순서를 정리한다: (1) Coolors 접속 → 스페이스바로 팔레트 고르고 메인 1·서브 1의 hex 복사, (2) 프로젝트를 Next.js + shadcn/ui로 세팅, (3) `tailwind.config.js`에 두 컬러를 primary·secondary로 넣기, (4) 레이아웃이 필요하면 Stitch에 primary/secondary hex를 넣고 레퍼런스 사이트는 (클로드로 추출한) 영문으로 구체 묘사, (5) 클로드 코드로 바이브 코딩할 땐 Skills의 디자인 가이드라인을 넣어 두고 MCP로 디자인 도구를 연결하지 않기. 이 순서만 따르면 바이브 코딩으로 만들었어도 꽤 괜찮은 퀄리티가 나온다고 한다(주장). 흔한 실수로 "기능 먼저, 디자인 나중"을 지적하며 처음부터 컬러·shadcn을 세팅하고 시작하라고, 나중에 바꾸면 열 배 더 힘들다고 조언한다.
