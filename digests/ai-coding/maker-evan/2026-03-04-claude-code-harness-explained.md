# 하니스 완전 공개 — Claude Code AI를 내 전담 직원으로 만드는 법

## 요약 (주제별)

### 하니스란 무엇인가
- 하니스(harness)는 AI를 쉽게 쓸 수 있게 도와주는 보조 도구다. Claude Code는 기본적으로 무엇이든 할 수 있는 범용 AI지만, 그 말은 반대로 "우리 프로젝트 방식"은 아무것도 모른다는 뜻이다.
- 비유: 새 직원은 머리가 좋아도 첫날부터 알아서 일하지 못한다. 보고서 작성법·고객 응대 방식·상황별 담당자를 하나하나 알려줘야 한다. 하니스는 Claude Code에게 이 "팀 방식"을 미리 다 알려주는 장치다.
- 한 번 설치해 두면 매번 "우리 팀 방식대로 해 줘"라고 설명하지 않아도 알아서 맞게 처리해 준다. 한 줄 요약: Claude Code를 범용 AI가 아닌 "우리 프로젝트 전담 전문가"로 만드는 것.

### 하니스의 세 구성요소
- **스킬(skill)**: 특정 분야의 전문 지식을 미리 담아둔 문서. 일반 Claude가 "동네 병원 의사"라면 스킬을 붙이면 그 분야 전문의가 된다. 사용법은 말할 때 앞에 슬래시로 스킬 이름을 붙이는 것(예: "그냥 회원가입 만들어 줘" 대신 "슬래시 FastAPI 백엔드 가이드라인 회원가입 만들어 줘").
- **에이전트(agent)**: "이걸 해결해 줘"라고 시키면 혼자서 생각하고 관련 내용을 찾아보고 고치고 확인까지 다 해 주는 전문 AI. 시키면 알아서 끝까지 하는 보조 직원. 중간에 사용자에게 되묻지 않는다.
- **커맨드(command)**: 자주 하는 복잡한 작업을 슬래시 명령어 하나로 바로 실행하는 단축키/매크로 같은 것.

### 스킬 vs 에이전트 차이
- 스킬은 "이 분야 전문 지식으로 답해 줘"라고 하는 것(지식 주입). 에이전트는 "이 문제 스스로 해결해 줘"라고 하는 것(자율 실행).
- 에이전트는 혼자서 관련 파일을 열어 읽고, 뭐가 문제인지 파악하고, 고치고, 제대로 됐는지까지 확인한다.

### 소개된 스킬 목록
- **FastAPI 백엔드 가이드라인**: 서버 쪽 코드 작성용. 스킬 없이 물어보면 빠르게 동작하는 코드를 만들지만 기능이 늘면 코드가 엉망이 되기 쉽다. 스킬이 있으면 처음부터 역할별로 깔끔하게 나눠 만들어 수정·협업이 쉽다(서랍장 정리 비유: 아무렇게나 던져 넣기 vs 양말칸·셔츠칸 구분).
- **Next.js 프론트엔드 가이드라인**: 웹사이트 화면 제작용. "로그인 화면 만들어 줘"라고 하면 단순 화면이 아니라 최신 방식에 맞게 속도 빠르고 구조 올바른 화면을 만든다. 웹 개발 기술이 빠르게 바뀌어 정답이 헷갈리는 상황에서 올바른 방식을 알아서 판단해 준다.
- **Mermaid 스킬**: 복잡한 내용을 그림으로 설명할 때. 흐름도(화살표 연결도), 데이터 관계도, 서비스 구조도를 자동 생성. Notion·GitHub에 붙여 넣으면 바로 그림으로 보인다. 팀원 설명에 유용.
- **PDF 스킬**: PDF 관련 작업 전반 — 글자 뽑아내기, 페이지 쪼개기, 여러 PDF 합치기, 스캔 문서 글씨 인식(OCR).
- **PPTX 스킬**: 보고서·제안서를 반복 제작할 때 효과적. "이 내용으로 발표자료 만들어 줘"라고 하면 코드로 PPT를 자동 생성.
- **웹 디자인 가이드라인 + React best practice 스킬**: 화면을 다 만든 뒤 "이 디자인 리뷰해 줘"라고 하면 색상 대비·여백을 체크. 혼자 개발할 때 없는 "디자인 리뷰어 + React 코드 리뷰어" 역할을 대신한다.

### 소개된 에이전트 목록
- **플래너(planner) 에이전트**: 새 기능 제작 전 "이거 어떻게 만들면 좋을까" 계획 자체를 짜는 역할. "사용자 기능 만들고 싶어"라고 하면 어떤 파일을 어떤 순서로, 어디서 어디로 연결되는지, 예상 위험 요소는 뭔지까지 단계별로 정리. 이걸 보고 시작하면 중간에 방향이 흔들리는 일이 줄어든다.
- **플랜 리뷰(plan reviewer) 에이전트**: 플래너가 짠 계획을 검토. "이 계획 리뷰해 줘"라고 하면 빠진 부분·보안 문제·나중에 문제될 지점을 하나하나 짚어 준다. 계획이 부실하면 나중에 다 뜯어야 하는 상황을 예방.
- **웹 리서치 스페셜리스트(web research specialist) 에이전트**: 개발하다 막힐 때. "이 오류 해결 방법 찾아 줘"라고 하면 개발자 커뮤니티·Q&A 사이트·공식 이슈 게시판을 직접 뒤져 정보를 모아 정리. 구글 검색을 대신하되 여러 곳을 동시에 찾아 정리해 오므로 직접 찾는 것보다 빠르다.

### 커스터마이징
- [의견] 하니스를 그대로 써도 좋지만 내 상황에 맞게 바꾸면 훨씬 강력하다.
- 회사만의 규칙이 있다면 스킬 파일을 열어 팀 규칙을 추가하면 된다(일반 텍스트 문서라 편집이 쉽다).
- 에이전트도 직접 만들 수 있다 — 파일 하나만 추가하면 된다(예: 우리 회사 방식으로 코드 검토하는 에이전트, 서비스 특성에 맞는 디버거).

### 설치·세일즈 구조 / 효용
- Claude Code와 화자가 만든 하니스 설치 방법은 영상 더보기(설명란)나 댓글의 가이드 문서 링크에서 단계별로 안내. 링크 확인 유도(세일즈/유입 장치).
- [의견] 처음 설치는 조금 번거롭게 느껴질 수 있지만, 한번 해 두면 같은 설명을 반복하지 않아도 되고 매번 일정한 결과물이 나와 장기적으로 시간이 훨씬 절약된다.

- 지난 영상에서 공유한 "하니스"가 무엇이고 어떻게 쓰는지, 코딩을 몰라도 이해하도록 비유 중심으로 풀어낸 입문 설명 영상이다.
- 핵심은 하니스가 스킬·에이전트·커맨드 세 요소로 구성되며, 각각 소개된 실제 스킬/에이전트 목록과 그 용도를 훑는다.
- 결론은 "Claude Code를 범용 AI가 아닌 우리 프로젝트 전담 전문가로 만드는 장치"이며, 커스터마이징과 설치 가이드 유도로 마무리한다.

## 전문 — 시간순 워크스루

### [00:00–00:11] 영상 취지 — "뭔지는 알겠는데 어떻게 쓰는지 모르겠다"
저번 시간에 자신의 하니스를 공유했는데, "이게 뭔지는 알겠지만 어떻게 쓰는 건지 모르겠다"는 반응이 너무 많았다고 한다. 그래서 이번 영상에서는 코딩을 전혀 몰라도 이해할 수 있게 하나하나 쉽게 뜯어 설명하겠다고 취지를 밝힌다.

### [00:11–00:18] 하니스란 무엇인가
하니스는 AI를 쉽게 쓸 수 있게 도와주는 보조 도구라고 정의한다. 먼저 이게 뭔지 이해하고 넘어가야 한다며 비유를 예고한다.

### [00:18–00:30] 비유 — 새 직원 온보딩
> "여러분이 새 직원을 뽑았다고 생각해 보세요. 이 직원이 아무리 머리가 좋아도 첫날부터 알아서 일 잘하진 않잖아요." [00:19]

새 직원에게는 보고서를 이렇게 쓰고, 고객 응대는 이런 방식으로 하고, 이 상황엔 이 사람한테 물어보라고 하나하나 알려줘야 한다는 것이다.

### [00:30–00:48] Claude Code도 똑같다 — 하니스가 푸는 문제
Claude Code도 기본적으로는 무엇이든 할 수 있는 범용 AI인데, 그 말은 반대로 우리 프로젝트 방식은 아무것도 모른다는 뜻이라고 설명한다. 하니스는 바로 이 문제를 해결해 준다 — Claude Code에게 "우리 팀은 이렇게 일한다"를 미리 다 알려주는 장치라고 보면 된다. 이걸 설치해 두면 매번 우리 팀 방식대로 해 달라고 설명하지 않아도 알아서 딱 맞게 해 준다.

### [00:48–01:14] 하니스의 세 구성요소 — 스킬·에이전트·커맨드
하니스에서 크게 세 가지만 알면 된다고 한다. 첫째 **스킬**은 특정 분야의 전문 지식을 미리 담아둔 문서로, 일반 Claude가 동네 병원 의사라면 스킬을 붙이면 그 분야 전문의가 되는 것이다. 둘째 **에이전트**는 "이걸 해결해 줘"라고 하면 혼자서 생각하고 관련 내용을 찾아보고 고치고 확인까지 다 해 주는 전문 AI로, 시키면 알아서 끝까지 하는 보조 직원이다. 셋째 **커맨드**는 자주 하는 복잡한 작업을 슬래시 명령어 하나로 바로 실행하는 단축키/매크로 같은 것이다.

### [01:14–01:23] 설치 가이드 안내 (세일즈/유입 장치)
Claude Code와 화자가 만든 하니스의 설치 방법은 영상 더보기(설명란)나 댓글을 통해 가이드 문서를 받으면 단계별로 안내가 나와 있다고 한다. 따라 하면 어렵지 않게 설치할 수 있으니 링크를 꼭 확인하라고 유도한다.

### [01:23–01:45] 스킬 사용법 — 슬래시로 이름 붙이기
스킬을 더 쉽게 설명하면 Claude에게 지어주는 "분야별 전문 가이드북"이다. 사용법은 간단해서, 말할 때 앞에 슬래시로 스킬 이름을 붙이면 된다. 예를 들어 "그냥 회원가입 만들어 줘" 대신 "슬래시 FastAPI 백엔드 가이드라인 회원가입 만들어 줘"라고 하면, Claude가 그 스킬에 담긴 전문 지식을 활용해 대답해 준다.

### [01:45–02:08] 스킬 ① FastAPI 백엔드 가이드라인
서버 쪽 코드를 짤 때 쓰는 스킬이다. 스킬 없이 그냥 물어보면 Claude가 빠르게 동작하는 코드를 만들어 주지만, 나중에 기능이 늘어나면 코드가 엉망이 되는 경우가 많다. 스킬이 있으면 처음부터 역할별로 깔끔하게 나눠 만들어 수정하기도 쉽고 다른 사람이 봐도 이해하기 쉬운 코드가 나온다.
> "마치 서랍장에 물건을 대충 던져 넣는 것과 양말은 양말칸, 셔츠는 셔츠칸으로 정리하는 것의 차이예요." [02:04]

### [02:08–02:27] 스킬 ② Next.js 프론트엔드 가이드라인
웹사이트 화면을 만들 때 쓰는 스킬이다. "로그인 화면 만들어 줘"라고 하면 단순한 화면이 아니라 최신 방식에 맞게 속도 빠르고 구조도 올바른 화면을 만들어 준다. 최근 웹 개발 기술이 빠르게 바뀌면서 어떤 방식이 맞는지 헷갈리는 경우가 많은데, 이 스킬이 있으면 Claude가 알아서 올바른 방식으로 판단해 준다.

### [02:27–02:46] 스킬 ③ Mermaid
복잡한 내용을 그림으로 설명해야 할 때 쓰는 스킬이다. "이 기능 흐름도 그려 줘"라고 하면 화살표 연결도로, "이 데이터 관계 보여 줘"라고 하면 관계도로, "서비스 구조 설명해 줘"라고 하면 구조도로 자동으로 그려 준다. 글로만 설명하면 이해하기 어려운 내용을 시각적으로 보여줄 수 있어 팀원들에게 설명할 때 특히 유용하고, Notion이나 GitHub에 붙여 넣으면 바로 그림으로 보인다.

### [02:46–02:56] 스킬 ④ PDF
PDF 파일 관련된 건 모두 다 할 수 있는 스킬이다. 글자 뽑아내기, 페이지 쪼개기, 여러 PDF 합치기, 심지어 스캔한 문서에서 글씨 인식하기(OCR)까지 가능하다.

### [02:56–03:04] 스킬 ⑤ PPTX
보고서나 제안서를 반복적으로 만들어야 할 때 특히 효과적인 스킬이다. "이 내용으로 발표자료 만들어 줘"라고 하면 코드로 PPT를 자동 생성해 준다.

### [03:04–03:22] 스킬 ⑥ 웹 디자인 가이드라인 + React best practice
마지막으로 웹 디자인 가이드라인과 React best practice 스킬이다. 화면을 다 만든 뒤 "이 디자인 리뷰해 줘"라고 하면 색상 대비가 적절한지, 여백은 충분한지 체크해 준다. 혼자 개발하다 보면 디자인 퀄리티를 검토해 줄 사람이 없는데, 이 스킬이 디자인 리뷰어 겸 React 코드 리뷰어 역할을 해 준다.

### [03:24–03:41] 에이전트란 — 스킬과의 차이
다음은 에이전트로, 알아서 일하는 보조 직원들이다. 스킬과 뭐가 다르냐면 — 스킬은 "이 분야 전문 지식으로 답해 줘"라고 하는 것이라면, 에이전트는 "이 문제 스스로 해결해 줘"라고 하는 것이다. 에이전트는 혼자서 관련 파일을 열어 읽고, 뭐가 문제인지 파악하고, 고치고, 제대로 됐는지까지 확인한다 — 중간에 사용자에게 물어보지 않고서.

### [03:41–04:00] 에이전트 ① 플래너(planner)
새 기능을 만들기 전에 "이거 어떻게 만들면 좋을까" 하고 계획 자체를 짜는 역할이다. "사용자 기능 만들고 싶어"라고 하면 어떤 파일을 어떤 순서로 만들어야 하는지, 어디서 어디로 연결되는지, 예상되는 위험 요소는 뭔지까지 단계별로 정리해 준다. 이걸 보고 시작하면 중간에 방향이 흔들리는 일이 훨씬 줄어든다.
