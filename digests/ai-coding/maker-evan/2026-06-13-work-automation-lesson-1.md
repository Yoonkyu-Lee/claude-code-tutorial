## 요약 (주제별)

### 강의 목표와 전제
- 이 세션은 "업무 자동화" 섹션으로, 브라우저를 자동 제어해 특정 사이트를 크롤링하고 반복 업무를 자동화하는 법을 다룬다.
- 핵심 전제: **코드는 한 줄도 직접 짜지 않는다.** 사용자는 클로드 코드에게 한국어로 명령만 하고, 코드 작성은 클로드 코드가 한다. 사용자는 "무엇을 시킬지"만 잘 정하면 된다.
- 학습 후 목표 역량: 터미널·포트·IP·브라우저·CDP·프록시 개념을 비유로 이해, 클로드 코드에 자동화 작업을 명확히 지시, 평소 쓰던 크롬을 그대로 자동화에 연결, 사이트의 봇 감지 방식 이해 및 회피 전략 요청, 안티디텍트 브라우저·프록시 흐름을 머릿속에 그리기.
- 섹션 구성: 클로드 코드 소개 → 웹 자동화 기초 → 봇 디텍션 → 어드밴스드(안티디텍트·프록시).

### 클로드 코드란 / 작동 방식
- [의견] 화자는 클로드 코드를 "옆자리에 앉은 신입 비서", "터미널에서 만나는 AI 비서"로 비유한다. 개발자가 아니어도 되고, 터미널은 AI와 대화하는 창구일 뿐이라고 본다.
- 할 수 있는 일: 자료 조사(인터넷 검색→보고서), 문서 작성(보고서·이메일·기획서 등 구조화 문서), 브레인스토밍(다양한 관점 아이디어), 데이터 정리(고객 데이터 분류·평균 계산·Excel/CSV 저장).
- 3단계 작동 구조: ① 자연어로 요청(일상 한국어로 원하는 것 설명) → ② 클로드 코드가 알아서 분석·실행·검색(파일 분석, 명령어 실행, 인터넷 검색) → ③ 결과물을 사용자에게 보여주고 승인 요청("오케이" 한마디). 이 과정에서 사용자가 직접 코드를 볼 일은 거의 없다.
- 이 세션에 필요한 것은 딱 둘: **클로드 코드**(한국어 명령을 받아 코드를 작성·실행하는 AI 엔진)와 **파이썬**(클로드 코드가 짜주는 코드의 언어).

### 프롬프트 설계 (지시·맥락·작업·출력)
- 프롬프트 = "AI에게 주는 업무 지시서". 신입 비서에게 "알아서 잘해 줘"라고만 하면 엉뚱한 결과가 나오듯, 무엇을·어떤 환경에서·어떻게·어떤 형식으로 받고 싶은지까지 정리해야 한다.
- 좋은 프롬프트 예: "네이버 메일에서 계약서 검색해서 상위 열 건을 마크다운 보고서로 만들어 줘." 나쁜 프롬프트 예: "매일 자동화 해 줘"처럼 모호한 지시 → AI가 추측으로 제멋대로 동작.
- 좋은 프롬프트의 4요소: **지시**(무엇을 할지 명확히, 예: "Cafe24 어드민에서 어제 주문 긁어줘") / **맥락**(환경·전제, 예: "Windows 크롬 디버그 모드, 이미 로그인된 환경에서") / **작업**(구체적 단계, 예: "주문 페이지 진입→어제 필터→CSV 다운로드") / **출력**(산출물 형식 지정, 예: ".md로").
- 네이버 메일 자동화 프롬프트 실례: "9222번 포트에 크롬을 열어놨어. 이걸 이용해 특정 주소에 들어가 네이버 메일에서 '계약서' 메일들을 각각 요약해 보고서(.md)로 저장해 줘. 사람인 척 각 단계별로 2~10초 랜덤으로 쉬어 줘." → 맥락(9222 포트 크롬=이미 로그인된 디버그 모드), 지시(URL+키워드 '계약서' 검색), 작업(요약·저장), 사람 흉내(랜덤 대기 = 봇 디텍션 회피 의도)를 모두 담은 예시.

### 핵심 기술 용어 (비유)
- **IP**: 인터넷상 컴퓨터의 "집 주소". 데이터를 보내려면 받는 컴퓨터 주소가 필요. 예: `142.250.66.78`(구글 서버 중 하나), `192.168.0.1`. **localhost**는 외부에 안 알리고 내 컴퓨터 자신만 가리키는 별명. 이 강의에서는 `http://localhost:9222`를 "내 컴퓨터의 9222번 호실"로 사용.
- **포트**: 컴퓨터의 "호실 번호"(0~65535). IP(건물 주소)만으론 어느 사무실인지 모르니 호실 번호가 필요. 80·443·25·587·22 등은 이미 쓰이는 포트라 사용 불가, 이 강의는 **9222번 포트**를 크롬 자동화 전용으로 약속. 포트를 연다 = 크롬이 9222번으로 노크를 받을 수 있게 띄운 상태. 보안상 9222는 본인 PC만 접속 가능하게(외부 PC 차단) 설정 가능.
- **CDP (Chrome DevTools Protocol)**: 크롬의 "리모컨 규격". TV마다 리모컨이 다르면 불편하듯, "이 신호를 보내면 클릭·이동·스크롤한다"는 공식 규격. "어태치(attach)"란 새 창을 띄우지 않고 **이미 떠 있는 크롬에 붙어서 조작**하는 것. 안전한 이유: 이미 로그인된 세션을 그대로 써서 새 로그인·2FA·OTP를 건너뛴다.

### 실전 절차 (네이버 메일 케이스 스터디)
- 왜 진짜 크롬을 쓰나: 브라우저 자동화의 본질은 봇 디텍션 회피 = "진짜 사람으로 보이기". 사이트는 JavaScript 환경·TLS·행동·IP를 전부 검사하므로, 본인이 평소 쓰는 크롬·프로필·쿠키·로그인 그대로 빌려 쓰고 CDP 9222 포트로 연결한다.
- 케이스: 이미 로그인된 크롬에 어태치 → 2FA/OTP 우회, 새 로그인 불필요, 봇 디텍션 입장에서 진짜 사람처럼 인식.
- 5단계 시나리오: ① CDP 어태치(9222 호실에 노크) → ② 메일 진입(naver.com 메일→받은편지함) → ③ 키워드 검색('계약서' 엔터, 결과 로딩) → ④ 상위 N건 추출(발신자·제목·날짜·미리보기 4개 필드) → ⑤ 보고서 생성(`네이버메일리포트.md` 자동 저장). 키워드·도메인만 바꾸면 Gmail·아웃룩·그룹웨어 메일에도 동일 패턴 적용 가능.
- 재사용성: 한 번 스크립트를 만들면 키워드만 바꿔 재실행 가능, 매번 코드를 다시 짤 필요 없다. 스크립트 동작은 "브라우저에 붙고→메일 페이지 진입→정보 추출→보고서 저장".

### 봇 디텍션 4단계 (공항 입국 심사 비유)
- 사이트는 방문자를 "여권·소지품·걸음거리·평판" 네 단서로 검사해 점수를 매기고, 일정선을 넘으면 차단/캡차. 막는 이유: 가격 정보 무단 수집, 회원가입 자동화, 표 매크로, 디도스 등을 막아 사이트와 진짜 사용자를 보호. 대표 솔루션: Cloudflare, DataDome, PerimeterX, Akamai. 걸리면 캡차·IP 블랙리스트·심하면 계정 정지.
- **① 여권 검사 (TLS/HTTP 헤더)**: 누가 노크했는지 가장 먼저 보는 표지. TLS(Transport Layer Security) 핸드셰이크에서 브라우저가 신원 정보를 보내는데, 진짜 크롬인지 파이썬 requests인지 구분. 걸리는 패턴: 파이썬 requests, 빈 User-Agent, 크롬과 다른 헤더 순서. [의견] 화자는 "이 여권 검사는 우리 케이스에서는 딱히 고려 안 해도 된다"고 봄.
- **② 소지품 검사 (JavaScript)**: 페이지 로드 후 사이트가 JS로 브라우저를 몰래 살핌. `navigator.webdriver` 변수가 켜져 있으면 "자동화 도구구나" 즉시 인식. 해상도·폰트·플러그인·WebGL도 검사 대상. 이 흔적 회피가 매우 중요.
- **③ 걸음거리 (행동)**: 마우스 좌표·클릭 좌표·타이핑 간격을 기록. 봇은 직선 이동·일정한 0.1초 타이핑·클릭 후 0.5초 만에 다음 페이지 등 "너무 매끄럽고 빠름". 진짜 사람은 곡선 이동·잠깐 멈춤·오타. → 클로드에 "각 단계 2~10초 랜덤 대기, 마우스는 곡선으로 움직여" 요청 권장.
- **④ 평판 (IP·계정)**: 데이터센터(AWS/GCP/Azure) IP는 즉시 의심, 가정용(SK/KT/LG U+) residential IP는 정상 분류. 방금 가입·즉시 결제한 IP에서 여러 계정 운영 시 봇 판단. 회피: residential 프록시, sticky session.

### 도구·자동화 (안티디텍트 브라우저 / 프록시)
- **핑거프린트(브라우저 지문)**: 화면 해상도·설치 폰트·그래픽카드·시간대·언어·오디오 출력 등 수십 항목을 모으면 그 PC만의 고유 지문. 사이트가 지문을 저장하면 쿠키를 지우고 IP를 바꿔도 같은 사람으로 추적 가능 → 다중 계정·광고 추적 회피가 어려운 진짜 이유. 쿠키처럼 지울 수 있는 값이 아니라 PC에서 자동 생성되는 값이라 강력. 관리 대상: WebGL·Canvas·오디오·폰트·해상도·시간대·WebRTC·언어.
- **안티디텍트 브라우저 — Dolphin Anty**: Chromium 기반 멀티프로필 도구. WebGL·Canvas·오디오·폰트·타임존·WebRTC 핑거프린트를 프로필별로 관리·가짜 생성. 한 PC에서 멀티 페르소나/계정 동시 운영, OS·해상도·타임존·언어 자동 매칭, 프록시 엔드포인트 연결, 디버깅 포트 노출 후 Playwright/Puppeteer를 CDP로 어태치.
- **프록시**: "우편물을 대신 보내주는 대리인". 내 PC가 사이트에 직접 접속하지 않고 중계 서버를 거쳐, 사이트는 중계 서버 IP만 보고 진짜 IP는 모름. 데이터센터 프록시(AWS 등 IP, 빠르고 싸지만 의심받기 쉬움) vs residential 프록시(실제 가정집 IP, 진짜 사람으로 인식, 비싸지만 효과 좋음). 용도: 특정 국가 전용 사이트, 봇 디텍션 강한 사이트, 멀티 계정.
- **추천 프록시 (주장/미검증)**: ① **Oxylabs residential** — 1억 개 residential IP풀, 다수 국가, 기가당 4~15달러, sticky session 30분 동안 동일 IP 유지, 국가 선택(CC 옵션), 로그인 후 IP가 갑자기 바뀌면 봇 의심받으므로 sticky로 자연스럽게. ② **911 Proxy** — 저렴하지만 덜 안정적, 소규모용.
- **Dolphin+프록시 통합 워크플로우 4단계**: ① Dolphin 프로필 생성 → ② 프록시 엔드포인트 입력(sticky session) → ③ Dolphin 실행(디버깅 포트 노출) → ④ Playwright 어태치. OS·해상도·타임존·언어·핑거프린트 자동 매칭.

### 프로젝트 셋업 & 실습 데모
- 셋업: 화자가 준 **GitHub 주소**([불명확: 영상에서 URL 미언급])에서 Code→Download ZIP으로 `automation-main`을 받는다. Antigravity IDE 또는 VS Code(혹은 Claude Desktop 앱)로 열고, 새 터미널/새 세션에서 `@`(골뱅이) 버튼으로 스크립트 셋업을 지시하면 클로드 코드가 크롬 프로필을 하나 세팅해 준다. 기존에 쓰던 창 말고 새 크롬 창을 하나 켜면 접속됨.
- 데모: 화자가 직접 네이버에 로그인 → 실습 자료의 프롬프트로 "9222번 포트 크롬을 조작해 네이버 메일에서 '넥슨' 검색되는 결과를 처리하는 파이썬 스크립트 작성" 지시 → 클로드가 스크립트를 짜고 실행, 상위 결과를 검색해 보여줌.
- 한 단계 더: "각 메일 내용을 읽고 정리해 보고서로 작성, `넥슨.md`로 저장해 줘" → 클로드가 화면을 알아서 움직이며 각 메일을 열어 확인, 사람처럼 처리하며 보고서 작성. 로그인은 사용자가 미리 해두면 AI가 로그인할 필요가 없어 이슈 없음. 결과: 메일 15개·그룹 정보를 담은 `넥슨.md` 생성.
- 마무리: [의견] 화자는 "코드는 내가 다 미리 만들어 놨으니 여러분은 GitHub에서 다운로드해 '해 줘'라고만 하면 된다"고 안내.

유튜버 "메이커 에반"이 오늘 보여줄 것은 업무 자동화 섹션이라고 인사한다. 이 섹션은 브라우저를 자동으로 컨트롤하고 특정 사이트를 크롤링해, 그걸 기반으로 본인의 루틴한 업무들을 자동으로 처리하는 법을 설명한다. [의견] 이 세션을 통해 시청자가 자기만의 루틴 업무를 자동화할 환경을 만들었으면 좋겠다고 밝힌다.

### [00:37–01:33] 학습 목표 — 코드는 한 줄도 안 짠다
이 강의가 끝나면 할 수 있게 될 것들을 나열한다: 기본 기술 용어(터미널·포트·IP·브라우저·CDP·프록시)를 비유로 이해, 클로드 코드에 자동화 작업을 프롬프트로 명확히 지시, 평소 쓰는 크롬을 그대로 자동화에 연결, 사이트가 봇을 어떻게 감지하는지 알고 회피 전략을 요청, 안티디텍트 브라우저와 프록시 흐름을 머릿속에 그리기. 핵심 전제를 강조한다.
> "이 과정 중에서 저희는 코드는 한 줄도 직접 짜지 않을 겁니다. 우리는 클로드 코드에게 한국어로 명령만 합니다. 코드는 클로드 코드가 짭니다. 사용자는 무엇을 시킬지만 잘 정하면 됩니다." [01:20]

### [01:35–01:49] 섹션 구성
강의 구성을 안내한다: 클로드 코드 간단 소개 → 웹 자동화 기초 → 봇 디텍션(bot detection) → 어드밴스드 과정 순서로 설명할 예정.

### [01:49–02:43] 클로드 코드란 — 옆자리 신입 비서
클로드 코드가 무엇인지 비유로 설명한다.
> "클로드 코드란 옆자리에 앉은 비서입니다. 터미널에서 만나는 AI 비서입니다." [01:59]
할 수 있는 일로 자료 조사(인터넷 검색→정보 정리→보고서), 문서 작성(보고서·이메일·기획서 등 구조화 문서 자동 작성), 브레인스토밍(다양한 관점 아이디어 제안·발전), 데이터 정리(고객 데이터 분류·평균값 계산·Excel/CSV 저장)를 든다. [의견] 개발자가 아니어도 괜찮고, 터미널은 그저 AI와 대화하는 창구일 뿐이며, 사무실 옆자리에 무엇이든 도와주는 신입 비서가 있다고 상상하면 좋다고 한다.

### [02:46–03:23] 클로드 코드의 3단계 작동 구조
"제가 한국어로 말하면 클로드가 알아서 실행하고 결과를 내가 확인하는 구조"라며 3단계로 안내한다. ① 자연어 요청 — 일상 한국어로 원하는 것을 설명. ② 분석·실행·검색 — 클로드 코드가 알아서 파일을 분석하고 명령어를 실행하며 인터넷도 검색. ③ 승인 — 결과물을 만들어 사용자에게 보여주고 승인 요청, 괜찮으면 "오케이" 한마디면 됨. 이 과정에서 사용자가 직접 코드를 보거나 짤 일은 없다고 본다.

### [03:23–03:49] 필요한 두 가지 — 클로드 코드 + 파이썬
모든 세션에 필요한 것은 딱 두 가지: 클로드 코드(사용자의 한국어 명령을 받아 코드를 작성·실행하는 AI 엔진)와 파이썬(클로드 코드가 짜주는 코드의 언어 종류). 컴퓨터가 알아먹는 언어(파이썬)를 이용해 작업을 지시할 예정이라고 정리한다.

### [03:49–04:20] 용어 — 프롬프트란
프롬프트를 "AI에게 주는 업무 지시서"로 설명한다. 신입 비서에게 "이거 알아서 잘해 줘"라고만 하면 엉뚱한 결과가 나오기 쉽다. 프롬프트는 무엇을·어떤 환경에서·어떻게·어떤 형식으로 받고 싶은지까지 정리해 주는 지시서이며, AI에게 입력하는 명령 문장이다.

### [04:20–05:00] 좋은/나쁜 프롬프트 예시
좋은 프롬프트 예: "네이버 메일에서 계약서 검색해서 상위 열 건을 마크다운 보고서로 만들어 줘." 나쁜 프롬프트 예: "매일 자동화 해 줘"처럼 모호한 지시. 후자는 AI가 무엇을 할지 추측해야 하고, 그 추측을 기반으로 동작하므로 제멋대로 행동할 확률이 높다. 프롬프트는 사람이 일상에서 쓰는 자연어(한국어·영어 등)를 기반으로 한다.

### [05:00–05:52] 좋은 프롬프트의 4요소 — 지시·맥락·작업·출력
좋은 프롬프트를 네 부분으로 구분한다. **지시**: 무엇을 할지 명확히(예: "Cafe24 어드민에서 어제 주문 긁어줘"). **맥락**: 환경과 전제를 확실히(예: "Windows 크롬 디버그 모드, 이미 로그인된 환경에서 해줘"). **작업**: 구체적 단계(예: "주문 페이지 들어가서 어제 필터, CSV 다운로드"). **출력**: 산출물 형식 지정(예: ".md처럼"). 형식을 지정해 줘야 AI가 헷갈리지 않고 정확한 산출물을 만든다.

### [05:55–07:17] 네이버 메일 자동화 프롬프트 실례 분석
실제 프롬프트 한 컷을 보여준다. 한국어로 한 문단처럼 적어도 되며 핵심은 빠진 정보가 없어야 한다.
> "내가 크롬 브라우저를 9222번 포트에 열어 놨어. 이걸 이용해서 어떤 특정 주소에 들어가서 네이버 메일에서 '계약서'라고 적힌 메일들을 각각 요약해서 보고서로 작성해 줘. .md로 저장하면 돼. 이때 사람인 척하기 위해 각 단계별로 2에서 10초 정도 랜덤으로 쉬어 줘." [06:16]
이를 4요소로 해부한다: 맥락 = 9222번 포트 크롬(이미 로그인된 디버그 모드 가정), 지시 = URL과 키워드('계약서' 검색)를 명확히, 작업 = 각 메일 요약과 마크다운 보고서 저장, 그리고 사람인 척 각 단계 2~10초 랜덤 대기(봇 디텍션 회피 의도).

### [07:17–08:37] 브라우저 자동화 — 왜 진짜 크롬을 쓰나
이제 브라우저 자동화로 들어간다. 이 자동화에서는 진짜 크롬 브라우저를 사용한다.
> "브라우저 자동화의 본질은 봇 디텍션 회피입니다. 진짜 사람으로 보이는 게 핵심이죠." [07:54]
사이트는 JavaScript 환경·TLS·행동·IP를 전부 검사하므로, 진짜 크롬으로 위장해야 한다. 본인이 평소 쓰는 크롬·프로필·쿠키·로그인을 그대로 자동화에 빌려 쓰고, CDP 9222번 포트로 연결한다. CDP(Chrome DevTools Protocol)는 진짜 브라우저를 외부에서 조작하는 "리모컨"이라고 소개하며, 용어들이 어려울 테니 하나씩 천천히 설명하겠다고 예고한다.

### [08:46–09:51] 용어 — IP (컴퓨터의 집 주소)
IP는 인터넷상 컴퓨터의 집 주소다. 택배를 보내려면 받는 사람의 집 주소가 필요하듯, 인터넷에서 데이터를 보내려면 받는 컴퓨터의 집 주소(IP)가 필요하다. 예로 `142.250.66.78`(구글의 한 서버), `192.168.0.1`을 든다. localhost는 외부에 안 알리고 내 컴퓨터 안에서만 통하는 별명("우리 집"). 오늘은 `http://localhost:9222` 주소를 쓸 텐데, "내 컴퓨터의 9222번 호실"이라고 이해하면 된다. 요점: "IP는 컴퓨터의 집 주소다."

### [09:51–11:01] 용어 — 포트 (컴퓨터의 호실 번호)
포트는 컴퓨터의 호실 번호다. 한 건물(IP)에 여러 사무실이 있으면 건물 주소만으론 어느 사무실인지 모르니 호실 번호가 필요하다. 예: 김변호사=305호, 박회계사=410호, 9222번=크롬 자동화 전용 사무실로 약속한 번호. 포트 번호는 0~65535 사이 숫자로 프로그램끼리 안 겹치게 쓰는 문 번호다. 80·443·25·587·22 등은 이미 쓰이는 포트라 사용 불가라서, 미리 약속한 9222번 포트를 쓴다. "포트를 연다"는 것은 크롬을 9222번으로 들을 수 있게 띄운 상태이며, 외부 도구가 노크할 수 있다. 보안상 9222번은 본인 PC만 들어올 수 있게(외부 PC 차단) 설정 가능하다.

### [11:01–11:50] 용어 — CDP (크롬의 리모컨 규격)
CDP는 Chrome DevTools Protocol, 크롬의 "리모컨 규격"이다. TV마다 리모컨이 다르면 불편하듯, "이 신호를 보내면 채널이 바뀐다"는 약속처럼 CDP는 "크롬에 이런 형식으로 신호를 보내면 페이지를 클릭·이동·스크롤한다"는 공식 규격이다. "어태치(attach)"란 이미 떠 있는 크롬에 붙어서 조작하는 행위로, 새 창을 띄우지 않는다. 안전한 이유는 이미 로그인된 세션을 그대로 사용하기 때문 — 새 로그인·2FA·OTP를 건너뛴다.

### [11:50–12:18] 프로젝트 셋업 예고 (GitHub)
화자가 주는 GitHub 주소로 들어가 특정 파일을 다운로드하고, 아래 스크립트로 "크롬 프로필을 세팅해 달라"고 명령하면 된다. 그러면 클로드가 브라우저를 셋업해 주고, 이를 기반으로 업무 자동화가 가능해진다.

### [12:18–13:23] 케이스 스터디 — 5단계 시나리오
이미 로그인된 크롬을 그대로 어태치해 2FA/OTP를 우회하는 케이스를 다룬다. 이미 로그인된 세션을 그대로 써서 새 로그인·2FA·OTP가 불필요하고, 봇 디텍션 입장에서 진짜 사람처럼 인식된다. 시나리오 5단계: ① CDP 어태치 — 이미 떠 있는 크롬의 9222번 호실에 자동화 도구가 노크. ② 메일 진입 — naver.com 메일 링크 들어가 받은편지함 진입. ③ 키워드 검색 — 검색창에 '계약서' 엔터, 결과 로딩. ④ 상위 N건 추출 — 발신자·제목·날짜·미리보기 4개 필드 자동 수집. ⑤ 보고서 생성 — `네이버메일리포트.md` 마크다운 자동 작성·저장. 동일 패턴으로 Gmail·아웃룩 웹·그룹웨어 메일도 키워드와 도메인만 바꾸면 가능하다.

### [13:23–14:35] 예시 프롬프트 + 스크립트 이해와 재사용
예시 프롬프트: "9222번 포트에 크롬 브라우저를 켰어. 이 브라우저를 조작해서 여기 주소에 들어가 메일 검색에 '넥슨'이라고 치고, 검색되는 결과를 (처리)하는 파이썬 스크립트 작성해 줘." 맥락을 자동 인식하게 하고, URL·키워드를 명확히, 출력을 명시적으로 해야 동작한다. 클로드가 만드는 스크립트는 "브라우저에 붙고→메일 페이지 진입→정보 추출→보고서 저장" 흐름으로 이해하면 된다.
> "키워드만 바꿔서 다시 실행 가능합니다. 매번 새 코드를 짜지 않으셔도 됩니다. 한 번 만들면 평생 재사용이 가능합니다." [14:03]
이미 로그인된 실제 크롬에 어태치하므로 새 로그인 시도가 불필요하다. 메일 진입은 지정 URL 이동→검색창에 키워드 입력→결과 로딩 대기, 정보 추출은 발신자·제목·날짜·미리보기 4개 필드 수집, 보고서 저장은 마크다운 파일로 자동 변환하며 날짜·키워드로 자동 명명한다.

### [14:35–15:44] 파트 3 — 봇 디텍션 개론
파트 3에서는 웹사이트가 봇을 어떻게 감지하는지 개론적으로 설명한다. 봇 디텍션은 사이트의 보안 검사대다. 공항 입국 심사를 떠올리라고 한다 — 여권·표정·동선을 종합해 들여보낼지 판단하듯, 사이트도 방문자를 검사한다. 네 가지 단서를 합산해 점수를 매기고, 점수가 일정선을 넘으면 차단/캡차가 발동된다. 왜 막는가: 가격 정보 무단 수집, 회원가입 자동화, 표 매크로, 디도스 등을 막아 사이트와 진짜 사용자를 보호하기 위함. 봇은 사람이 아니라 프로그램이 자동으로 클릭·요청을 보내는 행위다. 대표 솔루션으로 Cloudflare, DataDome, PerimeterX, Akamai 등을 든다. 걸리면 캡차·IP 블랙리스트, 심한 경우 계정 정지까지 당할 수 있다. 목표는 진짜 사람처럼 행동하는 크롬 브라우저로 이를 우회하는 것.

### [15:44–17:17] ① 여권 검사 — TLS / HTTP 헤더
네 단계 검사를 공항 입국 심사(여권·소지품·걸음거리·평판)에 대응시켜 설명한다. 첫째, 여권 검사: 사이트가 "누가 노크했는지" 가장 먼저 보는 표지로, 진짜 크롬인지 파이썬 requests인지 확인한다. TLS(Transport Layer Security) 핸드셰이크는 "첫 인사 때 내미는 여권"으로, 사이트와 처음 연결될 때 브라우저가 자기 신원 정보를 보낸다(인터넷 통신을 암호화하는 약속). 걸리는 패턴: 파이썬 requests, 빈 User-Agent, 크롬과 다른 헤더 순서. HTTP 헤더도 빡세게 검사하는데 "요청에 따라붙는 명함"으로 User-Agent·Accept-Language 등을 확인한다. [의견] 다만 이 여권 검사는 "우리 케이스에서는 딱히 고려 안 해도 될 것 같다"고 화자는 본다.

### [17:17–18:20] ② 소지품 검사 — JavaScript / navigator.webdriver
둘째, 소지품 검사: 사이트가 페이지 로드 후 슬쩍 "가방을 열어보는" 단계로, 자동화 흔적이 묻어 있는지 체크한다. 여권 통과 후 짐 검사대로 가듯, 사이트도 페이지를 띄울 때 JavaScript 코드를 몰래 실행해 브라우저 내부를 살핀다. `navigator.webdriver` 변수가 켜져 있으면 "자동화 도구가 켰구나"라고 즉시 알아챈다. 해상도·폰트·플러그인·WebGL 같은 것도 검사 대상. JavaScript는 사이트가 사용자 브라우저에서 실행하는 코드이며, 특정 변수(자동화 흔적)가 켜져 있으면 런타임에 사이트가 이 신호를 감지해 봇으로 판단·차단한다. 이 회피가 매우 중요하다.

### [18:20–19:15] ③ 걸음거리 — 마우스·타이핑 행동 패턴
셋째, 걸음거리: 입국 후 걸음걸이 관찰처럼, 여권 통과 후에도 부자연스럽게 똑같은 복장·식은땀·서두름이 단서가 된다. 사이트도 마우스 좌표·클릭 좌표·타이핑 간격을 기록한다. 봇은 보통 직선 이동, 일정한 0.1초 타이핑, 클릭 후 0.5초 만에 다음 페이지 등 "너무 매끄럽고 너무 빠르다". 진짜 사람은 곡선으로 움직이고, 잠깐 멈추고, 오타도 친다. 이런 봇 신호를 인지해 밴을 먹이기도 하므로 사람다운 액션을 넣어야 한다. → 클로드에 요청할 때 "각 단계별 2~10초 랜덤 대기, 마우스는 곡선으로 움직여"를 요청하는 게 좋다.

### [19:15–20:20] ④ 평판 — IP·계정 이력
넷째, IP·계정 = 입국자의 평판 조회. 여권만 봐도 위반 이력을 알 수 있듯 사이트도 비슷하다. 데이터센터에서 들어온 트래픽은 기업 봇이 만든 IP라 즉시 의심하고, 가정용 인터넷 IP는 진짜 사람으로 인식한다. 데이터센터 IP는 AWS·GCP·Azure 같은 알려진 IP 대역이라 즉시 의심 가능. 반면 residential IP(가정용, SK·KT·LG U+ 같은 회선)는 사이트가 정상으로 분류한다. 계정 이력도 중요 — 방금 가입 혹은 즉시 결제한 IP에서 여러 계정을 운영하면 봇으로 판단해 이상하게 처리한다. 회피법은 residential 프록시나 sticky session을 이용하는 것.

### [20:20–21:29] 핑거프린트 — 브라우저의 지문
넷째 단계까지 봇 디텍션을 알았으니, 다음(어드밴스드)에서는 안티디텍트 브라우저와 프록시로 우회하는 법을 설명한다. 먼저 핑거프린트(브라우저의 지문): 사람 지문이 다 다르듯 각 PC의 브라우저도 미세하게 다르다. 화면 해상도·설치 폰트·그래픽카드 정보·시간대·언어·오디오 출력 특성 등 수십 항목을 모으면 그 PC만의 고유 지문이 만들어진다. 사이트가 이 지문을 저장하면 쿠키를 지우고 IP를 바꿔도 같은 사람으로 추적 가능 — 다중 계정 운영이나 광고 추적 회피가 어려운 진짜 이유다. 관리 대상: WebGL·Canvas·오디오·폰트·해상도·시간대·WebRTC·언어. 강력한 이유는 쿠키처럼 지울 수 있는 값이 아니라 PC 자체에서 자동 생성되는 값이기 때문.

### [21:29–22:29] 안티디텍트 브라우저 — Dolphin Anty
핑거프린트 회피를 위해 안티디텍트 브라우저를 쓴다. 대표적으로 Dolphin Anty. Chromium 기반 멀티프로필 도구로, 핑거프린트 자체를 프로필별로 관리하고 가짜로 생성한다(WebGL·Canvas·오디오·폰트·타임존·WebRTC). 쓰는 이유: 한 PC에서 멀티 페르소나/멀티 계정 동시 운영, 프로필 자동 매칭(OS·해상도·타임존·언어), 프록시 엔드포인트 연결까지. 자동화는 Dolphin의 디버깅 포트 노출을 통해 Playwright·Puppeteer를 CDP로 어태치한다. 이 도구를 쓰면 핑거프린트 및 프록시로 간단히 우회 가능하다.

### [22:29–23:21] 프록시 — 우편물 대리인
프록시란 "우편물을 대신 보내주는 대리인"이다. 내 집 주소를 노출하기 싫으면 친구 집 주소를 빌려 거기서 발송하면 받는 사람은 친구 집 주소만 본다. 프록시도 똑같이 내 PC가 사이트에 직접 접속하지 않고 중계 서버를 거치므로, 사이트는 중계 서버 IP만 보고 진짜 IP는 모른다. residential 프록시는 그 "친구 집"이 데이터센터가 아닌 일반 가정집이라는 뜻이다. 데이터센터 프록시(AWS 등 IP)는 빠르고 싸지만 봇으로 의심받기 쉽고, residential은 실제 가정집 IP라 진짜 사람으로 인식되며 비싸지만 효과가 좋다. 쓰임: 특정 국가에서만 보이는 사이트, 봇 디텍션 강한 사이트, 멀티 계정.

### [23:21–24:18] 추천 프록시 ① — Oxylabs residential (주장/미검증)
일반적으로 많이 쓰이는 프록시 두 가지를 추천한다. 첫째, Oxylabs의 residential 프록시. 1억 개 residential IP풀이 있고(주장/미검증), 국가가 매우 많으며, 가격대는 기가당 4~15달러(주장/미검증), sticky session은 30분 동안 동일 IP가 유지된다(주장/미검증). 진짜 가정 IP로 위장하고 국가를 선택 가능(CC 옵션). 데이터센터 IP가 아니라 실제 가정집 IP로 위장하므로 사이트는 평범한 사용자로 인식한다. 한국 IP가 필요한 사이트에도 해외 PC로 접근 가능. sticky session으로 같은 IP를 30분 유지 가능 — 로그인 후 IP가 갑자기 바뀌면 봇이 의심하므로 sticky로 자연스럽게 사용 흉내를 낸다.

### [24:18–24:57] 추천 프록시 ② — 911 Proxy + Dolphin 통합 워크플로우
둘째, 911 Proxy. 저렴하지만 덜 안정적인 친구로, 소규모에서 쓰는 걸 추천한다. 이어 Dolphin+프록시 통합 워크플로우를 설명한다: 프로필 핑거프린트를 별도로 만들고, 프록시를 달고, Playwright 어태치로 4단계 흐름을 만든다. ① Dolphin 프로필 생성 → ② 프록시 엔드포인트 입력(sticky session 처리) → ③ Dolphin 실행(디버깅 포트 노출) → ④ Playwright 어태치. OS·해상도·타임존·언어·핑거프린트가 자동 매칭된다. 이렇게 하면 이후로는 손쉽게 활용 가능하다.

### [24:57–25:36] 강의 마무리
오늘 강의는 여기서 마무리한다. 이제 화자가 준 GitHub 코드를 기반으로 실제 실습해서 네이버 메일을 컨트롤하는 업무를 해보는 시간을 갖겠다며 감사 인사를 전한다.

### [25:36–27:03] 프로젝트 셋업 실습
프로젝트 세팅 방법을 보여준다. 화자가 준 GitHub 주소로 들어가 Code→Download ZIP으로 `automation-main`을 받는다. 그다음 Antigravity IDE나 VS Code 같은 도구로 여는 걸 추천한다. 새 창을 켜서 저장한 `automation-main`을 열고, 터미널이나 VS Code 익스텐션에서 New Session으로 연다. 먼저 셋업을 해야 하는데, `@`(골뱅이) 버튼을 눌러 스크립트 셋업을 지시하면 클로드가 셋업해 준다. 그러면 클로드가 크롬 프로필을 하나 세팅해 주고, 기존에 쓰던 것 말고 새 창을 하나 켜면 이런 식으로 접속된다.

### [27:03–29:11] 네이버 메일 실습 — '넥슨' 검색
실습으로 네이버에 들어가 네이버 메일을 검색해 본다. 화자가 실제로 로그인을 하고, 실습 자료에 있는 프롬프트(9222번 포트 크롬 관련)를 그대로 준다. 그러면 클로드가 스크립트를 작성하고, 그걸 기반으로 처리해 다운로드까지 알아서 한다. '넥슨'이라는 키워드를 네이버 메일에서 검색해 처리하는 스크립트를 작성하고, 상위 결과를 검색해 보여준다. 한 단계 더 나아가 "각각의 메일에 대해 내용을 읽고 정리해서 보고서로 작성해 줘, `넥슨.md`로 저장해 줘"라고 명령하면, 클로드가 각 메일을 읽고 내용을 정리해 보고서로 만든다. 클로드가 화면을 알아서 움직이며 각 메일을 열어 확인하는 것을 볼 수 있다. 사람처럼 움직이기 위해 각 처리를 하고 있고, 로그인은 사용자가 미리 해두면 AI가 따로 로그인할 필요가 없어 이슈가 없다.

### [29:11–30:34] 보고서 생성 확인 및 마무리
`넥슨.md` 파일이 생성됐는데, 메일 수가 15개 있고 그룹이 있다고 한다. 각각의 내용을 이런 식으로 정리·처리한 것을 확인할 수 있다. 이렇게 브라우저 자동화·업무 자동화가 가능하다. [의견] 화자는 "이런 코드를 미리 다 만들어 놨으니 여러분은 들어가서 그냥 '해 줘'라고만 하면 된다"고 안내한다. 방법은 준 GitHub 주소로 들어가 다운로드받고 ZIP을 기반으로 실행을 요청하는 것. Antigravity IDE에서도, Claude Desktop 앱을 통해서도 할 수 있으니 편한 걸 쓰면 된다. 감사 인사로 마무리.
