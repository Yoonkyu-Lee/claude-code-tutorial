# [시즌 3] 두 소 중 어떤 소가 더 일을 잘하냐? 두 에이전트로 랜딩 페이지 만들기 + 로티 스킬

> 정리 방식: 시간순 심층 워크스루 (paraphrase) + 주제별 상세 요약
> 검증 배너: 도구 성능 비교·작업 시간·토큰 사용률 등 수치는 화자의 1회 실습 관찰(주장/미검증). 승패 판정은 화자 주관 평가.

## 요약 (주제별)

### 실험 목적 · 관전 포인트
- 커뮤니티에서 "코덱스(Codex)가 일을 잘하냐, 클로드 코드(Claude Code)가 일을 잘하냐"는 질문이 많아 기획한 콘텐츠. 화자는 두 도구 차이를 "갤럭시냐 아이폰이냐" 수준의 취향/유행 차이로 보며, 압도적 우열은 없다고 본다 [의견].
- 동일 프롬프트를 두 에이전트에 동시에 넣어 결과물 차이를 확인하는 것이 1차 목표. 부차 목표는 (1) 로티(Lottie) 스킬 소개, (2) 데이터베이스 없이 스프레드시트로 랜딩 페이지 운영하는 법.
- 결론(화자 판정): 이번 랜딩 페이지 작업에서는 클로드 코드 쪽(자막상 "페이블 5" 모델)이 코덱스(GPT-5.5)보다 확실히 잘 만들었다며 클로드 코드에 승점 1점.

### 비교 셋업 (공정성 통제)
- C 드라이브에 프로젝트 폴더 2개 생성(예: `study-codex-lottie-site-1`, `...-2`), 화면 분할로 한쪽은 코덱스, 다른 쪽은 클로드 코드 실행.
- 양쪽에 동일한 `design.md`(디자인 가이드)를 넣고, 동일한 로티 스킬을 세팅하고, 동일한 스프레드시트 데이터를 소스로 줘서 조건을 맞춤.
- 모델: 코덱스 = GPT-5.5 미디움(패스트), 클로드 코드 = "페이블 5"를 미디움 에포트로 낮춰 공정하게 맞춤. [불명확: 자막 "페이블 5" — 클로드 코드에서 쓴 모델명, 정확한 표기 확인 불가]
- 코덱스의 이점 하나: 이미지 생성이 가능. 클로드 코드는 이미지 생성을 못 하므로 도서 표지 이미지는 사람이 직접 제공(스프레드시트에 URL로).

### 로티(Lottie) / 텍스트-투-로티 스킬
- 로티: 애플 로고가 멋지게 그려지듯 벡터 그래픽을 애니메이션으로 표현하는 기술. 화자도 로티 전문가는 아니고 쓰레드에서 최근 발견.
- 텍스트-투-로티(text-to-lottie) 스킬: GitHub 레포로 제공되며, 정지된 SVG를 넘기면 그것을 모션 애니메이션으로 표현해 준다. 설치법 + 프롬프트 가이드가 레포에 문서화됨(사람이 아니라 에이전트가 읽고 세팅).
- SVG 보충 설명: 확대·축소해도 깨지지 않는 벡터 이미지 형식. 태그/좌표 기반으로 그림을 그림. 에이전트에게 "그림 그려봐" 하면 SVG로 그리는 경우가 이것. 나노바나(nano banana) 같은 이미지 생성 툴과는 다른 방식. [불명확: 자막 "덕테이프" — 나노바나와 함께 언급된 다른 이미지 생성 툴명, 확인 불가]
- 프롬프트 가이드에는 "SVG를 제공하라", "모션 디자인 용어를 써라(카메라 오퍼레이터처럼 사고)" 같은 전문가용 지시가 있음. 비전문가는 이런 프롬프트를 못 짜므로, 이를 대신 생성해 주는 **메타프롬프트 스킬**("루티 프롬프트 라이터")을 하나 더 만들어 양쪽에 세팅.

### 스프레드시트를 DB로 쓰기 (Supabase 안 쓰는 이유)
- 대상: 이제 막 시작한 1인(소규모) 출판사(화자 선배가 운영, 실제 시프트/시프트북이라는 출판사 언급). 하루 수십만 방문이 아닌 사이트에는 Supabase 같은 데이터베이스가 과도하다 [의견]("집에서 요리하는데 몇천만 원짜리 칼 필요 없다"는 비유).
- 그렇다고 DB가 아예 없으면 상품 추가 때마다 재코딩·재배포가 불편. 해법: 약속된 형식의 **구글 스프레드시트**를 DB 삼아 전체 공개로 두고, 사이트가 그것을 읽어 상품을 나열.
- 스프레드시트 컬럼 예: 이미지(URL), 제목, 저자, 페이지(쪽수), 가격, 베스트셀러 유무(true/false), 신간 여부(is new, true/false), 설명(description). 도서 표지는 출판사 사이트의 이미지 주소를 복사해 URL로 삽입.
- 실습에서는 시간 절약을 위해 도서 3종만 대강 입력(원래는 에이전트에게 조사시켜 채워도 됨). 스프레드시트 입력 중 줄바꿈이 의도치 않게 들어가는 사소한 불편 언급.
- 장점(화자 주장): 이후 스프레드시트에 행만 추가하면 사이트가 알아서 읽어 즉시 반영 → 별도 DB 불필요. 보안 측면도 강조 [의견]: 구글 계정 비밀번호를 뚫지 않는 한 외부인이 데이터를 수정할 수 없어 "해킹 아무도 못 한다"고 표현(주장/미검증). 스프레드시트는 동시 수천 명이 읽어도 문제없는 튼튼한 백엔드라고 주장(주장/미검증).

### 메인 프롬프트 (사이트 요구사항)
- `design.md` 준수 + 1인 출판사 도서 상품 소개 랜딩 페이지 + 도서 상세(카탈로그) + 출판사 투고 유도 페이지가 있는 사이트를 **Next.js 기반**으로 제작.
- 데이터 소스: 공개된 구글 스프레드시트를 읽어 사이트에 적용. **1분 간격으로 스프레드시트를 읽어 업데이트**(행 추가 시 사이트에 즉시 반영)하는 기능 요구.
- 로티: 텍스트-투-로티 스킬로 출판사명(자막상 "골든 레드 레빗"/레드 레빗, Red Rabbit) 타이틀을 멋지게 표현. 단, 텍스트-투-로티를 쓰기 전에 먼저 메타프롬프트 스킬("루티 프롬프트 라이터")로 프롬프트를 생성한 뒤 사용하도록 지시.
- 디자인 제약: 색상 토큰은 한 테마당 최대 4개, 테마는 4개 중 택1 가능하게(총 16색). 한글 폰트는 반드시 Pretendard(프리텐다드).

### 결과 비교 · 판정
- 작업 시간: 코덱스 6분 47초 완료, "페이블 5" 15분 34초 완료(주장/미검증 — 1회 실측).
- 화자 평가: 코덱스 결과는 히어로 섹션이 엉망이라 "사이트라고 보기 어렵다", 로티도 엉망. 페이블 쪽은 카탈로그·출판사 소개·색상 테마 전환·로티가 비교적 깔끔("선녀 같다"). 두 결과 공통 문제: 도서 표지가 곧바로 안 보임.
- 원인 분석: 화자는 처음 준 `design.md`(드라이브 캐피털 참조)가 도서 출판사에 부적합했다고 반성 [의견]. 코덱스는 그 부적합한 design.md를 충실히 따르려다 더 불리했을 것으로 추정. 이후 도서 카탈로그에 적합한 design.md로 교체해 양쪽에 "다시 읽고 재구성해서 다시 만들라"고 재지시.
- 재작업 후에도 페이블 쪽이 훨씬 낫다고 인정. 화자는 언더독(코덱스)을 응원하는 편이지만 이번 코덱스 결과는 "도저히 사이트로 쓸 수 없을 것 같다"며 페이블 5에 승점.
- 교훈(화자 정리): (1) 디자인은 용도에 적합한 design.md를 주는 것이 중요, (2) 스프레드시트만으로도 충분히 좋은 DB 운용 가능.

### 도구·비용 · 도메인 (부가)
- 토큰/비용: 화자는 Max $100 요금제 사용, 이 사이트 작업에 페이블 사용량 약 42% 소모(주장/미검증). "페이블 5 있을 때 빨리 만들어 보라"는 권유.
- 도메인 연결: Vercel(버셀)에 배포하면 주소가 `.vercel.app`으로 끝나 아쉬움. 가비아(Gabia) 등에서 원하는 도메인 구매 → Vercel 프로젝트의 Domains에서 에디트 → 가비아의 DNS 값 보통 2개 수정(에이전트와 함께 진행). 실제 운영 예: `에디터.co.kr`, `시프트북.co.kr`. 상세 절차는 촬영 길이상 별도 영상으로 예고.

- 곁들여 (1) 정지 SVG를 애니메이션으로 바꿔주는 로티(text-to-lottie) 스킬 + 그 프롬프트를 대신 짜주는 메타프롬프트 스킬, (2) Supabase 대신 공개 구글 스프레드시트를 DB로 쓰는 저비용 운영법을 소개.
- 결과: 화자 판정으로 클로드 코드 쪽 완승(코덱스는 "사이트로 쓰기 어렵다"). 디자인은 용도에 맞는 design.md를 주는 게 핵심이라는 교훈으로 마무리.

## 전문 — 시간순 워크스루

### [00:00–01:15] 오늘 만들 사이트 소개와 관전 포인트
완성될 사이트를 먼저 보여준다. 진입 시 포인트가 되는 로티 애니메이션이 있고, 내려가면 판매 도서·베스트셀러 섹션, "출간 제안하기" 버튼, 깔끔한 푸터, 테마(색) 전환 기능이 있다. 카탈로그에서는 전체 도서를 필터링해 보고 출판사 소개도 확인할 수 있다. 오늘의 핵심은 **코덱스와 클로드 코드에 동일 프롬프트를 넣어 결과물이 얼마나 다르게 나오는지** 보여주는 것.
> "코덱스가 일을 잘하냐? 클로드 코드가 일을 잘하냐. 제가 느끼기에는 그냥 뭐 갤럭시 폰이 좋냐, 아이폰이 좋냐, 뭐 이거 물어보는 거 같은데" [00:53] [의견]

화자는 두 도구가 유행 따라 조금씩 달라질 뿐 압도적 우열은 없다고 느끼며, 실제로 같이 시켜보며 정말 차이가 있는지 확인하려 한다 [의견].

### [01:15–03:00] 로티 소개 예고 · 스프레드시트 떡밥
도입부에 서체가 예쁘게 그려지는 것이 "로티(Lottie)"인데, 화자는 로티를 잘 아는 사람은 아니고 최근 쓰레드에서 텍스트-투-로티 스킬을 발견해 알게 됐다고 밝힌다. 이런 애니메이션이 들어가면 일반 사이트보다 포인트를 잡아줘 괜찮다는 소감 [의견]. 두 에이전트에 같은 프롬프트를 동시에 넘겨 랜딩 페이지를 만들고, **데이터베이스를 쓰지 않는 것**이 오늘의 또 다른 포인트라고 예고한다.

### [02:00–03:00] "DB 없이 스프레드시트로" 예고
도서 데이터를 유동적으로 관리하면서도 공짜로 운영하는 법을 소개하는 것이 목적. 대상인 '시프트'라는 출판사는 이제 막 시작한 1인(직원 1명 합류 예정) 소규모 출판사로 화자 선배가 운영. 이런 규모는 비싼 Supabase 같은 DB를 도입하지 않아도 충분히 만들 수 있고 보안도 쉽게 챙길 수 있다며, 오늘은 **스프레드시트**를 쓸 것이라고 밝힌다. "비밀이 스프레드시트"라는 것. 로티 스킬 소개도 함께.

### [03:00–05:20] 텍스트-투-로티 스킬과 SVG 설명
텍스트-투-로티 스킬을 쓰레드에서 발견해 읽어보니, 애플 로고가 멋지게 그려지듯 벡터 그래픽을 애니메이션으로 만들어 주는 기술이 로티다. 어떤 사람이 이를 에이전트가 쓸 수 있게 스킬로 제공했고, GitHub 주소가 있으며 설치법·프롬프트 가이드가 문서화돼 있다. 이 문서는 사람이 아니라 에이전트가 읽고 세팅하면 된다.

이어 SVG를 설명한다. SVG는 확대·축소해도 깨지지 않는 벡터 이미지 형식으로, PNG처럼 확대 시 깨지지 않고 태그/좌표 기반으로 그림을 그린다. 에이전트에게 그림을 그리라고 하면 SVG로 그리는 경우가 많다. 나노바나(nano banana) 같은 이미지 생성 툴과 달리 태그·좌표 기반 표현이라는 점을 강조. 텍스트-투-로티 스킬은 정지된 SVG(예: Spotify, "Hello" 같은 단일 SVG)를 넘기면 그것을 애니메이션으로 표현해 준다.
> "이거 지금 안 움직이죠? 근데 이거를 주면서 이거 가지고 한번 요런 표현을 해 봐라고 하면은 이 텍스트 로티라는 스킬이 이걸 해 준다는 겁니다." [05:03]

### [05:20–07:13] Supabase를 안 쓰는 이유
