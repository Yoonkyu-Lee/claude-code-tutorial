## 요약 (주제별)

### 무엇을 하는 영상인가
- Bright Data 협찬 2부작 중 두 번째이자 마지막 편. 쿠팡 상품 데이터를 수집→간단 분석→이미지 포함 워드 보고서까지 만드는 전체 파이프라인을 시연한다.
- 방점은 "분석"이 아니라 "일반 코드로는 안 되는 쿠팡 수집을 Bright Data가 대신 쉽게 처리해준다"는 데 있다. (화자 명시)
- 대상 독자는 실무자 — 이커머스 데이터를 모아 전략을 짜야 하는 마케터·기획자. 일반 유저에게는 유료 결제 가치가 애매할 수 있다고 화자가 먼저 인정한다. [의견]

### 왜 Bright Data가 필요한가 (문제 정의)
- 쿠팡은 일반 코드로 크롤링이 안 된다(해본 사람은 안다는 화자 주장). 사이트 구조가 복잡하고, 이미지까지 수집하려면 테스트를 많이 해야 하는데 잘 안 된다.
- 크롤링 코드를 짜도 사이트 구조·내용이 바뀌면 유지보수가 매우 어렵다. 그런 걸 직접 하기 힘든 사람에게 이 과정을 추천. [의견]
- Bright Data에는 "언로커(Unlocker)"가 있어 차단(봇 차단)을 우회해준다. 돈을 내고 언로커를 이용해 링크를 가져온다.

### 비용 효율 주장 (주장/미검증)
- 바이브 코딩 툴만으로 복잡한 사이트(쿠팡) + 이미지까지 수집하려면 20달러 요금제로는 현실적으로 어렵고, 월 100달러급 결제가 필요하다는 주장. (주장/미검증)
- Bright Data 수집 비용은 1000건당 1.5달러라는 주장. (주장/미검증)
- 절감 포인트: 데이터를 실제로 수집하기 위해 반복 시도하며 소모되는 토큰 비용을 줄일 수 있다는 논리. (주장/미검증)
- ※ 02:00 부근에서 비용 비교 설명이 잘리고(자막 점프) 03:00 부근으로 넘어가 결론 수치는 부분적으로만 확인됨. [불명확]

### 전체 절차 — 3단계 방법론
- 1단계: Bright Data MCP(언로커)로 쿠팡 상품들의 **개별 링크 목록**을 수집 → CSV로 저장. 영상에선 1페이지 60개 + 2페이지 60개 = 링크를 모아 CSV로 만들었다.
- 2단계: Bright Data **사이트의 Scrapers 기능**으로 상세 데이터 수집(코드가 아니라 웹 사이트 UI에서). Scrapers Library에서 "쿠팡" 검색 → Scrapers 항목 선택(Datasets는 이미 수집 끝난 데이터 구매처라 구독자용으로 못 씀).
- 3단계: 수집 결과의 **스냅샷 ID**를 복사 → Cowork(코워크)로 불러와 나머지 작업(CSV 저장·분석·보고서) 진행.
- 화자는 실무자 기준으로 "이 정도면 굉장히 간단하다"고 평가. [의견]

### MCP 세팅 (언로커 사용 준비)
- 언로커를 쓰려면 MCP 세팅이 되어 있어야 함. 상세는 1편에서 설명했다며 축약.
- GitHub에 올린 `claude_desktop_config.json`에서 `mcpServers` 중괄호 안의 Bright Data 항목만 복사.
- Claude Desktop 설정 → 개발자 → 구성 편집을 편집기로 열어 `mcpServers` 맨 위에 붙여넣고, 발급받은 토큰을 입력(토큰은 Settings에서 확보). 화자는 오늘 노출됐으니 나중에 지운다고 언급.
- 저장 후 Claude Desktop을 **완전 종료**(윈도우는 우측 하단 트레이 아이콘 우클릭 → 종료)해야 완전히 적용됨. 재실행하면 언로커 MCP로 링크 수집 가능.

### Scrapers로 상세 수집 (실습)
- 준비한 CSV(링크만 든 파일)를 Scrapers에서 "Collect by URL" 기능으로 처리. "Discover by category URL"은 쓰면 안 됨 — 화면에서 보는 데이터가 아니라 Bright Data 크롤링 방식대로 다른 데이터가 와서, 일치율을 높이려 Collect by URL 사용. [의견]
- 업로드 시 "CSV missing field URL" 에러가 났는데, 원인은 헤더가 대문자 `URL`이라서. 소문자 `url`로 바꾸면 해결(시청자에게도 소문자로 바꾸라고 안내). 다시 업로드하니 헤더 포함 총 121개(=상품 120개) 인식.
- Scraper mode는 Synchronous, Run manual 방식으로 실행. 120개 상세 페이지를 하나씩 들어가 데이터를 가져옴.
- 예상 소요는 5~8분이라 했으나 실시간 진행바(120개 중 19개…)를 거쳐 실제로는 약 1분 만에 120개 완료, 성공률(Success rate) 100%. (주장/미검증)

### 스냅샷 → Cowork로 가져오기
- 완료 후 스냅샷 ID를 클릭해 복사. Cowork에서 새 작업 생성, 폴더는 1편에서 쓰던 "유튜브 브라이트 데이터" 폴더 사용.
- 이 폴더에 `.env` 파일이 있어야 하고, 그 안에 Bright Data API 키 값이 반드시 들어 있어야 함(없으면 이후 진행 불가). 열어서 API 키가 들어와 있는지 확인 필수.
- 스냅샷 ID를 넘기고 "이 데이터를 CSV로 저장해라"라고 지시 → 폴더의 API 키로 스냅샷 데이터를 가져와 CSV로 폴더에 저장. 영상 시작 10분도 안 돼 120개짜리 CSV 확보. 리뷰 순서대로 정렬 요청해 그 순서로 들어온 것도 확인.

### 분석·시각화 (Show Me / 대시보드)
- 기본 데이터 작업은 여기서 끝, 이후는 자유. 예로 Cowork에서 "리뷰 수·전망 좋은 분야를 확인할 대시보드를 만들어라"고 지시하고 **Show Me** 기능을 붙여 시각화.
- Show Me는 코드팩토리(유튜버)가 소개한 기능이라 출처를 밝힘. 오른쪽 아티팩트가 아닌 별도 영역에 열려 대시보드를 바로 만들어줌(코어클라우드라 안 열릴 수도 있다고 단서). 총 리뷰/평균 평점/분야 수 등 지표와 그래프가 나온다.
- 분석 결과(예시): 클렌징폼이 가장 많고 그다음 스크럽필링, 클렌징밤크림 등 클렌징 위주. 비건 상품은 리뷰가 거의 없어 인기가 없다는 해석. "기회 지도(수요 대비 경쟁)"에서 좌상단(경쟁 적고 수요 큼)에 해당하는 스크럽필링류를 들어가볼 만하다고 제시. [의견]

### 이미지 포함 워드 보고서
- 수집 데이터에 **이미지까지 포함**된 것이 특징. 화장품 회사 기획자·마케팅팀장이 잘 팔리는 상품을 벤치마킹하는 상황을 가정.
- 지시 예: "CSV 파일을 중심으로 팀장님에게 보고할 세 페이지 간단한 워드 보고서를 만들어라. 이미지를 파일에 첨부해서 보여주면 좋겠다."
- 새 페이지로 만들라 했더니 시간이 오래 걸려, 그냥 페이지를 만들 테니 진행하라고 지시. 결과적으로 이미지까지 들어간 보고서가 완성 → 보고서 작성 시간이 크게 줄었다는 주장. (주장/미검증)

### 세일즈 구조 / 부가 안내
- 유료로 올렸다고 안 써보지 말고 꼭 한번 해보라는 CTA. "이런 걸 봐줘야 화자에게도 도움이 된다"고 언급.
- 최근 영상을 많이 못 올린 이유가 있지만 여기서 말할 건 아니라며 넘어감.
- Bright Data가 25일 서울 파르나스 호텔에서 워크숍 진행 예정. 링크를 더보기에 첨부, 아젠다는 영어라 한글 번역해 확인하라고 안내.

- Bright Data(협찬) 2부작 마지막 편: 일반 코드로는 막히는 쿠팡 상품 데이터를 언로커·Scrapers로 수집하는 실습.
- 흐름은 MCP로 링크 수집 → Bright Data 사이트 Scrapers로 상세 수집 → 스냅샷 ID를 Cowork로 불러와 CSV 저장 → Show Me 대시보드 + 이미지 포함 워드 보고서.

## 전문 — 시간순 워크스루

### [00:00–00:44] 인트로 — 2부작 마지막, 쿠팡 수집 예고
Bright Data 두 번째 유료 영상이며, 원래 2편으로 기획됐던 것의 마지막이라 오늘로 Bright Data 영상은 끝이라고 밝힌다. 영상 준비를 위해 다양한 데이터 수집 시나리오를 직접 해보니 확실히 쓰면 좋다고 평가. 오늘 수집 대상은 쿠팡인데, 해본 사람은 알겠지만 일반 코드로 데이터를 수집하는 건 되지 않는다고 말한다.
> "쿠팡은 이제 해보신 분들은 아시겠지만 일반적인 코드를 작성을 해서 데이터를 수집하는 거는 되지가 않습니다." [00:23]

Bright Data를 거치면 매우 좋으며, 그 방식으로 데이터를 수집하고 간단히 분석까지 해본다. 다만 방점은 분석이 아니라, 안 되는 것을 Bright Data가 대신 아주 쉽게 처리해준다는 데 있다고 못 박는다.

### [00:44–01:14] 대상 독자 — 실무자용, 유지보수 문제
이 영상은 실무자용이며, 일반 유저는 "Bright Data에 돈을 쓸 필요까지 있을까" 싶을 수 있다고 인정한다. [의견] 하지만 이커머스 데이터를 수집해 전략을 짜야 하는 마케터·기획자에게는 이 과정이 매우 유용할 거라고 본다. 크롤링 코드를 짜도 사이트 구조·내용이 바뀌면 유지보수가 굉장히 어렵기 때문에, 그런 걸 하기 힘든 사람에게 오늘 과정을 추천한다. 바쁜 사람을 위해 이 내용을 준비했다고 덧붙인다.

### [01:14–02:00] 비용 효율 비교 (주장/미검증)
비용 효율을 직접 비교해봤다며 설명을 시작한다. 바이브 코딩 툴로 어떻게든 코드를 짜서 수집을 끝내겠다고 하면, 쿠팡처럼 복잡한 사이트는 수집할 내용도 복잡하고 이미지까지 가져오려면 테스트가 필요한데 잘 안 된다. 그래서 20달러로는 현실적으로 어렵고 월 100달러급 결제를 해야 한다고 주장한다. (주장/미검증) 반면 Bright Data는 1000건당 1.5달러이며, 바이브 코딩 툴에서 절감되는 부분은 데이터를 실제로 수집하려 반복 시도할 때 드는 토큰 비용이라고 설명한다. (주장/미검증)
> "브라이트데이터 비용 같은 경우는 1000건당 수집이 1.5달러인데…" [01:48]

이 지점에서 자막이 02:00→02:55로 점프해 비용 비교의 마무리 부분이 누락된다. [불명확]

### [02:55–03:26] 방법론 개요 — MCP로 개별 링크 수집
기능 소개 뒤 오늘은 개별 링크로 수집하는 과정을 보여준다. 그러려면 쿠팡 상품들의 개별 링크 목록이 필요하고, 그것을 Bright Data MCP로 수집한다(처음에 "바이브 코딩으로"라고 말했다가 MCP로 정정). MCP로 수집하는 이유는 Bright Data의 "언로커(Unlocker)" 때문. 언로커를 쓰면 봇 차단을 우회할 수 있어, Bright Data에 돈을 주고 언로커로 링크를 가져오게 한다.
> "브라이트 데이터에 보시면은 언로커라는 것이 있어요. … 이제 봇 차단이 되기 때문에 이것을 이제 브라이트 데이터를 통해서 우리가 얘한테 돈을 주고 언로커를 이용을 해서 링크를 가져오게 할 수 있습니다." [03:16]

### [03:27–03:53] 3단계 파이프라인 정리
상세 데이터 자체는 코드가 아니라 Bright Data 사이트에서 수집하게 하고, 이후 데이터를 가져올 때는 스냅샷을 쓴다. 정리하면 총 3단계: (1) MCP(언로커)로 개별 링크를 가져온다, (2) 상세 데이터 수집은 Bright Data 사이트에서 시킨다, (3) 스냅샷 ID를 떠와 Cowork(코워크)로 다시 불러들여 나머지를 진행한다. 실무를 해본 사람이면 이 정도면 굉장히 간단한 것이라고 평가. [의견]

### [04:44–05:07] 링크 CSV 준비 + MCP 세팅 예고
1페이지 60개와 2페이지 60개의 링크를 수집하게 만들어 CSV 파일로 저장했다. 언로커를 쓰려면 MCP 세팅이 되어 있어야 하는데, 1편에서 설명했으니 간단히만 언급하겠다고 한다. (1편은 많이들 안 봤다며 아쉬움 표현.)

### [05:01–06:05] MCP 세팅 상세 (config·토큰·완전 종료)
1편에 있는 GitHub 링크로 들어가면 `claude_desktop_config.json` 파일을 올려놨다. 그 안 `mcpServers` 중괄호 사이의 Bright Data 항목만 복사한 뒤, 자신의 토큰을 입력한다. Claude Desktop 설정 → 개발자 → 구성 편집을 편집기로 열어 `mcpServers` 맨 위에 붙여넣고, Settings에서 받은 토큰을 그대로 입력(자신은 노출됐으니 나중에 지운다고 언급). 저장 후 Claude Desktop을 완벽히 종료해야 하는데, 윈도우는 우측 하단 트레이 아이콘을 우클릭해 종료를 눌러야 완전 종료된다. 다시 켜면 언로커 MCP로 실제 링크를 가져올 수 있다.
> "완전 종료한 뒤 윈도우에서는 오른쪽 아래에 있는 아이콘 우클릭 하셔서 종료를 누르셔야 완벽하게 종료가 됩니다." [05:55]

### [06:05–06:57] 실습 시작 — Scrapers Library에서 쿠팡 검색, Collect by URL
링크가 담긴 CSV를 열면 링크만 있는 파일이며, 앞서 본 쿠팡 사이트 카테고리에서 가져온 것이다(순서는 다를 수 있으나 대부분 데이터가 잘 들어 있음). 이제 Bright Data에 들어가 Scrapers 기능으로 데이터를 가져오게 하는데, Scrapers는 Scrapers Library에서 시작한다. "쿠팡"을 검색하면 위쪽에 Scrapers가 나오고, 아래 Datasets는 이미 수집이 끝난 데이터를 구매하는 곳이라 구독자로서는 쓸 수 없다. Scrapers를 눌러 작업을 진행한다. URL이 다 준비돼 있으므로 "Collect by URL" 기능을 쓴다.

### [06:57–07:11] Discover by category URL은 쓰지 않는 이유
"Discover by category URL"은 쓰면 안 된다고 한다. 테스트해보니 실제로 자신이 보고 있는 화면의 데이터가 아니라, Bright Data가 하고 있는 크롤링 방식에 의해 다른 데이터가 오더라는 것. 그래서 일치율을 높이기 위해 Collect by URL을 사용했다고 설명. [의견]

### [07:11–07:41] CSV 업로드 — 대문자 URL 에러와 소문자 수정
들어온 뒤 할 일은 간단하다. 기존 내용을 지우고 "업로드 파일"로 바꾼 뒤 CSV를 드래그앤드롭한다. 그러자 "CSV missing field URL" 에러가 뜨는데, 살펴보니 헤더가 대문자 `URL`이라 그런 것 같다고 판단. 시청자에게는 소문자로 바꾸라고 안내한다. 소문자로 바꿔 다시 업로드하니 정확히 인식됐고, 맨 위 헤더 URL 1개를 포함해 총 121개(=상품 120개)라고 표시된다. 이후 Choose scraper mode에서 Synchronous를 쓰고 Run manual로 실행한다.

### [07:41–08:11] 실행 — 120개 상세 페이지 수집, 실시간 진행
120개의 상세 페이지를 하나씩 다 들어가 데이터를 가져온다. 소요는 5~8분 정도 걸리더라며 잠시 기다려야 한다고 안내(내부 사정에 따라 더 빠를 수도 있음). 실시간 업데이트로 "120개 중 19개 완료" 식으로 진행바가 계속 갱신되는 걸 보여준다.

### [08:11–08:24] 예상보다 빠른 완료 — 1분, 성공률 100%
생각보다 빨리 진행돼 약 1분 만에 데이터 120개를 다 가져왔고, Success rate가 100%로 전부 수집된 것을 확인한다. (주장/미검증) 다음으로 할 일은 간단한데, 스냅샷 ID를 클릭하면 그 ID가 복사된다.

### [08:24–08:55] Cowork 새 작업 + .env / API 키 확인
Cowork에서 "새 작업"을 누르고, 폴더는 1편에서 썼던 "유튜브 브라이트 데이터"로 세팅한다. 이 폴더에는 `.env` 파일이 있고, 그 안에 Bright Data API 키 값이 반드시 들어 있어야 한다. 이걸 확인해야 하며, 없으면 이후 작업이 진행되지 않는다. 열었을 때 API 키가 다 들어와 있는지 확인해야 한다고 강조한다.
> "이 점 인바이런먼트 파일 같은 경우는 반드시 … 브라이트 데이터 API 키는 … 값이 들어가 있어야 됩니다. … 없으면은 작업이 이후 더 진행되지 않습니다." [08:38]

### [08:55–09:37] 스냅샷 ID로 데이터 가져와 CSV 저장
그 상태에서 스냅샷 ID를 넘긴 뒤 "여기에 있는 데이터를 CSV 파일로 저장해라"라고 지시한다. 폴더 내 Bright Data API 키를 이용해 스냅샷의 데이터를 가져온다. 방금까지 URL 링크만 있던 폴더의 CSV 자리에, 120개 데이터가 수집돼 레디 상태인 스냅샷을 다운로드해 저장한다. 영상 찍은 지 10분도 안 됐는데 120개짜리 CSV 파일이 다 들어왔고, 쭉 나열돼 보여진다.

### [09:37–11:20] 데이터 확인 — 리뷰 순 정렬 반영
(자막이 09:37→11:06으로 점프.) 실제로 쓰는 사람은 데이터를 더 많이 가져올 것이라며 여기선 줄여서 보고 있다고 언급. 특정 항목을 그대로 검색해보면 실제로 그 데이터가 들어왔음을 확인. 리뷰 순서대로 해달라고 요청했기 때문에 그 순서로 들어와 있는 것도 보인다. 전반적으로 가져오기로 한 데이터를 다 잘 가져왔다고 정리하며, 기본 데이터 작업은 여기서 끝났다고 말한다.

### [11:20–12:03] Show Me로 대시보드 만들기
이후는 자유롭게 작업하면 된다며, 예로 Cowork에서 이 CSV를 바탕으로 "리뷰 수나 전망 좋은 분야를 확인할 대시보드를 만들어라"라고 지시하고 Show Me 기능을 붙인다. Show Me를 잘 모르는 사람이 있는데, 자신이 좋아하는 유튜버 코드팩토리가 소개한 기능이라고 출처를 밝힌다. Show Me를 쓰면 오른쪽 아티팩트가 아니라 별도 영역에 대시보드가 바로 만들어진다(코어클라우드라 안 열릴 수도 있다는 단서). 대시보드로 확인도 할 수 있다.
> "코드팩토리님이라는 유튜버분이 계시는데 그분이 쇼미라는 기능을 소개를 했어요." [11:44]

### [12:03–12:32] 워드 보고서 시나리오 + 이미지 포함 특징
동시에, 워드 문서로 보고서를 만드는 경우도 보여준다. 화장품 회사 기획자나 마케팅팀장이 요즘 잘 팔리는 상품을 벤치마킹해 가져가야겠다고 할 때, 이 데이터로 보고서를 만들 수 있다고 가정. 이 데이터의 특징은 이미지까지 포함돼 있다는 점이라, 보고서 안에 상품 이미지(패키징된 모습 등)를 함께 넣어 보여줄 수 있다. 지시 예: "CSV 파일을 중심으로 팀장님에게 보고할 세 페이지 간단한 워드 보고서를 만들어라. 이때 이미지를 파일에 첨부해서 보여주면 좋겠다."

### [12:38–13:28] Show Me 시각화 결과 + 카테고리 인사이트
Show Me로 전반적인 결과가 나온다. 총 리뷰·평균 평점·분야 수 등 지표가 뜨고, 아래에는 보고서로 나열해야 할 것들을 대시보드로 그래프와 함께 보여준다. Show Me는 데이터를 비주얼라이즈해 결과를 보여주는 기능이라고 정리. 결과를 보면 클렌징폼이 가장 많고 그다음 스크럽필링, 클렌징밤크림, 클렌징 순으로 클렌징 위주 상품들이며, 비건은 리뷰가 거의 없어 인기가 없다는 해석을 내놓는다. [의견]

### [13:28–14:00] 기회 지도 (수요 대비 경쟁)
"기회 주도 / 수요 대비 경쟁" 같은 항목도 분석해줬다며, 좌상단이 경쟁이 적고 수요가 큰 영역이라고 설명. 스크럽필링 같은 경우 그 영역에 해당하니 들어가보는 게 어떻겠냐는 식의 제안도 보여준다고 언급. [의견] (14:33에 "[콧방귀]" 같은 비언어 표시.)

### [14:56–15:33] 마무리 + 보고서 완성 확인
오늘 영상은 여기서 마치겠다고 한다. 최근 영상을 많이 못 올렸는데 이유가 있지만 여기서 말할 건 아니라며 넘어간다. 결과를 보니 보고서가 나왔고, 새 페이지로 만들어 달라니 너무 오래 걸려 "그냥 페이지 만들 테니 그냥 해라"라고 지시했다고 설명. 결과물은 이미지까지 싹 들어가 예쁘고 멋있게 완성됐고, 보고서 만드는 시간이 크게 줄었을 거라고 본다. (주장/미검증)
