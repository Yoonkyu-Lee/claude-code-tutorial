## 요약 (주제별)

### 무슨 일이 일어났나 — 개명
- 데스크톱용 "Codex 앱"이 사라진 게 아니라 이름을 **"ChatGPT"로 개명**됐다. 좌측 상단도 "ChatGPT Work / Codex"처럼 앞에 무조건 ChatGPT를 붙이려는 의도. "코덱스"로 검색하면 앱이 안 나오고 "ChatGPT"로 검색해야 나온다(로고도 곧 바뀔 듯).
- 화자 추정 [의견/미검증]: Claude 데스크톱의 코워크(cowork)를 의식한 움직임(Claude 쪽 코워크 버튼도 위치가 옮겨졌다고 언급). "코덱스"는 전문가·무서운 느낌이라 일반 유저에게 편안한 "ChatGPT" 이름으로 바꿔 마케팅하려는 것 — 본인도 "혼자 추측"이라고 명시.
- 결론: 이름만 바뀐 게 아니라 통합 시도도 있지만, 기능적으로 두려워할 건 없다.

### 이 앱의 본질 — 파일 시스템 접근
- 핵심은 "파일 시스템 접근". 웹 ChatGPT는 파일을 만들면 "다운로드해 가라", 업로드하려면 첨부해야 했다. 이 앱은 **내 PC 영역 안에서 실행되는 에이전트**라, 바탕화면·다운로드 등 폴더의 파일을 스스로 찾고 읽고·수정·삭제·복사·생성한다.
- "프로젝트 선택" = **폴더 선택**. 예: 데스크톱 폴더를 지정하면 이후 채팅이 그 폴더를 보고 작업. "이 폴더에 뭐 있냐" 물으면 실제로 보고 온다. 업/다운로드 없이 "내 폴더에 만들어, 찾아 읽어라"를 시킬 수 있어 작업이 매끄럽다.
- [의견] 경향성상 "에이전트가 파일 시스템에 접근해 작업"하는 게 앞으로 기본 방향이 될 것이며, 이를 받아들이고 자연스럽게 쓰는 게 중요.

### 웹 채팅 vs 앱 채팅, 토큰 분리
- 앱이 웹 브라우저 ChatGPT를 끌고 들어와, 앱 안 "최근 채팅"에서 웹에서 하던 채팅이 그대로 보이고 우측 하단 팝업으로 웹 채팅 기능을 쓸 수 있다 — 통합 시도.
- [의견] 앱 쪽이 더 편하진 않다. 웹의 고정(핀) 기능 등 편의가 아직 없고 "최근"밖에 없어 핀을 쓰던 사람에게 불편. "채팅이 안 꺼지는" 버그 지적, 고정 기능만 넣어도 완화될 것이라 개선 요청.
- 토큰(주장/미검증, 본인 재확인 예정): 앱에서 파일 시스템 접근 채팅을 쓰면 토큰이 소진되고, 웹 브라우저 채팅은 그 토큰과 무관하다는 게 커뮤니티 관찰. 현재 정리로는 **둘의 토큰이 따로 돈다**. 다만 통합을 하려는 이상 회사가 계속 따로 두길 원하진 않을 것이라 봄(지금은 혼선).

### Work vs Codex 모드 — 근본 동일, "경향성" 차이
- 두 모드는 근본 구조가 같고, 전환해도 채팅에서 크게 바뀌는 건 없다(예시만 바뀜). 딱 하나: **Work엔 플러그인이 노출되고 Codex엔 안 보인다**(단 Codex에서도 플러그인은 공유돼 실제로는 쓸 수 있고 화면에만 안 보임 — 예: Insane Search 검색하면 목록에 있음).
- [의견/전언] Work = 일상 업무 경향으로 "선입력"된 프리셋(PDF·문서·스프레드시트·프레젠테이션·조사·브리핑·업무 자동화에 더 잘, "뻘짓을 덜 함"). Codex = 코드 작업 특화 경향(코드 키워드에 더 민감).
- 이는 서비스사가 **하네스(harness)를 미리 세팅해서 주는** 흐름 — 유저가 헤매지 않게 "일상은 Work, 코딩은 Codex"로 최적화 제공. 회사 옷 입으면 회사원, 운동복 입으면 운동 모드처럼 같은 사람의 모드 전환일 뿐.

### Codex CLI와의 관계
- 화자 결론: **ChatGPT 앱(Codex 모드) = 터미널의 Codex CLI와 완전히 동일**한 녀석. 지역(local) 스킬도 이제 잘 감지하고(예전엔 못 함), 폴더에 스킬이 없으면 "없다"고 정확히 답한다.
- 매핑: ChatGPT Work ≈ Claude의 코워크(cowork), Codex 모드 ≈ Claude Code처럼 코드 탭에서 바이브 코딩. **터미널을 안 써도** 프로젝트를 오가며 바이브 코딩 가능한 환경이 마련됨.
- 앱은 상태 파일을 미리보기로 보여줘 터미널처럼 직접 파일을 열 필요가 없다는 점이 편의(화자는 자칭 "터미널 충"이나 이 편의는 인정).

### 바이브 코딩 데모 — 뱀서류 게임
- 라이브 강의에서 만들던 "Study 뱀파이어"(뱀파이어 서바이버류)를 이어 작업. **CLI에서 하던 채팅 히스토리를 앱이 완벽히 동일하게 기억**(라이브에서 이 앱으로 작업한 게 아닌데도) — 히스토리 통합 확인.
- 버그: 보이지 않는 벽으로 이동/도망 공간이 좁음 → "맵을 무한정 이동할 수 있게" 요청하니 무한 맵으로 수정됨(뱀서류다워짐).
- 애니메이션: 플레이어가 가만히 서 있어 **4프레임 걷기 모션**을 만들어 이동 방향으로 실행되게 요청. 효율 지시: "좌우는 뒤집기만 하면 되니 따로 그리지 마"(단 뒷모습은 앞모습과 달라 따로 필요). 작업을 큐(Q)에 넣어 무한 맵 → 애니메이션 순으로 처리. 결과 마녀가 걷지만 아래쪽 잘림·왼쪽 방향에 어색함 잔존 — 그래도 무애니메이션보다 나아짐. 이후는 "노가다의 영역".

### 스프라이트·이미지 생성 관찰
- 에이전트가 스프라이트 시트 생성: 앞으로 4프레임·옆으로 4프레임, 반대는 뒤집기, 뒷모습은 별도. [의견] 스프라이트는 축이 조금만 어긋나도 뒤틀리는데 이번엔 축을 잘 유지해 자연스러웠다(화자는 옛날 게임 도트 개발 경험이 있어 과정이 쉽게 느껴짐).
- 모델 추정 [의견/미검증]: 이미지 생성 모델 자체는 Image 1.5 또는 Image 2를 그대로 쓰는 듯하나, **Sol이 만들어주는 프롬프트가 좋아** 알아서 잘 만든다. [의견] 이 때문에 AI 스프라이트 제공 서비스들이 의미를 잃는 흐름 — 에이전트들이 다 삼키는 중.

### 화자 의견 — 게임 개발의 무게중심, 모델 리뷰 유보
- [의견] 게임 만들기는 고수 영역이 아니라 아이디어 싸움. 다만 게임성 개선 용어·로직·유사 게임 기법을 아는 사람이 더 잘 만든다. 컨셉만 바꾼 "카피 게임"은 메리트가 아니고 **게임성·배포·세일즈**가 중요.
- [의견/미검증, 진행 중 소감] GPT-5.6 **Sol** 리뷰는 유보 — 모델마다 웹페이지 만들어 비교하던 방식 대신 다른 방식으로 하고 싶어서. 현재 체감은 Sol 쪽으로 기울지만, Fable 5와 동일 조건 비교, 커서 앰배서더로서 Grok 4.5까지 써 본 뒤 종합해 말하겠다고 함.
- [의견] 이 게임으로 얻는 이득은 없어도 만들며 연습이 된다면 그것으로 좋다(자전거 타는 연습처럼 — 꼭 돈 벌 필요는 없다).

## 3줄 오리엔테이션
- 데스크톱용 "Codex 앱"이 사라진 게 아니라 "ChatGPT 앱"으로 개명됐을 뿐이며, 기능적으로 바뀐 건 거의 없다는 게 핵심 메시지.
- 이 앱의 본질은 "파일 시스템 접근" — 웹 ChatGPT와 달리 내 PC의 폴더/파일을 에이전트가 직접 읽고·수정·생성한다. Work 모드(일상 업무 경향)와 Codex 모드(코딩 경향)는 근본 구조가 같고 "경향성" 차이뿐이며, Codex CLI와도 동일한 녀석이라는 결론.
- 후반부는 이 앱(Codex 모드)에서 터미널 없이 뱀서류(뱀파이어 서바이버 스타일) 게임을 실제로 바이브 코딩하는 데모 — 무한 맵 수정, 4프레임 걷기 애니메이션 스프라이트 생성까지.

## 전문 — 시간순 워크스루

### [00:00–01:22] "코덱스 앱이 없어졌다" — 사실은 ChatGPT로 개명
화자는 코덱스(Codex) 앱이 없어져서 혼란스러울 사람이 많을 것 같아 정리 영상을 준비했다고 연다. 결론부터 말하면 앱이 사라진 게 아니라 이름을 "ChatGPT"로 바꿨을 뿐이다. 이름을 바꿨다는 건 의도가 다분하며, Claude 데스크톱의 코워크(cowork)를 의식한 움직임으로 본다 — 실제로 Claude 쪽 코워크 버튼도 위치가 옮겨졌다고 언급. 경쟁사들이 계속 일반 유저에게 애플리케이션 형태로 "에이전트가 내 파일 시스템에 접근해 내 PC를 조작하는" 경험을 부드럽게 제공하려 준비 중이라는 게 화자의 관찰이다.
> "코딕앱이 이제 이름을 최지T로 이제 바꿨고요." [00:13]

화면상 예전엔 "Codex 앱"이던 것이 "ChatGPT"로 바뀌었고, 좌측 상단도 "ChatGPT Work / Codex" 식으로 앞에 무조건 ChatGPT를 붙이려는 의도가 보인다고 짚는다. [의견] "코덱스"라는 표현은 전문가 영역 느낌이라 무섭고, 원조격인 "ChatGPT" 이름이 일반 유저에게 더 편안하기 때문에, 무서운 느낌을 줄여 마케팅하려고 이름을 바꿨을 것이라는 게 화자의 추측(본인도 "혼자서 추측"이라고 명시). 하지만 바뀐 건 거의 없고 두려워할 필요 없다고 정리.

### [01:22–02:37] 웹 채팅 vs 앱 채팅 — 통합은 시작됐지만 아직 불편
이름만 바뀐 건 아니다. 익숙한 웹 브라우저 ChatGPT(채팅으로 이미지 생성·글쓰기·웹 검색)를 이제 앱이 끌고 들어왔다. 앱 안에 "최근 채팅"이 있어 웹 브라우저에서 하던 채팅이 그대로 남아 있고, 그걸 클릭하면 우측 하단에 팝업이 떠 진짜 웹 브라우저 ChatGPT 채팅 기능을 쓸 수 있다. 즉 웹 채팅 영역과 (내 PC에서 실행되는) 앱 채팅 영역은 서로 다르며, 통합을 시도하는 느낌이다.

다만 [의견] 앱 쪽이 편하냐면 아니다. 웹 브라우저에는 고정 핀 기능 등 여러 편의가 있는데 앱은 아직 그런 편의를 제공하지 않고, 지금은 "최근"밖에 없어서 핀을 꽂아 쓰던 사람에게는 불편하다. 그래도 통합 의지 자체가 변화 지점이라고 평가.

### [02:37–04:02] 앱 채팅의 정체 — 내 PC에서 실행되는 에이전트 + 토큰 분리
ChatGPT가 맞긴 한데, 이 앱 채팅(Work)은 내 PC에서 실행되는 채팅이다. 그래서 이 녀석은 사용자의 파일 시스템 — 바탕화면, 다운로드 폴더 등 — 을 직접 보고, 거기 있는 파일을 스스로 찾고 수정·삭제·복사·생성할 수 있다.
> "얘는 여러분의 PC 에어리어에서 이미 실행이 된 녀석이기 때문에 영역만 정해 주면은 그 영역에서 이제 그 파일을 다 보고 이제 혼자 작업할 수 있기 때문에 스스로 파일을 가져옵니다." [03:20]

예전 웹 ChatGPT에서 엑셀 파일을 만들면 "다운로드해서 가져가세요"였고, 업로드하려면 파일 첨부를 해야 했다. 하지만 이 앱은 이미 PC 영역 안에서 실행되므로 영역만 정해 주면 그 안에서 알아서 작업한다 — 매끄러운 작업이 장점이다.

토큰 관련: 이 앱(구 Codex 앱, 현 ChatGPT 앱)에서 파일 시스템에 접근하는 채팅을 쓰면 토큰이 소진되고, 웹 브라우저 채팅은 그 토큰 소진과 관련이 없다는 게 커뮤니티 관찰이라고 전한다(주장/미검증 — 본인이 직접 써 보고 다시 정정 발표하겠다고 함). 기존에도 이 앱은 웹과 토큰이 따로 돌았기 때문에 그건 맞을 것이라고 봄. [의견] 통합을 하려는 이상 회사가 토큰을 따로 놓길 원하진 않을 것이고, 지금은 아직 혼선이 있다. 현재 정리로는 둘의 토큰이 따로 도는 게 맞다.

### [04:02–04:35] 순수 채팅 수요와 앱의 아쉬운 점
화자는 순수하게(파일 시스템 접근 없이) 채팅만 쓰고 싶을 때도 있다고 한다 — 예: 유튜브 썸네일 제작 시 늘 쓰던 프롬프트를 살짝 수정해 쓰는 경우, 웹에서는 고정(핀)을 자주 썼다. [의견] 이런 걸 앱에서 하기엔 아직 편하지 않으니, 관계자가 본다면 고정 기능만 넣어도 많이 완화될 것이라고 개선을 요청.

### [04:35–05:44] 앱의 핵심 = 파일 시스템 접근 (프로젝트=폴더 선택)
이 앱의 핵심은 결국 파일 시스템 접근이다. PC 영역 안에 바탕화면·다운로드 등 영역이 있고, ChatGPT 앱의 "프로젝트 선택"은 곧 폴더를 선택하는 것이다. 예컨대 데스크톱 폴더를 선택하면 이후 채팅은 그 폴더를 보고 작업할 수 있다 — "이 폴더에 뭐가 있냐" 물으면 실제로 보고 온다. 폴더를 조작·조회할 수 있으므로, 웹처럼 직접 업로드/다운로드할 필요 없이 "내 폴더에 네가 만들어, 폴더에 있는 거 찾아서 읽어라"를 시킬 수 있어 작업이 한결 매끄럽다. 그래서 대부분은 여기서 하는 게 나은 경우가 많을 거라고 권한다.

### [05:44–07:23] Work vs Codex — 근본 구조는 같고 "경향성" 차이 (하네스 프리셋)
Work와 Codex 두 개가 있어 마치 별도 모드 같지만, 전환해도 채팅에서 크게 바뀌는 건 없고 예시만 바뀐다. 딱 하나 차이: Work 모드일 때는 화면에 플러그인이 올라와 있고, Codex 모드일 때는 플러그인이 안 보인다.

리서치 결과 [의견/전언]: Work는 일상적 업무 쪽으로 경향성을 맞춰둔 무언가가 "선 처리"된 채팅창이다. PDF·문서·스프레드시트·프레젠테이션 등 일상 업무, 연결한 플러그인이 동작할 배경이 마련돼 있다. 파일·사이트 만들기, 조사·브리핑, 반복 업무 자동화 같은 일반 업무에 경향성이 맞춰져 "뻘짓을 덜 한다". Codex 모드로 맞추면 일상 업무보다 코드 작업에 특화돼, 코드 관련 키워드에 더 민감하게 반응한다.
> "하네스가 일상원부 쪽이 좀 맞춰져 있냐 아니면은 예 개발 쪽이 좀 맞춰져 있냐 이제 그 차이가 있고요." [06:56]

근본적으로 도구의 기본 구조 차이는 크게 없다. 여기서 읽을 수 있는 큰 흐름: 서비스 제공사가 하네스(harness)를 미리 세팅해서 준다는 것 — 유저들이 이렇게 저렇게 세팅하며 헤매니, "최적화된 하네스를 줄 테니 일상은 여기(Work), 코딩은 저기(Codex)서 하면 더 잘해준다"는 방향.

### [07:23–08:07] Insane Search 플러그인의 모드별 노출 차이
화자가 예전에 소개하며 설치했던 "Insane Search" 스킬이 어디에 설치됐는지 앱에 물어본다. [의견] Codex 모드에서는 그 버튼이 안 보이고 Work 모드에서만 버튼으로 보이는 게 좀 이상하고 재밌다고 함. 결국 파일 시스템에 접근해 둘 다 동작하는 건 다르지 않고, "경향성" 정도의 차이라는 점을 회사 옷 입으면 회사원 모드, 운동복 입으면 운동 모드로 바뀌는 것에 비유 — 완전히 다른 사람이 되는 게 아니라 같은 사람이 모드만 바뀌는 것.

### [08:07–09:11] Work=일상 프리셋, Codex=코딩 프리셋, 플러그인은 공유
Work는 일상 업무 위주로 선입력된 무언가가 있고 컨텍스트 방향성이 그쪽으로 잡혀 플러그인도 일상 업무를 많이 타겟한다. 앱이 로드하는 캐시가 "버전의 복사본"이라며 실제 위치를 확인하는 장면. Codex 모드에서 플러그인을 못 쓰는 건 아니고, 둘 다 동일하게 공유하며 그냥 화면에 안 보일 뿐이라고 결론. 실제로 Codex 모드에서도 "Insane Search" 검색하면 목록에 있어 쓸 수 있다 — 쓰는 데 아무 차이 없다. 근본적으로 동일하되 Codex 살펴보기는 코딩 업무 위주로 선입력돼 있다는 것.

### [09:11–10:17] Codex CLI와 앱은 동일 / Work≈코워크, Codex≈Claude Code
그럼 Codex CLI(터미널에서 `codex`로 쓰는, 고수처럼 보이는 것)는? 화자는 앱과 CLI가 근본적으로 같은 녀석이라고 결론 내린다. "이 폴더에 세팅된 스킬 있냐" 물으면 지역(local) 스킬을 감지한다 — 예전엔 지역 스킬을 감지 못 했는데 이제는 잘 감지한다. 데스크톱에 세팅된 스킬이 없으면 "없다"고 정확히 답한다(예전엔 이걸 몰랐음).
> "우리가 여기 있는이 최지T 애플리케이션의이 상태는 우리가 보고 있는 여기에서 실행한 멋진 코덱스의 CLI와는 이제 완전히 동일하다." [09:49]

정리하면: ChatGPT Work ≈ Claude의 협업 코워크(cowork), Codex 모드로 바꾸면 코드 탭으로 넘어가 Claude Code처럼 바이브 코딩 — 터미널을 굳이 안 써도 된다.

### [10:17–11:14] 방향성·통합, 그리고 버그
[의견] 경향성은 점점 "에이전트가 파일 시스템에 접근해 작업"하는 게 기본 방향이 될 것이며, 파일 시스템 접근이라는 사실을 받아들이고 자연스럽게 쓰는 게 중요하다. 둘의 차이는 없지만 경향성이 맞춰져 있다는 차이점, 그리고 ChatGPT라는 이름으로 바꿔 기본적으로 파일 시스템 접근을 허용하겠다는 게 회사의 의도. 웹에서 하던 채팅(예: 공룡 그림 요청)이 앱 안에서 동일하게 보이고, 이미지 생성 옵션도 똑같이 뜬다 — 통합 확인. 단, "채팅이 안 꺼지는 건 굉장히 불편하다"며 버그로 보이니 고쳐줬으면 한다고 지적.

### [11:14–12:09] 바이브 코딩 데모 시작 — 뱀서류 게임, 히스토리 이어받기
진짜로 바이브 코딩이 되는지 데모로 마무리한다. 이제 "코덱스"로 검색하면 앱이 안 나오고 "ChatGPT"로 검색해야 로고가 나온다(로고도 곧 바뀔 듯). 폴더를 열고, 오늘 라이브 강의에서 만들던 "Study 뱀파이어"(뱀파이어 서바이버류) 게임을 이어 작업한다. 흥미로운 점: 여기 채팅 기록은 CLI에서 작업했던 것인데, 앱이 그 히스토리를 완벽하게 동일하게 기억하고 있다 — 라이브 방송에서 이 ChatGPT 앱으로 작업한 게 아닌데도. 그때 "만들어 봐" 시켜서 개발·실행까지 했고, `Run Game` 커맨드로 게임을 실행해 봤다.

### [12:09–13:03] 게임의 버그 진단 — 보이지 않는 벽
파인더에서 보기 → Study 뱀파이어 → `Run Game`을 누르면 게임이 실행된다("20분 만에 만들었다"). 그런데 보이지 않는 벽이 있어 더 올라가지 못하고 움직일 공간·도망갈 공간이 매우 좁은 버그가 있다. 이 부분을 넓혀 달라고 요청하고 게임성을 올려볼 계획. Codex 모드로 해 놓고 이 상태에서 대화를 이어간다.

### [13:03–14:04] 무한 맵 수정 요청
게임을 직접 실행해 보니 보이지 않는 벽 때문에 뱀서류 느낌이 안 난다, 뱀서류는 맵을 무한정 이동할 수 있어야 한다고 지시. 에이전트가 직접 실행해 보거나 말을 이해하면 "맞아, 무한하게 이동할 수 있도록 바꿔줄게" 식으로 응할 것이라 설명. 바이브 코딩은 여기서부터 별게 없고, 마음에 안 드는 것을 디테일하게 수정하며 나아가는 것 — 에이전트가 출력했던 것들이 다 컨텍스트에 들어 있기 때문에 가능.

### [14:04–15:03] 4프레임 걷기 애니메이션 요청 (스프라이트 + 좌우 뒤집기)
플레이어 에셋이 가만히 서 있어 재미없으니 4프레임 걷기 모션 애니메이션을 만들어 움직이는 방향으로 실행되게 해 달라고 요청.
> "효율을 위해서 4프레임 S 작업을 할 때 왼쪽 오른쪽은 뒤집기만 하면 되니까 굳이 왼쪽 오른쪽을 따로 그리지 마." [13:56]

에이전트가 프레임 단위로 테스트하는 것까지 보인다. 화자는 작업을 미리 큐(Q)에 넣어 무한 맵 → 애니메이션 순으로 처리되게 한다. 이런 식으로 개발을 계속 이어가면 된다. 터미널 없이도 프로젝트를 왔다 갔다 하며 진짜 바이브 코딩이 가능한 환경이 마련됐다는 것.

### [15:03–16:16] 미리보기의 편의 vs 터미널 취향
자주 작업하는 건 핀을 꽂아 편하게 할 수 있다. 화자는 자칭 "터미널 충"이라 터미널을 좋아하지만, 앱은 상태 파일을 미리보기로 보여주기 때문에 터미널처럼 직접 들어가 파일을 열 필요가 없어 좋다고 인정. 에이전트가 테스트를 하며 좌표 제한이 없어졌는지·패턴이 잘 보이는지 확인 중. 맵을 파인더 → `Run Game`으로 다시 실행하면 맵이 무한으로 움직여 우리가 아는 뱀서류 게임이 된다.

### [16:16–17:12] 게임 개발은 이제 아이디어·게임성 싸움
[의견] 게임 만들기는 고수의 영역이 아니고 여기서부터는 아이디어 싸움이다. 다만 게임 개발 지식(게임성 개선 용어·로직·유사 게임 기법)을 아는 사람이 더 잘 만들며, 모르면 말로만 설명해야 해 한계가 있다. 이런 게 매우 쉬워졌고, 테스트해 보면 기본 게임성은 다 갖춘 게임이 나온 셈. 보스 레벨 설계·밸런스 패치는 게임 설계 잘하는 사람이, 그것도 Codex 도움을 받아 하면 된다. 사운드 등 기본은 잘 만들어졌다.

### [17:12–18:22] GPT-5.6 Sol 리뷰 유보 + 스프라이트 시트 생성 관찰
[의견] 화자는 GPT-5.6 Sol(솔) 리뷰를 아직 안 하고 있다 — 충분히 살펴봐야 리뷰가 되며, 모델 나올 때마다 웹페이지 새로 만들어 성능 비교하던 방식 대신 다른 방식으로 해 보고 싶어서. 현재 체감은 5.6 Sol 쪽으로 기울지만, Fable 5와 동일 조건으로 더 써 보고 커서 앰배서더로서 Grok 4.5도 써 본 뒤 다 비교해 말하겠다고 함(주장/미검증 — 진행 중 소감). 한편 에이전트는 스프라이트(sprite) 시트를 만드는데, 앞으로 걷기 4프레임·옆으로 4프레임을 만들고 반대 방향은 뒤집어 쓰되, 뒷모습(뒤로 가는 것)은 앞모습과 달라 따로 필요하다. 캐릭터별 시트를 만들어 마녀가 걷는 모습을 애니메이팅.

### [18:22–19:22] 스프라이트 품질 관찰 + AI가 삼키는 스프라이트 서비스
[의견] 스프라이트는 위치가 조금이라도 어긋나면 이미지가 뒤틀려 제대로 안 움직여 보이는데, 이번엔 축을 잘 유지해 발걸음 움직임을 자연스럽게 만들었다. 화자는 옛날 게임을 직접(도트 찍어) 개발한 경험이 있어 이 과정이 더 쉽게 느껴진다고 밝힘. 슬픈 점 [의견]: 최근 AI로 스프라이트 이미지를 제공하는 서비스를 만든 이들이 많았는데, Sol 모델과 이미지 제너레이션 스킬이 합쳐지며 그것도 의미가 없어진 것 같다 — 에이전트들이 다 삼키려는 흐름. 모델 자체(이미지 생성)는 Image 1.5나 Image 2를 그대로 쓰는 것 같지만, Sol이 만들어주는 프롬프트가 워낙 좋아 알아서 잘 만들어준다.

### [19:22–20:12] "카피 게임"의 무의미 + 게임성·배포·세일즈 중요
[의견] 테스트 삼아 컨셉만 바꿔(카피닌자처럼) 게임을 만드는 건 이제 메리트가 아니고, 진짜 게임성으로 승부해야 한다. 그다음 재미있는 게임을 얼마나 효율적으로 잘 배포하느냐, 세일즈 자체도 매우 중요하다. 이미지가 만들어졌으니 캐릭터가 움직이는 모습으로 곧 게임 완성. 이 게임은 캐릭터가 총 세 개(해금 3개)라 세 개 이미지를 만들려 하는데, 화자는 완성 의도가 없어 마녀 하나만 애니메이팅해 완성해 보라고 지시.

### [20:12–21:20] 걷기 애니메이션 완성 확인 + 마무리
"이미지 다 만들지 말라"고 했는데 끝까지 만들어 기다려줬다. 폴더에 들어가 게임을 실행·테스트하면 마녀가 잘 걷는다 — 다만 아래쪽에 이미지를 잘못 잘라 어색한 부분, 왼쪽으로 갈 때 약간 어색함이 있다. 그래도 애니메이션이 아예 없던 상태보다는 나아졌다. [의견] 이런 개선 요청을 반복하며 게임성을 올리면 되고, 여기서부터는 "노가다의 영역". 바이브 코딩을 재밌게 해 볼 수 있고, 이 게임으로 얻는 이득은 없지만 게임을 만들며 연습이 된다면 그것으로 좋다(자전거 사서 타는 연습처럼 — 무조건 돈 벌 필요는 없다). 다음 영상에서 유용한 소식으로 찾아오겠다며 마무리.
