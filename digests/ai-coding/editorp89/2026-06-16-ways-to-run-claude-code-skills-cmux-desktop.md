## 요약 (주제별)

### 1. 클로드 코드 실행 방식 — 4개 "파(派)"
- 화자는 클로드 코드를 실행하는 환경을 네 부류로 나눈다. 결론은 "실행 환경이 다양할 뿐 본질은 다 같은 클로드 코드"이며, 과거엔 서로 달랐지만 지금은 근본적으로 같아졌다.
- **순정파**: 검은 터미널 화면에서 직접 실행. 클로드 코드가 원래 터미널 실행이 정석이라 "순정".
- **아이디(IDE)파**: VS Code 같은 IDE(통합 개발 환경 = 개발 도구를 모아둔 프로그램) 안의 터미널에서 실행. 코드 수정·파일/폴더 탐색·터미널을 한 화면에서.
- **멀티파**: Cmux·Warp처럼 한 창에서 여러 터미널 세션을 관리. 폴더별로 클로드를 따로 운용할 때. (자막의 "시웍스/시목스/시머스"는 문맥·제목상 Cmux로 교정)
- **당의정파**: VS Code 확장(꽃 아이콘)이나 Claude Desktop의 Code 탭 같은 예쁜 GUI. 당의정 = 쓴 알약에 단맛을 발라 먹기 쉽게 한 것. 버튼·애니메이션으로 직관적이고 설치가 필요 없다.

### 2. 각 방식의 장점과 추천 순서
- 순정파(터미널): 모든 PC에 이미 깔려 있고 심플. "원소스 멀티유즈"라 원리 이해에 좋다.
- 아이디파: 왼쪽에 파일이 보여 에이전트가 보는 것을 시각적으로 확인하며 작업 가능.
- 멀티파(Cmux/Warp): 다중 세션. Cmux는 껐다 켜도 이전 작업 세션이 살아 있다. 화면 분할로 웹브라우저까지 함께 열 수 있음.
- 당의정파: 예쁘고 버튼 많고 직관적, 설치 불필요(Claude Desktop 하나면 됨).
- **추천 순서 [의견]**: 어려우면 당의정(Code/확장)으로 시작 → 욕구 생기면 터미널 → 코드·파일 볼 일 잦으면 IDE → 작업이 많아 다중 실행하면 Cmux/Warp.
- [의견] "뭘 써야 클로드 코드를 제대로 쓰는 것"은 없다. 작업 책상 취향 차이일 뿐. 다만 당의정은 자전거 보조 바퀴 같아서, 자유로운 주행(속도)을 위해선 언젠가 터미널로 넘어가야 한다.

### 3. 핵심 원리 — 실행한 폴더를 중심으로 활동
- 클로드 코드는 **사용자가 실행(소환)한 폴더**를 중심 무대로 삼아 그 안에서 주로 활동하고, 필요할 때만 바깥을 드나든다.
- 그래서 실행 위치가 매우 중요하다. 바탕화면에서 실행하면 바탕화면의 수많은 파일을 다 보게 되고, Windows 설치 직후 기본 위치인 `C:\Windows\System32`에서 실행하면 시스템 파일을 읽게 되어 하나라도 잘못 건드리면 위험하다.
- 대응: 공부할 때마다 **빈 폴더를 새로 만드는 습관**. 그 폴더로 이동한 뒤 클로드를 실행.

### 4. 설치 절차 (Windows) 및 위치 주의
- "claude code install"을 구글 검색 → 맨 위 공식 문서. Mac은 명령어 하나, Windows는 PowerShell과 CMD 명령어가 다름(화자는 CMD 추천이지만 데모는 PowerShell).
- 절차: PowerShell 명령어 복사 → 검색창에 powershell 실행(검은 화면) → 우클릭으로 붙여넣기 → 엔터로 설치.
- 설치 후 안내되는 위치(location)를 반드시 확인. `CD`(change directory) 명령을 외워 원하는 폴더로 이동. 폴더를 드래그앤드롭하면 경로가 자동 입력되고, 엔터로 위치가 바뀐 뒤 `claude` 실행.
- 데모에서는 바탕화면에 `study-cc` 빈 폴더를 만들어 이동 후 실행 → 폴더가 비어 "아무것도 없다" 응답.

### 5. 핵심 메뉴와 모드
- 슬래시(/) 명령들. **당의정에는 일부 명령이 없고 터미널엔 다 있음**: 로그인/로그아웃, clear, model, by the way, shift+space(모드 전환).
- 모델 설정(데모는 Opus 4.8), 컨텍스트 윈도우 조절, 빠른 모드(토큰을 더 씀 → 데모에선 끔).
- **by the way**: 작업이 길어질 때(길게는 7~10분) 중간에 끼어들어 다른 걸 먼저 시키는 명령.
- **모드**: 자동 / 계획 / 편집 승인 / 권한 요청.
  - 계획 모드: 계획만 세우고 코드 작성·파일 삭제 안 함.
  - 권한 요청 모드: 작업마다 물어봄(위험할 것 같을 때 유용).
  - 자동 모드: 확인 없이 알아서 실행. [의견] 결국 다 이걸 쓰게 됨(계속 물어보면 짜증나서). 대신 에이전트가 뭘 할지 예상하고 볼 줄 알아야 안전.
  - 당의정에서는 버튼/`shift+command+M`으로 모드 전환.

### 6. 스킬(Skill) 개념과 프론트매터
- [의견] 스킬은 "에이전트 다루기의 꽃"으로, 클로드·코덱스·커서 등 모든 에이전트 서비스에서 통용되는 정규 가이드 방법.
- 스킬 = 반복 작업을 위한 **매뉴얼(문서)**. 반복하고 있다는 느낌이 오면 스킬로 관리한다. 학습한 적 없는 작업도 스킬을 쓰면 많이 보완된다.
- 세팅 범위: 컴퓨터 **전역**과 특정 **프로젝트 폴더(지역)** 둘 다 가능. 클로드는 실행한 폴더 안의 스킬을 먼저 보고, 없으면 전역을 본다.
- **프론트매터(front matter)**: 스킬 파일 상단의 짧은 자기소개(이름 + "언제 쓰나"). 스킬의 인덱스 같은 것. 에이전트는 여러 스킬의 프론트매터만 먼저 훑어 적합한 스킬을 골라 실행한다.

### 7. 당의정 실습 — CEO 이메일 스킬
- Claude Desktop의 Code 탭 사용. 폴더 열기로 `study-cc` 지정, 자동 모드, 모델 Opus 4.8, 빠른 모드 off.
- "대표님께 정중한 메일 작성 환경을 세팅하고 싶다"고 상담 → 클로드가 폴더를 살핀 뒤 질답을 거쳐 **전용 스킬을 자동 생성**. 폴더 안에 숨김 폴더 `.claude`가 생기고 그 안에 `CEO 이메일` 스킬(SKILL.md)이 만들어짐(Mac은 `command+shift+.`로 숨김 폴더 보기).
- 스킬 내용: 프론트매터 + 양식 + 작성 절차 + 톤 + 상황별(보고/요청/사과·정정/감사) 문구 + 예시.
- 주의: 스킬을 만든 세션은 아직 스킬을 모름 → 슬래시에 안 뜸. **완전히 껐다 켜야**(command+Q) 등록됨.
- 사용: "매출 보고가 잘못된" 키워드만 줘도 정정·사과 메일을 형식대로 자동 작성. 수치가 없으면 "대충 입력"시킬 수 있고 이름/팀도 채움.
- 반복되는 이름·팀(에디터P / IT 모바일팀)은 스킬을 수정해 서명에 고정 → 이후 자동 서명. [의견] "반복 느낌이 오면 바로 스킬에 넣어라."

### 8. 지역 스킬 vs 전역 스킬
- `study-cc` 안에 만든 CEO 이메일 스킬은 **지역 스킬**이라, 전역(일반 Claude 채팅)이나 다른 폴더(예: 바탕화면)에서는 보이지 않는다.
- 그래서 이메일 작업은 `study-cc`에서, 코드 작업은 코드용 폴더에서 하는 식으로 **폴더로 스킬을 구분**해 쓴다.

### 9. 바이브 코딩 실습
- `study-cc-vibe-coding` 빈 폴더에서 진행. "아주 멋진 디자인의 엔터테인먼트사 랜딩 페이지를 화려한 오프닝과 함께 만들고, 초보라서 실행까지 해 확인하게 해 달라"고 지시.
- 클로드가 폴더에 파일을 만들고(토큰 대량 소진) 사이트를 띄움. 한글 폰트가 뚱뚱하게 나온 문제를 **화면 캡처를 붙여넣어** 지적 → 클로드가 수정 → 새로고침으로 확인. IDE/당의정은 좌우로 펼쳐 작업물을 보며 하는 게 포인트.

### 10. [의견] 바이브 코딩과 돈벌이는 무관 + 세일즈 장치
- [의견] 바이브 코딩으로 멋진 사이트·기능을 만드는 것과 돈을 버는 것은 아무 관계가 없다. 가게를 예쁘게 차려도 메뉴가 없거나 가게 존재를 알리지 못하면 사람이 돈을 안 쓴다.
- [의견] "바이브 코딩하면 누구나 돈 번다"고 하면 사기꾼. 돈은 세일즈(돈 쓰게 유도하는 노하우)로 버는 것이지 바이브 코딩 자체로 벌리는 게 아니다. 둘을 묶어 생각하지 말라.
- (세일즈 장치) 화자는 자신의 "바이브 코딩 책"을 언급하며, 그 책에 왜 이런 방식으로 접근하는지 다 설명해 놨다고 홍보.

## 3줄 오리엔테이션
- 클로드 코드를 실행하는 방법은 순정파(터미널)·아이디파(IDE)·멀티파(Cmux/Warp)·당의정파(Desktop Code 탭/VS Code 확장)로 나뉘지만, 본질은 전부 같은 클로드 코드다.
- 가장 중요한 원리 하나: 클로드는 "실행한 폴더를 중심 무대"로 활동하므로 어느 폴더에서 소환하느냐가 안전과 결과를 좌우한다.
- 이 편은 초보용 당의정(Claude Desktop Code 탭)으로 설치·핵심 메뉴·스킬(CEO 이메일 자동 생성)·바이브 코딩(랜딩 페이지)까지 실습하고, 터미널 중심은 다음(2번) 영상으로 넘긴다.

## 전문 — 시간순 워크스루

### [00:00–00:55] 시즌 3 오프닝, 코워크(Cowork)를 안 다루는 이유
시즌 3을 시작하며 클로드 코드 세팅·시스템 전반을 소개하겠다고 연다. 이 시리즈는 몇 편이 될지 모르지만 "클로드 코드 쓰는 법을 충분히 다뤘다" 싶으면 마치겠다는 계획. 코워크(Cowork)를 왜 안 하냐는 질문에는, 본인이 코워크를 주로 안 쓰고, 안 쓰는 걸 억지로 강의하는 건 웃기며, 어차피 클로드 코드를 쓰면 코워크까지 쓸 필요가 없다고 답한다. 하이엔드 도구인 클로드 코드부터 알려주고, 댓글로 요청이 오면 코워크 영상을 찍을지 고민하겠다고 한다.

### [00:55–01:16] 설치는 명령어 하나, 다만 실행 방법이 다양해짐
설치 자체는 명령어를 그냥 실행하면 끝이라 별게 없다. 다만 클로드 코드를 실행하는 방법이 굉장히 다양해졌기 때문에, 그 부분을 깔끔하게 설명한 뒤 함께 설치해 보겠다고 예고한다.

### [01:16–01:56] 공식 문서 찾기와 OS별 설치 명령어
docs 퀵스타트를 어렵게 찾을 필요 없이 구글에 "클로드 코드 인스톨"을 검색하면 맨 위에 공식 페이지가 나온다. Mac은 명령어를 쓰고, Windows는 PowerShell과 명령 프롬프트(CMD)에서 쓰는 명령어가 서로 다르다. 화자는 CMD를 추천하지만 이 영상에서는 PowerShell로 설치를 시연하겠다고 한다. 이 명령어를 복사해 실행하는 과정이 대부분의 사용자가 처음 만나는 셸 명령어이자 터미널과의 첫 대면이다.

### [01:56–03:07] 설치 없이 쓰는 대안 — Desktop Code 탭, VS Code 확장
명령어 설치가 싫으면 대안이 있다. Claude Desktop 가운데 끝에 있는 "코드(Code)" 탭이 곧 클로드 코드라서 그냥 써도 된다("이쁜 데서 쓰고 싶으면"). 또 VS Code를 아는 사람은 확장 프로그램으로 쓸 수 있다. VS Code 설치 후 왼쪽 확장 메뉴에서 "클로드 코드"를 검색해 Anthropic 파란 체크가 달린 것을 Install하면, 왼쪽에 꽃 모양 아이콘이 생기고 그걸 눌러 New Session으로 채팅창을 열어 쓰면 된다.

### [03:07–03:39] 결론 — 어디서 쓰든 근본은 같은 클로드 코드
방법마다 생긴 게 달라 다르게 느껴지지만 근본적으로는 같다. 터미널에서 하는 클로드 코드나 VS Code, Desktop Code 탭에서 하는 클로드 코드나 다 똑같다. 과거에는 똑같지 않았지만 이제는 똑같아졌다. 어떤 상황에서 어떤 걸 써야 하는지는 뒤에서 설명하겠다고 하고 넘어간다.

### [03:39–04:09] 순정파 — 터미널에서 실행
실행 방법에 파(派)가 나뉜다며 첫째로 "순정파"를 든다. 검은(화자 화면은 흰색) 터미널 화면에서 클로드를 실행하는 것이 순정파다.
> "클로드 코드는 맨 처음에 이 터미널에서 실행을 하는 게 정석이었습니다. 그래서 이걸 이제 저는 순정파라고 부릅니다." [04:03]

### [04:09–05:26] 아이디(IDE)파 — 통합 개발 환경 안에서
둘째는 "아이디파". IDE는 통합 개발 환경, 즉 개발에 필요한 도구들을 모아둔 프로그램이다. 대표 예가 VS Code로, 코드 수정·폴더 및 파일 탐색이 되고 아래 탭을 누르면 터미널도 열린다. 그 터미널은 순정파의 터미널과 똑같은 것이다. 다만 IDE 안에서는 창을 왔다 갔다 하지 않고 폴더·터미널·확장(클로드 코드)을 한 화면에 통합해 볼 수 있어 편하다. 이렇게 한군데 모아 보는 것을 IDE라 부른다.

### [05:26–07:04] 멀티파 — Cmux·Warp로 다중 세션
셋째는 "멀티파". 클로드 코드에 익숙해지면 폴더 A와 폴더 B를 오가며 폴더별로 클로드를 따로 운용하고 싶어지는데, 그때 창을 두 개 관리해야 해서 불편하다. 이를 위한 프로그램이 Cmux, Warp 등이다. 겉모습은 터미널과 똑같지만, 한 창에서 +를 누르면 터미널을 여러 개 열 수 있어, 각 폴더로 이동해 클로드를 실행하면 세션들을 한 창 안에서 관리하며 오갈 수 있다. 이것도 결국 검은 화면에 텍스트를 입력하는 터미널(CLI) 기반이다. (자막의 "시웍스/시목스/시머스"는 제목·문맥상 Cmux로 교정. 용어 정리는 나중에 하겠다고 함.)

### [07:04–09:02] 당의정파 — 예쁜 GUI로 감싼 클로드 코드
넷째는 "당의정파". 당의정은 쓴 알약 겉에 단맛(당)을 발라 먹기 쉽게 만든 것을 뜻한다. VS Code에서 꽃 모양을 누르면 뿅 나오는 예쁜 UI, 모델 버튼으로 모델을 바꾸고 볼륨 버튼 같은 직관적 요소들이 있어 쓰기 쉬워 보인다. Desktop Code 탭도 애니메이션이 있고 버튼만 눌러 모델을 바꿀 수 있어 편하다. 하지만 쓰는 방법과 본질은 터미널과 똑같은 클로드 코드다. 터미널이 어렵고 역겹게("디스거스팅") 느껴지는 사람을 위한 방식이다.
> "약을 먹기 편하게 만드는 거죠. … 약간 당의정을 바른 거죠." [07:14]

### [09:02–09:28] 당의정도 "가짜"는 아니다 — 자전거 보조 바퀴 비유
당의정을 쓴다고 "이런 것밖에 못 쓰는군요, 가짜다"라고 할 일은 아니다. 다만 그 안에서만 써야 해서 생기는 제약은 있다. [의견] 자전거 보조 바퀴처럼 중심 잡기엔 편하지만, 속도를 내는 자유로운 주행을 위해선 언젠가 보조 바퀴(당의정)를 떼야 한다. 끝까지 당의정이 편하면 그냥 써도 상관없다.

### [09:28–10:13] 순정파(터미널)의 장점 — 심플하고 어디에나 있음
[의견] 터미널은 모든 PC에 이미 깔려 있어 설치 안 된 PC가 없다. 별도 프로그램(Cmux/VS Code) 설치 없이 `claude` 하나만 실행하면 되니 심플하다. 여기저기서 쓰는 "원소스 멀티유즈"라 원리 이해에도 도움이 된다. 오늘은 간단히 여기서, 제대로 할 데선 따로 세팅하는 식.

### [10:13–11:26] 아이디파의 장점 — 파일을 눈으로 보며 작업
VS Code 터미널에서 클로드를 실행하면, 작업 폴더(예: 랩 hpx)를 열어 두었을 때 왼쪽에 파일이 보인다. "현재 폴더에 뭐가 있니?"라고 물으면 우리가 보는 것과 에이전트가 보는 것이 같음을 시각적으로 확인할 수 있어 편리하다. 에이전트에게 작업을 시킬 때 내 자료·파일을 보기 편한 환경에서 하고 싶으면 IDE로 오면 된다. 다만 여기서도 핵심은 터미널을 쓴다는 점이다.

### [11:26–14:24] 멀티파 실사용 — Cmux로 Gmail·로드맵 다중 작업
[의견/실사용] 화자는 업무에 클로드 코드 같은 에이전트를 매우 많이 쓴다. Cmux에서 `워크 GWS` 폴더로 이동해 주로 Gmail 작업을 한다. "최근 메일에서 중요한 개인정보는 똥똥으로 생략해 보여 줘. GWS 스킬을 사용해 메일을 읽어 봐"라고 하면, GWS(Google Workspace) CLI 명령을 통해 Gmail을 읽어 작업한다. 그 작업을 걸어 두고 새 터미널을 열어 에디터피 로드맵 사이트 작업을 별도로 진행하는 식으로 다중 세션을 돌린다. Cmux는 화면 분할로 웹브라우저(예: 네이버)까지 함께 열 수 있는 것도 장점. Warp도 좋은 비슷한 프로그램이지만 요즘은 Cmux를 많이 쓴다. Cmux는 껐다 켜도 이전에 하던 작업 세션이 살아 있는 장점이 있다.

### [14:24–15:35] 본질은 클로드 코드, 도구는 취향
앞서 소개한 것들의 본질은 다 "클로드 코드를 쓴다"는 것이고 추가 기능은 편한 걸 고르는 문제다. [의견] 무엇을 써야 클로드 코드를 잘 쓰는 것이라는 건 없다. 작업 책상 취향처럼, 깔끔한 순정을 좋아하는 사람도, 이것저것 갖춰야 잘 되는 사람도, 공통 세팅에서만 편히 쓰고 싶은 사람도 있다. 각자 장점만 다르다: 터미널은 심플, IDE는 결과물을 함께 보기, Cmux/Warp는 다중 실행, Desktop Code는 예쁘고 직관적이며 설치 불필요.

### [15:35–16:27] 추천 순서 — 당의정 → 터미널 → IDE → 멀티
오늘은 초보자를 위해 Desktop Code 탭으로 간단한 작업을 함께 해 본다. [의견] 추천: 어려우면 당의정(Code 또는 확장)으로 시작 → 터미널 욕구가 생기면 과감히 터미널로 이동 → 코드·파일 볼 일이 잦아지면 IDE → 작업이 많아 창을 여러 개 오가기 번잡하면 Cmux나 Warp. 이런 기준을 갖고 원하는 작업대에서 작업하면 된다.

### [16:27–17:35] 핵심 원리 — 실행한 폴더가 중심 무대
클로드 코드를 실행한다는 게 본질이므로 그 원리 하나만 기억하면 된다: 클로드 코드는 **사용자가 실행한 폴더를 중심으로 활동**한다. 소환해 둔 위치에서 주로 활동하기 때문에 위치가 중요하다. 바탕화면에서 실행하면 바탕화면의 수많은 파일을 다 보게 된다. 그래서 활동 범위를 어느 정도 지정하고 실행해야, 그 안에서 주로 활동하고 필요할 때만 바깥을 드나든다.

### [17:35–20:28] Windows 설치 실습 + 위치 경고(System32)
Windows로 이동해 실습. "클로드 코드 인스톨" 검색 → Windows PowerShell 명령어 복사 → 검색창에 powershell 실행(검은 화면) → 우클릭으로 붙여넣기 → 엔터로 설치. Windows는 설치 후 한 단계가 더 있으니 눈을 크게 뜨고 보라고 강조한다. 설치가 끝나면 설치 위치(location)를 알려주는데, 기본 실행 위치가 `C:\Windows\System32`라 그대로 클로드를 실행하면 시스템 파일들을 읽게 되어 위험하다("하나라도 잘못 수정하면 윈도우즈가 박살"). 그래서 폴더 이동 명령 `CD`(change directory)를 외워야 한다. CD + 띄어쓰기 + 이동할 폴더. [의견] 공부할 때마다 빈 폴더를 새로 만드는 습관을 들이라며, 바탕화면에 `study-cc` 폴더를 만들고 드래그앤드롭으로 경로를 자동 입력한 뒤 엔터로 위치를 옮긴다. 왼쪽 커서 위치가 곧 현재 폴더이며, `study-cc`로 이동한 상태에서 클로드를 실행해야 한다.

### [20:28–20:50] 빈 폴더 확인 후 Windows 실습 일시 종료
`study-cc`가 빈 폴더라 "뭐가 있냐" 물으면 "아무것도 없다"고 답한다. 이게 기본이지만 어렵게 느껴지는 사람이 있어 오늘은 당의정에서 하겠다고 한다. 터미널 방식은 따로 영상으로 준비해 올리겠다며 Windows 실습은 여기서 잠시 마친다.

### [20:50–23:24] 핵심 메뉴와 모드 — 슬래시 명령, by the way, 모드 4종
Desktop Code 탭으로 이동해 슬래시(/) 명령을 설명한다. 당의정에는 없는 명령이 여럿이다: 로그인/로그아웃(이미 Desktop에서 로그인됨), clear(새 대화로 지우면 됨), by the way, shift+space. 반면 터미널엔 다 있다. 모델 설정(Opus 4.8 등)은 엔터 두 번 연타로 고를 수 있고, 컨텍스트 윈도우 조절·빠른 모드도 가능하다. **by the way**는 클로드가 길게(7~10분) 작업하는 중에 끼어들어 "잠깐 이것부터 먼저 해 줄래?"라고 시키는 명령. **모드**는 shift+space(당의정은 `shift+command+M`/버튼)로 전환하며 자동·계획·편집 승인·권한 요청이 있다. 계획 모드는 계획만 세우고 코드 작성·파일 삭제를 안 한다. 권한 요청 모드는 작업마다 "해도 되나요?"를 묻는다(위험할 때 유용). [의견] 화자는 자동 모드를 쓰는데, 확인 없이 알아서 실행하며 결국 다들 이걸 쓰게 된다(계속 묻는 게 짜증나서). 대신 에이전트가 뭘 할지 예상하고 볼 줄 알아야 안전하다.

### [23:24–29:04] 스킬(Skill) — 에이전트 다루기의 꽃, 프론트매터 원리
[의견] 스킬은 클로드·코워크·코덱스·커서 등 모든 에이전트 서비스에서 통용되는 "에이전트 다루기의 꽃"이며, 잘 쓰면 에이전트를 잘 다루는 사람으로 보인다. 스킬은 일종의 매뉴얼(문서)로, 컴퓨터 **전역**에도, 특정 **프로젝트 폴더(지역)**에도 세팅할 수 있다. 클로드는 실행된 폴더 안에 스킬이 있는지 먼저 보고, 없으면 전역을 본다(같은 전역 스킬을 두 폴더에서 공유 가능하지만 무관한 폴더의 지역 스킬은 못 봄). 스킬 파일 상단에는 **프론트매터(front matter)**가 있는데, 스킬의 이름과 "언제 쓰나"를 짧게 적은 자기소개이자 인덱스다. 에이전트는 여러 스킬의 프론트매터만 먼저 훑어 상황에 맞는 스킬을 골라 실행한다. 스킬은 반복 작업을 할 때(예: 대표님께 늘 메일 쓰기), 작업 절차를 아주 디테일하게 적어두는 용도이며, 에이전트가 학습한 적 없는 작업도 스킬로 크게 보완된다.

### [29:04–33:24] 당의정 실습 (1) — CEO 이메일 스킬 자동 생성
Claude Desktop Code 탭에서 폴더 열기로 `study-cc`를 새로 지정하고, 자동 모드·모델 Opus 4.8·빠른 모드 off(토큰 절약)로 세팅한다. "대표님께 정중하게 메일 작성하는 게 어려워서 이 폴더에서 메일 작성 환경을 세팅하고 싶은데 어떻게 할까?"라고 상담한다. 클로드는 먼저 `study-cc` 폴더를 살핀 뒤, 부족한 정보를 질답으로 채운다. 대표님께 늘 메일을 쓰니 **전용 스킬**을 만들자고 제안하고, 보고·요청·승인·감사·축하·사과·정정·양해 같은 상황을 모두 체크한 뒤 "격식 있고 깍듯한" 톤을 반영한다. 그 결과 폴더 안에 숨김 폴더 `.claude`가 생기고, 그 안에 `CEO 이메일` 스킬(SKILL.md)이 작성된다(Mac은 `command+shift+.`로 숨김 폴더 표시). 스킬 파일 상단이 프론트매터이고, 아래에 양식·작성 절차·톤·상황별 문구·예시가 디테일하게 들어 있다.

### [33:24–36:04] 당의정 실습 (2) — 스킬 등록(재시작)과 사용, 서명 고정
스킬을 만든 세션은 아직 그 스킬을 모른다(만든 주체일 뿐 세팅 반영 전). 슬래시에 CEO가 안 뜨는데, 새 세션만으론 부족하고 클로드를 완전히 껐다(command+Q) 켜야 등록된다. 재시작 후 슬래시로 검색하면 `CEO 이메일` 스킬이 뜨고, 프론트매터 그대로 "이럴 때 쓰는 스킬"이 보인다. "매출 보고가 잘못된" 정도의 키워드만 줘도 스킬을 참조해 정정·사과 메일을 형식대로 자동 작성한다. 수치가 없으면 "대충 입력"시킬 수 있고 이름(에디터피)·팀(IT 모바일팀)도 채운다. [의견] 매번 같은 이름·팀을 넣는 반복이 보이면 바로 스킬을 수정해 기본 서명으로 고정하라. 실제로 "이름과 팀을 스킬에 넣어 줘"라고 하니 스킬 끝맺음에 서명이 추가되고, 이후 `CEO 스킬 + 5월 매출 보고`만 줘도 서명이 자동으로 들어간다.

### [36:04–37:12] 지역 스킬 vs 전역 스킬 — 폴더로 구분
`study-cc` 안에 만든 CEO 이메일 스킬은 **지역 스킬**이다. 전체 컴퓨터 영역에서 보면 바탕화면 아래 `study-cc`에만 세팅돼 있어, 일반 Claude 채팅(전역)이나 다른 폴더(예: 바탕화면으로 폴더를 바꾸면)에서는 CEO 스킬이 보이지 않는다. 이렇게 지역/전역이 나뉘므로, 이메일 작업은 `study-cc`에서, 코드 작업은 코드용 폴더에서 하는 식으로 폴더로 스킬을 구분해 쓰면 된다.

### [37:12–39:36] 바이브 코딩 실습 — 랜딩 페이지 생성과 캡처 수정
`study-cc-vibe-coding` 빈 폴더를 만들어 진행. "아주 멋진 디자인의 엔터테인먼트사(예: 레비) 랜딩 페이지를 화려한 오프닝과 함께 만들어 줘. 나는 초보라 네가 알아서 만들고 실행까지 해 확인하게 해 줘"라고 디테일하게 지시한다. 어느 폴더에서 실행되는지 확인하는 게 중요하다며, 빈 폴더에 파일들이 생기는 과정을 보여준다(토큰 대량 소진). 파일이 하나 생기고 오프닝과 함께 사이트가 뜬다. 다만 한글 폰트가 너무 뚱뚱해 표현이 안 됐다. 이때 화면을 캡처해 Code 탭에 붙여넣고 "한글 폰트가 너무 뚱뚱해서 이상하잖아"라고 지적하면 클로드가 수정한다. 화면을 붙여 좌우로 펼쳐 보며 작업하는 게 편하다.

### [39:36–40:27] [의견] 바이브 코딩 ≠ 돈벌이, 세일즈가 별개
[의견] 이런 걸 만들며 "누구나 돈 벌 수 있다"고 하면 사기꾼이다. 바이브 코딩과 돈 버는 것은 아무 관계가 없다. 바이브 코딩은 그냥 프로그래밍일 뿐 판매(수익화)와 무관하다. 가게를 아무리 예쁘게 차려도 메뉴가 없거나 가게 존재를 사람들이 몰라 오지 않으면 돈을 안 쓴다. 바이브 코딩으로 돈 번다는 사람은 실제로는 그것을 돈 버는 쪽으로 연결하는 노하우(세일즈)를 가진 사람이다. 멋진 사이트·기능을 만든다고 돈이 벌리는 게 아니라, 누군가 그것을 알아야 하고 돈 쓰게 유도할 수 있어야 한다. 세일즈와 바이브 코딩을 묶어 생각하지 말라.

### [40:27–41:10] 수정 확인 + 바이브 코딩 책 언급(세일즈)
수정이 끝나면 새로고침을 하라고 한다. "왜 새로고침해야 하는지, 어떤 사람은 가만히 있어도 바뀌던데"라는 의문에 대해선 (세일즈 장치) 자신의 "바이브 코딩 책"을 공부하라며, 그 책에 왜 이런 방식으로 접근하는지 다 설명해 놨다고 홍보한다. 새로고침하면 멋진 오프닝과 함께 폰트가 수정되어 한글이 잘 보이고, 아티스트·월드투어·"뉴 핫 넘버원" 같은 요소들이 나타난다.

### [41:10–42:05] 마무리 — 오늘 요약과 다음(2번) 영상 예고
오늘은 기본적인 클로드 코드 배경지식을 다 설명하고, 초보자가 좋아하는 예쁜 당의정 프로그램(Claude Desktop Code 탭)으로 스킬 생성과 바이브 코딩까지 실습했다. 다음 2번 영상에서는 결국 와야 하는 터미널을 중심으로, 오늘 한 스킬·전역/지역 사용법·바이브 코딩을 다시 쭉 보여주겠다고 예고하며 마친다.
