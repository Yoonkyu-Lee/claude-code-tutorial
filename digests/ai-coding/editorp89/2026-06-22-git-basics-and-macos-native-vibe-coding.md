## 요약 (주제별)

### 실행 환경 — Claude Desktop 코드 탭 vs 터미널
- 지난 회차 요약: Claude Code 실행법은 결국 두 갈래다. (1) Claude Desktop 안의 **코드 탭**, (2) **터미널**(iTerm 같은 것, 일반 터미널, VS Code/Cursor 안에서 여는 터미널 탭 등).
- 코드 탭은 반드시 Claude Desktop을 통해야 하므로 자율성이 떨어진다. 터미널은 여기저기(iTerm·기본 터미널·VS Code 터미널)에서 쓸 수 있어 다양성이 커서, [의견] 화자는 "웬만하면 결국 터미널을 쓰는 게 좋다"고 본다.
- 다만 이번 회차는 입문이라 Claude Desktop 코드 탭으로 바이브 코딩을 진행한다.

### 바이브 코딩 3원칙과 개발 지식의 필요성
- [의견] 화자 개인 정의: 바이브 코딩 3원칙 = **시키고 → 실행하고 → 문제 있으면 수정** 반복. 이 사이에 고수들의 다양한 기법이 있으나 이번엔 가장 단순한 방법만 쓴다.
- 용어 주: [의견] 화자는 에이전트로 코딩하니 "에이전트 코딩"이 맞다고 보지만, 아무도 그 말을 안 써서 통용어인 "바이브 코딩"을 쓴다. (용어는 계속 변화 중)
- [의견] 정설에 가까워지는 것: 개발 지식이 없어도 할 수는 있지만 있으면 더 좋다. 근거 사례로 "얼마 전 정부 지원 사업 사이트가 다 털린 사건"을 든다 — 개발 지식이 없어서라기보다 **당연히 해야 할 보안 조치를 안 해서** 터진 사고라고 본다. 최소한 "보안을 챙겨야 한다"는 인지는 있어야 에이전트와 함께 조사라도 한다는 것.
- 하네스 엔지니어링: [의견] 화자는 이 용어가 너무 어렵다고 보아, "AI가 동작할 환경을 먼저 만들어 주면 그 안에서 훨씬 잘한다"로 풀어 설명. 이건 거의 기본이라고 본다.

### Git 3개념 — 저장/이동/분기 (게임 비유)
- [의견] 화자는 Git을 "저장하기·이동하기·분기하기" 세 가지로만 이해하라고 제안. 게임의 세이브 비유(보스 잡기 전 저장 → 실패하면 돌아와 재시도).
- **저장하기 = 커밋(commit)**: 현재 프로젝트 폴더 전체 상태를 사진처럼 "찰칵" 스냅샷으로 기록. "커밋해줘"라고만 해도 에이전트가 저장으로 이해한다. 커밋에는 메시지를 남길 수 있다(예: "테스트 성공, 로고 이미지 등록 완료"). 커밋마다 **해시**(세이브 포인트의 ID 같은 것)가 붙는다. 비유로 영화 "나비효과"에서 사진/일기를 보고 그 시점으로 돌아가는 것.
- **이동하기 = 체크아웃(checkout)**: 특정 커밋 시점으로 앞뒤 이동. "그 커밋 시점으로 이동해라"라고 말로 시키면 된다. 세이브 포인트로 되돌아가는 것뿐 아니라 원하는 지점으로 자유롭게 왔다 갔다. (더 깊이 들어가면 복잡하나 이 수준이면 충분)
- **분기하기 = 브랜치(branch)**: 현재 시점에서 타임라인을 쪼개 가지를 침. 마블 "엔드게임"의 타임라인 분기 비유. 실험적 작업(예: 디자인 갈아엎기)을 할 때, 가지를 쳐서 하다가 마음에 안 들면 **가지째 버리고**, 마음에 들면 메인 가지로 **합치기(merge)** 한다. 분기는 원하는 만큼 여러 개 만들어 왔다 갔다 할 수 있다.
- 커밋+이동만 써도 충분하고, 브랜치는 관리가 편해서 실험적 작업 하나 떠올랐을 때 쓰면 좋은 선택지다.

### 왜 Git이 필수인가 — 컨텍스트 윈도우와 복구 불가
- [의견] Git 없이 하는 건 안전벨트 안 매고 자율주행하는 것과 같다. 사고 나면 복구가 안 된다.
- 사례: 화자 지인이 바이브 코딩 중 실수로 다 지웠는데 복구 방법이 없었다. 지운 것을 사람이 기억할 리 없고, **에이전트도 컨텍스트 윈도우가 살아 있을 때만 기억**한다. 대화를 나갔다 오거나 컨텍스트가 비워지면 바깥의 것은 다 잊으므로 "기억시켜서 되돌리면 되지 않나"는 성립하지 않는 질문이다.

### CLAUDE.md와 스킬(Skill)
- **CLAUDE.md**: 해당 폴더에서 Claude Code가 동작할 전체 방향성을 잡아 주는 그냥 문서 파일. 예로 "앞으로 중국어로만 말해라"라고 적으면 중국어로 답한다. **200줄 미만** 관리가 베스트로 알려져 있고, 길게 쓰는 건 요즘 권장 방식이 아니다. 디테일한 지시는 스킬로 빼고, 스킬 목록을 CLAUDE.md에 알리는 게 정석.
- **스킬**: 전역/지역으로 나눠 관리(게임 스킬 비유 — 다른 게임의 스킬이 섞이면 혼란스러우니 지역으로 분리, 전체 반영은 전역). CLAUDE.md가 문서 쪼가리라면, 스킬은 문서+예시 코드+이미지 등 작업 달성 수단을 가리지 않고 묶어 놓은 것으로 **실제로 압축 파일**이다.
- 스킬이 최적인 경우: (1) 에이전트가 아예 모르는 것 가이드(지식 커트오프 이후·새 애플리케이션 사용법 등), (2) 반복 작업(어제도 오늘도 같은 걸 시킬 때), (3) 아주 디테일한 지시가 필요해 에이전트가 갈피를 못 잡을 때("이거 시키면 무조건 이 스킬 써서 A→B→C로 개발").
- 스킬은 뉘앙스만으로 대충 시켜도 알아서 만들어 주고, 만든 걸 쓰라고 해도 잘 쓴다.

### 실습 목표 — macOS 네이티브 계산기 앱
- 이번 실습은 간단한 **macOS 네이티브 계산기 앱**을 바이브 코딩으로 만든다. macOS 앱 만드는 방법은 여러 가지(React Native, Electron 등 — 화자의 오마이오픈스냅/오마이그리드 같은 앱). 이번엔 상단 상태 표시·업데이트 확인·단축키 설정 등 네이티브하게 동작하는 앱을 목표로 한다.
- 화자가 GitHub에 올린 "macOS의 블루프린트"라는 CLAUDE.md + 스킬 세트를 clone해 프로젝트 폴더에 세팅하고 진행. 구조는 `.claude/skills/`(macOS 블루프린트 스킬) + `CLAUDE.md` + `README`. (세일즈 구조로서 한 번 기록: 시청자가 그대로 받아 쓸 수 있게 GitHub 링크로 공유하는 자료 배포)
- 주의(화자 고지): 이 CLAUDE.md/스킬은 화자가 직접 쓴 게 아니라 Claude가 작성한 것이라 완전히 정확하진 않다. 나중에 다듬어 다시 올릴 예정. 영상 중 에이전트가 "오마이오픈스냅 저장소"라고 잘못 답하는데, 이는 화자가 잘못 적어 올린 탓이라고 밝힘. `[불명확]` 남기지 않고 화자가 오류로 인정한 부분.

### 실습 진행 흐름과 곁다리 작업
- Claude Desktop 열고 **폴더 열기**로 새 작업 폴더 지정(`study/cc-calculator` 계열). 빈 폴더에 GitHub 저장소 URL을 복사해 "이 프로젝트 폴더에 세팅해라"라고 시키면 clone + Git 초기화된 프로젝트 폴더가 생성된다(이 시점에 이미 Git 사용 시작 — 초기화 상태).
- 새 대화를 열면 Git 표시가 나오고, "이 폴더는 무슨 작업 폴더냐" 물으면 CLAUDE.md와 내부를 읽고 답한다.
- 지시: "오마이오픈스냅 만들 건 아니고, 그 청사진(CLAUDE.md·스킬)을 활용해 간단한 macOS 계산기를 만들어 볼래?" — 이때 **릴리즈는 안 함**(업데이트 확인 기능이 붙는데 원하는 사람이 없을 듯해서 생략).
- 에이전트는 스킬/CLAUDE.md를 읽고 Swift 5, AppKit 등으로 계산기를 개발. 화자는 그동안 ChatGPT로 앱 **로고 이미지**를 생성("macOS 앱 공부 중, 간단한 계산기, 심플한 앱 로고 만들어줄래?")하고, 픽셀메이터 프로(약 5만 원, 1회 구매 영구 사용 — Adobe 대안)로 **배경 투명화** 처리해 앱 아이콘으로 등록. 작업 중 채팅을 넣으면 큐(queue)에 쌓여 위 작업이 끝난 뒤 실행된다.
- 앱은 `build/` 안에 만들어지고, 로고 이미지 위치를 에이전트에게 알려 아이콘 등록. 처음엔 아이콘이 안 떠서 껐다 다시 실행하기를 "될 때까지" 반복해 성공.

### Git 실전 시연 — 커밋 → 체크아웃 왕복 → 브랜치 → 머지 → DMG
- **커밋 시연**: 테스트가 잘 되면 "테스트 성공, 로고 등록 완료" 메시지로 커밋. 이때 `.gitignore`가 자연스럽게 등장 — 커밋에서 제외할(용량 크거나 불필요한) 파일을 골라 핵심 소스만 남기게 에이전트가 처리(맡기면 됨). 푸시(push)는 GitHub 온라인 저장용인데 이번엔 다루지 않음(별도 영상 예고).
- **커밋 이력/체크아웃**: "커밋 이력 목록 보여줘" → 최초 커밋 + 방금 커밋 두 개. 해시를 지목해 "여기로 돌아가자"(체크아웃). "커밋 취소가 아니라 이동만 하겠다"고 지시하면 build 제외 초기 상태로 되돌아감. 다시 두 번째 커밋으로 왕복하며 폴더 구조가 실시간 바뀌는 걸 시연.
- 곁다리 이슈: 되돌아왔을 때 "untracked"가 남는 것 관련해 에이전트와 문답. `.gitignore`가 그 커밋 시점에 없어서(생긴 시점 차이 때문에) 생긴 오해였고 "지금은 없다(정상)"로 정리. 화자 曰 이해 안 되면 무시해도 됨.
- **브랜치→머지**: "브랜치 하나 만들어라" → 에이전트가 `redesign` 브랜치 생성(타임라인 분기). "앱 디자인을 아주 독특하게 바꿔 실행해봐" → 신스웨이브/네온 레트로 퓨처 테마로 확 바뀜. 에이전트가 "디자인 잘못돼도 `git switch main`으로 안전 복귀 가능"이라 안내. 시각적으로 그래프(커밋·브랜치)를 보여 달라고 하면 시각화해 줌. redesign 변경분은 처음엔 커밋 안 된 상태 → "커밋해줘"로 커밋. [의견] 화자는 솔직히 그 디자인이 마음에 안 들지만 "마음에 든다" 가정 하에 메인으로 **머지**.
- **DMG 패키징**: "앱 설치용 DMG 만들어볼래?" → 에이전트가 DMG 패키징 생성(`dist/`에 위치). 더블클릭 → 드래그앤드롭 설치. 혼자 쓸 macOS 앱을 이렇게 배포 파일까지 만드는 것으로 실습 마무리.

### 마무리 메시지
- macOS 앱을 바이브 코딩으로 만들고 싶던 사람은 오늘 공유한 CLAUDE.md/스킬로 앱을 개발해 보고, 간단한 Git 3원칙(저장/이동/분기)을 쓰면서 진행하라. 짜잘한 문제는 에이전트와 함께 추가 공부하며 정확한 개념 사용법을 익히길 권함.
- [의견] 오늘로 Claude Code의 코드 탭 사용 기본은 다 다뤘다고 보며, 다음 영상부터는 (터미널의) Claude Code로 돌아온다고 예고.

코드 탭은 반드시 Claude Desktop을 통해야 해서 자율성이 조금 떨어지는 반면, 터미널은 iTerm·일반 터미널·VS Code 터미널 등 여기저기에 있어 다양성이 크다. [의견] 그래서 웬만하면 나중에 결국 터미널을 쓰면 좋겠다는 게 지난 회차 요약이었다. 다만 오늘은 입문이라 Claude Desktop 코드 탭으로 바이브 코딩을 진행한다.

### [01:14–02:05] 바이브 코딩 3원칙 · 용어 이야기
[의견] 바이브 코딩의 핵심은 화자만의 얘기라 걸러 들어도 된다는 전제로, "바이브 코딩"이라는 말 자체가 이미 옛말이 되어 간다고 언급. [의견] 화자는 에이전트로 코딩하니 "에이전트 코딩"이 맞다 보지만 아무도 그 말을 안 써서 대표적으로 통용되는 "바이브 코딩"을 쓴다(용어는 계속 변화). **3원칙 = 시키고 → 실행하고 → 문제 있으면 수정**을 반복. 이 사이에 고수들의 다양한 기법이 있으나, 오늘은 처음이니 아주 단순한 방법으로만 간다.

### [02:05–02:52] 개발 지식이 있으면 더 좋다 — 보안 사례
[의견] 정설에 가까워지는 것: 개발 지식이 없어도 할 수는 있지만 있으면 더 좋다. 근거로 "얼마 전 정부 지원 사업에서 사이트가 다 털려 난리 난 사건"을 든다. [의견] 개발 지식이 없어서가 아니라 **당연히 해야 할 것(보안)을 안 해서** 터진 보안 사고라고 본다. 최소한 "보안을 안 지키면 큰일 난다"는 인지라도 있어야 에이전트와 함께 조사라도 한다. 그런 지식이 있으면 더 잘하고, 없으면 에이전트가 운 좋으면 챙겨 주지만 안 챙기면 큰일 나니 있으면 더 좋다.

### [02:52–03:26] 하네스 엔지니어링 = 실행 환경 만들어 주기
에이전트의 실행 환경을 잘 만들어 주면 더 잘한다 — 최근 "하네스 엔지니어링"이라는 말인데, [의견] 화자는 너무 어려운 말이라 보아 "AI가 뭔가 하는 환경을 우리가 먼저 만들어 놓으면 그 안에서 훨씬 잘 동작하니 챙겨 주는 게 좋다"로 푼다. 거의 기본이다. 오늘은 CLAUDE.md·스킬의 기본 내용을 다 말해 주겠다고 예고. 다만 이런 세팅은 필요할 때마다 도입하는 게 맞지, 처음에 다 세팅해 놓고 간다는 건 말이 안 된다고 덧붙임.

### [03:26–04:11] Git 도입 — 저장/이동/분기, 게임 비유
[의견] 화자는 시청자가 지식 하나 더 얻어 가길 바라며 Git을 도입. 개발자가 억지로 떠먹인다는 반감을 의식하면서, 기술을 어렵게 볼 것 없이 **저장하기·이동하기·분기하기** 셋만 말한다. 게임 비유: 보스 잡기 전 저장하고, 실패하면 돌아와 재시도. 원래 앞의 두 개(저장·이동)만 써도 되지만 Git엔 특이하게 **분기(브랜치)** 기능이 하나 더 있다. 타임라인 그림으로 설명 시작 — 현재 지점에서 개발을 시작해 불안할 때마다 세이브하며 진행한다.

### [04:11–05:11] 분기(브랜치)를 언제 쓰나
현재 지점에서 분기를 칠 수 있다 — 마블 "엔드게임"에서 닥터 스트레인지가 타임라인을 건너가 작전 수행하는 장면 비유. 원래 시간선은 유지되고 분기한 쪽만 따로 나아간다. 언제 좋은가: 계속 앞으로 가고 싶은데 불안해서, 잘 안 됐을 때 그 통째로 날려 버리고 싶을 때. 그냥 세이브하며 가면 돌아올 때 발자취(세이브 포인트)가 남는데, 그 흔적을 남기고 싶지 않고 "안 되면 다 지워야지" 싶을 때 분기해서 가는 게 맞다. 분기는 수없이 많이 할 수 있어, 쓸 줄 알면 좋아서 여기까지 설명한다.

### [05:11–05:55] Git 안 쓰면 = 안전벨트 안 맴 · 컨텍스트 윈도우
[의견] 바이브 코딩에서 이 기술(Git)을 안 쓰는 건 안전벨트 안 매고 위험하게 자율주행하는 것과 같다 — 사고 나면 복구가 안 된다. 사례: 화자 지인이 바이브 코딩 중 실수로 다 지웠는데 복구 방법이 없었다. 지운 걸 사람이 기억할 리 없고, 에이전트도 **컨텍스트 윈도우가 살아 있을 때만** 기억한다. 대화를 나갔다 오거나 컨텍스트가 비워지면 다 까먹으므로 "기억시켜 되돌리면 되지 않나"는 성립하지 않는 질문이다. 그래서 Git을 쓰는 게 좋고, 저장/이동/분기 셋으로만 생각하면 된다.

### [05:55–06:47] 저장하기 = 커밋
저장하기는 전문용어로 **커밋(commit)**. 정확히는 커밋 외에도 더 있지만 커밋만 얘기한다 — "커밋"이라고만 해도 에이전트가 저장으로 이해해 준다. 커밋은 현재 상황을 사진 찍듯 **스냅샷**으로 딱 찍어 그 상태를 기록으로 남기는 것. 이후 그 사진을 보고 그 시점으로 돌아갈 수 있다. 비유: 영화 "나비효과"에서 주인공이 사진·일기를 바라보며 과거로 돌아가듯, 커밋한 시점으로 정확히 돌아갈 수 있다. 그러니 이후 뭔가 망가져도 커밋이 하나라도 있으면 거기로 돌아갈 수 있다.

### [06:47–07:35] 이동하기 = 체크아웃
커밋은 사진만 찍는 것(세이브 포인트 남기기)이고, 돌아가는 것은 **체크아웃(checkout)** — 정확히는 그 시점으로 앞이든 뒤든 **이동**하는 것. 세이브 포인트라 하면 이전으로만 돌아가는 걸 떠올리지만, 돌아온 뒤 그 이후 커밋들로 다시 이동해 앞뒤로 원하는 대로 왔다 갔다 할 수 있는 게 체크아웃이다. 용어가 입에 안 붙으면 그냥 "특정 커밋 시점으로 이동해라"라고 시키면 에이전트가 체크아웃으로 알아듣고 해 준다. 더 깊이 들어가면 복잡하니 이 정도만 알아도 충분히 다 쓸 수 있다.

### [07:35–09:11] 분기하기 = 브랜치, 메인 브랜치와 머지
분기하기는 영어로 **브랜치(branch)** = 가지를 새로 친다는 뜻. 앞으로 계속 나가고 싶은데 여기까지 개발한 게 마음에 들고, 이후에 실험적인 걸(예: 웹사이트 디자인 싹 갈아엎기) 도전하고 싶어졌다면 — 디자인 갈아치우다 완전 박살 날 수 있어 불안하니 일단 가지를 친다. 망하면 가지를 버리고, 잘 되면 메인 가지로 합칠 수 있다. 그림 설명: 커밋하며 나아가는 **메인 브랜치**가 있고(Git은 첫 커밋 순간 가지 하나를 만들어 줌), 어느 지점에서 브랜치를 치면 그 이후부터 새 가지가 생겨 거기서 커밋하며 진행한다. 디자인 개선이 잘 끝나 마음에 들면 그 가지를 메인에 합쳐 넣고, 다시 메인으로 이동해 작업을 이어 간다.

### [09:11–09:59] 머지 정리 · 브랜치는 선택
합치는 것을 **머지(merge)**라 한다. 브랜치를 쳐서 진행하면 마음에 안 들 때 버리고, 마음에 들면 메인으로 합쳐 나아갈 수 있다. 실험적 작업(메인 기능 완료 후 디자인도 바꿔 보고 새 기능도 해 보고 싶을 때)은 브랜치를 여러 개 만들어 왔다 갔다 하며 개발하다 마음에 들면 합친다. 복잡하면 브랜치는 안 쓰고 **커밋+이동만** 해도 충분하다. 실험적인 것 하나 떠올랐을 때 브랜치를 써 보면 관리 면에서 편해서 소개한 것.

### [09:59–10:37] CLAUDE.md 설명
CLAUDE.md는 영상에서 자주 언급한 것으로, 지금 이 폴더에서 클로드 코드가 동작할 **전체 방향성을 잡아 주는** 역할. **200줄 미만**으로 관리하는 게 베스트로 알려져 있다. 그냥 문서 파일이라 "앞으로 이렇게 동작해라", "중국어로만 말해라"라고 적으면 방향성이 잡혀 그대로 따른다(중국어로 대답). 다만 길면 좋지 않아 길게 쓰는 건 요즘 권장 방식이 아니다. 디테일한 지시는 (다음에 나올) **스킬**로 하는 게 좋고, 스킬들의 목록을 CLAUDE.md에 알리는 게 정석이다.

### [10:37–12:05] 스킬 — 전역/지역, 압축 파일, 쓰임새
스킬은 **전역/지역**으로 나눠 관리(게임 스킬 비유 — 서로 다른 게임의 스킬이 섞이면 혼란스러우니 지역으로 분리, 전체 반영되는 건 전역). CLAUDE.md가 문서 쪼가리라면, 스킬은 문서 + 예시 코드 + 이미지 등 작업 달성 수단·방법을 가리지 않고 리소스를 묶어 놓은 것으로 **실제로 압축 파일**이다. 최적 쓰임: (1) 에이전트가 아예 모르는 것 가이드 — 지식 커트오프(학습 날짜)까지만 알고 이후는 외부 데이터/검색으로 보완해야 하므로, 누가 새 애플리케이션을 만들었고 그 사용법을 알려줄 때 스킬로 하면 잘 동작. (2) 반복 작업 — 어제도 오늘도 같은 걸 하면 스킬로 만들어 두고 "이 스킬 활용해 진행해라". (3) 아주 디테일한 지시가 필요해 에이전트가 갈피를 못 잡을 것 같을 때 — "이거 시키면 무조건 이 스킬 써서 A→B→C로 개발해".

### [12:05–12:25] 스킬 사용 요령
스킬은 에이전트가 작업할 때 아주 디테일하게 따라야 할 요소들을 다 모아 압축 파일로 만든 것. 대충 뉘앙스로만 시켜도 알아서 해 준다 — "스킬 만들어"라고 해도 잘 만들고, 만든 걸 쓰라고 해도 잘 쓴다. 실습하며 보여 주겠다고 예고.

### [12:25–13:16] 실습 목표 — macOS 네이티브 앱
오늘 바이브 코딩으로 만들 것은 간단한 **macOS 앱**. macOS 앱을 만드는 방법은 여러 가지 — 웹 기술의 React Native, 혹은 Electron(화자가 소개했던 것들이 Electron으로 만든 예). 오늘 만들 앱은 화자가 얼마 전 공유한 **오마이오픈스냅**, **오마이그리드** 같은, 상단에 상태 띄우고 업데이트 진행하고 단축키 설정하는 등 macOS 네이티브하게 동작하는 앱류. 이런 걸 가이드하는 CLAUDE.md·스킬을 오늘 공유하고, 그걸로 간단한 macOS 앱을 만들어 본다.

### [13:16–13:56] 계산기 앱 · Git 실습 예고 · 정확성 고지
오늘 만들 앱은 아주 간단한 **계산기 앱**. 다만 가이드 자료(CLAUDE.md·스킬)는 화자가 만든 게 아니라 **Claude가 만든 것**이라 완전히 정확하진 않고, 나중에 읽어 보고 다듬어 다시 올릴 예정. 이 자료를 GitHub에 올려 시청자가 바로 받아 작업 시작할 수 있게 하겠다고 함(세일즈 구조: 자료 배포). 만드는 과정에서 불안할 때 Git의 **커밋·체크아웃·브랜치**를 써 보는 가이드도 함께 — 다만 살짝 억지로 끼워 만드는 거라 완전히 적합하진 않다고 미리 고지.

### [13:56–15:16] 프로젝트 폴더 세팅 — clone
Claude Desktop을 열고 **폴더 열기**로 작업 폴더를 지정(study 아래 `cc-calculator` 계열 이름). 빈 폴더 상태로 진행. GitHub에 올린 "macOS의 **블루프린트**" 저장소를 받아 세팅하는 걸 보여 줌 — 구조는 `.claude/skills/` 안의 macOS 블루프린트 스킬 하나 + CLAUDE.md + README. CLAUDE.md에는 화자가 오마이오픈스냅을 만들 때 설정했던 환경(업데이트 확인 누르면 최신인지 검사해 보여 주기, 설정 진입 등)을 구성해 놓았다. 이건 Claude가 적어 준 것을 그대로 갖다 쓴다.

### [15:16–15:47] URL 복사 → 세팅 지시 → Git 초기화
저장소의 **Code** 버튼을 눌러 나오는 주소를 복사해, Claude Code(코드 탭)로 가서 "이 프로젝트 폴더에 이걸 세팅해라"라고 시킨다. 그러면 알아서 세팅해 준다. 이 프로젝트 폴더에 Git 세팅을 하므로 **이미 Git을 쓰기 시작**한 것 — clone으로 가져온 뒤 커밋까지는 아니고 **초기화된 상태**의 프로젝트 폴더가 만들어진다.

### [15:47–16:32] 세팅 확인 · CLAUDE.md 자동 로드 · 에이전트 오답 고지
실제로 study의 calculator 폴더에 들어가 보면 아까 본 구성이 들어와 있다. 이 클로드 코드는 이 폴더에서 작업할 때 **무조건 CLAUDE.md를 읽고** 동작한다. 새 대화를 열면 Git이 만들어졌다는 표시가 나오고 이미 시작된 상태. "이 폴더는 어떤 작업을 위한 폴더냐" 물으면 CLAUDE.md와 내부를 읽고 답하는데, 에이전트가 "오마이오픈스냅의 저장소"라고 답한다. 이는 **오마이오픈스냅이 아니라 그게 있던 환경일 뿐**이라 잘못된 답이며, 화자가 잘못 적어 올린 탓이라 밝힘(다듬어 다시 올릴 예정). 요점은 지금 "스캐폴딩(청사진)만 있는 상태"라는 것.

### [16:32–17:18] 계산기 만들기 지시 · 릴리즈 생략
"나 오마이오픈스냅 만들 건 아니고, 그 청사진 자료(CLAUDE.md·스킬)를 활용해 간단한 **macOS 계산기**를 만들어 볼래?"라고 지시. 이때 **릴리즈는 안 함** — 릴리즈하면 업데이트 확인 기능이 동작하는데 [의견] 원하는 사람이 과연 있을까 싶어 없이 진행. 에이전트는 CLAUDE.md의 방법론(오마이오픈스냅 기준의 macOS 개발 방식)과 스킬을 보고 간단한 계산기를 만들어 준다. 만들고 나면 개발이 끝나고, 이후 커밋을 시켜 보고 체크아웃(이동)을 시연할 계획.

### [17:18–18:00] 커밋 vs 푸시 · 개발 시작
커밋을 시키는 데 두 가지 의미가 있는데 앞서 완전히 설명 안 했다며 보충: **커밋은 내 PC 안에서만** 세이브 포인트를 만드는 것. **푸시(push)**도 알아야 하지만 오늘은 Git(로컬)만 쓰고 GitHub는 안 다루니 푸시는 설명 생략(GitHub는 나중에 별도 영상 예고). 에이전트는 스킬이 스캐폴딩임을 정확히 인식하고 "이 스킬대로 하면 계산기를 만들 수 있겠다"며 진행. 마치 애플리케이션 하나를 만들어 준다.

### [18:00–19:00] 곁다리: 로고 이미지 생성 · 투명화
화자는 그동안 ChatGPT로 앱 **로고 이미지**를 만든다 — "macOS 앱 만들기 공부 중인데 간단한 계산기 만들고 있어, 심플한 앱 로고 이미지 만들어줄래?". 지금 PDF 교재도 하나 만들고 있는데 로고가 없어 엑박(깨진 이미지)으로 떠서, 그 배지가 싫어 로고를 만든다고. 에이전트가 계산기를 잘 만들고 있는지 지켜보며(Swift 5, AppKit 등 준비) 로고 생성을 병행. 로고가 잘 나오면 다운로드해, **픽셀메이터 프로**(화자가 구매한 앱, 약 5만 원, 1회 구매 영구 무료 — Adobe 대안)로 배경을 **투명화** 처리.

### [19:00–20:04] 로고 등록 지시 · 큐(queue) 동작
투명화한 로고를 앱 로고 이미지로 넣으라고 에이전트에 보냄. 앞서 말했듯 작업 도중 채팅을 넣으면 **큐에 쌓여** 위 작업이 끝난 뒤 실행된다(먼저 넣고 싶으면 바로 처리하는 버튼도 있으나 잘 안 돼 그냥 기다림). 그래서 앱 개발과 로고 등록을 모두 기다리면 됨. 계산기 앱은 잘 만들어지고 이름은 "오마이 칼" 식으로 나온다.

### [20:04–21:07] 앱 확인 · 테스트는 직접 · 로고 등록 시작
계산기 앱이 만들어지는 걸 확인. 로고 이미지 등록 후 **기능 테스트**를 해 볼 텐데, 테스트가 잘 되면 저장(커밋)을 해야 한다. 에이전트가 테스트하려고 화면을 요청하지만 테스트는 화자가 직접 하겠다며 멈추고, 로고 이미지 등록을 시킴. 앱은 `build/` 안에 만들어져 있어 그걸 실행하면 되고, 이후 이 앱을 **DMG**로 만들어 달라고 할 계획.

### [21:07–22:44] 아이콘 등록 · 안 뜰 때 반복
로고 이미지 위치(바탕화면)를 에이전트에게 알려 주면 찾아가 읽고 등록해 준다(프로젝트 폴더에 넣어 줬으면 더 좋았을 것). 등록 후 앱을 실행해도 아이콘이 안 뜨는 문제가 반복 — build/products/debug 등에서 표시가 안 됨. [의견] 화자도 macOS 네이티브 앱을 직접 만들어 본 적이 없고(오마이오픈스냅·오마이그리드도 다 직접 만든 게 아님) 그냥 시키는 것뿐이라, 껐다 다시 실행하기를 "될 때까지" 반복해 결국 아이콘이 뜨는 데 성공. 되면 커밋할 차례.

### [22:44–24:01] 커밋 시연 · .gitignore · 푸시 언급
"테스트 해 봤는데 잘 되네, 커밋 좀 해줘. 메시지에 '테스트 성공, 로고 이미지 등록 완료' 남겨줘"로 커밋. 커밋 = 현재 프로젝트 폴더 전체 상태를 "찰칵" 스냅샷. 이때 **.gitignore**가 자연스럽게 등장 — 커밋에서 제외할(엄청 중요하거나 용량이 쓸데없이 큰) 파일을 골라, 핵심 소스만 커밋 이력에 남기게 하는 것(에이전트에게 맡기면 됨). 에이전트가 "커밋 됐다, gitignore 제외했다" 안내. 이어 푸시 얘기가 나오는데, 이는 Git이 아니라 GitHub용(현재 상태를 온라인에 저장)이며 오늘은 다루지 않음.

### [24:01–25:11] 커밋 이력 · 체크아웃 왕복 · untracked
"커밋 이력 목록 보여줘" → 세이브 포인트 두 개(최초 커밋 + 방금 커밋). 각 커밋엔 **해시**(세이브 포인트의 ID 같은 것)가 있어, 해시를 지목해 "여기로 돌아가자"고 하면 **체크아웃**이 된다. "커밋 취소가 아니라 이동만 하겠다"고 하니 build 제외 초기 상태로 되돌아감(개발로 많이 만들어진 것들이 싹 사라짐). 되돌아오니 "untracked"가 남는데, 다시 두 번째 커밋으로 이동해 제대로 커밋을 시키려 한다. untracked는 커밋 이력에서 무시하는 것도 아니고 그냥 남아 있는 상태다.

### [25:11–26:03] untracked 원인 문답
왜 untracked가 있는지 에이전트에 물음. **.gitignore가 그 커밋 시점엔 없어서**(생긴 시점이 뒤라, 앞 커밋으로 가면 gitignore가 없어짐) 에이전트가 "무시 규칙이 없다 → untracked인가"로 생각한 것. "지금은 없다(정상)"로 정리. 이런 식으로 세이브 포인트를 만들며 가는 것이고, 이 문답이 이해 안 되면 무시해도 된다고 함.

### [26:03–27:31] 브랜치 생성 · 디자인 확 바꾸기
이제 **브랜치**를 만든다 — "디자인을 크게 바꿔 볼 예정인데 브랜치 하나 만들어라". 현재는 메인 가지라 메인 브랜치인데, 에이전트가 `redesign`이라는 새 브랜치를 파 준다(타임라인이 쪼개짐). 여기서 마음껏 디자인하다 마음에 안 들면 이 브랜치를 버리면 된다. "앱 디자인을 아주 독특하게 바꿔 실행해봐" 지시 → 에이전트가 "디자인 잘못돼도 `git switch main`으로 안전 복귀 가능"이라 안내. 신스웨이브/네온 레트로 퓨처 테마로 확 바꾼다. redesign 브랜치에서 하다 안 들면 그 브랜치의 커밋 이력을 잘라 버리고 메인으로 돌아오면 됨.

### [27:31–29:07] 히스토리 시각화 · 메인 합류 시나리오
디자인이 아주 화려하게 바뀜. [의견] 솔직히 마음에 들지 않지만 "마음에 든다" 가정 하에 메인으로 합칠 시나리오를 생각. 그림 정리: 메인에서 1차(초기)·2차(앱+로고 완성) 커밋 → 그 지점에서 브랜치를 만들어 redesign 작업 → 마음에 들면 메인으로 넘어와 머지(머지 커밋 생성). "여기까지 온 이력 보여줘" → Git 관점의 커밋·브랜치 상태. "초보라 시각적으로 보여줘"라고 하니 그래프로 시각화해 줌(현재는 redesign에 있음). 이 시각화는 가끔 쓰면 좋다.

### [29:07–30:04] redesign 커밋 여부 · 커밋
redesign 브랜치 생성 뒤 신스웨이브 리디자인 변경분은 **아직 커밋 안 된 상태**(저장 안 됨). 이 상태로 계속 진행하면 저장 안 된 채 나아가는 것이라, 가다 죽으면 처음으로 돌아온다. 그래프로 보면 청사진 상태에서 첫 커밋 후 가지를 뻗어 왔지만 커밋은 안 된 상태. "커밋 할까요?"에 "커밋해줘, 커밋 보고"로 브랜치를 판 상태에서 커밋 실행. 모르면 에이전트에게 다 물어보면 상태를 알려 준다.

### [30:04–31:00] 머지 방법 문답 · 메인으로 합치기
"이 브랜치가 마음에 들면 뭘 해야 하지? 메인으로 합칠 수 있나? 합치면 시각적으로 어떻게 되나?" 물음 → 에이전트가 "합칠 수 있고 합치기가 가장 쉽다"며 그림으로 보여 줌. 메인에서 브랜치로 나왔다가 마음에 들면 합치는 것. [의견] 오늘은 디자인이 마음에 들지 않지만 "합치자"고 해서 **머지** — redesign의 모든 작업이 메인으로 쓱 합쳐진다. 오늘 취지는 Git을 제대로 쓴다기보다, 바이브 코딩 중 사고 났을 때 돌아갈 수 있도록 커밋하고 가지를 쪼개는 걸 보여 주는 것.

### [31:00–32:19] DMG 패키징
합치기 전 메인이 한 칸 뒤에 있다는 그래프 설명(조금 어렵다며, 쪼개져 나왔다가 합쳐지는 그림으로 이해하라고). 머지 완료 후 메인으로 복귀. 매번 폴더를 찾아 들어가 실행하기 번거로우니, "앱 설치용 **DMG** 하나 만들어볼래?"라고 지시 → 에이전트가 DMG 패키징 생성. DMG는 더블클릭하면 드래그앤드롭으로 설치되는 것. "DMG 어디 있니?" → `dist/`에 있다. 실행하면 설치 화면이 뜬다. [의견] 이렇게 macOS 앱을 만들고 DMG까지 만드는 것으로 상당히 재밌는 실습이 됐다고.

### [32:19–33:05] 마무리
[의견] 이런 식으로 기본 개발을 하며 진행하는 게 일반적이고, 사고 방지(커밋·브랜치)는 당연히 기본으로 해야 한다. macOS 앱을 바이브 코딩으로 만들고 싶던 사람은 오늘 공유한 CLAUDE.md·스킬로 앱을 개발해 보고, 간단한 Git 3원칙을 쓰면서 짜잘한 문제는 에이전트와 함께 추가 공부하며 정확한 개념 사용법을 익히라고 권함. [의견] 오늘로 Claude Code 코드 탭 사용 기본은 다 다뤘다고 보며, 다음 영상부터는 (터미널의) Claude Code로 돌아온다고 예고하며 마무리.
