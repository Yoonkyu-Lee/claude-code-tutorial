### 크롤링 윤리 (화자 4대 주의)
- [의견] "크롤링 윤리라는 건 없다" — 정해진 규범은 없고, 학습용·개인용으로만 쓰되 **데이터 주인이 뭘 불편해하는지**를 알고 피하는 게 핵심이라는 관점 [03:40] [04:54].
- **① 짧은 시간에 대량 요청 금지**: 링크 클릭 한 번 = 서버에 "데이터 가져와" 요청. 1초에 수천 번 때리면 초인종(벨) 연타·집 냉장고 터는 것과 같다는 비유. 사람이 접속하는 정도의 시간차(예: 1초에 3~5회 수준)를 두라 [06:24] [07:53] [08:22].
- **② 판매 금지**: 수집 데이터를 팔면 문제. 숙박 사이트 데이터를 긁어 1,000원 싸게 복제 사이트를 만든 예시. 이미 **판례가 있어** 같은 짓 하면 거의 100% 걸린다(화자 주장/미검증) [08:34] [09:48].
- **③ 차단 우회 금지**: 사이트가 봇으로 판단해 막은 것을 우회하는 건 담 넘어 들어가는 것 = 합법 아님. 우회법을 묻지도 알리지도 말라 [10:05] [10:57].
- **④ robots.txt는 권고**: `도메인/robots.txt`에 Disallow(오지 마라)/Allow(와도 됨)가 있으나 **강제 아닌 권고**. Allow만 골라 다 긁어도 주인이 문제 삼으면 문제가 된다 [11:03] [12:37].
- 마무리 강조: 크롤링은 내가 안 가진 데이터를 프로그램으로 가져오는 것이라 문제 소지가 많으니 조심하라 [12:54].

### 크롤링의 본질과 도구
- 크롤링 = 사람이 손으로 하던 **복사→뒤로가기→복사** 반복을 코드(요즘은 AI)가 하는 것 [13:26] [14:17].
- 화자 도서에서는 Firecrawl과 Playwright를 소개했고, 이 강의에서는 **Playwright**를 사용 [03:17→교정: 파이어크롤/플레이라이트] [13:26].
- [의견] 예전엔 BeautifulSoup·Python·코드 작성을 배워야 했지만, 이제 AI가 코드를 다 짜주니 사용자가 배울 핵심은 **"원하는 데이터가 어디 있는지 위치를 설명하는 법"** 하나뿐 [14:40] [15:11].

### 핵심 개념 — 요소(element)와 위치
- 화면의 각 데이터(제목·가격·별점 등) = **요소(element)**. HTML은 여는 태그와 닫는 태그가 텍스트(알맹이)를 사탕봉지처럼 감싼 구조. 예: 여는 `<a>` + 닫는 `</a>` = **A 요소** [16:37] [24:39].
- 요소들은 **상위-하위(부모-자식) 트리** 구조 — 폴더 구조/아파트(동-층-호) 비유. 이 상하 관계를 주소처럼 표현하면 위치를 지정할 수 있다 [25:56] [26:16].
- 실습: 사이트 **books.toscrape.com**(대놓고 크롤링 연습용으로 만든 사이트)에서 진행 [17:56].
- **개발자 도구로 위치 찾는 법**: 원하는 데이터 우클릭 → 검사(Inspect) → 하이라이트된 코드 우클릭 → **Copy selector**(또는 Copy XPath / Copy full XPath) → 클립보드에 CSS 셀렉터 형태의 "주소지"가 복사됨. 셀렉터와 XPath 두 방식이 대표적(XPath는 구주소 방식으로 외워도 됨) [21:33] [28:02] [29:08].

### 실전 절차 — Claude 코워크로 크롤링
- 도구: **Claude Desktop 안의 코워크(Cowork)**를 사용(가장 무난하다고 선택) [21:00] [29:50].
- 셋업: 프로젝트 폴더 지정(예: `스터디 크롤링 클로드 코워크`) → 파일 변경 항상 허용 → **묻지 않고 실행(자동 실행)** 켜기. 코드도 파일이므로 폴더 쓰기 권한 = 코드 작성 권한 [29:58] [31:04].
- **패턴 1(한 단위 상세 지정)**: "이 주소에서, 이 셀렉터 위치의 데이터가 무엇인지 수집해봐" → 제목 하나 수집 성공. 이어서 한 책의 **이미지·별점·제목·가격·재고상태** 각각의 셀렉터를 붙여 한 책(한 단위) 전체를 수집 [31:29] [34:59].
- 수량 감각: 한 페이지에 책 20개 × 항목 4개 ≈ **80개 데이터**를 뽑는 코드가 뚝딱 완성 [37:00→교정] [37:56].
- **패턴 2(다음 위치만 지정)**: 한 단위 구조를 학습시킨 뒤 "다음 위치도 같은 항목으로 가져와" → 구조가 동일하므로 AI가 패턴 따라 자동 코드 작성 → "다음 위치가 없을 때까지 다 가져와"로 페이지 전체 순회. 결과 **26개** 데이터 수집 후 **CSV 파일**로 정리 [36:35] [37:56] [38:16].
- **더 큰 단위 이동**: 카테고리 전환 위치(셀렉터)를 알려주면 카테고리 순회까지 확장. "카테고리 5개가 될 때까지"로 지정해 5개 카테고리(sequential art / classics / philosophy / womens fiction 등) 데이터를 각 폴더에 수집 완료 [40:01] [41:06].
- 결과물은 코워크 프로젝트 폴더에 저장되어 파일로 확인 가능 [38:37].

### 왜 요소 위치를 직접 가르치나 (실무 가치)
- [의견] 요즘은 "그냥 다 가져와" 해도 **깔끔하게 정렬된 사이트**는 AI가 알아서 긁는다. 하지만 **복잡한 사이트에서 진짜 원하는 것만** 정확히 뽑으려면 요소 위치 지정 방법이 강력한 무기가 된다 — 그래서 기본기로 가르친다 [35:18] [40:47].
- 크롤링 설명은 여기까지가 "합법적으로 알려줄 수 있는 전부"이고, 그 이상(우회·봇 탐지 회피)은 대부분 불법 영역이라 다루지 않음 [42:01].

### 한글(HWPX) 스킬 연계 — 수집 데이터의 문서화
- 화자가 직접 만든 **한글 스킬**(GitHub 등에서 다운로드, 별점 200+ 받았다고 언급/미검증). Claude가 원래 못 하는 한글 문서 작성을 이 스킬로 가능케 함 [42:35] [43:02] [43:28].
- 설치: 초록색 Code 버튼 → ZIP 다운로드 → `HWPX 스킬 메인` 폴더 확보 → Claude Desktop **사용자 지정 → 스킬 → 스킬 업로드**로 등록 [43:44] [44:12].
- [의견] 스킬의 진짜 목적은 **기존 한글 템플릿(양식)을 그대로 이용해** 그 안에 데이터를 채워 넣는 것. 정부 보고서·민원 양식처럼 정해진 서식 반복 작업에 유용 [44:30] [45:03].
- 시연: 정부24 민원 서식(HWPX)을 받아 코워크 폴더에 넣고, "이 스킬로 이 민원 신청서에 수집한 도서 제목 10개를 넣어 '이 도서들이 불법 판매되고 있다'는 민원서를 써줘, 개인정보는 가상으로"라고 지시(화자 표현대로 억지 시나리오) [46:57] [47:59].
- 결과: 스킬이 **23행 3열의 표준 민원서 표 양식**을 인식하고 본문 구조·표를 채워 작성. 자간이 넓게 들어가는 등 서체 양식 이슈는 "자간 좁혀줘"·"정렬해줘" 식으로 후속 수정 가능 [49:00] [51:21] [52:07].
- [의견] 손으로 하면 파일 1개당 2~3분, 10개면 20~30분짜리 하루 일을 이렇게 자동화하면 이득. 정해진 정부 제출 양식을 반복하던 사람에게 특히 좋은 아이디어 [50:07] [51:38].

### Q&A — 인스타그램/쓰레드 크롤링, API
- [의견] 인스타그램·쓰레드는 SNS라 개인 소유 자산까지는 아니지만: ① 트래픽 과다 유발 시 **블락** 가능성 ② 남의 글을 긁어 내 것인 양 올리면 불법 소지. 초점은 "불법/합법"보다 **트래픽을 얼마나 유발하느냐** [53:21] [53:57] [55:02].
- [의견] 데이터 소유권은 글쓴이에게 있어 걸리면 문제되지만, 크롤링 행위 자체가 법원까지 갈 사안은 아니라고 봄. 인스타는 크롤링이 어렵고 쓰레드는 비교적 쉬운 편 [54:43] [55:11].
- **API가 정석**: 대기업 사이트는 크롤링 대신 공식 **API**로 정식 수집하는 게 맞다. 화자의 다른 유튜브 영상에 API 데이터 수집법이 있으니 참고 권유 [56:03] [56:07].
- 주의: 요즘 AI는 웬만한 데이터 수집을 잘 처리하고 해외 사이트가 법적으로 잡는 일은 거의 없으나, **트래픽 유발로 블락(접속 차단)** 되는 것은 조심하라 [56:11] [57:00].

## 3줄 오리엔테이션
- 크롤링을 "손으로 복붙하던 걸 AI가 대신하는 것"으로 정의하고, 핵심 스킬은 **원하는 데이터의 '요소 위치(셀렉터)'를 지정하는 법** 하나임을 가르친다.
- books.toscrape.com에서 Claude 코워크로 셀렉터 기반 크롤링(한 단위→다음 위치→카테고리 순회→CSV)을 실습하고, 시작 전에 크롤링 윤리 4가지를 강조한다.
- 마지막에 화자가 만든 한글(HWPX) 스킬로 수집 데이터를 정부 민원 양식에 채워 문서화하는 연계를 시연한다. 강의는 화자 도서의 부속물.

## 전문 — 시간순 워크스루

### [00:41–02:15] 오프닝·목 상태·오늘 다룰 것 예고
목 상태가 좋지 않다며 양해를 구하고 잘 들리는지 체크 후 시작. 오늘은 MCP를 이용한 크롤링, MCP 없이 하는 법, 그리고 한글 문서를 쓰기 위한 한글 스킬까지 여러 가지를 다 다루는 시간이라고 예고 [00:55] [01:09]. 최근 **개정한 도서**의 강의이며, 책을 보고 아쉬웠던 부분을 이 강의로 해결하면 좋겠다고 함 [01:19].

### [02:15–03:40] 오늘의 4가지 커리큘럼
> "오늘은 크롤링의 원리를 이해를 하고 한글 문서를 클로드를 통해서 정리를 하는 그 과정들을 같이 진행을 해 볼 예정입니다." [02:17]

네 가지: ① 크롤링 윤리 ② 크롤링 이해(원리)+하는 법 ③ 한글 스킬로 수집 데이터를 한글 문서화. 실습 대상은 준비를 안 했으나, **불법 소지 있는 곳은 제외하고 학습용으로 데이터 수집 가능한 곳**을 쓰겠다고 함 [02:35] [03:21].

### [03:40–05:16] 크롤링 윤리의 관점
[의견] "크롤링 윤리 같은 건 없다"며, 정해진 규범이 아니라 **학습용·개인용으로만** 쓰고 데이터 주인 입장을 헤아리라는 관점. 주인 아닌 이상 "해도 된다"고 말해줄 수 없고, 남의 집 냉장고 터는 느낌이라는 비유 [04:21] [04:26]. 다만 주인이 뭘 불편해하는지 보편적으로 알아두면 그걸 피해 갈 수 있다고 함 [04:48].

### [05:16–08:34] 윤리 ① 짧은 시간 대량 요청 금지
AI에게 크롤링 코드를 짜게 하면 요즘은 과속 코드를 잘 안 짜지만, 원리를 설명. 링크 클릭 = 서버에 "데이터 가져와" 요청인데, 이걸 10초에 만 번 하면 서버 입장에선 초인종을 1초에 수천 번 누르는(벨티) 격 [07:15] [07:41].
> "1초에 다섯 번 정도만, 또는 1초에 세 번 정도만 요청을 해서 데이터를 받아라. … 사람이 접속한다는 그런 느낌을 줄 수 있을 정도의 시간차를 두고 데이터를 가지고 와라." [08:05]

사람처럼 오가는 수준이면 서버가 기계라도 문제 삼지 않는다 [08:26].

### [08:34–10:05] 윤리 ② 판매 금지 (판례 존재)
수집 데이터를 팔면 문제. 숙박 사이트 예: 남의 사이트 데이터를 싹 긁어 **똑같은 복제 사이트를 만들고 1,000원 싸게** 팔면 손님을 뺏겨 소송 대상. **판례가 있어** 같은 짓 하면 거의 100% 걸린다(주장/미검증) [09:27] [09:48].

### [10:05–11:03] 윤리 ③ 차단 우회 금지
에이전트/AI에게 막힌 사이트 우회를 시키는 경우가 있는데, 사이트가 봇으로 판단해 차단한 것 = "오지 마라"는 뜻. 잠근 걸 굳이 넘어 들어가는 건 합법이 아니며, 우회법을 묻지도 알리지도 말라 — 걸리면 경찰서 [10:36] [10:57].

### [11:03–12:54] 윤리 ④ robots.txt는 권고 사항
`도메인/robots.txt`(예: google.com/robots.txt)에 Disallow(오지 마라)/Allow(와도 됨)가 있으나, 이는 **강제가 아닌 권고**. "잔디밭에 들어오지 마시오" 수준. Allow만 골라 다 긁었어도 주인이 기분 나빠 문제 삼으면 문제가 된다 [12:00] [12:29]. 네 가지를 늘 주의하라는 당부 [12:48].

### [12:54–14:40] 크롤링의 본질 — 손 복붙을 코드가 대신
크롤링은 내가 안 가진 데이터를 프로그램으로 가져오는 것이라 문제 소지가 많으니 조심하라 [12:56]. 도서에서 Firecrawl과 Playwright를 소개했고 오늘은 **Playwright** 사용 [13:26→교정]. 첫 페이지의 책 제목·별점·가격, 나아가 상품 설명(product description)까지 정리해 가져오려면 결국 **복붙**을 코드가(요즘은 AI가) 하는 것 [13:40] [14:17].

### [14:40–16:37] 이제 배울 건 "데이터 위치 설명하는 법"
[의견] 예전엔 BeautifulSoup·Python·코드 작성을 배워야 했지만 이제 AI가 코드를 다 짜준다. 사용자가 배울 핵심은 **"원하는 데이터가 어디 있는지 설명하는 법"** 하나 [14:56] [15:11]. 사람끼리는 "오른쪽 위"라고 말하지만 AI/컴퓨터에는 그렇게 설명이 안 됨 — 예: 설명 텍스트 중 괄호 앞 숫자(재고 22개) 같은 걸 어떻게 지정하나 [15:53] [16:15].

### [16:37–17:32] 요소(element)라는 개념
화면에 보이는 데이터 하나하나가 **요소(element/요소)**. 웹사이트는 이 요소들이 정렬돼 옷 입고 있는 것이며, 꾸밈 요소를 뺀 실제 텍스트 데이터가 요소. **요소의 위치를 설명할 줄 알면** 크롤링이 아주 쉬워진다 [17:18] [17:27].

### [17:32–19:00] 실습 사이트 접속 & 청중 투표
개발자 모드로 요소 위치를 찾는 실습 예고. **books.toscrape.com**(대놓고 스크레이프 연습용) 접속 안내 [17:56] [18:03]. 청중에게 실습 여부 투표를 받아 속도를 조절하겠다고 함(강의가 빠르다고 혼난 적이 있다며) [18:38] [19:23].

### [19:00–21:33] 도구 결정 — Claude 코워크
실습자가 6~7명 정도로 파악, 조금 천천히 진행. 인공지능 서비스는 **클로드**, 정확히는 클로드 데스크톱의 **코워크(Cowork)**로 진행(가장 무난) [21:00] [21:16].

### [21:33–24:39] 개발자 도구로 요소 찾기 — 태그의 구조
수집하고 싶은 데이터(책 제목)에 커서 → 우클릭 → **검사(Inspect)**. 하이라이트된 코드에 마우스를 올리면 화면 요소와 대응됨 = "이것이 이것"의 실체 [21:41] [22:56] [23:06]. `<a>...</a>`처럼 여는/닫는 한 쌍이 **태그**. 여는 태그+닫는 태그가 텍스트(알맹이)를 사탕봉지처럼 감싼 것이 **A 요소** [23:55] [24:39].

### [24:39–27:01] 요소의 상하 구조 — 폴더/아파트 비유
요소는 **상위-하위(부모-자식) 트리**. 개발자 도구의 펼침 구조가 폴더 구조와 같고, 아파트(동-층-호+엘리베이터)처럼 껍데기 안에 하위가 있음. 이 상하 관계를 주소처럼 표현해 위치를 지정 [25:24] [26:16] [26:43].

### [27:01–29:08] Copy selector로 "주소지" 얻기
수동으로 body→div.page_inner→div.row… 식으로 짚으면 한세월. 크롬은 요소 우클릭 → **Copy selector / Copy XPath / Copy full XPath** 제공. 화자는 **Copy selector** 사용 → 클립보드에 복사됨. 붙여넣으면 `>` 화살표로 이어진 CSS 셀렉터 = A 요소의 **위치(주소지)** [27:19] [28:02] [28:22]. 셀렉터와 XPath 두 표현 방식(XPath는 구주소 방식으로 외워도 됨) [29:02].

### [29:08–31:29] 코워크 프로젝트 셋업
얻은 주소로 "이 사이트 이 위치의 값이 궁금하다"고 하면 크롤링 시작 [29:40]. 클로드 데스크톱 → 코워크 → 다른 폴더 선택 → 폴더명 지정(예: `스터디 크롤링 클로드 코워크`) → 파일 변경 항상 허용 [30:04] [31:04]. 폴더 쓰기 권한 = 코드 작성 권한. **묻지 않고 실행(자동 실행)** 켜기 [31:04] [31:25].

### [31:29–34:05] 패턴 1 — 한 단위 상세 지정
"이 주소에서 이 셀렉터(제목) 위치의 데이터가 무엇인지 수집해봐" → 제목 하나 성공(historical fiction 목록의 책 제목 그대로 반환) [31:41] [32:58]. 이어서 한 책의 **이미지·제목·별점·가격·재고상태** 각 셀렉터를 검사→Copy selector로 붙여 한 단위 전체를 지정 [33:14] [33:46].

### [34:05–36:35] 한 단위 완성 & 왜 위치를 직접 가르치나
재고 상태 등 정확한 요소를 열어보며 지정, "위 위치를 고려해서 데이터 수집 후 말해라"로 한 단위 확정 [34:47] [35:08]. [의견] 요즘은 "다 가져와"만 해도 **정렬 잘 된 사이트**는 AI가 알아서 긁지만, **복잡한 사이트에서 원하는 것만** 뽑으려면 이 방법이 강력한 파워라서 가르친다 [35:18] [35:52].

### [36:35–38:37] 패턴 2 — 다음 위치만 지정 & CSV
한 단위 구조를 학습시킨 뒤 **"다음 위치"만** 알려주면 구조가 동일하므로 AI가 패턴 따라 자동 코드 작성 [36:35] [36:47]. "다음 위치가 없을 때까지 다 가져와" → 순회 → 총 **26개** 데이터 수집 후 **CSV 파일**로 정리 완료 [37:56] [38:16] [38:32].

### [38:37–40:01] 결과 파일 확인 & 더 큰 단위(카테고리) 이동

"카테고리 5개가 될 때까지 각 카테고리 안 데이터를 다 수집"으로 지시 → sequential art / classics / philosophy / womens fiction 등 5개 영역 데이터를 각 폴더에 수집 완료 [40:15] [41:06] [41:16]. [의견] 요소 위치 개념만 이해하면 나머지는 규칙 발견 후 지시만 하면 되고, 요즘 AI가 깔끔하게 처리 [40:47] [41:00]. 크롤링 설명은 여기까지가 합법적으로 알려줄 수 있는 전부, 그 이상(우회·봇 탐지 회피)은 불법이라 다루지 않음 [42:01].
