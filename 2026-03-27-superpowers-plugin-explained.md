## 한 줄 요약

GitHub 별 11만개를 받은 Jesse Vincent의 Superpowers 플러그인을 네 가지 핵심 원리(대화 먼저 · 테스트 먼저 · 역할 분리 · 독립 작업 공간)로 풀이하고, 화자가 직접 느낀 한계까지 솔직하게 덧붙이는 비코더 친화형 개념 설명 영상.

## 핵심 개념

### 문제 정의: AI 코딩 도구 = 판단력 없는 신입사원

> "AI한테 코딩시키면 잘할 거 같죠? 실제로는요. 열정만 넘치고 경험 없는 신입 사원이 설계도도 없이 집 짓기 시작하는 거랑 똑같아요." [00:00]

> "이 AI가 일하는 방식이 신입사원이랑 너무 비슷해요. 뭘 시키면 바로 달려들어요. '잠깐, 뭘 만들어야 하는 건지 먼저 물어볼게요' 이런 거 없어요. 그냥 코드부터 막 짜기 시작합니다." [00:23]

인테리어 비유 (반복적으로 등장):

> "[의견] 인테리어 업자한테 '카페 느낌으로 해 주세요' 했더니 물어보는 것도 없이 바로 벽을 부수기 시작하는 거예요. 몇 인석이요? 안 물어보고요. 예산은요? 안 물어보고요. 그냥 막 시작해요. AI 코딩 도구가 딱 이래요. 열정은 넘치는데 판단력이 없어요." [00:34]

### Superpowers의 정의

> "슈퍼파워즈는요. 한마디로 말하면 AI한테 일 잘하는 법을 가르쳐 준 매뉴얼이에요." [00:59]

> "만든 사람은 제시 빈센트(Jesse Vincent)라는 분인데요. 오랫동안 오픈소스를 만들어 온 개발자예요. 이분이 클로드 코드를 쓰면서 느꼈대요. '이 AI 똑똑하긴 한데 일하는 방식이 엉망이다.'" [01:04]

화자가 정리한 4단계 강제 순서:

> "1단계: 먼저 뭘 만들지 대화해. 2단계: 계획을 세워. (3단계:) 그다음에 코드를 짜. 4단계: 테스트에서 확인해. 이걸 스킬이라는 파일로 만들어서 AI한테 주입시키는 거예요." [01:20]

> *(원문에서 "3단계" 라벨은 발화 누락. 4단계 흐름은 명시됨.)*

규칙의 강제성:

> "이 스킬에는 AI가 절대 건너뛸 수 없는 규칙이 있어요. '이건 선택이 아닙니다. 반드시 따르세요' 이렇게 적혀 있거든요. 심지어 AI가 '이건 너무 간단해서 안 해도 되지 않나' 이렇게 빠져나가려 할 때를 대비해서 미리 대응 매뉴얼까지 넣어 놨어요." [01:28]

## 카탈로그 — 슈퍼파워즈의 네 가지 핵심

### 1) 대화 먼저 (Brainstorming)

> "슈퍼파워즈의 첫 번째 핵심은요. 코드 짜기 전에 대화부터 해예요. 이걸 브레인스토밍이라고 하는데요." [01:46]

화자가 든 시나리오:

> "여러분이 '로그인 기능 만들어 줘' 하면 보통 AI는 바로 뚝딱 만들기 시작하잖아요. 근데 슈퍼파워즈가 설치되어 있으면요. AI가 먼저 이렇게 물어봐요. '어떤 방식으로 로그인하게 할 건가요? 이메일로? 소셜 로그인으로? 비밀번호 찾기도 필요해요?' 그러면서 여러분이 뭘 원하는지 정확히 파악하고요. 두세 가지 방법을 제안하면서 장단점을 알려 줘요. 여러분이 '이걸로 가자' 승인을 해야 그때서야 만들기 시작해요." [01:56]

### 2) 테스트 먼저 (TDD)

> "두 번째 핵심은요. 만들기 전에 기준부터 정해라는 건데요. 개발 쪽에서는 TDD라고 부르는데 어려운 용어는 몰라도 돼요." [02:26]

요리 비유:

> "[의견] 요리할 때를 생각해 보세요. 보통은 레시피대로 만들고 마지막에 맛을 보잖아요. 근데 TDD는요. 맛을 먼저 정해 놓고 시작하는 거예요. '이 된장찌개는 짠맛이 이 정도야 해. 고춧가루는 이만큼 들어가야 해.' 이 기준을 먼저 만들어 놓고 그 기준에 맞춰서 요리하는 거예요." [02:31]

규칙의 엄격함:

> "슈퍼파워즈에서는 이걸 철칙이라고 불러요. '기준 없이 만들기 시작하는 거 금지. 기준 없이 먼저 만들었으면 지우고 다시해.' 진짜 이렇게 적혀 있어요." [02:59]

### 3) 역할 분리 — 팀장 AI / 팀원 AI / 컨텍스트 분리

> "세 번째 핵심은요. AI 하나가 혼자 다 하는게 아니라 팀을 나눠서 일한다는 거예요." [03:19]

두뇌 분리:

> "팀장 AI와 팀원 AI가 서로 다른 두뇌를 쓴다는 거예요. … 큰 그림을 그리는 팀장, 그러니까 전체 계획을 세우는 AI는 가장 똑똑하고 비싼 두뇌를 써요. 그걸 Opus라고 하는데 가장 고급 AI 모델이에요. 반면에 실제로 코드를 짜는 팀원 AI는요. 빠르고 가벼운 두뇌를 써요. 이걸 Haiku라고 하는데 속도가 빠른 모델이에요." [03:38]

건축 비유:

> "[의견] 마치 건축에서 설계는 경력 30년 건축가가 하고 벽돌 쌓는 건 숙련된 기술자가 하는 거랑 똑같아요." [04:09]

컨텍스트 분리:

> "각 팀원 AI는 자기 일에만 집중하게 돼 있어요. 다른 업무 내용이 섞이지 않아요. 이걸 컨텍스트 분리라고 하는데 쉽게 말하면 AI 머릿속을 깨끗하게 유지해 주는 거예요. … 코드를 읽는 AI, 코드를 짜는 AI, 검토하는 AI를 다 따로 분리해 놓은 거예요." [04:15]

화자의 직접적 호평:

> "[의견] 이런 세심한 설계를 보면요. '아, 이거 정말 잘 만들었다'라는 감탄이 나와요. AI를 그냥 쓰는게 아니라 AI가 어떻게 하면 가장 잘 일할 수 있는지를 깊이 고민한 흔적이 보이는 거거든요." [04:42]

### 4) 독립 작업 공간 (병렬 충돌 방지)

> "네 번째는요. 여러 AI가 동시에 일할 때 서로 방해 안 되게 하는 기술이에요." [04:53]

구글 문서 비유 → 책상 비유:

> "같은 프로젝트의 복사본을 하나도 만들어서 각 AI한테 '너는 여기서 일해' 하고 따로 주는 거예요. 마치 사무실에 각자 자기 책상이 있는 것처럼요. AI 1은 자기 책상에서 로그인 기능을 만들고 AI 2는 자기 책상에서 결제 기능을 만들어요. 서로 전혀 방해가 안 돼요. 다 끝나면 합치면 되거든요." [05:07]

전체 시스템 효과:

> "이 덕분에 제시 빈센트는 클로드가 몇 시간 동안 혼자서 알아서 일한다고 했어요. 사람이 옆에서 계속 지켜보지 않아도 AI가 알아서 계획대로 착착 진행한다는 거예요." [05:31]

## 화자의 주제적 메시지

> "[의견] AI 자체가 똑똑해지는 것보다 AI를 어떻게 쓰느냐가 더 중요하다는 거예요. 같은 클로드 코드를 써도 슈퍼파워즈가 있는 사람과 없는 사람의 결과가 완전히 달라요." [05:43]

ChatGPT 사용 비유로 일반화:

> "[의견] '요약해 줘' 하는 사람이랑 '이 문서의 핵심 논점 세 가지를 각각 두 문장으로 정리해 줘' 하는 사람이랑 결과물이 다르잖아요. 슈퍼파워즈는 이걸 시스템으로 만든 거예요." [05:55]

마무리 명제:

> "[의견] AI 시대의 진짜 경쟁력은요. 새로운 도구를 쓰는게 아니라 도구를 잘 쓰는 방법을 아는, 하니스를 아는 거예요." [07:57]

## 비판 / 한계

화자가 영상에서 직접 두 종류의 비판을 다룸.

### 외부의 비판 (화자가 인용)

> "솔직히 말씀드릴게요. '정말 효과가 있나? 벤치마크가 없잖아. 그냥 프롬프트를 잘 쓰면 되는 거 아니야?' 이런 의견도 있거든요. 맞는 말이에요. 정식 비교 실험 데이터는 아직 없어요." [06:15]

### 화자 본인의 한계 지적

> "[의견] 브레인스토밍적인 부분에서 어떤 물어보는 질문의 어떤 퀄리티라든지, 설계라든지 이런 부분이 좀 많이 부족하다고 느끼고 있고요." [06:30]

> "[의견] 두 번째로 QA 하는 파트로는 좀 부족한 거 같고, 결국에는 E2E 테스트가 돼야지 이게 진짜 제대로 돌아가는지 확인을 하는데, 그럼 QA 하는 파트가 아직은 좀 부족하다라는 느낌을 좀 많이 받고 있습니다." [06:43]

설치 후 동작:

> "설치하고 나면요. 클로드 코드를 켤 때마다 자동으로 슈퍼파워즈의 규칙이 AI한테 주입돼요. 그래서 여러분이 '앱 만들어 줘' 하면 AI가 바로 코드를 짜는 듯이 먼저 '몇 가지 물어볼게요' 하고 대화를 시작해요." [07:09]

## 외부 보강

현재 본 저장소가 띄운 Claude Code 세션에 실제로 설치된 superpowers 스킬 카탈로그(세션 system reminder 기반)와 영상의 네 핵심을 매핑하면 다음과 같다. 영상이 설명한 추상 원리가 실제 스킬 파일로 어떻게 분해돼 있는지 확인하는 용도.

| 영상의 핵심 | 대응 superpowers 스킬 |
|---|---|
| 대화 먼저 | `superpowers:brainstorming` |
| 테스트 먼저 | `superpowers:test-driven-development`, `superpowers:verification-before-completion` |
| 역할 분리 (팀 시스템) | `superpowers:subagent-driven-development`, `superpowers:dispatching-parallel-agents` |
| 독립 작업 공간 | `superpowers:using-git-worktrees` |
| (영상 미언급) 계획 단계 강제 | `superpowers:writing-plans`, `superpowers:executing-plans` |
| (영상 미언급) 디버깅 절차 | `superpowers:systematic-debugging` |

출처: 본 세션 시작 시점에 로드된 사용 가능 스킬 목록 (Claude Code system reminder). 별도 외부 문서 보강은 진행하지 않음.

## 미해결 / 영상이 다루지 않은 것

- "GitHub 별 11만 개, 5개월 만"의 정확한 리포지토리 URL과 시점 검증 (영상은 URL 미제공)
- Opus/Haiku를 실제로 어떻게 분기하는지 — 모델 라우팅 메커니즘이나 비용 추정 미공개
- "프로젝트 복사본"이 git worktree인지 단순 copy인지 — 영상은 비유(책상)만 제시
- 4단계 강제 순서를 우회하고 싶은 경우의 옵션 (스킬 비활성화·partial bypass 등)
- 화자 본인이 지적한 "브레인스토밍 질문 퀄리티 부족"·"E2E QA 미흡"에 대한 구체적 reproduction 예시
- 영상이 던지는 "AI 자체보다 시스템이 중요하다"라는 명제의 반례 — 모델 자체가 약하면 어떤 시스템도 못 살린다는 케이스
- "AI 시대의 경쟁력은 하니스" 주장의 정량적 근거 — 본 영상에서는 비유와 인상 위주
