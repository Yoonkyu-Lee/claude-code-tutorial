## 핵심 개념

### 1) 왜 한 스킬만으로는 부족한가 — AI 코딩의 4가지 실패 패턴

> "AI 코딩 에이전트는 똑똑합니다. 근데 시니어 엔지니어처럼 일을 하지 않습니다." [01:54]

화자가 나열한 네 가지 실패 패턴:

1. **요구사항 이해 전에 코딩** — 스펙이 흐릿한 상태에서 코드를 먼저 씀 [01:05]
2. **같은 맥락 반복 설명** — 세션마다 도메인 용어를 다시 설명, 게다가 매번 설명이 달라짐 [01:15]
3. **동작 안 하는 코드 생성** — "AI가 거짓말로 완성했다고 말하는 케이스" [01:28]
4. **AI 슬롭 양산** — 빠른 코딩으로 인한 결과

> "AI는 평균에 회기[회귀]하는 그런 특성을 지니고 있기 때문에 어떤 뾰족한 한 점으로 나아가지 못하고 특정한 구간을 좀 회기[회귀]하는 그런 패턴을 지닌다고 보시면 될 거 같습니다." [01:45]

### 2) 스킬이란 — 클로드의 매뉴얼

> "스킬이란 뭐냐? 클로드가 작업 전에 반드시 읽고 따르는 매뉴얼이라고 보시면 될 거 같습니다." [02:02]

- **포맷**: `SKILL.md` + 스크립트로 구성 [02:09]
- **호출**: 슬래시 커맨드(`/...`)로 호출 [02:13]
- **비유**: 회사의 매뉴얼. 모델의 재학습 없이 행동만 변경 [02:15]
- **목적**: 도메인 지식을 주입해서 알아서 일하게 만듦 [02:22]

### 3) 4스킬 분담 모델

화자가 자기 워크플로우에서 쓰는 4종 스킬과 각각의 역할:

| 스킬 | 역할 | 없으면 부재하는 것 |
|---|---|---|
| **GStack** | 기획 — "왜 만드는가" | 기획 [03:18] |
| **Superpowers** | 실행 — 효율적 개발 워크플로우 | 실행 [03:21] |
| **매포코 스킬** | 유지보수를 위한 코드 | 유지보수 [03:23] |
| **Codex Plugin CC** | 다른 모델로 다른 시각 검토 | 검증 [03:27] |

> "G스택이 없으면 기획에 있어 가지고 부제[부재]하고요. 슈퍼파워즈가 없으면 실행에 대해서 부제하고요. 매퍼코의 스킬이 없으면 유지보수에 대해서 부제하고요. 코덱스 플러그인 CC가 없으면 검증에 있어 가지고 부제하다고 보시면 될 거 같습니다." [03:18]

각 스킬이 묻는 질문:
- GStack: "정말 꼭 필요한 기능인가? 기획적으로 다양한 시각에서 충분히 보았는가?" [03:33]
- Superpowers: "효율적인 개발, 컨텍스트가 터지지 않고 개발을 하였는가?" [03:42]
- 매포코 스킬: "이 코드를 유지보수 가능하게 만드는 코드로 바꿔주는가" — 도메인 지식과 매핑함으로써 [03:50]
- Codex Plugin: "다른 시각으로 리뷰를 했나" [04:01]

## 카탈로그 — 4개 스킬팩 상세

### A. Superpowers — "시니어 엔지니어의 작업 습관을 마크다운으로 정리"

> "슈퍼파워즈는 시니어 엔지니어의 작업 습관을 마크다운으로 정리한 것입니다." [04:12]

화자가 실제로 쓰는 세 가지 하위 스킬:

#### A-1. Brainstorming [04:25]
- 맥락(프로젝트 파일 + 최근 커밋) 탐색
- **한 번에 질문 하나만** 함
- 목적·제약·성공 기준 수립
- 2~3가지 접근법과 트레이드오프 제시
- 섹션별로 승인 후에만 다음으로 진행
- 결과를 `spec.md`에 저장 후 기술 커밋
- writing plans를 자동 호출해 플랜 작성

> "설계에 승인 전까지 어떤 코드도 작성하지 않는다는 것이 바로 이 브레인스토밍의 특징이라고 보시면 될 거 같습니다." [04:52]

#### A-2. Writing Plans [04:56]
- Brainstorming 결과로 만들어진 스펙 기반으로 플랜 작성
- 스펙 확정 후 자동 실행

#### A-3. Subagent-Driven Development [05:16]

세 가지 코스로 나뉨:
1. **Implementer** — 구현, 테스트, 셀프 리뷰, 커밋 [05:24]
2. **Spec Compliance** — 요구사항 대비 검증과 통과 검수 [05:29]
3. **Code Quality** — 코드 품질 검토. "스펙 통과 후에만 시작한다"는 룰 [05:36]

> "이 친구를 이용을 하시면 어떤 구현 작은 단위에 구현을 실제로 할 때 서브 에이전트 기반으로 해 가지고 아주 작은 단위로 새로운 컨텍스트에서 한다라고 보시면 될 거 같습니다. 특히나 테스트가 굉장히 꼼꼼하게 붙어 있기 때문에 굉장히 도움이 많이 된다고 보시면 될 거 같아." [05:43]

### B. 매포코 스킬 — "도메인 언어와 모듈 깊이"

> "에이전트는 코딩만 빠르게 하지 않는다. 코드 부패도 빠르게 만든다라는 철학 아래에서 만들어졌습니다." [06:08]

#### B-1. Grill With Docs [06:17]
[불명확: 자막 원문 "그릴 위드스" / "그릴 위드 독스" — 정확한 스킬명은 영상에서 명확히 호명되지 않음. 본 노트는 `Grill With Docs`로 추정 표기]

참고 파일 두 종류:
- **`context.md`** — 도메인 언어 사전. "용어의 의미를 적는 파일" [06:26]
  - 구현은 쓰지 않음
  - 모든 스킬이 작업 시작 시 자동으로 읽음
  - 새 결정 발생 시 대화 도중 바로 갱신
  - "매 세션 맥락을 다시 설명할 필요가 없습니다" [06:44]
- **ADR** (Architecture Decision Record) — 결정 기록 [06:46]
  - 되돌리기 어려운 결정만 기록
  - 맥락 없이 보면 의아한 선택을 설명
  - 대안과 트레이드오프를 함께 기록
  - 이미 끝난 논의를 다시 꺼내지 않음

스킬 동작:
1. `context.md`와 ADR 로드 → 기존 도메인 용어를 먼저 읽음 [07:17]
2. 새 용어 발견 시 정의 부여
3. 한 번에 질문 하나만
4. 추측 대신 코드베이스를 봄
5. 추천 답변 + 근거 같이 제시
6. ADR에 기록된 선택은 재논의하지 않음. 새 결정만 챌린지
7. 대화 중 갱신 — 새 용어가 나오면 그 자리에서 `context.md`에 추가 [07:36]

> "이 친구를 통하면 코드의 유지보수성 AI에게 시킬 때 어떤 용어가 달라짐으로 인해해 가지고 생기는 이격들이 없어진다고 보시면 될 거 같아요." [08:01]

[의견] 화자 경험: 결제 쪽에서 생기는 이슈가 가장 컸음. context와 ADR이 잘 정의되어 있으면 "제가 잘못된 지시를 했더라도 AI가 반박을 하는 그런 신기한 현상"이 나타남 [08:10]

#### B-2. Improve Codebase Architecture [08:34]

목표: "얕은 모듈을 깊은 모듈로 만드는 것" [08:41]

> "깊은 모듈이라는 거는 뭐냐? 어 재사용 가능하고 그리고 테스트업[테스트가] 가능한 코드라고 보시면 될 거 같습니다." [08:45]

동작:
1. `context.md` 로드 → ADR 로드 [08:57]
2. 모듈의 경계와 마찰 지점 탐색 [09:04]
3. **삭제 테스트** — "지웠을 때 복잡도가 줄어드나 늘어나" [09:17]
4. Before/After 보고서 생성 [09:21]
5. 추천 → 사용자 선택 후 검증 [09:25]

> "게리탄[Gary Tan]이라는 친구가 만든 친구고요. Y이 콤비네이터의[Y Combinator의] 사장이에요. 그래서 한 명의 빌더가 올바른 도구를 가지면 팀보다 빠르게 움직인다라는 그런 철학 아래에서 만들어졌고요." [09:40]
