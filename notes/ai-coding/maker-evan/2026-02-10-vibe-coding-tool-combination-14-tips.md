## 한 줄 요약

## 핵심 개념

화자가 영상 전체를 관통해 반복하는 단일 명제:

> "바이브 코딩은 도구 선택이 절반이고 나머지 절반은 도구를 쓰는 순서와 조합이라는 겁니다." [00:10]

> "AI 도구는 하나만 잘 쓰는 게 아니라 잘하는 걸 조합하는 겁니다." [01:07]

마무리에서 같은 메시지를 한 번 더 압축한다:

> "바이브 코딩 잘하려면 도구 하나를 잘 쓰는 게 아니라 각 도구가 잘하는 걸 파악하고 조합하는 게 핵심입니다. 분석은 Gemini, 코딩은 Claude, 음성 추출은 Whisper. 그리고 마지막에 스킬로 퀄리티 체크." [05:06]

[의견] "똑같은 프롬프트인데 결과물이 다른 이유"라는 제목의 답을, 화자는 "프롬프트 표현력"이 아니라 "도구 조합/경유 단계"에서 찾는다. 즉 같은 의도라도 어떤 모델에 어떤 형태로 입력하느냐가 결과를 가른다는 입장.

## 카탈로그 — 14가지 팁

> 각 팁은 화자가 "직접 삽질하고 나서야 알게 된 것"이라고 전제한다. [00:18]
> 정정 각주: 자막의 "재미이/재미니" → **Gemini**, "클로드" → **Claude**, "스티치" → **Stitch (Google Stitch)**, "오픈 AI 위스퍼" → **OpenAI Whisper**, "넥스트 JS" → **Next.js**, "샤드 CN" → **shadcn/ui**, "무이/무고" → **MUI (Material UI)**, "버셀" → **Vercel** 로 컨텍스트 정정.

### 팁 1 — Gemini한테는 영어로 말하되, 영어를 직접 쓰지는 마라 [00:25]
- Gemini 계열은 한국어 프롬프트를 넣으면 결과물 퀄리티가 떨어진다.
- 그렇다고 사용자가 직접 영문 프롬프트를 쓸 필요는 없다.
> "Claude한테 먼저 한국어로 설명하고 영문 프롬프트로 변환해 달라고 하세요. 그다음 그걸 Gemini에 넣으면 됩니다. 이 한 단계만 추가해도 결과물 퀄리티가 체감될 정도로 달라집니다." [00:36]
- 패턴: **한국어 의도 → Claude가 영문 프롬프트로 번역 → Gemini에 투입**.

### 팁 2 — 이미지·영상 분석은 Gemini가 압도적 [00:46]
- 이미지 인식·영상 분석은 현재 Gemini 계열이 가장 잘한다 [의견].
> "Gemini로 분석한 다음에 그 결과를 Claude에 넘겨서 작업하는 겁니다." [00:54]
- 예시: 레퍼런스 웹사이트 스크린샷을 Gemini에 넣어 구조 분석 → 그 분석 결과를 Claude에 주면서 "이 구조로 만들어 줘" → 혼자 할 때보다 정확한 결과물.

### 팁 3 — 영상 분석할 땐 영상 자체를 던지지 마라 [01:11]
- 영상을 통째로 AI에 던져 분석시키면 퀄리티가 아쉽다. 특히 "후킹 포인트 찾기"는 현재 모델로는 한계가 분명 [의견].
> "OpenAI Whisper API로 음성을 먼저 추출하고 텍스트로 변환한 다음에 분석하는 겁니다. 영상 자체를 주는 것보다 [불명확: 원문 "을도파"] 컨셉 분석 정확도가 완전히 다릅니다." [01:23]
- 패턴: **영상 → Whisper로 음성→텍스트 → 텍스트 기반 분석**.

### 팁 4 — OpenAI는 평타, 압도적인 영역은 없다 [01:36]
- OpenAI 모델은 전반적으로 무난하지만 하나를 압도하는 영역이 없다 [의견].
> "코딩은 Claude가 낫고 이미지·영상 분석은 Gemini가 낫고, 그래서 저는 OpenAI를 메인보다는 보조로 활용하는 편입니다. Whisper 같은 특정 API는 정말 좋지만 코딩 메인으로 쓰기엔 아쉬운 부분이 좀 많아요." [01:49]

### 팁 5 — Stitch는 Pro와 Flash 차이가 크다 [01:57]
- Google Stitch의 Pro 모드와 Flash 모드는 결과물 차이가 꽤 크다.
> "빠르게 테스트할 때는 Flash도 괜찮지만 실제 프로젝트라면 모델을 Pro로 사용하세요. 디자인 퀄리티가 다릅니다." [02:07]

### 팁 6 — Stitch한테 레퍼런스 URL 주지 마라 [02:13]
- Stitch에 "이 홈페이지 참고해서 만들어 줘"라며 URL을 주면 잘 못 알아먹는다.
> "웹페이지의 레이아웃을 말로 묘사하고 컬러 코드를 정확하게 넣어 주는 겁니다. 이런 식으로 구체적으로 알려주면 결과물이 확 달라집니다." [02:21]

### 팁 7 — Stitch도 Claude가 답이다 [02:29]
- 팁 1(Gemini)·팁 6과 같은 "경유" 패턴을 Stitch에도 적용.
> "Claude한테 웹사이트 정보 읽게 하고 그걸 영문 프롬프트로 만들어서 Stitch에 명령하면 직접 프롬프트 쓰는 것보다 결과물이 훨씬 좋습니다." [02:36]
- 정리: Stitch 쓸 때 = **레퍼런스 URL 대신 / Claude가 정리한 묘사 + 영문 프롬프트**가 현재 가장 효과적인 조합.

### 팁 8 — Stitch 결과물을 그대로 쓰지 마라 [02:47]
- Stitch 결과물은 괜찮아 보여도 바로 실전에 쓰면 아쉬운 부분이 많다.
> "Stitch 결과물은 AI 스튜디오에서 빌드 다운로드 → Next.js 프론트엔드 가이드라인 스킬 같은 걸로 컨버팅. 이렇게 하면 Stitch의 디자인 감각은 살리면서 실제 프로덕션 수준 코드로 올릴 수 있습니다." [02:58]
- [불명확: 원문 "넥스트 JS 프론트 핸드 가이드라인 스킬" — "프론트엔드(front-end) 가이드라인 스킬"로 추정되나 정확한 스킬 이름은 영상에서 불명확]

### 팁 9 — MUI 말고 shadcn/ui를 써라 [03:08]
- UI 라이브러리로 MUI를 쓰는 사람이 많지만, shadcn/ui가 훨씬 감각적인 디자인을 뽑는다 [의견].
> "AI가 코드를 생성할 때 shadcn/ui 기반으로 만들면 모던하고 깔끔한 결과물이 기본으로 나와요. 특별히 커스텀 안 해도 '어 이거 괜찮은데' 하는 수준이 나옵니다." [03:20]

### 팁 10 — 코드 체크·디자인 체크를 스킬로 자동화하라 [03:26]
- 코드 짜고 그냥 끝내지 말 것.
> "Vercel의 React 베스트 프랙티스 스킬로 코드 품질 체크하고, 웹 디자인 가이드라인 스킬로 디자인 체크하면 퀄리티가 올라갑니다. 이 두 스킬은 마지막에 한 번씩 돌리는 습관만 들여도 결과물의 완성도가 확실히 달라져요." [03:38]
> "이 두 스킬의 경우 [불명확: 원문 "skill.shh.com / shc.com"] 이라는 사이트를 통해서 다운로드 받으실 수 있습니다." [03:44]
- [불명확: 다운로드 사이트 주소가 자막에서 "skill.shh.com" / "shc.com"으로 흔들림. `skills.sh` 또는 유사 마켓플레이스로 추정되나 정확 주소는 영상만으로 확정 불가]

### 팁 11 — 로고·미니멀 디자인은 Claude 웹이 최강 [03:48]
- 간단한 로고나 미니멀한 디자인은 Claude 웹에서 바로 만들어 보라.
> "복잡한 일러스트는 어렵지만 심플하고 세련된 디자인은 Claude가 의외로 잘 뽑아줍니다. 별도 디자인 툴 안 켜도 되는 경우가 많아요." [03:55]

### 팁 12 — 디자인 수정은 "미니멀하게"만 말해 보라 [04:03]
- Claude Code로 디자인 수정 시 디테일한 지시를 하나하나 내리기보다 방향만 주는 게 낫다.
> "'미니멀하게 다듬어 줘'라고 방향만 잡아주면 꽤 만족스러운 결과가 나옵니다. AI한테 디자인을 시킬 때는 구체적인 수치보다 방향성을 주는 게 효과적인 경우가 있어요." [04:09]
> "'깔끔하게 여백 살려서 미니멀하게' 이런 키워드가 생각보다 잘 먹힙니다." [04:19]

### 팁 13 — SNS 글은 AI로 쓰지 마라 (다듬기만) [04:25]
- SNS 글을 AI한테 통째로 쓰게 하면 티가 많이 난다.
> "원문은 직접 쓰세요. 여러분의 생각, 언어. 그다음 AI는 다듬기만 하는 겁니다. 문장 정리, 오탈자 수정, 흐름 개선 이 정도. 이렇게 해야 사람 냄새가 나는 글이 됩니다. AI가 100% 쓴 글은 읽는 사람도 압니다." [04:35]

### 팁 14 — 후킹 포인트는 아직 AI가 못 찾는다 [04:46]
- 영상 편집의 후킹 포인트 찾기는 현재 모델로는 한계 [의견].
> "시청자가 멈추는 순간, 감정이 바뀌는 타이밍 이런 건 아직은 사람의 감각이 필요해요. AI는 텍스트 기반 분석은 잘하지만 영상의 흐름과 감정을 읽는 건 아직 부족합니다. 이 부분은 현재 모델 수준으로는 직접 하시는 게 낫습니다." [04:57]

## 팁 (메타 패턴 요약)

영상 전체에서 반복되는 구조적 패턴 3가지:

1. **Claude 경유 패턴** (팁 1·7·8): 다른 도구(Gemini, Stitch)에 직접 한국어/URL을 넣지 말고, Claude가 영문 프롬프트·구조 묘사로 정제한 뒤 그 도구에 투입한다.
2. **도구별 강점 분업** (팁 2·3·4): 분석=Gemini, 코딩=Claude, 음성 추출=Whisper. 메인/보조를 명확히 나눈다.
3. **마지막 스킬 체크** (팁 10): 산출 직후 코드 품질·디자인을 전용 스킬로 한 번씩 점검하는 습관.

화자가 반복 강조한 표현: "이 한 단계만 추가해도", "확 달라집니다", "꼭 기억하세요", "습관만 들여도".

## 외부 보강

영상에서 음차로만 언급된 도구·모드를 공식 정보로 보강한다 (영상 내용과 분리).

- **Google Stitch** (팁 5~8): Google Labs의 무료 브라우저 기반 AI UI 디자인 도구로, Gemini를 사용해 텍스트 프롬프트·이미지에서 웹/모바일 UI를 생성한다. 두 모드가 존재 — **Standard(Flash)** 모드는 Gemini 2.5 Flash 기반으로 빠르고 텍스트 프롬프트에 적합(월 350회), **Experimental/Pro** 모드는 Gemini 2.5 Pro 기반으로 이미지 입력을 받고 더 정밀(월 50회). 영상의 "Pro vs Flash 차이가 크다"(팁 5)·"URL 대신 묘사+컬러코드"(팁 6) 주장과 정합. 출처: [Stitch - Google](https://stitch.withgoogle.com/), [Google Stitch Pricing 2026 - NxCode](https://www.nxcode.io/resources/news/google-stitch-pricing-plans-complete-guide-2026)
- **OpenAI Whisper** (팁 3): OpenAI의 음성→텍스트(speech-to-text) 모델·API. 영상의 "음성 먼저 추출 후 텍스트 분석" 워크플로의 핵심 도구.
- **shadcn/ui** (팁 9): Tailwind CSS 기반의 복사-붙여넣기형 React 컴포넌트 컬렉션. MUI와 달리 코드를 프로젝트에 직접 넣는 방식이라 커스터마이즈 자유도가 높다.
