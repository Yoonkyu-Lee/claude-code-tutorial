# DDD 구조가 AI 코딩 성패를 가르는 이유

## 한 줄 요약
같은 AI를 써도 코드를 담는 구조가 DDD(도메인 주도 설계)냐 아니냐에 따라 AI 코딩의 성패가 갈리며, 비즈니스 언어와 코드 이름을 일치시키고 도메인 단위로 경계를 나누면 AI가 패턴을 복제하고 필요한 맥락만 집중해서 환각이 줄어든다.

## 메타
- 영상 유형: **개념 설명형** (비유·정의 위주, 비개발자 대상 풀어쓰기)
- 대상 청중: 바이브 코딩을 직접 하는 사람 + 개발자에게 작업을 맡기는 비개발자 양쪽
- 자막: 한국어 자동 생성 자막, 타임스탬프 포함

## 핵심 개념

### 1. 문제 정의 — AI가 만든 코드가 손댈 수 없게 되는 악순환
화자는 첫머리에서 자신의 실패 경험을 제시한다.

> "AI한테 백엔드 만들어 달라고 했더니 진짜로 만들어 줬어요. 근데 일주일 뒤에 기능 하나 추가하려고 봤더니 아무도 손댈 수가 없는 코드가 돼 있었어요. 심지어 AI한테 다시 물어봐도 이 코드는 어떤 구조인지 파악하기 어렵습니다라고 하더라고요." [00:00]

핵심 대비:

> "같은 AI를 써도 어떤 팀은 AI가 거의 완벽한 코드를 뽑아냈고 어떤 팀은 AI가 만든 코드를 또 AI한테 고쳐 달라고 하는 악순환에 빠져요. 차이가 뭔지 아세요? 코드를 담아두는 구조예요." [00:14]

### 2. 구조 없는 AI 사용의 정량적 위험
화자는 틀(구조)이 없으면 AI 출력의 상당 비율이 부채가 된다고 주장한다.

> "AI는 [매번][^1] 조금씩 다른 방식으로 코드를 만들어요. 정해진 틀이 없으면요. 코드 구조 없이 AI를 그냥 쓰면 열 줄 중에 여섯 줄이 나중에 문제가 되는 코드가 돼요." [01:06]

> [의견] "열 줄 중 여섯 줄"이라는 비율은 화자의 경험에 기반한 표현으로, 측정된 통계 근거는 영상에서 제시되지 않는다.

### 3. DDD(Domain-Driven Design) — 편의점 비유
DDD를 비개발자용 비유로 설명한다.

> "편의점 생각해 보세요. 음료는 음료 코너에, 과자는 과자 코너에, 도시락은 도시락 코너에 있잖아요. 만약 음료 코너에 칫솔이 있으면 어떻게 돼요? 손님도 찾기 힘들고 편의점 알바생도 이거 어디다 진열하지 헷갈리겠죠? 코드도 똑같아요." [01:27]

DDD의 정의:

> "DDD는 비즈니스 개념을 코드로 그대로 표현하는 거예요. 유저, 결제, 주문, 배송 이게 도메인이에요. 비즈니스에서 실제로 쓰는 개념들이요." [01:39]

**도메인 안의 세 가지 기본 구성 요소** (화자가 풀어 설명한 역할):
- 이 개념이 어떻게 생겼냐를 정의하는 파일 — [의견 정리] 모델/엔티티에 해당
- 데이터베이스에서 꺼내오고 저장하는 부분 — [의견 정리] 리포지토리에 해당
- 실제 비즈니스 규칙을 처리하는 부분 — [의견 정리] 서비스에 해당

> "이 세 가지가 기본 [틀][^2]이에요." [01:56]

**핵심 — 단순한 폴더 나누기가 아니다:**

> "DDD는 단순히 폴더 나누는 게 아니에요. 비즈니스 사람들이 쓰는 말과 코드 이름을 완전히 일치시키는 거예요. 영업팀에서 결제 취소라는 용어를 쓴다고 하면 코드에도 똑같이 결제라는 이름이 그대로 있는 거예요. 다만 이때는 보통 영어를 사용을 합니다." [02:01]

### 4. 왜 DDD가 AI에게 효과적인가 — 세 가지 메커니즘

**(A) AI는 창의력이 아니라 패턴 인식으로 코드를 짠다**

> "AI가 코드를 잘 짜는 비결이 뭔지 아세요? 창의력이 아닙니다. 패턴 인식이에요." [02:24]

신입사원 비유: 기존 보고서 양식을 받으면 다음 보고서도 같은 양식으로 쓸 수 있다는 비유로 설명. 유저 도메인 패턴을 본 AI가 결제 도메인을 같은 패턴으로 인식해 복제한다.

> "결제 기능 만들어 줘. 한마디에 유저 도메인과 똑같이 생긴 결제 도메인을 그대로 [복제][^3]해서 만들어 줘요." [02:47]

화자의 직접 경험:

> "직접 해 보니까 이게 진짜였어요. DDD로 구조를 잡아두면 AI가 기존 패턴을 보고 새 기능을 거의 완벽하게 찍어냈어요. 코드 검토도 훨씬 편해졌고 같은 패턴이니까 확인해야 할 포인트도 줄었어요." [02:52]

반대 사례:

> "구조 없이 썼을 때는요. 초반엔 그랬는데 어느 순간 같은 역할을 하는 코드가 여기저기 흩어져 있더라고요. AI가 참고할 패턴이 없으니까 매번 조금씩 다르게 만든 거예요. 나중에 뭔가 고치려면 다 찾아서 다 고쳐야 하는 상황이 됐어요." [03:03]

**(B) 도메인 단위 분리 = AI가 볼 컨텍스트를 줄여 품질을 올린다**

> "AI도 사람이랑 비슷한 게 한 번에 너무 많은 정보를 주면 품질이 뚝 떨어져요. ... 전체 코드를 다 보여 주면서 결제 버그 고쳐 줘 하면 AI가 엉뚱한 파일을 건드리거나 이미 있는 코드를 중복으로 만들어요. 처음이랑 끝은 잘 기억하는데 중간에 있는 내용을 흐릿하게 처리하는 것 같더라고요." [03:15]

> [의견 보강] 화자가 묘사한 "처음과 끝은 기억하고 중간은 흐릿"은 LLM의 lost-in-the-middle 현상을 일상어로 표현한 것이다. (외부 보강 섹션 참조)

DDD가 이를 해결하는 방식 — 의사 진료 비유:

> "마치 의사한테 진료 받을 때 다리 아픈 건데 머리 엑스레이 심장 초음파를 다 갖다 주면 의사도 헷갈리는 것처럼요. 다리 관련 자료만 딱 가져가면 되잖아요." [03:48]

**(C) 말과 코드가 같은 언어를 쓰면 환각이 줄어든다**

> "AI한테 하는 말과 코드가 같은 언어를 써야 합니다." [03:57]

나쁜 예 — 모호한 파일명:

> "데이터 처리라는 파일에 처리해 줘라는 기능. 관리자라는 파일에 관리해 줘라는 기능. AI한테 유저 정보 가져와 줘 했을 때 데이터 처리가 그 역할인지 관리자가 그 역할인지 AI도 몰라요. 그러면 AI가 추측해서 뭔가 만들어 내요. 그게 환각이에요." [04:14]

화자의 경험:

> "저도 처음에 코드 이름을 대충 지었거든요. 그랬더니 AI가 자꾸 엉뚱한 파일에 코드를 만들었어요. 비[즈니][^4]스 언어로 이름 통일하고 나서부터는 AI가 아 이거 여기 있는 거네 하고 바로 찾더라고요. 환각이 확 줄었어요." [04:36]

### 5. DDD + Claude Code 훅 + 스킬 = 도메인 트리거
화자는 DDD의 위력을 완성시키는 요소로 Claude Code의 훅 시스템을 든다.

> "클로드 코드[^5]에는 훅 시스템이 있거든요. 특정 조건이 맞으면 AI가 자동으로 어떤 행동을 하도록 미리 설정해 두는 거예요. 여기에 도메인 트리거를 달아서 스킬을 불러서 사용하면 돼요. 스킬 안에는 각 도메인별로 읽어야 하는 파일들이 분류가 돼 있고 그에 맞춰서 행동해요." [04:55]

동작 예시:

> "제가 결제 환불 기능 만들어 줘라고 하면 AI가 알아서 결제 도메인 파일들을 먼저 쭉 읽어요. 기존에 결제 개념을 어떻게 표현하고 있는지 데이터는 어떻게 저장하고 있는지 어떤 비즈니스 규칙들이 있는지 다 파악한 다음에 코드를 만들어요." [05:12]

**전제 조건 — DDD 구조가 먼저 있어야 한다:**

> "근데 이게 DDD 구조가 잡혀 있어야 제대로 작동해요. DDD 없이 코드가 여기저기 흩어져 있으면 트리거를 달아도 AI가 어디를 읽어야 할지 결제 맥락을 파악할 수 있는지도 몰라요. 도메인 경계가 명확하니까 결제[는] 결제 폴더가 딱 떨어지는 거예요." [05:25]

> "직접 써 보면서 가장 놀랐던 게 이 부분이었어요. 구조 잡고 트리거 달아두니까 AI가 제 코드 스타일을 학습한 것처럼 [읽는][^6] 코드를 알아서 뽑아내더라고요. 이 도메인에서는 이렇게 해라고 매번 설명할 필요가 없어졌습니다." [05:39]

## 실행 가이드 (화자가 제시한 두 가지 상황)

### 상황 1 — 개발자에게 작업을 맡기는 비개발자
> "DDD 구조로 짜 주세요. 비즈니스 개념 단위로 폴더 나눠 주세요라고 요청하실 수 있어요. 이 구조로 짜면 나중에 AI 도구로 유지 보수할 때 훨씬 효율적이에요. 그리고 이 코드 어디 있어요라는 질문도 없어져요. 비즈니스 개념이 폴더 이름이니까요." [05:57]

### 상황 2 — 직접 바이브 코딩하는 사람
화자가 **강력하게 권장**한 핵심 팁:

> "처음 베이스 코드는 손으로 직접 짜시는 게 좋아요. AI한테 구조 잡아 달라고 하면 만들어 주기는 해요. 근데 첫 번째 도메인 하나는 직접 짜[시][^7]는 걸 강력하게 권합니다." [06:15]

이유:

> "어떤 개념을 어떻게 정의하고 데이터는 어떻게 꺼내 오는지 손으로 한번 써 봐야 나중[에] AI 코드가 맞는지 틀린지 판단할 수 있거든요. 이게 없으면 AI가 이상한 코드를 만들어도 뭐가 이상한지 전혀 모르고 그냥 쓰게 돼요." [06:24]

> "첫 도메인 하나는 직접 짜고 AI한테 이 패턴으로 다른 도메인도 만들어 줘 하면 훨씬 정확하게 동작해요. 그리고 이 구조를 유지하면서 기능을 추가할수록 AI가 점점 더 잘 따라와요. 처음 세팅이 중요한 거예요." [06:37]

> [의견] 화자는 자신이 만든 코드 베이스(레포지토리)를 가져다 쓰면 첫 도메인을 직접 짜는 단계를 생략할 수 있다고 안내한다. (구독자 이벤트 + 고정 댓글 링크 연계) [06:46]

## 핵심 요약 (화자가 마지막에 정리한 4가지)
화자가 영상 끝에서 직접 명시적으로 꼽은 4대 핵심:

1. **구조가 패턴이 된다** — "정해진 틀이 있으면 AI가 틀을 복사해서 일관된 코드를 만들어요." [06:58]
2. **컨텍스트 축소가 품질을 올린다** — "도메인 단위로 나누면 AI가 필요한 도메인만 집중해서 볼 수 있어요. 볼 코드가 작아질수록 품질이 올라가요." [07:03]
3. **이름 일치가 환각을 막는다** — "코드 이름이 실제 비즈니스 언어랑 같으면 AI가 어디를 수정할지 바로 찾아요. 이름이 맞지 않으면 AI가 추측을 하고 환각이 생깁니다." [07:10]
4. **도메인 트리거 = 자동 맥락 파악** — "[훅으][^8]로 도메인 트리거를 달아두면 AI가 알아서 관련 코드를 읽고 맥락을 파악합니다. 매번 설명 안 해도 돼요." [07:17]

마무리 비유:

> "편의점 진열[하듯이][^9] 비즈니스 개념 단위로 나눠서 같은 종류는 같은 곳에 두는 거예요. 그리고 진열[할] 때마다 여기엔 이것이라는 표시를 해 두는 거예요. 그게 DDD예요. AI가 여러분 코드를 처음 봤을 때 아 이 팀은 이렇게 일하는구나를 바로 파악할 수 있도록." [07:24]

## 외부 보강
영상이 일상어로 묘사한 개념의 정식 출처를 보강한다. 영상 내용과 섞지 않는다.

- **Domain-Driven Design (DDD)**: Eric Evans의 2003년 저서 *Domain-Driven Design: Tackling Complexity in the Heart of Software*에서 정립된 설계 방법론. 영상의 "비즈니스 언어와 코드 이름 일치"는 DDD의 핵심 개념인 **Ubiquitous Language(보편 언어)**에, "유저/결제 등 경계가 분명한 단위"는 **Bounded Context(경계 컨텍스트)**에 대응한다. (출처: Martin Fowler, "Domain Driven Design", https://martinfowler.com/bliki/DomainDrivenDesign.html)
- **Lost in the middle**: 화자가 "처음과 끝은 기억하고 중간은 흐릿"이라 표현한 현상은 LLM이 긴 컨텍스트의 중간부 정보를 잘 활용하지 못하는 경향을 가리킨다. (출처: Liu et al., "Lost in the Middle: How Language Models Use Long Contexts", TACL 2024, https://arxiv.org/abs/2307.03172)
- **Claude Code Hooks**: Claude Code는 특정 도구 호출 전후 등 이벤트 시점에 사용자 정의 동작을 자동 실행하는 hook 시스템을 제공한다. 영상의 "도메인 트리거 → 스킬 호출"은 hook에서 도메인별 Skill을 트리거하는 패턴으로 해석된다. (출처: Anthropic Claude Code 공식 문서 — Hooks)
