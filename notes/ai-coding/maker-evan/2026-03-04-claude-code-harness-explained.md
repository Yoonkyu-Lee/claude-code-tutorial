## 핵심 개념

### 하니스란?

화자의 핵심 정의:

> "하니스는 AI를 쉽게 쓸 수 있게 도와주는 보조 도구예요." [00:13]

이어 비유로 풀어 설명:

> "여러분이 새 직원을 뽑았다고 생각해 보세요. 이 직원이 아무리 머리가 좋아도 첫날부터 알아서 일 잘하진 않잖아요. 우리 회사에서는 보고서를 이렇게 쓰고 고객 응대는 이런 방식으로 하고 이 상황엔 이 사람한테 물어봐라고 하나하나 알려 줘야 하잖아요. 클로드 코드도 똑같습니다." [00:19]

문제 정의:

> "기본적으로는 무엇이든 할 수 있는 범용 AI인데요. 그 말은 반대로 우리 프로젝트 방식을 아무것도 모른다는 뜻이기도 해요. 하니스는 바로 이 문제를 해결해 줘요. 클로드 코드한테 '우리 팀은 이렇게 일해'를 미리 다 알려주는 하니스라고 보시면 돼요." [00:32]

설치 후 효과:

> "이걸 설치해 두면 매번 클로드한테 '우리 팀 방식대로 해 줘'라고 설명 안 해도 돼요. 알아서 딱 맞게 해 줍니다." [00:43]

### 하니스의 세 구성 요소

화자가 "이 하니스에서는 크게 세 가지만 알고 가시면 됩니다" [00:48]로 제시한 분류.

#### 1) 스킬 (Skill)

> "특정 분야의 전문 지식을 미리 담아둔 문서예요. 일반 클로드가 동네 병원 의사라면 스킬을 붙이면 그 분야 전문의가 되는 거예요." [00:51]

호출 방식:

> "클로드한테 말할 때 앞에 슬래시로 스킬 이름을 붙여 주면 돼요. 예를 들어 그냥 '회원 가입 만들어 줘'라고 하는 대신에 '슬래시 FastAPI 백엔드 가이드라인 회원 가입 만들어 줘.' 이렇게 해요. 이렇게 하면 클로드가 그 스킬에 담긴 전문 지식을 활용해서 대답해 줘요." [01:30]

> *[불명확: 원문 "슬패스트 APA" — 슬래시 + FastAPI로 보정. 호출 표기가 슬래시 명령으로 들어가는지, 일반 발화 안에 키워드로 들어가는지는 영상만으로 단정 불가.]*

#### 2) 에이전트 (Agent)

> "이걸 해결해 줘라고 하면 혼자서 생각하고 관련 내용을 찾아보고 고치고 확인까지 다 해 주는 전문 AI예요. 시키면 알아서 끝까지 해주는 보조 직원이라고 생각하시면 쉬워요." [00:57]

스킬과의 차이를 화자가 명시적으로 구분:

> "스킬은 '이 분야 전문 지식으로 답해 줘'라고 하는 거라면 에이전트는 '이 문제 스스로 해결해 줘'요. 에이전트는 혼자서 관련 파일을 열어서 읽고 뭐가 문제인지 파악하고 고치고 제대로 됐는지까지 확인해요. 중간에 저한테 물어보지 않아도요." [03:29]

#### 3) 커맨드 (Command)

> "자주하는 복잡한 작업을 슬래시 명령어 하나로 바로 즉시 실행하는 단축키나 매크로 같은 거라고 보시면 됩니다." [01:07]

> *영상에는 커맨드의 구체적 사용 예시가 등장하지 않음 — 정의만 제시되고 끝남.*

## 카탈로그

화자가 자신이 배포한 하니스에 포함된 스킬·에이전트를 차례로 소개한다. 커맨드는 정의 외에는 카탈로그가 등장하지 않는다.

### 스킬 6종

| # | 스킬 이름 | 용도 (화자 표현) | 등장 시점 |
|---|---|---|---|
| 1 | FastAPI 백엔드 가이드라인 | "서버 쪽 코드를 짤 때" | [01:46] |
| 2 | Next.js 프론트엔드 가이드라인 | "웹사이트 화면을 만들 때" | [02:10] |
| 3 | Mermaid 스킬 | "복잡한 내용을 그림으로 설명해야 할 때" (흐름도·관계도·구조도 자동 생성) | [02:26] |
| 4 | PDF 스킬 | "글자 뽑아내기, 페이지 쪼개기, 여러 PDF 합치기 … 스캔 문서에서 글씨 인식하기까지" | [02:46] |
| 5 | PPTX 스킬 | "보고서나 제안서를 반복적으로 만들어야 할 때 … 코드로 PPT를 자동 생성" | [02:56] |
| 6 | 웹 디자인 가이드라인 + React Best Practice | "디자인 리뷰자, 리액트 코드를 리뷰하는 리뷰어 역할" | [03:05] |

스킬 1(FastAPI)에 대한 화자의 주장:

> "[의견] 스킬 없이 그냥 물어보면 클로드가 빠르게 동작하는 코드를 만들어 줘요. 근데 나중에 기능이 늘어나면 코드가 엉망이 되는 경우가 많아요. 스킬이 있으면 달라요. 처음부터 역할별로 깔끔하게 나눠서 만들어 줘요. 나중에 수정하기도 쉽고 다른 사람이 봐도 이해하기 쉬운 코드가 나와요." [01:52]

비유:

> "[의견] 마치 서랍장에 물건을 대충 던져 넣는 것과 양말은 양말칸, 셔츠는 셔츠칸으로 정리하는 것의 차이예요." [02:04]

스킬 2(Next.js)에 대한 주장:

> "[의견] 로그인 화면 만들어 줘라고 하면 그냥 단순한 화면이 아니라 최신 방식에 맞게 속도 빠르고 구조도 올바른 화면으로 만들어 줘요. 최근 웹 개발 기술이 빠르게 바뀌면서 어떤 방식이 맞는 방식인지 헷갈리는 경우가 많거든요. 이 스킬이 있으면 클로드가 알아서 올바른 방식으로 판단해 줘요." [02:13]

스킬 3(Mermaid) 사용 흐름:

> "'이 기능 흐름도 그려 줘'라고 하면 화살표 연결도로, '이 데이터 관계 보여줘' 하면 관계도로, '서비스 구조 설명해 줘'라고 하면 구조도로 자동으로 그려 줘요. … 노션이나 깃허브에 붙여 넣으면 바로 그림으로 보여요." [02:32]

스킬 6(디자인/React 리뷰) 사용 맥락:

> "화면을 다 만들고 나서 '이 디자인 리뷰해 줘'라고 하면 색상 대비가 적절한지 여백은 충분한지 체크해 줘요. 혼자 개발하다 보면 디자인 퀄리티를 검토해 줄 사람이 없잖아요. 이 스킬이 디자인 리뷰자, 리액트 코드를 리뷰하는 리뷰어 역할을 해 줘요." [03:09]

### 에이전트 3종

|---|---|---|---|

플랜 리뷰가 필요한 이유:

> "[의견] 계획이 부실하면 나중에 다 뜯어야 하는 상황이 생기거든요." [04:12]

웹 리서치의 가치:

> "[의견] 여러 곳을 동시에 찾아보고 결과를 정리해서 가져오니까 직접 찾는 거보다 훨씬 빠르고 편합니다." [04:29]

## 팁

### 하니스는 커스터마이징해야 진가가 나옴

> "[의견] 하니스를 그대로 써도 좋지만 내 상황에 맞게 바꾸면 훨씬 강력해요." [04:34]

화자가 제시한 두 가지 커스터마이징 방법:

1. **스킬 파일 직접 편집**
   > "우리 회사만의 규칙이 있다면 스킬 파일을 열어서 우리 팀 규칙을 추가하면 돼요. 일반 텍스트 문서라 편집하기 쉬워요." [04:39]
2. **에이전트 신규 작성**
   > "파일 하나만 추가하면 돼요. 우리 회사 방식으로 코드 검토를 해 주는 에이전트, 우리 서비스 특성에 맞는 디버거 같은 것도 만들 수 있어요." [04:45]

### 초기 비용 vs 장기 절약 (화자가 강조한 대조)

> "[의견] 처음에는 설치하는 게 조금 번거롭게 느껴질 수 있습니다. 근데 한번 해 두면 그 이후엔 같은 설명을 반복하지 않아도 되고 매번 일관된 결과물이 나와서 장기적으로 시간이 훨씬 절약돼요." [05:00]

> *"처음에는 ~지만 한번 해두면" 구조는 CLAUDE.md의 "강조한 발언 누락 금지" 룰에 해당해 별도 인용.*

## 미해결 / 영상이 다루지 않은 것

- 각 스킬·에이전트가 실제로 어떤 파일 구조·프롬프트로 구성되는지 (정의·효과만 설명, 내부 구현 미공개)
- **커맨드의 구체적 예시 0건** — 정의만 제시하고 사용 사례·이름·동작을 보여주지 않음
- 슬래시 호출 시 스킬/에이전트/커맨드의 네임스페이스 충돌이나 우선순위 처리
- 하니스를 여러 프로젝트에 어떻게 분리·공유하는지 (전역 vs 프로젝트 단위 설치 구분)
- 화자가 주장하는 "코드 품질 차이"·"시간 절약" 효과의 정량적 근거 — 비유와 인상 위주
- 가이드 문서의 실제 URL — 영상 본문에는 노출되지 않고 더보기/댓글로 분기
