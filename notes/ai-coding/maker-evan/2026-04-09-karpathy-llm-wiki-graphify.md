## 핵심 개념

### 1. 문제 정의: AI에게는 기억이 없다
화자는 현재 대부분의 AI 사용 방식이 비효율적이라고 진단한다.

> "AI한테 물어보면 잘 대답하는데 다음날 또 물어보면 또 처음부터 설명해야 해요. 맥락이 없어요. 기억이 없어요. 매번 새로 시작이에요." [00:00]

> "파일 올리고 질문하고 답 받고 채팅창 닫으면 끝이에요. ... 지식이 전혀 안 쌓여요." [00:18]

구체 예시로, 회사 자료 20개가 있을 때 매 질문마다 AI가 20개를 전부 다시 읽어야 하고, 이것이 토큰(비용)과 속도 저하로 이어진다고 설명한다.

> "이게 토큰이에요. AI가 많이 읽을수록 토큰이 더 날아가거든요. 돈이 나간다고 생각하면 돼요. 그리고 느려요. 자료가 많아질수록 더 느려져요." [00:38]

### 2. 카파시의 LLM-Wiki 개념
테슬라 AI 팀 출신 Karpathy가 공유한 해법. 핵심은 **한 번 읽은 내용을 위키로 정리해 두고 재사용**하는 것.

화자가 든 비유 — 도서관 사서:
> "처음에 책이 들어오면 사서가 그 책 내용을 읽고 요약 카드를 만들어요. ... 그다음에 누가 이 주제 알고 싶어요 하면서 사서는 책 전체를 다시 읽지 않아요. 요약 카드만 봐요." [00:56]

LLM-Wiki의 동작:
- AI가 파일을 읽을 때마다 처음부터 읽는 게 아니라, 한 번 읽은 내용을 **위키 형태**로 정리 (위키피디아처럼 개념마다 페이지, 페이지 간 연결)
- 새 자료가 들어오면 AI가 읽고 기존 위키 페이지를 업데이트하거나 새 페이지를 생성
- 질문 시 AI는 원본 파일을 다시 읽을 필요 없이 위키만 본다

### 3. 핵심 명제: 유지 보수 비용이 거의 제로
영상이 강조한 카파시의 핵심 문장.

> "카파씨가 말한 핵심 문장이 있어요. 유지 보수 비용이 거의 제로에 가깝다고요." [01:35]

이유에 대한 화자의 설명:
> "왜냐면 인간은 이런 위키 유지하다 지쳐서 포기해요. ... 근데 AI는 안 지쳐요. 귀찮아하지 않아요. 파일 15개를 한 번 업데이트해도 불평 안 해요. 그래서 이게 되는 거예요. 인간이 포기했던 걸 AI가 대신해 주니까요." [01:38]

### 4. 카파시 방식의 한계 → Graphify의 등장
화자는 LLM-Wiki 개념의 약점을 지적한다.

> "카파시 방식은 개념이 좋은데 문제가 하나 있어요. 질문할 때마다 AI가 원본 파일을 전부 뒤져야 해요. 목차 읽고 관련 파일 찾고 그 파일 전체 읽고 이게 매 질문마다 반복돼요. 소스가 많아질수록 더 느려지고 토큰도 그만큼 날아갑니다." [02:09]

`[불명확: 원문 "목자 읽고"]` — 컨텍스트상 "목차(目次, table of contents)"를 가리킬 가능성이 높으나 원문 그대로 표기.

이 한계를 메우는 도구가 **Graphify**.

> "그래피파이가 뭐냐면 파일들을 한 번 자동으로 분석해서 지식 지도를 만들어 주는 도구예요." [02:25]

### 5. Graphify의 두 가지 분석 방식
1. **구조 분석**: 파일 안의 내용을 자동으로 훑음. "빠르고 정확" [02:31]
2. **의미 분석**: 서브 에이전트 여러 개가 동시에 문서를 읽어 개념들 사이의 관계를 추출. 문서뿐 아니라 이미지도 분석 [02:36]

> "서브 에이전트는 쉽게 말하면 AI 보조 일꾼들이에요." [02:43]

두 분석을 합치면 지식 지도가 나오고, `graph.json`과 위키 폴더가 자동으로 생성된다. 파일이 바뀌면 바뀐 파일만 다시 처리하고, 안 바뀌면 기존 지도를 재사용한다 (전체 재처리 불필요).

### 6. 토큰 절감 수치
> "최대 71.5배 [...] 절감할 수 있다는 보고가 있어요." [03:12]

`[불명확: 원문 "71.5배 5배 절감"]` — 자동 자막이 숫자를 중복 인식한 것으로 보임. 외부 보강에서 확인한 실제 수치는 71.5x. (아래 외부 보강 참조)

작동 원리:
> "그래프 레포트라는 파일이 있어요. 여기에 전체 지식 지도가 한 페이지에 요약돼 있어요. 어떤 개념이 핵심인지, 어떤 것들이 묶여 있는지, 어떤 질문에 잘 대답할 수 있는지까지 적혀 있어요. AI는 질문 받으면 요약 페이지 먼저 봐요. 그다음 관련 노드 두세 개만 읽어요. 파일 전체를 다 읽을 필요 없어요." [03:18]

> "노드는 개념 하나하나를 담은 카드라고 생각하면 돼요." [03:08]

### 7. 관계의 종류 + 신뢰도 표시 (Graphify의 차별점)
Graphify는 단순 연결이 아니라 **관계의 종류**까지 표현한다.

> "이걸 바탕으로 만들었다. 이걸 참조한다. 이게 근거다. 이런 식으로요. 단순히 연결돼 있다가 아니라 어떤 관계인지까지 알 수 있어요. 그러면 AI가 왜 그렇게 되는지 근거가 뭔지까지 답할 수 있어요." [03:46]

신뢰도(confidence) 표시:
> "원본에 명시된 내용은 확실해요. 합리적으로 추론한 건 중간이에요. 불확실한 건 낮아요. 그럼 AI가 답할 때 이건 확실한 내용이고 이건 추론한 내용이에요. 하고 구분해서 말할 수 있어요." [03:57]

### 8. [의견] 복리 성장
> "[의견] 새 노드가 기존 지도에 연결되면서 전체가 풍부해져요. 자료가 쌓일수록 AI 답변 품질이 올라가요. 이게 복리 성장이에요." [04:16]

## 구축 방법 (시간순 워크플로)

### A. 카파시 방식 (도구 없이, 폴더 + 텍스트 파일만)
> "구조가 진짜 단순해요. 폴더랑 텍스트 파일만 있으면 돼요." [04:28]

세 가지 요소만 필요:
1. **원본 파일** — `raw/` 폴더에 넣음 (기사, 논문, 이미지 무엇이든)
2. **AI가 만든 위키** — `wiki/` 폴더에 들어감
3. **`CLAUDE.md`** — AI에게 위키를 어떻게 관리할지 알려주는 지시. "새 파일이 들어오면 위키 업데이트해" 식으로 적어둠

> "새 파일이 들어오면 위키 업데이트해 이렇게 적어 두면 돼요. 그게 끝이에요. 진짜로요." [04:37]

운영 흐름:
- 자료를 읽고 파일로 저장 → AI에게 "이거 처리해 줘" → AI가 위키 페이지 10~15개를 알아서 업데이트
- 다음 질문 시 AI는 `INDEX.md`(목차)를 먼저 보고 관련 페이지만 읽어 답함

### B. Graphify를 더하는 방식
파일이 많거나 이미지가 섞여 있으면 Graphify가 자동으로 지식 지도를 생성.
- `graph.json`과 위키 폴더, 전체 요약 페이지(GRAPH_REPORT.md)가 생성됨
- 이를 `CLAUDE.md`에 연결: "질문 전에 요약 페이지 먼저 봐" 한 줄

> "질문 전에 요약 페이지 먼저 봐. 이 한 줄이에요. 그러면 AI가 매 질문 전에 자동으로 지도부터 탐색해요." [05:06]

## 팁 (화자가 강조한 주의사항)
Graphify 사용 시 `CLAUDE.md`에 세세한 프롬프팅이 필요하다는 점을 화자가 강조한다.

> "그래픽 파일을 사용할 시 클로드에다가 그래 명령어라든지 이런 거를 쓰지 말라고 요청을 해야 되고 무조건 그래프.json 파일을 읽고 관련 노드만 읽으라고 명령을 좀 해 줘야 돼요." [05:23]

`[불명확: 원문 "그래 명령어"]` — 외부 보강(Graphify 공식 동작)에 비추면 `Grep`/`Glob` 같은 원본 파일 전수 검색 명령을 막으라는 의미로 추정되나, 자막만으로는 단정 불가하여 원문 표기.

> "클로드.md에 대해서 좀 세세한 프롬프팅만 있다면 그 뒤에 프로세스들은 문제없이 작동하는 걸 확인할 수 있습니다." [05:36]

즉 **CLAUDE.md 프롬프트 설계가 이 전체 워크플로의 신뢰성을 좌우한다**는 것이 화자의 핵심 당부.

## 카탈로그 (영상에 등장한 산출물·파일)
| 이름 | 역할 |
|---|---|
| `raw/` 폴더 | 원본 파일 보관 (기사·논문·이미지) |
| `wiki/` 폴더 | AI가 만든 위키 페이지 |
| `CLAUDE.md` | AI에게 위키 관리 방식을 지시하는 룰 파일 |
| `INDEX.md` | 위키 페이지 목차 (카파시 방식에서 질문 시 첫 진입점) |
| `graph.json` | Graphify가 생성하는 지식 지도 데이터 |
| GRAPH_REPORT.md (영상: "전체 요약 페이지"/"그래프 레포트") | 전체 지식 지도를 한 페이지로 요약한 진입점 |

### Karpathy의 LLM-Wiki 개념
- Karpathy가 공유한 `llm-wiki.md` 문서가 원전. 기존 RAG와 달리, AI가 마크다운 라이브러리를 스스로 만들고 유지하는 패턴. "수집(쉬움) → 정리(어려움) → 유지(불가능에 가까움)"에서 인간이 포기하는 '유지' 단계를 LLM이 거의 제로 한계비용으로 대신한다는 것이 핵심. (출처: [VentureBeat](https://venturebeat.com/data/karpathy-shares-llm-knowledge-base-architecture-that-bypasses-rag-with-an), [MindStudio: LLM Wiki Pattern](https://www.mindstudio.ai/blog/karpathy-llm-wiki-knowledge-base-pattern))
- 개념적으로 Vannevar Bush의 1945년 에세이 "As We May Think"의 Memex 장치를 참조 — 수동 유지 때문에 실패했던 아이디어를 LLM이 실현한다는 맥락. (출처: [Starmorph blog](https://blog.starmorph.com/blog/karpathy-llm-wiki-knowledge-base-guide))

### Graphify 도구
- 오픈소스 지식 그래프 스킬. Claude Code, Codex, OpenCode, Cursor, Gemini CLI 등 다중 에이전트 지원. 코드·SQL 스키마·R/셸 스크립트·문서·논문·이미지·영상을 쿼리 가능한 지식 그래프로 변환. (출처: [GitHub safishamsi/graphify](https://github.com/safishamsi/graphify))
- 내부 구현: Tree-sitter(구조 파싱), NetworkX(그래프), Leiden clustering(개념 군집화). 산출물은 `graphify-out/` 폴더에 `graph.html`(인터랙티브), `GRAPH_REPORT.md`(한 페이지 감사 리포트), `graph.json`(영속 데이터). (출처: [graphify.net](https://graphify.net/))
- **토큰 절감 수치(영상의 "71.5배" 확인)**: 52개 혼합 코퍼스(코드+논문+이미지)에서 쿼리당 약 1.7k 토큰 vs 원본 직독 약 123k 토큰 → **71.5배 감소**. 500+ 파일 프로젝트에서는 최대 70배 절감 보고. (출처: [MindStudio: Graphify 70x](https://www.mindstudio.ai/blog/graphify-claude-code-knowledge-graph-large-codebase-70x), [CLSkills](https://clskillshub.com/blog/graphify-claude-code-integration))
- **CLAUDE.md 프롬프팅 주의(영상 [05:23] 확인)**: Claude가 `Grep`/`Glob`로 원본 파일을 뒤지기 전에 hook이 "knowledge graph exists. Read GRAPH_REPORT.md ... before searching raw files"를 주입. 영상에서 화자가 말한 "그래 명령어... 쓰지 말라"는 이 `Grep`/`Glob` 전수 검색 차단을 가리키는 것으로 보강 확인됨. (출처: [GitHub safishamsi/graphify](https://github.com/safishamsi/graphify))
