## 카탈로그 — 도구 6개 + 보너스 2개

### 1. 슈퍼파워즈 플러그인 (Superpowers)

> "스킬이 아니라 플러그인입니다. 안에 여러 스킬들이 같이 있어요." [01:19]

화자가 정의하는 역할:

> "클로드가 시니어 개발자처럼 일하게 강제해 주는 워크플로우 묶음이에요. 바로 코드 짜는게 아니라요. 일단 멈춰서 계획부터 세워요. 테스트 먼저 쓰고 그다음에 코드 쓰고. 끝나면 자기가 두 번 리뷰해요. 한 번은 스펙이 맞는지, 한 번은 코드 품질 어떤지요." [01:24]

해결하는 문제:

> "클루드 코드 쓸 때 가장 큰 실패 패턴이 급하게 짠 코드거든요. 시키면 바로 코드 뱉터요. 보면 그럴 듯. 근데 막상 돌리면 엣지 케이스 다 깨져요. 혹은 더 나쁘게는 클라이언트가[가] 유저가 돌리는데 깨져요." [01:39]

> "슈퍼파워즈는 클로드 속도를 충분히 늦혀서요. 문제를 제대로 생각하게 만들어요. 단계를 건너뛰지 못하게요." [01:52]

화자가 실제로 쓰는 하위 스킬은 **2개**:

#### 1-a. 브레인스토밍

> "세[새] 기능 만들기 전에 무조건 켜는 거예요. 뭐 만들까 같은 모[모호]한 거 들고 가면요. 질문을 던져 줘요. 누가 쓸 거예요? 성공 기준이 뭐예요? 어떤 문제 해결하는 거예요? 이 질문들 답하면서 요구 사항이 정리되고 그다음에야 코드가 나와요. 이거 한 단계 끼우는 것만으로도 헛코드가 진짜 많이 줄어요." [02:06]

#### 1-b. 서브에이전트 드리븐 디벨로먼트 (Subagent-Driven Development)[^2]

[^2]: 자막의 "sub브에 드리브로먼트" / "서브에 드리븐 디벨로먼트"는 영문 표기 `Subagent-Driven Development`의 한글 음역. 본 저장소 다른 노트(`2026-04-15-gstack-superpowers-workflow-review.md`, `2026-03-27-superpowers-plugin-explained.md`)에서 동일 용어가 정리되어 있다.

> "계획이 정리된 다음에 쓰는 거예요. 큰 일을 서버[서브] 에이전트들한테 나눠서 시켜요. 가장 깨끗한 컨텍스트에서 자기 일만 해요. 끝나면 다른 서브 에이전트가 결과물을 두 단계로 리뷰해요. 하나는 스펙대로 됐는지 하나는 코드 품질이 괜찮은지요. 메인 세션이 안 들[더]어지니까 한 세션이 훨씬 오래 가요." [02:23]

설치:

> "클로드 코드에서 이 명령을 한 줄만 치시면 됩니다. 플러그인 인스톨 수퍼파워즈 클로드 플러그인 오피셜 공식 마켓플레이스에서 바로 받아줘요." [02:41]

> "저는 글로벌로 다 깔아놨거든요. 어떤 프로젝트에 들어가도 자동으로 켜지게요." [02:52]

### 2. GStack — 그중 `office`와 `browser` 두 개만

> "Y 콤비네이터 게리타[게리 탄]이 직접 쓰는 셋업이라고 공개한 거예요." [02:58]

> "GStack 안에 들어 있는 것 중에서 제가 매일 쓰는 건 두 개뿐입니다. 오피스랑 브라우즈[브라우저]예요." [03:02]

#### 2-a. office

> "새 아이디어 잡을 때 쓰는 스킬이에요. 이거 만들면 어떨까? 들고 가면 Y 콤비네이터에서 사무실에서 받는 거랑 비슷한 질문을 여섯 개 던져 줘요. 진짜 수요가 있어요. 기존에 다들 어떻게 해결하고 있어요? 가장 좁은 진입점은 뭐예요?" [03:10]

> "이 질문들 답하다 보면 막연했던 사업 아이디어가 굉장히 좁아지고 날카로워져요. 저는 새 영상 기획할 때도 켜는 거 같습니다. 새 사이드 프로젝트 시작할 때도 켜요." [03:24]

> "스크린샷 찍고 로그인하고 폼[을] 채[우]고 콘솔 에러 잡고요. 저는 만든 사이트 QA 할 때 매번 켜요. 진짜 빠르고 컨텍스트도 안 더러워져요." [03:46]

### 3. 매포의 `구워줘`(Grill Me)[^3]

[^3]: 자막은 "그릴미"·"그림미"로 흔들리며 화자가 "이름은 그대로 구워줘라는 뜻이에요"라고 직접 의미를 풀어준다. 영문 스킬명은 `grill-me` 계열로 추정되나 정확한 식별자는 [불명확].

> "매퍼[매포][^4]이라는 분이 만든 스킬이에요. 이분 타입스크립트 쪽에서 진짜 유명한 분이시거든요. 강의도 하시고 유튜브도 운영하시고 하고 있어요." [04:24]

[^4]: 화자가 사람 이름을 "매퍼" 또는 "매포"로 발음 — 자막은 양쪽이 섞여 있다. 본 노트는 후반의 정리 발언("셋 매포의 그릴미") 기준으로 `매포`로 통일.

> "이름은 그대로 구워줘라는 뜻이에요. 제가 짱[짠] 코드나 설계를 크로드[클로드]한테 드리밀면요. 여기 이상한데 이거 왜 이렇게 했어요? 이런 식으로 막 [캐]물어요." [04:36]

해결하는 문제 — 1인 개발자의 코드 리뷰 부재:

> "1인 개발자분들이 진짜 많이 공감하실 거예요. 동료가 없으니까 코드 리뷰 받을 데가 없어요. 혼자 짜고 혼자 [넘]어가고[부끄러운][^5] 실수가 그대로 나가요. 그릴미는 그 동료 역할을 해 줘요." [04:45]

[^5]: 자막 "혼자 짜고 혼자 뭐지하고 부끄러운 실수가 그대로 나가요"는 발음 인식 노이즈. 의도는 "혼자 짜고 혼자 넘어가다 보면 부끄러운 실수가 그대로 나간다" 정도로 읽힘 — 단정 보정 대신 원문을 남기고 [브래킷]으로 추정 보강.

사례:

> "리액트 컴포넌트 하나 짰어요. 잘 짠 거 같아요. 근데 그림미[구워줘] 돌리면요. 이 유즈 스테이트[useState] 왜 여기 있어? 이게 부모 컴포넌트로 올려야 되는 거 아니야? 뭐 이런 식으로 잡아내요." [04:56]

> "저는 PR 올리기 전에 무조건 그림이[구워줘] 한번 돌리거든요. 그런 실수가 거의 안 나가도록 설정을 할 수 있습니다." [05:06]

### 4. 매포의 `Improvement Architecture`[^6]

[^6]: 자막은 "임프로브먼트 아키텍처" / "인프로먼트 아키텍처" / "임프먼트 아키텍처"로 흔들린다. 영문 표기는 `improvement-architecture` 계열로 추정.

> "구조 통째로 다시보기 스킬이라고 보시면 될 거 같습니다. 이미 있는 코드[들][고]임이면서[을 보면서] 어떻게 더 좋게 만들 수 있을까 물어보면요. 여기 이렇게 분리하면 테스트 쉬워져. 뭐 이 부분 추상[화]하면 재사용 가능해 뭐 등등 해 줘요." [05:12]

해결하는 문제 — 리팩토링 판단의 막막함:

> "리팩토링 할 때 진짜 막막한 부분이거든요. 내가 너무 과하게 분리하는 건가? 아니면 더 분리해야 하나? 이게 혼자 보면 잘 안 보여요. 근데 이 스킬 돌리면 한두 개 진짜 좋은 제[안]이[제]한이 나와요. 물론 100% 다 받아들이진 않습니다." [05:31]

> "저는 솔직히 그릴미나 인프로먼트[Improvement] 아키텍처는 둘 다 매일 쓰는 거 같습니다." [05:43]

### 5. Codex Review

> "이건 오픈 AI 코덱스 쪽 스킬인데요. 클로드 안에서 쓸 수 있도록 플러그인 형태로 오픈 AI 측에서 만든 거더라고요." [05:49]
