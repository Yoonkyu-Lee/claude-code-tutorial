## 한 줄 요약
바이브 코딩에서 AI에게 어떤 언어로 코드를 짜게 하느냐가 첫 시도 성공률에 영향을 주며, Go는 단순한 설계·실행 전 오류 검사·내장 동시성 덕분에 백엔드·AI 에이전트 영역에서 "한 번에 돌아가는 코드"를 뽑아낼 가능성이 상대적으로 높다.

## 핵심 개념

### 1. 언어 선택이 바이브 코딩 성공률을 좌우한다
화자는 같은 "만들어 줘" 요청인데도 어떤 사람은 오류만 나고 어떤 사람은 수정 없이 바로 돌아가는 차이를 짚으며, 그 요인 중 "의외로 많이 간과하는" 것이 언어 선택이라고 본다.

> "AI한테 이거 만들어 줘 하고 코드 받았는데 실행만 하면 오류만 계속 났던 적 많으시죠? 근데 저 아는 사람은 AI한테 만들어 줘 하면 그냥 바로 돌아가거든요. 수정 한번 없이." [00:00]

> "요인이 여러 가지 있는데 그 중에 의외로 많이 간과하는게 언어 선택이에요. 어떤 언어로 AI한테 코딩을 시키느냐에 따라 성공률이 꽤 달라질 수 있거든요." [00:11]

### 2. 바이브 코딩의 정의 (영상 내 정의)
> "원래 코딩이라고 하면 코드를 직접 한 줄 한 줄 타이핑하는 거잖아요. 근데 바이브 코딩은 달라요. AI한테 이런 기능 만들어 줘라고 말로 설명하면 AI가 코드를 짜 줘요. 저는 그걸 그냥 실행만 하는 거고요." [00:21]

### 3. 부엌 비유 — 잘 정리된 재료 = 일관된 언어
화자가 언어별 차이를 설명하는 핵심 비유.

> "요리를 부탁할 때 재료가 잘 정리된 부엌에서 요청하면 요리사가 바로 만들어 주잖아요. 근데 재료가 뒤죽박죽이면 요리사가 재료 찾다가 시간 다 보내고 뭘 빠뜨리기도 하죠. 언어도 똑같아요. AI가 실수를 덜하는 언어가 있고 실수를 자꾸 하는 언어가 있어요." [00:40]

### 4. Go가 AI 코딩에 유리한 세 가지 이유

**(1) 단순함을 최우선으로 설계 + 코드 스타일 자동 통일**
- Go는 2009년 구글이 만든 언어이며, 목표가 "단순하게 만들자" 하나였다.
- 당시 이미 언어가 수십 가지였지만 구글 엔지니어들 보기엔 다들 너무 복잡했고(기능 많고 배우기 어렵고 사람마다 다르게 써서 코드가 제각각), 그래서 처음부터 단순함을 최우선으로 설계했다.
- 저장하면 코드 스타일을 자동으로 통일해 주는 도구가 언어 자체에 내장돼 있다 (gofmt — 외부 보강 참조).

> "이게 왜 중요하냐면요. AI가 학습한 그 코드는 어디서 봐도 비슷하게 생겼거든요. 사람마다 스타일이 달랐던 다른 언어들과 달라요. 그래서 AI가 Go스러운 코드를 훨씬 일관성 있게 배울 수 있었어요. 결과적으로 AI한테 Go로 코드 짜 달라고 하면 처음 시도에서 작동할 확률이 높은 편이에요." [01:20]

**(2) 실행 전 오류를 미리 잡아준다**
- Go는 코드를 실행하기 전에 먼저 검사하는 단계가 있다.

> "마치 요리 레시피에서 재료 목록을 미리 확인하는 것처럼요. AI가 뭔가 잘못 만들면 실행도 하기 전에 여기 문제 있어요라고 알려 줘요." [01:41]

> "실행해 보고 오류 나고 다시 물어보고 또 오류 나는 그 사이클이 훨씬 짧아지는 거예요." [01:48]

[불명확: 원문 "검사 단기가 있어요" / "올린하고" — 컨텍스트상 "검사 단계가 있어요" / "오류 나고"로 보이나 자막 오인식. 의미는 컴파일/타입 검사 단계를 가리킴]

**(3) 동시 작업(동시성)이 몸에 배어 있다**
- AI 에이전트는 여러 API에 동시에 묻고 여러 곳에서 정보를 가져와 합치는 작업이 잦은데, Go는 이런 동시 작업을 빠르게 처리한다.

> "비유하면 Go는 멀티태스킹이 원래부터 몸에 밴 사람이에요. 다른 언어들은 멀티태스킹 하려면 별도 도구를 설치하거나 설정해야 하는 경우도 많거든요." [02:03]

### Go로 만들어진 실제 AI 인프라 (영상 언급 사례)
| 도구 | 영상 내 설명 |
|---|---|
| **Ollama** | 내 컴퓨터에서 AI를 직접 돌릴 수 있게 해주는 도구. Go로 만들어짐 |
| **Milvus** | AI가 기억을 저장하는 데이터베이스(벡터 DB). Go로 만들어짐 |
| **구글 AI 에이전트 개발 도구** | 구글이 2025년 AI 에이전트 개발 도구를 공식 발표하면서 Go를 공식 지원 언어로 추가 |

> "실제로 AI 인프라를 만든 사람들 중에 Go를 선택하는 경우가 늘고 있다는 거예요." [02:28]

> 각주: 구글의 "2025년 AI 에이전트 개발 도구"는 영상에서 이름이 명시되지 않음. 정황상 ADK(Agent Development Kit)로 추정되나 영상이 이름을 말하지 않았으므로 [불명확]로 둔다. 외부 보강 참조.

### 언어별 적합 영역 (화자가 짚은 구분 — Go가 만능이 아님)
| 영역 | 화자 추천 언어 | 근거 (영상 내) |
|---|---|---|
| AI 모델을 직접 만들고 학습 | **Python** | 생태계가 훨씬 크고 자료가 많음 |
| 에이전트 프레임워크 사용 | **Python** | 생태계가 워낙 방대하고 다양해서 유리 |
| 웹사이트·앱 화면단(프론트엔드, 사용자가 직접 보는 쪽) | **TypeScript** | 더 익숙하고 예시도 많음 |
| 서버 API·백엔드·AI 에이전트 로직 (뒤에서 돌아가는 쪽) | **Go** | 이 영역에서 강점 |

> "여기서 한 가지 말씀드리자면 Go가 모든 상황에 맞는 건 아닙니다." [02:33]

> "근데 AI 에이전트나 백엔드 쪽에서 뭔가 만들어 보려는 분이라면 그거를 한번 고려해 보시는 것도 충분히 가치 있다고 생각해요." [03:14]
- [의견] 언어 선택이 "생각보다" 성공률에 영향을 준다는 것이 영상의 일관된 주장이며, "꽤 달라질 수 있다"·"높은 편"·"가능성이 상대적으로 높은" 같은 절제된 표현으로 단정은 피한다.

## 외부 보강

> 아래는 영상이 다루지 않았거나 이름을 명시하지 않은 부분을 공식·1차 출처로 보강한 것이다. 영상 내용과 섞지 않는다.

- **gofmt**: 영상이 말한 "저장하면 코드 스타일을 자동으로 통일해 주는 내장 도구"는 Go의 `gofmt`(및 에디터 통합 `goimports`)를 가리킨다. Go 표준 배포에 포함되며, Go 팀은 "단 하나의 표준 포맷"을 강제해 스타일 논쟁을 없애는 것을 명시적 설계 목표로 삼는다. (출처: go.dev — "gofmt formats Go programs ... using tabs for indentation")
- **Go의 동시성**: 영상이 말한 "몸에 밴 멀티태스킹"은 Go의 goroutine·channel을 가리킨다. 언어 키워드 `go`로 경량 동시 실행 단위를 만드는 것이 언어 차원에 내장돼 있다. (출처: go.dev/tour — Concurrency)
- **구글 ADK (Agent Development Kit)**: 영상이 이름을 말하지 않은 "구글 2025년 AI 에이전트 개발 도구"는 정황상 Google의 Agent Development Kit로 보인다. ADK는 Python으로 먼저 출시된 뒤 Go 지원이 추가되었다. 다만 영상이 명시하지 않았으므로 동일 대상이라는 단정은 피한다. (출처: google.github.io/adk-docs — 교차 확인 권장)
- **Ollama / Milvus**: 둘 다 주 구현 언어가 Go인 것은 사실이다. Ollama는 로컬 LLM 실행 도구, Milvus는 오픈소스 벡터 데이터베이스. (출처: 각 프로젝트 GitHub 저장소)
