## 한 줄 요약

## 핵심 개념

### 1. Anthropic이 푼 진짜 문제 세 가지

화자는 회사 데이터 분석을 AI에 넘길 때 막혔던 세 가지 근본 원인을 정리한다.

> "이거 진짜 중요하니까 천천히 들어 주세요. 첫 번째가요. 개념이랑 실제 데이터 사이 애매함이에요." [01:47]

1. **개념과 실제 데이터 사이의 애매함**
   - 예: "활성자"가 무엇인가? 로그인하면? 글 써야? 가짜 계정 포함? 며칠 기준?
   - 비유: "신입 직원한테 '단골 손님 수 좀 세 줘'라고 한 거랑 같아요. 단골의 기준이 뭔지 안 정해 주면요. 신입은 자기 맘대로 세요." [02:09]

2. **데이터가 자꾸 낡는다**
   - 테이블 구조·정의·로직이 계속 바뀜 → "어제 맞던 게 오늘 틀려요." [02:21]

3. **정보를 못 찾는다 (분류표 부재)**
   - 비유: "도서관의 책은 다 읽는데요. 분류표가 없는 거예요. 찾고 싶은 책이 있어도 수만 건 사이에서 헤매는 거죠." [02:32]

### 2. 4층짜리 구조 — Anthropic이 지은 "집"

> "이제부터 그 유명한 4층짜리 구조를 볼 거예요. 집 짓는다고 생각하시면 편해요. 1층부터 차근차근 올라가요." [02:40]

#### 1층: 데이터 기초 공사

핵심 개념: **표준 데이터셋 (Single Source of Truth)**[^1]

> "이게 뭐냐면요. 단 하나의 진실의 원천이에요. One True Source of Truth라고 부르는 그런 친구입니다." [02:54]

- "매출"이 시트마다 다르게 계산되는 문제를 "매출은 무조건 이 데이터를 봐라"로 못 박음
- "40개의 그럴듯한 후보가 아니라 딱 하나로요." [03:07]
- 비유: 음식점 표준 레시피 — 알바가 바뀌어도 김치찌개 맛이 같으려면 레시피가 하나여야 함

**강제 메커니즘**: CI/CD[^2]
> "코드 올릴 때 자동으로 검사해 주는 컨베이어 검수대예요. 규칙 안 지키면 아예 통과를 못 시켜요." [03:27]

**메타데이터를 1등 시민으로**
- 설명, 정의, 출처, 소유자 정보 → 부록이 아니라 제품처럼 관리

#### 2층: 진실의 원천들 (Sources of Truth)

신뢰도 순으로 쌓은 4단 구조:

| 단(위→아래) | 이름 | 역할 |
|---|---|---|
| 최상 | 시멘틱 레이어 | 회사가 공식 인증한 지표 계산기. "활성자는 이렇게 계산한다"를 시스템이 보장 |
| 2단 | 데이터의 족보 (lineage) | 데이터가 어디서 와서 어떻게 변했는지 흐름 추적 |
| 3단 | 과거 쿼리 모음 | 예전 SQL을 그냥 쌓는 게 아니라 **정리된 참고 문서로 다듬은** 것 |
| 최하 | 회사의 맥락 지식 (Context) | 로드맵, 의사결정 기록, 조직도 |

**중요한 팁** (영상에서 두 번 강조):
> "날것으로 SQL을 그냥 검색하게 했더니만 정확도가 1%도 안 올랐대요. 다듬어야 의미가 있다는 거예요. 이거 뒤에서 또 나와요. 기억해 두세요." [04:28]

핵심 문장:
> "매출이라는 단어가 40개 후보가 아니라 딱 하나의 데이터로 연결되면 문제는 거의 사라진다." [04:45]

#### 3층: 스킬 — 정확도를 뒤집은 마법의 층

> "이제 나올 3층이요. 정확도를 21%에서 95%로 뒤집은 마법의 층이거든요." [04:56]

**스킬 = 절차적 지식 = 일하는 방법**
- 어떤 자료를 먼저 봐야 하는지
- 애매할 땐 어떻게 헤쳐 나가는지
- 완성된 분석은 어떻게 생겨야 하는지

> "비유하자면요. 신입 직원한테 주는 선배의 업무 매뉴얼이에요." [05:22]

**충격적인 숫자**:
- 스킬 없을 때: 정확도 21% (5번 중 1번)
- 스킬 도입 후: 95%, 일부 영역 99%
> "같은 AI인데 매뉴얼 하나 줬다고 이렇게 바뀐 거예요." [05:43]

**스킬을 둘로 나눈 영리한 설계**:

1. **지식 스킬 (Knowledge Skill)** — "안내 데스크"
   - 화자가 부른 이름: "최상의 라우터"[^3]
   - 질문이 들어오면 영역을 판단해 참고 문서 약 30개를 골라 불러옴
   - "수만 개를 다 뒤지는 게 아니라요. 잘 고른 30개만요." [05:58]
   - 비유: 도서관 분류표

2. **언북 스킬 (Unbook Skill)**[^4] — "베테랑 분석가의 일하는 순서"
   - 질문 명확화 → 자료 탐색 → 실행 → **적대적 검토** → 패턴 재사용
   > "적대적 검토를 해요. 자기가 낸 답을 자기가 의심하면서 다시 따져 보는 거예요." [06:15]
   - 재사용 가능한 검증된 툴(예: 리텐션 곡선, 문제 분해 패턴) 호출

**참고 문서의 표준 구조**:
- 메뉴 / 빠른 참조 (데이터 맥락 + 기본 필터)
- 차원 설명
- 핵심 테이블
- 주의 사항 + 자주 쓰는 패턴
- 다른 문서로의 연결

> "군더더기 없이 딱 필요한 것만 짜여 있어요." [06:44]

#### 3층의 함정: 한 달 만에 95% → 65%

> "스킬을 다 만들고요. 정확도 95%를 찍었어요. 다들 좋아했겠죠? 근데요. 딱 한 달이 지났어요. 정확도가 95%에서 65%로 떨어졌어요." [06:48]

원인: 회사는 계속 변하는데 스킬 문서는 그대로 → 문서가 낡음.

비유:
> "가게 메뉴는 바뀌었는데 신입한테 준 매뉴얼은 작년 거인 거예요. 신입이 열심히 일할수록 더 틀리는 거죠." [07:11]

**Anthropic의 해결책**:
- 스킬 문서를 데이터 코드와 **같은 저장소**에 넣음
- CI 훅을 걸어 "데이터 모델을 바꾸면 스킬 문서도 같이 바꿔야 통과"로 강제
- 결과: 코드 수정의 90%가 스킬 업데이트를 동반

핵심 교훈:
> "낡은 문서는요. AI 성능을 모델 한 개보다 더 빨리 죽여요." [07:36]

#### 4층: 검증

세 갈래로 구성:

**① 오프라인 평가 (Offline Eval)**
- 정답이 알려진 문제 세트로 시험
- 시험 문제 자체도 Claude가 만들고 사람이 검수
- 영역별 90% 미만이면 출시 차단

**② 반전 실험 — 과거 SQL 다 던져주기**
> "AI한테 과거 SQL 수천 개를 통째로 주면 어떨까? ... 무려 그 파일들의 80%가 관련을 갖고 있었어요. 정답이 거기 다 있었던 거예요. 결과가 어땠을까요? 정확도가 1%도 안 올랐어요." [08:05]

> "진짜 문제는 구조였어요. 질문을 데이터에 어떻게 연결하느냐 그 다리가 없었던 거예요." [08:24]

비유:
> "신입한테 10년치 영수증을 박스째 던져 준 거예요. '여기 답 다 있어' 하고요. 근데 정리가 안 돼 있으니 못 쓰는 거죠." [08:33]

> "데이터는 쌓는 게 아니라 구조화하는 거예요. 이게 핵심 교훈입니다." [08:38]

**③ 실시간 검증 — 적대적 검토 (Adversarial Review)**
AI가 답을 내놓으면 다른 AI 검토관이 의심하고 따짐.

**트레이드오프 (공짜 점심 없음)**:
| 지표 | 변화 |
|---|---|
| 정확도 | +6% |
| 토큰 | -32% (비용 절감) |
| 응답 시간 | +72% (느려짐) |

> "정확도를 얻는 대신 속도를 내주는 거죠. 이게 현실이에요. 공짜 점심은 없습니다." [09:06]

**출처 푸터 (Source Footer)**
답에 출처 등급, 신선도, 소유자를 꼬리표로 붙임 → "식당 메뉴판에 원산지 표시"하는 것과 같음 [09:20]

**자동 교정 수집**
예약된 에이전트가 채팅 채널을 돌아다니며 "이 숫자 틀렸어요"를 잡아 수정안을 만듦 → 시스템이 스스로 학습

### 3. Anthropic이 직접 짚은 6가지 핵심 교훈

> "여러분 회사에 그대로 적용될 수 있는 거니까 잘 들어 주세요." [10:07]

1. **데이터는 소프트웨어가 아니다.** 코드는 맞는지 증명할 수 있지만, 데이터 분석은 정답을 딱 증명할 방법이 없다. 진짜 문제는 코드 생성이 아니라 애매함 그 자체.
2. **정확도는 결국 맥락과 검증의 문제.** 모델이 똑똑한가의 문제가 아니다.
3. **강제가 없는 규칙은 빠르게 썩는다.** 정해만 두고 강제 안 하면 무너진다.
4. **낡은 문서는 모델 한 개보다 성능을 더 빨리 죽인다.** (한 달 만에 65%로 떨어진 사례)
5. **AI는 문서 만드는 건 잘한다. 근데 정의는 사람이 책임져야 한다.** "활성자가 뭔지는 AI가 정하면 안 돼요. 사람이 정하는 거예요." [10:43]
6. **과거 쿼리는 그냥 쌓지 말고 정리된 패턴으로 다듬어야 한다.** (1%도 안 올라간다는 사실이 두 번 등장)

### 4. 최소 시작 지점 — 거창하지 않게 딱 3개

Anthropic이 안내한 시작점:

> "거창하게 시작 안 돼요. 딱 세 개면 됩니다." [11:00]

1. **표준 데이터셋 몇 개** — 모든 걸 정리할 필요 X, 핵심 데이터 몇 개만 Source of Truth로 지정
2. **오프라인 평가 몇 십 개** — 정답 있는 시험 문제 수십 개를 만들어 측정에 사용
3. **얇은 지식 스킬 하나** — 처음부터 두껍게 만들지 말고, 안내 데스크 역할만 하는 아주 얇은 스킬 하나

이 셋으로 시작해 상황(오류 허용도, 비즈니스 복잡도, 사용자 기술 수준, 정확도 vs 속도, 접근 관리)에 맞춰 한 층씩 쌓아 올림.

> "잡기 시작해서 측정하고 강제하고 관리한다. 이 네 박자예요." [11:52]

## 카탈로그 — 영상에 등장한 도구·개념

| 항목 | 분류 | 영상 내 설명 |
|---|---|---|
| Single Source of Truth (표준 데이터셋) | 1층 데이터 기초 | "매출은 이 데이터를 봐라"로 못 박는 단 하나의 원천 |
| CI/CD | 1층 강제 메커니즘 | 규칙 안 지키면 통과 못 시키는 컨베이어 검수대 |
| 시멘틱 레이어 | 2층 최상단 | 회사 공식 지표 계산기 |
| 데이터 lineage (족보) | 2층 | 데이터 출처와 변환 흐름 |
| 과거 쿼리 정리본 | 2층 | 날것 SQL 아닌 다듬어진 참고 문서 |
| 컨텍스트 지식 | 2층 | 로드맵, 의사결정, 조직도 |
| 지식 스킬 (Knowledge Skill) | 3층 | 안내 데스크 / 라우터, 30여 개 문서를 선별 호출 |
| 언북 스킬 (Unbook Skill) | 3층 | 베테랑의 일하는 순서 + 적대적 검토 + 패턴 재사용 |
| 적대적 검토 (Adversarial Review) | 3·4층 | 자기 답을 자기가 의심하는 검토 단계 |
| 오프라인 평가 | 4층 | 정답 있는 시험 세트, Claude가 문제 생성·사람 검수 |
| 출처 푸터 (Source Footer) | 4층 | 답의 출처 등급·신선도·소유자 꼬리표 |
| 자동 교정 수집 | 4층 | 채팅 채널 모니터링 → 자동 수정안 생성 |

## 팁 (영상 내 반복 강조)

- "**날 것으로 주면 1%도 안 올라간다**" — 과거 자료(SQL이든 문서든)는 정리·구조화 후 제공해야 효과. 영상 내 두 번 강조 [04:28], [10:51]
- "**낡은 문서는 모델 한 개보다 빨리 성능을 죽인다**" — 스킬 문서를 코드와 같은 저장소에 두고 CI 훅으로 동기화 강제 [07:36]
