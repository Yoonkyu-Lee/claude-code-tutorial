## 한 줄 요약

## 핵심 개념

### 1. 영상의 출발점 — "이건 아직 에이전트 코딩이 아니다"
화자는 다른 개발자들이 보여주는 "창 여러 개 띄우고 병렬 실행하는 미친 코딩"을 동경하지만, 정작 본인의 일상 코딩은 ChatGPT를 왔다 갔다 하며 매번 사람이 개입하는 흐름이라는 자기 진단으로 시작.

> "거진 휴먼인더 룸이 거진 매번 들어가는 인간의 개입이 끝이 없는 내가 그런 에이전트 코딩을 하고 있다. 이런 거 살짝 에이전트 코딩이라 부르기엔 조금 그렇죠. 그죠?" [00:30]

[의견] 화자는 이 상태를 "진짜 에이전트 코딩"이라 부르기 어렵다고 보고, 그 격차를 메우기 위해 OhMyClaude Code를 소개한다고 프레이밍.

### 2. OhMyClaude Code 플러그인이란
Claude Code 위에서 동작하는 플러그인. 화자가 "기똥툴"이라며 들고 온 도구.

> "오마이클드 코드라는 플러그인입니다. 이게 여러분들 특히 초보자분들이 이거 쓰시면은 너무 좋습니다." [00:54]

핵심 장점 (화자 정리):
- **ChatGPT ↔ Claude Code 왕복이 사라진다**. 계획부터 실행까지 Claude Code 한 곳에서 끝남.
- **에이전트 오케스트레이션이 잘 갖춰져 있다**. 기능 하나 요청하면 "완성될 때까지" 멈추지 않는다.

> "그냥 얘 설치해서 쓰면은요. 내가 어떤 기능 만들어 줘 하면은 그냥 그거를 완성이 될 때까지 될 때까지 해요. 어. 그러니까 중간에 뭐 이렇게 에러나고 그런 거 없어. 그냥 그냥 될 때까지 무한 루프 자기네들끼리 핑퐁 치면서 막 하는 거야." [01:45]

[불명확: 화자가 참고한 "블로그 글"의 URL은 영상에서 노출되지 않음. "공식 문서는 아니고요. 그냥 제가 블로그 글이 좀 잘 정리가 돼 있어서 하나 들고 왔습니다." [01:25]]

### 3. 모드 강도 스펙트럼 (Mode Intensity Spectrum)
화자가 가장 중요하다고 강조한 개념. OhMyClaude Code가 제공하는 모드를 "에이전트한테 얼마나 강한 채찍질을 할지"의 스펙트럼으로 설명.

> "내가 얼마나 강력하게 에이전트한테 채찍질를 할 것이냐 여러분들이 결정할 수가 있거든요." [02:20]

스펙트럼(약 → 강) (자막에서 화자가 거론한 순서대로):
1. **단순 대화 (Chat)** — 그냥 묻고 답하기
2. **Plan (계획)** — 계획만 세우고 코드는 안 짬
3. **Autopilot (자율 실행)** — 플랜 → 개발 → 테스트까지 알아서 진행
4. **Ralph Mode** — 끝까지 멈추지 않음. 수용 기준 모두 통과할 때까지 반복
5. **Team Ralph Mode** — 에이전트들이 팀으로 서로 검증·핑퐁

> "랄프 모드 뭐다? 끝까지 멈추지 않는 모드. 절대 멈추지 않을 거예요. 완벽한 거만 전 갖고 올 거예요." [03:10]

### 4. Deep Interview 기능 — 초보자용 명세화 어시스턴트
화자가 "내가 좋아하는 기능"이라며 가장 시간을 들여 시연한 기능.

> "내가 단순히 나 개발 잘 못 해 가지고 나 디테일화 뭐 상세화 시키는 거 정말 못 하는 사람이야. 그런 사람들이 이거 쓰시면 너무 좋습니다." [03:53]

작동 방식 — 화자가 "쇼핑몰을 만들어 줘"라는 모호한 요청을 입력하면:
- 플러그인이 **모호성(ambiguity) 점수**를 계산. 100%면 아무것도 모름, 20% 정도로 떨어지면 개발 진입 가능.
- 떨어뜨리기 위해 **개발자가 클라이언트에게 취조하듯 질문을 던진다**.
- 질문 → 답 → 모호성 재계산 → 다음 질문의 루프.

> "이거를이 딥 인터뷰가 말 그대로 딥 인터뷰 당신을 인터뷰를 해 드립니다." [04:48]

화자는 이 경험을 "개발자와 일하는 경험"이라 부르며, 본인이 "클라이언트(갑)의 입장"이 되어 본다고 표현.

> "저는 지금 제가 굉장히 클라이언트 입장에서 어, 그래서 제가 상당히 갑의 입장에 있는 느낌입니다." [06:26]

### 5. 시연 — 쇼핑몰 명세화 인터뷰 단계별 모호성 감소
화자가 실제로 진행한 인터뷰 시퀀스와 그때그때 모호성 % 변화:

| # | 질문 요지 | 화자 답 | 모호성 |
|---|---|---|---|
| 1 | 상품 탐색·회원·결제가 만들고 싶은 게 맞나? | MVP만, 탐색·결제·회원가입 정도 | (시작) |
| 2 | 이 쇼핑몰을 만드는 본질적 목적은? (사업/프로토타입/데모/포트폴리오) | 포트폴리오 데모용 | (감소 시작) |
| 3 | 무엇을 파는 가게? (단일 브랜드/전자제품/일반 잡화 등) | 패션 의류 | 62% |
| 4 | 기술 스택은? (또는 추천 받기) | Next.js full-stack | 48% |
| 5 | 포트폴리오용인데 회원가입이 정말 필요? 게스트 결제로 충분하지 않나? | 게스트 결제만 | 41% |
| 6 | 데모의 "완성 기준"은? (핵심 end-to-end / 반응형 / 배포 / 깔끔한 코드 등) | 핵심 end-to-end | 27% |
| 7 | 실결제 안 붙을 텐데 결제 흐름 깊이는? (바로 주문 완료 / 배송지 폼 / 가짜 카드 UI) | 바로 주문 완료 | (임계값 도달) |

> "이거 굉장히 중요하죠. 그죠? 완성은 이제 뭐 핵심으로음 엔드투 엔드냐 아니면은 뭐 반응 디자인까지 돼야 되냐 뭐 이렇게 되어 있죠." [08:36]

화자 강조 포인트: **개발자에게 발주할 때 클라이언트가 미리 말해줘야 하는 항목들을 플러그인이 거꾸로 발굴해 준다**.

> "이렇게 나한테 내가 사실 말해 줘야 되는 거를이 친구가 찾아서 알아서 질문해 주는게 굉장히 좋지 않나요?" [06:23]

### 6. 명세 산출물 — `specs/` 디렉터리에 저장
인터뷰가 끝나면 그동안 주고받은 질문·답을 **시스템 요구 사항**으로 정리해 `specs/`에 저장. 이 파일이 그대로 다음 단계(개발)의 입력이 된다.

> "저희가 지금까지 했던 딥 인터뷰 질문을요. 여기에 또 다 저장을 해 놨습니다. 그래서 여기 스펙스에가 보시면요. 어, 우리가 했던 인터뷰 질문들을 이렇게 정리를 해 놓은 걸 보실 수가 있습니다. 그러니까 이게 이제 우리 시스템 요구 사항이 되겠죠." [08:46]

### 7. 인터뷰 후 분기 — Plan / Autopilot / Ralph / 더 인터뷰
명세 완료 후 플러그인이 다음 선택지를 제공:
- **Plan Mode**: 계획만 세우고 코드 작성은 멈춤
- **Autopilot**: 바로 개발 진행
- **Ralph Mode**: 다 만들어질 때까지 무한정 진행, 사용자는 지켜보기만
- **인터뷰 계속**: 모호성 18%(잔여)를 더 없애기

화자는 "더는 기다릴 수 없다"며 **Ralph Mode**로 진입.

### 8. Ralph Mode 실제 동작 — 리뷰어 에이전트 자동 분리
Ralph Mode 진입 후 영상에서 관찰된 것:
- 스토리들이 차례로 PASS 처리됨.
- **별도 Reviewer(verification) 에이전트가 자동으로 추가**되어, 메인이 만든 코드를 따로 검증.

> "메인에서는 메인은 코드를 만든 앤데 코드를 만든 애가 또 검증한다는게 좀 약간 말이 안 되잖아요. 그죠? 그래서 새로운 이제 검사하는 리뷰 에이전트를 하나 만들어 가지고 얘가 이제이이 코드를 검사할 수 있게끔 에이전트를 하나 새로 띄어 두신 거라고 보시면 됩니다." [10:00]

[의견] 화자는 이 자동 분리를 "에이전트를 굳이 직접 구성하지 않아도 되는" OhMyClaude Code의 핵심 효용으로 본다. 다운로드만으로 코드 생성 에이전트 ↔ 검증 에이전트 분업이 따라온다는 것.

수용 기준이 모두 통과할 때까지 아키텍처 검증과 함께 반복.

> "랄프는 수용 기준이 모두 통과할 때까지 어, 놓치는 거 하나 없이 모두 통과해야 됩니다. 아키텍처 검증과 함께 반복해서 이제 작업에 들어갑니다." [09:26]

## 카탈로그 — OhMyClaude Code 모드 선택지 (화자가 영상 후반에 언급)

| 모드 | 특징 | 화자가 든 추천 시나리오 |
|---|---|---|
| Chat | 단순 대화 | (스펙트럼 좌측 끝) |
| Plan Mode | 계획만 수립, 코드 X | 다른 도구에도 흔히 있음 |
| Autopilot | 플랜→개발→테스트 자율 실행 | **기능 하나 정도** 만들 때 |
| Ralph Mode | 수용 기준 통과까지 무한 반복 | **대규모 기능, 완료될 때까지 반복** 필요할 때 |
| Team Ralph Mode | 에이전트 팀 협업 + Ralph 반복 | **풀스택 팀 프로젝트** |
| UltraWork (병렬 모드) | 동시에 여러 작업(예: 5개 엔드포인트) 병렬 실행 | "다섯 개 엔드포인트 전부 구현해 줘"처럼 병렬화 가능한 작업 |
| Deep Interview | 모호성 점수 기반으로 질문해 명세화 | 초보자가 요구사항 디테일화 못 할 때 |

> "여기 보시면은 이제 라프 모드처럼 그냥 모든게 수용 기준이 통과할 때까지 쭉 반복하는 모드가 있는 반면에 또는 내가 병렬 실행 여러 개 하고 싶다. 그러면은 이제 어, 요런 울트라워크라는 요런 병렬 모드도 있네요." [10:29]

## 팁 (영상에서 드러난 실용 메모)

- **신규 플러그인 설치 후 `reload plugin` 실행 권장.** 화자가 인스톨 직후 한 행동.

  > "플러그인 새로 설치했기 때문에 한번 리로드 플러그인 요거 해 주면 좋습니다." [03:34]

- **설치 스코프 선택**: 화자는 "프로젝트 스코프" 설치를 시연. (전역/사용자 스코프 옵션이 더 있는지는 영상에서 다루지 않음.)
- **모호성 임계값**: 영상에서 관찰된 진입 임계값은 **약 20% 이하**. 27%에서 한 번 더 질문을 받고, 그다음 답으로 임계값 도달.
- **Ralph Mode는 "지켜보기"**. 인터뷰가 끝난 뒤 Ralph로 들어가면 사용자는 더 이상 개입하지 않고 결과를 기다리는 흐름이라고 화자가 묘사.

## 환경 셋업

영상에서 다룬 설치 단계 (구체적 명령은 영상에 명시되지 않고 블로그 글을 화면에 띄워 진행):
1. OhMyClaude Code 플러그인을 Claude Code에 추가 (블로그 글에 설치 절차 정리되어 있다고 화자가 언급).
2. 두 번째 플러그인 install 수행 (영상에서는 이름까지 명시되지는 않음. [불명확: 두 번째 플러그인의 정확한 이름은 자막상 식별 안 됨]).
3. 프로젝트 스코프(`project scope`)로 설치.
4. `reload plugin` 실행.
5. 명령 입력 시작 (예: `쇼핑몰을 만들어 줘`).

## 외부 보강
영상이 OhMyClaude Code의 공식 문서가 아닌 "잘 정리된 블로그 글"을 참조한다고 명시했고, 그 URL을 노출하지 않아 이 노트에서 별도 외부 보강을 수행하지 않았다. 사용자가 실제 설치 시점에 공식 저장소·문서를 직접 조회하는 편이 안전하다.
