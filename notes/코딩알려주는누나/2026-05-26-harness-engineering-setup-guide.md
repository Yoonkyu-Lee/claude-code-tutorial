## 핵심 개념

### 1. 하네스 엔지니어링이란 (화자 정의 발췌)
영상은 별도 정의 섹션을 두진 않고 시연이 끝난 뒤 한 문장으로 정리한다.

> "그 내가 하는 작업을 AI가 그대로 잘 할 수 있게 어, 그거 길을 잘 닦아 놓은 거. 요게 하네스 엔지니어링입니다." [17:16]

부연으로:
> "만약에 얘가 실수해도 다시 돌아올 수 있도록 이렇게 뭐 QA 같은 걸로 이렇게 좀 더 이렇게 깎아 주기도 하고요." [17:24]

### 2. 에이전트는 "인간의 뇌 프로세스를 잘게 쪼개 담아둔 것"
워크플로 단계 설명 중 즉흥적으로 나온 정의지만, 화자가 강조한 비유다.

> "에이전트가요? 사실 별거 없어요. 그냥 인간의 뇌를 담아둔 거라고 생각하시면 돼. 인간의 뇌에 있는 그 프로세스를요 하나 하나 잘게 잘게 쪼개서 담아둔 거를 어 에이전트라고 보시면 됩니다." [08:48]

### 3. 스킬 vs 에이전트의 차이 (컨텍스트 윈도우 관점)
영상 중반에 한 번, 후반에 한 번 — 같은 주제를 두 차례 다루며 강조한다.

**메인 vs 서브 에이전트 (시연 화면 기준):**
> "이 메인은 어쨌든 제가 제 가장 메인으로 쓰는 친구니까 여기 컨텍스트가 꽉 차면은 이제 다음 거 다시 새 A4용지 들고 와야 돼. 좀 아쉬운 건데. 근데 여기에서 이제 작업을 하고 결과만 쏴 주다 보니까 뭐다? 얘가 조금 더 오래 쓸 수 있다." [14:12]

> "이 설계하는 과정에 있는이 모든 컨텍스트는 메인은 필요 없는 거야. 메인은 결과만 알면 되니까." [13:15]

**스킬 vs 에이전트:**
> "스킬 같은 경우에는요. 이 메인에서 틀어서 쓰는 거고요. 그래서 이 메인에 이 스킬의 어, 컨텍스트가 다 쌓이게 됩니다. 근데 [에이전트][^a]를 틀면은 이 새로운 창이 하나 더 뜨는 거라고 생각을 하시면 돼요. 그렇기 때문에 여기에 또 컨텍스트가 새로 쌓이게 되는 거예요. 그래서 여기에서 결과만 받아볼 수 있게끔." [14:23]

[^a]: 자막 원문은 "리전트". 컨텍스트상 "에이전트"의 자동 자막 오인식으로 보여 보정.

**오케스트레이터를 스킬로 만든 이유 — 화자의 해석:**
> "[의견] 제가 봤을 때는이 오케스트레이터도이 스킬에 만들어 둔 이유는 오케스트레이터처럼이 메인에서 동작해야 되는 친구가 있잖아. 얘는 에이전트로 만들 필요가 없는 거예요. 그죠? 왜냐 스킬은 원래부터 원래가 원래가 메인에서 동작하는 애니까. 근데 뭐다? 우리가 구조 짜고 로직 짜고 UI 짜고 이런 거는 이제 다 이제 서브로 나갈 수 있는 개념들이잖아. 그러니까 걔네들은 뭐다? 에이전트로 만들어 준 거다." [15:10]

(이 한 문장이 영상에서 가장 중요한 설계 원칙 발화다 — "메인에서 직접 굴려야 하는 통제·라우팅 책임 = 스킬, 결과만 받으면 되는 전문 작업 = 서브에이전트".)

### 4. CLAUDE.md의 "트리거" 항목
플러그인이 자동 생성한 `CLAUDE.md`에서 화자가 가장 강조한 항목.

> "여기서 중요한게 뭐냐면요. 바로 이겁니다. 트리거. 어, 트리거. 내가 뭐 어, 뭐 만들어줘, 수정해 줘. 뭐 기능 추가해 줘. 이거 검증해 줘라고 할 때에 무엇을 사용하라고 해 놨냐? 지금 샵 아, 샵 빌더 오케스트라. 오케스트레이터 스킬을 사용하라고 지금 되어 있죠." [06:03]

→ 사용자의 자연어 요청을 받았을 때 어떤 스킬을 호출할지를 `CLAUDE.md`에 명시해두는 것이 하네스 작동의 출발점.

### 5. 스킬의 description 필드 = AI의 자동 호출 트리거
> "이 디스크립션이 굉장히 중요합니다. 이 디스크립션을 베이스로 AI가 얘를 자동으로 호출할 수가 있는 거거든요. 몰[쇼핑몰]을 뭐 만들어 줘, 제작해 줘, 뭐 페이지 추가해 줘, 뭐 등등 이런 걸 요청할 때에 이 친구를 부르세요라고 디스크립션에 굉장히 잘 써둔 걸 보실 수가 있죠." [06:42]

### 6. 팀 통신 프로토콜 (에이전트끼리 대화 규약)
> "팀 통신 프로토콜은요. 이 에이전트들끼리 어쨌든 대화를 해야 되잖아요. 얘가 제대로 했으면 어, 너 제대로 했어. 어, 나 이거 수정해 줄래? 에이전트끼리 대화를 할 수가 있어야 팀메이트로서 의미가 있는 거잖아요." [10:04]

각 에이전트 정의 파일에 "어떤 형태의 메시지를 누구에게 보낼지"를 명시한다. 예: 아키텍트는 UI 빌더에게는 "페이지 목록 / 파일 경로 / 디자인 토큰 / 구조"를, 로직 빌더에게는 "페이지별 필요한 동작 / 데이터 형태"를, QA에게는 "계약 문서"(검증 기준)를 보낸다.[10:21]

> "이 아키텍트 에이전트는요 뭐다 각각에게 주로 말하는 거예요. 야 UI는 요렇게 할 거야. 야 로직 이렇게 할 거야. 야 QA 너 내가 계획한 거 이거니까 딱 이대로 테스트해 줘. 이거를 다 뭐 한다? 샌드 메시지 메시지를 보내는 역할을 하는 에이전트다." [10:52]

## 시연 워크플로 (시간순)

### Step 1. 플러그인 설치 (마켓플레이스 등록 + 인스톨)
**대상 플러그인**: 카카오 개발자가 만든 `harness-marketplace` (영상에서는 GitHub 페이지만 보여주고 작성자 핸들은 시각적으로만 언급).[01:21]

> "바로 이 카카오 개발자분이 만드신 이 하네스 엔지니어링입니다. 어, 카카오 이이 하네스 엔지니어링 이제 플러그인을 만들어서 요거 설치하고 나서 하네스 구성해 줘라고 요청을 딱 하면은요. 그냥 하네스가 그냥 저르락 세팅이 됩니다." [01:21]

**설치 절차 — 영상에서 본 손동작 재구성:**
1. GitHub README에서 마켓플레이스 등록 명령 복사 → Claude Code에 붙여넣기 (영상에서는 명령어 텍스트가 클로즈업되진 않음).[02:07]
2. "마켓플레이스 한네스나 파운드"[^b] 에러가 나서 실수 발견 → 명령에서 끝부분을 `harness`만 적으면 안 되고 **`harness-marketplace`로 풀 네임을 넣어야 인식된다.**[02:28]
3. 설치 범위 선택: 유저 레벨 / 프로젝트 레벨 중 화자는 **프로젝트 레벨**을 선택.[02:45]
4. 설치 완료 후 `/reload-plugins`(영상 표현 "대시 리로드 플러그인스")로 플러그인 리로드.[03:00]

[^b]: 자막 원문 그대로. "harness not found"의 한국어 자막 오인식으로 추정.

> "여기 끝에 여기가 그냥 하네스가 아니라 하네스 대시 마켓플레이스로 넣어 주셔야 됩니다. 요렇게 해야 추가가 됩니다." [02:36]

### Step 2. 하네스 구성 요청
빈 프로젝트(코드 없음) 상태에서:

> "쇼핑몰 웹사이트를 만들려고 한다. 어 하네스를 구성해 줘" [03:09]

→ 엔터 하나로 `CLAUDE.md`, `.claude/skills/shop-builder-orchestrator/SKILL.md`, 4개 에이전트 정의 파일이 자동 생성된다.

화자 강조: **기존 프로젝트가 있다면** "쇼핑몰 만들려고 한다" 같은 도메인 설명을 안 붙이고 그냥 "하네스를 구성해 줘"만 해도 프로젝트 베이스에서 자동으로 도메인을 인식해 구성된다.[03:28]

### Step 3. 자동 생성 결과물 점검
영상은 생성된 산출물을 위에서 아래로 훑는다.

**CLAUDE.md** (목표/트리거/구성/실행 방법/변경 이력 섹션 포함) — 위 핵심 개념 §4 참조.[05:53]

**SKILL.md (shop-builder-orchestrator)** — 워크플로 정의:
- 페이즈 0: 컨텍스트 확인 (워크스페이스 존재 여부 체크 → 신규 vs 부분 수정 분기)[07:35]
- 페이즈 1: 사용자 요구 사항 분석 → `requirements.md`[^c]에 저장[08:13]
- 페이즈 2: 팀 구성[09:01]
- 페이즈 3: 아키텍트가 설계 → UI/로직 병렬 구현 → QA 검증 → 결과물 파일화 → 정리·보고[09:07]

[^c]: 자막 원문은 "requires.md"지만 표준 명명상 `requirements.md`로 보정.

화자 코멘트:
> "이 워크플로가 이 일하는 순서를 적어 놓은 거예요. 개발자로서 일하는 순서. 개발자도 뭐예요? 신규 개발 들어오면은 어 이거 기존에 있던 건가 아니면 진짜 새로운 건가? 한번 확인을 하고 그리고 나서 아 요구 사항 뭔지 분석하고" [08:32]

**4개 서브에이전트 카탈로그** (화자 설명 순):

| 에이전트 | 역할 | 주요 통신 | 영상 타임스탬프 |
|---|---|---|---|
| shop-architect | 데이터 구조 정의·쇼핑 설계 전문가 | UI 빌더·로직 빌더·QA에게 메시지를 **보내는** 라우팅 노드 | [09:38] [10:52] |
| shop-ui-builder | UI/화면 구성 (예쁘게) | 로직 빌더와 양방향 대화 + 아키텍트로부터 구조 수신 + QA로부터 피드백 수신 | [07:08] [11:07] |
| shop-logic-builder | 비즈니스 로직 | UI 빌더와 양방향 + 아키텍트로부터 요구 수신 | [07:14] |
| shop-qa-inspector | 테스트·검증 | 아키텍트로부터 "계약 문서" 수신 → UI/로직에 피드백 송신 | [07:24] [16:04] |

에이전트 정의 파일이 공통으로 가지는 섹션 (아키텍트 기준):[09:46]
1. 페르소나 ("에이전트엔 무조건 페르소나 넣어 줘야 되고요")
2. 핵심 역할
3. 작업 원칙
4. 입출력 프로토콜
5. 팀 통신 프로토콜

> "[의견] 에이전트엔 무조건 뭐다? 페르소나 넣어 줘야 되고요." [09:46]
(→ "무조건"이라는 단정적 단어 사용. 화자의 주관 규범으로 분리.)

### Step 4. 실제 요청 — "쇼핑몰 랜딩 페이지를 만들어 줘"
프롬프트 1줄 입력 후 시연되는 흐름:[11:51]

1. 워크스페이스 디렉터리 자동 생성[12:09]
2. `requirements.md` 자동 작성 (요청 사항 분석 결과)[12:19]
3. 메인 + 아키텍트(설계) 에이전트 2개가 동시에 떠 있는 상태가 보임 → 설계 완료 후 결과만 메인에 전달[12:47]
4. UI 빌더 + 로직 빌더 병렬 실행 → 메인 + 2개 서브 = 동시 3개 에이전트 화면[15:35]
5. QA 에이전트가 마지막에 실행되어 "치명적 결함 + 추가 내건" 잡아냄[16:04]
6. 워크스페이스에 최종 산출물 정리 — 유저 요청 / 계획 / UI 기획 / 로직 구현 / QA 발견 에러 / 수정 내역 / 총 정리[16:24]

> "어, 끝이 났고요. 이거에 대해서 이제 결과물로 뭐다? 이제 어, 코드도 지금 잘 쌓여 있는 걸 지금 보이시나요?" [16:13]

### Step 5. 다음 확장 방향 (화자 제안)
> "여기에서 이제 좀 더 들어가면은 이제 후[^d] 같은 거 이용을 해서 뭐 린트 검사라던가 뭐 빌드 검사라던가 이런 것도 여러분들이 어, 워크플로우에 추가를 해 줄 수 있겠죠." [17:31]

[^d]: 자막 원문 "후". Claude Code의 **Hook** 기능을 가리키는 음차로 추정.

> "여러분들이 한번 요거 설치를 해 보시고 여기에서 이제 내가 또 필요할 거 같은 워크플로 또 추가하거나 또 너무 많은 거 좀 덜어내고 요런 작업을 여러분들이 하시면서 여러분들만의 하네스 엔지니어링 만들어서 저희한테 공유 좀 해 주세요." [17:40]

## 화자가 반복 강조한 메시지

1. **"문장 하나로 한 세팅 완성"** — 진입 장벽 낮추기가 이 영상의 목적.[00:51]
2. **"내 상황과 안 맞는 코드는 공유해도 의미가 없다"** — 공유보다 도구 소개를 선택한 근거.[00:43]
3. **"메인에 컨텍스트를 안 쌓는 구조 = 더 오래 작업"** — 위 §3의 핵심.[14:12]
4. **"실수해도 돌아올 수 있게 QA로 깎아준다"** — 하네스의 회복 능력.[17:24]
5. **"AI가 빠르면 뭐 하냐, 내 PC가 느린데"** — 동시 작업(다중 에이전트, 프런트/백/DB 서버, 도커) 때문에 PC 성능이 결국 발목.[04:18] (단, 이 발언은 이어지는 협찬 코너로 직접 연결됨 — 아래 별도 섹션 참조)
