## 핵심 개념

### 1. Claude Code — "옆자리에 앉은 비서"

> "클로드 코드란 옆자리에 앉은 비서입니다. 터미널에서 만나는 AI 비서입니다." [01:53]

화자가 제시한 4가지 활용 영역:
- **자료 조사**: 인터넷 검색 → 정보 정리 → 보고서
- **문서 작성**: 보고서, 이메일, 기획서 등 구조화된 문서 자동 작성
- **브레인스토밍**: 다양한 관점의 아이디어 제안·발전
- **데이터 정리**: 고객 데이터 분류, 평균값 계산, Excel/CSV 저장

> "개발자가 아니어도 괜찮습니다. 터미널은 그저 AI와 대화하는 창구일 뿐입니다." [02:32]

### 2. Claude Code의 3단계 작업 흐름

> "제가 한국어로 말하면 클로드가 알아서 실행을 하고 결과를 내가 확인하는 구조를 가지고 있습니다." [02:50]

1. **자연어 요청**: 일상 한국어로 원하는 것 설명
2. **분석·실행·검색**: Claude Code가 파일 분석, 명령어 실행, 인터넷 검색
3. **결과 승인**: 결과물을 사용자에게 보여주고 승인 요청. "괜찮으면 오케이 한 마디면 됩니다"

> "이 모든 세션에서 필요한 것은 딱 두 가지입니다. 클로드 코드와 파이썬 언어입니다." [03:28]

Claude Code는 한국어 명령을 받아 코드를 작성·실행하는 AI 엔진, Python은 Claude Code가 짜주는 코드의 언어. 사용자는 Python을 읽거나 쓸 일이 없다고 화자는 설명.

### 3. 프롬프트 = "AI에게 주는 업무 지시서"

> "신입비서에게 일을 시킬 때 '이거 알아서 잘해 줘'라고만 하면 결과가 어떻게 나올까요? 엉뚱한 결과가 나오기 쉽습니다. 프롬프트는 무엇을 어떤 환경에서 어떻게 어떤 형식으로 받고 싶은지까지 정리해 주는 지시서입니다." [04:01]

#### 좋은 프롬프트 vs 나쁜 프롬프트

| 좋은 예시 [04:20] | 나쁜 예시 [04:33] |
|---|---|
| "네이버 메일에서 계약서 검색해서 상위 열 건을 마크다운 보고서로 만들어 줘" | "메일 자동화 해 줘" |

> "AI가 무엇을 할지 추측을 해야 되고 이 추측을 기반으로 해 가지고 동작을 해야 되기 때문에 자기 마음대로 행동을 할 확률이 높습니다." [04:38]

#### 좋은 프롬프트의 4요소

> "지시, 맥락, 작업, 출력 이 네 가지 부분에 대해 가지고 명확해야 된다고 생각합니다." [05:09]

| 요소 | 설명 | 예시 |
|---|---|---|
| **지시(instruction)** | 무엇을 할지 | "카페24 어드민에서 어제 주문 긁어줘" |
| **맥락(context)** | 환경과 전제 | "Windows 크롬 디버그 모드로 이미 로그인 환경에서 해줘" |
| **작업(task)** | 구체적 단계 | "주문 페이지 들어가서 어제 필터, CSV 다운로드" |
| **출력(output)** | 산출물 형식 | "MD처럼" |

#### 화자가 제시한 통합 프롬프트 예시 [06:16]

> "내가 크롬 브라우저를 9222번 포트에 열어 놨어. 이걸 이용해서 어떤 특정 주소에 들어가서 네이버 메일에서 계약서라고 적힌 메일들을 각각 요약해서 보고서로 작성해 줘. document.md로 저장하면 돼. 이때 사람인 척하기 위해 각 단계별로 2에서 10초 정도 랜덤으로 쉬어 줘."

이 한 문단에 4요소가 모두 들어있음을 화자가 분해해 보여줌:
- 맥락: 9222번 포트 크롬 (이미 로그인된 디버그 모드 가정)
- 지시: URL + 키워드 (계약서)
- 작업: 각 메일 요약 + 마크다운 저장
- 출력: document.md
- 추가 맥락(봇 디텍션 회피): 2~10초 랜덤 대기

### 4. 왜 "진짜 크롬"을 쓰는가

> "브라우저 자동화의 본질은 봇 디텍션 회피입니다. 진짜 사람으로 보이는 게 핵심이죠." [07:54]

> "사이트는 의심을 합니다. 자바스크립트 환경, TLS, 행동, IP 전부 검사합니다. 그렇기 때문에 저희는 진짜 크롬으로 위장을 해야 되는데요. 본인이 평소 쓰는 크롬, 프로필, 쿠키, 로그인 그대로 자동화에 빌려서 쓴다고 보시면 될 거 같습니다." [08:04]

연결 방식: **CDP 9222번 포트**.

> "이 친구는 진짜 브라우저를 외부에서 조작하는 리모컨이라고 보시면 될 거 같습니다." [08:23]

### 5. 기본 용어 (비유 사전)

#### IP — "컴퓨터의 집 주소"

> "택배를 보내려면 받는 사람의 집 주소가 필요합니다. 인터넷에서도 데이터를 보내려면 받는 컴퓨터의 집 주소가 필요합니다. 이게 IP 주소입니다." [08:56]

- 예: `142.250.66.78` — 구글의 한 서버
- `localhost` = "내 컴퓨터 자신을 가리키는 약속된 이름". 외부에 알리지 않고 내 컴퓨터 안에서만 통하는 별명
- 형식: 숫자 4개로 구성 (예: `192.168.0.1`)
- 오늘 실습 주소: `http://localhost:9222`

#### 포트 — "컴퓨터의 호실 번호"

> "한 건물 안에 여러 사무실이 있다고 상상해 보세요. 건물 주소(IP)만 가지고는 어느 사무실로 갈지 모릅니다. 그래서 호실 번호가 필요합니다." [09:53]

- 포트 번호 범위: 0 ~ 65535
- 예약된(이미 쓰이는) 포트: 80, 443, 25, 587, 22 등 — 이런 포트는 사용 불가
- 약속된 자동화 포트: **9222** ("크롬 자동화 전용 사무실"로 우리가 약속한 번호)
- 9222 포트 보안: "본인 PC만 들어올 수 있게 셋업 가능. 외부 PC는 차단" [10:50]

#### CDP — "크롬의 리모컨 규격"

> "TV마다 리모컨이 다르면 불편합니다. 그래서 이 신호를 보내면 채널이 바뀐다는 약속이 있습니다. CDP도 같은 개념인데요. 크롬에 이런 형식으로 신호를 보내면 페이지를 클릭, 이동, 스크롤 한다는 공식 규격입니다." [11:09]

- 정식 명칭: Chrome DevTools Protocol (크롬 개발자 도구 프로토콜)
- **어태치(attach)**: 이미 떠 있는 크롬에 붙어서 조작하는 행위. 새 창을 띄우지 않음
- 안전한 이유: 이미 로그인된 세션을 그대로 사용 → **새 로그인, 2FA, OTP 등을 건너뜀** [11:37]

## 시나리오: 네이버 메일 자동화 5단계 [12:42]

화자가 제시한 케이스 스터디 — 이미 로그인된 크롬에 어태치해 2FA/OTP를 우회한다는 전제.

1. **CDP 어태치**: 이미 떠 있는 크롬의 9222번 호실에 자동화 도구가 노크
2. **메일 진입**: `naver.com` → 메일 링크 → 받은편지함 페이지 진입
3. **키워드 검색**: 검색창에 "계약서" 엔터, 결과 로딩 대기
4. **상위 N건 추출**: 발신자, 제목, 날짜, 미리보기 4개 필드를 자동 수집
5. **보고서 생성**: `네이버메일리포트.md` (마크다운) 자동 작성·저장

> "동일 패턴으로 Gmail이나 아웃룩 웹, 사 그룹웨어 메일도 가능합니다. 키워드와 도메인만 교체하시면 됩니다." [13:17]

### 예시 프롬프트 [13:25]

> "9222번 포트에 크롬 브라우저를 켰어. 이 브라우저를 조작해서 여기 주소에 들어가서 메일 검색에 '넥슨'이라고 치고 검색되는 결과를 가져오는 파이썬 스크립트 작성해 줘."

화자는 "한 번 만들면 평생 재사용이 가능합니다. 이미 로그인된 실제 크롬에 어태치를 하는 거기 때문에 새 로그인 시도를 하지 않아도 됩니다" [14:10]라고 강조.

## 보트 디텍션 4단계 — 공항 입국 심사 비유 [14:35]

> "봇 디텍션은 사이트의 보안 검사 대비대입니다. 공항 입국 심사를 떠올려 보세요. 여권, 표정, 동선을 종합해서 이 사람을 들여보내도 되는지 판단합니다." [14:47]

[불명확: "보안 검사 득배 때입니다" — 자막 오인식, 문맥상 "검사대"로 추정]

사이트가 막는 이유 [15:06]:
- 가격 정보 무단 수집
- 회원 가입 자동화
- 표 매크로
- DDoS

> "유명 제품 — 클라우드플레어, 데이터돔, 퍼미터X, 아카마이 — 이런 AI 사이트가 도입해서 방어를 합니다. 걸리면 캡차, IP 블랙리스트, 심한 경우 계정 영구 정지까지 당하는 거 같습니다." [15:20]

[자막 정정] "클라우드 프렌드 플레어" → Cloudflare, "데이터이나" → DataDome, "퍼미터 X" → PerimeterX, "아카마이" → Akamai.

### 단서 1: 여권 검사 (TLS·HTTP 헤더)

> "사이트가 누가 노크했는지 가장 먼저 보는 표지입니다. 진짜 크롬인지 파이썬 requests인지 그런 것들을 확인한다고 보시면 될 거 같습니다." [16:15]

- **TLS 핸드셰이크**: 사이트와 처음 연결될 때 브라우저가 자기 신원 정보를 보냄. TLS = Transport Layer Security, 인터넷 통신 암호화 약속
- 걸리는 패턴: Python requests, 빈 User-Agent, 헤더 순서가 크롬과 다름
- **HTTP 헤더**: User-Agent, Accept-Language 등 — "요청에 따라붙는 명함"

> "근데 이 여권 검사 같은 경우에는 저희 케이스에서는 딱히 고려를 안 하셔도 될 거 같습니다." [17:12]

[의견] 화자는 진짜 크롬을 그대로 쓰는 본 강의 시나리오에서는 TLS·헤더 단계가 자동으로 통과되므로 신경 쓰지 말라고 안내.

### 단서 2: 소지품 검사 (JavaScript 런타임)

> "사이트가 페이지 로드 후 슬쩍 가방을 열어보는 단계라고 보시면 될 거 같습니다. 자동화의 흔적이 묻어 있는지 이런 것들을 체크한다고 보시면 될 거 같아요." [17:19]

> "navigator.webdriver라는 변수가 켜져 있으면 '자동화 도구가 켰구나'라고 즉시 알아챕니다. 해상도, 폰트, 플러그인, WebGL 같은 것도 다 검사 대상입니다." [17:46]

[자막 정정] "내비게이터 웹드라이버" → `navigator.webdriver`, "알아ract니다" → "알아챕니다", "웹지" → WebGL.

### 단서 3: 걸음거리 (행동 패턴)

> "사이트도 마우스 좌표, 클릭 좌표, 타이핑 간격을 기록합니다. 봇은 보통 직선 이동, 일정한 0.1초 타이핑, 클릭 후 0.5초/5초 만에 다음 페이지 — 너무 매끄럽고 너무 빠르다고 보시면 될 거 같고요. 진짜 사람은 곡선으로 움직이고 잠깐 멈추고 오타도 칩니다." [18:35]

[자막 정정] "씨근땀, 서두음" → 자연스러운 한국어로는 "땀, 서두름" 추정 [불명확: 원문 정확치 않음]

회피 전략 [19:05]:
> "클로드한테 요청을 하실 때 '각 단계별 2에서 10초 랜덤 대기, 마우스는 곡선으로 움직여' 이런 것들을 요청을 하시는 게 좋을 것 같습니다."

### 단서 4: IP·계정 평판

> "데이터 센터에서 들어온 트래픽은 기업이 만든 IP라서 즉시 의심합니다. 반면 가정용 인터넷 IP는 진짜 사람으로 인식합니다." [19:28]

| 구분 | 예시 | 사이트 인식 |
|---|---|---|

## 외부 보강 (자발적)

영상이 용어를 비유로만 설명하고 끝나는 부분에 대해 공식 출처로 짧게 보강 — 영상 본문과 분리되어 있고 출처가 명시됨.

- **Chrome DevTools Protocol**: 공식 문서는 `https://chromedevtools.github.io/devtools-protocol/`. 영상의 "리모컨 규격" 비유와 일치. Domain(Page, Network, Runtime 등) 단위로 명령·이벤트가 정의되어 있고, `--remote-debugging-port=9222` 옵션으로 크롬을 띄우면 외부 클라이언트가 WebSocket으로 어태치 가능.
- **`navigator.webdriver`**: W3C WebDriver 표준에 정의된 boolean 속성. 자동화 도구(Selenium, Playwright 기본 launch)가 `true`로 설정하기 때문에 봇 탐지에 즉시 활용됨. 영상에서 언급된 "내비게이터 웹드라이버"가 이 속성.
- **봇 디텍션 솔루션**: Cloudflare Bot Management, DataDome, Akamai Bot Manager, PerimeterX(현 HUMAN Security)가 영상에서 언급된 4개 — 모두 영상의 4단계 단서(TLS 핑거프린트, JS 런타임, 행동 분석, IP 평판)를 종합해 점수를 매김.
