### 바이브 코딩이 잘 굴러가는 전제 3가지
화자가 명시적으로 강조한 핵심. 아무 스택에서나 잘되는 게 아니라는 주장.

> "바이브 코딩은 아무 기술 스택에서나 잘되는 게 아닙니다. AI가 잘 아는 프레임워크를 써야 하고, AI가 읽기 좋은 코드 구조여야 하고, 컨텍스트 윈도우 안에 핵심 코드가 다 들어가야 해요." [00:42]

1. **AI가 잘 아는(=많이 학습한) 프레임워크**
2. **AI가 읽기 좋은 코드 구조**
3. **컨텍스트 윈도우 안에 핵심 코드가 다 들어가는 크기**

> "이걸 모르고 아무 스택이나 쓰면 바이브 코딩이 아니라 그냥 AI한테 삽질을 시키는 겁니다. 1년 동안 그걸 몸으로 배운 거예요." [00:55] `[의견]`

### 왜 타입(Type)이 핵심인가
프론트·백 양쪽 결론을 관통하는 화자의 핵심 논거.

> "AI는 타입이 있는 언어에서 훨씬 강합니다. (...) 타입이 없으면 AI가 추론을 해야 돼요. '이 변수가 스트링인지 넘버인지 모르겠는데 일단 이렇게 해 볼게' 하면서 코드를 짜거든요. 타입이 있으면 함수 시그니처만 보고 '아 이건 이렇게 하면 되겠다' 확신을 가지고 코드를 짜요." [01:50]

> "바이브 코딩 관점에서 TypeScript는 AI한테 지도를 쥐어 주는 거랑 같아요. 타입이 곧 문서이자 가이드거든요." [02:09] `[의견]`

## 팁 — "타입을 빡세게 달아라"
화자가 "이거 모르면 FastAPI 있어도 바이브 코딩 제대로 안 됨"이라며 가장 강조한 부분.

> "Python은 타입을 안 달면 AI가 헤매요. (...) Python에도 타입 힌트가 있거든요. 근데 선택 사항이에요. 안 달아도 돌아가요. 문제는 타입 없는 Python 코드를 AI한테 주면 '이 함수가 뭘 받고 뭘 리턴하는지 모르겠는데' 이러면서 엉뚱한 코드를 짭니다." [05:45]

- **나쁜 예**: 그냥 `dict`로 데이터를 넘김 → AI가 내용물을 추측해야 함
- **좋은 예**: `name: str`, `email: str`, `age: int`처럼 명시 → "AI가 보고 바로 이해" [06:03]
- FastAPI + Pydantic 시너지: "요청 바디를 Pydantic 모델로 정의하면 자동으로 검증도 되고 API 문서도 자동 생성돼요" [06:24]

> "타입을 빡세게 다는 게 귀찮다고 느낄 수 있는데, 바이브 코딩에서는 그 타입이 AI한테 주는 설명서예요. 설명서가 좋으면 AI가 일을 잘하는 거고, 설명서가 없으면 AI가 헤매는 거예요." [06:34] `[의견]`

## 외부 보강

영상이 인용한 수치 중 검증 가능한 항목을 공식/2차 출처로 교차 확인했다. (영상 본문과 분리)

- **FastAPI vs Flask 처리량 (15,000~20,000 vs 2,000~3,000 RPS)**: TechEmpower 벤치마크 기준으로 FastAPI가 약 15,000~20,000 req/s, Flask가 약 2,000~3,000 req/s로 보고되며, 영상이 인용한 "5~10배" 차이와 정합한다. 단 DB·Redis가 끼는 혼합 워크로드에서는 양쪽 모두 DB에서 병목이 걸려 차이가 줄어든다는 단서가 붙는다. (출처: [FastAPI 공식 Benchmarks](https://fastapi.tiangolo.com/benchmarks/), [Strapi: FastAPI vs Flask 2025](https://strapi.io/blog/fastapi-vs-flask-python-framework-comparison))
- **SQLModel 정체성**: SQLModel은 FastAPI 제작자(Sebastián Ramírez)가 만든 ORM으로, SQLAlchemy + Pydantic을 통합해 DB 모델과 API 스키마의 중복 정의를 없앤다 — 영상 설명과 일치. (출처: 동일 비교 자료)
- 위 보강은 "성능·정체성" 사실 확인용이며, 영상이 인용한 Stack Overflow 순위·기업 수·CrewAI 투자/매출 수치는 원본 출처 확인이 안 되어 검수 대상으로 남긴다.
